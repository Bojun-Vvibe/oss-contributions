## Summary
When running `codex-exec` with the legacy workspace-write sandbox profile
(no active permission profile), the command-line `--add-dir` writable roots
were silently dropped from the thread-lifecycle config sent to the runtime.
On macOS in particular this means a `codex-exec --add-dir backend frontend`
invocation could write to `frontend` (cwd) but not `backend`, with no error
message — just sandbox denials at write time. This PR rebuilds the
"sandbox_workspace_write" override block from the resolved
`FileSystemSandboxPolicy` and threads it into both
`thread_start_params_from_config` and `thread_resume_params_from_config`.

## Notes
- The early-return-when-active-profile branch is correct: if the user has
  selected a structured permission profile, that profile carries its own
  roots through a different code path. Doubling them here would diverge.
- `exclude_tmpdir_env_var` / `exclude_slash_tmp` are computed as the
  *negation* of the special-entry presence — i.e., if no writable
  `Tmpdir` special entry exists, exclude `$TMPDIR` from writable.
  Correct, but easy to mis-read; a one-line comment by each derive
  would help future maintainers.
- Test is `#[cfg(not(target_os = "windows"))]` — fine, since this is
  specifically about the macOS exec sandbox path. A follow-up could add
  Linux Landlock coverage.

## Verdict
`merge-after-nits` — add a clarifying comment on the negated

covering Linux as well as macOS in the lifecycle test.
