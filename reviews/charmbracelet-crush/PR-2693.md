# Context

`internal/agent/tools/mcp/init.go` builds the `exec.Cmd` for
stdio-transport MCP servers from a `config.MCPConfig`. The
config has three string-shaped surfaces a user can write:
`command`, `args` (a string array), and `env` (a map).
Today `command` is run through `home.Long(...)` for `~`
expansion *and* through the same `resolver.ResolveValue` that
expands `$VAR`-style env-var references. The `env` map has
its own `ResolvedEnv()` accessor that does the same. The
`headers` map (for HTTP MCP servers) has `ResolvedHeaders`.

`args` is the only field that ships a user string into a
subprocess unmodified.

# Problem

A user writing:

```json
"vestige": {
  "type": "stdio",
  "command": "vestige-mcp",
  "args": ["--data-dir", "$VESTIGE_DB"]
}
```

reasonably expects `$VESTIGE_DB` to be substituted from
their environment, the way `command` and `env` already are.
Today that expectation is wrong — the literal string
`"$VESTIGE_DB"` is passed to the child as argv[2]. The user
sees confusing errors from the MCP server about a directory
called `$VESTIGE_DB` that doesn't exist. Closes #2684.

# Design

```go
resolvedArgs := make([]string, len(m.Args))
for i, arg := range m.Args {
    resolved, resolveErr := resolver.ResolveValue(arg)
    if resolveErr != nil {
        slog.Error("Error resolving MCP arg", "error", resolveErr, "arg", arg)
        resolvedArgs[i] = arg
        continue
    }
    resolvedArgs[i] = resolved
}
cmd := exec.CommandContext(ctx, home.Long(command), resolvedArgs...)
```

The resolver is the same `config.VariableResolver` already
threaded into `createTransport`. On error the original arg is
preserved and the failure is logged. This matches the
`ResolvedHeaders` precedent the PR cites. Eleven lines, one
file. The PR is exactly the size it needs to be.

# Risks and edges

- **Logging-without-failing changes operator visibility.** A
  resolver error today (e.g. `$NOT_A_VAR` with strict-mode
  resolver) will silently log and pass the literal arg
  through. The MCP server then fails downstream with a
  cryptic error and the operator has to find the slog line
  to understand why. The `ResolvedHeaders` path has the same
  shape so consistency is the right choice here, but if
  strict resolution were ever desirable, an opt-in
  `MCPConfig.StrictResolve bool` would let CI catch missing
  vars at startup. Out of scope, worth noting.
- **`ResolveValue` is not specified in the diff.** Reviewer
  needs to confirm: does it expand only `$VAR` and `${VAR}`,
  or does it also expand `~` (which `home.Long` is doing for
  command), or does it run shell expansion (globs, command
  substitution)? Answer matters because `args` are passed to
  `exec.CommandContext` directly — they do *not* go through
  a shell — so shell-style expansions (globs, command
  substitution, redirections) would be silently honored
  before exec but not after, creating a confusing inverse-
  parity with users' shell experience. The likely answer
  (based on the resolver's purpose) is var-only, which is
  fine.
- **No `home.Long` on args.** The PR resolves env-vars but
  does not expand `~/...` paths in args. So
  `args: ["--data-dir", "~/data"]` will still pass the
  literal `~/data`. The PR scope is env-var expansion only,
  which is consistent with the issue title, but the
  asymmetry with `command` (which gets both resolve + home)
  is now codified. A follow-up should add `home.Long` to
  args too — same reasoning, same precedent. Cheap to do
  here actually:
  ```go
  resolvedArgs[i] = home.Long(resolved)
  ```
  with a one-liner test. The PR author should be asked.
- **Pre-allocated slice + index assignment is correct.** No
  `append` race, no off-by-one. Good Go.
- **`continue` after the error is structurally clear** —
  `resolvedArgs[i] = arg` is set before the continue so the
  slot is never zero-string. Good.
- **No new tests.** The PR notes "existing MCP tests pass"
  and "manually verified that args without `$` are passed
  through unchanged". Both true but neither is a regression
  test for the actual fix. A 5-line test that sets an env
  var, runs `createTransport` against a config with that
  var in args, and asserts the resulting `cmd.Args` contains
  the expanded value would be the right floor. Two more for
  the unset-var fallback path and the `${VAR}` syntax (if
  supported) would be ideal.
- **No tests for the error path either.** If `ResolveValue`
  returns an error, the original arg is preserved — but
  what *triggers* that error in `ResolveValue` is opaque
  from this diff. Reviewer should look at the resolver
  implementation to know what to test.
- **`slog.Error` for a recoverable user-config issue is
  arguably too loud.** It's user-config-level — wrong env-
  var name, missing `~/.envrc` entry — not a system
  failure. `slog.Warn` would be more honest about who's at
  fault. Minor.
- **Performance comment in the PR ("no performance impact
  for the common case") is correct** — `ResolveValue` on a
  string with no `$` is essentially a single `strings.
  Contains` check. The pre-allocated slice keeps the
  allocation profile flat too.

# Verdict

Approve. The fix is exactly right, the precedent is sound
(matches `command`/`env`/`headers` resolution), and the
scope is correctly minimal. Two requests before merge:

1. Add a regression test that sets an env var, calls
   `createTransport` (or factor the resolution into a
   testable helper), and asserts the resulting argv.
2. Either expand `home.Long` to args in this PR (one line)
   or open a follow-up issue noting the asymmetry.

Optional: downgrade the resolver-error log to `slog.Warn`.

# Skim path

1. `init.go` — confirm the `ResolveValue` import path
   matches what `command` uses two lines above.
2. `internal/config` — find `ResolveValue` and confirm it
   is var-only (not shell-expanding). If it does shell
   expansion, the security review widens considerably
   because args become an injection surface.
3. `internal/agent/tools/mcp` test files — confirm there is
   not already a test fixture that constructs `MCPConfig`
   with `args`, which would be the right place to extend.
