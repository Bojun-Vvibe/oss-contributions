## Summary

Replaces direct `tea.SetClipboard(d.userCode)` calls in the OAuth dialog (`internal/ui/dialog/oauth.go`) with the existing `common.CopyToClipboard` / `common.CopyToClipboardWithCallback` helpers. The helpers presumably wrap the Bubbletea clipboard primitive with a fallback path for terminals that don't speak OSC 52 (xterm clipboard escape) — typically rendering a "press y to copy"-style prompt or shelling out to `pbcopy`/`xclip`/`wl-copy`.

## Verdict: `merge-after-nits`

The change itself is right (centralizing clipboard handling through a single helper is the obvious correct factoring), but the diff is incomplete to evaluate without seeing `common.CopyToClipboard`'s implementation. From the diff alone the only assertion I can make is "this PR delegates to a shared helper," which is good but not sufficient as a review.

## Specific references

- `internal/ui/dialog/oauth.go:350-355` (`copyCode`) — the old form was `tea.Sequence(tea.SetClipboard(d.userCode), util.ReportInfo("Code copied to clipboard"))`. The new form is `common.CopyToClipboard(d.userCode, "Code copied to clipboard")`. Whether the new helper still emits the success toast on failure (which would be a regression) depends on the helper; please confirm `common.CopyToClipboard` only fires the success message after the clipboard write succeeds, otherwise users on non-OSC52 terminals will see "Code copied to clipboard" and then nothing be on their clipboard.
- `internal/ui/dialog/oauth.go:357-371` (`copyCodeAndOpenURL`) — same delegation, with the browser-open `func() tea.Msg` passed as the third positional arg to `CopyToClipboardWithCallback`. The relative ordering matters: in the old `tea.Sequence`, the order was `SetClipboard → openURL → ReportInfo`. The new helper signature `CopyToClipboardWithCallback(content, message, callback)` likely runs callback after the copy; please confirm the message-vs-callback ordering matches the original (i.e. browser opens *then* toast appears, not the other way around). If the helper reverses that, the user gets a "Code copied and URL opened" toast a frame before the URL actually opens — minor UX regression but worth checking.
- The `util` import (used by the deleted `util.ReportInfo` calls) likely becomes unused in this file after the diff. The Go compiler will catch that, but worth scanning the imports block to keep the diff `goimports`-clean.
