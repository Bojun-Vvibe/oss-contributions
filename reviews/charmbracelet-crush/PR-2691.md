# Context

Crush runs a per-language LSP client in-process so the agent can
surface diagnostics (compile errors, type errors, lints) into
prompts — the model prefers "fix this concrete error on line 47"
over "figure out what's wrong". The client speaks standard
LSP-over-stdio and keeps a map of request-id → pending `chan
Response` so that async replies can be routed back to callers.

Diagnostics are pulled two ways: the pull-based
`textDocument/diagnostic` (LSP 3.17+) and the push-based
`textDocument/publishDiagnostics` notifications. This PR only
touches the pull path.

# Problem

When a buffer closes (user opens another file, the agent's edit
loop moves on), Crush calls `client.CloseBuffer(uri)` which
sends `textDocument/didClose`. The pending
`textDocument/diagnostic` request for that URI is, however,
**not cancelled**. Two things happen:

1. The LSP server keeps computing. For rust-analyzer and
   gopls this can be 100s–1000s of ms of CPU on a project the
   user no longer cares about.
2. When the response eventually arrives, the router delivers
   it to the pending channel, which may already have been
   garbage-collected if the caller used a bounded-lifetime
   context. If the caller is still alive but has moved on,
   stale diagnostics flash into the UI for a file the user
   isn't looking at anymore.

The bug report attached to the PR describes exactly this:
pasting a gopls-using project into a fresh session and
bouncing through three files produces diagnostics for file 1
appearing on the status line after file 3 has been opened.

# Design

The fix wires up `$/cancelRequest` (LSP's standard cancel
verb) into `CloseBuffer`:

```go
func (c *Client) CloseBuffer(ctx context.Context, uri DocumentURI) error {
	c.mu.Lock()
	pending := c.diagReqByURI[uri]
	delete(c.diagReqByURI, uri)
	c.mu.Unlock()

	for _, reqID := range pending {
		_ = c.notify(ctx, "$/cancelRequest", cancelParams{ID: reqID})
	}

	return c.notify(ctx, "textDocument/didClose", didCloseParams{TextDocument: TextDocumentIdentifier{URI: uri}})
}
```

With a matching bookkeeping change in `requestDiagnostics`:

```go
reqID := c.nextID()
c.mu.Lock()
c.diagReqByURI[uri] = append(c.diagReqByURI[uri], reqID)
c.mu.Unlock()
```

Four things done right:

1. **Cancel before close.** The order matters. If you send
   `didClose` first some servers (gopls in particular) treat
   the document as gone and either error the diagnostics
   response or silently drop it without responding at all.
   The response-never-arrives case leaks a pending channel.
   Cancelling first lets the server return a normal "request
   cancelled" error, which the router handles correctly.
2. **Best-effort notify.** `_ = c.notify(...)` — cancel is
   fire-and-forget by LSP spec. Erroring out on a failed
   cancel would block `CloseBuffer`, which is called from
   hot paths.
3. **Per-URI id bucket, not per-request.** Multiple pending
   diagnostics requests per URI can exist (the agent can
   re-request during edit flurries). Tracking them as a
   slice and cancelling all of them on close is correct.
4. **Lock released before I/O.** The mutex is dropped before
   the notify calls. Otherwise a slow server would hold the
   client lock for the duration of a network round trip.

# Risks

- **Server compliance.** `$/cancelRequest` is SHOULD in the
  spec, not MUST. Some smaller LSP implementations ignore it.
  For those servers, this PR reduces the cost from "CPU
  spent + UI flicker" to "CPU spent, UI silent" because the
  router now discards the late reply (the URI is no longer
  in `diagReqByURI`, so the response can't match). That's
  strictly better, but worth noting: it does not *stop the
  CPU*, only hides the result.
- **Response router cleanup.** I don't see the matching
  change in the response router — when a cancelled request
  does reply with "cancelled" error, the pending
  `chan Response` needs to be closed or selected-into,
  otherwise the goroutine that's waiting on it leaks. The
  PR adds the bookkeeping *removal* in `CloseBuffer` but
  not the *drain* on the caller side. If callers use
  `ctx.Done()` as their escape, they're fine; if any caller
  is blocked on an unbuffered receive, this PR could
  introduce a leak. Worth a grep before merge.
- **Test coverage.** The new test verifies that
  `CloseBuffer` sends a `$/cancelRequest` for each
  outstanding URI. It does not verify the order (cancel
  before didClose) nor the cleanup of `diagReqByURI`. Two
  more 5-line assertions would lock both.
- **Concurrent close races.** Two goroutines closing the
  same buffer simultaneously would both read the same
  slice, both send cancels (harmless duplicate), and both
  try to delete the map key (harmless idempotent). Safe,
  but a `sync.Once` or a `LoadAndDelete`-style swap would
  be slightly cleaner.

# Suggestions

1. Add an order-of-operations assertion in the test:
   record the sequence of `notify` calls and assert the
   cancel arrives before the didClose.
2. Consider a generalized `cancelPendingForURI(uri)`
   helper and wire it to `textDocument/didChange` too —
   any in-flight diagnostics on a URI that's just been
   edited are stale by definition.
3. Document the fire-and-forget nature of cancel in a
   package doc comment. Future contributors will want to
   know why the returned error is discarded.

# Verdict

**Ship it.** Small, correctly ordered, fixes a real
user-visible bug. The nit about response-router cleanup
should be verified before merge, but isn't a blocker if
all current callers are context-scoped.

# What I learned

LSP clients have a shape that shows up in many other async-
request-over-long-lived-transport systems: Jupyter kernels,
MCP servers, LiteLLM proxies, Language Model provider
streams. The invariant you have to maintain is *every
pending id either receives a response or is explicitly
cancelled, exactly once, before the lifetime scope ends*.
If you track pendings by request-id alone, cleanup-by-scope
(user closed buffer, context expired, session ended)
becomes an O(N) scan. Tracking pendings by *scope key*
(here, the URI) makes cleanup O(1) per scope. The pattern
generalizes: whenever you find yourself scanning a pending
map to cancel things, you have an indexing bug.
