# Context

Two semi-independent issues bundled into one PR: (a) corrupted
sessions when tool results don't match any tool call, and (b)
rogue `</think>` tags leaking into session titles. Both are
agent-flow integrity issues but live in different layers.

# Problem

**Primary: orphaned `tool_result` blocks.** The Anthropic-style
conversation invariant requires `tool_use` and `tool_result`
blocks to come in matched pairs. When the persistence layer
serializes a conversation that contains a `tool_result` with
no matching `tool_use` — which can happen when history is
trimmed, a rollback is partial, or a streaming buffer is
truncated mid-write — the next API call rejects the
conversation and the session is bricked.

This PR is the *mirror image* of PR #2622 in this same repo.
That one handles "orphaned `tool_use` (call without result)";
this one handles "orphaned `tool_result` (result without
call)." Together they enforce the pairing invariant in both
directions.

**Secondary: `</think>` tags in titles.** Crush auto-generates
short titles for sessions, often by asking the model to
summarize the first user turn. Reasoning-model outputs (DeepSeek
R1, Claude with extended thinking, etc.) emit `<think>...</think>`
blocks that the parsing layer sometimes leaves in. The result:
session list shows literal `</think> blah blah` as a title,
which is both ugly and a sign the upstream stripping is
incomplete.

# Design

The two fixes occupy two files (plus a routine
`go.mod`/`go.sum` bump for the upstream
`charmbracelet/fantasy#202` dependency that this PR requires):

**`internal/agent/agent.go` (+54/-1)** — the orphan-result
handling. Symmetric to PR #2622: walk persisted history, find
`tool_result` blocks whose `ToolCallID` isn't present in any
preceding `tool_use`, and either (a) drop them or (b) inject
a synthetic preceding `tool_use`. From the description
("validate tool call/results"), it's most likely a
drop-on-validate strategy: the orphan result reflects no real
call, so synthesizing a call would be inventing history.
Dropping is honest.

**`internal/agent/tools/fetch.go` (+1/-1)** — the title-tag
strip. A one-line regex/string change to strip `</think>`
(and presumably `<think>...</think>` blocks) from the
generated title. Trivial fix.

**`go.mod`/`go.sum`** — bump `charmbracelet/fantasy` to a
version containing PR #202, which presumably exposes the
APIs needed for the validation logic above.

The interleaving of two unrelated fixes in one PR is a
soft norm violation — they should ideally be separate
commits at minimum, separate PRs ideally — but the
secondary fix is one line and bundling it is forgivable.

# Risks

- **Drop semantics rewrite history.** If the orphan
  `tool_result` came from a legitimate-but-partial state
  (say, the matching `tool_use` was somehow lost in a
  rollback), dropping it loses information. The user
  doesn't see the result anymore. The trade-off is
  reasonable — losing one cell of history is much better
  than losing the entire session — but a debug log
  ("dropped orphan tool_result for unknown call ID xxx")
  would help future diagnosis.
- **No symmetry test against PR #2622.** Both PRs landed
  in the same window and address the two halves of the
  same invariant. A combined property test ("after
  validation, every `tool_use` has a matching
  `tool_result` and every `tool_result` has a matching
  `tool_use`") would lock both directions in. Without
  it, a future contributor who refactors the validation
  could re-introduce one direction's bug while preserving
  the other's tests.
- **`</think>` strip is regex-specific.** Different
  reasoning models emit slightly different tags
  (`<thinking>`, `<reasoning>`, `[REASONING]`, etc.). A
  single-tag fix paints over the symptom. A more general
  "strip any `<X>...</X>` block whose tag isn't in an
  allowlist" would harden against the next model's tag.
- **Upstream dependency coupling.** Requiring
  `charmbracelet/fantasy#202` means this PR can't merge
  before that one. Standard cross-repo dependency
  pattern but worth flagging in CI gating.

# Suggestions

1. Split the title-strip fix into its own PR with its
   own test. One line of code shouldn't ride along with a
   session-recovery fix; the two have completely
   different review concerns.
2. Add the property test described above:
   `forall conversations c: validate(c) preserves the
   pairing invariant in both directions`.
3. Generalize the title strip: instead of stripping
   `</think>` specifically, strip any leading XML-ish
   tag block from the model output before using it as a
   title. The next reasoning model with a different tag
   gets covered for free.
4. Logging on drop: each time a tool_result is dropped,
   log the call ID and the surrounding context (turn
   number, file). Cheap, makes the next bug report
   actionable.
5. Cross-reference PR #2622 in the commit message so the
   pair is discoverable in `git log` archaeology.

# Verdict

**Ship it.** Bundling notwithstanding, both fixes address
real bugs, the primary fix is the right symmetric companion
to the orphaned-`tool_use` direction, and the upstream
dependency on `fantasy#202` is a clean way to share the
validation primitive. The combined system (this PR + #2622)
should fully enforce the pairing invariant.

# What I learned

When two PRs land that address two halves of one invariant,
that's the right moment to write the *invariant* as a test,
not just each half independently. Otherwise the maintenance
burden grows linearly with the number of half-tests, and
future refactors can break one direction without touching
the test of the other. Also: bundling unrelated fixes is a
small cost when both are tiny, but it pays a hidden tax in
review attention — reviewers split focus, follow-up
discussion threads get tangled, and the eventual git
archaeology is harder. The right default is one fix per PR;
the right exception is "they share a dependency bump" or
"they're both one-liners and reviewing them together saves
time."
