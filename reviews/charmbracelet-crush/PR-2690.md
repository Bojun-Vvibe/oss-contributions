# Context

`internal/db` opens a single shared `*sql.DB` against an
on-disk SQLite file in WAL mode. Crush spawns sub-agents
through `coordinator.runSubAgent`; each writes session,
message, and cost rows on its own goroutine. Go's
`database/sql` defaults `MaxOpenConns` to unlimited, so each
concurrent sub-agent's writes can grab a fresh underlying
connection — i.e. a fresh SQLite handle on the same file.

In WAL mode, multiple writer handles do not actually parallelize
writes (SQLite serializes at the file level via the reserved
lock), but they *do* each observe the WAL independently and
can each trigger an auto-checkpoint. When the process is
killed mid-checkpoint (Ctrl-C, OOM, parent crash), the WAL
header and main-DB header can desync, and the next open of
the file fails with `SQLITE_NOTADB (26)` — "file is not a
database". Closes #2682.

# Problem

Two questions for the reviewer:

1. Does limiting `MaxOpenConns` to 1 actually prevent the
   race, or just narrow it?
2. Is the `_txlock=immediate` change defense-in-depth or load-
   bearing on its own?

The PR's claim is that (1) is the structural fix and (2) is
belt-and-suspenders. I think that's right — see analysis.

# Design

Three small changes:

1. **`connect.go`**: `db.SetMaxOpenConns(1)` after the
   driver-specific `openDB`. This forces `database/sql` to
   queue all callers through one pooled connection, which in
   turn means one underlying SQLite handle. Sub-agents
   serialize on the connection-pool semaphore instead of on
   SQLite's own lock-arbitration.
2. **`connect_modernc.go`**: adds `_txlock=immediate` to the
   query string so all transactions begin with `BEGIN
   IMMEDIATE` instead of `BEGIN DEFERRED`. IMMEDIATE acquires
   the reserved-writer lock at transaction start; deferred
   waits until the first write statement and may collide with
   another writer that has already acquired the reserved lock.
3. **`connect_ncruces.go`**: same `_txlock=immediate`, but
   also adds the `file:` URI prefix so ncruces' DSN parser
   actually sees the query parameters (without the prefix the
   driver treats the whole string as a path).

# Risks and edges

- **`MaxOpenConns(1)` makes reads serialize too.** In WAL
  mode SQLite supports concurrent readers + one writer; this
  change collapses that to one-at-a-time everything. Crush's
  workload is write-heavy and per-session, so the contention
  budget is dominated by writes anyway, but a future feature
  that does parallel read-only queries (e.g. a "search across
  sessions" UI) will lose all parallelism. Worth a comment in
  the file flagging that the next person who wants reader
  parallelism needs to reintroduce a reader pool with
  `OpenDB` returning a separate read-only handle.
- **`SetMaxIdleConns` is not also set.** Default for
  `database/sql` is 2 idle, but capped at `MaxOpenConns`. So
  the effective idle is 1 here, which is fine. Just worth
  noting to anyone who later raises `MaxOpenConns` without
  also raising idle.
- **`_txlock=immediate` changes failure surface for read-
  only transactions.** Drizzle/`sql.Tx` started without a
  write intent will now still acquire the reserved lock
  upfront, blocking any other transaction from starting. In
  the single-connection world this matters less (only one
  transaction at a time anyway), but if `MaxOpenConns` ever
  goes back up, read-mostly transactions will newly serialize
  with writes. Not a correctness issue, but a perf cliff if
  the pool cap is ever relaxed.
- **The `_txlock=immediate` rationale in the modernc comment
  cites "deferred-to-writer upgrade deadlocks"** — this is
  the well-known SQLITE_BUSY pattern when two deferred txns
  both upgrade to writer and one has to lose. With
  `MaxOpenConns(1)` there is only one transaction at a time,
  so the upgrade-deadlock is structurally impossible. The
  `_txlock=immediate` change therefore has no observable
  effect *under the new pool cap*. It's belt-and-suspenders
  for a future where the cap is relaxed. Worth saying so in
  the commit message — currently both changes are presented
  as load-bearing for the same bug.
- **The ncruces DSN change requires the `file:` prefix.**
  The PR correctly notes this. But: the existing
  `Connect(ctx, dataDir)` builds a `dbPath` that is a plain
  filesystem path, and the ncruces driver previously opened
  it directly. Now it gets `file:<dbPath>?_txlock=immediate`
  — which means any path containing characters that need URI
  encoding (spaces in `dataDir`, on macOS/Windows) will
  break. The fix is `url.PathEscape(dbPath)` or equivalent
  before formatting. Probably out-of-scope for this hotfix
  but a real bug for anyone with a space in their data dir.
- **The fix does not address the original WAL corruption
  files in the wild.** Existing crush installs that already
  have a desynced WAL+DB header will still fail to open
  after this PR — the recovery path is `rm -f db.db-wal
  db.db-shm` (with the main DB intact) or restoring from
  backup. A graceful-degradation path that detects
  `SQLITE_NOTADB` on open and rebuilds from scratch with a
  loud log line would be a kind follow-up.
- **No regression test for the original race.** The bug is
  hard to reproduce deterministically — the failure window
  is ~ms wide and depends on the OS scheduler killing mid-
  checkpoint. A stress test that spawns N goroutines
  hammering writes while a separate goroutine sends SIGKILL
  to the test process at random intervals could catch
  regressions. Heavy, but the bug class deserves it.

# Verdict

Approve. Both changes are correct and small; the diagnosis
matches SQLite's documented failure mode for unbounded WAL-
mode concurrency. Recommendations:

1. Move the `_txlock=immediate` rationale to "defense in
   depth for any future relaxation of the pool cap" so future
   maintainers don't think it's load-bearing for this bug.
2. Add a code comment over `SetMaxOpenConns(1)` flagging
   that read concurrency is intentionally lost and what the
   recovery path is if you want it back.
3. Open a follow-up issue for graceful-degradation on
   `SQLITE_NOTADB` for already-corrupted installs.

# Skim path

1. `connect.go` — confirm `SetMaxOpenConns(1)` runs *before*
   `PingContext`, not after, so the ping itself respects the
   cap.
2. `connect_ncruces.go` — confirm the `file:` prefix and the
   `?_txlock=immediate` are syntactically correct for the
   ncruces driver's parser. `url.PathEscape(dbPath)` would
   harden this against path-with-spaces.
3. `git log issue/#2682` — confirm the reproduction matches
   the "concurrent sub-agents + interrupted checkpoint"
   pattern, not some other corruption vector this PR
   wouldn't address.
