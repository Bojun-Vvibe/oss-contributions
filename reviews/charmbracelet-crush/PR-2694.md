# Context

`internal/skills/skills.go`'s `DiscoverWithStates`
walks a list of root paths looking for `SKILL.md`
files. Each match is registered, parsed, and surfaced
to the UI sidebar. The walk uses `fastwalk` with
`Follow: true`, which means symlinked directories are
descended into.

A common operator pattern is to symlink
`~/.config/crush/skills` → some project's
`.claude/skills/` directory so that one set of skill
definitions is shared across multiple agents. The
project's own scan also picks up `.claude/skills/`
directly. Both roots end up walking the same physical
tree. Closes #2683.

# Problem

The dedup `seen` map is keyed on the *logical* path
that `fastwalk` reports. When the user-config root
and the project root both reach the same physical
`SKILL.md` via different logical paths, the two
strings hash differently, neither hits the existing
key, and both insert. The skill registry happens to
dedup by `name` later, so commands work fine, but
the sidebar renders one row per `SkillState` — and
states are not name-deduped. The user sees every
skill twice.

The PR's framing of why `crush_info` looks correct
but the sidebar is wrong is exactly right and worth
quoting: "the skill registry deduplicates by name
internally, but the sidebar rendered one row per
`SkillState` entry without deduplicating." That
diagnosis is the actual fix-locator — knowing that
two surfaces dedup at different layers tells you
the real bug is in the discovery layer's set
membership, not in either rendering layer.

# Design

```go
realPath := path
if resolved, resolveErr := filepath.EvalSymlinks(path); resolveErr == nil {
    realPath = resolved
}
mu.Lock()
if seen[realPath] {
    mu.Unlock()
    return nil
}
seen[realPath] = true
mu.Unlock()
```

Resolve every candidate `SKILL.md` to its canonical
path before checking the dedup set. If
`EvalSymlinks` fails (dangling symlink, permission
issue), fall back to the raw path so behavior is at
worst no-worse-than-before. Three new fields, one
new branch, a fallback. Tight.

The accompanying test
(`TestDiscoverDeduplicatesSymlinks`) builds a real
dir, builds a symlink to it from a different parent,
calls `Discover` with both roots, and asserts the
skill list and the normal-state list each have
length 1. That's the right shape — it's the actual
operator scenario reproduced minimally, and it would
have caught the bug pre-fix.

# Risks and edges

- **`EvalSymlinks` reads the filesystem.** Once per
  candidate `SKILL.md`. In practice that's bounded
  by the number of skills (small, dozens), so the
  perf cost is negligible. Worth knowing it's not
  free if a future refactor ever pushes this loop
  to thousands of entries.
- **Race window between resolve and stat.** Between
  `EvalSymlinks` and `Parse`, the underlying file
  could be replaced (TOCTOU). Not new — the prior
  code had the same window between dedup and parse.
  Not exploitable in any meaningful sense for skills
  discovery (you'd be racing your own filesystem).
- **`EvalSymlinks` collapses the entire path,
  including parent components.** That's the right
  behavior here — the goal is to identify the *file*
  uniquely. But it means two distinct `SKILL.md`
  files that happen to have intermediate-component
  symlinks will still be correctly distinguished
  (different inodes after full resolve).
- **The fallback to raw path on
  `EvalSymlinks` error means a dangling-symlink
  skill file can still be double-counted.** That's
  an acceptable degradation — a dangling symlink
  during the walk would also cause `Parse` to fail
  for one of the two entries, so the visible
  duplication would self-resolve in practice.
- **Symlink loops.** `fastwalk` with `Follow: true`
  is the documented hazard here — an
  `a/b -> ../a` loop would walk forever before this
  PR. After this PR it still walks forever
  (`EvalSymlinks` on a loop returns an error, so
  the fallback path lets the loop continue).
  `fastwalk` itself has loop detection (it tracks
  visited inodes) so the practical risk is low,
  but a comment in `DiscoverWithStates` noting
  "loop protection is delegated to fastwalk" would
  help future readers.
- **Inode-based dedup vs path-based dedup.** A
  more aggressive version would `os.Stat` and key
  on `(dev, ino)`. That's strictly stronger
  (handles hardlinks too) but introduces another
  syscall and a platform-specific
  `syscall.Stat_t` cast, and hardlinked
  `SKILL.md` files are vanishingly rare. Path-
  canonicalization is the right choice for the
  cost/benefit here.
- **Test is `Not parallel`.** Comment says it
  "shares global broker with other Discover
  tests." That's an existing pattern in the file,
  consistent. Worth noting that the global broker
  is itself a code smell (test-isolation boundary
  through global state), but out of scope.
- **Test asserts `normalStates == 1` and
  `len(skills) == 1`.** Both the right invariants.
  Missing one assertion that would harden the
  test: that the *path* recorded for the surviving
  skill is the canonical (real) one, not the
  symlinked one. If a future refactor flipped the
  fallback so the symlinked path won the race, the
  current test would still pass but the UI would
  show the wrong path. One more `require.Equal`
  on the resolved path would close that.
- **Concurrency.** The walk is parallel
  (`fastwalk` defaults). `mu` already guards
  `seen`. The new resolve happens *outside* the
  lock, which is correct (it's the hot path and
  doesn't touch shared state) and matches the
  existing pattern of "do read-only work under no
  lock, then take lock for the membership
  check/insert." Good Go.

# Verdict

Approve. Bug diagnosis is correct, fix is the
canonical one, test is the right shape. One
optional improvement before merge: extend the test
to assert the surviving entry's path is the
resolved one. Otherwise ready as-is.
