# PR: fix(ui): Skill display as loaded despite being disabled in config

## Summary

The skill status panel was iterating `m.skillStates` and the global skill registry without consulting `Options.DisabledSkills` from config, so a user who explicitly disabled a skill still saw it rendered as "online" in the UI. This patch builds a `disabledSet` once per call and skips matching entries in both loops, with a unit test covering the disabled case.

## Specific observations

- `internal/ui/model/skills.go:61-66` — the new `disabledSet` is built behind a `m.com != nil && m.com.Workspace != nil` guard and then `cfg := m.com.Config(); cfg != nil`. Good defensive style for a UI render path that can be entered before the workspace is fully wired. Nit: `make(map[string]bool, len(...))` would shave one allocation if `cfg.Options.DisabledSkills` is non-empty; not worth blocking on.
- `internal/ui/model/skills.go:79-81` and `:102-104` — disabled-skip is applied in both the `skillStates` loop and the registry loop. Without the second one, a disabled built-in skill that has no live state would still render. Catching both paths is correct.
- `internal/ui/model/skills_test.go:60-81` — the test seeds `DisabledSkills: []string{"go-doc", "crush-config"}` and asserts neither name appears in the returned items. It only asserts absence; it does not assert that other skills are still present. A second positive-case bullet (one disabled, one enabled, expect length 1 with the enabled name) would harden this against a future regression that overzealously filters everything.
- The fix uses `string` map keys against `state.Name` (with the `filepath.Base(filepath.Dir(state.Path))` fallback at `:75-77`). If a user's `DisabledSkills` config entry is the directory-derived name rather than the manifest `Name`, the fallback path is still matched correctly because `name` is computed before the disabled check. Worth a one-line comment so the next reader does not "simplify" by moving the check above the fallback.

## Verdict

`merge-after-nits`

## Reasoning

The behavior is correct, the change is small and localized, and the test prevents the obvious regression. The two nits — asserting positive presence in the test and a one-liner comment about why `name` must be resolved before the disabled check — are the kind of polish a maintainer would normally drop in a follow-up commit. The PR is already merged, which matches that posture; in pre-merge review I would have asked for the positive-case test before approving.
