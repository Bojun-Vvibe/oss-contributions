# Context

Crush persists conversation history across sessions. The
Anthropic API enforces a strict invariant: every assistant
message containing a `tool_use` block must be followed by a
user message containing a matching `tool_result` block. This
invariant is what allows the model to follow the
"call → result → reasoning" loop reliably across turns.

Earlier (commit c41a1a0e) crush added
`filterOrphanedToolResults` to handle one direction of
violation: a `tool_result` with no matching `tool_use`. That
shape arises from history pruning or partial deserialization.

# Problem

The inverse direction was unhandled. When a session is
interrupted mid-tool-call — Ctrl-C, network timeout, OOM kill
on the tool subprocess — the assistant message gets persisted
*with* the `tool_use` block, but the tool execution never
completes, so no `tool_result` is recorded. On resume, the
Anthropic API rejects every subsequent message with:

```
invalid_request_error - tool_use ids were found without
tool_result blocks immediately after: toolu_xxx
```

The session is permanently bricked. The user's only recourse
is "start a new session," losing the entire context. This is
exactly the failure mode that destroys trust in long-running
agent tools — the user feels they did nothing wrong (Ctrl-C is
a normal action) and yet the work is gone.

The bug is structural: the persistence layer captured the
intermediate state of a two-phase commit (`tool_use` written,
`tool_result` not yet written) and the resume path didn't
know how to repair it.

# Design

The fix lives in `preparePrompt()` in `internal/agent/agent.go`.
After appending an assistant message to the conversation
history, walk every `tool_use` in that message. If the
corresponding `tool_result` ID is not present in the next user
message, inject a synthetic one:

```go
fantasy.ToolResultPart{
    ToolCallID: tc.ID,
    Output: fantasy.ToolResultOutputContentError{
        Error: errors.New("Tool call was interrupted and did not produce a result."),
    },
}
```

Three things right about this design:

1. **Synthesizes a real `tool_result` rather than dropping the
   `tool_use`.** Dropping the call would technically satisfy
   the API but would also rewrite history — the model would
   no longer "see" that it had attempted the call. Keeping
   the call and adding an error result preserves the truth:
   "you tried to do X, it didn't complete, here's why." The
   model can decide whether to retry, route around, or give
   up.
2. **The error message is honest.** "Tool call was
   interrupted and did not produce a result" tells the model
   exactly what happened. Anything more vague risks the model
   re-issuing the same call into the same condition.
3. **Symmetric to the existing fix.** Same code path, mirrored
   logic: `filterOrphanedToolResults` removes orphan results;
   the new code injects synthetic results for orphan calls.
   Together they enforce the invariant in both directions.

Test budget: 141 LOC against 47 LOC of production change. That
ratio is right for a session-recovery fix — the cost of a
regression here is "all your sessions break again," and the
tests are how the fix becomes permanent.

# Risks

- **Synthetic error may confuse a re-tool-using model.** The
  model sees `Tool call was interrupted` and may immediately
  retry the same call. If the underlying cause was network
  timeout, retry is sensible; if it was a Ctrl-C because the
  user wanted to stop, retry is exactly wrong. There's no
  way to distinguish from the persisted record. Worth
  considering whether to include a hint to the user in the
  next assistant turn ("the previous tool call was
  interrupted; would you like to retry?").
- **No upper bound on synthetic results.** A pathological
  history with N orphan `tool_use` blocks gets N synthetic
  results. Almost certainly fine in practice (tool calls
  per turn rarely exceed 5–10), but a sanity bound with a
  log line on overflow would harden the contract.
- **Coupling to Anthropic's specific shape.** The fix is
  motivated by Anthropic's strict invariant; other providers
  (OpenAI, Bedrock) may have different rules. If they
  *don't* require the matching pair, the synthetic results
  are harmless extra context. If they require it
  *differently* (e.g., not "immediately after" but "anywhere
  in the conversation"), the fix may need provider-specific
  variants. Worth a follow-up audit.
- **Manual reproduction was not checked off.** The PR has
  `[ ] Manual: interrupt a tool call mid-execution, resume
  session, verify conversation continues` as unchecked. A
  reviewer should run that path before merge — the unit
  tests verify the synthesis logic but not the end-to-end
  resume flow.

# Suggestions

1. Surface the recovery to the user. After a successful
   resume that injected synthetic results, print a one-line
   notice: "Recovered N interrupted tool calls from previous
   session." Visibility builds trust.
2. Distinguish "interrupted" from "errored" in the synthetic
   result. If a future signal can tell us why the call
   didn't complete (Ctrl-C vs. timeout vs. crash), reflect
   it in the error message so the model can route
   accordingly.
3. Property test: for any randomly-generated conversation
   history, after `preparePrompt`, every `tool_use` ID has a
   matching `tool_result` ID. Locks the invariant beyond the
   specific scenarios in the unit tests.
4. Consider running the symmetric pair (filter + inject) as
   a single `repairConversation()` step with a shared
   contract, so the next provider-specific quirk has one
   place to land.

# Verdict

**Ship it** (after the manual reproduction is verified). The
fix is structurally sound (symmetric to the existing direction),
preserves user history truthfully, has dense test coverage, and
addresses a high-severity recovery failure that would otherwise
silently destroy user sessions.

# What I learned

When a persistence layer can capture half of a two-phase write
(here: `tool_use` written, `tool_result` not yet written), the
resume path needs explicit repair logic for every possible
half-state. The bug class is "every API invariant the producer
assumes will hold across writes can be violated by an
interruption between writes." The fix is twofold: (a) at
write-time, atomically pair the writes if possible (often hard
because tool execution is unbounded); (b) at read/resume-time,
detect violations and repair them honestly — preferring
"synthesize a truthful record of what happened" over "delete
the partial record and pretend nothing happened." The honest
record gives the consumer (here: the model) the information it
needs to recover gracefully.
