# Context

LLM agents that can only emit text get stuck the moment they
need a decision they can't reasonably make alone. They either
guess (and frustrate the user with bad assumptions), or stop
and ask in prose ("which approach would you like?"), at which
point the user has to type a free-form response that the model
then has to re-parse. Both flows are bad UX.

This PR adds an `ask_user_questions` tool to crush. The model
emits a structured tool call: array of questions, each with a
short prompt, an array of options, and a `multi_select` flag.
Crush displays a dialog (the existing dialog system handles
rendering), the user clicks options, and the answers go back
to the model as a JSON `[{id, selected}]` payload that the model
can correlate with its original questions. The whole round-trip
is one tool call from the model's perspective.

The implementation is appropriately layered: a `questions`
package owns the request/response types, a tool wraps the
service call, a dialog renders the UI, and the
`pubsub.Broker` pattern (used elsewhere in crush, see PR 2663)
plumbs questions from the agent goroutine to the UI goroutine.

+1014/-5 across 17 files. Big surface area but most of it is
boilerplate (UI scaffolding, workspace plumbing, tests).

# Diff walkthrough

**The tool itself is tiny** — 43 lines:

```go
func NewAskUserQuestionsTool(questionsService questions.Service) fantasy.AgentTool {
    return fantasy.NewAgentTool(
        askUserQuestionsToolName,
        string(askUserQuestionDescription),
        func(ctx context.Context, params AskUserQuestionParams, call fantasy.ToolCall) (fantasy.ToolResponse, error) {
            sessionID := GetSessionFromContext(ctx)
            req := questions.NewQuestionsRequest(sessionID, call.ID, params.Questions)

            resp, err := questionsService.Ask(ctx, req)
            if err != nil {
                return fantasy.ToolResponse{}, err
            }

            jsonResp, err := json.Marshal(resp.Answers)
            if err != nil {
                return fantasy.ToolResponse{}, err
            }
            return fantasy.NewTextResponse(string(jsonResp)), nil
        },
    )
}
```

The tool is a thin adapter: pull session ID from context,
build a request, call `service.Ask` (which blocks until the
user answers or context cancels), marshal the answers back as
JSON. Almost no logic in the tool layer; that's correct — the
interesting state lives in the service.

**The tool description** (markdown, embedded via `//go:embed`):

```
Ask user questions to gather information, clarify ambiguity, taking decisions,
determining user preferences and/or taste. When in doubt, use this tool to ask the user.

<usage>
- Provide array of questions to ask user
- Each question needs:
  - UUID to correlate originating question and answer
  - short question text to ask user
  - array of options from which user will select one or more answers
  - boolean indicating if user can answer selecting multiple options
...
<tips>
- If you recommend specific options, put them first in array of options and append "(Recommended)" to label
- To let user pick something else, offer "None of the above" option
- When planning
  - use this tool to clarify requirements or choose between approaches BEFORE finalizing your plan.
  - IMPORTANT: do not ask for feedback about the plan while you are working on it, because user cannot see it until you send it back 
```

This prompt is *good*. Several specific design choices worth
calling out:
1. **The "(Recommended)" convention** — by giving the model a
   concrete way to surface its preference within the structured
   format, you preserve the model's reasoning ("I think A is
   best, but the user might prefer B") without forcing it back
   into prose.
2. **The "None of the above" suggestion** — anticipates the
   "model offered three options, none fit" failure mode and
   gives the model a pre-canned escape hatch.
3. **The "do not ask for feedback about the plan while you
   are working on it"** — addresses a subtle UX bug where a
   model could ask "is this plan good?" while still mid-tool-
   call, and the user can't see the plan until the response is
   posted. That's the kind of guidance you only write *after*
   you've watched a model actually do this and confuse a user.

The fourth observation: this is one of those prompts where
the *limitations* section is as valuable as the *usage*
section. "Keep question and option labels short" + "If
option's description is too long, it will be truncated" tells
the model what *not* to do, which is often more efficient than
specifying everything it should do.

**The service** is the architecturally interesting bit:

```go
type questionsService struct {
    *pubsub.Broker[QuestionsRequest]
    pendingRequests *csync.Map[string, chan QuestionsResponse]
}

func (s *questionsService) Ask(ctx context.Context, req QuestionsRequest) (QuestionsResponse, error) {
    ch := make(chan QuestionsResponse, 1)
    s.pendingRequests.Set(req.ID, ch)
    defer s.pendingRequests.Del(req.ID)

    s.Publish(pubsub.CreatedEvent, req)

    select {
    case <-ctx.Done():
        return QuestionsResponse{RequestID: req.ID}, ctx.Err()
    case resp := <-ch:
        return resp, nil
    }
}

func (s *questionsService) Answer(res QuestionsResponse) {
    if !res.IsComplete() {
        slog.Warn("Incomplete response - missing answers", ...)
    }
    if ch, found := s.pendingRequests.Get(res.RequestID); found {
        ch <- res
    } else {
        slog.Warn("Received answers for unknown questions", ...)
    }
}
```

This is the *request-reply over pubsub* pattern. `Ask`:
1. Creates a per-request reply channel.
2. Stashes it in `pendingRequests` keyed by request ID.
3. Publishes the request (any UI subscriber sees it).
4. Blocks on either (a) context cancel or (b) reply channel.
5. Cleans up its slot on return (`defer Del`).

`Answer` is the symmetric side: look up the request ID, push
the response into its reply channel.

This is the correct shape for "fire-and-forget + reply" over
a fan-out broker, and it shows the author understood the
pattern rather than reinventing it. A few specific touches:
- `chan QuestionsResponse, 1` (buffered) means `Answer` doesn't
  block if the asker has already cancelled and is about to
  cleanup — the send goes into the buffer, the channel is
  GC'd when `pendingRequests.Del` runs.
- `csync.Map` for the pending registry — concurrency-safe
  without explicit locking.
- The `defer Del` survives panic. If `Ask` panics for any
  reason, the slot is still cleaned up.
- The "received answers for unknown questions" warning catches
  the case where `Answer` arrives after a context cancel — a
  diagnostic, not a panic.

One subtle issue: in `Answer`, if the channel send `ch <- res`
is sent but the asker has *just* cancelled and `Del`'d, the
buffered slot holds the response until GC. Fine. But if there's
a way `Ask` could put two requests in flight with the same ID
(uuid.New() collision is essentially impossible, so this is
hypothetical), the `ch` slot would be racy. The UUIDv4
assumption holds; worth a one-line comment.

**The `IsComplete` check** is interesting:

```go
func (res *QuestionsResponse) IsComplete() bool {
    return len(res.Answers) == cap(res.Answers)
}
```

This relies on the response being constructed with
`make([]Answer, len(req.Questions))` so `cap == len == numQuestions`.
Then `SetAnswerAt(idx, ...)` mutates in place without extending.
If the dialog ever uses `append` on `res.Answers`, the `cap`
invariant breaks. Fine for the current flow but worth a
defensive comment on `QuestionsResponse.Answers`.

# Risks

1. **`Ask` blocks the agent goroutine indefinitely.** Until
   the user answers (or context cancels), the agent is parked.
   For a long-running unattended session, the model could ask
   a question and then sit there for hours. Worth a per-call
   timeout *or* a UI affordance ("agent is waiting for you").
   The blocking is intrinsic to the design — request-reply —
   but the duration deserves UX consideration.
2. **No question-arrived notification when crush is
   backgrounded.** If the user has the TUI in another
   workspace/tmux pane, they could miss the question for a
   long time. A bell, a desktop notification, or a "session
   blocked" status indicator would help.
3. **The dialog's pubsub subscription has its own lifetime.**
   If the UI dialog goroutine restarts (re-render), there's
   a window where a question is published with no subscriber.
   `pubsub.Broker` typically buffers per-subscriber, but the
   subscriber has to exist *before* publish for buffering to
   help. Worth a test for "publish during UI restart".
4. **The model can ask N questions per call** (the param is
   `[]Question`). A misbehaved model could ask 100 questions
   at once. The dialog presumably renders them in some
   sensible way, but no soft cap exists. A `len(questions) > 10`
   warning or auto-paginate would help.
5. **The "received answers for unknown questions" warning**
   means the system silently drops answers for cancelled
   requests. That's the correct behaviour, but it could mask
   bugs where IDs are getting corrupted in transit. Counting
   these warnings (and exposing as a debug metric) would help
   catch ID corruption.
6. **Tool parameter validation isn't visible.** What if
   `params.Questions` is empty? What if a question has zero
   options? Probably the dialog handles these gracefully but
   worth explicit checks at the tool boundary so the model
   gets a clear error response rather than an empty UI.
7. **The JSON marshal returns `resp.Answers` directly** —
   not the request ID. The model has the original `id` per
   answer to correlate, but if a model emits two parallel
   `ask_user_questions` calls (unlikely but possible), the
   correlation depends solely on the inner question UUIDs.
   Including the request ID in the response payload would
   be more robust.
8. **`Selected []string`** holds option labels. If two
   options share a label, the answer is ambiguous. Should
   probably be option indices or option IDs. Labels are
   user-friendly for logging but semantically fragile.
9. **17 files changed**, including workspace plumbing in
   `internal/workspace/*`. The questions service has to be
   constructed somewhere and threaded through to the agent
   tool registry. Each new service follows this same pattern,
   which suggests room for a service registry abstraction
   eventually — but adding one in this PR would balloon scope.
10. **The tool description recommends "(Recommended)"
    suffix** but doesn't specify what the dialog does with
    it (e.g., highlight, default-select). Worth confirming
    the dialog actually treats it specially — otherwise the
    convention is just text the user happens to see.

# Suggested questions for the author

1. Is there a timeout policy for `Ask`? Indefinite blocking
   on user input has obvious UX implications for unattended
   sessions.
2. `Selected []string` uses option labels — what if two
   options have the same label? Should this be option IDs
   instead?
3. The "(Recommended)" suffix convention — does the dialog
   render this specially (highlight, default-select), or
   is it pure text the user happens to see?
4. Should there be a soft cap on `len(params.Questions)`
   per call? A model asking 100 questions at once seems
   like something to defend against.
5. Background-notification story: any plan for a bell or
   desktop notification when an `ask_user_questions` is
   pending and the TUI is not focused?
6. Should the JSON response include the request ID, not
   just the per-answer IDs, for redundant correlation?

# Verdict

**Approve.** This is a high-value addition: it adds a
structured channel for model-user interaction that didn't
exist before. The implementation respects the existing
patterns in the codebase (`pubsub.Broker`, `csync.Map`,
`fantasy.NewAgentTool`), the tool description prompt is
thoughtfully designed (especially the "(Recommended)" and
"None of the above" conventions), and the request-reply
shape is correct.

The main thing I'd want before merge is the timeout policy
question. Indefinite blocking on user input is the right
default for interactive sessions but the wrong default for
unattended ones, and crush has both modes. Even a
conservative default ("if no answer in 30 minutes, return
ctx.Err and let the model decide") with config override
would be safer than indefinite block.

# What I learned

Three takeaways, all about the *human-in-the-loop interface*
for LLM agents.

**First**, the tool-call vs prose-question distinction
matters enormously. When a model needs a user decision, it
has two options: emit free-form prose ("which approach do
you prefer, A or B?"), or emit a structured tool call. The
prose path forces the user to type a response, which the
model then has to re-parse, which can fail in 50 different
ways (the user types "the second one", "B sounds good",
"option two", "yeah let's go with B"). The structured path
gives the user a clickable UI and gives the model a
guaranteed-parseable answer.

For agent products that already have a UI surface, *every*
user-facing decision should have a structured tool. The
prose option is a fallback for "I have to ask but the answer
shape isn't constrained" — and even then, structuring it as
"text input field with a label" is better than "let the model
ask in chat".

The general lesson: the cost of a structured tool is one
schema. The cost of free-form prose is *every* round-trip
where the model has to parse the user's response. Structure
once, save forever.

**Second**, the *request-reply over pubsub* pattern is worth
memorizing. The challenge: an agent goroutine needs to send
a request, wait for a reply, and the reply will come from a
*different* goroutine (the UI) over a *fan-out* broker (the
broker doesn't know about request IDs).

The pattern:
- Asker: register `(requestID -> reply channel)` in a shared
  map, publish the request, block on `select { ctx, replyCh }`,
  defer-delete on return.
- Answerer: look up the request ID, send the response into
  its reply channel, log if not found.

This solves all three concurrency hazards at once: the asker
can be cancelled (context wins the select), the answerer
doesn't block (buffered reply channel), and the cleanup is
automatic (defer + nil-safe registry). I have *seen this
pattern in five different codebases* — it's the canonical
way to bridge "fire-and-forget broker" with "I need a reply".
Worth keeping in your mental toolkit.

**Third**, the *tool prompt as user-facing API*. The 48-line
markdown description of `ask_user_questions` is, in effect,
a user manual for the model. It tells the model:
- when to use the tool (clarify, gather preferences, decide)
- when *not* to use the tool ("do not ask for feedback while
  working on a plan the user can't see yet")
- how to format the input (UUIDs, short labels, options)
- what conventions to follow ("(Recommended)" suffix, "None
  of the above" option)
- what limitations to expect (truncation, label length)

Models behave dramatically better when the tool description
includes *negative space* ("do not", "limitations",
"truncated") in addition to the positive specification. The
"do not ask for feedback about the plan while you are working
on it" line is the kind of guidance that comes from observing
the model fail in production — and folding the lesson back
into the prompt is the cheapest, fastest fix.

When you write a tool prompt, ask: what failure mode have
I seen the model fall into? Document the *avoidance*
explicitly. The model's instruction-following is good enough
that "do not X" works, and it's much cheaper than re-training
the model.

The fourth meta-lesson: a "+1014 line" feature can still be
small in concept. Most of those 1014 lines are scaffolding
(workspace plumbing, dialog rendering, test fixtures). The
*idea* is 200 lines: a service with `Ask`/`Answer`, a tool
wrapper, a UI dialog. When reviewing big PRs that introduce
a single feature, separate the conceptual diff from the
plumbing diff. The plumbing is largely mechanical; the
concept is what to scrutinize. This PR's concept is sound
(request-reply over pubsub for human-in-the-loop), so the
review focuses on edge cases (timeout, label collisions,
soft caps) rather than the plumbing.
