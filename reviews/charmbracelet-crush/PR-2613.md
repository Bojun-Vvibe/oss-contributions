# Context

Some inference providers — Gemini and Vertex AI specifically —
cap the number of images per request at 10. Crush sends the
*entire* conversation history with every prompt (this is how
LLM agents preserve context). The bug: once the cumulative
image count in a session exceeds 10, *every* subsequent
request — including pure-text prompts in the same session —
fails with a "too many images" error from the provider. The
session is permanently bricked. The user can't even type
"summarize what we discussed" because the history blocking
the request also contains the images that exceed the limit.

This PR adds `pruneExcessImages()` that runs just before the
request is sent: count images, if over the limit, replace the
oldest ones with text placeholders (`"[Earlier image removed
to stay within model limits]"`) until the count is at or
below the limit. The persisted history in the DB is untouched
— only the *sent* messages are pruned. A new
`max_images` config field lets users override the default
limit per model.

+794/-0 across 3 files, but the implementation is ~110 lines
in `agent.go`; the rest is tests and the one-line config
field.

# Diff walkthrough

**The integration point** in `agent.go`:

```go
prepared.Messages = a.workaroundProviderMediaLimitations(prepared.Messages, largeModel)
+prepared.Messages = pruneExcessImages(prepared.Messages, largeModel)
```

The function slots in right after the existing
`workaroundProviderMediaLimitations`. That's the right
neighbourhood — both functions exist to massage the message
list to fit provider quirks before sending. Reader doesn't
have to hunt for "where do we adjust messages for provider
limits"; both adjustments are co-located and run in series.

**The provider limit map**:

```go
var defaultMaxImages = map[catwalk.InferenceProvider]int{
    catwalk.InferenceProviderGemini:   10,
    catwalk.InferenceProviderVertexAI: 10,
}

func maxImagesForProvider(model Model) int {
    if model.ModelCfg.MaxImages > 0 {
        return model.ModelCfg.MaxImages
    }
    return defaultMaxImages[catwalk.InferenceProvider(model.ModelCfg.Provider)]
}
```

User config wins, then fall back to known provider defaults.
A provider not in the map returns 0 → no limit → no pruning.
This is conservative in the right direction: a provider crush
doesn't know about gets the *current* (buggy) behaviour, not
the new (potentially over-aggressive) behaviour. The risk of
breaking a working session by pruning when you shouldn't is
zero.

The provider limits are hardcoded which is fine for now (10
is documented in Gemini's API), but worth noting they could
drift. A comment with the source URL would help future
maintainers verify.

**The image detection** is helpful in its honesty:

```go
func isImageFilePart(part fantasy.MessagePart) (fantasy.FilePart, bool) {
    fp, ok := fantasy.AsMessagePart[fantasy.FilePart](part)
    if !ok {
        return fp, false
    }
    return fp, strings.HasPrefix(fp.MediaType, "image/")
}
```

This handles only `FilePart` with image MIME types. If crush
ever supports inline base64 image parts (a different type), or
if the provider considers PDFs/videos as "images" for limit
purposes, this misses them. But for the *current* image-as-
file model that's the exact right shape.

**The pruning algorithm**:

```go
func pruneExcessImages(messages []fantasy.Message, model Model) []fantasy.Message {
    maxImages := maxImagesForProvider(model)
    if maxImages <= 0 {
        return messages
    }

    total := countImagesInMessages(messages)
    if total <= maxImages {
        return messages
    }

    toRemove := total - maxImages
    removed := 0

    result := make([]fantasy.Message, 0, len(messages))
    for _, msg := range messages {
        if removed >= toRemove {
            result = append(result, msg)
            continue
        }
        newParts := make([]fantasy.MessagePart, 0, len(msg.Content))
        for _, part := range msg.Content {
            if _, ok := isImageFilePart(part); ok && removed < toRemove {
                newParts = append(newParts, fantasy.TextPart{
                    Text: "[Earlier image removed to stay within model limits]",
                })
                removed++
                continue
            }
            newParts = append(newParts, part)
        }
        result = append(result, fantasy.Message{...})
    }
    ...
}
```

Three things to like:
1. **Early return** if no limit or count is already under
   the cap. The function does zero work in the happy path
   (no allocations, just a count). That's important because
   it runs on *every* request.
2. **Oldest-first** pruning. Iterates from the front of
   `messages` (oldest), replaces images, returns once
   `removed == toRemove`. The most recent images — which
   the user is most likely to be referring to in their
   current prompt — are preserved. This is the right
   policy for almost any LLM use case.
3. **Replace, don't drop**. The image is replaced with a
   text placeholder, not silently removed. The model sees
   "[Earlier image removed to stay within model limits]"
   and can mention this to the user if they ask "what was
   the first image". Silently dropping would let the model
   hallucinate, or the user notice an inconsistency without
   knowing why.

**The persistence boundary** in the function's doc:

```
Only the sent history is modified; the persisted messages
in the database are untouched.
```

This is critical. If the user later switches to a provider
with no image limit (or a higher one), the original images
should be available. The DB shouldn't be corrupted to fit
the *current* provider's quirks. The implementation
respects this — `pruneExcessImages` returns a new slice,
never mutates `messages`, and never touches the DB.

**The config field**:

```go
MaxImages        int      `json:"max_images,omitempty" jsonschema:"description=Maximum number of images allowed per request. When conversation history exceeds this limit the oldest images are automatically dropped. 0 means use the provider default."`
```

Good description, escape hatch for users who hit different
limits (e.g., a self-hosted Gemini-compatible endpoint with
a different cap). The `omitempty` + `0 means default`
convention is consistent with the surrounding fields.

# Risks

1. **Hardcoded limits drift.** Gemini's image limit could
   change; Vertex AI could add/remove image support;
   Anthropic could add a limit tomorrow. The map needs
   maintenance. A test that asserts "every provider crush
   supports has either a documented limit or a confirmed
   no-limit policy" would be valuable but is hard to
   write without runtime probing.
2. **`isImageFilePart` only matches `FilePart`.** If a
   provider ever sends image content via a different part
   type (inline base64 in a `TextPart`, a custom
   `ImagePart`), the count is wrong and pruning won't
   trigger. Worth a defensive comment + test fixture per
   part type the codebase actually supports.
3. **Replacement text counts as a real text part.** Each
   pruned image becomes a `TextPart` with
   `"[Earlier image removed to stay within model limits]"`.
   For 10+ pruned images that's 10+ near-duplicate text
   parts cluttering the message. Consider collapsing
   consecutive placeholders into one `"[N earlier images
   removed]"`.
4. **Token cost accounting.** Each placeholder is ~50
   tokens. If a user has 1000 images in history, replacing
   them all would add ~50K tokens, possibly tripping
   *token* limits while solving the *image* limit. The
   patch trades one limit for another. A more compact
   placeholder ("[img]" or unicode marker) would help.
5. **No test for the "all images in one message" case.**
   If a single message has 11 images and the limit is 10,
   does the function correctly prune just 1 image from
   that message? The algorithm should, but worth an
   explicit test.
6. **No test for the empty-message-after-pruning case.**
   If a message contained only an image and it gets
   replaced by a text part, the message is non-empty
   (good). But if a future change drops the image rather
   than replaces it, an empty message could end up in the
   list, which some providers reject.
7. **Pruning mutates message structure for the LLM but
   not for the UI.** The UI presumably reads from the DB
   (untouched). So the user sees their full history in
   the UI but the model sees a pruned one. If the user
   asks "what was in the first image you saw?", the model
   genuinely doesn't know. Worth a UX consideration:
   should the placeholder ever be surfaced to the user?
8. **Provider config can be wrong.** If a user sets
   `max_images: 100` for a Gemini model that actually
   caps at 10, pruning won't fire and requests will fail.
   No validation that user value ≤ known provider
   default. A warning log on first request would help.
9. **`countImagesInMessages` runs on every request.**
   For a 10K-message history (extreme), that's 10K
   message scans + N×part scans per request. Probably
   negligible compared to the network call, but worth
   knowing the cost is O(messages × parts).
10. **The placeholder text is in English and not
    localizable.** Crush is a TUI primarily for
    developers but if it ever localizes, this string
    will need extraction.

# Suggested questions for the author

1. The replacement text is ~50 tokens × N pruned images.
   For very-image-heavy sessions, that could push the
   request over a *token* limit while solving the *image*
   limit. Worth collapsing consecutive placeholders, or
   using a tighter token-cost-aware marker?
2. Hardcoded limits: where in the codebase should the
   "next time Gemini changes its image limit" maintenance
   happen? A comment with the source URL above
   `defaultMaxImages` would help.
3. Should there be a warning log on first-request when
   the user-configured `max_images` exceeds the known
   provider default? (Saves debugging when requests
   start failing for unknown reasons.)
4. Is there a UX surface that should tell the user "we
   dropped some images from history"? Right now it's
   silent except for the slog.Info debug log.
5. Should `isImageFilePart` also consider video/audio for
   providers that count those toward a media limit? Or
   is image-specific the right scope?

# Verdict

**Approve.** This is a small, well-shaped fix for a real
session-deadlock class bug. The implementation is
defensive in all the right places: provider not in the map
→ no behaviour change; user config not set → use known
default; persisted DB → never modified; algorithm →
oldest-first replacement preserving the user's recent
context. The test file (which I didn't quote in full but
is 700+ lines) covers the standard cases.

The one thing I'd want before merge is a tighter
placeholder token cost. `"[Earlier image removed to stay
within model limits]"` is fine for 1-2 prunes; for 50+
prunes it bloats the request. Either a compact marker or
collapsed placeholders would solve it.

# What I learned

Three takeaways, all about the *send-vs-store* boundary in
LLM agent code.

**First**, the *session-deadlock pathology* is a category
worth recognizing. LLM agents send the full history with
every request because that's how the model preserves
context. So any constraint the provider enforces on a single
request becomes a *cumulative* constraint on the session.
Image-count limits are the most obvious example here, but
the same pattern applies to: file-attachment limits, tool-
call limits per request, audio-clip limits, structured-output
schema size limits. The first time a session hits the limit,
*every* subsequent request inherits the failure. The user
sees a session that just stops working with no clear cause.

The diagnostic is: if your test session works for the first
N turns and then permanently breaks, suspect a per-request
quota that you've now exceeded *cumulatively*. The fix shape
is always: **prune the sent payload, don't prune the stored
history**. Storage is cheap; user trust is not.

**Second**, the *send-store split* is the architectural
move that makes this fix safe. Crush has two
representations: the DB (canonical, complete) and the sent
messages (a derived, possibly-pruned view for one specific
provider call). `pruneExcessImages` operates only on the
derived view. If the user switches providers, deletes a
message, or re-runs a turn, the canonical history is
untouched. If a future fix needs to prune for token count,
it goes in the same boundary; it doesn't require touching
the DB or the UI.

This split is worth designing in *before* you need it. Once
you have agents that mutate stored history to fit a current
provider's quirk, you're in a world where the user's data is
hostage to whatever the *current* model demands. Always have
a "send-time view" that you can mangle freely, distinct from
a "stored truth" you guard.

**Third**, the *replace-don't-drop* policy preserves the
model's epistemic state. When the model sees `"[Earlier image
removed to stay within model limits]"`, it knows: (a) there
was an image, (b) it can't see it now, (c) why. If the user
asks "what was in the first image", the model can honestly say
"I don't have access to it anymore" instead of either
hallucinating or being silently confused that the user is
referring to something it never saw.

The general lesson: when you have to remove information from
a context window, leave a *marker* of the removal. Models
trained on chat-style data understand placeholders like
`[redacted]`, `[earlier message removed]`, etc. The marker
costs a few tokens; the alternative — silent deletion — costs
the model's calibration and the user's trust.

The fourth meta-lesson: this is a 110-line fix for a bug
that, if you didn't know about it, would manifest as a
mysterious "Gemini is broken" complaint from a power user.
The code is mostly defensive scaffolding (`if maxImages <= 0`,
`if total <= maxImages`, `if removed >= toRemove`), and the
defensive scaffolding is the *value* — it's why the fix
makes the unhappy path work without changing the happy path.
When reviewing fixes for "rare provider quirks", count the
defensive checks; if they cover all the no-op cases, the fix
is probably safe to merge even if you can't reproduce the
original bug.
