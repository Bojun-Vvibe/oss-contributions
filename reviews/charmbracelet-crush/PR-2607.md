# Context

`crush` is a terminal-based chat UI for LLM agents. MCP tools
(GitHub, GitLab, etc.) routinely return text content that
*happens* to be a unified diff — for example, the GitHub MCP
server's `get_pull_request_diff` tool returns the literal diff
text. Pre-PR, that text rendered as plain prose: a wall of
`+` and `-` lines, no syntax highlighting, no per-hunk
formatting, just unstyled output.

This PR adds a small content-classifier
(`internal/diffdetect`) and routes any tool-result content that
*looks like* a unified diff through the existing `diffview`
renderer. The before/after screenshots in the PR show the
visible UX win: a wall of monospace text becomes a styled diff
with red/green hunks and file headers.

# Problem

Two adjacent UX problems being solved together:

1. **Generic tool results were rendered as prose.** Three
   files (`docker_mcp.go`, `generic.go`, `mcp.go`) each had
   their own custom rendering paths that just dumped the
   tool's text content verbatim. None of them detected
   structured content (diffs, JSON, tables) and routed to a
   richer renderer.
2. **The diff renderer existed but was siloed.** `diffview`
   was already in the codebase but only invoked from the
   built-in `apply_patch` tool path. Generalizing it to *any*
   diff-shaped content required a detector and a refactor of
   the per-tool rendering paths to share a common pipeline.

# Design

Three components, neatly separated:

**1. `internal/diffdetect/detect.go` (+36 new):**

```go
type Signal struct {
    HasHunk       bool   // @@ ... @@
    HasFileHeader bool   // --- a/foo or +++ b/foo
    HasGitHeader  bool   // diff --git a/foo b/foo
}

func IsUnifiedDiff(content string) bool {
    s := Inspect(content)
    if s.HasGitHeader && s.HasFileHeader { return true }
    return s.HasHunk && s.HasFileHeader
}
```

The detector is correctly **structural, not regex-based**. It
scans line-by-line and tracks three independent signals.
"Looks like a diff" requires at least *two* of those signals
in combination — preventing false positives from text that
incidentally has `@@` somewhere but isn't a diff. The
combination rules are conservative: require both hunk and
file header, OR git header and file header. Pure hunk-only
text (which exists, e.g. partial patches) doesn't match,
which is the right call — false positives are worse than
false negatives here, because mis-rendering plain text as a
diff is noisier than mis-rendering a diff as plain text.

The +152-line test file (`detect_test.go`) covers positive
cases (real GitHub diffs, partial patches, multiple hunks),
negative cases (text containing `+++`, log lines with `@@`),
and edge cases (empty input, single-line input).

**2. `internal/ui/chat/unified_diff.go` (+171 new) +
`tool_result_content.go` (+59 new):**

These are the routing layer: take a tool result, run it
through `diffdetect.IsUnifiedDiff`, and either route to the
diff renderer or fall back to the existing prose path.

The unified-diff renderer here parses the diff into hunks,
calls into `internal/ui/diffview/diffview.go` (extended +43
to expose the rendering primitives), and produces the styled
output. The diffview extensions look like a clean factor —
existing apply_patch path is unchanged.

**3. Per-tool simplifications:**

```
internal/ui/chat/docker_mcp.go    +1/-14
internal/ui/chat/generic.go       +2/-27
internal/ui/chat/mcp.go           +3/-46
```

Each file gets its custom rendering ripped out and replaced
with a single call into `tool_result_content.go`. **Net deletion
of ~85 lines** of duplicate rendering code is a real win —
three implementations collapse into one.

# Risks

- **False positives.** The detector is conservative but not
  bulletproof. A document that includes diff-like content as
  *part* of its prose (e.g. a tutorial about diffs) would
  trigger the detector and render as a diff. Worst case the
  user sees a misformatted document; not catastrophic but
  annoying. The PR has a 152-line test file but I don't see
  a "documentation that mentions diffs" negative case in the
  list.
- **Mixed content.** A tool that returns "Here's the diff:\n
  --- a/foo\n+++ b/foo\n@@..." in prose would either render
  as all-prose (if the prefix throws off detection) or
  all-diff (if the hunk markers dominate). Mixed-content
  rendering would need a more sophisticated detector that
  finds *regions* of diff inside larger prose. Out of scope
  for this PR but worth noting as a known limitation.
- **Performance on large content.** `diffdetect.Inspect`
  splits on `\n` and walks every line. For a multi-megabyte
  tool result, that's an O(n) scan that runs on every
  tool-result render. Probably fine, but worth a benchmark
  for the worst case (a 10MB log file from some MCP tool).
- **Net 1015/-88 is a big PR.** Most of it is the new tests
  (`mcp_test.go` +406, `tool_result_content_test.go` +116,
  `detect_test.go` +152) which is excellent test coverage,
  but the test-to-code ratio (~70% test) is an outlier vs the
  rest of crush. Worth confirming the tests are exercising
  meaningful paths and not just expanding line count.
- **`strings.SplitSeq`** is a Go 1.24+ feature. If crush
  supports older Go versions, this would be a build break on
  CI. The PR doesn't mention Go version requirements; worth
  verifying the `go.mod` minimum is bumped if necessary.

# Suggestions

1. Add a "tutorial-about-diffs" negative test case to
   `detect_test.go` to lock in the false-positive boundary.
2. Add a benchmark on `diffdetect.Inspect` against a large
   synthetic input (10MB log file) and decide if early-out
   logic is needed (e.g. "stop scanning after 10k lines").
3. Long-term: consider region-based detection so mixed
   content can render the diff *parts* as styled diffs while
   keeping the prose around them readable. This is the next
   natural extension.
4. The `diffdetect` package is generic enough that other
   crush features (search results, git log views) might
   reuse it. Worth promoting it to a shared package if
   reusability emerges.

# Verdict

**Ship it.** Genuine UX improvement with the right
architecture: small detector + routing layer + collapsing
three duplicate rendering paths into one. The test budget is
generous (~70% test code), false-positive risk is acknowledged
and mitigated by conservative detection rules, and the net
deletion in the per-tool files is a structural cleanup
disguised as a feature PR.

# What I learned

The structural pattern here is worth memorizing: **detect →
route → render**, with each phase isolated in its own package.
The detector knows *only* about content shape; the router
knows *only* about content type → renderer mapping; the
renderer knows *only* about how to display its content type.
Each phase can be tested in isolation, swapped independently,
and reused across call sites.

The conservative-detection bias is a recurring pattern in any
"auto-detect this kind of content" system. False positives
(rendering a non-diff as a diff) are *much* worse than false
negatives (rendering a diff as prose) because the user can
mentally parse a poorly-rendered diff but a prose-mis-rendered-
as-diff is gibberish. The right policy is "require strong
signal" — two corroborating markers, not just one. That bias
should be encoded in the detector's combination rules, not
left to a downstream tweak.

The third lesson is the **collapse-N-duplicate-implementations**
pattern as the most common high-leverage refactor in any
mature UI codebase. Three files (`docker_mcp.go`, `generic.go`,
`mcp.go`) each had their own slightly-different rendering
logic — totaling ~87 lines pre-PR, ~6 lines post-PR. The new
`tool_result_content.go` is ~59 lines, so the *net* code is
smaller and centralized. Whenever you find three files doing
"the same thing slightly differently", the question to ask is
"what's the smallest detector + router that lets all three
share one renderer". That's the leverage.
