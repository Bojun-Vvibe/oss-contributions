# Context

Crush's `App` had a single `chan tea.Msg` (buffered 100) that
multiple goroutines wrote to and a single TUI consumer read
from. As soon as a second consumer arrived — the HTTP server's
`/events` SSE handler in `proto.go` — the two consumers
*competed* for events: every event went to whichever consumer
read first, and the other one missed it. The "fix" up to now
was a `subscriberSendTimeout` (2 s) that dropped messages when
the channel filled up, and a goroutine that drained a timer in
a way that could deadlock if you didn't tiptoe around it (see
the now-deleted `TestSetupSubscriber_DrainAfterDrop`).

This PR replaces the single channel with `pubsub.Broker[tea.Msg]`
(crush already has a generic broker, used elsewhere for agent
notifications). Every caller of `app.Events(ctx)` now gets its
*own* subscription channel; every event is delivered to every
subscriber. The drop-on-full and timer-juggling logic is
deleted. Net diff is +157/-152 across four files.

# Diff walkthrough

**The type change in `app.go`**:

```go
-events          chan tea.Msg
+events          *pubsub.Broker[tea.Msg]
...
-events:             make(chan tea.Msg, 100),
+events:             pubsub.NewBroker[tea.Msg](),
```

The other `pubsub.NewBroker` next to it (`agentNotifications`)
is the prior art that proves this pattern already works in the
codebase. Reusing it for events is the right call — fewer
concurrency primitives in the project, less state for readers
to track.

**The API change**:

```go
-func (app *App) Events() <-chan tea.Msg {
-    return app.events
+func (app *App) Events(ctx context.Context) <-chan pubsub.Event[tea.Msg] {
+    return app.events.Subscribe(ctx)
 }
```

Two important shape changes:
1. `Events()` now requires a `ctx`. This is correct: a
   subscription's lifetime must be bound to *something*, and
   passing the caller's context lets the broker auto-clean up
   when the caller goes away. Without this, every caller leaks
   a goroutine and a buffered channel for the lifetime of the
   broker.
2. The element type went from `tea.Msg` to `pubsub.Event[tea.Msg]`.
   Callers now pull `.Payload` out of the event wrapper. The
   `proto.go` change (`ev.Payload` instead of `ev`) shows the
   migration is straightforward.

**The deletion** is the satisfying part:

```go
-const subscriberSendTimeout = 2 * time.Second
...
-                sendTimer.Reset(subscriberSendTimeout)
-                select {
-                case outputCh <- msg:
-                case <-sendTimer.C:
-                    slog.Debug("Message dropped due to slow consumer", "name", name)
-                case <-ctx.Done():
-                    ...
-                }
+                broker.Publish(pubsub.UpdatedEvent, tea.Msg(event))
```

Twenty lines of timer-juggling, message-dropping, and
"slow consumer" handling collapse to one `Publish` call. The
broker handles all of that internally — and presumably better,
because (a) it's the same pattern used everywhere else in the
codebase and (b) per-subscriber channels mean one slow
subscriber doesn't starve fast ones.

**The TUI consumer change**:

```go
+    events := app.events.Subscribe(tuiCtx)
     for {
         select {
         case <-tuiCtx.Done():
             return
-        case msg, ok := <-app.events:
+        case ev, ok := <-events:
             if !ok {
                 return
             }
-            program.Send(msg)
+            program.Send(ev.Payload)
         }
     }
```

The TUI is now a peer subscriber instead of *the* consumer.
That's the whole point of the refactor: the SSE handler in
`proto.go` and the TUI loop here are now equal citizens of
the broker, and adding a third subscriber tomorrow (a
metrics emitter, a debug logger, anything) is a one-line
`broker.Subscribe(ctx)` away.

**The HTTP server side** in `proto.go`:

```go
-    events, err := c.backend.SubscribeEvents(id)
+    events, err := c.backend.SubscribeEvents(r.Context(), id)
...
-            c.server.logDebug(r, "Sending event", "event", fmt.Sprintf("%T %+v", ev, ev))
-            wrapped := wrapEvent(ev)
+            c.server.logDebug(r, "Sending event", "event", fmt.Sprintf("%T %+v", ev.Payload, ev.Payload))
+            wrapped := wrapEvent(ev.Payload)
```

The `r.Context()` plumbing here is exactly why the API
change to take a `ctx` was the right call — when the HTTP
client disconnects, the request context cancels, the
broker auto-unsubscribes, the channel closes, the
goroutine exits. Zero leak, zero extra code at the call
site.

**The test file rewrite** is the most interesting part. The
old tests used Go 1.24+ `testing/synctest` to deterministically
test the `subscriberSendTimeout` timer logic — including a
specific test (`TestSetupSubscriber_DrainAfterDrop`) for a
deadlock that the old timer code could hit. All of that is
gone. The new tests cover the broker semantics:

```go
func TestEvents_NConsumers(t *testing.T) {
    for _, n := range []int{2, 5, 10} {
        t.Run(fmt.Sprintf("consumers=%d", n), func(t *testing.T) {
            t.Parallel()
            testNConsumers(t, n)
        })
    }
}

func testNConsumers(t *testing.T, n int) {
    ...
    channels := make([]<-chan pubsub.Event[tea.Msg], n)
    for i := range n {
        channels[i] = broker.Subscribe(ctx)
    }
    require.Equal(t, n, broker.GetSubscriberCount())

    const numEvents = 20
    for i := range numEvents {
        broker.Publish(pubsub.UpdatedEvent, tea.Msg(i))
    }

    var wg sync.WaitGroup
    for i, ch := range channels {
        wg.Go(func() {
            for j := range numEvents {
                select {
                case ev := <-ch:
                    require.Equal(t, tea.Msg(j), ev.Payload, ...)
                case <-time.After(5 * time.Second):
                    t.Errorf("consumer %d: timed out waiting for event %d", i, j)
                    return
                }
            }
        })
    }
    wg.Wait()
}
```

This is the *behavioural* property that justifies the whole
refactor: N consumers, each receives every event in order.
The old code couldn't even *express* this property because
the model was wrong (single channel, competing readers).
That's the deepest signal that the refactor is right — the
new test is one you couldn't have written against the old
code.

# Risks

1. **The broker has no per-subscriber buffer limit visible
   in this diff.** If one subscriber stops draining, the
   broker either blocks all publishers (worst case) or
   buffers unboundedly (memory leak). The diff doesn't
   show `pubsub.Broker`'s policy. Worth confirming —
   ideally the broker drops oldest events for a slow
   subscriber rather than blocking everyone.
2. **The deleted `TestSetupSubscriber_NoTimerLeak`** used
   `goleak.VerifyNone` to catch leaked timer goroutines.
   The new tests don't use goleak. If the broker leaks
   subscriber goroutines on `Shutdown`, this PR removes
   the test that would have caught it.
3. **Behavioural change**: under the old code, slow
   consumers caused message drops with a debug log. Under
   the new code, slow consumers presumably cause
   accumulation in their per-subscriber buffer. *Different*
   pathology, possibly *worse* if memory is tighter than
   message-loss tolerance. Worth a one-line note in the
   PR description.
4. **The HTTP `/events` handler now holds an SSE connection
   open with a per-client subscription**. A client that
   opens 1000 SSE connections without consuming gets 1000
   broker subscribers. No rate limit visible. This is a
   pre-existing issue (the old single-channel design was
   even worse — one slow client could starve the TUI), but
   the refactor doesn't fix it.
5. **`Events()` signature changed from no-arg to `(ctx)`.**
   Any external caller (plugin, embedded use) breaks.
   Probably fine for crush's stage of project, but worth
   calling out.
6. **`pubsub.UpdatedEvent` vs `CreatedEvent`** — the old
   code didn't distinguish event kinds (single channel).
   The new code uses `UpdatedEvent` everywhere. If a
   subscriber wants to filter by kind, they can't from
   the looks of it (the broker doesn't filter). If
   filtering becomes desirable, the kind field is unused
   noise.
7. **`Shutdown()` ordering**: the new cleanup func calls
   `app.serviceEventsWG.Wait()` *then* `app.events.Shutdown()`.
   That's correct (drain producers before closing). But if
   any goroutine starts publishing *after* `Shutdown`,
   does the broker panic, drop, or block? Not visible
   from this diff; worth a one-line invariant comment.
8. **The TUI subscriber is created with `tuiCtx`**, which
   cancels when the TUI shuts down. After cancel, the TUI
   `for` loop exits on `<-tuiCtx.Done()`. But the subscription
   channel close is not the trigger — the context is. If
   the broker is shutdown *before* the TUI context cancels,
   the `events` channel closes and the loop exits via the
   `!ok` branch. Both paths work; just worth noting both
   are exercised.
9. **No metric for broker subscriber count.** A simple
   `broker.GetSubscriberCount()` (already used in tests)
   could be exposed via a debug endpoint to detect leaks
   in production.
10. **The 100-message buffer is gone from this layer**, but
    `pubsub.Broker` presumably has its own per-subscriber
    buffer. The size of *that* buffer determines the new
    backpressure characteristics. Worth confirming and
    documenting.

# Suggested questions for the author

1. What's `pubsub.Broker`'s policy when a subscriber stops
   draining? Drop oldest, drop newest, block publisher,
   unbounded buffer? The answer determines whether a
   wedged HTTP client can hurt the TUI.
2. The old code had a `TestSetupSubscriber_NoTimerLeak`
   using `goleak.VerifyNone`. Should the new test suite
   have a `TestEvents_NoSubscriberLeak` that asserts
   no goroutines leak after `Shutdown`?
3. Any rate limit on `/events` SSE connections? With
   per-client subscriptions, a malicious client can now
   spawn N subscribers cheaply.
4. The kind field (`pubsub.UpdatedEvent` vs `CreatedEvent`)
   is uniform here. Is there a roadmap to filter by kind,
   or should we use a single canonical kind to keep things
   simple?
5. Is `broker.GetSubscriberCount()` worth exposing on a
   debug endpoint?

# Verdict

**Approve.** This is a textbook "replace a single-consumer
channel with a fan-out broker" refactor. The diff is symmetric
(net +5 lines), the new tests cover the new contract
(N-consumer fairness) that the old tests couldn't express, and
the deleted code (timer juggling, drop-on-full, deadlock
workaround) is the kind that you're glad to see go. The reused
`pubsub.Broker` already powers `agentNotifications` in the
same struct, so this PR isn't introducing a new primitive —
it's normalizing on the one that already worked.

The single thing I'd want before merge is one sentence in the
PR description about how `pubsub.Broker` handles slow
subscribers — drop policy, buffer size, blocking behaviour.
That's the new failure mode the refactor introduces, and it
should be explicit.

# What I learned

Three things, all about the moment a single-consumer channel
turns into a multi-consumer one.

**First**, the *shape* of the bug. Crush had `App.events
chan tea.Msg`, perfectly fine when only the TUI reads it.
Then someone added an HTTP `/events` SSE handler — also
reading `app.Events()` — and the bug appeared. Both consumers
ranged over the same channel; every event went to whoever
read first. This is the canonical "Go channels are great for
producer-consumer, terrible for fan-out" lesson, and you
*will* re-learn it the first time you grow a second consumer
on a system that started with one.

The fix is a fan-out primitive: a broker that internally
holds one channel per subscriber and pushes every published
event to every channel. crush's codebase already had
`pubsub.Broker[T]` for this. The right move was to *reuse* it,
not to invent a new fix on the spot.

The diagnostic is: when you find yourself writing
"send-with-timeout-and-drop" code on a single channel because
of a "slow consumer", you are almost certainly trying to do
fan-out with a tool that doesn't fan-out. Stop and reach for
the broker.

**Second**, the *test rewrite* is more important than the
code rewrite. The old tests used `testing/synctest` to
deterministically simulate timer behaviour, including a test
specifically for a deadlock the old timer-drain code could
hit (`TestSetupSubscriber_DrainAfterDrop`). Those tests were
*great* for what they tested — but they tested an
implementation detail (the timer dance), not the property the
system was supposed to deliver (every consumer gets every
event). When the implementation changed, the tests had to be
thrown out.

The new test (`TestEvents_NConsumers`) tests the *property*:
N consumers, M events, every consumer sees every event in
order. That test would survive a future broker reimplementation
(e.g., switching from goroutine-per-subscriber to a more
sophisticated scheme). Property tests survive refactors;
implementation tests die with the implementation.

The lesson: when you rewrite a system, take the chance to
rewrite the tests at the property level. If the new test
catches bugs the old test couldn't even *describe* (like
"consumer 5 missed event 12"), that's the signal that the
refactor was structurally right, not just cosmetically
prettier.

**Third**, the *context plumbing* through the API. The old
`Events()` took no arguments. The new `Events(ctx context.Context)`
requires a context. This is a breaking signature change but
it's the right one: subscriptions must have a lifetime, and
the cleanest way to express "this subscription lives as long
as my work" is to pass the work's context.

Look at the HTTP handler payoff:

```go
events, err := c.backend.SubscribeEvents(r.Context(), id)
```

When the HTTP client disconnects, `r.Context()` cancels, the
broker unsubscribes, the goroutine exits. *Zero cleanup code
at the call site.* That is the entire promise of `context.Context`
in Go — wire it through everywhere and lifecycle just works.

The general pattern: any API that returns a long-lived channel,
goroutine, subscription, lock, file handle, or network
connection should take a `context.Context` so the caller's
lifecycle drives the resource's lifecycle. If you find an API
that returns a long-lived thing without taking a context, that
API has a leak waiting to happen.

The fourth meta-lesson: the satisfying refactors are usually
*net-zero or net-negative* line counts. This one is +5 net.
You replaced 20 lines of timer juggling with one `Publish`
call, replaced six lines of test fixture with a clean
`pubsub.NewBroker`, and added new test coverage for properties
the old code couldn't express. If a "fix" diff is +500 lines,
you're probably patching around a structural problem instead
of fixing it. If a "fix" diff is +5 with the right tests, you
probably found the structural problem and removed it.
