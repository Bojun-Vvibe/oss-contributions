# Context

`crush` agents support multimodal tool results — an MCP tool
can return an image (PNG, JPEG, etc.) which gets sent to the
LLM as base64-encoded data. The base64 is supposed to be
clean, RFC-4648-compliant input, but in practice MCP servers
sometimes return:

- Already-base64'd data that crush then *re-base64s* (double
  encode → garbage).
- Raw binary that wasn't base64 at all but had a base64-ish
  MIME type set.
- Truncated base64 with a missing padding character.
- Base64 with embedded null bytes or non-ASCII bytes.

Any of those landed straight into the conversation history.
The model received it, choked on it, sometimes returned an
unparseable response, and the *session* — which persists the
conversation — was now permanently corrupt. Reload the session,
the bad bytes get re-sent, the model chokes again. No way out
short of deleting the session.

The author's PR description: "this fixes a gnarly bug where bad
multimodal tool results could brick sessions. Oof." That's the
right frame.

# Problem

Three distinct failure modes being collapsed into one fix:

1. **Invalid base64 from MCP tool.** Not all MCP servers are
   well-behaved. A tool returning malformed image data
   shouldn't corrupt the session — it should be detected and
   replaced with a placeholder error message.
2. **Double-encoding.** The `tools/mcp/tools.go` path was
   re-encoding data that was *already* base64. The receiving
   model would get gibberish that decoded to gibberish.
3. **No validation at the persistence boundary.** Bad bytes
   went straight into the session store. On reload, the bad
   bytes returned. The corruption was *durable*.

Issue #1782 is referenced — likely the user-reported symptom.

# Design

Three layers of fix, each at a different boundary:

**Layer 1: A validator (`internal/stringext/string.go`):**

```go
// +18 / 0
func IsValidBase64(s string) bool {
    _, err := base64.StdEncoding.DecodeString(s)
    return err == nil
}
```

The simplest possible thing — try to decode, return success.
+38 lines of tests cover empty input, valid b64, malformed
b64, padding edge cases. Trivially correct, easy to reuse.

**Layer 2: Validate at the agent's tool-result conversion
boundary (`internal/agent/agent.go:1091`):**

```go
case fantasy.ToolResultContentTypeMedia:
    if r, ok := fantasy.AsToolResultOutputType[
            fantasy.ToolResultOutputContentMedia](result.Result); ok {
        if !stringext.IsValidBase64(r.Data) {
            slog.Warn("Tool returned media with invalid base64 data, discarding image",
                "tool", result.ToolName, "tool_call_id", result.ToolCallID)
            baseResult.Content = "Tool returned image data with invalid encoding"
            baseResult.IsError = true
        } else {
            // ... existing happy path ...
        }
    }
```

Three things happen on detection of bad base64:
1. **Log it** with `tool` and `tool_call_id` for forensics.
2. **Replace** the data with a human-readable error string.
3. **Set `IsError = true`** so the downstream message
   formatting flags it as an error result, not as a successful
   image load.

The replacement message — "Tool returned image data with invalid
encoding" — is sent to the model. The model sees an error tool
result, not garbage. The session stays clean.

**Layer 3: Refactor MCP tool handling
(`internal/agent/tools/mcp/tools.go`, +17/-32):**

The pre-PR code path had its own ad-hoc base64 wrapping. The
PR removes most of that (-32 lines) and threads the data
through the validated path. The negative diff in the test file
(-70 lines in `tools_test.go`) suggests the test was exercising
the pre-PR ad-hoc wrapping, which doesn't exist anymore.

`internal/message/content.go` (+14/-3) is the persistence-side
fix: validation also happens at the message-store boundary, so
even if a future code path bypasses Layer 2, Layer 3 catches
the bad data before it lands in the session store.

# Risks

- **Replacement message is fixed text.** All bad-base64
  failures get the same message, regardless of cause (truncated,
  double-encoded, raw binary). The model receives no diagnostic
  info beyond "invalid encoding". A more informative message
  ("base64 decode failed at byte N", "data length suggests
  double-encoding") would help the model retry intelligently or
  surface a useful error to the user. Marginal improvement.
- **`base64.StdEncoding` only.** RFC-4648 has two variants
  (standard and URL-safe). MCP servers *should* use standard,
  but a non-compliant server using URL-safe would now be
  rejected as "invalid encoding" when previously it might have
  worked accidentally. Worth supporting both with a fallback:

  ```go
  func IsValidBase64(s string) bool {
      if _, err := base64.StdEncoding.DecodeString(s); err == nil { return true }
      if _, err := base64.URLEncoding.DecodeString(s); err == nil { return true }
      return false
  }
  ```

- **Session-store cleanup not addressed.** Existing sessions
  that already contain corrupt base64 from before this fix will
  *still* fail on reload — the validator catches *new* bad data
  at write time but doesn't sweep the existing store. A
  one-time migration that scans existing sessions, validates
  base64 fields, and replaces corrupt ones with the placeholder
  would close the loop. Maybe a follow-up PR.
- **`tools_test.go` deletes 70 lines.** The PR replaces them
  with newer tests that cover the new code path, but deleting
  test coverage during a fix-PR is always worth a careful
  read — make sure the deleted tests' invariants are still
  enforced by the new tests.
- **No fuzz test.** Base64 validation is exactly the kind of
  thing a fuzzer eats for breakfast. A 5-line `FuzzIsValidBase64`
  in `string_test.go` would catch any future regression in the
  validator. Cheap to add.

# Suggestions

1. Support URL-safe base64 in the validator — costs nothing,
   handles a class of well-meaning-but-wrong MCP servers.
2. Make the replacement message include the tool name —
   "Tool `screenshot` returned image data with invalid
   encoding" gives the user actionable info on which tool to
   blame.
3. Add a `FuzzIsValidBase64` fuzz test.
4. Land a one-time session-store sweep that fixes pre-existing
   corrupted sessions. Without it, users who already hit this
   bug have no path forward except deleting their sessions.
5. Long-term: any data crossing the agent boundary into a
   persistent store should pass through a typed-validator
   layer. Base64 is just the first instance; URLs, JSON,
   timestamps, tool-call IDs all have similar
   write-and-corrupt failure modes.

# Verdict

**Ship it.** Real fix to a real session-bricking bug. The
three-layer defense (validator + agent boundary + message
boundary) is the right shape — defense in depth means the next
code path that bypasses one layer still gets caught by the
others. Follow-ups (URL-safe base64, fuzz, session sweep) are
all small and non-blocking.

# What I learned

The pattern "untrusted external data → no validation at the
persistence boundary → permanent session corruption" is the
worst class of input-validation bug because it's *durable*.
Most input-validation bugs cause a transient failure: bad
input → bad request → user retries with good input → all is
well. Persisted-corruption bugs cause failures that re-occur
*every time* the user loads the session, with no path to fix
them short of nuking state. The user impact is much higher per
bug than transient failures, but the bug class gets
under-prioritized because each individual report looks small.

The defense pattern — **validate at every boundary the data
crosses, not just the outermost one** — is the right
architectural response. Three layers in this PR catch the
same bad data three times. That's not redundancy; it's
defense in depth. If a future contributor adds a new code
path that bypasses one layer, the others still hold the line.

The other reusable insight is the **error result with
human-readable replacement** instead of a hard error. The
naive fix would be to throw an error and refuse to send any
tool result. That breaks the conversation flow entirely. The
better fix sends a tool *error* result that the model
understands ("the tool failed, here's why") so the
conversation can continue. The user sees a clean error message;
the model has a chance to retry, ask for clarification, or
move on. Worth applying to any tool-result handling that can
fail: replace with an error result, don't drop the call.
