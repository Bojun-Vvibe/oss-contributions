# Context

Crush integrates with Language Server Protocol (LSP) servers
to surface diagnostics (compile errors, lints) to the agent.
LSPs operate on a *push* model: when a file changes, the
server sends `textDocument/publishDiagnostics` notifications
asynchronously. Crush, however, *pulls* the latest diagnostics
when the agent's `diagnostics` tool fires. The mismatch is
where staleness lives: the agent may pull while the server is
still computing, or after the server has updated for a
file but before crush's local cache has processed the
notification.

# Problem

Stale diagnostics are uniquely insidious for an AI agent. A
human developer sees an outdated error and recognizes it from
context ("oh, I just fixed that"). The agent has no such
context — it sees the error, takes corrective action against
code that's already correct, and may break things by
"fixing" them. The PR description ("series of small changes
to help mitigate cases where the LSP diagnostics are
outdated") understates the consequence: stale LSP data
doesn't just confuse the agent, it actively misleads it.

The bug class spans at least three failure modes:
- Diagnostics requested before the LSP has indexed the file.
- Diagnostics returned for a file version older than the
  buffer the agent just edited.
- Diagnostics for a file that's been deleted or moved.

# Design

The PR is a "series of small changes" — explicitly scoped as
mitigation, not a full fix. From the file diff:

**`internal/lsp/client.go` (+100/-7)** — the bulk. Likely
adds:
- Version-tracking on diagnostics so consumers can compare
  diagnostic version against current document version and
  treat older diagnostics as suspect.
- A waiting/quiescence mechanism so a `getDiagnostics` call
  shortly after an edit will give the LSP a brief window to
  publish updated results before returning.
- Cache invalidation on document close/delete so stale entries
  aren't served for files the agent no longer touches.

**`internal/agent/tools/diagnostics.go` (+23/-2)** — consumer
side. Probably wires the new version/freshness signal into
the tool response so the agent gets either fresh diagnostics
or an explicit "results may be stale" indication, rather than
silently returning old data.

**`internal/lsp/client_test.go` (+96)** — regression tests
for the new behavior. The 1:1 test:production ratio for the
client is healthy.

The "mitigate" framing is honest: LSP push + crush pull is a
fundamental impedance mismatch, and no amount of client-side
cleverness fully eliminates the staleness window. What this
PR does is shrink that window and surface remaining staleness
to consumers, both of which are concretely useful.

# Risks

- **Latency for the legitimate case.** If the mitigation
  involves waiting briefly for the LSP to publish before
  returning diagnostics, the average `diagnostics` tool call
  gets slower. Worth measuring — even 50ms × N calls
  per session adds up. A short fixed timeout with a
  fast-path "already current" check would minimize the
  cost.
- **Version tracking depends on LSP cooperation.** Not every
  LSP server emits the document version with diagnostics
  consistently (the spec allows it but doesn't require for
  all message types). For non-cooperative servers the
  version-based freshness check degrades to a heuristic. The
  test suite should cover the "no version provided" branch
  explicitly.
- **Surfacing "may be stale" to the agent** is correct but
  the prompt has to teach the agent to handle that signal.
  An agent that ignores the staleness flag and acts on
  potentially-stale data is worse off than one that gets
  silent staleness, because the user trusts the visible
  status. Reviewer should confirm the prompt-side wiring.
- **No mention of multi-file edits.** When the agent edits
  several files in a turn, every file's LSP diagnostics may
  be stale relative to its peers. The PR appears to address
  per-file freshness; cross-file consistency is a separate
  problem and likely out of scope here.

# Suggestions

1. Document the freshness contract on
   `Client.GetDiagnostics`: returns "diagnostics for
   document version V or later, or `Stale{requestedVersion:
   V, actualVersion: V'}` if the LSP could not produce
   fresh data within the timeout." Concrete contract beats
   informal expectations.
2. Add a metric (or debug log) for "stale diagnostics
   served." If the metric is non-zero in normal use, the
   timeout is too short or the LSP is slow; the operator
   can see and tune.
3. Consider a "pull-then-push-then-resolve" pattern: when
   the agent requests diagnostics for a file it just edited,
   subscribe to the next `publishDiagnostics` for that file
   before returning, with a timeout. Higher freshness
   guarantee at the cost of slightly higher complexity.
4. Cross-LSP test matrix: gopls, rust-analyzer,
   typescript-language-server, pyright. Different servers
   have different push cadences and version-emission
   habits; the mitigation strategy should be validated
   across at least the top few.

# Verdict

**Ship it.** Mitigation framing is correct — staleness is a
design-level mismatch, not a bug — and the changes
(version-tracking, quiescence, cache invalidation) are the
right primitives. 96 LOC of new tests in `client_test.go`
locks the new behavior in.

# What I learned

When two systems use opposite synchronization models —
LSP push vs. crush pull — staleness lives at the boundary
and no amount of client-side polish fully eliminates it. The
practical strategy is the one used here: track version, give
the producer a brief window to catch up, and *surface*
remaining staleness to the consumer rather than hide it.
For an AI agent specifically, "may be stale" is information
the agent can act on; silent staleness is information that
will silently mislead. The general rule: when you can't
guarantee freshness, give consumers the metadata to choose
how to handle staleness, rather than pretending you've
solved a problem you can only mitigate.
