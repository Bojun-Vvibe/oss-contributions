# Context

`crush` ships a telemetry / event subsystem that fires events
for product analytics — `AppInitialized()`, `SessionListed()`,
`SessionShown()`, `SessionDeletedCommand()`, etc. The subsystem
is *opt-in* and depends on a config flag; it's initialized via
`event.Init()` only when the user has telemetry enabled.

The bug shape is straightforward: in several command paths,
events were being fired *before* `event.Init()` had run. The
events hit a not-yet-initialized event sink and were silently
dropped. From the analytics side, this looked like "huge
fraction of users do nothing in their first session" — actually
those were users whose first action fired before init.

# Problem

Two compounding shapes of the same bug:

1. **In `internal/cmd/run.go`**: `event.AppInitialized()` was
   being fired *before* the config-load + telemetry-init step.
   Any user with telemetry enabled was missing the
   `AppInitialized` event for every session. The downstream
   metric "session-start funnel" had a permanent ~zero
   conversion rate at step 1 because the step-1 event simply
   didn't fire.
2. **In `internal/cmd/session.go`**: `event.SessionListed()`,
   `event.SessionShown()`, and `event.SessionDeletedCommand()`
   were each fired at the top of their respective command
   handlers, *before* `sessionSetup` had been called — which is
   where config loads and (newly) `event.Init()` runs. Same
   silent-drop pattern, three command paths.

# Design

Two structural changes:

**1. `sessionSetup` is now responsible for `event.Init`**:

```go
// internal/cmd/session.go (new)
cfg, err := config.Init("", dataDir, false)
if err != nil { return ... }
if dataDir == "" {
    dataDir = cfg.Config().Options.DataDirectory
}
if shouldEnableMetrics(cfg.Config()) {
    event.Init()
}
```

This is the right colocation: `sessionSetup` is already the
single entry point for config + DB + workspace setup, so making
it also responsible for event-system bring-up matches the
"setup" semantics. The previous design had config loaded inside
`sessionSetup` but `event.Init` was somewhere else (or, often,
*not at all* for session subcommands), creating the race.

**2. Move every event call to *after* the corresponding setup**:

```go
// Before:
event.SetNonInteractive(true)
event.SessionListed(sessionListJSON)   // <- dropped, init hasn't run

ctx, svc, cleanup, err := sessionSetup(cmd)

// After:
event.SetNonInteractive(true)
ctx, svc, cleanup, err := sessionSetup(cmd)
defer cleanup()
event.SessionListed(sessionListJSON)   // <- captured, init has run
```

Same shape applied four times (once for `AppInitialized`, three
times for the session subcommands).

The `event.SetNonInteractive(true)` is *correctly* left at the
top — that's setting a property, not firing an event, so it's
fine before init.

# Risks

- **`event.Init()` failure modes are unclear from the diff.**
  If `event.Init()` can fail (network probe, file write for
  identity persistence, etc.), the new code path doesn't surface
  the error. Better than silent-drop, but a logged warning would
  help debug the case where init *is* called but fails.
- **`shouldEnableMetrics(cfg.Config())` is the gate.** If the
  user has metrics disabled, `event.Init()` doesn't run, which
  means subsequent event calls are *also* silent no-ops —
  intentional, but the call sites still issue them. That's the
  designed shape (events themselves should check init internally
  and no-op when disabled), but worth verifying that all event
  funcs handle the disabled case without crashing.
- **No regression test.** The fix is in command flow ordering,
  which is hard to unit-test, but an integration test that
  starts crush with telemetry enabled, intercepts the event
  sink, and asserts `AppInitialized` was sent would lock this
  in. Without it, the next refactor that re-orders these
  command paths will silently re-introduce the bug.
- **Multiple `event.AppInitialized()` call sites in `run.go`.**
  The diff shows the call moved into *both* code branches
  (continue-session vs new-session), which is correct but
  duplicates the call. Future maintainers might think one of
  them is redundant and remove it. A `defer event.AppInitialized()`
  pattern (or extraction into the setup function) would be more
  robust.
- **Stats command (`stats.go`) got the same treatment** — good
  for consistency, but the PR description doesn't mention
  whether *all* event-firing command paths were audited or just
  the visible ones. Worth a grep across `internal/cmd/` for
  remaining `event.X(...)` calls that aren't preceded by
  setup.

# Suggestions

1. Audit `internal/cmd/` for any remaining event calls that
   precede `sessionSetup` / config init. Catch the next instance
   in this PR rather than as a follow-up.
2. Make `event.Init()` log on failure rather than silently no-op.
3. Add an integration test (or a CLI smoke test) that asserts
   `AppInitialized` is emitted when telemetry is enabled.
4. Consider moving `event.AppInitialized()` itself *into*
   `sessionSetup` (or a small helper called from it), so the
   call sites can't accidentally fire it before init. Removes
   the entire bug class.

# Verdict

**Ship it.** Real fix to a real silent-drop in the analytics
pipeline. Co-locating `event.Init` with config/setup is the
right structural call. Follow-ups on test coverage and
exhaustive call-site audit are non-blocking.

# What I learned

This is a textbook "asynchronous-init + early-call" bug that
shows up in any system with two-phase startup (config-load,
then "real" init). The temptation is to fire events as early
as possible to capture the full session, but if init hasn't
run, the early events go nowhere. Worse, the bug is *invisible*
locally — telemetry sinks usually log errors only when the
network call fails, not when they're dropping events because
init hasn't happened. So the bug ships, the analytics dashboard
shows weirdly bad funnel conversion, and engineers blame the
users.

The defensive pattern is **"buffer until init, flush on init"**:
the event subsystem accepts calls before init and queues them
internally, then flushes the queue when `Init()` runs. That
shape eliminates ordering bugs in the call sites entirely. The
trade-off is unbounded buffering if init is *never* called —
needs a cap. But for any system where init is guaranteed
to happen exactly once, the buffer-and-flush pattern is the
high-leverage move.

The reusable insight is **"setup colocation"**. When you have
one function that's the canonical entry point for setup
(`sessionSetup` here), make it responsible for *all* setup
work — config, DB, telemetry, lock files, the lot. Distributing
setup across multiple call sites is exactly how this bug class
gets created, because every new caller has to remember to do
all the steps in the right order. One setup function, one
ordering, one place to forget something. That's the failure
mode this PR converges on the fix for.
