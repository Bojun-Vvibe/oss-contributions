# fix(ci): checkout PR branch instead of main in bot workflow

Two-arm checkout-ref fix in `.github/workflows/gemini-cli-bot-brain.yml`: prepends a `Determine Checkout Ref` shell step before each `actions/checkout@08c6903cd8c0...` call (one at line ~56, the other at ~234), and changes the second checkout from the hardcoded `ref: 'main'` to `ref: '${{ steps.determine_ref.outputs.ref }}'`. The new steps resolve `$ISSUE_NUMBER` from `github.event.issue.number || github.event.pull_request.number || github.event.inputs.issue_number`, query `gh pr view "$ISSUE_NUMBER" --json headRefName --jq .headRefName`, and fall back to either `${{ github.ref }}` (first arm) or `'main'` (second arm) if the gh-pr-view returns empty.

The bug this fixes is real and load-bearing for the bot's correctness: the second checkout at `:236` was unconditionally checking out `main`, which means when the bot was invoked via a `/gemini` comment on PR #N, it generated diff suggestions against `main`'s HEAD rather than against PR #N's head. Result: every bot suggestion on a PR was either (a) a no-op patch (the change was already in main), (b) a conflicting patch (main and PR head had diverged), or (c) a wrong-target patch (the bot was suggesting changes for a file that PR #N had renamed/deleted). The fix correctly threads the PR's `headRefName` through to the checkout step so the bot operates on the actual PR head.

The fallback hierarchy at the new `Determine Checkout Ref` steps is the right shape: try the issue/PR number from the event payload, look up its head ref via `gh pr view`, and fall back to a sensible default if the lookup fails. The first arm's fallback to `${{ github.ref }}` is correct for the "this workflow was triggered by something other than an issue/PR comment" case (e.g. a `workflow_dispatch` without an issue_number input). The second arm's fallback to `'main'` preserves the prior behavior for that same dispatch case, which is the safe default for the apply-patch step (you don't want a `workflow_dispatch` accidentally landing patches on a non-main branch).

The third change at `:286` (`git checkout -b "$BRANCH_NAME"` → `git checkout -B "$BRANCH_NAME"`) is the load-bearing companion to the ref-fix: with `-b`, if the workflow re-runs (retry on transient failure, second `/gemini` comment on the same PR) and the branch already exists locally, `checkout -b` errors out and the patch-apply step never runs. With `-B`, the local branch is force-reset to the current HEAD if it exists, which is the right semantic for a bot-managed working branch — the bot should always start from a clean state derived from the resolved checkout ref.

Three nits worth flagging:

(1) The `gh pr view "$ISSUE_NUMBER" ... 2>/dev/null || echo ""` pattern silently swallows real failures (gh API rate-limit, permission errors, malformed event payload) into the empty-string fallback path. For the first arm that's relatively safe — it falls back to `github.ref` which is the workflow's natural ref. For the second arm it falls back to `'main'`, which could mask a real bug (gh API misconfigured, PR #N legitimately exists, but checkout-ref-resolution silently lands on main and the bot generates wrong-target patches). A `2>&1 | tee /tmp/gh-output` and a one-line `if [ "$REF" = "main" ] && [ -n "$ISSUE_NUMBER" ]; then echo "::warning::issue $ISSUE_NUMBER did not resolve a PR head, falling back to main" ; fi` would surface the silent fallback as a workflow warning.

(2) The two `Determine Checkout Ref` steps are byte-identical except for the fallback (`github.ref` vs `'main'`). Composite-action extraction or a YAML anchor would deduplicate, but GitHub Actions YAML has limited support for both — the duplicated-step shape is the practical workaround. A comment naming the two arms ("Step 1: PR-context checkout for read-only analysis. Step 2: PR-context checkout for write/apply, falls back to main if no PR context.") would help the next reader understand why the duplication exists and which arm to update when adding a new caller.

(3) The fallback to `${{ github.ref }}` in the first arm assumes `github.ref` is always a valid checkout target. For `pull_request_review_comment` events on a PR from a fork, `github.ref` is `refs/pull/N/merge` (the merge-ref), not the PR head — `actions/checkout@v5` with `ref: refs/pull/N/merge` checks out the *merge commit*, not the PR head. The new code path with `gh pr view ... --json headRefName` returns the actual head branch name (e.g. `feature/foo`) which `checkout` resolves differently. The two paths produce different working-tree contents for fork PRs. Worth a one-line comment ("note: gh-pr-view returns headRefName which is the source branch, not the merge ref github.ref provides") so the next reader doesn't reflexively prefer the github.ref path "for simplicity."

Otherwise: this is the right fix to the right bug. The bot has been silently generating wrong-target patches, and the diff is mechanical and limited to the workflow file. No reviewable code-path concerns. Ready to merge once the silent-fallback-warning nit is addressed.

## what I learned
"Hardcoded ref: 'main'" in any GitHub Actions workflow that responds to PR-context events is almost always a bug — the workflow needs to operate on the PR's content, not main's content, and the only correct way to express that is to resolve the PR's headRefName from the event payload. The defensive shape for this kind of workflow is a single composite action ("checkout-pr-context") that owns the ref-resolution logic, used everywhere a PR-aware checkout is needed; the duplicated-step shape this PR ships is the practical compromise when composite actions are too heavy for a single repo's needs.
