# Protect HttpApi web UI fallback with auth

Closes the auth-bypass that drip-219's #25163 inadvertently opened: the catch-all `HttpRouter.add("*", "/*", ...)` UI fallback added there sat *outside* the `authorizationRouterMiddleware`, so under the experimental HttpApi backend the web UI was reachable without server credentials even when `OPENCODE_SERVER_PASSWORD` was set. This PR puts the UI route inside the same auth boundary as everything else, *and* rewrites the proxy path from a Hono-bridged `globalThis.fetch` to a real `Effect`-typed `HttpClient.execute` call so the UI can be tested with a stub client instead of by monkey-patching globals.

The load-bearing change is at `httpapi/server.ts:124-130`: the `uiRoute` now wraps `HttpRouter.add("*", "/*", (request) => serveUIEffect(request).pipe(...))` in a trailing `.pipe(Layer.provide(authorizationRouterMiddleware.layer.pipe(Layer.provide(ServerAuthConfig.defaultLayer))))`, which is the same middleware ordering that protects the API routes. The OPTIONS-preflight test at `httpapi-ui.test.ts:230-241` proves the boundary is correct: an unauthenticated `OPTIONS /` with a `origin: http://localhost:3000` header returns 204 with the CORS headers, but a `GET /` against a server with `OPENCODE_SERVER_PASSWORD=secret` returns 401 — auth gates real requests but doesn't break preflight, exactly the right contract.

The second half of the change is the `serveUIEffect` rewrite in `routes/ui.ts:124-185` that promotes the prior `serveUI(request: Request)` async function into an `Effect.Effect<HttpServerResponse>` returning function. The shape change has three concrete wins: (1) the `AppFileSystem.Service` for embedded-UI reads is service-injected via `yield* AppFileSystem.Service` instead of the bare `node:fs/promises` import, so tests can stub the filesystem layer; (2) the upstream proxy fetches via `yield* HttpClient.execute(...)` so tests can stub the HTTP client layer (the old code had to monkey-patch `globalThis.fetch` and could only catch URL-shape regressions, not response-shape ones); (3) the new `proxyResponseHeaders` helper at `:32-39` strips `content-encoding` and `content-length` from the upstream response before forwarding, with the comment naming the exact bug ("FetchHttpClient exposes decoded response bodies, so forwarding upstream transfer metadata makes browsers decode already-decoded assets again") — that's a real bug hidden in the original Hono-`proxy()`-based code that wasn't testable until this rewrite exposed the seam.

The new test "strips upstream transfer encoding headers from proxied assets" at `:135-173` pins this exact contract: stub upstream returns a response with `content-encoding: br`, `content-length: 999`, and `content-type: text/javascript`; the test asserts the response back to the browser has neither `content-encoding` nor `content-length: 999` set, but does have `content-type: text/javascript` and the body `console.log('ok')`. That's the right test shape — assert what gets stripped, what gets preserved, and what the actual body is, not just "no error thrown."

The auth tests at `:188-228` cover three positive arms (`auth_token` query param via `btoa("opencode:secret")`, `Authorization: Basic` header, and the negative arm where no credentials returns 401), which together pin the contract that the UI fallback respects all the same auth mechanisms as the API routes. The `accepts auth token for the web UI` test specifically uses the query-param form because that's the form the embedded web UI uses to bootstrap (no header-injection mechanism in a static HTML bundle), so testing that arm is load-bearing for the actual UX.

The HoNo-shaped `UIRoutes` factory at `routes/ui.ts:185` is preserved as a thin shim (`new Hono().all("/*", (c) => serveUI(c.req.raw))`) so the legacy non-experimental Hono server keeps working — that's the right backwards-compat shape, the experimental backend gets the new Effect-shaped path while the stable backend keeps the Hono path with no behavioral change.

Combined fix-and-rewrite in one PR is justified by: (a) the rewrite is what makes the auth-fix testable (the prior monkey-patched-global tests couldn't have caught the auth-bypass), (b) the upstream-header-strip bug is real and was un-fixable in the Hono path without rewriting it, (c) the diff stays under 460 lines net and the test coverage is comprehensive. Ready to merge.

## what I learned
When a "wrap a route in middleware" fix and "rewrite the route to be testable" fix land in the same PR, the question is whether the test coverage proves both halves. Here it does: the OPTIONS-preflight + 401 + 200-with-auth tests prove the middleware wrapping, the strips-upstream-encoding test proves the rewrite caught a real bug, and the legacy-Hono shim preserves backwards compat. The anti-pattern would be wrapping the existing Hono code in middleware without rewriting it — that would have fixed the auth-bypass but kept the upstream-header bug invisible.
