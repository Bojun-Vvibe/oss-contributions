# [Fix] Replace subprocess startup-import diff with static source scan

Test-shape rewrite at `tests/test_litellm/proxy/test_proxy_server.py:5513-5544` replacing the `subprocess.run([sys.executable, "-c", check], timeout=120)` startup-import diagnostic with a static `re.search` scan of `litellm/proxy/proxy_server.py` source against the registered `LAZY_FEATURES` module paths. The PR body names the win directly: "Test is 13 seconds locally instead of timing out past two minutes."

The original test (which landed with drip-218's lazy-loaded-imports infra in #26802) tried to verify the lazy-loading contract by spawning a subprocess that imported `proxy_server` and then diff'd `sys.modules` against a hardcoded list of "heavy modules that should not be present at startup." The shape was load-bearing for the goal — actually proving "module M is not imported at startup" requires actually doing the import in a clean process — but it had three real costs that this PR fixes: (a) 60-120 second runtime on slow CI runners, (b) flaked when other tests in the same vitest/pytest run had triggered lazy loads via the TestClient and polluted the module cache (the prior subprocess-isolation was supposed to handle this but timed out instead), (c) hardcoded the heavy-module list as a literal string inside the subprocess `-c` argument, so adding a new lazy feature required updating both `LAZY_FEATURES` and the test string, with no compile-time guard against drift.

The new shape at `:5532-5538` reads `LAZY_FEATURES` directly and constructs the regex `^(from\s+{module_path}\s+import|import\s+{module_path})` per feature, anchored at column 0. The anchor is the load-bearing detail — the comment at `:5530-5531` explains it correctly: "Anchor at column 0 — indented imports inside function bodies are fine (deferred until the function runs)." That's the exact distinction between top-level imports (which fire at module import time and would defeat lazy loading) and function-body imports (which fire only when the function is called and are exactly the lazy-loading mechanism the LAZY_FEATURES infra uses). A naive `re.search(rf"from\s+{module_path}\s+import", src)` without the `^` anchor would have flagged every legitimate lazy-import inside a `LazyFeatureMiddleware.dispatch()` body as a "leak."

The error message at `:5541-5544` is the right shape: it lists the actual leaked module paths, points at the file (`proxy_server.py`), and names the contract being violated ("loaded via LazyFeatureMiddleware") — so the next person who accidentally adds a top-level `from litellm.proxy.guardrails.guardrail_endpoints import ...` to `proxy_server.py` sees a single-screen error that names the file, the offending module, and the fix path, instead of "PRESENT_AT_STARTUP: ['litellm.proxy.guardrails.guardrail_endpoints']" which requires the reader to know what that means.

Trade-off worth naming: the static scan can produce false negatives that the subprocess test couldn't. Specifically, if `proxy_server.py` does `from litellm.proxy.guardrails import guardrail_endpoints as _ge` (importing the *parent* package and then attribute-accessing the leaf), Python *does* import the leaf module at top level, but the regex `from litellm.proxy.guardrails.guardrail_endpoints import` won't match. The PR doesn't add a sibling test for the import-parent-then-attribute-access path. In practice this isn't a real risk because `LAZY_FEATURES` registers full leaf paths and the convention in proxy_server.py is direct leaf imports, but a future contributor who imports `from litellm.proxy import guardrails` to "stay aesthetic" would silently defeat lazy loading. A 5-line follow-up adding `import\s+{parent_path}` patterns for each `LAZY_FEATURES` entry's parent would close this gap, but it's a follow-up not a blocker.

The deletion of the subprocess machinery (`subprocess.run`, `out_lines`, `report` parsing) at `:5516-5547` is clean — no orphaned imports, no dead error-message-formatting code. The trade is "test loses end-to-end runtime verification but gains 6× speedup and zero-flake guarantee" which is exactly the right trade for a CI test on a hot lazy-loading code path.

## what I learned
Test-as-spec versus test-as-experiment is the recurring tension in CI. The deleted shape (subprocess + sys.modules diff) was an experiment — actually run the system, observe the output. The new shape (static regex scan) is a spec — assert the source code conforms to a stated invariant. For a contract like "no top-level imports of these modules," the spec form is strictly better: it's faster, it's deterministic, it catches the contract violation at PR-review time instead of at runtime, and the failure mode names the contract directly. The experiment form is only needed when the spec form can't capture all the ways the contract could fail (here, the import-parent-then-attr-access edge case is the spec form's only blind spot, and it's narrow enough to live with).
