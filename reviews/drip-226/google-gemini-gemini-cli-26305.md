## Notes
- Closes the asymmetry between `/mcp add` (existed via settings file edit) and `/mcp enable|disable` (existed) by adding the missing `/mcp remove <name> [--scope user|workspace|all]` (alias `rm`). Single-file `cli/src` surface ‚Äî uses the existing `McpClientManager.restart()` path rather than adding a new `removeServer()` method to core, which is the right minimal-surface move that doesn't require coordinating with a core-API release.
- The load-bearing UX decision is the multi-scope refusal at the dual-defined case: when a server exists in *both* user and workspace scopes, the command refuses with a teaching error message and asks for explicit `--scope`. This avoids the silent zombie-config pitfall where deleting one copy leaves a forgotten copy in the other scope still active after restart. `refuses to delete when server exists in both scopes without explicit --scope` test at `mcpCommand.test.ts:441-454` pins this anti-behavior contract ‚Äî `setValueMock` must *not* be called and the error must contain `"BOTH workspace and user"`.
- `--scope all` semantics at `mcpCommand.test.ts:472-498` correctly assert `setValueMock.toHaveBeenCalledTimes(2)` with both scope arms invoked in lockstep ‚Äî the right shape for "delete from everywhere" since one-arm-only would leave the zombie. The success message names which scopes were affected (`"workspace and user"`) so the user gets immediate audit visibility.
- Extension-provided servers are rejected at `:386-393` with a "use `/extensions` instead" hint ‚Äî correct because they live in the extension manifest (read-only from the slash-command's POV) rather than in user/workspace settings, and silently failing or partially-removing would leave a broken state. Test pins `setValueMock` is *not* called.
- The completion handler at `:511-514` filters out extension-origin servers from tab-completion suggestions for `/mcp remove` ‚Äî the right shape because they can't actually be removed via this command, so suggesting them would mislead. Pinned by `completion returns servers without extension origin matching prefix` test.
- Documentation entry at `mcp-server.md:1207-1227` names the load-bearing zombie-config rationale inline ("avoids the surprise where deleting the workspace copy leaves a forgotten 'zombie' copy in your user settings still active after the next restart") so a future doc-maintainer doesn't simplify away the dual-scope explanation.

## Nits
- `--scope` is the right name (mirrors the existing `gemini mcp remove --scope` CLI subcommand) but the parallel `/mcp enable|disable` uses `--session` for in-memory persistence ‚Äî the orthogonality is documented in the PR description but not in the slash-command's own help text. A one-line "see /help for /mcp enable's --session flag" near the docs at `:1219` would close the loop.
- After `setValue(scope, 'mcpServers', {})` at the test arms (`:413, :463, :476, :479`) the entire `mcpServers` map is replaced with `{}` ‚Äî correct for the single-server fixture but worth a regression test arm where the scope contains *two* servers and only the named one is removed (asserting the other server's entry survives in the new map).
- `restartMock` is called once after every successful removal ‚Äî for the `--scope all` two-arm case it's called once at the end which is correct, but for a future "remove three servers in one command" extension the call count would need to stay at 1 to avoid restart thrash. Worth a comment near the `restart()` call site naming the "exactly once per `/mcp remove` invocation" contract.
- The `'unknown'` server-name test at `:374-381` correctly expects `"'unknown' not found"` ‚Äî but the actual error string in the new arm at `mcpCommand.ts` isn't visible in the diff snippet. Worth confirming the error message exactly contains the user's input string (not a sanitized form) so a typo like `/mcp remove sevrer1` produces `"'sevrer1' not found"` rather than `"'server1' not found"` (which would be confusingly close to a valid name).
- The previous attempt at this issue (#19025) is named in the PR description ‚Äî worth a one-line `// supersedes #19025` comment near the new subcommand definition for future archaeology.
