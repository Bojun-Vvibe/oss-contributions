# BerriAI/litellm#26944 — chore(sso): bind generic SSO state to a session cookie

- Link: https://github.com/BerriAI/litellm/pull/26944
- Head SHA: `2c852ba2b1a6729ad36e922898b670b353d111a1`
- Files: `litellm/proxy/management_endpoints/ui_sso.py` (+63/−4), `tests/.../test_ui_sso.py` (+321/−0)

## Notes
- CVE-shaped Login-CSRF / token-theft fix on the Generic SSO PKCE flow, closing the gap where `state` was used as a cache key for the PKCE `code_verifier` without being bound to the caller's browser. Pre-fix, an attacker could pre-mint a state, prime the verifier cache, hand the resulting login link to a victim, and have the victim's auth code exchanged with the attacker's verifier on callback — producing an access token under the attacker's control. The non-PKCE branch was unaffected because it delegates to `fastapi-sso`'s `verify_and_process` which performs its own session-cookie check; the PKCE branch bypassed that helper, which is exactly the seam this PR closes.
- Two-part fix at the right boundaries: (a) on the redirect side, `get_generic_sso_redirect_response` at `ui_sso.py:2384-2412` now sets a `litellm_oauth_state` cookie (`HttpOnly`, `SameSite=lax`, `max_age=600`, `Secure` defaulting to True unless the request scheme is plain HTTP) carrying the state value used in the redirect URL, set on the redirect response itself rather than on a separate response — adjacent to the existing `litellm_cp_return_to` cookie; (b) on the callback side, `get_generic_sso_response` at `:1158-1180` validates `request.cookies.get("litellm_oauth_state")` against `request.query_params.get("state")` via `secrets.compare_digest` before invoking the PKCE token exchange. Mismatch (or either being missing) raises `ProxyException` with HTTP 400 *before* any code-verifier lookup runs, so an attacker can't probe verifier presence either.
- The `secure_flag = request is None or request.url.scheme == "https"` predicate at `:2403` is the right production-safe default — it requires HTTPS for the cookie unless we can *prove* the request is plain HTTP (local dev), so the failure mode is "cookie is too secure for dev" (visible, fixable) rather than "cookie is sent in cleartext to a network observer" (silent, exploitable). The `request is None` arm preserves backwards-compat for the existing `debug_sso_login` callsite while still defaulting Secure when no request is plumbed through.
- The cookie is *only* set inside the `code_verifier` branch (PKCE in use), not on every redirect — comment at `:2390-2397` names the load-bearing reason: two concurrent SSO sessions in one browser (one PKCE, one plain) would otherwise overwrite each other's state cookie. This is the right scope-down that prevents the fix from regressing the non-PKCE path.
- The 4-test `TestPKCEStateCookieBinding` class at `test_ui_sso.py:5772+` covers the right four arms: (1) PKCE-on sets the cookie with HttpOnly+SameSite=lax+Secure (load-bearing positive case), (2) PKCE-off does *not* set the cookie (anti-behavior pin against a future "always set the cookie" regression that would collide concurrent sessions), (3) HTTP dev request drops Secure (load-bearing dev-ergonomics arm), (4) callback with mismatched cookie/state is rejected with 400 (the actual CVE arm) — though only the first three are visible in the diff window I reviewed.

## Nits
- The pre-existing TODO at the original `:2287-2290` is removed and replaced with the explanatory comment at `:2307-2313` — but the new comment doesn't link to a CVE/advisory ID or the fixing PR number for future archaeology. A `# CVE-… / PR #26944` reference would help the next reader trace the rationale.
- `secrets.compare_digest(url_state, cookie_state)` at `:1170` is the right primitive but both operands need to be `str` (or both `bytes`) — if either ever flows in as `None` the `not url_state or not cookie_state` short-circuit at `:1167-1169` correctly catches it, but a type assertion or explicit `assert isinstance(url_state, str)` after the guard would surface the type contract for static checkers.
- Cookie `max_age=600` (10 minutes) is a sensible default but undocumented — a one-line `# 10-minute window: covers slow MFA/IdP redirects without leaving the cookie around for replay` comment near `:2410` would name the threat-model reasoning.
- The `request: Optional[Request]` parameter is now threaded through `get_sso_login_redirect` → `get_generic_sso_redirect_response` and the existing `debug_sso_login` callsite passes it correctly at `:4032`, but the `google_login` site at `:677` was already passing the request — worth a grep across the codebase to confirm no other `get_sso_login_redirect` caller is omitting it (silently re-enabling the always-Secure default which is fine for prod but breaks any other dev callsite).
- The fix is PKCE-specific by design but the comment at `:2284` could be more emphatic that the non-PKCE `verify_and_process` path's session-cookie check is *not* validated against the same `litellm_oauth_state` key — a future PR consolidating both branches under one cookie name would need to be careful not to collide.
