# block/goose#8929 — fix(providers): refresh GCP metadata server token on expiration

## Notes
- Closes a real long-running-session bug at the GCP ADC metadata-server credential boundary. Pre-fix, `AdcCredentials::DefaultAccount(TokenResponse)` cached the *one-time-fetched* token response in the credentials struct itself at process start, and `get_default_access_token` simply cloned it — meaning the access token was effectively immortal for the lifetime of the goose process, and the only time a fresh token was obtained was on restart. For long-running goose sessions on GCE/Cloud Run/GKE this manifests as opaque 401s on every GCP-API call after the standard ~1h token TTL elapsed, with no recovery short of restarting the whole process.
- The fix correctly restructures the variant to store the metadata-server *base URL* instead of the token: `DefaultAccount(String)` at `:60-62` with `#[serde(skip)]` (the right serde attribute since the in-memory variant should never be persisted to disk — it's specific to a runtime-discovered URL). The discovery path at `AdcCredentials::default_account` is shortened: instead of parsing the token response into `TokenResponse {token_type, access_token, expires_in}` and stashing it, it now just returns `DefaultAccount(base_url.to_string())` at `:265-266` after the metadata-server health check succeeds.
- `get_default_access_token` at `:539-563` now actually does the metadata-server HTTP call (`GET <base_url>/computeMetadata/v1/instance/service-accounts/default/token` with `Metadata-Flavor: Google` header) on every invocation — this is the right shape because the standard `GcpAuth` caching mechanism upstream (which respects `expires_in`) wraps this call, so each call is cached for the token's lifetime and refreshed automatically on expiration. Comment at `:537-538` correctly names the rationale ("token expires. Better to keep the base_url, and fetch a new token when needed").
- Error handling at `:548-562` is correct: non-success status codes return `AuthError::TokenExchange(format!("Status {}: {}", ...))` with the response body included for diagnostics, and JSON-parse failure is mapped to `AuthError::TokenExchange("Invalid response: ...")`. The existing test at `:1037-1040` is updated in lockstep to assert `base_url == mock_server.uri()` rather than the prior token-content assertions — preserving the test's original intent (verify the discovery path correctly identifies the metadata server) while shifting the assertion to the new representation.

## Nits
- `#[serde(skip)]` on the `DefaultAccount(String)` variant at `:60-62` is correct but unaccompanied — the other three variants (`AuthorizedUser`, `ServiceAccount`) presumably *do* serialize. Worth a `// not serialized: this variant carries a runtime-discovered metadata server URL that is process-local` doc-comment naming why the asymmetry exists, so a future "let's persist credentials to disk for fast restart" PR doesn't silently drop this case.
- The `Metadata-Flavor: Google` header is correctly added at `:546` but `Metadata-Flavor` is also the standard `403 Forbidden` short-circuit on non-GCE hosts — the new error at `:550-562` would catch this but emit "Status 403" rather than "this host has no GCE metadata server." A small `if status == 403 { return Err(AuthError::Credentials("not running on GCE/GCP host: ...")) }` arm would produce a more actionable error for developers who deploy goose to a non-GCP host with stale ADC config pointing at the metadata server URL.
- No test for the *refresh* path itself — the diff only updates the discovery test. A second test arm where two consecutive `get_default_access_token` calls hit the mock server twice (asserting `mock_server.received_requests().await.len() == 2`) would pin the "actually re-fetches" contract — pre-fix this would have failed (would only hit once or zero times depending on caching), post-fix it should pass exactly twice.
- `client.get(format!("{}{}", base_url, metadata_path))` at `:543` does naive string-concat — fine for the well-known metadata server URL but fragile if `base_url` ever ships with a trailing slash. A one-line `base_url.trim_end_matches('/')` or use of `reqwest::Url::join` would defend against the inevitable config-drift bug.
- The PR description's "Related Issues" section is empty — worth linking to the originating bug report or user log for future archaeology, even just as a `// fix: long-running goose sessions on GCE 401 after ~1h` comment near the new method.

**Verdict: merge-as-is**
