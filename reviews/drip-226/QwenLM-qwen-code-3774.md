## Notes
- Closes the read-less-content-oracle attack surface introduced by the cache-only design of #3717. Pre-PR, a model could call Edit with a candidate `old_string` it never saw via ReadFile, and the response (`OCCURRENCE_MATCH` vs `NO_OCCURRENCE_FOUND` vs `EXPECTED_OCCURRENCE_MISMATCH`) leaked information about the file's contents ŌĆö letting a model probe candidate strings ("does the file contain `sensitive token: hunter2`?") without ever reading it. The new enforcement at Edit / WriteFile rejects with `EDIT_REQUIRES_PRIOR_READ` *before* `calculateEdit` runs, regardless of whether the candidate would have matched. Test `does not let an unread file be probed via NO_OCCURRENCE_FOUND` at `edit.test.ts:1099-1117` directly pins this anti-behavior.
- Two new error codes at `tool-error.ts:+11` ŌĆö `EDIT_REQUIRES_PRIOR_READ` (no cache entry) and `FILE_CHANGED_SINCE_READ` (mtime/size drifted since last read) ŌĆö with model-actionable error messages telling the model to use `read_file` first or re-read before retrying. The two-code split is the right granularity: a model receiving `FILE_CHANGED_SINCE_READ` can re-read and retry without reasoning about why, while `EDIT_REQUIRES_PRIOR_READ` signals "you skipped a step in your plan" which deserves a different recovery strategy.
- The "ranged read does not satisfy enforcement" arm at `edit.test.ts:957-976` is load-bearing ŌĆö `lastReadWasFull = false` (recorded when ReadFile is called with offset/limit) is correctly treated as "the model has not seen the bytes a free-form `old_string` would touch", because the model has no way to know whether its candidate string falls inside or outside the read range. Pinned by `rejects an edit when the previous read was ranged (offset/limit)` exact-error-code assertion. Same shape for non-cacheable reads (binary/PDF/image at `lastReadCacheable=false`) at `:978-1000`.
- Auto-memory bug fix at `read-file.ts:+9/-2` decouples two previously-coupled behaviors: auto-memory paths still skip the `file_unchanged` fast-path (the per-read freshness `<system-reminder>` must always reach the model) but now *do* record into the cache, so a follow-up Edit on `AGENTS.md` after Read sees `fresh`. Pre-fix, #3717's blanket cache-skip for auto-memory paths meant the new enforcement would have rejected every Edit on AGENTS.md immediately after a successful Read of it ŌĆö a self-regression #3717 + this PR would have shipped together. New regression test in `read-file.test.ts` asserts the cache-record-but-no-fast-path contract.
- The compaction-edge-case rationale ("first Edit after compaction will be rejected with `EDIT_REQUIRES_PRIOR_READ`; the model re-reads; one extra Read per compaction event is the cost") is correctly identified in the PR description as the most-likely false-positive ŌĆö and the existing `Config.fileReadCacheDisabled` flag is the right escape hatch (one toggle reverts to pre-cache behavior). Sub-agent inheritance via per-Config own-property machinery from #3717 means a sub-agent's first Edit requires its own Read, which is the correct safety property for tasks delegated to sub-agents.
- The `seedPriorRead` test helper at `edit.test.ts:108-118` is the right pattern ŌĆö every business-logic test that exercises Edit's diffing/encoding/replace_all behavior calls `seedPriorRead(filePath)` after `fs.writeFileSync(filePath, ŌĆ”)` so the test isolates Edit-business behavior from prior-read enforcement. The doc-comment at `:101-107` names the contract so a future contributor adding an Edit business test doesn't forget to seed.

## Nits
- The two `requirePriorRead` helpers live separately on `EditToolInvocation` and `WriteFileToolInvocation` (PR description acknowledges this). The function bodies are nearly identical ŌĆö extracting to `core/src/tools/util/require-prior-read.ts` taking `(config, path) ŌåÆ Result<void, ToolError>` would dedupe and pre-position for the inevitable third caller (`MultiEdit` is named in the PR description). Cheap to do now while the diff is small.
- Error messages like `"File /work/config.ts has not been read in this session. Use the read_file tool first..."` are model-actionable but include the absolute path ŌĆö for sandboxed environments where the path is sensitive this leaks more than necessary. Consider `cwd`-relative path rendering when possible.
- The PR description mentions auto-memory regression test in `read-file.test.ts` but the diff window I reviewed shows only the test file delta size (+36); worth confirming the test asserts both the positive arm (Read AGENTS.md ŌåÆ cache record exists) AND the negative arm (Read AGENTS.md ŌåÆ file_unchanged `<system-reminder>` still emitted on next Read).
- No test for the cross-tool case "Edit `/foo`, then WriteFile `/foo` (overwrite)" ŌĆö the cache state after Edit's `recordWrite` should let WriteFile's enforcement see `fresh`. Worth an integration test arm.
- The `FILE_CHANGED_SINCE_READ` error message ("mtime or size changed") tells the model to re-read but doesn't include the *direction* of change ŌĆö for very large files, mtime-only-changed-but-size-same might warrant a different recovery (full re-read vs ranged re-read). One-line ergonomic improvement.
