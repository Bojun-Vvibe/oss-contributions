## Notes
- Closes a silent-drop bug at three sites in the Bedrock Converse message-conversion pipeline by adding `elif element["type"] == "document"` arms at the async user-message path (`factory.py:4592-4594`), the sync user-message path (`:5031-5033`), and the tool-result path (`_convert_to_bedrock_tool_call_result`, `:4095-4100`) — all three call into a new `_process_document_message` static method at `:4857-4912` that maps the Anthropic-style `{type:"document", source:{type:"base64", media_type, data}}` shape to the Bedrock `DocumentBlock` shape with format string, base64 bytes, deterministic name, and the previously-unreachable `citations`/`context` fields. The fix is at the converter source-of-truth (one helper, three call sites) rather than per-call-site duplication.
- The deterministic-name derivation at `:4889-4895` uses SHA-256 over a length-prefixed sample (first 64KB of normalized base64 + ascii-encoded full length suffix) producing `Document_<16hex>_<format>` — the load-bearing detail is the `str(len(normalized)).encode("utf-8")` length suffix that prevents two distinct documents whose first 64KB happen to match (e.g. two PDFs with the same header) from colliding on `name`. The 16-hex truncation gives 64 bits of entropy which is plenty for a per-conversation document-name disambiguator. `test_document_block_deterministic_name` at `test_…:121-129` pins the contract by exact value-equality.
- The `DocumentBlock` `TypedDict` flip at `types/llms/bedrock.py:46-55` from `total=True` to `total=False` + explicit `Required[…]` markers on the three pre-existing fields (format/source/name) is the right shape — it lets the new `citations`/`context` keys be omitted without losing static-type-check coverage on the load-bearing trio. New `CitationsConfig(TypedDict, total=False)` at `:46-47` matches the Bedrock API's `{enabled: bool}` shape.
- The 9-test arm at `test_bedrock_document_blocks.py` is well-shaped: `test_document_block_not_dropped_with_text` directly reproduces the silent-drop bug ("mixed document + text preserves both blocks"), `test_document_block_citations_not_set_when_disabled` asserts the absence-when-`enabled:False` contract (anti-behavior pin against a future "always set citations" regression), `test_document_format_mapping` covers the seven MIME → Bedrock-format-string entries, and `test_non_base64_source_raises` pins the `BadRequestError` exact-message contract for the non-base64 source-type rejection.

## Nits
- `_process_document_message` raises `litellm.BadRequestError(model="", llm_provider="bedrock", …)` with empty `model` at `:4869-4874` — the upstream callers all have `model` in scope as a parameter (`_bedrock_converse_messages_pt(messages, model, custom_llm_provider)`), so threading it into the helper would produce a more actionable error message ("model X received non-base64 document"). Cheap to thread.
- The MIME→format dict at `:4877-4887` uses `media_type.split("/")[-1]` as the fallback — for novel MIME types like `application/x-bzpdf` this produces `x-bzpdf` which Bedrock will reject with a less-actionable error than the helper itself could give. A stricter "raise on unknown MIME unless `media_type` matches `^application/[a-z]+$`" would make the failure mode legible at the right boundary.
- The `HASH_SAMPLE_BYTES = 64 * 1024` cap at `:4892` is silently truncating the digest input — for two near-identical documents that diverge only after the first 64KB, the SHA-256 prefix will collide and only the length suffix differentiates them. Consider documenting the sample-then-length-suffix scheme inline at `:4892` so a future contributor doesn't "fix" it to a full-content hash and introduce a memory regression on large PDFs.
- No test arm for the tool-result path's document handling at `_convert_to_bedrock_tool_call_result:4095-4100` — `test_document_in_tool_result` is mentioned in the PR description but the diff window I reviewed cuts off mid-test-file; if it's not in the final 60 lines, it should be added.
- The `BedrockSourceBlock(bytes=data)` shape at `:4904` passes the base64 string directly as `bytes` — relying on the downstream serializer to handle the encoding. Worth a comment naming the `bytes` field's expected wire shape (`base64-encoded str`, not raw `bytes`) so the next contributor doesn't add a `base64.b64decode` "fix."
