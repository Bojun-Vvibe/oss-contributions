## Rationale

This PR walks back the `notify` deprecation that was rolled out in a previous cycle: it strips the `DeprecationNotice` event emission from `core/src/session/session.rs:573-590`, removes the `LEGACY_NOTIFY_CONFIGURED_METRIC`/`LEGACY_NOTIFY_RUN_METRIC` metric counters from `otel/src/metrics/names.rs:35-36` (and their callsites in `session/turn.rs:580-585`), deletes the `emits_deprecation_notice_for_notify` test in `core/tests/suite/deprecation_notice.rs:117-148`, and re-words the README/config schema/struct doc-comments to drop the "deprecated" language. The new file `codex-rs/hooks/src/user_notification.rs` resurrects the legacy notify wire format (`type=agent-turn-complete`, kebab-case JSON keys) as a `legacy_notify` hook — keeping the historical contract intact for users who already wired up `notify-send` / `terminal-notifier` scripts.

**Why "needs-discussion" rather than approve:** the *implementation* is clean — the new `user_notification.rs` file ships with two unit tests (`test_user_notification` and `legacy_notify_json_matches_historical_wire_shape` at lines 282-329) that pin the JSON wire format byte-for-byte against an `expected_notification_json()` golden, which is exactly the right test to have for a backward-compat surface. The README rewrite at `codex-rs/README.md:46-49` is a clean reversion to the pre-deprecation copy and reads well. The doc-comment edits in `core/src/config/mod.rs:480-499` correctly drop both the "Deprecated" prefix and the "Use lifecycle hooks for new automation" trailer. So mechanically this is a fine revert.

**However**, this is a policy reversal, not a code fix. The original deprecation was a deliberate, telemetry-instrumented attempt to herd users off `notify` toward the new `Stop` hook (the deleted `DeprecationNoticeEvent` literally pointed users to `https://developers.openai.com/codex/hooks`). Reverting it raises questions the diff doesn't answer:

1. **What's the new long-term plan?** Is `notify` now a permanently-supported config knob, or is the plan to deprecate again later with better hook ergonomics? The README/doc-comment edits *remove* the deprecation language entirely (e.g. `core/src/config/mod.rs:499` now just says "If unset the feature is disabled.") — readers will reasonably conclude this is a stable supported feature. If that's wrong, this PR is setting up a worse second deprecation later.
2. **Is `legacy_notify` hook the canonical path going forward?** The new file `hooks/src/user_notification.rs:215-232` registers a hook named `"legacy_notify"`. If `notify` is *not* deprecated, why is the hook prefixed `legacy_`? That naming will read as deprecation-by-implication to anyone reading the source. Either rename it to `notify_hook` (and drop the "Legacy notify payload" doc string at line 13) or be explicit in the README that `notify` is internally serviced by a hook called `legacy_notify` and that's intentional.
3. **The deleted telemetry counters (`codex.notify.configured`, `codex.notify.run`) were how the team was *measuring* migration progress.** Killing them with no replacement means there's no longer a way to know how many users still depend on `notify` for the next conversation about its future. Worth keeping at least `codex.notify.configured` even without the deprecation event, purely for adoption telemetry.
4. **Wire-format test asserts kebab-case** (`thread-id`, `turn-id`, `input-messages`, `last-assistant-message` at `hooks/src/user_notification.rs:268-275`). That matches the `serde(rename_all = "kebab-case")` on the enum at line 16, which matches the README example. Good — the contract is properly pinned. But there are no integration tests in this PR proving the `legacy_notify` hook actually fires when `config.notify` is set; the deleted `emits_deprecation_notice_for_notify` test should be replaced (not just removed) with an `emits_legacy_notify_hook_when_notify_configured` test that runs through `Session` and asserts a real spawn.

**My recommendation:** before merging, the PR description should answer (1) and (2) explicitly, and the diff should add (3) the adoption-telemetry counter back and (4) an integration test that proves the hook actually wires up. Until that conversation happens, holding at `needs-discussion`.
