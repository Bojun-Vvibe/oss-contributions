## Stated purpose
The `externalAgentConfig/import` JSON-RPC endpoint previously blocked on session-history import for all selected sessions before responding — a single import of a several-thousand-message external agent transcript could leave the request hanging long enough for upstream IDE clients to time out. This PR moves session import work to the background, returns from the RPC immediately after validation/preparation, and emits the existing `externalAgentConfig/import/completed` notification when the actual write-to-disk work finishes. It also serializes session imports so duplicate concurrent requests cannot produce duplicate imported threads.

## What actually changed
1. **`external_agent_config.rs:144-167`** — `detect_recent_sessions` is replaced by `external_agent_session_source_path(path)` which canonicalises the input `path`, canonicalises the configured `external_agent_home/projects` root, and only returns `Some(path)` if the canonical path is a `.jsonl` file under the projects root. This is the security gate that prevents an attacker-supplied `migrationItems[].cwd` from referencing an arbitrary file on disk; `fs::canonicalize` is the right tool here because it follows symlinks before the prefix check, so a symlinked `.jsonl` outside the projects root will be rejected after resolution. Returning `Ok(None)` for `NotFound` (rather than propagating the IO error) is the correct UX choice — a missing source file at import time is a "skip silently" not a "blow up the whole batch".
2. **`external_agent_config_api.rs`** — switches from `prepare_pending_session_imports` (which surfaced `PrepareSessionImportsError`) to `prepare_validated_session_imports`, suggesting the validation step has been hoisted up so the prepare-phase no longer fails on per-session validation. Adds `Arc<Semaphore>`, `HashSet<...>`, and uses these to serialize session imports — the semaphore caps concurrency (likely permits=1 for the duplicate-prevention guarantee), and the HashSet of in-flight `ThreadId`s is the dedup gate.
3. **`codex_message_processor.rs:516`** — `CodexMessageProcessor` gains `#[derive(Clone)]`. This is the load-bearing change for the background-task pattern: the message processor is now `Clone`-able so the import handler can clone it before `tokio::spawn(async move { ... })`, then call back through the cloned processor to emit the `externalAgentConfig/import/completed` notification when the spawned task finishes.
4. **README.md** — protocol contract updated to reflect that `externalAgentConfig/detect` may now return `details` for both plugin and *session* migration items, and that `externalAgentConfig/import` may emit `externalAgentConfig/import/completed` after the full import finishes — "immediately after the response when everything completed synchronously, or after background remote imports finish" is upgraded to "after background imports finish" (the qualifier "remote" is dropped because session imports now also go async). Good docs hygiene — the wire contract change is documented in the same PR as the implementation.

## Quality / risk observation
The two-phase split — validate synchronously inside the RPC, do work in the background, fire-and-forget the completion notification — is the textbook async-import pattern, and the PR gets the three load-bearing details right: (a) `Clone` on the message processor so the spawned task can callback, (b) semaphore + in-flight set for duplicate-prevention which is *necessary* because clients may retry an apparently-hung request and you'd otherwise produce two copies of the same imported thread, (c) `fs::canonicalize` before the projects-root prefix check so the `.starts_with(projects_root)` test isn't defeated by symlinks. The `Ok(None)` for `NotFound` on either the source path *or* the projects root canonicalization is also correct — a non-existent source file should be a per-item skip, and a missing projects root means "no external agent installed" which should also be a skip rather than a 500. One observation about test coverage: the verification list mentions `cargo test -p codex-app-server external_agent_config_` and `cargo test -p codex-external-agent-sessions` but no explicit test for the "duplicate concurrent request → exactly one imported thread" property. Given that's the whole point of the semaphore + in-flight-set pair, a focused concurrent-request test using `tokio::join!` of two `import` calls with the same migrationItems and asserting exactly one `record_imported_session` call would tighten the regression net considerably. The 465 added / 57 deleted ratio is what you'd expect for this kind of behavioural lift — most of the additions are the new background-task plumbing plus the validated-imports API surface.
