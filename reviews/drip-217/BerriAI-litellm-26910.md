# Review: BerriAI/litellm#26910 — fix: drop milvus dbName and partitionNames from MILVUS_OPTIONAL_PARAMS

- PR: https://github.com/BerriAI/litellm/pull/26910
- Head SHA: `dfc080f58043f98222840b12d038dbd8ef6ad593`
- Author: yassin-berriai
- Files: 3 (+~165/-6)
- Merged: 2026-04-30

## Stated purpose
Close a multi-tenant data-leak vector in the Milvus vector-store passthrough. Previously, `dbName` and `partitionNames` were in the `MILVUS_OPTIONAL_PARAMS` set, meaning they were forwarded from the *user-supplied* request body straight into the Milvus search API request. Because litellm acts as a proxy in front of a backend Milvus instance shared across tenants, a user could pass `dbName: "other_tenant_db"` or `partitionNames: ["other_tenant_partition"]` in their search request and read another tenant's vectors. The fix removes both keys from the user-passthrough allow-list and re-adds them as backend-configured `litellm_params` (`milvus_db_name`, `milvus_partition_names`).

## What actually changed
1. **`llms/milvus/vector_stores/transformation.py:25-34`** — `"dbName"` and `"partitionNames"` are removed from the `MILVUS_OPTIONAL_PARAMS` frozenset that gates which user-supplied search params get forwarded. The remaining `annsField`, `limit`, `filter`, `groupingField`, `outputFields`, `searchParams`, `consistencyLevel` are all per-query semantic options with no cross-tenant blast radius.
2. **`llms/milvus/vector_stores/transformation.py:171-188`** — `request_body` now reads `db_name = litellm_params.get("milvus_db_name")` and `partition_names = litellm_params.get("milvus_partition_names")` and only sets the corresponding keys on the outgoing Milvus request *if they are truthy*. This is the load-bearing change: the values now flow from the operator-controlled `litellm_params` (configured server-side per route) instead of the user-controlled request body.
3. **`types/router.py:237-241`** — adds `milvus_db_name: Optional[str] = None` and `milvus_partition_names: Optional[List[str]] = None` to `GenericLiteLLMParams`. This is the type-level contract that operators can now set these in `config.yaml` under `vector_store_config:` blocks.
4. **`tests/vector_store_tests/test_milvus_vector_store.py`** — adds `_extract_request_body(mock_post)` helper that handles both `data=` (JSON-stringified) and `json=` (dict-shaped) call shapes, plus two new tests: `test_user_supplied_db_and_partition_are_dropped` (positive: pass `dbName="other_tenant_db"`, `partitionNames=["other_tenant_partition"]` in the user request, assert neither key appears in the outgoing Milvus request body) and `test_backend_configured_db_and_partition_are_forwarded` (positive-side: pass them via `litellm_params`, assert they appear). The former is the security regression test, the latter pins the new operator-controlled shape so a future refactor doesn't silently strip them on both sides.

## Quality / risk observation
This is a textbook "wrong layer for trust" fix — the param was being sourced from the wrong actor (user) when it controls cross-tenant scope (which database, which partition). The fix moves the source of truth to `litellm_params`, which is the right call. One observation about completeness: the security model implied here is that operators set `milvus_db_name` and `milvus_partition_names` per route and users cannot influence them. But the request-body strip path uses the `MILVUS_OPTIONAL_PARAMS` allow-list as the gate — if a user sends `dbName: "x"` in the body, it's now silently dropped (not 400-rejected). Silent strip is the more conservative behaviour for a security fix (don't leak the existence of the partitioning to the caller), but the test only asserts absence in the outgoing request, not behaviour to the caller — worth confirming that the user gets a 200 with results from the *operator-configured* db, not a confusing "your dbName was ignored" warning. Second observation: `if db_name:` and `if partition_names:` use truthy checks — `partition_names: []` (empty list) would be treated as "unset" and not sent. That's the right Milvus-API behaviour (an empty `partitionNames` arg is meaningless) but worth a comment. Third: there's a subtle backward-compat concern for existing operators who were *intentionally* relying on the user-passthrough behaviour (e.g. single-tenant deployments with a per-user Milvus db model) — they'll see their workflow silently break after upgrade. A `litellm` changelog note flagging this as a security-driven breaking change would help. The vulnerability is real and the fix is correct; ship it.
