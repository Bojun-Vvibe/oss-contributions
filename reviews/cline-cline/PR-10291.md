# Context

cline's Windows CI job has been the flakiest leg of the matrix
for several months. The failures cluster in three places — the
CLI skills-panel tests, the BannerService cache/retry tests,
and the VS Code integration setup — and each one has the same
structural cause: tests that *guess* timing instead of
*observing* state. On a fast macOS runner the guess is right
often enough to pass; on a slow Windows runner the guess is
wrong often enough to flake.

# Problem

Three concrete classes of flakiness, each with a different
underlying mechanism:

1. **Skills panel tests use `setTimeout`-based readiness
   checks.** The pre-PR test pattern was "wait 50ms then
   assert". On a fast runner the panel had finished loading
   in <50ms; on Windows CI it hadn't. The test would then
   interact with a half-loaded panel, which interpreted the
   input differently than the test expected. Failure mode:
   intermittent assertion failure that disappears on rerun.
2. **BannerService tests have racy fake-timer + async
   interleaving.** The service does cache lookups, retries,
   and auth refreshes — all on async/timer schedules. The
   pre-PR test code mixed `vi.useFakeTimers()` with
   `await Promise.resolve()` flushing in a way that worked
   *most* of the time but had at least one ordering where
   the fake-timer advance landed before the async resolution.
3. **VS Code integration setup downloads the VS Code runtime
   over the network at test-suite startup.** On a slow Windows
   runner the download could time out, fail mid-flight, or
   race with the test suite's own startup deadline. Failure
   mode: setup error, no test results at all.

# Design

Three separate fixes, each addressing one cause:

**1. Replace timing guesses with state observation.**

```tsx
// Before (cli/src/components/SkillsPanelContent.test.tsx):
await new Promise((r) => setTimeout(r, 50));
expect(panel).toHaveTextContent("Skills");

// After:
await waitFor(() => expect(panel).toHaveTextContent("Skills"));
```

`waitFor` (Testing Library) polls until the assertion passes
or hits a per-test timeout. The runtime-side change is
complementary: the panel itself ignores input while
`isLoading`, so the test's interaction with a still-loading
panel becomes a no-op rather than a wrong-state interaction.
That removes a *class* of test bugs by making the runtime
robust to the same race the test was racing.

**2. Deterministic fake-timer + explicit async drain.**

The BannerService test rewrite (+42/-23) moves to an
explicit pattern:

```ts
vi.useFakeTimers({ shouldAdvanceTime: false });
// ... arrange ...
await vi.advanceTimersByTimeAsync(retryDelayMs);
await vi.runAllTimersAsync();
// ... assert ...
```

`advanceTimersByTimeAsync` and `runAllTimersAsync` are the
Vitest APIs that *interleave* timer advance with microtask
flushing. They eliminate the "fake-timer advance landed
before pending promise" race that the older
`vi.advanceTimersByTime()` + `await Promise.resolve()` pattern
left open. This is the right modern pattern for Vitest fake
timers and should be the default for any test that mixes
timers and async work.

**3. Pin, cache, and retry the VS Code download.**

`.github/workflows/test.yml` (+24/-1) and `.vscode-test.mjs`
(+2/-1) pin the VS Code runtime version, cache it across
CI runs, and add a workflow-specific retry on download failure.
The retry is narrow (only the download step, not the test
run), which is the right discipline — retrying *tests* hides
real bugs; retrying *infrastructure* hides infrastructure
flake.

# Risks

- **`waitFor` has its own timeout.** If the underlying race
  condition gets *worse* (e.g. the panel takes 30s to load
  because of a real bug), `waitFor`'s default timeout (1s)
  will fail with a timeout error rather than the original
  "wrong state" assertion error. That's still better than the
  pre-PR behavior — the failure is now consistent and
  diagnostic — but reviewers should note that test failures
  may now look like timeouts rather than assertion mismatches.
- **`isLoading`-gated input handling is a runtime change, not
  just a test change.** The skills panel now silently drops
  input during loading. From a UX perspective this is correct
  (clicking a button on a half-loaded panel shouldn't do
  anything), but it's worth a UX check that the loading state
  is *visible* — silently dropping input on an invisible
  loading state is a worse UX than "click does nothing".
- **Vitest fake-timer pattern requires discipline.** The new
  pattern works but is verbose. If a future contributor
  reverts to the older `vi.advanceTimersByTime()` +
  `await Promise.resolve()` pattern in any *new* test, the
  same race re-emerges. A custom ESLint rule banning the old
  pattern, or a wrapper helper that enforces the new pattern,
  would prevent regression.
- **Pinned VS Code version becomes stale.** The CI now runs
  against a pinned VS Code version. If the upstream VS Code
  changes API behavior, cline's tests pass but real users on
  the new VS Code break. A separate, less-frequent CI job
  that runs against the *latest* VS Code (allowed to fail)
  would catch this drift early.
- **Network-cache-retry layering.** Three mitigations stacked
  on the same problem. If the VS Code marketplace is fully
  down, all three fail and the job is red. That's fine — the
  goal is "less flake", not "zero flake". Worth being explicit
  in the PR that this reduces but doesn't eliminate the
  network-dependent failure mode.

# Suggestions

1. Land a custom ESLint rule (or just a doc-comment in
   `vitest.config.ts`) that says "use
   `advanceTimersByTimeAsync` + `runAllTimersAsync`, never
   `advanceTimersByTime` + `await Promise.resolve()`".
2. Add a UX check / screenshot test that confirms the skills
   panel shows a visible loading indicator while
   `isLoading=true`. The runtime change requires the user
   feedback loop to remain intact.
3. Add an "allowed to fail" CI job pinned to *latest* VS Code
   that runs nightly. Catches API drift before users hit it.
4. Add a CI metric ("flake rate per workflow per week") so
   the team can quantify the improvement. Hard to manage what
   you don't measure.

# Verdict

**Ship it.** Three real flakiness sources, three correct
mechanism-specific fixes, no shortcut "rerun three times and
hope". The runtime-change-to-make-tests-easier
(`isLoading`-gated input) is the highest-leverage move
because it removes a *class* of races rather than fixing one
test.

# What I learned

The single most valuable technique in this PR is "make the
runtime robust to the race the test is racing". The skills
panel ignoring input while `isLoading` isn't just a test fix
— it's also the *correct* runtime behavior for the same
reason. Whenever a test is racing with the runtime, ask
whether the runtime should be racing with the user too.
Often the answer is "no, the runtime should be robust" and
the test fix and the runtime fix collapse into the same
patch.

The fake-timer + async pattern is worth memorizing. Mixing
fake timers with promise-based code is a known footgun across
all JS test frameworks. Vitest's
`advanceTimersByTimeAsync` / `runAllTimersAsync` are the
correct primitives — they interleave timer advance with
microtask flushing — but they're not the default and many
older test files still use the pre-async APIs. Standardizing
on the async variants in any new test work is one of those
low-effort, high-payoff team norms.

The third reusable bit is the **pin + cache + retry** triad
for any CI dependency that comes from the network. Each
layer addresses a different failure mode (pin = version
drift, cache = transient network, retry = momentary outage)
and they compose. Skipping any one layer leaves a known
failure class uncovered. Worth applying anywhere a test
suite depends on `npm install`, `apt install`, model
downloads, or marketplace assets.
