# Context

cline supports Claude across **five distinct provider surfaces**:
direct Anthropic API, Bedrock, Vertex, Claude Code subscription,
and discovery-based providers (OpenRouter / Cline router /
Vercel AI Gateway). Every new Claude model has historically
required a synchronized wiring pass across all five — model
metadata, pricing, prompt-caching enablement, reasoning support
flags, and the *1M context variant* for the eligible models.

This PR is the "release prep" pass for Opus 4.7: model registry
entries, dynamic-discovery wiring, UI gating for
reasoning/thinking budget, all the parity work — but
*deliberately* no defaults change, no banner, no recommendation
copy. The release-positioning surface is held back for a
separate PR.

# Problem

Three structural problems get partially resolved:

1. **Provider-specific 1M-context naming variations.** Every
   provider represents the 1M-context Claude variant
   differently:
   - Anthropic: `claude-opus-4.7:1m` (raw `:1m` suffix)
   - Claude Code: `claude-opus-4.7[1m]`, plus aliases like
     `opus[1m]`
   - Bedrock: provider-specific full IDs
   - OpenRouter / Cline router: synthesized from base model,
     not a separate listing
   - Vercel AI Gateway: depends on OpenRouter's synthesized ID
     being normalized *before* the upstream call

   Adding a model means converting between five naming dialects
   in the right places. Get one wrong → the synthetic ID leaks
   to the upstream provider and the request 400s.

2. **Reasoning support is per-model, but the UI has been a
   per-provider switch.** Some providers expose a "thinking
   budget" UI; some don't. The `ReasoningEffortSelector`
   component was gated on provider; with Opus 4.7 introducing
   reasoning across providers, the gating needs to move to a
   per-model check (`shared/utils/reasoning-support.ts` is the
   new home).

3. **Discovery vs static listing.** OpenRouter and Cline router
   discover models at runtime; Anthropic / Bedrock / Vertex /
   Claude Code list them statically. New model support requires
   *both* paths know how to enrich a discovered
   `anthropic/claude-opus-4.7` with the correct pricing,
   caching, reasoning, and 1M variant treatment.

# Design

The PR is large (+662/-135 across 25 files) but the structure
is uniform — each layer of the stack gets the same shape of
addition:

- **`src/shared/api.ts`** (+88/-2): registry entries for
  Opus 4.7 and Opus 4.7 1M, mirroring the Opus 4.6 shape.
- **`src/shared/utils/reasoning-support.ts`** (+29 new): the
  new per-model reasoning-support helper.
- **`src/core/api/providers/{anthropic,bedrock,vertex,litellm,
  requesty}.ts`**: provider-side dispatch updates.
- **`src/core/api/transform/{openrouter-stream,
  vercel-ai-gateway-stream}.ts`** (+79/-40): the synthetic-ID
  normalization layer that strips OpenRouter-style IDs before
  upstream calls.
- **`src/core/controller/models/refresh{Cline,OpenRouter}Models.ts`**:
  discovery enrichment paths.
- **`webview-ui/src/components/settings/`** (5 files): UI
  gating updates for the new model.
- **Tests**: `+93/-0` for Anthropic, focused additions for
  Bedrock, Claude Code, OpenRouter stream transform.

The design choice that matters most: **`shared/utils/reasoning-
support.ts`** as a single source of truth for "does this model
support reasoning". Previously the check was scattered across
provider files; centralizing it means the next reasoning-capable
model needs *one* registry entry, not five provider-side
dispatch updates. That's the right direction.

The intentional **scope cut** is also a design choice: defaults,
banners, onboarding, featured-model UI are explicitly *not*
in this PR. The reasoning is twofold: (a) avoid shipping
product-positioning before the model is externally available;
(b) keep the rollback surface tight if Anthropic's launch IDs
differ from the predicted ones. Both are sound.

# Risks

- **Predicted model IDs.** The PR uses `claude-opus-4-7`,
  `claude-opus-4.7:1m`, etc. as predicted IDs. If Anthropic
  ships dated IDs (`claude-opus-4-7-20260501`) instead, every
  registry entry needs an additional alias. The PR notes this
  in its "Additional Notes" but doesn't add the alias
  infrastructure proactively. A `model_id_aliases: string[]`
  field per registry entry would make the day-of-launch fix a
  one-line change instead of a 25-file follow-up.
- **The OpenRouter / Vercel AI Gateway path is implicit.** The
  comment in the PR description — "Vercel AI Gateway depends
  on the OpenRouter-style synthetic ID being normalized before
  the upstream request is made" — describes a real correctness
  invariant that's enforced *only* by the order of two
  function calls in `vercel-ai-gateway-stream.ts`. A test that
  passes a synthetic 1M ID through the gateway path and
  asserts the upstream payload contains the *normalized*
  (non-synthetic) ID would lock this in. The PR has a small
  openrouter-stream test (+26/-4) but I don't see a Vercel-
  specific normalization test.
- **Bedrock global-endpoint metadata.** The PR notes parity
  was checked vs. Opus 4.6 but flags day-of-launch uncertainty
  about JP / global inference profiles. If AWS exposes these
  late, users on Bedrock will see a "model not available in
  region X" error. Not a regression vs. Opus 4.6 but worth
  a follow-up readiness item.
- **No e2e gating for the reasoning UI.** The webview changes
  add reasoning toggles; there's no test that exercises
  selecting Opus 4.7 → seeing the reasoning UI appear → seeing
  the budget setting persist. Manual validation is mentioned
  but not encoded.
- **Test coverage is asymmetric.** Anthropic gets +93 lines of
  tests; Bedrock gets +15; Claude Code gets +20. Bedrock's
  cross-region inference profiles are arguably the most
  fragile surface, so the test budget feels backwards.

# Suggestions

1. Add `model_id_aliases: string[]` to the registry entry shape
   so day-of-launch ID renames are a one-line patch.
2. Add a Vercel AI Gateway normalization test that asserts the
   synthetic OpenRouter `:1m` ID never reaches the upstream
   request body.
3. Beef up Bedrock test coverage — it's the most provider-
   variant-prone surface and has the thinnest test budget here.
4. Add a webview integration test for "select Opus 4.7 →
   reasoning UI appears → budget persists across reload".
5. Land a follow-up PR that, once the model ships, just
   updates IDs / aliases / defaults, with the entire
   architectural readiness already in place from this PR.
   That PR should be tiny by design; if it isn't, this PR
   missed something.

# Verdict

**Ship it.** Disciplined release-prep PR that does the entire
provider-wiring matrix without contaminating the product-
positioning surface. The scope discipline is the most
valuable part — most "prepare for Model X" PRs sneak in
defaults / banners / promo copy and become impossible to
revert if the launch slips.

# What I learned

The "five-dialect-naming-of-the-same-model" problem is the
canonical failure mode of any abstraction that fronts multiple
LLM providers. Each provider has its own ID conventions, its
own context-window-variant scheme, its own auth-and-pricing
quirks. The temptation is to encode the conversion at every
call site. That breaks down within two model generations.

The right pattern is **one canonical internal ID + N provider
adapters that translate inbound and outbound at the boundary**.
This PR moves in that direction with `reasoning-support.ts` as
a centralized helper, but the model-ID dialect translations
are still distributed across the stream-transform files. A
future PR that consolidates "synthetic ID → upstream ID"
translations into a single `provider-id-normalize.ts` would
close this loop.

The other reusable insight is **release-prep PR scope
discipline**. The temptation is to bundle "support the model"
with "make it the default" because they ship together. Doing
them as one PR means a launch slip → revert burns the entire
support work too. Doing them as two PRs means the support
PR can land weeks early, the defaults PR is tiny, and a
launch slip just delays the tiny PR. Worth establishing as
a team norm: "support" PR + "positioning" PR are always
separate, even when they ship the same day.
