# Context

`cline-core` is the standalone Node.js process variant of
cline (used for CLI / headless runs vs. the VS Code
extension). Long-running cline-core processes — agents that
run for hours or days — have been hitting memory pressure
that's hard to diagnose after the fact: when the process
OOMs, all you have is the exit code. No memory trajectory,
no heap snapshot, no clue whether it was a slow leak or a
sudden allocation spike.

This PR adds three layers of memory observability with a
combined +108/-1 footprint.

# Problem

Three concrete diagnostic gaps:

1. **No memory trajectory.** Operators have no way to see
   "memory grew from 200MB to 7GB over 4 hours" — only "the
   process is gone now". Without a trajectory, leak debugging
   requires reproducing on a developer machine with manual
   profiling. Most users can't do that.
2. **No heap snapshot at OOM.** When V8 runs out of heap, it
   can dump a `.heapsnapshot` file via the
   `--heapsnapshot-near-heap-limit=N` flag, but cline-core
   wasn't passing that flag. The most diagnostic moment of a
   memory bug — the moment of failure — was producing no
   diagnostic data.
3. **No pointer to the snapshot path.** Even when a snapshot
   *is* written, V8 puts it in the current working directory
   with a generic filename. Operators reading the log
   afterward have no idea where to look.

# Design

Three layers stacked, each one cheap on its own:

**Layer 1: V8 heap-snapshot-near-OOM enabled.**

```sh
# scripts/runclinecore.sh:50
node --max-old-space-size=${NODE_MAX_OLD_SPACE_SIZE:-8192} \
     --heapsnapshot-near-heap-limit=1 \
     cline-core.js
```

One V8 flag. When the process is within 0.99x of the heap
limit, V8 writes a `Heap.<pid>.<seq>.heapsnapshot` file to
cwd. The `=1` cap means *one* snapshot per crash — without
the cap, a thrashing GC would write a snapshot every
allocation cycle and fill the disk.

**Layer 2: Periodic in-process memory logging.**

`src/standalone/memory-monitor.ts` (+69 new): a small
`startMemoryMonitoring()` / `stopMemoryMonitoring()` pair
that logs `process.memoryUsage()` on an interval (RSS, heap
total, heap used, external, array buffers). This is the
trajectory data — when an OOM happens 3 hours into a
session, the log shows the curve.

**Layer 3: Snapshot-path discovery on abnormal exit.**

```ts
// src/standalone/cline-core.ts:170
process.on("exit", (code) => {
    if (code !== 0) {
        const snapshots = fs.readdirSync(process.cwd())
            .filter(f => f.endsWith(".heapsnapshot"));
        for (const snap of snapshots) {
            log(`[MEMORY]   ${fullPath} (${sizeMB}MB)`);
        }
    }
});
```

On non-zero exit, the process scans cwd for `.heapsnapshot`
files (left by Layer 1) and logs their paths and sizes. The
operator reading the log now has a literal path to point
DevTools at.

The implementation has the right defensive shape: a
`try`/`catch` that swallows errors silently, because if the
process is too far gone to do `fs.readdirSync` there's
nothing useful to do anyway. The `log` helper still works
because it writes to a stream that was set up at startup.

The startup log line —
```
log(`Heap snapshots (if triggered near OOM) will be written to: ${process.cwd()}`)
```
— is small but matters. Operators reading the log *before*
a crash now know where to look in advance, which means they
can `tail` the directory or pre-configure log shipping for
`*.heapsnapshot` files.

# Risks

- **Periodic log spam.** Without seeing
  `memory-monitor.ts`'s interval, the volume could be
  significant on long sessions. Default interval matters: 10s
  → 8.6k log lines/day, 60s → 1.4k log lines/day. The PR
  body doesn't specify. If the interval is short and the
  output is unconfigurable, this becomes the dominant log
  noise on quiet sessions.
- **Heap snapshots are large.** A snapshot of an 8GB heap
  is ~8GB on disk. If the process OOM-loops (crash → restart
  → crash → restart), the disk fills with snapshots quickly.
  The `=1` flag prevents *multiple per crash*, but does
  nothing across restarts. A cleanup step on startup
  (`rm Heap.*.heapsnapshot older than 24h`) would prevent
  the disk-fill loop.
- **`process.on("exit")` may not fire on hard OOM.** The PR's
  own comment notes "if V8 has no heap left, this handler
  may not run at all". That's the honest framing, but it
  means the snapshot-path logging is best-effort. If V8
  crashes hard, the operator is back to scanning cwd
  manually. A *startup-time* scan ("hey, I see leftover
  `.heapsnapshot` from a previous run, here's the path")
  would catch this case and is cheap to add.
- **`require("fs")` inside the exit handler** is technically
  correct (sync `require` always works) but slightly
  contrarian to the rest of the file's `import` style.
  Switching to a module-scope `import` keeps the style
  consistent with no behavioral cost.
- **`fs.statSync` on every snapshot.** If there are many
  snapshots, this is N sync I/O calls during shutdown. Fine
  for the common case (1-2 snapshots), but worth bounding if
  the directory has dozens.
- **No metrics emission.** The data is in logs only. For
  fleet operators running many cline-core instances, having
  the same data emitted as Prometheus metrics or OpenTelemetry
  would let them dashboard memory trajectories across the
  fleet without log parsing.

# Suggestions

1. Make the monitoring interval configurable via env var
   (`CLINE_CORE_MEMORY_LOG_INTERVAL_MS`, default 60s).
2. Add a startup-time scan for leftover `.heapsnapshot`
   files and log them — catches the hard-OOM case where the
   exit handler didn't fire.
3. Add a startup-time cleanup of `.heapsnapshot` files older
   than N days. Prevents the OOM-loop disk-fill.
4. Switch `require("fs")` to module-scope `import * as fs
   from "fs"` for style consistency.
5. Future PR: emit memory data as OpenTelemetry metrics in
   addition to logs.

# Verdict

**Ship it.** Three small layers that compose into a real
diagnostic surface for the most user-painful failure mode of
long-running cline-core. The defensive coding (try/catch on
the exit handler, the `=1` cap on snapshots) is appropriate
for code that runs in extreme-resource situations. The
follow-ups are all small and non-blocking.

# What I learned

The pattern "enable diagnostic data at the moment of failure
+ log the path to it on the way out" is the right default
for any long-running process. The temptation is to either
(a) skip diagnostic capture because "it's noisy", or
(b) capture diagnostics constantly and try to find the
needle later. The middle path — capture *only at the moment
of failure*, but capture *everything* at that moment, and
log the breadcrumb — is the right tradeoff. V8's
`--heapsnapshot-near-heap-limit` is the perfect example of
the OS giving you exactly that primitive.

The reusable insight is **pre-failure logging of post-failure
artifact paths**. The startup line "snapshots will be written
to X if triggered" is high-leverage because it's read by
operators *before* anything has gone wrong, while they're
configuring log shipping or tail commands or whatever. Once
the failure happens, it's too late to add the snapshot path
to the log shipper. The same pattern applies to core dumps,
crash reports, debug builds — log the *expected location of
the artifact* at startup, not just at failure.

The third lesson is the layered-defense shape: V8 flag (Layer
1) writes the file; periodic monitor (Layer 2) gives
trajectory; exit handler (Layer 3) surfaces the path. Each
layer is independent and graceful — Layer 1 works without
Layer 3, Layer 3 works without Layer 1, but together they
cover every diagnostic question an operator will have. This
"each layer answers a different question, all layers compose"
shape is a good template for any observability work.
