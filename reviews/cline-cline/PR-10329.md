# Context

cline supports both its own first-party LLM provider ("Cline
provider", which goes through Cline's own gateway) and direct
third-party providers (Anthropic, OpenAI, etc.) using the
user's own API keys. When an organization wants to enforce
budget caps, the Cline provider can do this server-side — it
sees every request and can refuse them past a daily or monthly
threshold. But for *third-party* providers, the org's spend is
opaque from Cline's side: requests go directly from the
extension to the provider, billed to the user's API key.

This PR adds a client-side enforcement layer. Periodically (on a
TTL'd cache) the extension fetches the org's overbudget status
from Cline's account API; before dispatching any non-Cline
provider request, it consults the cached status and blocks the
request if the org is over budget. The block is shaped as a 429
`ClineError` so the existing `SpendLimitError` UI can render it
without webview changes.

+1000/-11 across 17 files: new service, Task integration, new
RPC method, proto changes, error type, error UI updates, and
six test files. I focused on the service, the Task integration,
and the error/UI shape.

# Diff walkthrough

**The service shape.** `ThirdPartySpendLimitService` is a
classic singleton TTL cache:

```ts
async fetchIfNeeded(): Promise<void> {
    const activeOrgId = this.getActiveOrgId()
    if (!activeOrgId) {
        this.cachedStatus = null; this.cachedOrgId = null; this.cachedAt = 0
        return
    }
    const cacheHit = this.cachedOrgId === activeOrgId && !this.isStale()
    if (cacheHit) return

    if (this.fetchPromise) {
        await this.fetchPromise
        if (this.cachedOrgId === activeOrgId && !this.isStale()) return
    }
    this.fetchPromise = this.doFetch(activeOrgId).finally(() => {
        this.fetchPromise = null
    })
    return this.fetchPromise
}
```

Three things this gets right: (a) the cache key is the *active
org ID*, so switching orgs immediately invalidates rather than
serving stale cross-org data; (b) the in-flight `fetchPromise`
deduplicates concurrent calls (a real issue in cline where Task
can fire many tool-call dispatches in parallel); (c) the
post-await re-check guards against the case where the previous
fetch resolved with a different org context (e.g. the user
switched orgs while a fetch was pending).

**The fail-open `doFetch`** is the most important security
choice in the diff:

```ts
private async doFetch(organizationId: string): Promise<void> {
    try {
        const status = await ClineAccountService.getInstance()
            .fetchOverbudgetStatusRPC(organizationId)
        this.cachedStatus = status ?? null
        this.cachedOrgId = organizationId
        this.cachedAt = this.now()
    } catch (err) {
        Logger.error("Unexpected error fetching overbudget status:", err)
        this.cachedStatus = null  // ← fail-OPEN
        this.cachedOrgId = organizationId
        this.cachedAt = this.now()
    }
}
```

When the status fetch fails, the cache becomes `null`, which
means `isOverbudget()` returns `false`, which means requests
proceed. This is **fail-open**: if Cline's account API is down,
users can still spend on third-party providers. That's the
right call for a feature where the alternative is "an unrelated
Cline outage blocks all non-Cline traffic", but it's worth
calling out explicitly. A fail-closed posture would be more
secure but would couple every third-party request to Cline
backend availability — a bad trade for a non-security-critical
budget cap.

**The TTL configuration.** The default is 5 minutes, with an
env override:

```ts
function resolveTtlFromEnv(): number | undefined {
    const raw = process.env.CLINE_SPEND_LIMIT_TTL_MS?.trim()
    if (!raw) return undefined
    const shorthand = raw.match(/^(\d+)\s*(ms|s|m)?$/i)
    if (!shorthand) return undefined
    const value = Number(shorthand[1])
    if (!Number.isFinite(value) || value < 0) return undefined
    const unit = (shorthand[2] ?? "ms").toLowerCase()
    switch (unit) {
        case "s": return value * 1000
        case "m": return value * 60 * 1000
        default: return value
    }
}
```

The shorthand parsing (`"5m"`, `"30s"`, `"60000"`) is small UX
polish that pays for itself — env vars with raw millisecond
counts are universally hated. Worth noting that the regex
deliberately doesn't accept fractional values or hours; that's
an appropriate limit for a cache TTL.

**The Task integration** is the meaningful part:

```ts
private async checkThirdPartySpendLimit(providerId: string | undefined): Promise<void> {
    if (providerId === "cline") return  // server-side enforcement

    const svc = ThirdPartySpendLimitService.getInstance()
    await svc.fetchIfNeeded()

    const status = svc.getStatus()
    if (!status?.overbudget) return

    const hitDaily = status.limits.dailyLimitUsd != null
    const budgetPeriod: "daily" | "monthly" = hitDaily ? "daily" : "monthly"
    // ...
    throw ClineError.transform({
        status: 429,
        code: "SPEND_LIMIT_EXCEEDED",
        message,
        details: { code, budget_period, limit_usd, spent_usd, resets_at, message },
    })
}
```

The "throw a synthetic 429 to reuse the existing UI" pattern is
the right shape — the `SpendLimitError` component already knows
how to render 429s with budget metadata, and reusing it means
no webview changes for the third-party path. The "daily takes
precedence over monthly" rule is correctly explained in the
comment ("it will reset sooner") and is the right user-visible
prioritization (showing the *next* reset time, not the latest).

The shape `status.limits.dailyLimitUsd != null` to detect "daily
is configured" is a typed null check, not just truthiness —
which is right, because `0` is a meaningful daily limit value
(an org could set it to 0 to fully disable daily and rely on
monthly only). Using `!= null` instead of `!status.limits.dailyLimitUsd`
preserves that distinction.

# Risks

1. **TTL of 5 minutes can leak significant spend.** If an org
   hits its limit at minute 0:01 and the cache was refreshed at
   minute 0:00, the user has 4:59 of unrestricted third-party
   spend until the next refresh. For a daily $50 limit and a
   user running parallel agents, that could easily be $20+ of
   overspend per stale-cache window. The TTL is a knob; the
   default may be too generous. A status-changing event (org
   switch, manual refresh button) doesn't bust the cache today.
2. **Fail-open on Cline backend outage.** This is the right
   default but should be documented and ideally exposed as a
   user/org policy: a strict-mode org might prefer "block on
   uncertainty". Currently there's no escape hatch.
3. **No webhook / push channel from the backend.** When the
   org genuinely hits the limit server-side, the extension
   doesn't learn about it for up to TTL ms. A small "the next
   API response with a 429-shaped body invalidates the cache"
   shortcut would tighten the loop without a push channel.
4. **`fetchPromise` deduplication doesn't bound the wait.** A
   slow `fetchOverbudgetStatusRPC` (cold backend, network
   stall) blocks every concurrent task that calls
   `fetchIfNeeded`. A timeout on `doFetch` (say 5s) with the
   timeout treated as fail-open would cap the worst case.
5. **The synthetic 429 `ClineError.transform` doesn't carry a
   `Retry-After` header** in the way a real provider 429 would.
   The auto-retry logic elsewhere in Task may behave differently
   for this error than for a "real" rate limit. The diff shows
   `isSpendLimitError` is checked; worth confirming the retry
   loop short-circuits cleanly.
6. **The status cache is keyed only by org ID.** If a user is
   in multiple orgs and switches via a path that doesn't go
   through `AuthService.getActiveOrganizationId()`, the cache
   could serve the wrong org's status. The double-check after
   the in-flight promise resolves catches the race within a
   single fetch cycle but not the case where org-switch
   bypasses the auth service.
7. **`process.env.CLINE_SPEND_LIMIT_TTL_MS` is read at
   construction time only.** Changing the env var at runtime
   has no effect (process env is immutable from inside Node
   anyway, but a future "reload config" path would need to call
   `setTtlMs`). Worth a comment.
8. **The "daily takes precedence over monthly" rule** is right
   for the user message, but the actual block decision is
   `status.overbudget === true` from the backend — meaning the
   *backend* decides which limit triggered. If the backend
   returns `overbudget: true` with both `dailyLimitUsd` and
   `monthlyLimitUsd` set but the *daily* hasn't actually been
   hit, the client message will incorrectly attribute the block
   to the daily limit. Worth a backend invariant: when
   `overbudget: true`, populate only the limit that triggered.
9. **No telemetry for "blocked due to spend limit".** This
   would be the kind of metric an org admin wants — how often
   their developers hit the cap. The PR adds the enforcement
   but not the visibility.
10. **`Logger.error` on fetch failure runs on every call after
    the first failure.** If the backend is down for an hour
    and the user has a busy session, the log will fill up. A
    backoff or "log once per N minutes" would be kinder.

# Suggested questions for the author

1. The 5-minute TTL allows up to 5 minutes of overspend after
   the limit is hit server-side. Has the team modeled the worst-
   case spend per stale window for typical org limits?
2. Could a 429 *response* from a third-party provider also bust
   the cache, on the assumption that "if the user just got rate-
   limited, they're probably also at their internal limit"?
3. Is there a strict-mode policy for orgs that want fail-closed
   on Cline backend outage rather than fail-open?
4. What does the auto-retry loop do for `SpendLimitError`?
   Should it never retry (the cache won't refresh until TTL),
   or retry-after-cache-bust?
5. Is the backend invariant "when `overbudget: true`, only the
   triggering limit is populated" enforced? The client's
   "daily takes precedence" rule depends on it.

# Verdict

**Approve.** The core architecture is sound: TTL'd cache with
in-flight dedup, fail-open on backend outage, synthetic 429 to
reuse the existing UI, separate from Cline-provider's server-
side enforcement. The most important follow-ups are operational
rather than structural: tighten the TTL default (or expose it
prominently), document the fail-open posture, and consider a
push-channel or response-driven cache bust to reduce the stale-
window spend leak. None of these block the value the PR
delivers — orgs that today have *no* third-party limit
enforcement get a working approximation now, and a tighter one
later.

# What I learned

Three takeaways, all about client-side enforcement of server-
side policy.

**First**, the *fail-open vs fail-closed* choice for budget
enforcement is fundamentally different from the same choice for
security enforcement. For security (auth, capability gates,
network egress), fail-closed is almost always right — a partial
outage shouldn't become an authorization bypass. For budget
caps, fail-closed turns "Cline backend hiccup" into "all
third-party traffic blocked", which is a much worse UX than
"5 extra minutes of spend". The PR makes the right call. The
general principle: fail-open is correct when the consequence of
a wrong decision is *bounded* (5 more minutes of spend) and
fail-closed is correct when it's *unbounded* (data exfiltration,
permission escalation). Always classify the consequence before
choosing the posture.

**Second**, the *synthetic-error pattern for UI reuse* is one
of the cheapest ways to get a polished error experience for a
new failure mode. The PR's `checkThirdPartySpendLimit` throws a
`ClineError` shaped like a real backend 429 with the same
`code: "SPEND_LIMIT_EXCEEDED"` and a populated `details` object.
The existing `SpendLimitError` React component, which was
written for the Cline-provider 429 path, renders it identically
— no webview changes, no per-feature UI work. The cost is the
small obligation to keep the synthetic error's shape in sync
with the real one (a contract test would lock this down). The
benefit is that every UI improvement to the spend-limit error
display automatically applies to both backends. This pattern
generalizes: when a client-side check needs to surface as if it
were a server response, *throw the response*.

**Third**, the *in-flight promise dedup* pattern in
`fetchIfNeeded` is an underused TypeScript/JavaScript idiom for
"cache + coalesce concurrent misses":

```ts
if (this.fetchPromise) {
    await this.fetchPromise
    // re-check after wait — context may have changed
}
this.fetchPromise = this.doFetch(...).finally(() => {
    this.fetchPromise = null
})
return this.fetchPromise
```

The naïve cache reads "if hit, return; else fetch and store" —
which means N concurrent misses fire N concurrent fetches.
Storing the *promise* (not just the result) and awaiting it
collapses N into 1. The crucial detail is the post-await
re-check: by the time the awaited fetch returns, the context
that triggered the fetch (org ID, user identity, target URL)
may have changed, and a stale fetch result must be discarded.
The PR gets this right. Most cache implementations don't, and
the resulting bugs are subtle ("sometimes the cache serves the
previous user's data after a fast org switch"). When you write
a TTL cache for an async fetcher, always add the in-flight
slot, and always re-check the context after awaiting it.
