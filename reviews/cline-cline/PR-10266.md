# Context

Cline shows token usage per turn. For Anthropic-style prompt
caching, that usage breaks down into four counts:
non-cached input tokens (`inputTokens`), cache *read* tokens
(reused from a prior cache), cache *write* tokens (newly
seeded into the cache this turn), and output tokens. The
cache-write count matters because it's billed differently
(typically more expensive than reads) and because it directly
explains why a multi-turn conversation's first turn looks
expensive and subsequent turns look cheap.

Different providers expose cache writes under different field
names. Anthropic and OpenRouter use
`cache_creation_input_tokens`; some surface it under
`prompt_tokens_details.cache_write_tokens`. Provider handlers
in Cline normalize all of these into the unified usage shape.

# Problem

The `openrouter` handler already read
`prompt_tokens_details.cache_write_tokens` and reflected it as
`cacheWriteTokens`. The `cline` and `vercel-ai-gateway` handlers
hardcoded `cacheWriteTokens: 0` even when the provider's usage
chunk exposed Anthropic-style cache writes via
`cache_creation_input_tokens`. Two consequences:

1. **Cache-write tokens were invisible** in the UI for these
   providers. Users running multi-turn Claude conversations
   through `cline` or `vercel-ai-gateway` saw "0 cache writes"
   even when caching was active and being billed.
2. **`inputTokens` was inflated.** The provider reports total
   input including cache writes; subtracting cache writes is
   how you derive the "fresh" input tokens. Without that
   subtraction, `inputTokens` double-counted the cached
   portion.

Either symptom alone is annoying; together they make
cache-effectiveness analysis impossible — users can't tell
whether their prompts are caching well because the numbers
don't add up.

# Design

A targeted normalization in two handler files:

```ts
const cacheWriteTokens =
  usage.cache_creation_input_tokens ??
  usage.prompt_tokens_details?.cache_write_tokens ??
  0;
const inputTokens = (usage.prompt_tokens ?? 0) - cacheWriteTokens;
```

Read the field if present, fall through to the secondary shape,
default to 0. Subtract cache writes from the reflected
non-cached `inputTokens`. That's the entire production change:
+20/-5 in `cline.ts`, +13/-3 in `vercel-ai-gateway.ts`. The
remaining +90 LOC is regression tests across both providers
covering the new branch.

The validation is the load-bearing part of the PR description:
a 3-turn live matrix across `{cline, openrouter,
vercel-ai-gateway}` × `{gpt-5.4, gemini-3-flash-preview,
claude-sonnet-4.6}` — 27 runs total — with reflected usage
matching raw provider fields in every run after the patch.
That's the right shape of validation for a normalization fix:
provider behavior is the source of truth, and you have to
exercise the matrix to be sure none of the providers expose
the field under yet a third name you missed.

# Risks

- **Field-shape assumption.** The fix assumes
  `cache_creation_input_tokens` is always Anthropic-style
  (count of newly-cached *input* tokens). If a future provider
  reuses the name to mean something else (e.g., count of cache
  *entries* created), the subtraction would be wrong. A
  per-provider switch on `model.includes('claude')` would be
  more defensive but adds branchiness; the current
  field-presence check is cleaner and the matrix run validates
  the assumption holds across the three providers tested.
- **Negative `inputTokens` if provider reports cache writes >
  prompt tokens.** Shouldn't happen, but a `Math.max(0, ...)`
  would prevent a UI rendering bug if it ever does.
- **Two near-identical patches in two files.** The fix is the
  same logic duplicated. A shared
  `normalizeCacheUsage(usage)` utility would prevent the next
  provider from being added with the bug re-introduced. Worth
  a follow-up refactor.
- **Provider matrix is live, not recorded.** The 27-run
  validation gives high confidence today but isn't reproducible
  in CI. A recorded-fixture test (capture one real usage chunk
  per provider/model and replay it in unit tests) would catch
  provider field-name drift over time.

# Suggestions

1. Extract a shared `extractCacheWriteTokens(usage)` helper in
   a provider-utils module and have all three handlers
   (`openrouter`, `cline`, `vercel-ai-gateway`) call it. Then
   the next provider added gets it for free.
2. Record real usage chunks from each provider as fixtures and
   assert the normalization output against them. Live matrix
   for confidence; recorded fixtures for permanence.
3. Add a sanity check in the unified usage-shape constructor:
   `if (cacheWriteTokens > inputTokens + cacheWriteTokens)
   warn`. Catches the "field means something else" failure
   mode before it reaches the UI.

# Verdict

**Ship it.** The fix matches the bug, the test coverage hits
the new branches in both providers, and the live matrix is
strong evidence the normalization is correct across the
relevant model space.

# What I learned

When normalizing semantically-equivalent fields across
multiple upstream providers, the failure mode is not "one
provider gets it wrong" — it's "we ported the normalization
to N-1 of N providers and forgot one." The fix is structural:
extract the normalization into a shared helper *the moment*
you have two providers doing it, even if it feels premature.
The cost of the helper is one indirection; the cost of
forgetting one provider is silent UI miscalibration that
users may attribute to the model rather than to your handler.
