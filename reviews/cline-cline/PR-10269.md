# Context

Cline runs shell commands inside the user's editor and streams
output back to the model. The flow uses a request/response
pattern called `ask`: when the agent wants to interrupt the
model loop and surface output (e.g., to ask "should I continue
running this long command?"), it issues
`Task.ask("command_output", ...)` and awaits a response. The
terminal side is `CommandOrchestrator`, which manages the
process lifecycle through `started`, `running`, `completed`,
`error`, and timeout transitions.

# Problem

`Task.ask("command_output", ...)` could remain pending after the
underlying terminal process had already completed, errored, or
timed out. The producer (terminal) was done; the consumer (the
awaited `ask`) was still parked. With no resolver, the agent
loop deadlocked — the user's UI showed a permanent "waiting for
your input" prompt for a command that had already exited
seconds ago. The only recovery was a full session restart.

This is the canonical "the channel between two state machines
got out of sync." The terminal's lifecycle transitioned to a
terminal state without telling the ask layer, because there was
no channel from terminal lifecycle → ask resolution; the ask
layer assumed the user (or the next output chunk) would settle
it.

# Design

Two pieces, narrowly scoped:

**(1) Explicit release path in `CommandOrchestrator`.** On the
`completed`, `error`, and timeout transitions, the orchestrator
now invokes a release callback that resolves any pending
`command_output` ask tied to this command. The callback is wired
via `types.ts` (+2 LOC adding the callback shape) and consumed
in `CommandOrchestrator.ts` (+54/-1).

**(2) Targeted callback from `Task`.** `Task` (+3 LOC) provides
the orchestrator with a function that resolves the underlying
pending-ask state for `command_output` specifically. The PR
intentionally does *not* touch `Task.ask()` semantics globally
— other ask types (e.g., `tool_approval`, `command_approval`)
keep their existing lifecycle. The scope is exactly "terminal
orchestration releases its own asks," nothing wider.

The test budget tells the story: 244 LOC of regression coverage
for 57 LOC of production change. The tests exercise both the
completion and error paths and assert the pending ask is
settled in each.

# Risks

- **Timeout transition is the trickiest.** "Completed" and
  "error" are unambiguous terminal states. "Timeout" is more of
  a *policy* applied by the orchestrator on top of a still-live
  process. If the timeout fires *and* the process then writes
  output before being killed, the ask must already be resolved
  (won't double-resolve) and the late output must not
  re-trigger an ask. The test file should explicitly cover that
  race; from the file list it's not obvious it does.
- **Scope discipline cuts both ways.** Keeping `Task.ask()`
  semantics global is right for blast radius, but other ask
  types that share the same pending-channel pattern likely
  have the same latent bug. A follow-up audit of every
  `ask(...)` site for "what producer eventually settles this?"
  would prevent the next instance.
- **Callback is fire-and-forget.** If the callback throws (e.g.,
  the task was already disposed), the orchestrator's
  lifecycle transition probably shouldn't fail. Wrapping the
  callback in a try/catch with a debug log is cheap insurance.
- **No integration test against the actual VS Code terminal.**
  Unit tests stub the terminal; the real `VscodeTerminalProcess`
  may have its own quirks around `completed` event timing
  (e.g., shell integration vs. fallback paths). The original
  bug was reported against real VS Code use, so a smoke that
  spawns and kills `sleep 2` would close the loop.

# Suggestions

1. Add a "kill switch" timeout on every `ask()` itself, not
   just the terminal — say, 30 minutes. If no producer ever
   resolves it, the ask should self-cancel with a logged
   reason rather than block the agent indefinitely. This is a
   defense-in-depth pattern for the bug class.
2. Audit every `ask(...)` call site against this checklist:
   "what producer settles this? what producer-state means it
   will *never* settle?" Document the answer in a comment at
   the call site.
3. Extend the `command_output` resolver pattern to a generic
   `releasePendingAsk(askType: string)` on `Task`, then
   migrate other producer state machines onto it. Beats
   re-implementing the terminal-specific callback for every
   producer.

# Verdict

**Ship it.** Narrow, well-tested, and surgical. The
asymmetric LOC (+244 test vs. +57 production) is exactly right
for a deadlock fix — the tests are the permanence of the
contract.

# What I learned

A pending `Promise`/`ask` with no producer-side timeout is a
deadlock waiting to happen the first time the producer
transitions to a terminal state without notifying its
consumer. The two reliable defenses are: (a) every consumer
`await` should have an upper-bound timeout that logs and
cancels, even if it should never fire in practice; (b) every
producer state machine should enumerate every consumer
waiting on it and resolve them all on every terminal
transition. This PR adds (b) for the specific instance; the
absence of (a) is what made the bug visible to users instead
of failing fast in CI.
