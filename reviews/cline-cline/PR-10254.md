# Context

cline exposes MCP servers' tools to the model as native function-
call names. The wire format is `{key}0mcp0{tool_name}` where
`{key}` is a 6-character identifier for the MCP server, kept
short to stay under the 64-character tool-name limit imposed by
LLM providers (and to avoid Gemini's "function name must start
with a letter" rule). Previously the key was generated as
`"c" + nanoid(5)` — a *random* 5-character suffix.

The bug (#8087, referenced in the new docstring) is that the
random key is regenerated on every extension restart or MCP
reconnect. The model, mid-conversation, has cached the previous
session's tool names — `cAB12X0mcp0read_file` — and emits tool
calls against them. After the restart, the *same MCP server* is
now keyed as `cP9KQR0mcp0read_file`, so the routing layer can't
find a server matching `cAB12X` and the tool call fails with no
useful diagnostic.

The fix: replace `nanoid(5)` with a deterministic hash of the
server name. Same server name → same key, forever.

+116/-7 across 3 files. Two new files (`server-key.ts` with the
hash function, and a test file), one edit to `McpHub.ts`.

# Diff walkthrough

**The hash function.** `server-key.ts` introduces a tiny djb2
implementation:

```ts
export function hashServerName(name: string): string {
    let hash = 5381
    for (let i = 0; i < name.length; i++) {
        hash = (hash * 33) ^ name.charCodeAt(i)
    }
    return (hash >>> 0).toString(36).padStart(5, "0").slice(0, 5)
}

export function buildServerKey(serverName: string): string {
    return `c${hashServerName(serverName)}`
}
```

This is the canonical djb2 hash (`hash * 33 ^ char`), which is
not cryptographic but has good distribution for short strings.
The output is base-36 (0-9a-z), 5 characters, padded if the
hash happens to be small. The `c` prefix preserves the Gemini
function-name compatibility constraint.

The output space is `36^5 = 60,466,176` distinct keys. Birthday
collision probability is meaningful around √60M ≈ 7700 servers,
which is far above any realistic deployment. For "< 100 MCP
servers" (the docstring's stated target), collision probability
is well under one in a million.

**The integration in `McpHub.ts`** is a four-line swap:

```ts
// Before:
const uid = "c" + nanoid(5)

// After:
const uid = buildServerKey(server)
const existing = McpHub.mcpServerKeys.get(uid)
if (existing && existing !== server) {
    Logger.warn(
        `MCP server key collision: "${server}" and "${existing}" both hash to "${uid}". ` +
        `"${existing}" will be overwritten. Consider renaming one of the servers.`,
    )
}
McpHub.mcpServerKeys.set(uid, server)
return uid
```

The collision warning is the right defense: hash collisions are
unlikely but possible, and silently overwriting one server's
mapping with another's would produce maddening "tool calls go
to the wrong server" bugs. The warning is *actionable* — it
names both colliding servers and tells the user what to do
("rename one") rather than just complaining about the collision.

The test file is thorough: it covers determinism (called twice,
same output), distinctness (different inputs, different
outputs), edge cases (empty string, special characters, very
long names), and the format invariant (matches `/^c[a-z0-9]{5}$/`).
That's the right shape for a tiny pure function — exhaustive
property tests rather than examples.

# Risks

1. **The collision overwrite is `silent at the data layer` and
   loud only in logs.** If two servers hash to the same key,
   `McpHub.mcpServerKeys.set(uid, server)` blindly overwrites
   the previous mapping. A user who doesn't read warnings will
   experience tool-call routing failures with no in-product
   indication. A stronger shape: refuse the second registration
   and surface it as an MCP server status error in the UI.
2. **Renaming an MCP server changes its key.** This is the
   intentional shape (the key is derived from the *name*) but
   the user-visible consequence isn't documented: a model with
   cached tool names from before a server rename will hit the
   same "key not found" issue the PR is trying to fix. Worth a
   note in the docs that server names are now part of the
   stable identity.
3. **djb2 is not collision-resistant against adversarial
   inputs.** A malicious MCP server config could craft a name
   that collides with a known good one. In practice MCP server
   configs are user-controlled, so the threat model is "user
   shoots own foot", but if MCP server discovery is ever
   automated (registry-based), the collision could be exploited.
   FNV-1a or xxhash would be the same length with stronger
   distribution and still no crypto guarantees, but the
   marginal benefit doesn't justify the rewrite here.
4. **No migration path for in-flight conversations.** A user
   updating cline mid-conversation will see *all* their MCP
   tool keys change exactly once (from random to deterministic).
   Their model will fail every cached tool call until it
   re-discovers the new keys. A note in the release changelog
   would be kind; a more complete fix would persist the
   previous random key and use it as a fallback for one
   release.
5. **The base-36 padding masks the loss of entropy for very
   small hashes.** A server name that happens to djb2 to a
   small integer will pad with leading zeros, which is
   structurally fine but means the realistic effective entropy
   is slightly less than `36^5`. Not a real concern at < 100
   servers.
6. **The `mcpServerKeys` map is `static`.** That means *all*
   `McpHub` instances share one global key registry. In a
   multi-window VSCode session this is correct (the user wants
   the same server in window A and window B to share a key),
   but it's a subtle invariant that's worth a comment on the
   field declaration.
7. **Test file uses `should`** (the assertion library) which is
   valid but inconsistent with the rest of the codebase if
   other test files use `expect` or `assert`. Worth a
   quick consistency check.
8. **No test for the collision warning path.** The
   `Logger.warn` branch is only reached when two distinct
   server names hash to the same key. A test that constructs
   such a pair (e.g. via finding two short colliding strings
   experimentally and pinning them in a test) would protect
   against silent regression of the warning behavior.

# Suggested questions for the author

1. Has the team considered making collision a *hard error*
   (refuse the second registration, surface in MCP status UI)
   rather than a warning + silent overwrite?
2. Is there a migration story for users updating mid-
   conversation, or is the assumption "the next tool call will
   fail once and the model will recover"?
3. Would a stronger hash (FNV-1a, xxhash) be worth the cost?
   The collision probability is fine for <100 servers but
   shifts adversarial-input considerations.
4. Should there be a test that pins two known-colliding inputs
   to lock down the warning code path?

# Verdict

**Approve.** Tiny, focused, well-tested fix for a real bug
(#8087). The deterministic-hash approach is the obvious right
answer; the implementation is clean (djb2 in eight lines,
base-36 output for identifier safety, `c` prefix for Gemini
compatibility). The collision warning is appropriately
defensive without being overengineered. The test file covers
the function's properties exhaustively. This is the shape every
small bug-fix PR should aspire to: one hypothesis, one fix, one
test file that exhausts the function's behavior, and one
diagnostic for the foreseeable failure mode.

# What I learned

Two takeaways, both about identifier stability across sessions.

**First**, the *cached-by-the-model* invariant is one of the
most subtle classes of LLM-integration bugs. The model's
context contains tool names from the previous turn; if those
names change between turns, the model emits dead references
and the tool-call layer has no way to know whether the model
"meant" the new name or genuinely got the wrong one. The fix
isn't "make the model adapt" (you can't change the model) — it's
"make every identifier the model sees stable across the
session and ideally across sessions". This applies to MCP
server keys, tool names, file IDs in agent context, conversation
IDs in resumption flows, and anything else the model might
quote back. Random IDs are tempting because they're easy; they
become bugs the moment the model starts caching them.

The general rule: **if the model can quote it back, it must be
stable**. Generate from a deterministic function of stable
inputs (the server name, the file path, the tool's natural
identifier), and add a `c` or `c_` prefix to satisfy provider
identifier rules. Never use `nanoid` for anything the model
will see.

**Second**, the collision-warning shape: when defensive code
detects an unexpected condition that *is* recoverable but
indicates configuration drift, the warning should (a) name the
two conflicting parties explicitly, (b) tell the user what
action to take, and (c) describe the consequence of inaction.
The PR's warning hits all three:

> MCP server key collision: "foo" and "bar" both hash to "cX9Y2".
> "bar" will be overwritten. Consider renaming one of the
> servers.

Compare to the typical defensive log line ("warning: server
key collision detected") which tells the user *that* a problem
exists but not *which* problem, *what* will happen, or *how*
to fix it. Naming the parties matters because grep-by-name is
how users investigate; describing the consequence matters
because it tells the user how urgent the fix is; suggesting an
action matters because it short-circuits the support ticket.
This is the shape every defensive `console.warn` should take.

The third meta-lesson: when swapping out a random identifier for
a deterministic one, the migration moment is one-time and
mostly harmless if the cache layer is the model itself
(conversations get a transient hiccup, then recover). It would
be vastly more harmful if the cache layer were *the user's
config file* (saved tool aliases, recorded macros). The PR
gets away without an explicit migration because the only
consumer of the random IDs was ephemeral. Worth tracking what
*else* in the system might consume server keys before the
deterministic version becomes load-bearing — once a key shape
is persisted to disk somewhere, changing it again becomes
expensive.
