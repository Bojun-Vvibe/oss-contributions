# Context

cline ships a small set of built-in slash commands (`/newtask`,
`/smol`, `/compact`, `/newrule`, `/reportbug`, `/deep-planning`,
`/explain-changes`). Each is a hardcoded prompt template
injected into the conversation when the user types it.
`/deep-planning` was the most elaborate of these — a multi-step
workflow that put cline in "architect mode" with model-family-
specific prompt variants (anthropic, gemini, gemini3, gpt51,
generic) and ~1500 lines of templated prompt text living in
`src/core/prompts/commands/deep-planning/`.

This PR removes it. The stated rationale (visible from the doc
rewrite) is that `/deep-planning` overlapped substantially with
*staying in Plan mode for longer* — cline's existing two-mode
(Plan/Act) workflow already supports the same outcome, and
having a dedicated slash command was confusing users about when
to use which. The replacement guidance is "stay in Plan mode and
ask cline to write a markdown plan; if your team does this
often, turn it into a reusable workflow".

+56/-1773 across 23 files. Almost all the deletion is the
prompt module + variants + the dedicated docs page; the code-
plumbing change is tiny.

The single-line nature of the runtime change is the right shape:

**The doc rewrite carries most of the work.** Three files of

-## Using `/deep-planning`
+## Planning Large Tasks

-For complex tasks that need thorough analysis, use the `/deep-planning` slash command.
+For complex tasks that need thorough analysis, stay in Plan mode long enough to:
1. Explore the codebase systematically
2. Identify affected files, dependencies, and edge cases
3. Discuss tradeoffs and clarify requirements

That's the substantive replacement: the *behavior* the slash

# Risks

1. **Hard removal with no deprecation period.** A user who
   types `/deep-planning` after this update will see "command
   not recognized" with no hint that the feature has been
   removed or what to do instead. A one-release deprecation —
   command still parses, prints a friendly migration message,
   no longer triggers the prompt — would have been kinder. The
   removal is small enough that adding a transitional shim
   would have been ~10 lines.
2. **Existing custom workflows that reference `/deep-planning`
   will break silently.** If a user wrote a workflow file
   that includes `/deep-planning` as a step (the docs
   explicitly suggested this in older revisions), that
   workflow now silently does nothing for the deep-planning
   step. The PR's release notes should call this out
   prominently, and ideally the slash-command parser should
   emit a one-time warning when a recognized-but-removed name
   is used.
3. **Loss of model-family-specific prompt variants.** The
   deleted directory had separate prompts for anthropic,
   gemini, gemini3, gpt51, and generic. That's real prompt-
   engineering work, refined over time. If the replacement
   guidance is "stay in Plan mode longer", then *Plan mode's
   system prompt* probably needs to absorb some of the
   model-specific tuning that was in the deep-planning
   variants. Without that, models that performed well in
   `/deep-planning` may regress in Plan mode.
4. **The docs say "ask Cline to create a markdown planning
   document"** but there's no slash command or canonical
   prompt for this. Users will phrase the request differently
   each time and get inconsistent results. A new `/plan`
   workflow shipped *together with* the removal would give
   users a clear migration target.
5. **The "turn it into a reusable workflow" pointer assumes
   familiarity with the workflows feature.** A user who's been
   using `/deep-planning` happily for a year may not know that
   workflows exist. The doc rewrite could include a one-line
   example workflow that replicates the deep-planning shape, as
   a starter.
6. **Test coverage of the removal.** The test files
   (`src/core/slash-commands/__tests__/index.test.ts`,
   `src/test/slash-commands.test.ts`) lose entries for
   deep-planning, but it's worth checking that no test was
   asserting "all built-in commands are present" — such an
   assertion would now silently match the smaller list.
7. **The deleted prompt content has historical/research
   value.** Deleting 1500 lines of carefully-tuned prompt
   variants without preserving them in a separate examples
   repo or docs annex loses the prompt-engineering work. A
   companion PR that moves the variants to a `prompt-examples/`
   directory (or to the docs as "advanced custom workflow
   templates") would preserve the IP without keeping it as
   shipped code.
8. **CLI-compatibility flag.** The deleted entry had
   `cliCompatible: true`. CLI users may have shell aliases or
   scripts that invoke `/deep-planning`; those break with no
   warning. The CLI probably has a different surface area to
   consider than the GUI for migration messaging.
9. **The PR title and commit message tell users what changed,
   but the *why* is buried in the doc rewrite.** A short
   "Removed because Plan mode supersedes it; here's the
   migration story" paragraph in the PR description would help
   users understand the deletion isn't a regression.

# Suggested questions for the author

1. Has a deprecation-then-removal path been considered?
   Specifically: one release where `/deep-planning` parses,
   emits a one-time migration message, and falls back to the
   prompt; then a removal release.
2. Is there a plan to ship a canonical "create-plan" workflow
   template alongside this removal, so users have a clear

3. The deleted prompt variants encoded model-family-specific

# Verdict

**Approve, with a follow-up requested for the migration UX.**
The decision to remove a redundant feature is correct — feature
overlap erodes user trust ("which one am I supposed to use?")
and inflates maintenance cost. The code change is clean and
small. The doc rewrite is thoughtful, articulating the
replacement practice clearly.

The main gap is *user-facing migration*. A user with
`/deep-planning` in muscle memory will get a confusing "command
not found" experience, and a user with the command embedded in
a workflow will see silent breakage. Both are addressable with
one release of deprecation messaging and a canonical
replacement workflow template. Ship the removal, but file the
follow-ups.

# What I learned

Three takeaways, all about deprecation discipline.

**First**, the *removal-vs-deprecation* trade-off in built-in
features. There are two camps in API removal:

- "Hard removal: rip the bandage off, force the migration in
  one release, smaller surface area to maintain."
- "Soft deprecation: at least one release of warnings before
  removal, so users can migrate without breaking their
  workflows."

For *internal APIs* hard removal is fine — you control the
callers. For *user-facing features in a tool with shell
integration and stored workflows*, soft deprecation is almost
always the right call. The cost is one extra release; the
benefit is that no user wakes up to a broken muscle-memory
workflow with no diagnostic. This PR makes the harder choice
without (visibly) addressing the migration UX. That's not
wrong but it's a noticeable rough edge.

The general principle: **the cost of soft deprecation scales
with the surface area of the deprecation; the cost of hard
removal scales with the user count.** A small feature with
many users (typing `/deep-planning` is muscle memory for a
real cohort) should get soft deprecation. A large feature
with few users can get hard removal.

**Second**, the *replace-with-equivalent* discipline. The
strongest deprecations come bundled with their replacement.
"Don't use X, use Y" — where Y is shipped in the same release
and ideally takes a `--migrate-from-X` flag — is vastly easier
on users than "X is gone, here's a paragraph about doing it
yourself". The PR's doc rewrite gestures at "use Plan mode
+ ask for markdown" as the replacement, but doesn't ship a
canonical `/plan` command or workflow template. The user is
left to figure out the prompt themselves. For a feature whose
*entire value* was a carefully-tuned prompt, removing the
prompt and asking the user to invent their own loses
information that took real engineering to develop. A companion
"workflow library entry" PR would close the gap.

**Third**, the *prompt-as-code* preservation question. The
deleted prompt variants represent ~1500 lines of prompt-
engineering work, refined across model families. Deleting
them from the codebase deletes them from the project's
collective memory unless they're preserved somewhere. Other
projects in this space (model-card repos, prompt-engineering
guides) keep an "archive of deprecated prompts" — not as
shipped code but as documentation/reference. cline's repo
deletes them from history (well, not history, but from the
discoverable surface). For prompts in particular, where the
craft is in the wording and the wording is the artifact, this
is worth a separate preservation step. The one-line summary:
*shipped code can be deleted; engineered prompts deserve to be
archived even when retired.*

The fourth meta-lesson: removing features is good. Most
codebases under-remove because removal is uncomfortable and
stays uncomfortable until the unused feature finally bites
someone. Cline's willingness to remove a non-trivial built-in
feature is a healthy signal about project maintenance culture.
The improvement opportunity is to make removal *gracefully
loud* (deprecation warning + canonical replacement + preserved
artifact) rather than *silently complete*.
