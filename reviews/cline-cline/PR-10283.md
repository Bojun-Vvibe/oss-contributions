# Context

Cline supports "skills" — markdown files with frontmatter that
describe agent capabilities. Three sources exist: project skills
(workspace), global skills (user's home dir), and now remote
skills delivered via enterprise org config. The remote schema
(`globalSkills: GlobalInstructionsFile[]`) had been merged
earlier (#10236) and the dashboard could already write into it,
but the extension had no read-side wiring — saved skills sat in
the remote config doing nothing.

# Problem

Three intertwined problems:

1. **The feature was half-built.** Remote schema existed but
   the extension didn't surface remote skills in discovery,
   the system prompt, the UI toggles, or the tool handler.
   Users who had skills pushed to them by their org
   administrator could not see or use them.
2. **Identity drift between `entry.name` and `frontmatter.name`.**
   The dashboard could produce remote entries where the
   metadata `name` and the frontmatter `name` disagreed. A
   strict implementation would reject these; a permissive one
   would silently use one over the other and confuse the user.
3. **Latent architectural issues** the wiring effort exposed:
   a race in `applyRemoteConfig`, missing `alwaysEnabled`
   enforcement in the tool handler, the View button not
   knowing how to open `remote://skill/` URIs, and tooltip
   text hardcoded to "rule" for all instruction types.

# Design

The PR's structural choices are unusually disciplined for a
feature wiring of this scope. Five worth highlighting:

**(1) `frontmatter.name` is canonical, drift is warned not
rejected.** Rejecting drift would silently hide
org-configured skills (the worst possible failure for an
enterprise feature). Warning + including means admins see the
drift in logs and end users still get the skill. The
trade-off is correctly biased toward "show what you have."

**(2) `remote:` path prefix.** Remote skills get a synthetic
path prefix that flows through toggle stores, content
loading, and UI rendering. It's never written to disk. This
is the right shape for a "looks like a path but isn't"
identifier — it composes with all the existing path-keyed
data structures without forcing a parallel "remote skills
table" everywhere.

**(3) Single shared validator.** `parseRemoteSkillEntries`
replaces three duplicated frontmatter-parsing call sites
across `skills.ts`, `refreshSkills.ts`, and
`remote-config/utils.ts`. This is the right time to
consolidate — wiring a new feature is when duplication
becomes obvious and the consolidation is cheapest.

**(4) `skills.ts` stays a pure utility.** Zero `StateManager`
coupling: remote entries are *passed in* by callers that
already have state access. Keeps the module testable in
isolation and inverts the dependency direction correctly.

**(5) Atomic `replaceRemoteConfig`.** The original
`applyRemoteConfig` did `clearRemoteConfig()` followed by
field-by-field repopulation. A concurrent reader landing in
that window saw an empty cache. Replaced with an atomic
swap. This fix benefits rules and workflows too, not just
skills — finding it during the wiring effort and fixing it
broadly is the right move.

Precedence is documented and enforced: remote (enterprise) >
disk-global (user) > project (workspace). `alwaysEnabled`
remote skills cannot be toggled off, enforced both in the UI
(disabled toggle with tooltip) and at execution
(`UseSkillToolHandler` re-checks the flag).

# Risks

- **`remote:` prefix collisions.** If a user ever names a
  local skill file with a literal `remote:` prefix in its
  on-disk path, behavior is undefined. Filesystems on the
  three target OSes don't make this likely, but a defensive
  validator at toggle/load time would harden the contract.
- **Drift-tolerant lookup is two source-of-truth.** Warning on
  `entry.name ≠ frontmatter.name` is the right call now, but
  it leaves a quiet operational debt: which is "the name" in
  logs, telemetry, error messages? Pick one (frontmatter)
  consistently and document it.
- **`alwaysEnabled` is enforced in two places** (UI + handler)
  to defense-in-depth against a stale toggle. Correct, but if
  the two checks ever drift (e.g., a third entry point —
  subagent runner — gets added), the user-facing contract
  will silently weaken. A single
  `isSkillEnabledForExecution(entry, toggles)` helper that
  every entry point calls would prevent the drift class.
- **Race fix is silent.** `replaceRemoteConfig` fixes a real
  race but the PR doesn't document how it was discovered
  (presumably from flaky tests or production logs). A short
  comment at the new method explaining "do not split into
  clear+repopulate" would prevent a future "cleanup" from
  re-introducing the bug.
- **53 unit tests, no integration test.** The wiring spans
  state, remote-config, system prompt, tool handler, UI, and
  proto. Each layer has unit coverage; an end-to-end test
  that pushes a remote config and asserts the skill appears
  in the system prompt after a refresh would lock in the
  wiring as a contract.

# Suggestions

1. Centralize `isSkillEnabledForExecution(entry, toggles)`
   and route every consumer through it (handler, system
   prompt, subagent runner, UI). One source of truth for
   `alwaysEnabled` enforcement.
2. Add a leading comment to `replaceRemoteConfig` explaining
   *why* it exists (the previous clear+repopulate window).
   Critical context that a future contributor will lack.
3. Add an integration test fixture: a remote config with one
   `alwaysEnabled` skill, one user-toggleable skill, and a
   drift case. Assert all three render correctly in the UI
   and resolve correctly in the system prompt.
4. Telemetry event when drift is observed (`name ≠
   frontmatter.name`) so admins can detect dashboard bugs
   before users notice.

# Verdict

**Ship it.** This is feature wiring done right: the original
gap is closed, five latent bugs uncovered along the way are
fixed in scope, the architectural choices (canonical
frontmatter name, `remote:` prefix, shared validator,
StateManager-free utility, atomic swap) are all defensible,
and the test coverage is dense (53 new unit tests, 1392
total passing). The PR description itself is a model — it
documents not just what changed but *why each design
decision*, which is exactly what cross-layer review needs.

# What I learned

Wiring a feature across 16 files is the cheapest moment to
fix structural debt that's been hiding in any of those
files. The discipline is: when you touch a module to add the
new path, audit what you find and fix it in the same PR if
the fix is in scope; spin off a separate PR if it isn't. The
five "bugs fixed beyond the original feature" in this PR are
exactly that pattern. Also: when an enterprise feature
*could* fail loudly (reject on drift) or *quietly include*
(warn on drift), bias toward inclusion + warn. The cost of
hiding an admin-pushed asset is much higher than the cost of
showing a slightly mislabeled one.
