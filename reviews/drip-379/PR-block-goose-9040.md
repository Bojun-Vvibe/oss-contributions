## What it does

Extends the goose ACP `_goose/sources/*` CRUD endpoints (previously skill-only) to support a second source type: agents. Adds an optional `metadata: Option<BTreeMap<String, serde_json::Value>>` field to `SourceEntry`, `CreateSourceRequest`, and `UpdateSourceRequest` for source-specific frontmatter, regenerates `acp-schema.json` accordingly, and introduces a full agent-CRUD pipeline in `crates/goose/src/sources.rs` covering parse (`parse_agent_markdown`), build (`build_agent_md`), slug-derivation (`slugify_agent_name`), filesystem layout resolution (`global_agents_dir`, `project_agents_dir`, `inferred_agent_root`), CRUD operations (`create_agent_source`, `update_agent_source`, `delete_agent_source`), and directory scanning (`scan_agents_from_dir`). Reworks the existing `parse_frontmatter` helper at `:11-32` to use line-by-line delimiter scanning instead of the prior `split("---")` approach (correctly handles `---` inside YAML values), and refactors `summon.rs` to consume the new shared `parse_agent_markdown` helper instead of its own private `AgentMetadata` struct.

## Observations

- **`parse_frontmatter` rewrite at `sources.rs:11-32` is a real correctness improvement.** The previous `split("---").collect()` model would mis-split any agent file whose YAML contained a literal `---` (e.g. inside a multi-line string or comment). The new implementation scans for delimiter lines, normalizes `\r` line endings via `is_frontmatter_delimiter` at `:34`, and only treats a bare `---` line as a delimiter — semantically much closer to standard YAML frontmatter parsers. Worth a regression test pinning the "YAML body contains `---`" case.
- **Reserved-key contract is enforced in both directions.** `RESERVED_AGENT_METADATA_KEYS = ["name", "description"]` at `:38` is filtered out on write (`build_agent_md:99-103` skips reserved keys when extending frontmatter from user-supplied `metadata`) and on read (`sanitize_agent_metadata` is invoked in `create_agent_source:213` and `update_agent_source:236`). The doc comment on `SourceEntry.metadata` at `custom_requests.rs:18-21` correctly documents the contract for SDK consumers.
- **`update_agent_source` rename path at `:255-269` does write-new-then-delete-old with rollback** — if `fs::remove_file(&current_path)` fails after the new file is created, it `fs::remove_file(&target_path)` to undo, then returns the original error. Correct ordering, but the rollback `let _ = ...` swallows the cleanup error silently. If the rollback also fails the user is left with two files (original + duplicate) and no signal. A `warn!` log on rollback failure would be cheap.
- **Slug derivation at `slugify_agent_name:94-128` collapses any non-alphanumeric run to a single `-`.** Two distinct names ("My Agent" and "My_Agent") produce the same slug `my-agent` and therefore collide on the filesystem. The `source_already_exists` check at `write_new_file:32-40` (via `create_new(true)` + `ErrorKind::AlreadyExists`) catches the collision at write time, which is correct, but the resulting error message says "A source named X already exists" referring to the *display name* the user just typed — confusing if the conflicting file was created from a different display name. Worth surfacing the colliding existing path in the error.
- **`inferred_agent_root` at `sources.rs:129-148` resolves containment by canonicalize-then-`starts_with`.** Solid against `..` traversal in user-supplied paths. The fallback `canonicalize_or_original` at `:125-127` returns the original path if canonicalization fails (e.g. for to-be-created files), which is correct for the create path but means `is_global_agent_file` could return a wrong answer if the file doesn't exist yet. Only matters in narrow edge cases.
- **`require_crud_source_type` at `:60-67` correctly tightens the previous `require_skill_type` to `matches!(source_type, SourceType::Skill | SourceType::Agent)`.** The error message is updated in lockstep ("Only 'skill' and 'agent' are currently supported"). Anyone calling the CRUD endpoints with a recipe `SourceType` will continue to get a clear validation error rather than a silent path mismatch.
- **`summon.rs` migration is the canary for the shared parser.** The previous private `AgentMetadata` struct at lines 87-95 (deleted) hardcoded the recognized fields (`name`, `description`, `model`); the new `parse_agent_markdown` returns a generic `BTreeMap` for non-reserved keys, which `summon.rs:179-186` then accesses via `parsed.frontmatter.metadata.get("model").and_then(|m| m.as_str())`. Behavior should be equivalent, but worth checking whether any prior `model: 123` (numeric) frontmatter would have errored under `serde_yaml::from_str::<AgentMetadata>` (yes — type-required string) but now silently returns `None` from `as_str()`. Subtle behavior change, probably benign but worth a release note.
- **`acp-schema.json` regeneration is mechanical** but reviewers should confirm the schema was actually regenerated from the Rust types via the project's codegen step rather than hand-edited; the diff at `:1509-1531` and `:1668-1675` is consistent with auto-generation but no signal of which command produced it.
- **No tests visible in the truncated diff for the new agent CRUD operations.** A 1693-line diff with such substantial new logic should ship with at least round-trip coverage (create → list → read → update → list → delete → list-empty) and edge cases (slug collision, metadata reserved-key filtering, rename across dir = create-then-delete rollback). If those tests exist in the diff sections beyond line 700 they're fine; if not, this is the major gap.

## Verdict: needs-discussion

The core design is sound — shared frontmatter parser, reserved-key contract, canonicalize-based path containment, write-new-then-delete-old rename — and the `parse_frontmatter` rewrite is a genuine correctness fix unrelated to the agent-CRUD goal. But three things hold this back from a clean merge-after-nits: (1) test coverage for the new agent CRUD pipeline isn't visible in the reviewed portion of the diff and a +1200 net-line change deserves explicit round-trip + edge-case coverage, (2) the slug-collision error message confuses users when two display-names canonicalize to the same slug, and (3) the silent `let _ = fs::remove_file(&target_path)` rollback at `update_agent_source:265` should at minimum log on failure. Worth a maintainer round on whether the testing story is adequate before this lands on a release branch.
