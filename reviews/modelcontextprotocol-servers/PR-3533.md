# Context

The `sequential-thinking` reference MCP server exposes a single
tool (`sequentialthinking`) whose input schema requires several
typed fields: `thoughtNumber`, `totalThoughts` (positive ints),
`nextThoughtNeeded`, `isRevision`, `needsMoreThoughts` (booleans),
plus optional `revisesThought` / `branchFromThought`. The schema
is declared with strict `z.number()` and `z.boolean()` validators,
which Zod enforces structurally — a value of `"1"` is rejected
even though every reasonable consumer would interpret it as the
integer `1`.

# Problem

LLM clients calling the tool intermittently send these fields as
*strings* — `{"thoughtNumber": "1", "nextThoughtNeeded": "true"}`
— instead of native JSON numbers/booleans. The PR description
attributes this to LLM-side serialization quirks; from observation
this is a real, recurring failure mode for *every* model behind
*every* MCP client. Different vendors marshal tool arguments
through different code paths (some go through a Python `json.dumps`
of structured args, some pass the model's literal text), and the
text path produces JSON with stringified primitives more often
than anyone expects.

The result is a Zod validation failure on the server side, the
tool call gets an error response, the model sees the error,
re-tries with a slightly different argument shape, often retries
*incorrectly*, burns a few thousand tokens, and the user gets a
broken sequential-thinking session with no clear root cause.

# Design

The fix replaces strict validators with coercive ones:

```ts
thoughtNumber: z.coerce.number().int().min(1).describe("..."),
totalThoughts: z.coerce.number().int().min(1).describe("..."),
revisesThought: z.coerce.number().int().min(1).optional()...,
branchFromThought: z.coerce.number().int().min(1).optional()...,
```

`z.coerce.number()` runs `Number(value)` first, then validates the
result. Constraints (`.int().min(1).optional()`) still apply to
the post-coercion value, so injection of `"-1"` or `"abc"` still
fails — just at the right validation layer.

Booleans are subtler. `z.coerce.boolean()` is a *trap*:
`Boolean("false")` is `true` in JavaScript (any non-empty string
is truthy), so `z.coerce.boolean().parse("false") === true`. The
PR correctly avoids this by hand-rolling a preprocessor:

```ts
const coercedBoolean = z.preprocess((val) => {
  if (typeof val === "boolean") return val;
  if (typeof val === "string") {
    if (val.toLowerCase() === "true") return true;
    if (val.toLowerCase() === "false") return false;
  }
  return val;
}, z.boolean());
```

The preprocessor canonicalises the two known string forms; any
other value flows through unchanged so `z.boolean()` rejects it
with a real error. This is exactly the right shape — the
preprocessor's job is *only* to normalize the known-confused
representations, not to rescue every truthy/falsy.

# Risks

- **`z.coerce.number()` is more permissive than just `"1" → 1`.**
  It will also coerce `null` → `0`, `""` → `0`, `[]` → `0`,
  `["5"]` → `5`. A client that accidentally sends
  `"thoughtNumber": null` will silently land at thought `0`, fail
  the `.min(1)` check, and return an error — but a client that
  sends `"thoughtNumber": [3]` will silently succeed and the
  server will treat it as `3`. That coupling-of-array-to-scalar
  is unlikely to bite in practice but is worth a future test.
- **The preprocessor only matches `"true"`/`"false"`** (case
  insensitive). Real LLM output occasionally uses `"True"`,
  `"yes"`, `"1"`, `"0"`, `"on"`. The current set is the
  conservative one (good) but the next bug report is predictable
  — someone's model writes `"yes"` and gets a Zod error. Worth
  considering whether `"1"`/`"0"` should also coerce to
  `true`/`false`, since `z.coerce.number()` *will* accept those
  strings as numbers and the user's mental model already maps
  them to truthy/falsy.
- **No tests added.** The PR description says "existing tests
  pass unchanged since they already use native types", which is
  exactly the gap — the bug class only arises with the
  *non*-native types, so existing tests cannot lock the fix in.
  A `test("accepts thoughtNumber as string")` and
  `test("rejects nextThoughtNeeded as 'maybe'")` pair would be
  ten lines and prevent the next regression.
- **Schema description text mentions "numeric value, e.g., 1, 2,
  3"**. After this PR the field also accepts `"1"`, but the
  description does not say so. Models will keep occasionally
  sending strings because nothing tells them otherwise; this is
  fine (the coercion absorbs it), but a description note like
  "(string forms also accepted)" would steer well-behaved
  callers toward the native form.

# Suggestions

1. Add the missing tests (3-5 lines each).
2. Consider extending the preprocessor to accept `"yes"`/`"no"`
   and `"1"`/`"0"` as boolean shorthand — these are documented
   common LLM outputs and the cost of accepting them is one map
   entry each.
3. Update the schema description text to match the new accepting
   behaviour.
4. The same coercion gap likely exists in *other* reference MCP
   servers in this repo. A quick grep for `z.number()` /
   `z.boolean()` in tool input schemas would surface them.

# Verdict

**Ship it.** The fix is correctly scoped, the boolean preprocessor
in particular is *exactly* the right shape (rather than the wrong
shape `z.coerce.boolean()` provides), and the production impact
is meaningful — every silent retry this prevents saves real model
tokens and user patience.

# What I learned

`z.coerce.boolean()` is a footgun across the ecosystem. The Zod
docs warn about it, but it lives in the same `z.coerce.*`
namespace as the other (correct) coercers, so it gets used
uncritically. The general rule: *coercion is only safe when the
target type has a canonical string representation that round-
trips*. Numbers have one (`"42"` ⟷ `42`), so `z.coerce.number()`
is fine. Booleans don't (no agreed canonical "true"/"false"
spelling — JavaScript chose "any non-empty string is true",
Python chose "non-empty string is true *but* `bool("False")` is
also true, while `ast.literal_eval("False")` is `False`), so
generic boolean coercion is unsafe and any boolean coercer must
be hand-rolled to enumerate the explicit string forms it accepts.

A second insight worth keeping: schema validation at the MCP
server boundary is a *robustness boundary*, not a correctness
boundary. The server cannot guarantee the LLM will send well-
typed JSON; it can only choose to either accept the messy
reality (with explicit, bounded coercion) or reject it (and force
every client to filter their LLM output). The accept-with-bounded-
coercion path scales; the reject path does not.
