# Context

`mcp-server-git` is one of the reference MCP servers shipped under
`modelcontextprotocol/servers`. It exposes git primitives (`git_log`,
`git_show`, `git_branch`, `git_create_branch`, etc.) as MCP tools that
an LLM agent can call. Under the hood it uses GitPython, which
ultimately shells out to the `git` binary with caller-controlled
arguments. The threat model that matters here is twofold: (a) the LLM
itself emits a tool call with a hostile argument, and (b) the
filesystem already contains a hostile artifact (e.g. a ref file at
`.git/refs/heads/--format=evil` planted by a sibling MCP tool with
filesystem write access). Either way, an argument that begins with
`-` will be parsed as a flag by `git`, allowing the attacker to flip
formatting flags, inject `--exec`, or otherwise change the command
semantics under the agent's nose.

# Problem

Two of the surface's tools — `git_diff` and `git_checkout` — already
had a leading-dash guard with a corresponding malicious-ref test.
The other four did not. So an attacker who could persuade the agent
(or a sibling tool) to pass `--format=evil` as a `git_show` revision,
`--track=evil` as a branch name, `--exec=rm -rf /` as a `git_log`
timestamp, or `--exec=...` as a `git_branch` `contains` filter could
have flag-injected the underlying `git` invocation. This is a
defense-in-depth bug rather than a 0-day — GitPython's higher-level
APIs do escape some forms — but the existence of the partial fix
makes the gap obvious to any reader.

# Design

The fix is the simplest possible: copy the existing pattern to the
four missing call sites. Each site gets a leading-`-` guard with
the same comment style ("Defense in depth: reject … starting with
'-' to prevent flag injection") and raises `BadName` (the
GitPython error class already used for invalid refs) or `ValueError`
(for non-ref string params like timestamps).

```py
def git_show(repo: git.Repo, revision: str) -> str:
    # Defense in depth: reject revisions starting with '-' to prevent flag
    # injection, even if a malicious ref with that name exists (e.g. via
    # filesystem manipulation)
    if revision.startswith("-"):
        raise BadName(f"Invalid revision: '{revision}' - cannot start with '-'")
```

The `git_log` site adds the guard to *both* `start_timestamp` and
`end_timestamp`, raising `ValueError` (matching how the function
treats other timestamp validation failures rather than `BadName`,
which is for ref names).

`git_create_branch` guards both `branch_name` and the optional
`base_branch`. `git_branch` guards `contains` and `not_contains`.

The 59-line test addition is exemplary in matching the existing
malicious-ref test pattern. `test_git_show_rejects_malicious_refs`
plants a `--format=evil` file under `.git/refs/heads/`, confirms it
exists on disk (so the guard isn't relying on the ref being
non-resolvable), then asserts `BadName` is raised. This is the
right test shape — the test would fail if a future refactor moved
the guard *after* the GitPython resolution step (which would let
the malicious ref be loaded before the guard rejected it).

# Risks

- **The guard is a `startswith("-")` check, not a parser-aware
  check.** A `--` separator (which is how callers normally pass
  user-controlled values to `git`) is not used inside these
  GitPython call sites. The right long-term fix is to switch to
  `git ... --` invocations where applicable, but that requires
  GitPython's lower-level `repo.git.execute([...])` rather than
  the high-level wrappers used here. The PR opts (correctly) for
  the smaller change.
- **`git_log` guards only `start_timestamp` / `end_timestamp`.**
  The `max_count: int` parameter is type-int-narrowed by the
  function signature so it cannot carry a flag string, but if a
  future PR adds a string-typed parameter to this function, the
  guard pattern needs to be remembered. A small `_reject_flag(name,
  value)` helper invoked at the top of every git-shelling function
  would be more drift-resistant than scattered inline guards.
- **No mention of `git -c key=value` injection.** That family of
  flags is not currently exposed, but is worth a tracked follow-up
  to either explicitly deny or audit for any new function.
- **Error message echoes the rejected value verbatim.** Generally
  fine for an MCP server (the LLM gets the value back), but if
  the rejected value is itself a long secret-looking string, it
  will end up in the MCP transcript. Truncate to e.g. 64 chars
  for log hygiene.

# Suggestions

1. Factor a `_reject_flag_arg(name: str, value: Optional[str])`
   helper to centralize the pattern. Cost: one helper, ~4 LOC.
   Benefit: any future contributor who adds a new arg-bearing
   function gets a one-line addition with a single review-time
   reminder.
2. Add a CONTRIBUTING note under `src/git/` documenting the
   "every user-controlled string passed to a `git` shell-out must
   be flag-guarded" rule.
3. Consider moving to `repo.git.execute(["git", "show", "--",
   revision])` style for the truly user-controlled paths; this is
   a separate hardening but worth tracking.

# Verdict

**Ship it.** The PR is the right shape: it copies an established
pattern to the call sites that were missing it, with matching tests
including the "exists on disk" variant that catches future
refactors. The defense-in-depth framing is correct — this isn't a
known exploit, it's the closing of an obvious-on-inspection gap.

# What I learned

When a security guard exists at *some* call sites but not others,
the missing call sites are not just a bug — they are a documentation
bug too. A reader scanning `server.py` sees the guard on `git_diff`
and reasonably assumes the absence on `git_show` means
"intentional, the tool is safe by some other mechanism." Closing
the gap is the obvious fix; *centralizing* the guard so the
inconsistency cannot recur is the better fix. The pattern
generalizes: any time you find yourself adding the same 3-line
defensive check to function N, your real PR is "extract a helper,
audit all current call sites, add the helper to all of them, and
write a CONTRIBUTING note that future call sites must call it." A
linter rule (e.g. a Semgrep pattern: "any call to GitPython's
`repo.commit/refs/git.execute` with a string parameter must be
preceded by `_reject_flag_arg`") would lock the contract in CI.
