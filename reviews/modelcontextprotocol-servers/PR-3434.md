# Context

`mcp-server-filesystem` is the reference MCP filesystem server. It
gates every tool through a path-validation pipeline whose first
hop is `normalizePath()` in `src/filesystem/path-utils.ts`. That
function reduces user-supplied paths to a canonical form — handles
`.` and `..`, normalizes separators per platform, etc. — so that
downstream allow-list checks can compare strings without worrying
about equivalent representations of the same location. The
function delegates to Node's `path.normalize()`, which is generally
well-behaved but inherits a Windows-specific quirk that breaks the
filesystem server's gatekeeping.

# Problem

Node's `path.normalize("C:")` on Windows returns `"C:."` rather
than `"C:\\"`. The reason is historical: a bare drive letter on
Windows refers to the *current working directory on that drive*,
which is per-drive state preserved by the OS for compatibility
with DOS. Node treats `C:` as the equivalent of "current dir on
C:" and emits `C:.` to make that explicit.

For an MCP server that needs canonical paths to compare against an
allow-list (e.g. "is `requested_path` inside `/allowed/root`?"),
this is broken. A tool call asking for `C:` gets normalized to
`C:.`, which doesn't equal `C:\` (the actual drive root), so the
allow-list check either rejects an intentionally allowed access or
— worse, depending on the rest of the validation pipeline — accepts
something the user did not intend. The reported issue (#3418)
describes the symptom: downstream path validation logic breaks
because the normalized form is not a real, resolvable filesystem
location.

# Design

The fix is six lines of detection-and-prefix:

```ts
// On Windows, if we have a bare drive letter (e.g. "C:"), append a separator
// so path.normalize doesn't return "C:." which can break path validation.
if (process.platform === 'win32' && /^[a-zA-Z]:$/.test(p)) {
  p = p + path.sep;
}
```

Placed *before* the `path.normalize()` call, this preempts the
quirk: `normalizePath("C:")` becomes `path.normalize("C:\\")`,
which returns `"C:\\"` correctly. Lowercase drive letters
(`d:`) also match the regex; the test case asserts the resulting
canonical form is `D:\\` (uppercased by `path.normalize`).

The 11-line test addition is correctly written: it forces
`process.platform = 'win32'` via `Object.defineProperty` so the
test runs identically on a Linux CI runner. The two assertions
cover both casing variants (`C:` and `d:`).

# Risks

- **`process.platform` is overwritten via
  `Object.defineProperty(...{writable: true})`.** This works in
  the surrounding test file (which has other tests doing the same
  thing) but is global mutable state across test files in
  parallel runs. Vitest's default `forks` pool isolates this per
  worker, but a future migration to `threads` mode could see test
  pollution. Worth tracking.
- **The regex `^[a-zA-Z]:$` is correct for ASCII drive letters
  but does not cover the rarely-encountered `\\?\C:` or
  `\\.\C:` extended-length / device-path prefixes.** Those
  prefixes change the resolution semantics in ways
  `path.normalize` already handles, but worth a comment naming
  the in-scope set explicitly.
- **The fix runs on `win32` only.** Same code path on
  POSIX-named pipes or WSL paths is untouched. This is correct
  because `C:` is meaningless on POSIX, but a future contributor
  who wants to run the filesystem server inside a WSL2 process
  serving a Windows-side path may want to extend the
  detection — worth a note.
- **No path-traversal-equivalence test.** It would be nice to
  also assert that `normalizePath("C:/foo/../") === "C:\\"`
  works correctly *after* the bare-drive-letter fix, just to
  prove the new pre-prefix doesn't interfere with downstream
  `..` resolution. The existing test file probably covers this
  generically, but a co-located case would be tighter.

# Suggestions

1. Add a third test case asserting that the fix is a no-op on
   non-Windows: `process.platform = 'linux'; normalizePath("C:")`
   should return whatever the existing POSIX behavior is, to
   guard against future code motion accidentally applying the
   prefix on POSIX.
2. Add an inline comment naming the in-scope drive-letter set
   (e.g. "ASCII drive letters only; extended-length and device
   paths fall through to `path.normalize`'s native handling").
3. Consider a follow-up that adds a Windows-specific test matrix
   (forward slashes in mixed paths, UNC paths, drive-relative
   `C:foo`) — `C:foo` is the *other* DOS-era ambiguity that
   means "foo relative to current dir on C:", and `path.normalize`
   handles it but it would be good to lock the behavior in tests.

# Verdict

**Ship it.** The fix is small, correctly localized to the
platform that has the quirk, and the test runs cross-platform via
`platform` injection. The PR description and code comment together
clearly explain the failure mode (`C:` → `C:.`), which is the kind
of context future maintainers absolutely need when seeing a magic
six-line block before a `path.normalize` call.

# What I learned

`path.normalize` is one of those Node APIs that looks like a
universal canonicalizer but is actually a thin wrapper over
platform-specific resolution rules with platform-specific edge
cases. The bare-drive-letter case is one of three or four
Windows-specific gotchas (the others being drive-relative paths
like `C:foo`, UNC paths, and the `\\?\` extended-length prefix);
treating `path.normalize` as opaque means inheriting all of them.
The right architectural posture for code that *gates security
decisions* on canonical form (allow-list checks, sandbox
boundaries) is to wrap `path.normalize` in a project-specific
`canonicalize()` function with explicit pre-processing for the
known quirks and a doc comment listing what is and is not
handled. This PR is the first step of that pattern; a future
PR that consolidates `normalizePath` plus drive-relative handling
plus UNC handling into a single annotated function would be the
finished form.
