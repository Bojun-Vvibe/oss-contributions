# Context

The `fetch` MCP server is one of the original reference servers in
`modelcontextprotocol/servers`. Its job is the simplest possible:
download a URL, convert the HTML to Markdown, return it as a tool
result. Despite the simplicity it is one of the most heavily used
servers — almost every coding-agent stack ships it as a
default-enabled tool because "fetch a URL" is a primitive every agent
needs.

The pipeline is `httpx → readabilipy → markdownify`. Readabilipy wraps
Mozilla's Readability.js, which is the same library Firefox's "Reader
Mode" uses. Readability is *aggressive* — it filters out anything that
looks like chrome (nav, footer, sidebar) and anything CSS-hidden.
That's the right default for a typical news article, but it produces a
silent failure on the modern progressive-SSR pattern: the server emits
the real article content inside hidden containers (`display:none` or
`visibility:hidden` until hydration), and the client unhides them
after JS boot. Readability strips the hidden containers. The agent
then sees a single line of "Loading..." text and reports the page as
empty.

This PR adds a three-stage fallback. **Stage 1**: try Readability as
before. **Stage 2**: if the result is shorter than 1% of input HTML
length, retry with `use_readability=False` (readabilipy's pure-Python
content extractor that doesn't filter on CSS visibility). **Stage 3**:
if stage 2 also falls below the threshold, BeautifulSoup-strip
`<script>`/`<style>` and pass the raw HTML through markdownify.
+135/-4, half of which is tests, in two files. Small but worth
careful reading because the heuristic ("1% of input") is a behavior
change that could degrade quality on some pages.

# Diff walkthrough

The core change is in `extract_content_from_html`:

```python
min_expected_length = max(1, len(html) // 100)

# Stage 1: Try Readability (best quality for standard pages)
ret = readabilipy.simple_json.simple_json_from_html_string(
    html, use_readability=True
)
content_html = ret.get("content", "")
if content_html:
    content = markdownify.markdownify(
        content_html,
        heading_style=markdownify.ATX,
    )
    if len(content.strip()) >= min_expected_length:
        return content
```

Three things to notice. First, the `.get("content", "")` replaces the
old `ret["content"]` direct subscript — a real defensive improvement
because readabilipy can return `{"content": None}` (the test
`test_readability_none_content_triggers_fallback` covers this).
Second, the threshold gates *after* markdownify conversion, not on
the raw HTML — that's correct because Readability tends to produce
short HTML that converts to long Markdown (table-of-contents
expansion, etc.) and you want to measure the user-visible artifact.
Third, the threshold itself is `len(html) // 100` floored at 1 — see
risks below for why this bound is debatable.

Stage 2 is the same shape with `use_readability=False`. Stage 3 is
the interesting one:

```python
# Stage 3: Convert full HTML directly with markdownify (last resort).
# Strip <script> and <style> first — markdownify renders them verbatim
# as plain text, which injects large blobs of JS/CSS noise into the
# output.
soup = BeautifulSoup(html, "html.parser")
for tag in soup(["script", "style"]):
    tag.decompose()
content = markdownify.markdownify(
    str(soup),
    heading_style=markdownify.ATX,
)
if content.strip():
    return content

return "<error>Page failed to be simplified from HTML</error>"
```

The script/style decomposition is the right call: markdownify by
default treats script/style as text content and writes the entire JS
or CSS body verbatim into the markdown output, which both wastes the
agent's context window and confuses the agent's downstream parser.
The author's inline comment explains the rationale, which is
exactly the kind of comment that prevents this from being
"optimized away" by a future contributor.

The test file adds 90 lines of new coverage in a single
`TestExtractContentFallback` class. The cases are well-chosen:
sufficient content (no fallback), Readability strips → stage 2 wins,
both readabilipy modes fail → stage 3 wins, completely empty input,
Readability returns `None`, and an explicit threshold test that
confirms "1% of input" is the gate. The mocking pattern is a bit
brittle (patches a module-level import path) but it's consistent
with how the rest of `test_server.py` mocks dependencies, so it's
fine for this codebase.

# Risks / things a maintainer would catch

1. **The 1% threshold is a magic constant with no `--threshold`
   override.** A 50KB HTML page with a legitimate 200-character
   summary triggers a fallback (200 < 500). That's almost certainly
   not what the user wanted — they got the article-summary page and
   the fallback drowns it in nav/footer noise. The threshold should
   either be configurable via tool input, or use a more sophisticated
   heuristic (e.g., "fall back only if Stage 1 produced a single text
   node that looks like a loader/spinner"). At minimum, a test that
   asserts "short-but-real article passes Stage 1" would lock down
   the no-regression intent.
2. **Stage 3 produces low-quality Markdown.** Raw HTML through
   markdownify yields the entire page including navigation, footer,
   ads, cookie banners. For an agent operating in a fixed context
   window, "verbose noisy result" is sometimes worse than "empty
   result" — the noise crowds out the actual answer. A user who is
   debugging why their fetch returns garbage now has a harder time
   distinguishing "page was empty" from "page was successfully but
   uselessly extracted at stage 3". A `_meta` field reporting which
   stage succeeded would help debugging and would let downstream
   agents weight the result.
3. **`markdownify(html=str(soup), ...)` throws away formatting.**
   `str(soup)` re-serializes the parsed DOM, which can subtly change
   whitespace and self-closing tag handling versus the input HTML.
   For markdownify this is mostly fine but it's a behavior change
   from "pass HTML through directly".
4. **Threshold compares stripped Markdown length to raw HTML
   length.** A 100KB HTML page with 5KB of meaningful Markdown will
   trigger the fallback (5000 < 1000... wait, actually 5000 > 1000,
   so it passes). OK but the *units* are mismatched (HTML chars vs
   Markdown chars), and someone reading the code has to do this
   arithmetic in their head. Either compare like-for-like (Markdown
   length vs HTML text-content length, where HTML text-content is
   the BeautifulSoup `get_text()` length), or document the asymmetry.
5. **No CPU bound.** A 10MB HTML page now goes through Readability
   *twice* and BeautifulSoup once. Previously it would do
   Readability once and return — the worst case is now ~3x slower
   on pathological pages. For an MCP server invoked from an
   interactive agent loop, that could be a noticeable wait. A size
   guard ("if len(html) > 5MB, skip stage 2 and 3") would be cheap
   defense.
6. **`ret = readabilipy.simple_json.simple_json_from_html_string(...)`
   does not have a timeout.** Readabilipy spawns a Node subprocess
   under the hood when `use_readability=True`. On a deadlocked
   subprocess this hangs forever. Pre-existing bug, but worth
   noting because the new code path *doubles* the exposure.
7. **`BeautifulSoup(html, "html.parser")` uses the stdlib parser.**
   On malformed HTML the stdlib parser is markedly worse than `lxml`
   or `html5lib`. Users running fetch against scraped/dirty HTML may
   see unexpected stage-3 output.
8. **The error message `<error>Page failed to be simplified from
   HTML</error>` is in pseudo-XML.** Agents that try to parse it as
   actual XML will fail. Either return a structured `{"error": ...}`
   per the MCP error convention or use a plain string.
9. **Test brittleness around `Date.now()`-style ordering.** The
   tests mock the module-level import path
   `readabilipy.simple_json.simple_json_from_html_string`. If
   readabilipy ever exports a different alias (or `from
   readabilipy.simple_json import ...` is used at the top of
   `server.py` instead of qualified access), the tests silently
   stop intercepting. A fixture that patches the function as it's
   *used* in the server module would be more robust.
10. **No coverage for the `<script>`/`<style>` decomposition.** A
    test like "HTML containing `<script>alert('x')</script>` →
    output does not contain `alert`" would lock that defensive
    behavior in.

# Suggested questions for the author

1. Was the 1% threshold tuned empirically against a corpus, or
   chosen as a defensible default? If the latter, would you accept
   exposing it as a `min_extracted_length` tool argument with the
   current value as default?
2. Have you considered emitting a `_meta` (or warning header) that
   tells the downstream agent which stage produced the result?
   That feels like the right MCP-shaped affordance for "I had to
   fall back".
3. The stage-3 output can be very noisy on real-world pages. Is
   there appetite for an *aggressive* boilerplate-stripper at
   stage 3 (drop `<nav>`, `<footer>`, `<aside>`, `[role=banner]`,
   `[role=navigation]`, etc.) before markdownify? It would catch
   most of the worst-case noise without any extra dependencies.
4. Is the BeautifulSoup parser choice (stdlib `html.parser`) the
   intentional one, or would `lxml` (already a transitive
   dependency of readabilipy) be preferable for stage 3?
5. There's no test for "raw HTML containing inline JS bigger than
   the article" — should we add one to lock down the
   script/style decomposition?

# Verdict

**Approve with the threshold-configurability follow-up.** The
underlying problem (Readability strips hidden SSR markup) is real
and well-known, and the three-stage fallback is the standard fix.
The 1% threshold is the only debatable design choice; everything
else (script/style decomposition, defensive `.get`, threshold-
based gating after markdown conversion) is correct. The tests
exercise the new code paths well enough for a reference server.

# What I learned

The reusable insight here is about *fallback chains over
quality-vs-completeness tradeoffs*. When you have N extractors
ranked by quality (Readability > readabilipy > raw markdownify)
and they sometimes lie by omission (returning empty/short content
that they wrongly think is "the right answer"), you cannot just
use the first non-error response. You need a *gate* that classifies
the response as "good enough" or "trigger fallback". The hard part
is the gate itself — get it wrong in either direction and you get
either pages dropping good content or pages drowning in noise.

This PR's gate is a length heuristic. That's a reasonable starting
point, but the *real* signal is structural: a "Loading..." page has
a single text node, no headings, no links — versus a real article
which has headings, paragraphs, and outbound links. A more
sophisticated gate would compute a "structure score" (number of
unique heading levels + paragraph count + link count) and trigger
the fallback when the score is below a per-stage threshold. That
generalizes better than character count, and it's the pattern
worth pulling out into a reusable helper for any tool that does
"best-effort content extraction with fallback".

The second insight is the more general one: any time you build a
fallback chain, **emit which leg won**. That makes the system
debuggable from the outside without adding logging that you have
to plumb through. In MCP specifically, the `_meta` field on tool
results is the right place for this signal.
