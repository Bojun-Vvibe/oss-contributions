# Context

`mcp-server-fetch` is one of the reference MCP servers — a small
Python process that reads JSON-RPC frames from stdin, fetches a
URL, and writes the response back to stdout. It is the canonical
example anyone follows when building their own MCP server.

# Problem

A single malformed byte on stdin kills the server process.

The PR description quotes a fuzzing comparison:

| Server | Crashes | Survives |
|--------|---------|----------|
| `mcp-server-fetch` (`raise_exceptions=True`)   | 61/65 | 4/65 |
| `mcp-server-sqlite` (`raise_exceptions=False`) | 0/65  | 65/65 |

`server.run(read_stream, write_stream, options,
raise_exceptions=True)` makes the SDK propagate any
deserialization or handler error as an `ExceptionGroup` up out of
`run()`. With nothing catching it, the process exits. The next
JSON-RPC request from the client never lands; the user sees the
MCP integration go silent. Restart, re-send, hope the prior frame
isn't replayed.

The blast radius is larger than the immediate crash: an MCP host
that has spawned this process now has a stale handle, possibly an
in-flight request whose result will never arrive, and (depending
on the host) a model that timed out waiting and continued the
conversation with a confused tool-call history.

The other reference servers (`sqlite`, `time`, `filesystem`) all
already pass `raise_exceptions=False`, so the SDK swallows the
error, logs it, and continues serving subsequent requests. `fetch`
and `git` were the outliers.

# Design

The diff is one character flip:

```py
# before
await server.run(read_stream, write_stream, options, raise_exceptions=True)
# after
await server.run(read_stream, write_stream, options, raise_exceptions=False)
```

That's the entire change. The MCP SDK's stdio loop, with
`raise_exceptions=False`, internally catches per-request exceptions
in its `try/except` handler, returns a JSON-RPC error frame to the
client, and continues reading the next message. With `True` the
exception breaks out of the loop.

What the PR gets right:

- **Aligns with sibling reference servers.** Picking the same
  default as `sqlite`/`time`/`filesystem` removes one source of
  ambiguity for downstream implementors who copy from the
  reference set. The default behaviour for an MCP server should
  be "survive a bad frame, surface it as a JSON-RPC error".
- **Single-line surface.** No accidental change in any other
  behaviour. Easy to backport, easy to revert if the SDK's error
  path is ever broken.
- **Cites the fuzzing data.** Comparison numbers (61/65 vs 0/65)
  give reviewers a concrete reason to believe the change is
  load-bearing.

# Risks

- **`raise_exceptions=False` may hide real bugs.** The original
  `=True` was probably set during development so server-side
  Python errors would be loud. Switching to `=False` means a
  legitimate handler bug (e.g. an unhandled `KeyError` in
  `fetch_url`) is now logged-and-swallowed rather than crashing
  the dev's process. For the reference *production* server this
  is the right tradeoff (uptime > developer ergonomics), but
  there is no follow-up logging review to verify that the SDK's
  internal error-logging is loud enough that real bugs still get
  noticed.
- **No new test.** The fix targets a class of failure (malformed
  input causes process exit) that is *exactly* what fuzz tests
  are for. A pytest fixture that pipes 65 hand-written malformed
  frames through the server and asserts the process stays up
  would lock in the fix permanently. The PR description says
  "Test plan" with three checkboxes but they are all "verify
  manually" — those break first.
- **Doesn't fix `git`.** The PR description acknowledges
  `git` has the same bug but is out of scope. That's defensible
  for a single-purpose patch, but means the next person to file
  the bug for `git` re-discovers the same fuzz harness, the same
  PR pattern, and the same review. Bundling them as one
  cross-cutting fix would have been a stronger move.
- **Behavior on persistent malformed input.** With `=False`, a
  client that streams *only* malformed frames now keeps the
  server alive forever, replying to each with an error. There
  is no rate limit or "after N malformed frames in a row, give
  up" backstop. Probably benign over stdio (the host is the
  only writer), but worth noting.

# Suggestions

1. Add a fuzz-style test: a fixture file with 30+ malformed
   JSON-RPC frames, plus a test that pipes each through the
   server and asserts both (a) no process exit and (b) a JSON-RPC
   error frame is written back. Five lines of fixture, ten lines
   of test, permanent regression coverage.
2. Sweep the rest of the reference set for the same `=True`
   pattern. The PR mentions `git`; verify there are no others.
3. Add a section to the reference-server `README.md`
   documenting that *new* reference servers should default
   `raise_exceptions=False` and explaining why. Without that the
   next reference server PR re-introduces the bug.
4. Consider whether the SDK should default `raise_exceptions=False`
   inside `server.run()` itself. Library-level safe defaults are
   a much wider safety net than per-server review.

# Verdict

**Ship it.** A 1-line fix backed by a 61/65 vs 0/65 fuzz
comparison is a textbook "merge this immediately" PR. The
follow-ups (test fixture, sibling sweep, SDK default change) are
the real engineering work and belong in separate PRs.

# What I learned

The "raise vs swallow" toggle in protocol-handling library code
is a recurring API design problem. The library can't know whether
its caller wants development-time loudness or production-time
robustness. Three observations from this PR:

1. **The default matters more than the option.** Whatever the
   library defaults to, that is what 90% of users get. If
   `raise_exceptions` defaults to `False`, only people who
   actively chose `True` will crash; if it defaults to `True`,
   only people who actively flipped it will survive bad frames.
   The right default for a "long-running protocol server" SDK is
   `False`. The MCP SDK's choice (default unclear; reference
   servers split) creates a footgun.

2. **Reference implementations are also documentation.** Five
   reference servers, four with `=False`, one with `=True`,
   means readers of the reference set learn the wrong lesson at
   a 20% rate. Consistency across reference implementations is
   itself a form of doc.

3. **Fuzz testing surfaces this bug class instantly.** 65 hand-
   crafted malformed frames is a small fixture. The fact that
   it took until 2026 for someone to actually run it against
   `mcp-server-fetch` is the bigger signal: every JSON-RPC
   server should have this fuzz harness from day one, and the
   right place to put it is *in the SDK's test suite*, run
   against any registered `Server` instance. Then no individual
   server has to remember to fuzz itself.
