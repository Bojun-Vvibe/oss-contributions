# Context

`mcp-server-memory` is the reference MCP "knowledge graph" server.
It stores entities and directed relations and exposes
`create_entities`, `create_relations`, `read_graph`, `search_nodes`,
and `open_nodes` as MCP tools. The semantics are simple: callers
either ask for the whole graph (`read_graph`), narrow by query
(`search_nodes("Acme")`), or explicitly open a subset by name
(`open_nodes(["Alice"])`). In each filtered case the returned
`KnowledgeGraph` carries the matched entities and a slice of the
relation set.

# Problem

The relation slice was wrong. Both `searchNodes` and `openNodes`
filtered the relation set with `&&`:

```ts
const filteredRelations = graph.relations.filter(r =>
  filteredEntityNames.has(r.from) && filteredEntityNames.has(r.to)
);
```

This required *both* endpoints to be in the result set, which
meant: any time you opened a single node, you got that node's
data with `relations: []`. Any relation `Alice → Bob` that touched
the opened set on only one side was silently dropped. The reported
issue (#3137) describes the canonical confusion: open a single
"date" node looking for related incidents, get back zero relations,
incorrectly conclude the node is isolated, and miss the entire
point of having a knowledge graph in the first place.

The deeper problem is that this is the *opposite* of standard graph
query semantics. "Open node X" in any graph database (Neo4j, RDF
triple stores, NetworkX) returns X plus all edges *touching* X,
because that is the only useful definition — the whole reason
graphs are interesting is that traversal from X reveals neighbors
not yet known. A "both-endpoints-must-match" filter degenerates to
"only return relations within an already-known closed set", which
makes `open_nodes` strictly less useful than `read_graph` for any
discovery purpose.

# Design

The fix is one operator per call site:

```ts
// Include relations where at least one endpoint is in the requested set.
// Previously this required BOTH endpoints, which meant relations from a
// requested node to an unrequested node were silently dropped — making it
// impossible to discover a node's connections without reading the full graph.
const filteredRelations = graph.relations.filter(r =>
  filteredEntityNames.has(r.from) || filteredEntityNames.has(r.to)
);
```

The two call sites (`searchNodes` and `openNodes`) get the same
change with comments tuned to each function's purpose. The author
keeps the entity filtering identical — only the relation predicate
changes — which is correct: the entity set is what the caller
asked for; the relation set is the answer to "what edges should
the caller know about given that they care about these nodes."

The 41-line test addition is thorough and worth highlighting
because it codifies the contract in four orthogonal cases:

- `should include outgoing relations to unmatched entities`
- `should include incoming relations from nodes not in the open set`
- `should include relations connected to opened nodes` (the
  one-node case, which is the issue's repro)
- `should include all relations connected to opened nodes` (the
  multi-node case, where one relation matches at one endpoint and
  another at both endpoints)

The pre-existing test `should exclude relations to unopened nodes`
is deleted (it was encoding the bug).

# Risks

- **Result-set growth.** A single `open_nodes(["X"])` on a hub
  node with thousands of edges will now return all those edges,
  whereas previously the response was a pleasingly empty list.
  This is the correct behavior but it is a payload-size change
  that could affect downstream consumers tuned to the old
  (broken) shape. A `max_relations` parameter could provide a
  cap, but adding it now without a use case is YAGNI.
- **Returned relations reference entities not in `entities`.**
  After the fix, `result.relations[i].to` may name an entity not
  in `result.entities`. Consumers that assume "every endpoint of
  every relation is present in entities" will break. This is the
  *intended* shape (it's how every graph DB returns adjacency
  results), but should be called out in the server's tool
  description. I did not see a corresponding update to the tool
  schema or description in the diff.
- **Symmetric behavior in `search_nodes` may surprise callers.**
  A query for `"Alice"` that hits one entity will now return
  Alice's outgoing edge to Bob even though Bob did not match the
  query. That is correct, but worth documenting in the
  `searchNodes` description.
- **Performance.** Both filters are still O(R) over relations.
  No change in complexity. For very large graphs the bigger
  payload will dominate.

# Suggestions

1. Update the tool descriptions for `open_nodes` and
   `search_nodes` to explicitly state "returns relations
   connected to the result set, including edges to entities not
   in the result set." This is now a load-bearing semantic.
2. Optionally add an `include_orphan_endpoints: boolean` param
   defaulting to `true` for callers who *do* want the
   closed-subgraph view. Cheap to add, future-proofs for the
   `read_graph`-equivalent-on-subset use case.
3. Consider also returning the *entities* referenced by the
   returned relations (i.e. an "expansion frontier") as a
   separate field — common in graph APIs and saves the caller a
   round-trip when they want to follow edges.

# Verdict

**Ship it.** The fix is one operator change per call site,
correctly motivated by both the user-reported bug and the
universal graph-DB convention. Test coverage is precisely the
right shape (the four orthogonal cases above), and the deleted
test that previously encoded the bug is the cleanest possible
signal that the contract has flipped. Documentation of the
semantic change in the tool description should be a tracked
follow-up.

# What I learned

`&&` vs `||` in a relation filter is a tiny code change that
happens to be a fundamental disagreement about what your data
structure *is*. The `&&` shape says "the result is a *subgraph
induced* by the entity set" — only edges within the set survive.
The `||` shape says "the result is the entity set plus its edge
neighborhood" — edges crossing the boundary survive. Both are
valid graph operations, but they have different names: induced
subgraph vs ego-network/neighborhood. A graph-database design
that doesn't make this choice explicit in the API is going to
ship the wrong default to half its users. The lesson: any time
your filter predicate joins two membership checks with a boolean
operator, the operator choice is encoding a semantic that
deserves a name in the docstring, not just a code comment.
