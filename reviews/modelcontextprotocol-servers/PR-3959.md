# Context

The `memory` MCP server is the reference knowledge-graph memory:
entities (with name, entityType, and a list of free-text
observations) and relations between them. Long-running agent
deployments — coding agents that persist project context across
sessions, customer-support agents accumulating user facts — push
the graph to thousands of entities, each with dozens of
observations. The three read tools (`read_graph`, `search_nodes`,
`open_nodes`) return the *entire* matched subgraph including all
observation text. At scale this single response can blow past the
agent's context window.

This PR adds optional projection/limit parameters to all three
read tools so the agent can ask for cheaper views: metadata only,
filtered by entity type, capped observation count per entity, and
(for `open_nodes`) observation pagination via offset+limit. +86/-65
in a single file. Half of the deletion count is mechanical
formatting cleanup; the substantive change is option plumbing
through the three tool handlers and the `KnowledgeGraphManager`
methods.

# Diff walkthrough

**The shared slice helper.** A new private static at the top of
the manager:

```typescript
private static sliceObservations(obs: string[], options?: { limit?: number; offset?: number }): string[] {
  if (!options || (options.offset === undefined && options.limit === undefined)) return obs;
  const offset = options.offset ?? 0;
  return options.limit !== undefined ? obs.slice(offset, offset + options.limit) : obs.slice(offset);
}
```

Three small details. First, the no-op fast path returns `obs`
directly (not a copy) when no slicing is requested — avoids
allocations on the common path. Second, the function correctly
handles all four corner cases (no limit + no offset, limit only,
offset only, both). Third, returning `obs.slice(offset)` when
`limit` is undefined is safe even when offset > obs.length
(JavaScript `slice` clamps).

**`readGraph` becomes parameterized.**

```typescript
async readGraph(options?: {
  includeObservations?: boolean;
  observationLimit?: number;
  entityTypes?: string[];
  metadataOnly?: boolean;
}): Promise<KnowledgeGraph> {
  const graph = await this.loadGraph();
  let entities = graph.entities;
  if (options?.entityTypes?.length) {
    entities = entities.filter(e => options.entityTypes!.includes(e.entityType));
  }
  const withObservations = options?.metadataOnly || options?.includeObservations === false ? [] : undefined;
  const processedEntities = entities.map(e => ({
    name: e.name,
    entityType: e.entityType,
    observations: withObservations ?? KnowledgeGraphManager.sliceObservations(e.observations, { limit: options?.observationLimit })
  }));
  const names = new Set(processedEntities.map(e => e.name));
  return {
    entities: processedEntities,
    relations: graph.relations.filter(r => names.has(r.from) && names.has(r.to))
  };
}
```

The conditional `withObservations ?? sliceObservations(...)` is
the projection: if the caller wants no observations
(`metadataOnly: true` *or* `includeObservations: false`), every
entity gets `[]`; otherwise the observations are sliced by
`observationLimit`. The cleverness is that `withObservations`
sets to `[]` once, *outside* the map, so the map body doesn't
re-evaluate the predicate per entity — minor but correct.

The relation filter is `names.has(r.from) && names.has(r.to)` —
**both endpoints must be in the result set**. This is consistent
with the prior `readGraph` behavior (which returned all
relations), but it's *different* from `searchNodes` and
`openNodes` (see next sections) which use `||`. The asymmetry is
intentional but unstated. With `entityTypes` filtering, this
means: a relation from an entity-of-included-type to an
entity-of-excluded-type gets dropped silently. That's likely
*not* what the user wanted — they wanted to see the connections
out of the included set.

**`searchNodes` uses OR.**

```typescript
return {
  entities: processedEntities,
  relations: graph.relations.filter(r => names.has(r.from) || names.has(r.to))
};
```

This matches the old behavior exactly (search has always returned
relations where either endpoint matches the query). Good.

**`openNodes` adds offset.**

```typescript
async openNodes(names: string[], options?: {
  includeObservations?: boolean;
  observationLimit?: number;
  observationOffset?: number;
}): Promise<KnowledgeGraph> {
  ...
  observations: withObservations ?? KnowledgeGraphManager.sliceObservations(e.observations, { offset: options?.observationOffset, limit: options?.observationLimit })
  ...
  relations: graph.relations.filter(r => entityNames.has(r.from) || entityNames.has(r.to))
}
```

`openNodes` is the only one of the three with `observationOffset`,
which makes sense: it's the per-entity-detail view, the place
where pagination through long observation lists matters. This is
also the only of the three that *previously* used the OR semantics
for relations — preserved here. Note the comment from prior PRs
about "previously this required BOTH endpoints, which silently
dropped relations to unrequested nodes" — the OR fix is preserved.

**The Zod schemas at the tool registration.** Three updates of
the same shape:

```typescript
inputSchema: {
  includeObservations: z.boolean().optional().describe("Whether to include observation content (default: true)"),
  observationLimit: z.number().int().positive().optional().describe("Max observations per entity"),
  entityTypes: z.array(z.string()).optional().describe("Filter entities by types"),
  metadataOnly: z.boolean().optional().describe("Return only entity names and types, no observations")
},
```

The `.describe()` strings are the part the LLM actually sees in
the tool catalog. They're concise and accurate. The
`z.number().int().positive()` rejects zero and floats, which is
the right validation (a `limit: 0` would silently return nothing,
a `limit: 1.5` would be confusing). One ergonomic miss: there's
no upper bound. A caller that asks for `observationLimit:
1000000` will get the entire observation list back, defeating the
point of the parameter.

# Risks / things a maintainer would catch

1. **Asymmetric relation filter between `readGraph` (AND) and
   `searchNodes`/`openNodes` (OR).** `readGraph` with
   `entityTypes` set will silently drop relations that connect
   to filtered-out entities. The user's mental model is "show me
   the People subgraph" — they expect relations from People to
   Organizations to appear. Either align all three to OR, or
   document the asymmetry, or add a third explicit
   `relationFilter: "any" | "both"` option.
2. **No upper bound on `observationLimit`.** A confused agent that
   asks for `observationLimit: Number.MAX_SAFE_INTEGER` defeats
   the optimization. Cap at, say, 1000 and emit a warning when
   clamped.
3. **`metadataOnly` and `includeObservations: false` mean the
   same thing.** Two flags with overlapping semantics. The
   computed `withObservations` predicate handles both
   (`metadataOnly || includeObservations === false`), but the
   tool surface should pick one and deprecate the other.
4. **`entityTypes: []` (empty array) is silently ignored** —
   `options?.entityTypes?.length` is falsy. A user who passes
   `[]` probably means "filter to nothing" (zero results). The
   current behavior returns *everything*. Either treat `[]` as
   "no entities match" or reject `[]` at the schema level via
   `.min(1)`.
5. **Case sensitivity in `entityTypes` filter.** `includes` is
   exact-match. If the graph has `entityType: "Person"` and the
   user asks for `entityTypes: ["person"]`, zero matches. Should
   be lowercased on both sides, or documented as case-sensitive.
6. **`observationOffset` exists only on `openNodes`.** A natural
   extension to `searchNodes` would be `entityOffset` for
   pagination across the result set, which is the more common
   need at scale. This PR doesn't add it; not a blocker but
   worth a follow-up.
7. **`processedEntities.map` always re-allocates the entity
   object** — even on the no-op happy path where no slicing is
   needed and observations should be returned as-is. The
   short-circuit in `sliceObservations` returns the same array
   reference, but the wrapping object is freshly constructed. On
   a 10k-entity graph that's 10k allocations on the hot read
   path. A `if (!options) return graph;` early-exit in
   `readGraph` would save it.
8. **No streaming.** The whole result graph is JSON-serialized in
   one shot. `JSON.stringify(graph, null, 2)` on a large graph
   is both an allocation spike and an MCP-frame size hazard.
   Beyond this PR's scope, but a real concern at the scale this
   PR aims for.
9. **`relations` are not filtered by the *projected* entity list
   — they're filtered by the entity *names*.** Subtle but
   matters: if `metadataOnly: true` removes observations from an
   entity, the entity name is still in the projected set, so
   relations referring to it stay. This is correct, but the same
   logic isn't applied uniformly: in `readGraph`, relations are
   filtered against `processedEntities` (which has had
   `entityTypes` filter applied), so they correctly drop to the
   filtered set. Good.
10. **Schema description claims `default: true` for
    `includeObservations`.** That's accurate behaviorally
    (`undefined !== false`), but the schema itself doesn't set a
    default. `.default(true)` on the Zod schema would make this
    explicit and ensure agents that introspect the schema see the
    same default behavior the docstring claims.

# Suggested questions for the author

1. The relation-filter asymmetry between `readGraph` (AND) and
   `searchNodes`/`openNodes` (OR) — was that intentional? With
   the new `entityTypes` filter, `readGraph` users will likely
   expect to see relations *out of* the filtered set, which AND
   semantics drops. Would you align them, or expose an explicit
   `relationFilter` option?
2. `metadataOnly` and `includeObservations: false` behave
   identically. Could we deprecate one (probably `metadataOnly`,
   since `includeObservations` composes better with
   `observationLimit`)?
3. Is there appetite for capping `observationLimit` /
   `observationOffset` at a sane upper bound (e.g., 1000) and
   emitting a `_meta.warning` when clamped?
4. Should `observationOffset` also exist on `searchNodes`, and
   should there be a sibling `entityOffset` / `entityCursor` for
   paginating across entities? That's the actual "scalability"
   primitive most agents will want at scale.
5. Are there any existing memory-server clients in the wild that
   pass `entityTypes: []` expecting "filter to nothing"? If so,
   the current behavior (silently returning everything) is a
   foot-gun.

# Verdict

**Approve.** This is a focused, useful PR for a real scaling
problem. The implementation is clean, the additions are
backward-compatible (every new parameter is optional), the test
hygiene of the underlying file is preserved. The relation-filter
asymmetry between `readGraph` and the others is a real semantic
inconsistency that should be addressed in a follow-up — either
alignment or explicit documentation. The lack of an upper bound
on `observationLimit` is a minor footgun. Otherwise ready.

# What I learned

The interesting design lesson is about **projection in tool
APIs for LLM clients**. When designing a read tool, the natural
default is "return everything" because that's the simplest
contract and any caller can ignore what they don't need. For
*human* clients that's correct. For *LLM* clients it's exactly
backwards: the most expensive resource is the LLM's context
window, and "return everything" forces the agent to scroll past
mountains of irrelevant text on every read. The correct default
for a tool that an LLM will call is "return the *cheapest* view
that's likely to be useful, with explicit opt-ins to widen
projection." This PR is moving the memory server in that
direction — `metadataOnly` is the projection a smart agent should
actually use first ("does the graph contain a Person named
Foo?") before fetching the heavyweight observation lists.

Concretely: any MCP read tool that returns variable-size data
should default to a *summary* shape (counts, names, schema)
and require an explicit "give me details" flag for the heavy
projection. That inverts the common API design instinct, but
matches LLM economics. A future iteration of the memory server
might even want a separate `count_*` family of tools for the
metadata-only path, so that the LLM doesn't have to negotiate
flags on the heavyweight tools at all.

The second insight is smaller: when you have N read tools that
operate on the same underlying object (the knowledge graph),
the projection options should be *consistent* across all N. The
fact that this PR adds `observationOffset` to `openNodes` only
and not `searchNodes` is a small inconsistency that callers will
notice and complain about. Designing the option set as a
*single* TypeScript type (`type ReadOptions = {...}`) shared
across all read tools, with per-tool subtypes only where
necessary, prevents this drift.
