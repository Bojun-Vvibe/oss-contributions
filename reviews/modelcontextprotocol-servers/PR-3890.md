# Context

The `filesystem` MCP server is one of the most-used reference
servers — every coding agent that runs against a local repo wires
it in for the read/write/list/search primitives. The current tool
surface covers single-file ops (`read_file`, `write_file`,
`edit_file`), directory listing (`list_directory`,
`directory_tree`), and search (`search_files`). What's been
missing is a *cross-directory* primitive: "given two checkouts /
two branches / two extracted tarballs, tell me what differs".

Today an agent that needs to answer "did anything change between
these two folders?" has to enumerate both with `directory_tree`,
diff the JSON in its head, and re-issue `read_file` for any pair
where it suspects a content change. That's expensive in context
tokens and slow over the MCP wire. This PR adds a single
`compare_directories` tool that returns four buckets:
`onlyInDir1`, `onlyInDir2`, `differentContent`, `identical`. It
adds 344 lines, no deletions, across three files (server
registration, library impl, vitest tests). The shape is
straightforward but the file has two real correctness issues
that a careful reviewer should catch.

# Diff walkthrough

**The tool registration.** In `index.ts:702`:

```typescript
server.registerTool(
  "compare_directories",
  {
    title: "Compare Directories",
    description:
      "Compare two directories and report differences. Returns files only in first dir, " +
      "only in second dir, files with different content (size/mtime), and identical files. " +
      "Useful for syncing and finding changes between directory versions.",
    inputSchema: {
      dir1: z.string(),
      dir2: z.string(),
      compareContent: z.boolean().optional()
    },
    outputSchema: { content: z.string() },
    annotations: { readOnlyHint: true }
  },
  ...
)
```

The `annotations: { readOnlyHint: true }` is the right MCP signal —
hosts that gate write operations behind user approval don't need to
prompt for this. The description is honest about the default
behavior (compares by size and mtime, not content). The
`outputSchema: { content: z.string() }` is wrong, though: the
handler returns `{ content: [{type: "text", text}], structuredContent:
{ result } }`, where `structuredContent.result` is the rich
`DirectoryComparisonResult`. The schema declared on the tool says
the output is a single string `content`, which doesn't match the
shape the agent will actually receive — clients that validate
output against schema will reject this.

**The comparison core.** In `lib.ts:278`:

```typescript
export async function compareDirectories(
  dir1: string,
  dir2: string,
  compareContent: boolean = false
): Promise<DirectoryComparisonResult> {
  // Temporarily set allowed directories for validation during comparison
  const previousAllowed = getAllowedDirectories();
  setAllowedDirectories([dir1, dir2]);

  try {
    const dir1Files = await searchFilesWithValidation(dir1, "**/*", [dir1]);
    const dir2Files = await searchFilesWithValidation(dir2, "**/*", [dir2]);
    ...
  } finally {
    setAllowedDirectories(previousAllowed);
  }
}
```

This is the most concerning part of the PR. The function takes
the *global* `setAllowedDirectories` setting, mutates it to allow
`dir1` and `dir2`, runs the comparison, and restores the previous
setting in `finally`. Two problems.

First, **this completely bypasses the security boundary** that
`setAllowedDirectories` exists to enforce. The point of the
allowlist is that an agent cannot read paths outside of it — even
indirectly. Here, `compareDirectories` adds whatever the caller
passes in, even if those paths were never granted. The
`validatePath` call in `index.ts` is what's *supposed* to enforce
this, but inside `lib.ts` we re-grant. Note that `validatePath` is
invoked in the tool handler before reaching `compareDirectories`,
so on the *normal* call path the dirs are already validated against
the original allowlist. But the underlying `compareDirectories`
function is `export`ed and now has the property "if you call me
with any two paths, I'll widen the allowlist for you." Anyone
adding a future tool that calls this helper has just inherited a
silent path-traversal escape hatch.

Second, **`setAllowedDirectories` is not concurrency-safe**.
Module-level mutable state means two concurrent
`compareDirectories` calls (or any other tool fired in parallel)
will race: caller A widens the allowlist, caller B's
`validatePath` checks against A's widened list, then A restores.
The MCP server processes one request at a time *today*, but
nothing in the server framework guarantees that — and the moment
streaming/parallel tool calls land, this becomes a real bug. The
fix is to thread allowed-dirs as a parameter, not as global state.

**The size+mtime heuristic.** In the same function:

```typescript
if (stat1.size !== stat2.size) {
  differentContent.push({...});
} else if (compareContent) {
  const contentsEqual = await compareFileContents(file1, file2);
  if (contentsEqual) identical.push(relPath);
  else differentContent.push({...});
} else {
  // compareContent is false, use mtime as indicator
  if (stat1.mtimeMs !== stat2.mtimeMs) {
    differentContent.push({...});
  } else {
    identical.push(relPath);
  }
}
```

Default behavior is "same size + same mtime ⇒ identical". This is
fast but is a meaningful definition of "identical" that the
docstring doesn't fully advertise. Specifically: a file edited
in-place, restoring the original content (same size), and
back-dated via `touch -t` to its original mtime, will be reported
as identical despite differing intermediate state. For sync use
cases that's fine; for forensic / "did anything change?" use cases
it's wrong. The `compareContent: true` flag is the escape hatch.

The byte comparator:

```typescript
const [content1, content2] = await Promise.all([
  fs.readFile(file1),
  fs.readFile(file2)
]);
if (content1.length !== content2.length) return false;
return content1.equals(content2);
```

This loads the **entire file** into memory before comparing. For
two 4GB tarballs you've now allocated 8GB of buffer. A streaming
comparator (`createReadStream` + chunk-by-chunk `Buffer.equals`)
would bound the memory at one chunk per file. Less critical than
the security issue but a real footgun for the "compare two large
trees" use case, which is exactly what this tool is for.

**The `searchFilesWithValidation(dir1, "**/*", [dir1])` call.**
This globs everything recursively. There's no pagination, no
result-count cap, no excluded patterns (`.git`, `node_modules`,
`__pycache__`). On a real-world repo comparison, the result will
include tens of thousands of `node_modules` entries and the agent
will get back a multi-megabyte response. The MCP spec doesn't
require pagination but the result will likely exceed many client
buffers.

# Risks / things a maintainer would catch

1. **Allowlist re-grant pattern in `compareDirectories`.** The
   single most important issue. The function mutates global state
   to grant itself access. Even though the *current* call path
   validates first, the function is `export`ed and will be reused.
   The fix: thread allowed-dirs as a parameter to
   `searchFilesWithValidation` instead of relying on global state.
2. **Concurrency hazard from `setAllowedDirectories` global
   mutation.** Two parallel comparisons race; an unrelated
   parallel tool call sees the widened allowlist for the duration
   of either comparison.
3. **Output schema mismatch.** `outputSchema` declares
   `content: z.string()` but the handler emits both `content`
   (array of text blocks) and `structuredContent` (the full
   result). Either widen the schema or remove `outputSchema`.
4. **Symlink handling is not addressed.** `fs.stat` follows
   symlinks; `fs.lstat` doesn't. A tree containing symlinks to
   outside-the-allowlist targets will be silently followed and
   compared. For a security-sensitive tool, `lstat` + explicit
   symlink-equality semantics is the correct default.
5. **No exclude patterns.** `node_modules`, `.git`, `dist`, build
   artifacts will all be compared, blowing up the result. A
   `excludePatterns: string[]` parameter (defaulting to common
   noise dirs) would make the tool actually usable.
6. **Whole-file `readFile` in `compareFileContents`.** Allocates
   2× file size in memory. Streaming chunk compare is the standard
   pattern.
7. **No file-count or output-size cap.** A repo with 1M files
   will produce a 1M-line text response. The structured result
   alone would be tens of megabytes.
8. **No handling for differing file *types*** (one is a directory,
   the other is a file with the same name). `searchFilesWithValidation`
   probably returns only files, but if a future variant returns
   both, the comparison will compare a dir's stat to a file's stat
   silently.
9. **`path.relative(dir1, file)` can produce different separators
   on Windows vs POSIX.** A test that runs on both OSes (the
   repo's CI does) would catch the rare case where `path.join`
   semantics diverge between the two `Set<string>` lookups.
10. **Test setup uses `Date.now()` for unique tmp dirs.** Two
    test runs invoked within the same millisecond (which can
    happen on fast CI) will collide. Use `fs.mkdtemp` (the
    standard Node API for unique tmp dirs) instead.
11. **`differentContent[]` shape leaks `mtimeMs` in raw form.**
    Fine for programmatic use but the human-readable `lines.push`
    output omits mtime entirely — inconsistent surface.

# Suggested questions for the author

1. The `setAllowedDirectories` mutation pattern in
   `compareDirectories` looks like a security regression — am I
   reading the threat model correctly? Would you accept threading
   the allowlist as a parameter instead?
2. Is the default of `compareContent: false` (size+mtime only) the
   right ergonomic choice? Most users will assume "identical"
   means byte-equal. A safer default would be `compareContent:
   true` with a `fastMode` opt-out for performance-sensitive
   syncs.
3. Whole-file `readFile` for content compare is a memory hazard
   on large files. Would you accept a streaming comparator?
4. The `outputSchema` on the tool registration doesn't match the
   handler output (string vs array+structuredContent). Should it
   be removed or widened?
5. No exclude patterns means `node_modules` and `.git` get
   compared — should the tool default-exclude common
   build/cache dirs, take an `excludePatterns` arg, or both?
6. `fs.stat` follows symlinks. Is the intended behavior "compare
   resolved targets" or "compare symlink as a unit"? Should be
   documented.

# Verdict

**Request changes.** The feature is useful and the implementation
is mostly clean, but the `setAllowedDirectories` mutation pattern
is a real security regression in a security-sensitive server. The
output-schema mismatch and the whole-file-read memory issue are
also material. I'd recommend the author:
1. Refactor `compareDirectories` to accept allowed-dirs as a
   parameter rather than mutating global state.
2. Switch byte-compare to streaming.
3. Either remove the output schema or fix it to match the actual
   shape.
4. Add at least one `excludePatterns` test.

The tests as written are solid for the happy paths; they just
don't cover the security or memory concerns above.

# What I learned

Two patterns worth pulling out.

**First**, the "temporarily mutate global state, restore in
`finally`" pattern is a common code smell that almost always
masks a missing parameter. When you find yourself writing
`setX(newValue); try {...} finally {setX(prevValue)}`, the
correct refactor is to make `X` a parameter to the inner work,
not a property of the module. The reason isn't just clean code —
it's that the `finally` restore is *only* correct in
single-threaded contexts. In a concurrent setting, the restore
unwinds someone else's intended state. JavaScript's single-event-
loop semantics make this *usually* fine for synchronous bodies,
but the moment you introduce `await` (as this PR does), other
tasks can interleave on the event loop and observe the mutated
state. This is the canonical way to introduce a TOCTOU bug in
async JS code.

**Second**, when you're building a tool whose primary use case is
"compare two large trees", treat *result size* as a first-class
input dimension. The MCP server framework doesn't enforce
response size caps, but real clients (LLM context windows, JSON
parsers, websocket frame limits) all do. A tool that can return
arbitrarily-large responses on innocuous-looking inputs will
eventually be the source of a "MCP server hung" support ticket.
The right defaults are: implicit excludes for noisy dirs, an
explicit max-result count, and structured *summary* output (the
counts) with a separate `details` paginated query for the
per-file diff. This PR ships the unsummarized-details version,
which will be fine for tiny demos and painful in production.
