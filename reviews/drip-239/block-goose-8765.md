## Context

Adds a complete Simplified Chinese translation (1574 message ids, verified via `pnpm i18n:check`) plus the structural fixes needed to make the rest of the desktop app actually localizable: hardcoded English strings in `ChatInput`, the navigation renderers (`CondensedRenderer` / `ExpandedRenderer`), the navigation-items hook, and the keyboard-shortcut formatter were all bypassing `react-intl` and producing English text regardless of the active locale.

## What's right

- **Translation coverage is complete and verified.** PR body cites `pnpm i18n:check` as the verification primitive — the standard "every extracted message id has a translation" check. 1574 ids translated means no fall-through to English in normal operation.
- **The locale-detection fix at `i18n/index.ts:32-65`** (the new `matchSupported(tag)` function) is the load-bearing structural improvement. The pre-fix detection logic only did exact match + base-language fallback (`pt-BR` → `pt`), which would map a Chinese user's `zh-Hans-CN` browser locale to base `zh` (which doesn't exist in the catalog) and fall through to English. The new function correctly handles the BCP 47 Chinese-script complexity:
  - **Simplified-vs-Traditional discrimination at `:51`** via `/^zh-(hant|tw|hk|mo)\b/.test(lower)` correctly matches the four Traditional-Chinese region/script tags (`zh-Hant*`, `zh-TW`, `zh-HK`, `zh-MO`) and *only* those. The `\b` word boundary is critical — without it, a tag like `zh-twozz` would falsely match.
  - **Simplified-Chinese-aliases at `:52-56`**: `zh`, `zh-Hans`, `zh-Hans-CN`, `zh-SG`, `zh-MY` all map to `zh-CN`. This matches the locale-aliasing convention used by Chrome's `navigator.language` (which often reports `zh-CN` or `zh` for Simplified-using systems regardless of region) and Windows (which reports `zh-Hans-CN` for Simplified Chinese on Chinese Windows).
  - **The "any zh-* that is NOT Traditional counts as Simplified" comment at `:53`** is the right discipline statement — it makes the fallback semantics explicit so a future contributor doesn't see `zh-SG` and assume it should be Traditional (Singapore Chinese is Simplified).
- **`navigator.languages` consultation at `:71-78`** is the right priority order: full preference list first (so a Chinese user on an English Windows UI who has Chinese as a fallback in their language preferences still gets Chinese), then primary `navigator.language` (which may be the OS-locale rather than user-preference). Pre-fix only consulted `navigator.language`, so cross-locale users (Chinese-speaker on English-OS) would always get English.
- **The `Intl.getCanonicalLocales(tag)` use at `:90`** correctly canonicalizes the tag for date/number formatting *separately* from the message-catalog key — so a user with `zh-Hans-CN` gets `messageLocale: "zh-CN"` (catalog key) but `locale: "zh-Hans-CN"` (formatting tag), preserving regional formatting preferences while routing messages through the canonical catalog. The try/catch fallback at `:90-94` correctly handles invalid BCP 47 input (`en-` etc.) by falling back to the matched key.
- **The component-level fixes follow the same pattern.** `ChatInput.tsx:1318` flips `placeholder={getNavigationShortcutText()}` to `getNavigationShortcutText(intl)` (passing the intl object to the formatter so it can localize keyboard-shortcut labels). `:1451` flips `<span className="text-sm">Send</span>` to `<span className="text-sm">{intl.formatMessage(i18n.send)}</span>`. The three navigation renderers (`CondensedRenderer` / `ExpandedRenderer` and a third unseen) all flip `{item.label}` to `{getNavItemLabel(item, intl)}`. This is the right shape: the *renderer* doesn't know what locale to use, but it has access to the `intl` context — so the renderer passes `intl` to a labeling helper that does the actual localization. Each call site change is one line and structurally identical, which is what makes a sweep like this reviewable.

## Risks / nits

- **The `zh-(hant|tw|hk|mo)\b` regex at `:51` will fail to match the case-variant `zh-Hant-TW`** — wait, actually the pre-lowercase at `:50` (`const lower = tag.toLowerCase()`) handles this. Good. But worth a unit test pinning each of `['zh-TW', 'zh-Hant-TW', 'zh-Hant', 'zh-HK', 'zh-Hant-HK', 'zh-MO']` *all* fail to match Simplified, since these are real BCP 47 tags that real Traditional Chinese users will have, and a regression here would silently serve them Simplified Chinese (which is read-but-strongly-disliked by most Traditional users — the political and cultural sensitivity is real).
- **No fall-through unit test for `zh-XX` where `XX` is not in the Traditional list and not a known Simplified region.** E.g. `zh-Latn` (pinyin transliteration), `zh-Hans-MO` (Macau-but-Simplified, theoretically valid). The fallback rule says "any zh-* that is NOT Traditional counts as Simplified", so these would map to `zh-CN` — probably fine, but worth a test arm so the discipline is pinned.
- **The 5026-line zh-CN.json catalog cannot be reviewed line-by-line in a PR** — translation quality will need post-merge native-speaker review. The PR body terminology table (`Session 会话`, `Recipe ...`) is a good start but only covers a handful of high-frequency terms. Worth either (a) a dedicated translation-review issue tagging native-speaker contributors, or (b) a CHANGELOG note flagging that `zh-CN` is "initial translation, refinements via follow-up PRs welcome".
- **The hardcoded native-menu fix is mentioned in the PR body but not visible in this diff slice.** "The native menu bar and sidebar nav — previously hard-coded — now also respect the active locale" — this is the kind of fix that's easy to verify visually but easy to regress because Electron menus are constructed once at startup. A test that reconstructs the menu after a locale change and asserts the menu items match the new locale would pin the regression.
- **The `intl` object threading through `getNavigationShortcutText(intl)` and `getNavItemLabel(item, intl)`** is the right shape but adds a per-callsite parameter that wasn't there before. Worth confirming all call sites of these helpers have access to an `intl` context — a missed call site would show a TypeScript error (good) but if the helper has a default-parameter fallback (`intl?: IntlShape = ...`) the failure mode would be silent re-introduction of English text. A grep for `getNavItemLabel(item)` (no second arg) and `getNavigationShortcutText()` (no arg) across the codebase would close that.
- **`navigator.languages` is `Array<string>` per the spec but real-world implementations sometimes return `null` or undefined values inside the array.** The `if (tag) candidates.push(tag)` guard at `:74` correctly filters those out, but worth a test arm with `Object.defineProperty(navigator, 'languages', { value: ['en', null, 'zh-CN'] })` to pin the discipline.

## Verdict

**merge-after-nits.** Locale-detection fix is structurally correct (Simplified-vs-Traditional discrimination, navigator.languages priority, separate formatting-locale-vs-catalog-key resolution). The component-level i18n sweep is one-line-per-callsite and structurally consistent. The risks are about translation-quality review (a separate post-merge concern), test coverage of the locale-detection edge cases (Traditional-Chinese variants, missed call sites of the threaded `intl` arg), and the unsewn native-menu fix. Add the BCP 47 edge-case unit tests + a grep-confirmation that no helper call sites are missing the new `intl` arg + a translation-review issue, then merge.
