- **PR**: https://github.com/sst/opencode/pull/25273
- **Author**: Brendonovich
- **Head SHA**: `46cd9c09cd9eed785d8b34f535a77bb4c2b8351a`
- **Files**: `packages/desktop-electron/src/main/{index,ipc,menu,server,windows}.ts`, `preload/types.ts`, `renderer/loading.tsx` (+410 / -303)
- **Verdict**: **merge-after-nits**

## Context

The pre-fix `main/index.ts` was the canonical module-scope-mutable-singletons-with-EventEmitter Electron main-process shape: `let mainWindow: BrowserWindow | null = null`, `let server: Server.Listener | null = null`, `const initEmitter = new EventEmitter()`, `let initStep: InitStep`, `const pendingDeepLinks: string[] = []`, `const serverReady = defer<ServerReadyData>()`, `const loadingComplete = defer<void>()`, plus the side-effecting `setupApp()` / `initialize()` chain mutating those module-level bindings from inside `app.on(...)` callbacks. This is the Electron-main-process shape that always drifts: ordering is implicit (deep links arrive before `mainWindow` exists → push to array → flush later, but flush is in *another* callback), the `EventEmitter` lifecycle is unowned, and `defer<T>()` is a homegrown Promise-with-resolver.

This PR replaces all of that with `effect`'s typed concurrency primitives at one layer.

## What's right

- **State is replaced with explicit, named primitives at module top.** `index.ts:80-87`: `initStep = SubscriptionRef.make<InitStep>(...)`, `serverReady = Deferred.make<ServerReadyData>()`, `loadingComplete = Deferred.make<void>()`, `deepLinkQueue = Queue.unbounded<string[]>()`, `deepLinksConsumed = Deferred.make<void>()`, `server = Ref.make<Option.Option<Server.Listener>>(Option.none())`, `menuCommands = PubSub.unbounded<string>()`, `sqliteProgress = PubSub.unbounded<SqliteMigrationProgress>()`. Each is a typed primitive that *names* the concurrency contract: `SubscriptionRef` for "value with subscribers", `Deferred` for "single-shot future", `Queue` for "buffered FIFO", `Ref` for "atomic mutable cell", `PubSub` for "broadcast". The prior `EventEmitter + module-mutable + defer()` salad collapses to "pick the right primitive per slot".
- **App events are a `Data.TaggedEnum` with a `$match`** at `:97-127`: `SecondInstance | OpenUrl | BeforeQuit | WillQuit`, dispatched through one `handleAppEvent(event, deepLinkQueue, mainWindow, server)` function. This is the load-bearing structural fix vs. four parallel `app.on(...)` handlers that each closed over different subsets of module-mutable state — now the closure surface is *explicit parameters* and the four cases share one match. Adding a fifth event type becomes "add a variant + a `$match` arm" instead of "add a fifth handler that reaches into module scope".
- **Closures are eliminated by passing dependencies explicitly.** `focusMainWindow(win: BrowserWindow)` at `:131-134` takes the window as a parameter rather than reading the module-scope `mainWindow` (which had to be null-checked at every call site). `stopServer(ref: Ref.Ref<Option.Option<Server.Listener>>)` at `:136+` takes the ref as a parameter. The "explicit parameters — no hidden closures" comment at `:129` is the right discipline statement.
- **Deep-link ordering becomes a Queue, not an "array + flush flag".** Pre-fix logic was "if `mainWindow` exists, send; else push to `pendingDeepLinks` and hope the renderer-ready callback drains it". Post-fix, deep links go to `deepLinkQueue` (an unbounded `Queue`), and a fiber waits on `deepLinksConsumed: Deferred<void>` to know when the renderer has drained. The "renderer became ready before deep links arrived" race goes away because `Queue.take` blocks until something is offered, and `Deferred.await` blocks until the renderer signals readiness — both are first-class.
- **`Server.Listener` lifecycle is `Ref<Option<Listener>>`**, not `let server: Listener | null`. `Option.isSome(srv)` at the `stopServer` body forces a structural check at the type level rather than a `if (server) server.close()` bare-truthy check, which silently treats `0`/`""`/`false` as "no server" if the type ever drifts.
- **The `import type { InitStep, ... } from "../preload/types"` flips to value import** at `:36` because the new `Data.TaggedEnum` constructors need runtime variants — which is why `InitStep` had to gain a `_tag: "ServerWaiting"` discriminant in `preload/types.ts:+8`. This is the right direction (discriminated union over flat object) but does add a runtime dep on the preload-types module from main.

## Risks / nits

- **Eight `Effect.runSync(...)` calls at module load** (`:80-87`) is the load-bearing trade-off for synchronous module-scope initialization. Each is fine because all eight primitives' constructors are pure and never touch I/O — but a future refactor that swaps any of them for an effect that *does* I/O (e.g. `SubscriptionRef.makeWithInitial(loadFromDisk)`) will silently block the Electron main thread at module load. Worth a comment at `:78-79` pinning that "all `runSync` here must be on pure constructors only — no I/O at module load" so the discipline is explicit.
- **`Effect.runSync(...)` at module top means import-order matters more than before.** If any test imports `index.ts` for typecheck purposes the eight primitives get allocated globally, which is fine but worth confirming there's no `import "./main"` anywhere in test setup.
- **`Data.TaggedEnum` for `AppEvent` doesn't include the `Effect`-side fiber that consumes them.** `:97-127` only defines the *event shape* and the `handleAppEvent` dispatcher; the actual event-loop fiber that pulls from a queue and calls `handleAppEvent` lives elsewhere. Without seeing the full file (the diff cuts off mid-refactor) it's not possible to confirm the consumer is one fiber per app, single-threaded, with proper `Fiber.interrupt` on `WillQuit`. A test asserting `WillQuit → server stops within N ms → app.exit(0) called` would pin the ordering contract.
- **`pendingDeepLinks: string[]` → `deepLinkQueue: Queue.unbounded<string[]>` doubles the array level.** Each offer is `Queue.offer(deepLinkQueue, [url])` (`:115`) — a single-element array per deep link. The renderer-side consumer presumably flattens, but the type is now `string[][]` of pending batches rather than `string[]` of pending links, which is a subtle shape change that any consumer needs to handle. Worth a contract comment at the queue declaration (`:84`) explaining "each batch is one second-instance argv slice" so a future contributor doesn't add `Queue.offer(deepLinkQueue, url)` (singular string) by accident.
- **No tests in the diff.** A 410/-303 refactor of the load-bearing main-process module without any test coverage is the highest-risk part of this PR. At minimum a "boot path" smoke test asserting `initStep` transitions through `ServerWaiting → ServerReady → MainWindowReady` in order (via `SubscriptionRef.changes` stream) would pin the lifecycle.
- **`Deferred.make<void>()` for `deepLinksConsumed`** (`:84`) is single-shot — meaning the *first* time the renderer drains the queue it signals consumed, and any *subsequent* deep links arrive into a queue that is no longer being awaited (or, depending on consumer-side wiring, into a queue that the consumer has stopped reading from). Either the comment needs to clarify that `deepLinksConsumed` is a one-time "renderer is alive enough to receive" signal (and the queue is then drained on every renderer-side ready event independently), or the type should be `SubscriptionRef<boolean>` if it's truly meant to be observable repeatedly.

## Verdict

**merge-after-nits.** The structural direction is right — `EventEmitter + module-mutable + defer()` collapses to `Queue + SubscriptionRef + Deferred + Ref + PubSub` with explicit parameter passing and a `Data.TaggedEnum` event type. The risks are all "comments that pin the discipline" rather than structural concerns, plus the missing test coverage which is the one thing that could let this regress silently after merge. Land it with a follow-up test PR for the boot-path lifecycle.
