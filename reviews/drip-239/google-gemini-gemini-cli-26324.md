## Context

Issue #19985: the interactive CLI hangs when wrapping long unbroken ghost-text words (the prompt-completion preview rendered after the cursor) for inputs like `@getskill.sh:3`. The pre-fix wrapping loop in `InputPrompt.tsx` sliced ghost-text words by terminal width, but if the available width was zero (terminal narrower than expected) or narrower than the next code point's display width (a fullwidth CJK char in a 1-column slot), the loop would repeatedly slice from index 0 and never advance — locking the render loop indefinitely.

## What's right

- **The new `splitWordForWidth(word, maxWidth)` helper at `InputPrompt.tsx:162-195`** is the right shape: it pulls the wrapping logic out of the inline render path into a named, exported, unit-testable function. Exporting it (rather than keeping it module-private) is what makes the test arms at `InputPrompt.test.tsx:484-500` possible without reaching into the component's render-time internals.
- **The two structural fixes are at the exact failure points.** First, `:163`: `const width = Number.isFinite(maxWidth) && maxWidth > 0 ? maxWidth : 1` clamps any non-positive or non-finite incoming width to `1`. This closes the "terminal width is zero or negative" case (e.g. terminal hasn't reported a size yet, or a math overflow upstream produced `NaN`). The `Number.isFinite` check is the right primitive — `Number.isNaN(maxWidth) || maxWidth <= 0 ? 1 : maxWidth` would also work but `Number.isFinite` is the canonical "is this a usable real number" predicate that catches `NaN`, `Infinity`, and `-Infinity` in one check.
- **Second, the inner-loop progress guarantee at `:175-180`.** The `if (partWidth + charWidth > width)` arm only fires when adding the *next* char would exceed the budget; the inner `if (partWidth === 0)` arm at `:176-179` is the load-bearing fix: it forces consumption of the wide character anyway (`part += char; splitIndex = i + 1;`) and breaks. This guarantees that *at least one code point* is consumed per outer-loop iteration, even when a single character is wider than the entire available width — making the outer `while (stringWidth(wordToProcess) > width)` loop structurally bounded by `wordToProcess.length` per iteration.
- **Use of `toCodePoints` and `cpSlice` instead of `.split('')` and `.slice()`** at `:172` and `:185` correctly handles surrogate pairs (emoji, supplementary-plane characters). A naive `word.split('')` would split a `🎉` into two broken UTF-16 code units, which the `stringWidth` lookup would then mis-classify and potentially render as garbage.
- **Test coverage at `InputPrompt.test.tsx:484-500` is the right two arms.** `splitWordForWidth('@getskill.sh:3', 0)` (the bug-report input verbatim, with the pathological `maxWidth=0`) asserts `lines.join('') === '@getskill.sh:3'` AND `lines.toEqual('@getskill.sh:3'.split(''))` — i.e. each char ends up on its own line because of the width-1 clamp. `splitWordForWidth('表x', 1)` asserts the wide-char-narrower-than-width case lands `['表', 'x']` — the wide CJK char gets its own line via the `partWidth === 0` forced-consume arm, and the narrow `'x'` follows. Both arms pin the structural fix exactly: the input-from-the-bug-report and the synthetic edge case that exercises the inner forced-consume branch.

## Risks / nits

- **The first test only asserts `lines.toEqual('@getskill.sh:3'.split(''))`** (each char on its own line). That's the *output* of the width-clamp + forced-consume path, but the test doesn't have a timeout/loop-bound assertion that would catch a regression where the function correctly produces the right output but takes O(n²) iterations doing it. A `splitWordForWidth(very_long_word, 0)` test with a `performance.now()` upper bound (say `<10ms` for a 10k-char input) would pin the structural property "this function is O(n) regardless of width" rather than just "this function eventually terminates". Not blocking — the structural argument is solid — but worth considering for hardening.
- **`Number.isFinite` and `> 0` conjunction handles all the bad-input cases except one: `maxWidth = 0.5`**, which is `> 0` and finite, would pass the clamp and then the inner loop would compare `partWidth + charWidth > 0.5`. For any printable char (`charWidth ≥ 1`), this would always trigger the forced-consume arm — fine, just slightly weird that the function doesn't quantize to integer first. A `Math.max(1, Math.floor(maxWidth))` or similar normalization would be cleaner. Not a real bug since the caller likely never passes fractional widths, but worth a comment explaining the clamping discipline.
- **The function is exported from `InputPrompt.tsx`** which is a React component file. Pure utility helpers in a component file are a known mild smell — they bundle with the component instead of living in a `utils/text.ts` or `utils/wrapping.ts` module that other components could share. If `splitWordForWidth` is genuinely component-local, fine; if any other render path needs the same wrapping behavior in the future, it should move.
- **No assertion on the existing ghost-text rendering integration**, just the new helper unit tests. The PR body says "the existing ghost-text rendering path now uses that helper" — a render-time test ("render `<InputPrompt value=... ghostText='@getskill.sh:3' terminalWidth={20} />` and assert it doesn't hang and renders the expected wrapped output") would pin the integration. The helper-level test pins the algorithm; an integration test would pin the wiring.
- **Bug-fix attribution**: the PR body says "Fixes #19985" — confirm the linked issue actually describes the `@getskill.sh:3` reproducer. If the issue is more general ("ghost text wraps weirdly" without the infinite-loop framing), the fix is still right but the closes-issue link should be tightened.

## Verdict

**merge-after-nits.** The structural fix is correct: clamp the width to ≥1 at the entry, force-consume at least one code point per outer iteration when the next char doesn't fit. Two well-chosen unit tests pin both fixes. The nits are about test scope (timing assertion, integration test) and code organization (extract to utils file, normalize width to int) that don't block the fix.
