## Context

Sister PR to #26321 (drip-237) which targeted the same sanitization concern via a different filename (`keychain-token-storage.ts`). This PR targets the `KeychainService` class in `keychainService.ts` and introduces:

1. A new `SecureStorageError` class (and a `SECURE_STORAGE_GENERIC_MESSAGE` constant) in `utils/errors.ts:+12`.
2. A wrapping `runSecureStorageOperation(...)` helper on `KeychainService` that catches every native-keychain error from `getPassword`/`setPassword`/`deletePassword`/`findCredentials` and re-throws as `SecureStorageError(SECURE_STORAGE_GENERIC_MESSAGE)`.
3. Sanitized debug logging (`debugLogger.debug('Keychain initialization failed')` instead of `debugLogger.debug(expect.stringContaining('encountered an error'), 'locked')` which leaked the raw native error string).
4. A new `expectSecureStorageFailure` test helper that asserts (a) the error is `instanceof SecureStorageError`, (b) `.message === SECURE_STORAGE_GENERIC_MESSAGE`, and (c) the message contains *none* of the sensitive fragments `account=acc1`, `/Users/secret`, `AccessDenied`.

## What's right

- **All four operations get the same wrapper.** `keychainService.ts:53-104`: `getPassword`, `setPassword`, `deletePassword`, `findCredentials` are each wrapped via `this.runSecureStorageOperation(async () => { const keychain = await this.getKeychainOrThrow(); return keychain.X(...) })`. This is the right shape — one chokepoint that owns "convert any thrown native error to a sanitized SecureStorageError" — rather than try/catch sprinkled at four call sites that could drift apart over time.
- **The test discipline at `keychainService.test.ts:330-407` is the headline.** The `sensitiveFragments = ['account=acc1', '/Users/secret', 'AccessDenied']` array (`:332-336`) is the *positive* discipline: a real-shaped sensitive native error message (`'AccessDenied account=acc1 /Users/secret'`) is constructed, fed to each of the four operations, and the `expectSecureStorageFailure` helper at `:336-355` asserts the error type AND that *none* of the three sensitive fragments leaked into `.message`. This pins the sanitization contract at the observable-behavior level rather than relying on "we wrapped it, trust me".
- **The four parallel test arms at `:376-405`** are symmetric: one per operation, each constructing the same sensitive native error, each calling `expectSecureStorageFailure`. The symmetry is what guarantees no operation drifted out of sanitization coverage — a future contributor adding a fifth operation will notice the missing test arm.
- **The initialization-path sanitization at `:153-167`** (the `should sanitize initialization logging when native keychain throws` test): asserts that `debugLogger.debug` is called with `'Keychain initialization failed'` (a fixed string) AND that no debug call's joined-args contains `'denied'`, `'/Users/alice'`, or `'login.keychain-db'`. Three independent fragments per call, asserted across all calls — the right shape for "no log line anywhere leaks the raw error".
- **The companion test update at `:155-167`** (the existing `'should return true (via fallback), log sanitized error...'` test) flips both the assertion shape (`debugLogger.debug` is called with the fixed message, not with `expect.stringContaining('encountered an error'), 'locked'`) AND the test name (`'log error'` → `'log sanitized error'`). The name change is the right discipline — the test description now matches the actual contract.
- **The structural-validation test at `:198-201`** (`'should return true (via fallback), log validation error, and emit telemetry on module load failure'`) drops the second `expect.objectContaining({ getPassword: expect.any(Array) })` argument from the `debugLogger.debug` assertion. Pre-fix that argument was exposing the *shape* of the broken keytar mock (`getPassword: undefined → Array of validation issues`) which is a structural-information leak — the post-fix assertion only checks the human-readable message string.

## Risks / nits

- **`runSecureStorageOperation` itself is not visible in the diff hunk.** The PR shows the four call sites that wrap with it, but the helper definition (presumably the `try { return await op(); } catch { throw new SecureStorageError(...); }` shape) isn't in the diff slice I have. If the helper *also* logs the original error via `debugLogger.debug(...)` for diagnostic purposes, that's the right discipline — but only if the log call is itself sanitized (just the fixed message string, no stringified original error). Worth pinning at the helper definition.
- **No `cause` chain test.** The standard pattern for "sanitized public message + diagnostic preservation" is `new SecureStorageError(GENERIC, { cause: originalError })` so a verbose-logger code path can opt in to the raw error for debugging without it leaking through default `.message` access. The test asserts that `.message === GENERIC_MESSAGE` (sanitization works for the default path) but doesn't assert anything about `.cause` — if `cause` is silently set, the four `sensitiveFragments` assertions are *not* in fact closing the leak (any caller that does `JSON.stringify(error)` or `String(error.cause)` would re-leak). A test arm asserting either `(error as Error).cause === undefined` (cause is dropped — strict sanitization) OR `(error.cause as Error).message === originalSensitiveString && JSON.stringify(error) does not include sensitiveFragments` (cause preserved but doesn't serialize through the public surface) would close the gap.
- **Three-fragment sensitivity list is good but not exhaustive.** Real native keychain errors often include OS-level paths (`~/Library/Keychains/login.keychain-db`), library-internal C-string error codes (`errSecAuthFailed`), or pointer-shaped strings in stack traces. The chosen fragments are representative but a `'errSecAuthFailed'` arm would also be valuable since that's the most common real macOS keychain rejection signature.
- **`expectSecureStorageFailure` helper structure** uses a `let thrown: unknown; try { await promise; } catch (error) { thrown = error; }` pattern at `:339-345` — fine, but `await expect(promise).rejects.toBeInstanceOf(SecureStorageError)` is the more idiomatic vitest pattern. The custom helper is preferred here because it lets the helper run *additional* assertions (the message check + the fragment loop) that `.rejects.toBeInstanceOf` can't compose. So the choice is correct, just worth a comment explaining why the more-idiomatic shorthand wasn't used.
- **PR title is "...restore test execution, and fix install/build issues"** but the diff slice I have only shows the keychain sanitization changes. If there are install/build fixes elsewhere in the diff (presumably in `package.json` or a build script), they should be reviewed separately — bundling unrelated changes makes bisection painful and increases the per-review surface area. Worth confirming the install/build piece is genuinely related (e.g. needed to make the keychain tests run) rather than just hitchhiking.
- **Sister PR coordination.** This PR (#26322) targets `keychainService.ts`; the prior #26321 (drip-237) targeted `keychain-token-storage.ts` with the same `SecureStorageError` pattern. The two PRs may be touching the same `utils/errors.ts` file and racing to add the same `SecureStorageError` class. Whichever lands second needs to either rebase or no-op the duplicate definition. PR-body cross-link to #26321 would surface the coordination question.

## Verdict

**merge-after-nits.** The sanitization contract is closed at the right layer (one wrapper, four call sites flow through it), tests assert the *observable* contract (sanitized message + no sensitive fragments leak) rather than internal state, and the parallel four-arm structure makes future drift visible. The main blocker for `merge-as-is` is the missing `cause` chain assertion — without it, the sanitization is one `JSON.stringify` away from re-leaking. Add the `cause`-chain test arm + clarify the `runSecureStorageOperation` helper's own logging discipline + cross-link sister PR #26321, then merge.
