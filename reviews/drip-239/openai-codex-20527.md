# openai/codex#20527 — Support updatedInput hook rewrites

- **PR**: https://github.com/openai/codex/pull/20527
- **Author**: abhinav-oai
- **Head SHA**: `bcccc6c208ec91e882de9b0a75f0ea88dc8a35dd`
- **Files**: 20 files, +1220 / -66 across `core/src/{function_tool,hook_runtime,mcp_tool_call,stream_events_utils}.rs`, `core/src/tools/{events,registry,orchestrator,sandboxing,network_approval}.rs`, `core/src/tools/handlers/{apply_patch,mcp,shell,unified_exec}.rs`, `core/tests/suite/{hooks,hooks_mcp}.rs`
- **Verdict**: **merge-after-nits**

## Context

The `PreToolUse` and `PermissionRequest` hook contracts both document an `updatedInput` field in their hookSpecificOutput — i.e. a hook is supposed to be able to say "allow this tool call but rewrite its arguments first" (e.g. a hook that allow-lists `git commit` but injects a `--no-verify` flag, or an MCP `create_entities` hook that swaps an entity name before the call goes to the server). The pre-fix runtime *parsed* `updatedInput` but then *rejected* it instead of applying the rewrite, so the documented surface was a no-op. This PR wires the rewrite through a per-tool dispatch seam.

## What's right

- **The `FunctionCallError::UpdatedInput(Value)` carrier** at `function_tool.rs:7-8` is the right structural choice for "this tool call should be retried with rewritten input" — it's an `Err` variant (so the normal-path tool execution short-circuits) but carries the rewritten `serde_json::Value` payload up the stack to the dispatch layer that knows how to retry. Encoding "retry with new input" as an error variant rather than a separate `Result` shape means it composes with all the existing `?` propagation without changing every signature.
- **`PreToolUseHookResult` enum** at `hook_runtime.rs:46-49` (`Continue { updated_input: Option<Value> }` | `Blocked(String)`) flips the prior `Option<String>` ("Some(reason) → blocked, None → continue") to a typed three-state result. The `Continue { updated_input: None }` arm is the no-rewrite happy path; `Continue { updated_input: Some(v) }` is the rewrite path; `Blocked(reason)` is the deny path. This makes the rewrite case structurally distinct from the no-op case at the type level, so a future caller can't accidentally collapse the three cases into a two-state boolean.
- **The deny-path is fail-closed.** `hook_runtime.rs:170-194`: when `should_block` is true, the match builds a `PreToolUseHookResult::Blocked(...)` with the same Bash-vs-other branching as before; the `block_reason: None` case (which was previously a no-op `None` return) is now `PreToolUseHookResult::Continue { updated_input: None }` — i.e. "blocked but no reason given" is treated as "not actually blocked". This preserves the pre-fix behavior for the malformed-deny case (defer to the next hook in the chain) while making the new "allow with updatedInput" case structurally first-class.
- **The per-tool `with_updated_hook_input` dispatch seam** is the right design. Each concrete tool that supports rewrites implements `ToolHandler::with_updated_hook_input(invocation, updated_input) -> Result<ToolInvocation, FunctionCallError>`, returning a rebuilt `ToolInvocation` from the hook-facing JSON shape. This lives at the *tool* level rather than at the dispatcher level, which is correct because each tool knows its own `ToolPayload` shape (e.g. `apply_patch`'s `command` field vs. MCP's `server`/`tool`/`raw_arguments` triple vs. shell's `command` array). Trying to do the rewrite generically at the dispatcher would require a schema-aware JSON-to-payload mapping that doesn't exist.
- **`apply_patch` rewrite at `apply_patch.rs:321-358`** correctly handles both `ToolPayload::Function { arguments }` (the JSON-tool-call shape, where `arguments` is a JSON string that needs parsing, mutating the `input` field, and re-serializing) and `ToolPayload::Custom { input }` (the custom-tool shape, where the patch is the literal payload). The fall-through `payload => payload` arm at `:354` is the right discipline for "if the payload type doesn't have an obvious slot for the rewritten command, leave it alone" — though it would be more correct to *fail* in that case (see nits).
- **MCP rewrite at `mcp.rs:37-56`** is a simpler shape: the rewritten input *is* the new `raw_arguments` (serialized as JSON), so the rewrite is just "replace `raw_arguments` with `serde_json::to_string(&updated_input)`". No field-level merging, which matches the MCP semantic (the entire arguments object is what the tool consumes).
- **Re-running approval after `PermissionRequest` rewrites** is the correct safety property. A hook that rewrites a `git push` to a `git push --force` shouldn't bypass the *new* approval that `--force` would have triggered — re-running approval evaluation closes that loophole.
- **Bounded rewrite loop with no-op fast-path.** The "bounded rewrite loop and a no-op fast path once the hook returns the same input again" (PR body) is the right termination strategy: a hook that always rewrites the input to the same value would otherwise loop forever; the fast-path means "if the hook is idempotent on already-rewritten input, exit after one round" which is the common case.
- **Test coverage is the headline.** `hooks.rs` gets +438 lines and `hooks_mcp.rs` gets +114 lines. The MCP test at `mcp_tool_call_tests.rs:2072-2143` (`permission_request_hook_can_update_mcp_tool_input`) is the right kind of end-to-end pin: it installs a `PermissionRequest` hook for `mcp__memory__.*` that returns `behavior: "allow"` + `updatedInput: {entities: [{name: "Grace", entityType: "person"}]}`, calls `maybe_request_mcp_tool_approval` with input `{name: "Ada"}`, and asserts the decision is `McpToolApprovalDecision::AcceptWithUpdatedInput({name: "Grace"})` — not just "the hook ran" but "the rewritten input survived round-trip and is structurally accessible".
- **The `UpdatedInput` escape-hatch detection at `stream_events_utils.rs:339-343`** turns the "updated tool input escaped dispatch" case into a `CodexErr::Fatal` — i.e. if the dispatch layer ever fails to handle an `UpdatedInput` error variant (e.g. because a new tool type was added without a `with_updated_hook_input` implementation), the runtime fails loudly rather than silently dropping the rewrite. This is the right discipline for a state-machine where a missed transition should be a process-level failure, not a silent no-op.

## Risks / nits

- **The `ToolPayload` fall-through arm in `apply_patch.rs:354` (`payload => payload`)** silently passes through unhandled payload types without applying the rewrite. If a future contributor adds a new `ToolPayload` variant that *should* have a rewrite slot but forgets to add an arm here, the rewrite silently no-ops — which is the same bug class as the original (`updatedInput` accepted but ignored). Worth turning this into a `payload => return Err(FunctionCallError::RespondToModel(format!("apply_patch hook rewrite not supported for payload type {payload:?}")))` so a missed arm is a fail-loud signal at runtime.
- **The 1808-line `mcp_tool_call.rs` match** (visible in the diff at the `Some(PermissionRequestDecision::Accept ...)` block around `:1768-1810`) gains a new `McpToolApprovalDecision::AcceptWithUpdatedInput(_)` arm in the trailing exhaustive-match `| McpToolApprovalDecision::Accept | McpToolApprovalDecision::AcceptWithUpdatedInput(_) | McpToolApprovalDecision::Decline { .. } | McpToolApprovalDecision::Cancel | McpToolApprovalDecision::BlockedBySafetyMonitor(_) => {}`. This is correct (the `{}` empty arm matches the prior `Accept` semantics for these branches), but a comment explaining "AcceptWithUpdatedInput is structurally Accept-equivalent at this point because the rewrite has already been applied upstream at the registry layer" would help the next reviewer understand why the new variant gets the same `{}` body as `Accept`.
- **The bounded-rewrite-loop bound itself is invisible from the diff.** PR body says "bounded rewrite loop and a no-op fast path once the hook returns the same input again", but a hard ceiling (`MAX_HOOK_REWRITES = 8`?) on the rewrite count is the right structural protection against a non-idempotent hook that always returns *slightly different* input each round. Worth surfacing the bound in the PR body or a constant comment so it's reviewable.
- **`UpdatedInput(serde_json::Value)` as an `Err` variant** is a structural choice that costs *informativeness* on the unhappy path: a tool that genuinely fails plus a hook that rewrites the input both flow through the same `Err(FunctionCallError::...)` channel, distinguished only by variant tag. A `Result<ToolOutput, FunctionCallError>` extended to `Result<ToolOutput, ToolFailure>` where `ToolFailure` is a parallel enum (`{Retry(Value), Reject(String), Fatal(String)}`) would split the "this is success-with-redirect" case from the "this is failure" case at the type level. Not blocking — the current shape works — but worth considering for a follow-up if more retry-shaped variants accumulate.
- **438 lines of `hooks.rs` test additions plus 114 of `hooks_mcp.rs` is good coverage but doesn't include a test for the bounded-loop bound itself.** A `permission_request_hook_loops_until_bound_exceeded` test that installs a hook always returning a different `updatedInput` and asserts the loop terminates with a clear error after N rounds would pin the structural protection.

## Verdict

**merge-after-nits.** This closes a real "documented surface is a no-op" bug at the right structural layer (per-tool rewrite seam + typed three-state hook result), with the correct safety discipline (re-run approval after `PermissionRequest` rewrites, fail-closed deny path, bounded loop). The nits are: turn the `apply_patch` fall-through into a fail-loud, surface the loop bound as a constant comment, add a bound-exceeded regression test, and add a comment explaining why `AcceptWithUpdatedInput` shares the `Accept`-equivalent `{}` arm in the trailing match.
