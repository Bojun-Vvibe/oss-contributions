## What it does

Refs #3760. Every "side-query" one-shot LLM call in the codebase
(session-title generation, tool-use recap, `/rename`, `/summary`,
follow-up suggestions, ACP message rewrite, arena per-agent approach
summaries, chat compression, web-fetch summarization, insight
analysis, subagent spec generation) used to call
`config.getContentGenerator().generateContent({...})` directly, each
rebuilding its own model selection, abort signal, prompt-id, and
`thinkingConfig`. Several were silently sending `enable_thinking:
true` for workloads with no use for chain-of-thought (60-token
labels, kebab-case titles).

This PR funnels them all through `runSideQuery(config, options)` in
`packages/core/src/utils/sideQuery.ts`, which now:

- Has a text-mode overload returning `{ text, usage }` (in addition
  to the pre-existing JSON-mode path used by the structured-output
  callers).
- Sets `fastModel` as the default when the caller doesn't override.
- Disables thinking by default (`thinkingConfig.includeThoughts:
  false`) and strips any `thought`-marked parts from the response on
  the way out.
- Calls a new `BaseLlmClient.generateText` primitive in
  `packages/core/src/core/baseLlmClient.ts` that goes straight to the
  underlying `ContentGenerator` — bypassing the user-memory wrapping
  and main-prompt fallback that `geminiClient.generateContent`
  applies on the main turn (which is wrong for side queries: a
  recap of a turn shouldn't get the user's memory injected).

Six call sites explicitly override the new defaults because they
genuinely need different policy:

- `LlmRewriter.ts` (ACP rewrite) keeps custom temperature
  `0.3` / `maxOutputTokens: 1024`.
- `/summary` and chat compression pin to the main model because
  cache locality matters.
- Web-fetch summarization pins to main because summarization quality
  matters.
- Subagent spec gen and follow-up-suggestion direct path likewise
  override.

The `LlmRewriter` test at
`packages/cli/src/acp-integration/session/rewrite/LlmRewriter.test.ts:11-44`
is rewritten so the `@qwen-code/qwen-code-core` mock now stubs
`runSideQuery` itself, which forwards into `mockGenerateContent` —
so existing call-arg assertions keep working without each test
having to know that the chokepoint exists.

The most consequential semantic shift is in arena: pre-PR, each
agent's approach summary was generated by *that agent's own
model* via `InProcessBackend.getAgentContentGenerator`. Post-PR, all
arena summaries route through the chokepoint against `fastModel`,
removing both `InProcessBackend.getAgentContentGenerator` and
`ArenaManager.getAgentSummaryGenerator`.

A bug-fix worth calling out separately: `/summary`'s text extractor
did not previously strip `thought` parts, so on a thinking model the
saved `.qwen/PROJECT_SUMMARY.md` could contain `reasoning_content`.
The chokepoint strips thoughts on the way out, and the request now
goes out with thinking off — closing the leak two ways.

## What works

The "single chokepoint with grep-able per-site overrides" shape is
the right pattern here. The pre-PR state had ~12 call sites each
making independent decisions about model/thinking/abort, and the
silent `enable_thinking: true` on workloads-that-don't-need-thinking
was an exact instance of the cost: nobody set it, but they all
inherited it from the default `geminiClient.generateContent` path.
Centralizing the policy makes the floor explicit (`fastModel` +
thinking-off) and the deviations grep-able (each overriding site has
an explicit `model:` or `thinkingConfig:` in its options object).

The new `BaseLlmClient.generateText` primitive is the right
abstraction layer: side queries don't want the user-memory wrapping
or the prompt-template fallback that the main turn gets, and
yanking that policy up to the dedicated primitive means side queries
can never accidentally pick it up again.

The `LlmRewriter` test rewrite is particularly clean — by stubbing
`runSideQuery` to forward into the existing `mockGenerateContent`
spy, the test assertions on call args keep working unchanged. That's
the right way to migrate a test through a chokepoint refactor:
forward the spy, don't move it.

The `/summary` thought-stripping fix is a real bug being closed in
passing — the kind of thing that should arguably be its own PR but
is genuinely entangled with the chokepoint move (the strip happens
in the chokepoint).

## Concerns

1. **Default-model regression is silent.** The PR description
   acknowledges the risk: a future caller that inherits the default
   without thinking gets `fastModel` when they need `main`, and
   output quality silently drops. The cure ("grep-able per-site
   overrides") works for the six current divergent sites but
   relies on every future contributor reading
   `runSideQuery`'s docstring before calling it. A
   `model: 'fast' | 'main'` required field with no default would be
   marginally safer, at the cost of one more line per call site.
2. **Arena per-agent summary semantic shift is buried.** The PR
   description correctly flags this, but the `ArenaManager.test.ts`
   diff doesn't (as far as the visible chunk shows) add an
   end-to-end assertion that all per-agent summaries now share a
   model. A test like "run two-agent arena, assert both summaries
   went through `fastModel`" would lock in the new semantics
   against drift back to per-agent generators.
3. **Test for `runSideQuery` itself.** The chokepoint is now load-
   bearing for ~12 call sites and has its own default policy
   (`fastModel`, thinking-off, thought-strip) that's not covered
   by a focused unit test in the visible diff — only the call
   sites' tests exercise it transitively. A direct test of
   `resolveDefaultModel`, `applyThinkingDefault`, and the
   thought-strip would harden the floor.
4. **`ContentGenerator` API surface narrows silently.** Removing
   `InProcessBackend.getAgentContentGenerator` and
   `ArenaManager.getAgentSummaryGenerator` is correct internally,
   but the PR description says "no external consumers" — worth
   verifying that with a `grep -r getAgentContentGenerator
   .qwen/extensions/` or similar against the published extension
   surface, since extensions aren't in-tree.
5. **`signal ?? new AbortController().signal` at LlmRewriter.ts.**
   The fallback creates a fresh `AbortController` whose `.signal`
   is never aborted — which means a caller that passes `undefined`
   gets a never-cancelled side query. Pre-PR the
   `contentGenerator.generateContent` path would inherit the
   global session abort; this PR's fallback explicitly opts out of
   that. Probably benign but worth confirming the
   global-session-abort wiring still applies upstream of
   `runSideQuery`.

Verdict: merge-after-nits

## What I learned

When a codebase has 12 independent call sites making the same kind
of one-shot LLM call, the bug isn't usually the call sites — it's
the *defaults each site ends up with* because they all inherited
them from the main turn's path. The fix is rarely "tighten the
defaults at the main path" (that breaks the main turn) and almost
always "introduce a side-query primitive with its own opinionated
defaults, and either migrate the call sites or let them stay."
This PR does the latter migration, which is more work but yields
the grep-ability that's the actual deliverable: a year from now,
"which side queries use thinking?" is `grep -r 'thinkingConfig' src/`
and you find the deliberate three, not the accidental nine.
