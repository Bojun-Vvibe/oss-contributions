# BerriAI/litellm#27266 — fix(responses): handle response.incomplete streaming event in Responses->Chat transform

- **Head SHA**: `40bd8d5f31b9282ecfb7587a44dced7f9c59e27b`
- **Stats**: +51 / -0, 1 file (`litellm/completion_extras/litellm_responses_transformation/transformation.py`)

## Summary

OpenAI's Responses API emits a terminal `response.incomplete` event when generation halts early due to `max_output_tokens` being reached or a content filter triggering. The Responses → Chat-Completions translation layer in litellm did not have a branch for this event type, so it fell through to the trailing `else: pass` and was silently discarded — downstream callbacks, guardrails, and finish-reason-aware UIs never saw the terminal chunk and treated the stream as having stopped without a finish reason. The fix adds an explicit `elif event_type == "response.incomplete":` arm that maps `incomplete_details.reason` to the corresponding Chat-Completions `finish_reason`, surfaces `incomplete_details` and `content_filters` via `provider_specific_fields`, and translates `response.usage` if present.

## Specific citations

- `litellm/completion_extras/litellm_responses_transformation/transformation.py:1358-1361`: enters the new branch and pulls `response_data = parsed_chunk.get("response", {})` then `incomplete_details = response_data.get("incomplete_details") if response_data else None` then `reason = incomplete_details.get("reason") if incomplete_details else None`. The triple-guarded chain is defensive but correct — the `if response_data else None` is technically redundant after `.get("response", {})` (which never returns None for the truthy default), but the symmetry with the `incomplete_details` guard is fine.
- `:1366-1372`: the reason → finish_reason mapping. `max_output_tokens` → `length` (correct per OpenAI Chat Completions spec — `length` is the canonical finish reason for token-cap truncation), `content_filter` → `content_filter` (literal pass-through, also canonical), and the catch-all `else: finish_reason = "stop"`. The catch-all defaulting to `"stop"` is debatable — when an unknown future `reason` value lands, defaulting to `"stop"` makes the stream look like a clean completion when it actually didn't complete; a `verbose_logger.warning(...)` on the unknown-reason branch would let consumers find new reason values without rolling out a fix-by-fix cycle.
- `:1376-1383`: surfaces both `incomplete_details` and `content_filters` (note plural — OpenAI's Responses API returns an array of per-output-index filter triggers) via `provider_specific["incomplete_details"]` / `provider_specific["content_filters"]`. The `if content_filters:` guard correctly handles the common case where only `incomplete_details` is present (max_tokens path); the final `provider_specific_fields=provider_specific if provider_specific else None` at `:1408` guards against emitting an empty dict that would break downstream `if chunk.provider_specific_fields:` checks.
- `:1385-1391`: usage translation lazy-imports `from litellm.responses.utils import ResponseAPILoggingUtils` and calls `_transform_response_api_usage_to_chat_usage(...)`. The lazy import is the correct pattern here — top-level import would create a circular dep (the responses utils module imports from this transformation file). Worth a one-line `# lazy import to avoid circular dependency` comment.
- `:1393-1407`: the synthesized `ModelResponseStream` has `delta=Delta(content="")` paired with the resolved `finish_reason`. Empty-content + finish_reason is the canonical OpenAI Chat Completions terminal-chunk shape (the spec allows finish_reason on a chunk with no content delta), so any consumer respecting the wire format will treat this as a clean termination.

## Verdict

**merge-after-nits**

## Rationale

Correct fix for a real protocol-shape gap that silently dropped early-termination signals — the impact is concrete (Azure OpenAI's content-filter and max-output-tokens paths were invisible to litellm's logger/guardrail callbacks). The reason → finish_reason mapping matches OpenAI's documented terminal semantics, the `provider_specific_fields` channel is the right surface for the structured `incomplete_details` / `content_filters` payloads (Chat Completions has no first-class field for them), and the `Delta(content="")` + `finish_reason` shape is wire-compliant. Three nits worth raising: (1) the catch-all `else: finish_reason = "stop"` at `:1372` silently masks unknown future reason values — a `verbose_logger.warning(f"Unknown response.incomplete reason: {reason!r}, defaulting to stop")` would surface new OpenAI reason variants for the next maintenance pass; (2) **no test coverage** — given that drip-384's #27269 (the sibling `function_call_arguments.done` fix) added three tests, the symmetric expectation here is at least one test per code path: `max_output_tokens` → `length`, `content_filter` → `content_filter`, unknown-reason → `stop`, and a `usage` round-trip case. Without tests this regresses the moment someone refactors the dispatch chain; (3) the lazy `ResponseAPILoggingUtils` import deserves a one-line comment naming the circular-dep reason. None block merge — the implementation is correct, the surface is right, and the gap is real — but the test gap is the strongest "after-nits" signal.
