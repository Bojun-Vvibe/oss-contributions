# QwenLM/qwen-code#3861 — fix(cli): preserve comments and formatting in settings.json during migration write-back

- **Head SHA**: `be9ce3c573562b3a3a96855e594ea943d9d5389c`
- **Stats**: +440 / -23, 5 files (Fixes #3843)

## Summary

`persistSettingsObject()` in `loadAndMigrate()` was using `JSON.stringify(settingsObject, null, 2)` to write back migrated settings, which destroys all comments, trailing commas, and custom formatting in `settings.json`. The PR swaps the serializer to `comment-json`'s `stringify()` so format-preserving writes work end-to-end (parse → migrate → stringify keeps the AST's comment/trailing-comma metadata intact). The bulk of the diff is actually unrelated: ~250 lines of new `mcp-client-manager.ts` concurrency hardening (in-flight discovery promise coalescing) — a separate logical change that should arguably be split.

## Specific citations

- `packages/cli/src/config/settings.ts:36`: adds `import { stringify as commentJsonStringify } from 'comment-json'`. The aliased import is correct given Node's `JSON.stringify` is in-scope.
- `packages/cli/src/config/settings.ts:735-741`: the load-bearing change in `persistSettingsObject`. `JSON.stringify(settingsObject, null, 2)` → `commentJsonStringify(settingsObject, null, 2)`. The behavior depends on whether `settingsObject` is the *parsed* `comment-json` AST (carries comment metadata as non-enumerable properties on the parsed value) or a plain JS object (no metadata). Since this function is called inside `loadAndMigrate()` on the post-migration object, and migrations typically `setNestedPropertySafe` mutate the parsed AST in-place, comments on *unchanged* keys should survive — but comments on keys that migrations remove or replace via reassignment will not. The PR comment "deprecated keys removed by migration don't persist as zombies" implicitly acknowledges this — the contract is "preserve comments on keys that survive, drop comments on keys that migrations renamed/removed."
- `commentJson.test.ts:189-296`: three new tests pin the contract — `should preserve comments on keys that exist in both original and updates`, `should add new keys while preserving existing comments`, `should preserve inline comments and trailing commas`. The trailing-comma test at `:269` fixture has both `"theme": "dark",` (trailing) and `},` (trailing) which are JSON5/JSONC-style — the assertion `expect(result).toContain('// inline comment')` proves inline comments survive but doesn't assert the trailing commas survive (only that values were updated). A dedicated `expect(result).toMatch(/"theme": "light",\s*$/m)` would harden this.
- `settings.test.ts:633-647,738-757,850,882,915`: updates 6 existing migration tests because `writeWithBackupSync` now writes to a `.tmp` path then renames (the `.find((call) => call[0] === ${USER_SETTINGS_PATH}.tmp)` filter at multiple sites). These were already updated in a prior PR for the rename-pattern; the new PR adds `expect(writeCall).toBeDefined()` + early-throw to make the test failure mode explicit when the temp write doesn't fire.
- `mcp-client-manager.ts:61,156-225,283,540,541` (the unrelated bulk): adds `inFlightDiscoveries: Map<string, Promise<void>>` field, gates `discoverMcpToolsForServer` on existing in-flight promise (coalesce-don't-spawn), tracks the in-flight promise *before* any `await` to close the TOCTOU race window during `disconnect()`, disconnects the failed client in the catch block before deleting it (closes orphaned-process gap), and clears the in-flight tracking in `stop()` and `removeServer()`. Five new tests at `mcp-client-manager.test.ts:227-460` cover (a) concurrent discovery coalescing, (b) post-completion cleanup, (c) re-discovery after `stop()`, (d) cleanup when removing a server via incremental discovery, (e) failed-client disconnect on connect error. The concurrency hardening is genuinely good code but is a *separate concern* from settings.json comment preservation.

## Verdict

**merge-after-nits**

## Rationale

Two solid changes correctly implemented but bundled into one PR — recommend split before merge. The `comment-json` swap in `persistSettingsObject` is the headline fix and is exactly right: `comment-json`'s `stringify` round-trips the comment metadata that `comment-json`'s `parse` captured, so any settings file that was loaded via the existing comment-aware loader will now preserve its comments through migration write-back. The three new test cases cover the meaningful axes (preserve-on-overlap, add-new-keys, inline + trailing commas) and the `writeWithBackupSync` test updates make the temp-file pattern explicit. The MCP discovery concurrency rework is also genuinely good — the in-flight promise coalesce-don't-spawn pattern correctly closes a real race (TOCTOU during disconnect when two `discoverMcpToolsForServer` calls land concurrently), the failed-client-disconnect-before-delete addresses the orphaned-child-process gap from prior failure modes, and the test coverage is thorough (5 new tests covering coalesce, cleanup, post-stop, incremental-removal, failed-connect-cleanup). Three nits/concerns: (1) **the two changes should be separate PRs** — the title and body talk only about settings.json comment preservation, but ~60% of the diff is MCP concurrency. A reviewer skimming the title would miss the concurrency hardening, and a future bisect of an MCP regression would find this PR via a misleading title; (2) the trailing-comma test at `commentJson.test.ts:269` asserts inline comment survival but not trailing-comma survival — recommend adding `expect(result).toMatch(/,\s*}\s*$/m)` to lock the contract; (3) the contract "comments on keys that migrations removed don't survive" should be documented at the `persistSettingsObject` callsite — the inline `// deprecated keys removed by migration don't persist as zombies` comment is good but doesn't address the comment-on-removed-key case explicitly. None block merge if the maintainer accepts the bundled scope; recommend split.
