## Summary

Resolves the `case 'openIdConnect': // TODO: Implement` stub in the A2A auth-provider factory by adding a real `OpenIdConnectAuthProvider` that performs OIDC Discovery (`.well-known/openid-configuration`), validates HTTPS on both the issuer URL and discovered endpoints, runs the standard PKCE-backed authorization-code flow via the existing `oauth-flow.ts` helpers, and persists tokens in `MCPOAuthTokenStorage`. The provider extends `BaseA2AAuthProvider` and is wired into `A2AAuthProviderFactory.create` via dynamic `await import(...)`.

- `openIdConnect-provider.ts:58-61`: HTTPS enforcement on the issuer URL (`if (!config.issuer_url.startsWith('https://')) throw`) before any network call. This is the correct ordering — fail before fetching — and the symmetric check at `:79-84` validates that the *discovered* `authorization_endpoint` and `token_endpoint` are also HTTPS, preventing a MITM that returns an HTTPS issuer document with HTTP downgrade endpoints. Strong defaults.
- `:23-26`: the `OidcDiscoverySchema = z.object({ authorization_endpoint: z.string().url(), token_endpoint: z.string().url() })` — only requires the two endpoints actually used. The PR title says "OIDC" but `jwks_uri` (which would be needed for ID token signature verification) is not in the schema. This is fine for an auth-only provider that uses the access_token as a Bearer credential and doesn't actually verify the ID token, but **the PR is mis-titled if no ID-token validation is happening** — calling this "OAuth2 with discovery" would be more accurate. The test fixture at `openIdConnect-provider.test.ts:35` includes `jwks_uri` in the mock response but the provider never reads it, so the test is over-specified.
- `:69-77`: `headers()` uses `cachedToken` if `!storage.isTokenExpired(...)`, otherwise calls `authenticateInteractively()`. **No refresh-token branch** — if the cached token is expired but a `refreshToken` was returned by the token endpoint (line 116, captured into the stored token), the code unconditionally re-runs the interactive browser-launch + PKCE flow rather than first attempting a `grant_type=refresh_token` exchange. For a CLI tool this is a real UX regression for any user with a longer-than-token-lifetime session. Should at minimum be a TODO, ideally a follow-up issue link in the PR body.
- `:97-127`: `authenticateInteractively()` correctly chains `getConsentForOauth(agentName)` → `generatePKCEParams()` → `startCallbackServer(pkce.state)` → `buildAuthorizationUrl(...)` with `scopes: ['openid', ...(this.config.scopes || [])]` — prepending `openid` is correct (required scope per OIDC spec) and the spread preserves user-configured additional scopes. `openBrowserSecurely(authUrl)` uses the existing secure launcher (good — matches the pattern in the MCP OAuth flow). The `await port` is awaited *twice* (`:107` and `:122`) — `startCallbackServer` returns `{ port: Promise<number>, response: ... }` so awaiting twice on the same promise resolves to the same value (no double-listen), but a `const callbackPort = await port` hoisted once would be clearer.

## Verdict

**merge-after-nits**

## Rationale

Solid implementation of a meaningful gap — enables real enterprise OIDC flows (Okta, Auth0, Azure AD, internal corp IdPs) for A2A remote agents instead of forcing API-key-only setups. The HTTPS double-validation (issuer + discovered endpoints) is the right hardening, the PKCE flow reuses existing battle-tested helpers, and the static `create()` factory pattern correctly defers construction until after async discovery completes (preventing partial-init resource leaks). Three nits, one of which approaches block-worthy: (1) **no refresh-token branch** in `headers()` — every token expiry triggers an interactive browser relaunch, which is a major UX regression for long-running CLI sessions; recommend adding a `tryRefreshToken()` branch before falling through to `authenticateInteractively()`, even if the initial implementation just logs "refresh failed, falling back to interactive" on any error; (2) the PR is titled "OpenID Connect" but the implementation only treats the response as an OAuth2 Bearer token — no ID-token signature verification via `jwks_uri`, no ID-token claims extraction. This is a defensible scope cut, but the title should either be relaxed to "OAuth2 with OIDC discovery" or a follow-up issue should be opened to add ID-token verification, since OIDC's value proposition over plain OAuth2 is the verified identity claims; (3) `agentName ?? 'unknown'` in the factory means multiple anonymous agents share a credential slot — a `crypto.randomUUID()` fallback or a hard error on missing agentName would be safer. The `should discover endpoints` test should also have a positive-cache-hit case so `initializeFromCache()` is exercised. None block merge if the maintainer accepts the refresh-token gap as a follow-up.
