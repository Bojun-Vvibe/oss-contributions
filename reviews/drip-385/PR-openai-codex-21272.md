# openai/codex#21272 — Support compact SessionStart hooks

- **Head SHA**: `5760bc2545c4b7b8bc66605d9dbc5aabfefd837f`
- **Stats**: +340 / -31, 8 files

## Summary

Compaction replaces the live conversation history with a summary, which silently invalidates whatever durable model context a `SessionStart` hook had injected on `startup`/`resume`. This PR adds `compact` as a fourth `SessionStartSource` so hooks can re-run after the rewrite to restore that context, and converts the pending-source slot from a single `Option<...>` to a `VecDeque` so a `resume`-then-`compact` sequence (resumed thread that immediately auto-compacts because token-count exceeds limit) preserves both events instead of clobbering the first.

## Verdict

**merge-after-nits**

## Rationale

Solves a real semantic gap (compaction wipes hook-injected context) with the architecturally correct fix (queue, not slot, since multiple lifecycle events can stack between hook drains). The schema regeneration is included, the enum string mapping is consistent, and the test coverage is unusually thorough — both single-source and multi-source-stacked cases are pinned, and the second test is the kind of integration test that would catch any future LIFO/FIFO regression. Three nits: (1) the `while let Some(...)` loop in `run_pending_session_start_hooks` early-returns `true` on the first non-empty additional-context, but the subsequent queued sources stay in `pending_session_start_sources` for the *next* turn — this is documented behavior but worth a code comment naming "drain semantics" so a future reader doesn't 'fix' it into an unconditional drain that breaks the rollout-injection contract; (2) the `non_openai_model_provider` helper at `hooks.rs:80-87` introduces a stutter `built_in_model_providers(/* openai_base_url */ /*openai_base_url*/ None)` — the duplicate doc comment is a copy-paste artifact, drop one; (3) `VecDeque::new()` allocates lazily so unbounded growth in pathological repeat-compact scenarios (which can't happen — compaction generates one queue entry per occurrence and drains them) isn't a real concern, but a `VecDeque::with_capacity(2)` would make the typical `[startup, compact]` / `[resume, compact]` case zero-realloc. None block merge.
