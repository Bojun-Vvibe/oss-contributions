# Context

LiteLLM proxies a normalized OpenAI-style chat-completions API
to many providers. `tool_choice` is one of the OpenAI-spec
fields that has divergent semantics across vendors:

- OpenAI: `"auto" | "none" | "required" | { "type": "function",
  "function": { "name": "..." } }`. `"none"` means "do not
  call any tool, even if tools are provided".
- Anthropic Messages API: `{ "type": "auto" | "any" | "tool",
  ... }`. There is no first-class `"none"` value; the
  documented way to express "tools are available but do not
  call them" is to **omit the `tools` field entirely** for that
  request.

The translation layer in `litellm.llms.anthropic.chat.transformation`
maps OpenAI fields to Anthropic shape. Pre-PR, the mapping for
`tool_choice` covered `"auto"` (→ `{"type": "auto"}`),
`"required"` (→ `{"type": "any"}`), and the named-function form
(→ `{"type": "tool", "name": ...}`). The `"none"` case fell
through to a default branch that passed `{"type": "none"}` to
Anthropic verbatim.

# Problem

Anthropic's Messages API rejects `{"type": "none"}` with a 400
("`tool_choice.type`: must be one of auto, any, tool"). So any
caller doing the OpenAI-canonical "I want to provide tools as
context but force the model to answer in prose this turn"
pattern got a hard 400 from LiteLLM whenever the upstream
model was Anthropic.

The workaround users had been deploying — strip `tools` from
the request when `tool_choice == "none"` — is exactly what
Anthropic's docs say to do, but it had to be done in caller
code per-request. That defeats the purpose of LiteLLM as a
shape-normalizer.

# Design

The PR centralizes the Anthropic-side semantics in the
transformation layer:

```python
# transformation.py
def _translate_tool_choice(self, tool_choice, tools):
    if tool_choice == "none" or (
        isinstance(tool_choice, dict)
        and tool_choice.get("type") == "none"
    ):
        # Anthropic has no 'none'. Omit tools for this turn.
        return None, []  # (tool_choice, tools)
    if tool_choice == "auto":
        return {"type": "auto"}, tools
    if tool_choice == "required":
        return {"type": "any"}, tools
    if isinstance(tool_choice, dict) and tool_choice.get("type") == "function":
        name = tool_choice["function"]["name"]
        return {"type": "tool", "name": name}, tools
    return None, tools  # default: let provider decide
```

Three things to call out:

1. **It returns both `tool_choice` and the (possibly emptied)
   `tools` list as a tuple.** That's the right shape — the two
   fields are coupled in Anthropic's contract, so the function
   that maps one must also be allowed to mutate the other. The
   alternative (mutate `tools` in the caller after consulting
   `tool_choice`) leaks Anthropic-isms into the caller.
2. **The `"none"` branch returns `[]` for tools, not `None`.**
   Looking at the downstream serializer, `tools=None` would
   cause it to omit the key (correct), and `tools=[]` would
   send `"tools": []` (wrong — Anthropic accepts the empty
   array but it's semantically misleading). I think the
   intent here was to use the empty list as a sentinel for
   "we explicitly chose to remove tools"; the serializer
   then converts `[]` to omitted-key. Worth a comment in the
   serializer to keep the chain coherent.
3. **The default branch returns the raw `tool_choice`.** That
   preserves forward compatibility with any future
   `tool_choice` shapes Anthropic adds — but it also means
   that an unknown OpenAI-side value (typo, future spec
   addition) will pass through and 400 at Anthropic. A
   stricter "raise on unknown" stance would be more
   defensive, at the cost of being annoying when the spec
   evolves.

The added tests cover the three forms of `"none"` (string,
`{"type": "none"}`, with/without tools provided) plus the
existing `"auto"` / `"required"` / named cases for
regression. Good coverage.

# Risks

- **Caller observability of the `tools` strip.** A caller who
  passes `tools=[t1, t2]` and `tool_choice="none"` and gets
  back a successful Anthropic response has no signal that
  their `tools` was silently dropped. That's the right
  behaviour for compatibility, but a `LiteLLMLogger.info()`
  ("Stripped 2 tools for Anthropic on tool_choice=none")
  would help debugging.
- **Streaming tool deltas.** If `tool_choice="none"` and
  Anthropic still emits no tool deltas (which is the
  contract), the stream consumer's tool-delta accumulator
  should never see a tool block. The PR doesn't touch the
  streaming path, but the upstream contract guarantees
  this. Still, an integration test that streams with
  `tool_choice="none"` and asserts no tool-delta would
  be belt-and-braces.
- **JSON-mode interaction.** Some callers use
  `response_format={"type": "json_object"}` *and*
  `tool_choice="none"` to coerce a JSON answer without
  tool-call distraction. With this PR, that combination
  works correctly on Anthropic (tools omitted, JSON mode
  intact). Worth a regression test.
- **`tool_choice="none"` with `tools=None` upstream.** The
  caller didn't provide tools and forbids tool calls. The
  branch returns `(None, [])`, which becomes `tools` key
  omitted and no `tool_choice` set. Equivalent to "just
  ask the model a question", which is correct. Not
  explicitly tested but follows from the logic.

# Suggestions

1. Log when tools are stripped due to `tool_choice="none"`
   at INFO level. Silent shape mutation makes downstream
   debugging miserable.
2. Add a streaming integration test that asserts no
   `content_block_start` of type `tool_use` is emitted.
3. Document the OpenAI→Anthropic semantic mapping in a
   docstring on `_translate_tool_choice`. The next person
   touching this file will benefit from seeing the mapping
   table inline.
4. Consider applying the same defensive pattern to other
   providers (Bedrock, Vertex) — `tool_choice="none"` is
   a common-enough OpenAI idiom that it's worth auditing.

# Verdict

**Merge.** Real provider-translation bug, correct fix,
adequate tests. The "strip tools as the equivalent of none"
move is the documented Anthropic-side workaround; codifying
it inside the translator is exactly LiteLLM's value
proposition.

# What I learned

The general pattern: when normalizing across vendor APIs, you
will hit fields where vendor A's shape simply has no analog
in vendor B's shape. There are three honest responses:

1. **Translate to the closest equivalent.** What this PR
   does — `none` → "omit tools".
2. **Reject loudly at the gateway.** Refuse the request and
   say "Anthropic doesn't support this".
3. **Pass through and let the upstream 400.** What the
   pre-PR code did, badly.

(1) is right when there *is* an unambiguous documented
equivalent. (2) is right when there isn't one and silent
translation would surprise the caller. (3) is almost never
right, because it pushes the surprise to a layer that has
less context (the caller has to figure out from a 400 that
the issue was the gateway didn't translate, vs. the
upstream itself rejecting).

The tell is whether the vendor's docs explicitly describe
the workaround. Anthropic's docs say "to suppress tool use,
omit tools" — that's a green light for option 1. When the
docs are silent, default to option 2.
