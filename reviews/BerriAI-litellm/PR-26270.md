# Context

litellm's proxy aggregates per-request spend in six in-memory queues
(user, team, end-user, org, key, agent), then a periodic scheduler
job drains them into Redis via `async_rpush_pipeline`. Redis is
treated as the durable buffer; a separate consumer reads from there
into Postgres. A flaky integration test
(`test_basic_spend_accuracy`) had been failing in CI with `Failed
to calibrate SPEND_PER_REQUEST after 5 attempts` after burning ~10
minutes per attempt. The PR fixes both the test *and* — incidentally,
in the course of writing better assertions — uncovers a real
data-loss bug in the buffer flush path.

# Problem

**The data-loss bug.** `store_in_memory_spend_updates_in_redis`
follows this shape:

```py
# 1. Drain all 6 queues into local lists
db_spend_update_transactions = await spend_update_queue.flush_all_updates_from_queue()
# ... five more drains ...

# 2. Push everything in one pipeline
result_lengths = await self.redis_cache.async_rpush_pipeline(rpush_list=rpush_list)
```

If step 2 raises (transient Redis hiccup, network blip, server
restart), the locally-bound `db_spend_update_transactions` and the
five sibling locals are garbage-collected with the function frame
and the spend they aggregated is *gone forever*. The source queues
were drained in step 1; they cannot be re-read. Every spend dollar
aggregated during that ~5-second tick disappears with no log line
loud enough to flag it. Operators see "spend tracking is sometimes
off by a few percent" and never trace it back.

The refactored test made this visible: 22 requests, ground-truth
expected total computed locally, only 5 requests' worth of spend
landed in Postgres. The discrepancy was reliable enough that it
masqueraded as a calibration failure for months.

**The test design bug.** The previous test had two phases: (1) make
*one* "calibration" request and poll `/key/info` for up to 5 ×
120 s for that single spend to land, deriving `SPEND_PER_REQUEST`
from the result; (2) make N–1 more requests and assert `N *
SPEND_PER_REQUEST` matches. If the calibration request hit the data-
loss path, phase 1 timed out with a vacuous error, having burned
10 minutes for nothing.

# Design

**Buffer fix:** wrap the rpush in try/except. On failure, restore
the drained transactions back into their respective queues so the
next scheduler tick retries:

```py
try:
    result_lengths = await self.redis_cache.async_rpush_pipeline(rpush_list=rpush_list)
except Exception as e:
    verbose_proxy_logger.error(
        "Spend tracking - failed to push aggregated spend updates to Redis. "
        "Restoring %d transaction sets to in-memory queues for retry on next tick. "
        "Error: %s", len(rpush_list), str(e),
    )
    await self._restore_spend_updates_to_in_memory_queues(...)
    return
```

The `_restore_*` helper walks the eight entity sub-types
(`USER, END_USER, KEY, TEAM, TEAM_MEMBER, ORGANIZATION, TAG,
AGENT`) and re-queues each `(entity_id, cost)` pair as a
`SpendUpdateQueueItem`, then re-inserts the five daily-spend
dicts into their queues. The drain-and-restore symmetry is good:
no aggregation work is repeated, only the wire push.

**Test refactor:** ground-truth-locally-then-poll, instead of
calibrate-then-multiply:

1. Make all 22 requests up-front, keep each response.
2. Compute `expected_total = sum(litellm.cost_per_token(model,
   prompt_tokens, completion_tokens) for each response)`. This is
   the canonical cost calculator the proxy uses internally, so
   "expected" comes from the same code that produces "actual",
   minus the buffer pipeline.
3. Fail-fast `/health/readiness` check before polling, so an
   infra outage stops misleadingly looking like a spend bug.
4. On poll timeout, the failure message includes `expected`,
   `observed`, `diff`, and the proxy's readiness state.

`PROXY_BATCH_WRITE_AT=2` is also passed to the docker run in CI so
the scheduler flushes every 2–7 s instead of 10–15 s, cutting the
poll deadline by ~3×.

# Risks

- **Restore path memory pressure.** When Redis is down for an
  extended period, every tick re-aggregates the same transactions
  + new ones. The in-memory queues grow unboundedly until Redis
  recovers or the process OOMs. A bounded retry (e.g. drop the
  oldest tick after N consecutive failures with a *single loud
  log line per drop*) would be safer for production. The current
  code prefers data preservation over OOM safety; that may not be
  the right tradeoff for a proxy that has to stay up.
- **Restore is per-entity-type, not all-or-nothing.** If
  `_restore_spend_updates_to_in_memory_queues` itself raises mid-
  loop (rare but possible — a queue is full, an invariant is
  violated), partial restoration leaves a subset of entities
  silently dropped. A try/except around each restore-loop with a
  per-entity error log would surface this.
- **`_restore_*` is `@staticmethod` with 11 parameters.** That is
  a maintenance hazard. The next contributor who adds a
  `daily_skill_spend_update_queue` has to remember to thread it
  through both the call site and the restore path. Bundling the
  six queues + six transactions into a single dataclass
  (`AggregatedTickState`) would make the call site +
  restore-site parity automatic.
- **Test ground truth uses `litellm.cost_per_token`.** If the
  function is itself buggy, the test silently agrees with the
  bug. The old calibration approach had the property that an
  off-by-X cost calculation cancelled out across N requests
  (still ratio-correct). The new approach catches absolute
  discrepancies but mis-blesses systematic errors. A separate
  regression test for `cost_per_token` itself (probably already
  exists) would close the loop.
- **The new error log is `error`-level**, but most "Redis blip"
  events are transient. In a healthy production run with one
  blip a week this is fine; in a Redis-flapping environment it
  spams. A rate-limited log (one per minute per error type)
  would be friendlier.

# Suggestions

1. Add a counter `spend_buffer_restore_total{reason}` so
   operators can see *how often* this path fires without
   tailing logs.
2. Bundle the 12 parameters into an
   `@dataclass AggregatedTickState`.
3. Consider an upper bound on restored items per tick.
4. Promote this to a CHANGELOG entry — silent spend loss is
   the kind of bug that operators want to know was fixed.

# Verdict

**Ship it, with a follow-up for backpressure.** The data-loss
fix is correct and the test refactor is a real improvement
(local ground truth + fail-fast readiness check is a much
sturdier shape than calibrate-then-multiply). The remaining
question is what to do when Redis stays down — preserve forever
or drop after N — and that is operationally significant enough
to warrant explicit policy rather than implicit "preserve
everything".

# What I learned

The shape "drain into a local, then push" is a classic data-
loss pattern. The drain is *destructive* (the source no longer
holds the data); the push is *fallible* (the destination might
not). Without an explicit restore-on-failure step, every push
failure is a permanent data loss. The fix has three flavors:

1. **Restore-on-failure** (this PR): drain → push → except: re-
   enqueue. Correct, but unbounded memory growth on extended
   outages.
2. **Peek-then-ack**: read without removing → push → ack-and-
   remove on success. Adds one round-trip but fails safe.
3. **Two-phase WAL**: append to durable log first, push to
   Redis from log. Most robust, most overhead.

For spend tracking specifically, restore-on-failure is the
right tradeoff (financial data, occasional Redis blips). For
critical-path async work (auth events, audit logs), two-phase
WAL is usually worth the cost. The diagnostic for "which
pattern do I need" is: *what is the cost of duplicate delivery
vs the cost of permanent loss?* Restore-on-failure trades
"maybe duplicate during retry" for "never lose"; peek-then-ack
is the symmetric "maybe lose during ack-failure" for "never
duplicate".
