## Summary

When a post-call guardrail raised (e.g. content blocked), the proxy's `finally` block was still firing the deferred async-success logging closure, then the exception path also fired `post_call_failure_hook` → `async_failure_handler`, producing a duplicate (Success + Failure) log entry per request. The Success row exposed the blocked LLM response, defeating the guardrail's purpose. This patch extracts the flush into a static helper `_flush_deferred_async_logging(logging_obj, exception_raised)` that returns early when `exception_raised` is true, while still clearing the slot so a downstream flush cannot double-fire.

## Specific observations

- `litellm/proxy/common_request_processing.py:1241-1247` (call site) — the `finally` now delegates to `ProxyBaseLLMRequestProcessing._flush_deferred_async_logging(logging_obj=..., exception_raised=_exception_raised)`. The static helper is the right call: previously the `finally` block carried four lines of logic with two failure modes (slot still set after closure mutation, exception swallowing). Centralizing it makes the contract testable.
- `litellm/proxy/common_request_processing.py:1442-1481` — the helper docstring is unusually thorough and explicitly explains the duplicate-row symptom and the streaming early-return case. Good. One concern: the helper clears `_enqueue_deferred_logging = None` *before* the `if exception_raised: return` branch runs, which means a later attempt to flush is a no-op. That is correct here, but the ordering is load-bearing — worth an inline comment ("clear slot before returning so a later flush cannot double-fire") because a future refactor that moves the assignment after the guard would silently regress.
- `litellm/proxy/common_request_processing.py:1474` — `if _enqueue_fn is None: return` returns before clearing. That is fine because the slot is already empty, but consider the path where `getattr(logging_obj, "_enqueue_deferred_logging", None)` raises (mock with side_effect, or a logging_obj that overrides `__getattribute__`). The helper does not catch — the `finally` would propagate. Probably acceptable, but a `try/except` around the getattr would harden against pathological logging_obj implementations.
- `tests/test_litellm/proxy/guardrails/test_deferred_guardrail_logging.py:235-249` — `test_flush_deferred_async_logging_fires_on_success` exercises the happy path against the real helper. Good.
- `tests/test_litellm/proxy/guardrails/test_deferred_guardrail_logging.py:252-296` — `test_flush_deferred_async_logging_suppressed_on_exception` is the regression test for the duplicate-row symptom. The assertion that `logging_obj._enqueue_deferred_logging is None` after the call is the load-bearing one — it locks in "slot is cleared even when we choose not to flush", which is exactly the property a regression that re-orders the helper would break.
- The test file removes the old `test_deferred_logging_fires_on_guardrail_exception` test that asserted the *opposite* behavior (closure must still fire on guardrail exception). That is the right move because the old test encoded the bug; the PR description should mention the test was inverted, not just removed, so reviewers do not assume coverage was lost.
