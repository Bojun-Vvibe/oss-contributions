# Context

LiteLLM's proxy supports Bedrock Guardrails as one of several
guardrail providers. Guardrails run at three hook phases —
`pre_call`, `during_call`, `post_call` — and every invocation
emits a `StandardLoggingGuardrailInformation` record into the
spend/compliance store so operators can audit which guardrails ran
on which requests. Two related production bugs were hitting the
Bedrock integration: (1) the spend log's `guardrail_mode` field was
showing `pre_call` for Bedrock calls that were actually `during_call`
or `post_call`, because the mode was being inferred from Bedrock's
own `INPUT`/`OUTPUT` source field rather than from the actual proxy
hook phase; (2) the spend log was persisting raw `match` and
`regex` strings — i.e. the literal text spans that the guardrail
matched on (PII, profanity, etc.) — into compliance metadata,
which is exactly what compliance is supposed to *prevent*.

# Problem

**(1) Mode inference from the wrong source.** The original code in
`bedrock_guardrails.py` mapped Bedrock's `source: "INPUT" |
"OUTPUT"` directly to `guardrail_mode`, with `INPUT → pre_call` and
`OUTPUT → post_call`. But Bedrock can be invoked at *any* hook
phase against either input or output content (e.g. `during_call`
on the streaming output stream uses `source: "OUTPUT"` but the
hook phase is `during_call`, not `post_call`). The unified
`apply_guardrail` path always looked like `pre_call` in spend logs
because the path's `source` defaulted to `INPUT`.

**(2) Match/regex leakage.** The guardrail hook returns a
provider-shaped JSON like `{assessments: [{type: "PII",
matches: [{match: "555-1234", regex: "\\d{3}-\\d{4}"}, ...]}]}`.
That JSON was being copied into
`StandardLoggingGuardrailInformation.guardrail_response` and
persisted. Anyone with read access to the spend store could see
the literal matched PII spans. The existing Bedrock
`_redact_pii_matches` helper redacted *some* shapes but only the
ones it knew about — anything from a different upstream
shape (e.g. HTTP `detail.assessments`) leaked through.

# Design

**(1) Native `during_call` hook.** A new
`use_native_during_call_hook: ClassVar[bool] = False` is added to
`CustomGuardrail`, with `BedrockGuardrail` opting in (`= True`).
When set, `during_call` invocations route through Bedrock's native
`async_moderation_hook` rather than the unified `apply_guardrail`
path. `ProxyLogging._execute_guardrail_hook` checks the flag.
Result: spend logs record the *actual* hook phase, because the
hook phase is now passed in explicitly via `logging_event_type`
rather than reverse-inferred from Bedrock's `source` field.

In `make_bedrock_api_request`:

```py
def make_bedrock_api_request(
    self, ..., logging_event_type: Optional[GuardrailEventHooks] = None,
):
    ...
    # When provided, drives `event_type` for the standard logging payload.
    # If omitted, keep legacy mapping from `source` for backwards-compat.
```

The "if omitted, keep legacy" branch is the right backwards-compat
move: existing custom callers that subclass `BedrockGuardrail`
without setting `logging_event_type` continue to behave as before;
the proxy hook path opts in to the new behavior explicitly.

**(2) Centralized redaction.** A new
`redact_nested_match_and_regex_keys()` in `core_helpers.py` does
an iterative, cycle-safe deep-walk over any nested dict/list and
replaces every `"match"` and `"regex"` string field with
`"[REDACTED]"`:

```py
seen: set = set()
stack: List[Any] = [redacted]
while stack:
    node = stack.pop()
    node_id = id(node)
    if node_id in seen: continue
    seen.add(node_id)
    if isinstance(node, dict):
        if "match" in node: node["match"] = "[REDACTED]"
        if "regex" in node: node["regex"] = "[REDACTED]"
        stack.extend(node.values())
    elif isinstance(node, list):
        stack.extend(node)
```

The implementation is iterative (avoids stack-overflow on deeply
nested provider JSON), cycle-safe via the `seen` set, and outer-
wrapped in a `try/except` that returns the original `payload` on
any unexpected error — i.e. it is *fail-open* with respect to
function correctness but *fail-closed* would be safer for
compliance. (See risks below.)

The function is then called from
`CustomGuardrail.add_standard_logging_guardrail_information_to_request_data`,
which is the single chokepoint where standard logging payloads
are built. Bedrock's local helpers (`_redact_pii_matches`, the
HTTP `detail.assessments` path) now delegate here too, so there
is exactly one redaction implementation and exactly one call
site.

**(3) Bonus dedupe.** `add_guardrail_to_applied_guardrails_header`
gains a `if guardrail_name not in _metadata["applied_guardrails"]`
guard, preventing the same guardrail from being recorded N times
in the response header when N hooks fire (which Bedrock does
naturally for input + output).

# Risks

- **Fail-open redaction.** If `redact_nested_match_and_regex_keys`
  raises (e.g. a custom dict subclass with a `__contains__`
  override that throws), the original `payload` is returned
  *unredacted*. For a compliance-motivated fix this is the wrong
  default: a redaction function should fail-closed (return `None`
  or `"[REDACTION_FAILED]"`) so that a bug never silently leaks
  PII. Worth flagging to maintainers.
- **`copy.deepcopy` is expensive on large guardrail JSONs.** For
  high-throughput proxies this could measurably increase latency
  in the logging path. A streaming visitor that mutates a
  pre-built payload would be faster, but harder to make safe.
- **`use_native_during_call_hook` is a `ClassVar[bool]`.** It is a
  class-level toggle, not per-instance. A user with two
  `BedrockGuardrail` instances cannot configure them
  independently. Probably fine for current use cases but a
  growing pain point.
- **Dedup of `applied_guardrails` is by name.** If a deployment
  configures two guardrails with the same display name (different
  configs), only the first is recorded. A `(name, hook_phase)`
  tuple key would be more correct.
- **`logging_event_type` is `Optional`.** A future contributor
  could forget to thread it through and silently fall back to
  the legacy inference path. A future-PR cleanup could promote
  it to required and delete the legacy branch.

# Suggestions

1. Switch redaction to fail-closed: catch the exception, log it
   loudly via `verbose_logger.error`, and return a sentinel
   `{"_redaction_error": True}` instead of the raw payload.
2. Add a test that exercises a deeply-nested cyclic structure to
   prove the `seen` guard works under realistic provider shapes.
3. Consider promoting `match`/`regex` redaction to a configurable
   list of sensitive keys (`STANDARD_LOGGING_REDACTED_KEYS`) so
   future providers don't each need to re-implement the same
   pattern under different field names (`pattern`, `value`,
   `text`, etc.).

# Verdict

**Ship it (with a follow-up to fail-closed the redaction).** The
hook-phase fix is correct and well-motivated; the redaction
centralization is the right architectural move (single chokepoint,
single implementation). The test coverage (+47 in
`test_custom_guardrail.py`, +35 in `test_core_helpers.py`, +269
in `test_bedrock_guardrails.py`) is appropriate for the scope.
The one design quibble — fail-open redaction — is small enough
to land as a follow-up rather than block this PR.

# What I learned

Two reusable lessons. First, when an inferred field can be wrong
(here: `guardrail_mode` inferred from `source` instead of from
the actual hook phase), the right architectural move is to pass
the value in *explicitly* rather than to write a smarter
inference. Inference is technical debt that compounds: every new
edge case adds a branch to the inference logic, while explicit
threading of the value scales linearly with new call sites. The
PR's `Optional[GuardrailEventHooks]` parameter with a "keep
legacy mapping if omitted" backwards-compat branch is the
canonical way to make that migration without a hard break.
Second, redaction functions should always fail-closed: a
correctness bug in a feature normally degrades the feature, but
a correctness bug in a redaction function leaks regulated data.
Default-deny is to redaction what default-allow is to
encryption.
