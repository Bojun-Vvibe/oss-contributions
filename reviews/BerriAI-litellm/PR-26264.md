# Context

`litellm`'s proxy mode lets one operator host the gateway and many
end users invoke it through their own API keys. Years ago a
researcher disclosed (Huntr bounty
`4001e1a2-7b7a-4776-a3ae-e6692ec3d997`) that any caller could pass
`api_base` in their request body and route the request to an
attacker-controlled host *that the proxy would still sign with the
operator's OpenAI API key*. The fix at the time was a
`banned_params` list inside `is_request_body_safe` blocking
`api_base`, `base_url`, and `user_config` from request bodies. This
PR extends that list to cover the same exfiltration shape across
cloud-provider auth fields that the original list missed.

# Problem

The original `banned_params = ["api_base", "base_url",
"user_config"]` covered the OpenAI-shaped attack surface
(redirect-the-host) but not the AWS- or GCP-shaped equivalents.
Specifically:

- **`aws_sts_endpoint`** — the host litellm hits to exchange an
  IAM role for temporary creds. A caller who controls this can
  point STS exchange at their own server, which then either steals
  the AWS access key/secret being exchanged or returns
  attacker-controlled temporary credentials that subsequent calls
  are made with.
- **`aws_web_identity_token`** — the OIDC web-identity token used
  in the federated `AssumeRoleWithWebIdentity` flow. A caller who
  injects their own token can have litellm assume *their* role and
  make AWS calls under that identity, and have the response
  surfaced back through the proxy's response.
- **`aws_role_name`** — the IAM role litellm assumes. A caller who
  swaps this can have litellm pivot into a different role within
  the operator's AWS account (privilege escalation, lateral
  movement).
- **`vertex_credentials`** — the GCP service account JSON / token
  used to authenticate to Vertex. A caller who injects credentials
  controls *which* GCP project/identity litellm acts as.

All four are the same bug class as the original `api_base`: a
field that *operators* set in deployment config but should never be
caller-overridable. The fix is to add them to the same list:

```py
banned_params = [
    "api_base",
    "base_url",
    "user_config",
    "aws_sts_endpoint",
    "aws_web_identity_token",
    "aws_role_name",
    "vertex_credentials",
]
```

# Design

The PR is +9/-1, all of it inside `is_request_body_safe`. The
existing escape hatches stay in place: operators who *want* to
allow client-side credential passing can still set
`general_settings.allow_client_side_credentials = true` or define
per-model `configurable_clientside_auth_params`. So the change is
strictly additive in safety: more parameters refused by default,
no new policy.

There are two meaningful design choices worth naming:

1. **List-based ban, not allowlist.** The function's defense is
   `for param in banned_params: if param in request_body: raise`.
   The shape is *blocklist*: any field not enumerated here passes
   through. This is the right choice given litellm's surface
   (hundreds of provider-specific fields, many legitimate), but it
   also means *every new credential-shaped field added by a
   provider integration must be reviewed for inclusion here*. The
   existence of this PR is evidence that step is currently manual
   and easy to miss.
2. **Single chokepoint.** `is_request_body_safe` is called once
   from each request-handling path (chat completions, embeddings,
   etc.). Adding to its list patches every entry point at once.
   This is the right shape — the alternative ("each handler
   re-checks its own auth fields") is exactly how regressions
   ship.

# Risks

- **Blocklist drift.** As cloud providers add new auth modes, new
  fields appear (e.g. `aws_session_token`, `aws_profile`,
  `gcp_credentials_file`, `azure_ad_token`, `azure_client_secret`).
  Each one is a potential bypass. There is no test or lint that
  forces the next contributor adding a new auth-shaped field to
  the codebase to also add it here. A property test would help:
  parse every provider's transformation module, extract its
  `litellm_params` keys, and assert that any key matching
  `(.*credentials.*|.*token.*|.*role.*|.*endpoint.*)` is either
  in `banned_params` or in an explicit allowlist.
- **`for param in banned_params: if param in request_body`.** This
  is a substring/key check; it only catches top-level keys with
  exact matches. `aws_sts_endpoint_url` would slip through if a
  future field used that name. Stricter normalization (lowercase,
  strip trailing punctuation) is unlikely to matter today but
  cheap insurance.
- **Tests verify the negative path** ("each new field rejected
  with 401") and "`api_base` still rejected", but I do not see a
  *positive* test for the allow-list opt-in
  (`allow_client_side_credentials=true`). Without that, a future
  refactor that breaks the escape hatch lands silently — no
  regression — and operators who depended on it discover it in
  production.
- **No corresponding warning in docs** (visible in the PR diff).
  Operators who configured `aws_role_name` in their own
  `litellm_params` and used to "test it from the client" will see
  401s after upgrading. The CHANGELOG should call this out.

# Suggestions

1. Add a positive-path test: with
   `allow_client_side_credentials=true`, all seven banned params
   are accepted. This locks in the escape hatch.
2. Add a property test that introspects each provider's
   `transformation.py` to find candidate auth-shaped fields and
   asserts every match is in `banned_params` or explicitly
   exempted. This is the only way to scale defense as providers
   grow.
3. Consider promoting `banned_params` to a `frozenset` and adding
   a docstring linking to the original Huntr disclosure. Both
   help future contributors recognise the security weight of the
   list.
4. Bump litellm's CHANGELOG and call this out as
   `[breaking] [security]` for any operator who relied on
   client-side cloud auth params.

# Verdict

**Ship it.** The patch surface is minimal, the security weight is
high, and the additions are correct. It belongs in a security
release note even though the title is `[Fix]`.

# What I learned

The "extend an existing security list" PR is the right *form* of
security maintenance. It is also a recurring symptom: every
codebase with a blocklist eventually accumulates this PR pattern,
because new attack-surface fields keep being added without
matching list updates. The structural fix is one of:

(a) **Allowlist, not blocklist** — explicitly enumerate which
    request-body fields are caller-settable; everything else is
    rejected. Hard for litellm given the size of the provider
    surface, but achievable per-provider. The Anthropic provider's
    request schema, for example, *could* be enumerated.

(b) **Schema-driven** — every provider's `litellm_params` schema
    declares per-field `caller_settable: bool`. The proxy refuses
    anything where `caller_settable=False`. This shifts the
    decision into the provider's module, where it is naturally
    co-located with the field's definition. New auth field added
    → required to declare `caller_settable=False` → no separate
    blocklist update needed.

(c) **Status quo + property test** — keep the blocklist, but add
    a property test that asserts every auth-shaped key in the
    codebase is either banned or explicitly opt-in. Cheap, and it
    converts the next missing-entry bug from "ships silently" to
    "CI fails red".

This PR is (c) waiting to be written. Without it, the next
provider integration to ship will reopen the same gap.
