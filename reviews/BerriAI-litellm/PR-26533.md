## Summary

Extends `SemanticMCPToolFilter._name_matches_canonical` so the proxy's tool-allow-list logic also recognises clients (LibreChat-style) that mangle tool names by **appending** a `<sep><uid>` segment for cross-server disambiguation, in addition to the existing **prefix** case (e.g. opencode wrapping the canonical with an alias). Both branches stay gated on the canonical containing `MCP_TOOL_PREFIX_SEPARATOR`, so unprefixed local function names can't be claimed by accident.

## Specific references

- `litellm/proxy/_experimental/mcp_server/semantic_tool_filter.py:265-294` — the new branch. The `MCP_TOOL_PREFIX_SEPARATOR not in rest` guard is the right invariant: a unique-ID segment by definition contains no separator, so allowing only `<canonical><sep><singleSegment>` correctly rejects `svc-search-extra_tool` (covered by `test_suffix_does_not_match_another_namespaced_tool` at test file line 146).
- `tests/.../test_semantic_tool_filter.py:202` — `test_prefix_and_suffix_both_match_same_canonical` asserts the **prefix** form wins when both are present in `available_tools`. That ordering happens implicitly because `_get_tools_by_names` is iterating `available_tools` and the prefixed entry was listed first; it isn't a property of `_name_matches_canonical` itself. Worth a one-line comment in the test, or a deterministic tie-break in the filter, to lock that contract in.
