# Context

LiteLLM's `Router` is the fallback/retry orchestrator: callers
hand it a `model_list` with multiple deployments (primary +
fallbacks), and Router decides when to retry, when to fall
over, and when to surface a terminal error. A key input to
those decisions is the *type* of error coming back from the
underlying `litellm.completion(...)` call — a
`RateLimitError` should trigger backoff, an
`AuthenticationError` should fail fast (retry will not help),
a `ContextWindowExceededError` should route to a larger-
context fallback rather than retry the same model.

This classification only works if the exception type is
preserved end-to-end. The `exceptions.py` mapping table is
the contract: every provider-specific error is translated
into one of the ~12 canonical exception classes, and the
Router switches on `type(e)`.

# Problem

Several recent issues (linked on the PR) report that the
Router *retries* errors that should be terminal. Classic
example: an Anthropic `overloaded_error` bubbles up as
`APIError` (generic), but should be `InternalServerError`
per the mapping table — Router treats `APIError` as
retryable with short backoff, pounds the overloaded
upstream, gets throttled harder, eventually 429s, and the
user sees a confusing `RateLimitError` that didn't actually
come from rate limiting.

Root cause, tracked down in the PR description:

```python
# router.py, in _handle_deployment_error (before):
try:
    response = await litellm.acompletion(**kwargs)
except Exception as e:
    # attempt to classify
    mapped = exception_type(
        model=deployment["model_name"],
        custom_llm_provider=provider,
        original_exception=e,
        completion_kwargs=kwargs,
    )
    raise mapped from e
```

`exception_type(...)` is the classifier. It's idempotent in
theory: if `e` is already a mapped LiteLLM exception, it
should be returned as-is. In practice, a regression in
`exceptions.py` from six weeks ago made it unconditionally
re-wrap — so an already-classified `InternalServerError`
went through a second trip, lost its `code` attribute
during the re-wrap (because the re-wrap path keys on
provider string format, not on attribute introspection),
and came out as generic `APIError`.

The downstream consequence is that the Router never saw a
terminal class and retried forever.

# Design

The PR takes two complementary fixes:

1. **Make `exception_type` actually idempotent.**

   ```python
   _MAPPED_BASES = (
       RateLimitError, AuthenticationError, BadRequestError,
       InternalServerError, ContextWindowExceededError,
       ContentPolicyViolationError, Timeout, APIConnectionError,
       ServiceUnavailableError, UnprocessableEntityError,
       NotFoundError,
   )

   def exception_type(..., original_exception, ...):
       if isinstance(original_exception, _MAPPED_BASES):
           return original_exception
       ...
   ```

   This is the *right* fix at the classifier layer. An
   exception that already satisfies the contract should
   not be touched — re-wrapping is where
   `response_format`, `status_code`, `llm_provider`,
   and other carried attributes get dropped.

2. **Stop catching-and-remapping inside the Router for
   already-classified exceptions.**

   ```python
   # router.py (after):
   try:
       response = await litellm.acompletion(**kwargs)
   except _MAPPED_BASES:
       raise                          # pass through untouched
   except Exception as e:
       mapped = exception_type(..., original_exception=e, ...)
       raise mapped from e
   ```

   Belt-and-suspenders against the classifier regressing
   again. Good — the classifier is a large switch over
   provider error text, and it is exactly the kind of
   code that gets re-broken during provider integrations.

The test in `test_router_error_passthrough.py` seeds a
`FakeProvider` that raises pre-classified exceptions and
asserts the Router receives the same exception objects
(not re-wrapped copies), using `assert exc is raised_exc`
— identity check, not equality. That's the correct
assertion for "pass-through" semantics.

# Risks

- **Subclass drift.** Provider adapters occasionally
  subclass `BadRequestError` to tag them with
  `retry_safe=True` or similar attributes. `isinstance`
  checks pass those through correctly. But if any adapter
  raises a `BadRequestError` that *should* be reclassified
  to `ContentPolicyViolationError` based on body contents
  (Anthropic does this — a `stop_reason: "content_filter"`
  is body-level, not class-level), the idempotence check
  short-circuits before that promotion can happen. A
  `needs_reclassify(e)` predicate the classifier can
  consult would preserve the option to promote.
- **`exception_type` callers outside Router.** The
  classifier is called from several sites — completion,
  embedding, image_generation, audio_transcription. The
  idempotence change affects all of them. That's almost
  certainly desirable (none of them want re-wrapping
  either), but the PR only adds the passthrough test at
  the Router layer. A parametrized test over callers
  would lock the wider contract.
- **Serialization.** If a caller catches an already-
  mapped exception, serializes it (e.g. FastAPI JSON
  error handler), and re-raises the JSON-decoded object,
  the object identity is lost and the `isinstance` check
  fails. Not a regression — it's the same behavior as
  before — but worth noting that the passthrough is an
  in-process optimization, not a cross-boundary
  guarantee.
- **Logging parity.** Router previously logged the
  re-wrap step ("mapped provider X error to Y"). That
  log line no longer fires for the passthrough branch.
  If anyone had dashboards keyed on that log, they will
  silently under-count. Adding a
  `logger.debug("passthrough: %s", type(e).__name__)` in
  the passthrough branch keeps the observability
  surface flat.

# Suggestions

1. Add a "promotion allowed" hook the classifier can
   call *before* the isinstance check, so future body-
   content-based promotions (like the
   content-policy case above) are not blocked by the
   short-circuit.
2. Parametrize the passthrough test across all mapped
   exception types — currently the test only covers
   three. Twelve tests at 4 lines each is cheap
   insurance.
3. Add a module-level note in `exceptions.py` spelling
   out the idempotence contract, so the next contributor
   who "simplifies" the classifier doesn't re-introduce
   the regression this PR fixes.

# Verdict

**Ship it.** This fixes a real classification bug that's
been causing spurious retries and blowing up upstream
rate-limit budgets. The belt-and-suspenders approach
(fix classifier + Router passthrough) is the right
defense for a code path that has regressed before and
will regress again.

# What I learned

Classifier idempotence is a load-bearing property of
any system that translates heterogeneous inputs into a
small canonical vocabulary — HTTP status normalization,
exception mapping, log-level mapping, content-type
sniffing. The failure mode when idempotence breaks is
always the same shape: a classified value passes
through twice, loses metadata in the second pass
because the second pass keys on a different signal than
the first, and comes out as a weaker class. The fix,
every time, is the shape this PR uses: check
`isinstance(value, CanonicalBases)` first, short-circuit
if true, document the contract next to the classifier.
And the *right* test is an identity assertion (`assert
out is in`), not a structural assertion, because
structural equality hides mutation-in-place.
