# Context

`litellm` is the multi-provider LLM gateway used as glue under many AI
coding agents (Aider, Cline, Crush, OpenHands, OpenCode all support it
either natively or as an "OpenAI-compatible" backend). Its Anthropic
adapter has two related responsibilities: translate Anthropic-shaped
requests/responses to/from the OpenAI ChatCompletions shape (the
"experimental pass-through" adapter), and process raw Anthropic responses
in the canonical `AnthropicConfig` chat path. Both surfaces have to
deal with a class of provider that emits **chain-of-thought as inline
`<think>...</think>` text inside the assistant text block** rather than
as proper Anthropic `thinking` content blocks — DeepSeek-R1, several
self-hosted reasoning fine-tunes, and at least one Bedrock provider all
do this.

The bug that triggers this PR is concrete and ugly: when such a
provider is wrapped behind litellm's Anthropic interface, the leaked
`</think>` tag survives into the assistant's text block. Downstream
agents that do tool-calling then see assistant text like
`"I need to call the tool first.\n</think>\n\ntool-loop-ok"` followed
by a `tool_use` block. Two failure modes follow. (1) The agent's UI
renders the literal `</think>` to the user. (2) Worse, when that
assistant message is sent back as conversation history on the next
turn, models that recognize `</think>` as a control token can be
prompt-confused or refuse to continue. The PR closes both by stripping
everything up through the first `</think>` from leaked text blocks at
three points in the adapter, plus a separate fix to preserve
`reasoning_content` when an assistant message has both thinking and
tool calls.

The diff is small (+228/-23, 6 files, half tests) but it touches
three independent translation paths and one identity bug — exactly
the kind of change that benefits from line-by-line review.

# Diff walkthrough

**Path 1: `AnthropicConfig.extract_response_content` (chat
transformation).** Three lines added at `transformation.py:1621`:

```python
if "</think>" in text_content:
    text_content = text_content.split("</think>", 1)[1].lstrip()
```

This runs on the canonical Anthropic response decode path. The split
limit of `1` is the right defensive choice — if a model emits two
`</think>` tags (which happens with looped reasoning or with chained
adapters), only the first is treated as a sentinel and everything
*after* the first is preserved as user-visible text. The `lstrip()`
removes the leading newlines that `</think>` is conventionally
followed by. No corresponding `<think>` open-tag handling, which is
also correct: by the time `extract_response_content` sees the text,
the open tag (if any) is already in front of the `</think>` and gets
discarded by the split.

**Path 2: `_translate_openai_content_to_anthropic` (OpenAI →
Anthropic adapter).** The same three-liner applied in mirror, at
`adapters/transformation.py:1242`:

```python
text_content = choice.message.content
if isinstance(text_content, str) and "</think>" in text_content:
    text_content = text_content.split("</think>", 1)[1].lstrip()

if text_content:
    new_content.append(
        AnthropicResponseContentBlockText(
            type="text", text=text_content
        ).model_dump()
    )
```

Two notes. First, the `if text_content:` guard is *new* — previously
the code unconditionally appended a text block whenever
`choice.message.content is not None`. The new guard correctly drops
the text block entirely if the only content was a leaked think
section. Without this, the adapter would emit an empty
`{"type": "text", "text": ""}` block, which is a separate Anthropic
spec violation that a downstream Anthropic client would reject.
Second, the `isinstance(..., str)` guard is necessary because
`message.content` can also be a list of structured content parts on
some providers — applying `"</think>" in <list>` would raise
`TypeError`.

**Path 3: `_sanitize_think_tag_text_blocks` in
`messages/handler.py`.** This is the most conservative change — a
new helper that operates on the already-decoded
`AnthropicMessagesResponse` dict at the very top of the handler:

```python
def _sanitize_think_tag_text_blocks(
    response: Union[AnthropicMessagesResponse, AsyncIterator]
) -> Union[AnthropicMessagesResponse, AsyncIterator]:
    if not isinstance(response, dict):
        return response

    content = response.get("content")
    if not isinstance(content, list):
        return response

    sanitized_content: List[Dict[str, Any]] = []
    for block in content:
        if not isinstance(block, dict):
            sanitized_content.append(block)
            continue

        if block.get("type") != "text":
            sanitized_content.append(block)
            continue

        text = block.get("text")
        if not isinstance(text, str) or "</think>" not in text:
            sanitized_content.append(block)
            continue

        cleaned_text = text.split("</think>", 1)[1].lstrip()
        if cleaned_text:
            sanitized_block = dict(block)
            sanitized_block["text"] = cleaned_text
            sanitized_content.append(sanitized_block)

    response["content"] = sanitized_content
    return response
```

The carefully-defended `isinstance` ladder is the right shape for an
adapter that has to be tolerant of partial-shape responses from many
provider backends. It also `dict(block)` clones before mutating —
important because adapters in this codebase are sometimes called with
borrowed/cached dicts. The streaming case (`AsyncIterator`) is a
deliberate no-op (`if not isinstance(response, dict): return
response`) — see Risks below.

The handler then invokes the sanitizer at all three return paths
(`anthropic_messages` async wrapper, the responses-API route, the
completion-transformation route, and the base HTTP handler). That's
the four-call-site change in `handler.py`. It would be tidier to do
this once at a single ingress point, but the dispatch shape of
`anthropic_messages_handler` makes that hard without a wrapping
decorator — the explicit four-site approach is fine, and the test
covers at least the completion-transformation path.

**Path 4: the `reasoning_content` identity fix.** This is the most
interesting bit and almost separate from the think-tag work. In
`adapters/transformation.py:612`, where the adapter walks an
incoming Anthropic assistant message with both `thinking` and
`tool_use` blocks:

```python
elif content.get("type") == "thinking":
    thinking_text = content.get("thinking") or ""
    thinking_block = ChatCompletionThinkingBlock(
        type="thinking",
        thinking=thinking_text,
        signature=content.get("signature") or "",
        cache_control=content.get("cache_control", {}),
    )
    if thinking_text:
        reasoning_content_parts.append(thinking_text)
    thinking_blocks.append(thinking_block)
```

And then later, when the adapter assembles the OpenAI-shape assistant
message that has tool calls:

```python
if len(tool_calls) > 0:
    assistant_message["tool_calls"] = tool_calls  # type: ignore
    assistant_message["reasoning_content"] = (
            "".join(reasoning_content_parts)
            if len(reasoning_content_parts) > 0
            else None
    )  # type: ignore
```

Before this PR, `reasoning_content` was simply not populated when
tool_calls were present, which meant downstream OpenAI-shape
consumers (e.g., the agent's chat history serializer) would lose the
reasoning entirely and only see the tool calls. Reasoning + tool
calls is a *very* common Claude pattern in agent loops: the model
thinks, decides to call a tool, emits both. Losing the reasoning on
this exact path would silently degrade quality on the next turn
when the history gets replayed.

# Risks / things a maintainer would catch

1. **Streaming is unhandled.** `_sanitize_think_tag_text_blocks`
   short-circuits `AsyncIterator` responses unchanged — so a
   streaming completion that emits leaked `</think>` chunks would
   still surface them to the caller. The PR is honest about this
   (the type signature acknowledges the iterator case) but doesn't
   ship a stream-rewriting equivalent. For agent tool loops on
   streaming providers the bug is still live.
2. **`</think>` is treated as global sentinel.** If a model
   legitimately emits the literal string `</think>` in user-visible
   text (e.g., explaining tag syntax to the user, or echoing back
   user-pasted XML), it will be silently truncated. Low-probability
   in coding-agent traffic but documented behavior change. A
   stricter form would require `<think>` to also be present, or
   require the tag to start at the beginning of the block.
3. **Three independent copies of the same sanitizer.** Path 1, 2,
   and 3 all implement `text.split("</think>", 1)[1].lstrip()`.
   When a future provider emits `<reasoning>...</reasoning>` or
   `<|reasoning|>...<|/reasoning|>` (both exist in the wild), three
   places need editing. A `litellm/llms/_helpers/reasoning_strip.py`
   with a single `strip_leaked_reasoning(text, sentinels=...)`
   would centralize this.
4. **No `<think>` open-tag handling.** If a provider emits an open
   `<think>` but the response is truncated before the close, the
   reasoning leaks in full. Also: if a provider emits *only*
   `<think>...content...</think>` with no trailing visible text,
   path 1 (`extract_response_content`) returns empty `text_content`
   which then suppresses the assistant text block — but the
   *reasoning* itself is also dropped, since this code path doesn't
   capture the stripped portion into `reasoning_content`. Path 4
   captures reasoning from structured `thinking` blocks; path 1
   doesn't capture from leaked text. Asymmetric.
5. **`response["content"] = sanitized_content` mutates the
   caller's dict.** The function clones individual *blocks* via
   `dict(block)` but reassigns `response["content"]` in place. If
   the response is shared (e.g., cached and replayed across multiple
   downstream consumers), the second consumer will see the
   already-sanitized content. Probably fine for litellm's request
   lifecycle but worth confirming there are no caches in front.
6. **Test coverage is path-specific.** Three new tests cover paths
   1, 2, and 3 each with a happy case. None test: empty result
   after stripping (entire content was reasoning), multiple
   `</think>` tags, no `</think>` (no-op), non-string `text`,
   non-list `content`, mixed-case `</Think>`. The defensive
   `isinstance` ladder in `_sanitize_think_tag_text_blocks` is
   *uncovered*.
7. **`reasoning_content` is `None` vs `""`.** The new code emits
   `None` when no reasoning parts are present. The downstream
   `assistant_message` shape on tool-call paths now sometimes has
   `reasoning_content: None` and sometimes lacks the key entirely.
   Inconsistent — preferable to either always set or always omit.
8. **`type: ignore` cluster.** Three new `type: ignore` comments
   in `adapters/transformation.py:660`. The underlying type
   `ChatCompletionAssistantMessage` (or whatever it's called)
   should be widened to include `reasoning_content` instead of
   silencing the type checker.
9. **Indentation looks off** in the `assistant_message[...]=`
   block — the inner `"".join(...)` and `else None` are indented
   12 spaces, not 8, which suggests this got hand-edited rather
   than autoformatted. `ruff format` / `black` would reflow it.
10. **No changelog / docs update.** A user-facing behavior change
    (leaked `</think>` is now silently stripped) deserves a one-line
    note in the Anthropic-provider docs and the changelog so
    operators don't get surprised when previously-visible text
    disappears from their logs.

# Suggested questions for the author

1. Is there appetite for a streaming-aware sanitizer? Even a
   single-buffer implementation (look for `</think>` across chunk
   boundaries, hold up to ~16 bytes back) would cover the streaming
   tool-loop case where this bug was reported.
2. Would you accept a follow-up that extracts a single
   `strip_leaked_reasoning(text, sentinel="</think>") -> (visible,
   reasoning)` helper and routes all four call sites through it,
   *also* capturing the stripped portion into `reasoning_content`
   for paths 1 and 3?
3. Should the sentinel list be configurable? At least three other
   open-source reasoning models use different sentinels
   (`<|reasoning|>`, `<reasoning>`, `<thinking>`); a per-model
   override map in `AnthropicConfig` would make this future-proof.
4. The `reasoning_content` identity fix touches a quality-sensitive
   path — has it been A/B'd against a multi-turn tool-calling eval?
   Specifically: does Claude's own `extended_thinking` path
   round-trip correctly when the assistant message goes back as
   history with both `thinking_blocks` *and* `reasoning_content`
   populated (no double-counting)?
5. On path 3, do you want to bias toward emitting an empty-text
   block versus dropping the block entirely? Some downstream
   Anthropic-shape consumers may rely on the text block being
   present (even if empty) to count tool-result indices.

# Verdict

**Approve with the streaming-coverage gap noted.** The fix is
minimal, defensive, and correct for the non-streaming case. The
`reasoning_content` identity fix on the tool-call path is a real
quality improvement and is the more important of the two changes
in this PR — losing reasoning on tool turns would silently degrade
agent performance on subsequent turns. The streaming case is the
one piece of unfinished business; either ship a follow-up or call it
out as a known gap in the changelog.

# What I learned

Two takeaways. **First**, on the API-shape side: when a "thin
adapter" sits between two LLM protocols, leaked control tokens are
the most underrated source of correctness bugs. The provider that
emits `<think>...</think>` is technically returning syntactically
valid text — the adapter has no signal to treat it specially other
than the sentinel literal. The right architectural place for this
kind of stripping is at the *protocol boundary* (the inbound decode
in `extract_response_content`), not at the protocol *outbound* — but
because litellm has multiple translation paths (chat,
pass-through-adapter, messages-handler), the sanitizer ends up
duplicated. The lesson is that any adapter codebase that has more
than one decode path will benefit from a shared "normalize provider
output" stage that runs *before* per-path translation, and reasoning
sentinel-stripping is a textbook member of that stage.

**Second**, on the `reasoning_content` fix: when assembling a
target-protocol message that has multiple optional fields
(`reasoning_content`, `tool_calls`, `thinking_blocks`,
`cache_control`), the assembly code wants to be a single-pass
projection from a structured intermediate, not a sequence of
conditional `if len(x) > 0: assistant_message[k] = ...` blocks. The
bug here was exactly that the `reasoning_content` projection was
guarded behind the wrong condition (or absent). A typed builder
pattern — `AssistantMessageBuilder().with_text(...).with_tool_calls(
...).with_reasoning(...).build()` — would have made the missing
field visible at construction time. In a codebase as wide as
litellm, that refactor isn't free, but the failure mode here
("silently lose reasoning when tool calls are present") is the
canonical reason to do it.
