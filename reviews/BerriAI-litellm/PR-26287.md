# Context

`litellm` builds outbound provider URLs by interpolating runtime
parameters into f-strings. Snowflake's account id, AWS regions,
Vertex AI's `vertex_location`, and the region segment of a Bedrock
ARN are *never* validated before they go into the host portion of an
HTTPS URL. Most callers pass clean values and the URL comes out
right. But `litellm` is also a multi-tenant proxy: those values can
arrive from request bodies, untrusted callers, or upstream config
that was never schema-validated.

# Problem

`f"https://{account_id}.snowflakecomputing.com/api/v2"` is a textbook
URL-construction injection sink. If `account_id` is allowed to
contain a dot, a slash, a fragment, or an `@`, the *constructed*
hostname is no longer Snowflake's. A value of
`evil.attacker.com#` produces `https://evil.attacker.com#.snowflakecomputing.com/...`,
which most HTTP clients resolve to `evil.attacker.com` because the
fragment terminates the authority. Same shape applies to the other
four sinks. The end-to-end risk: SSRF / credential exfiltration,
because litellm signs the request with the legitimate provider's
credentials before sending it to the attacker-controlled host.

This is exactly the bug class that gets you a CVE. Of note: the PR
went out as `[Fix]` in the title rather than as a `[Security]`
advisory, which is *probably* defensible because the parameters in
question are usually operator-controlled, but anyone running litellm
as a multi-tenant proxy should treat this as a security patch.

# Design

Five sinks, five `re.match` guards, the same regex skeleton:

```py
# snowflake
if not re.match(r"^[a-zA-Z0-9_-]+$", account_id):
    raise ValueError("Invalid account_id format")

# s3 vectors / vertex / bedrock region
if not re.match(r"^[a-z][a-z0-9-]*$", aws_region_name):
    raise ValueError("Invalid aws_region_name format")
```

The regexes are *strict* (the safe direction): only ASCII letters,
digits, hyphen, and (for Snowflake) underscore. `^...$` anchors lock
out fragment / path / query / port injection. The Vertex one
correctly special-cases `"global"` *before* the regex check so the
literal `"global"` is allowed even though it would also pass the
regex.

What the PR gets right:

1. **Validation at the construction site, not at the entry point.**
   Validating `account_id` only at request body parsing would miss
   values that arrive via `litellm_params`, env var
   (`SNOWFLAKE_ACCOUNT_ID`), or a config file. Validating *at the
   place the value gets concatenated into a URL* is the only
   chokepoint that catches every path.
2. **Two copies of `get_vertex_base_url` are patched in lockstep**:
   `litellm/llms/vertex_ai/common_utils.py` and the local copy in
   `litellm/proxy/pass_through_endpoints/llm_passthrough_endpoints.py`.
   Forgetting one of two near-duplicate functions is exactly how
   security patches regress; finding and patching both here is good.
3. **Bedrock parses the region out of an ARN string**, then
   validates the parsed value. This is correct: an attacker who
   controls the ARN could embed a region segment with metacharacters
   that pass ARN structural parsing but break URL construction.

# Risks

- **Two copies of `get_vertex_base_url` is a red flag.** The PR
  patches both, which is the right short-term move. But the next
  refactor that adds a third sink will forget. A single shared
  helper (`vertex_ai.urls.location_to_base_url`) imported by both
  call sites would prevent the next regression.
- **The error message leaks the invalid value.** `f"Invalid region
  in ARN: {batch_id}"` puts the attacker-supplied string into
  whatever logs the proxy emits. Usually fine, but for a security
  fix you ideally also redact: `"Invalid region in ARN"` (no
  interpolation), with the offending value going only to a debug
  channel. Tiny improvement, large win for log hygiene.
- **No regression test for the security path.** `pytest
  test_auth_utils.py — 39 passed` only exercises auth, not the URL
  validators added here. Five lines per sink (one positive, one
  injection-shaped negative) would lock these in. As-is, the next
  refactor that "simplifies" the validation can silently delete a
  guard with green CI.
- **Snowflake regex `[a-zA-Z0-9_-]+` allows mixed case.** Snowflake
  account IDs are documented as case-insensitive in URLs. Allowing
  uppercase here means the same logical account can be addressed
  with two distinct URL strings, which can confuse downstream
  caching, rate-limiting, and per-key budget tracking. Lowercasing
  via `.lower()` before validation would normalize.
- **The fix does not protect existing log lines or telemetry** that
  may already include the unvalidated `vertex_location` /
  `aws_region_name`. Out of scope for this PR, but worth a follow-
  up: where else does this string go, and is any of it
  user-visible (e.g. in error responses to the caller)?

# Suggestions

1. Extract the four regex constants into one shared module
   (`litellm/security/url_format.py`) so the next provider added
   to litellm cannot forget the pattern.
2. Add per-sink unit tests with a parameterized list of
   injection payloads (`evil.com#`, `evil.com?x=`, `..`,
   `evil.com:8080`, `evil.com/path`, etc.) — five lines per sink,
   permanent regression coverage.
3. Lowercase Snowflake `account_id` before validation.
4. Promote this PR to a security advisory with CVE if the proxy is
   in fact accepting these values from request bodies in any
   shipped configuration. The `is_request_body_safe` PR (#26264)
   that landed adjacent to this one suggests they are.

# Verdict

**Ship it, and follow up.** The 19-line surface is the right
*tactical* fix — every sink is patched, every patch is correct,
the merge into `main` should not wait. The strategic follow-up is
to deduplicate the validators and add the missing tests so the
fix doesn't quietly regress in three months.

# What I learned

URL construction by f-string is an ambient pattern in Python
codebases because `f"https://{host}/path"` *looks* exactly like
the safe thing to write. The bug class is the same as SQL
injection circa 2005: parameter substitution into a structurally
significant string without honoring the structure of that string.
The fix is the same shape too: either build the URL through
`urllib.parse.urlunparse` (which validates each component against
its allowed grammar) or guard the components with strict regexes
at the *construction* site (not the entry point). The
construction-site rule is the more general one: it catches values
that arrive from env vars, config files, secret managers, and
anywhere else parsing might have missed. The fact that this PR
patches *five* sinks is a good signal that the codebase has more
of the same shape — anyone doing a routine "find every f-string
that interpolates into `https://`" sweep will likely find a sixth.
