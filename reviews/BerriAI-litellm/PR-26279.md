# Context

`litellm` ships a model-routing proxy whose authentication layer is the
single chokepoint between an inbound HTTP request and downstream LLM
billing, model-access policy, budget enforcement, team membership, and
guardrail invocation. The function `_user_api_key_auth_builder` in
`litellm/proxy/auth/user_api_key_auth.py` is the sprawling decision tree
that decides "is this request allowed and as whom." Built up over
several years of integrations — virtual keys, JWT, OAuth2, OAuth2 reverse
proxy header, master_key, custom-auth callbacks, pass-through custom
headers, the `/user/auth` route, and an HA fallback for the `prisma`
database being unreachable — it has accreted multiple paths that each
return a `UserAPIKeyAuth` object on success. All of those paths are
supposed to terminate by calling `common_checks(...)`, the function that
actually enforces "this team is not blocked, this key is allowed to
target this model, this budget is not exhausted, the route is permitted
for this role, the org permission is satisfied." If any path forgets to
call `common_checks`, you have an authorization bypass — the request
authenticates fine but skips the gate.

This PR audits every return-without-`common_checks` path in the builder
and consolidates the call into one wrapper. The author lists the
specific bypass paths they observed: OAuth2, OAuth2 proxy header, JWT
admin shortcut, `master_key`, pass-through custom headers, `/user/auth`,
and the `allow_requests_on_db_unavailable` fallback. Several of these
are common production deployment patterns; the OAuth2 ones in
particular are how every upstream-OIDC integration ends up in
`litellm`'s codebase. The blast radius of "OAuth2 token never gets
budget-checked" or "JWT admin shortcut never gets team-blocked-checked"
is exactly as bad as it sounds — paying customers can transcend their
configured limits and admins-by-claim can hit routes that were
restricted from them.

The PR also closes a second, narrower bypass in the HA fallback itself:
under the old code, a `prisma` data-layer error (e.g. a unique-violation
during a normal write) would funnel through the same code path as a DB
outage, and `allow_requests_on_db_unavailable=true` would grant the
caller an anonymous *proxy-admin* token. So a deployer with that flag
on, taking a normal Postgres unique-constraint hit, could have a
request escalate to admin. This PR (a) shrinks the predicate that
decides "is this a DB outage" to actual connectivity errors, and (b)
demotes the fallback token to `INTERNAL_USER` with a sentinel `user_id`.

# Diff walkthrough

three files (the rest are tests).

**`user_api_key_auth.py` — the centralization.** Several deletions
remove the in-builder `common_checks` calls. Two illustrative ones:

The deletes are mechanical: every place the builder used to fetch
context, call `common_checks`, then return is now just the return.
The corresponding fetches are recentralized into a new helper:

```python
@tracer.wrap()
async def _run_centralized_common_checks(
    user_api_key_auth_obj: UserAPIKeyAuth,
    request: Request,
    request_data: dict,
    route: str,
) -> None:
    ...
    fetch_coros = []
    if user_api_key_auth_obj.team_id is not None:
        fetch_coros.append(
            _safe_fetch("team", get_team_object(...))
        )
    # ... user, end_user, project, global spend ...
    results = await asyncio.gather(*fetch_coros)
    await common_checks(
        request=request,
        request_body=request_data,
        team_object=team_obj,
        user_object=user_obj,
        end_user_object=end_user_obj,
        ...
    )
```

`_safe_fetch` is a load-bearing helper:

```python
async def _safe_fetch(label: str, awaitable):
    try:
        return await awaitable
    except (HTTPException, ProxyException, litellm.BudgetExceededError) as e:
        verbose_proxy_logger.debug(
            "centralized auth: %s fetch failed (%s: %s)", label, type(e).__name__, e,
        )
        raise
    except Exception as e:
        verbose_proxy_logger.debug(
            "centralized auth: %s fetch swallowed (%s: %s)",
            label, type(e).__name__, e,
        )
        return None
```

The split between "re-raise" and "swallow to None" is the
authorization-relevant decision in the entire function. `HTTPException`,
`ProxyException`, and `BudgetExceededError` — which `get_end_user_object`
raises for end-user budget violations — propagate so denial decisions
are not lost. Everything else (transient DB error, network blip while
fetching the `team_object`) becomes `None`, and `common_checks` runs
against whatever's already on the token. That is the right call: the
token itself carries `team_max_budget`, `team_models`, `team_blocked`
etc., so policy *can* still be enforced even when the side fetch fails.
But it is also the riskiest line in the patch — anything new that
extends `_safe_fetch`'s re-raise list later will need surgical care.

The skip predicate at the top of `_run_centralized_common_checks` is
the second load-bearing decision:

```python
# No-auth dev mode: master_key unset AND no JWT/OAuth2 auth configured.
if master_key is None and not (
    general_settings.get("enable_jwt_auth", False)
    or general_settings.get("enable_oauth2_auth", False)
    or general_settings.get("enable_oauth2_proxy_auth", False)
):
    return

if user_custom_auth is not None and not general_settings.get(
    "custom_auth_run_common_checks", False
):
    return
```

The first skip preserves the "litellm with no auth at all" dev-mode UX:
running `common_checks` in that mode would block every admin route on
deployments that intentionally have no `master_key` set. Reasonable.
The second skip is the explicit opt-out for callers who registered a
`user_custom_auth` callback and have not opted into running
`common_checks` afterward — backward-compatible with the documented
custom-auth contract. The skip is gated correctly: custom-auth
deployments do not use OAuth2 or DB-fallback paths, so this skip does
not re-open the bypass. Worth flagging in maintainer review: this is
the only intentionally-preserved bypass and it should be documented in
the migration notes.

**`_team_obj_from_token` — fallback reconstruction.** An interesting
helper reconstructs a `LiteLLM_TeamTableCachedObj` from fields already
on the token:

```python
def _team_obj_from_token(valid_token: UserAPIKeyAuth) -> LiteLLM_TeamTableCachedObj:
    assert valid_token.team_id is not None
    return LiteLLM_TeamTableCachedObj(
        team_id=valid_token.team_id,
        max_budget=valid_token.team_max_budget,
        soft_budget=valid_token.team_soft_budget,
        spend=valid_token.team_spend,
        ...
        object_permission_id=valid_token.team_object_permission_id,
    )
```

This is the recovery path when `_safe_fetch("team", get_team_object(...))`
returned `None`: rather than skipping the team check entirely, the
function reconstructs a stand-in team object from the token's own
denormalized fields and runs `common_checks` against that. It is the
right pattern — fail-closed where you can, fail-degraded where you must.
It does, however, mean that if the token's denormalized team fields are
stale (e.g. `team_blocked=False` cached on issue, then a human flips
`team_blocked=True` in the DB), the cached `False` wins during the DB
outage window. That is acceptable but worth a comment — it is a
deliberate availability/correctness trade.

**`auth_exception_handler.py` — demoting the HA-fallback token.** The
old code:

```python
return UserAPIKeyAuth(
    key_name="failed-to-connect-to-db",
    token="failed-to-connect-to-db",
    user_id=litellm_proxy_admin_name,  # ← the bypass
    request_route=route,
)
```

The new code:

```python
DB_UNAVAILABLE_FALLBACK_USER_ID = "__db_unavailable_fallback__"

verbose_proxy_logger.warning(
    "Auth: DB unavailable — issuing restricted INTERNAL_USER "
    "fallback token (allow_requests_on_db_unavailable=True)"
)
return UserAPIKeyAuth(
    key_name="failed-to-connect-to-db",
    token="failed-to-connect-to-db",
    user_id=DB_UNAVAILABLE_FALLBACK_USER_ID,
    user_role=LitellmUserRoles.INTERNAL_USER,
    request_route=route,
)
```

Two changes: the `user_id` is a sentinel that downstream code can key
off (and that *cannot* collide with any real user_id), and the
`user_role` is `INTERNAL_USER` instead of `PROXY_ADMIN`. The warn-level
log when the fallback fires is also an upgrade — DB-outage windows are
exactly when you want the on-call to know the proxy is degraded.

**`db/exception_handler.py` — narrowing the predicate.** The old
predicate was "any `PrismaError` is a connectivity error"; the new one
explicitly excludes data-layer errors:

```python
data_layer_errors = (
    prisma.errors.DataError,
    prisma.errors.UniqueViolationError,
    prisma.errors.ForeignKeyViolationError,
    prisma.errors.MissingRequiredValueError,
    prisma.errors.RawQueryError,
    prisma.errors.TableNotFoundError,
    prisma.errors.RecordNotFoundError,
)
if isinstance(e, data_layer_errors):
    return False
```

This is the structural fix to the second bypass. A unique-violation no
longer triggers the HA fallback. The fallback is reserved for actual
connectivity failure or the explicit `no_db_connection` marker. Bare
`PrismaError` defaults to True (preserve safety for unknown future
subclasses) — a reasonable conservative choice.

# Risks / things a maintainer would catch

1. **Behavior change for OAuth2 deployments.** The PR body calls this
   out: tokens issued via `enable_oauth2_auth` /
   `enable_oauth2_proxy_auth` are now subject to `common_checks`. If
   an OAuth2 layer issues tokens with non-trivial `team_id`, `models`,
   or `max_budget` fields, those will now be enforced where they were
   silently ignored before. This may surface latent
   misconfigurations as 401/403 in production. A migration note in the
   release CHANGELOG is warranted.
2. **`_safe_fetch`'s exception list is the only thing standing between
   "denial reaches caller" and "denial silently swallowed."** A future
   denial-emitting fetcher that raises a custom exception type not in
   `(HTTPException, ProxyException, BudgetExceededError)` will be
   silently dropped. The list should arguably be inverted: re-raise
   *anything that isn't* a known transient (DB error, network error,
   timeout), and log loudly otherwise.
3. **Five DB fetches in parallel per request.** `asyncio.gather`
   fans them out, but on a cold cache (e.g. a new key) every request
   pays five round trips. The author mentions the cache is hit on the
   hot path; a regression in cache TTL or invalidation would amplify
   here. A simple budget for "max number of DB fetches per
   `_run_centralized_common_checks` call" with metrics would catch a
   future cache regression early.
4. **Sentinel `__db_unavailable_fallback__` user_id is implicit
   contract.** Downstream code that filters logs, computes spend, or
   correlates requests now has to know about this sentinel. A grep
   for places that group by `user_id` would surface sites that need
   to special-case it.
5. **The skip for `master_key is None and not jwt/oauth2_auth`
   preserves a real bypass in dev-mode.** Documented, but a
   production deploy that accidentally launches without a
   `master_key` and without OAuth2 would now silently disable
   authorization — same as before, just newly load-bearing. A
   startup warning when both conditions hold would be an
   inexpensive guardrail.
6. **`_team_obj_from_token` synthesizes from cached fields that may
   be stale.** If an admin flips `team_blocked=True` in the DB during
   a DB outage, callers with already-issued tokens get the cached
   `False` for the duration of the outage. Probably acceptable, but
   should be in the migration notes.
7. **No new dedicated test for the
   `INTERNAL_USER`-vs-`PROXY_ADMIN` demotion.** The author updated
   `test_auth_exception_handler.py` but a single targeted assertion
   that "after a DB outage with the flag enabled, the returned token
   has `user_role == INTERNAL_USER` and `user_id == sentinel`" would
   be a clean regression anchor.
8. **Telemetry: the `_safe_fetch` swallow is at debug level.** A swallowed
   fetch is exactly the kind of event you want at info or warn. If a
   transient network blip causes 1% of requests to skip the team
   check and run against denormalized fields, a debug-level signal is
   not going to surface in dashboards. Suggest warn-level with rate-
   limited logging.
9. **`is_database_connection_error` deny list could go stale.** The
   `data_layer_errors` tuple lists today's known prisma exception
   subclasses. If a future prisma version adds a new data-layer
   error, the predicate will silently classify it as a connectivity
   error and re-open a narrower version of the bypass. Inverting the
   logic to "True only for an explicit allow-list of connection
   errors" would be more conservative.
10. **Migration story for users who *want* the old admin-fallback
    behavior.** Some operators might have built tooling on the
    assumption that DB outage → proxy-admin escalation. Probably
    deserved-to-break, but a migration note + a config flag
    `db_unavailable_fallback_role` (defaulting to INTERNAL_USER, with
    PROXY_ADMIN explicitly opt-in) would make the regression
    discoverable.

# Suggested questions for the author

1. Have you considered inverting `_safe_fetch`'s exception filter —
   re-raise everything *except* an explicit allow-list of known
   transient (network, prisma-connection, timeout) errors? Today a
   future custom denial type added to a fetcher will silently be
   swallowed.
2. The fallback `_team_obj_from_token` reconstructs from cached
   fields that may be stale during a DB outage. Should the
   reconstruction at least set a `_cache_origin: "fallback"` flag so
   `common_checks` can decide which checks to relax?
3. The "`master_key is None and no JWT/OAuth2 auth`" skip preserves
   the no-auth dev-mode UX. Would a startup-time warning when both
   conditions hold be welcome — or is this too aggressive for users
   who deliberately run an unauthenticated proxy?
4. The `data_layer_errors` tuple is a deny-list. Prisma may add new
   error subclasses upstream; would you consider inverting it to a
   single-purpose allow-list of connection-error types so future
   subclasses default to "data-layer, do not fall back"?
5. Should `_safe_fetch`'s swallow path log at warn rather than debug?
   A measurable rate of swallowed fetches in production is exactly
   the kind of authentication-quality signal that deserves dashboards.
6. Is the new `DB_UNAVAILABLE_FALLBACK_USER_ID` sentinel surfaced
   anywhere users of the proxy can grep for? Downstream consumers
   (spend reporting, audit logs) will need to know about it.

# Verdict

**Ready to merge after Q&A.** The structural change is correct and
the right shape — moving `common_checks` from "called from N return
sites" to "called once at the boundary" is the only sustainable way
to make this gate maintainable. The HA-fallback demotion and the
`is_database_connection_error` narrowing are independently valuable
correctness fixes. The behavior change for OAuth2 deployments is
significant and warrants a CHANGELOG entry; once the maintainer is
satisfied with the answers on `_safe_fetch`'s exception policy and
the `data_layer_errors` deny-list direction, this is a high-value
security fix that should ship.

# What I learned

The reusable insight here is the architectural pattern: when a single
function has N return sites and an invariant ("do X before returning")
that has to hold at each, the right fix is to move the invariant
*outside* the function, not to add the call site to each return.
`_user_api_key_auth_builder` had grown to the point where every new
auth method (OAuth2, OAuth2-proxy, JWT, master_key, custom-auth, HA
fallback) was a fresh opportunity to forget the invariant. The PR
doesn't repair the function — it surrounds it with a wrapper that runs
the invariant once on whatever the function returns. That is the
"decorator over decorator over decorator" pattern applied to
authentication: separate "who is this caller" from "is this caller
allowed to do this." Whenever you find yourself reviewing auth code
where the same ten-line block has been copied to N return sites and
sometimes one of them is missing, the refactor is always the same —
hoist the invariant to a wrapper. The complementary lesson is the
`_safe_fetch` design: making the re-raise/swallow split *explicit*
rather than implicit means the file can be audited by reading three
lines instead of three hundred. That's worth the modest indirection.
