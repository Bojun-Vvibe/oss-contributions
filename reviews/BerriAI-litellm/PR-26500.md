## Summary

`add_provider_specific_params_to_optional_params` only wrapped unknown kwargs into `extra_body` for providers in the hard-coded `litellm.openai_compatible_providers` list. Providers registered via `providers.json` (Scaleway in the test, but the registry is the actual abstraction) were missed, so non-OpenAI parameters like `chat_template_kwargs` landed as top-level kwargs on the OpenAI SDK call and were rejected with `unexpected keyword argument`. The fix adds an `or JSONProviderRegistry.exists(...)` clause so any JSON-registered provider goes through the same wrapping path.

## Verdict: `merge-as-is`

Targeted three-line behavioural change with two precise tests that cover both directions: a JSON provider wraps the unknown param into `extra_body`, and a non-OpenAI-compatible provider (`bedrock`) keeps the legacy top-level pass-through. Closing-issue reference (#26443) is in the docstring.
