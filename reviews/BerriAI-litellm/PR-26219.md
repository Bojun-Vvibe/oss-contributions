# Context

LiteLLM offers a "Responses → Chat Completions" bridge that lets
clients call `/v1/chat/completions` against models that natively use
the new Responses API (the post-Chat-Completions OpenAI shape with
`response.output[]` items, `response.completed`, `output_text.done`
SSE events, etc.). The bridge transforms the Responses-shaped reply
into the Chat-Completions shape the client expects. Two failure
modes had been observed in production: (a) the router would
sometimes downgrade a model that should hit `/responses` to the
legacy `/chat/completions` endpoint and surface as `403 Forbidden`,
and (b) even when routing was correct, some non-streaming Responses
payloads arrived with `response.completed.output == []` while the
real assistant message had been emitted earlier in the SSE stream.
The bridge then raised `ValueError: Unknown items in responses API
response: []`.

# Problem

**(1) Routing downgrade.** Router startup re-registers shared
backend keys (e.g. `chatgpt/gpt-5.4`) for every deployment that
mentions them. If two deployments share a backend and one of them
sets `model_info.mode = "chat"`, the *last write wins* in
`litellm.model_cost`, downgrading the shared backend's `mode` from
`responses` to `chat`. Subsequent `/v1/chat/completions` requests
get routed to the legacy endpoint and fail with `403`.

**(2) Empty `raw_response.output`.** The non-streaming bridge path
was:

```py
choices = self._convert_response_output_to_choices(
    output_items=raw_response.output,
    ...)
if not choices:
    raise ValueError(f"Unknown items in responses API response: {raw_response.output}")
```

If the upstream sent the assistant message via earlier SSE events
(`response.output_item.done`, `response.output_text.done`) and then
emitted `response.completed` with `output: []`, the bridge had no
data to convert. The whole SSE log was sitting in
`logging_obj.model_call_details["original_response"]`, but the
bridge never looked there.

# Design

**Routing fix (router.py +12).** When deployment registration
encounters a `model_cost` key that already exists, the new code
preserves the existing `mode` rather than overwriting it. This is
the correct conflict-resolution rule: `responses` is strictly more
capable than `chat` for shared backends (a `responses`-mode key can
serve both Chat-Completions-via-bridge and native Responses
clients), so preserving the more-capable mode is always the safe
direction.

**SSE recovery (transformation.py +137).** A new
`_recover_output_items_from_raw_sse(raw_sse)` walks the buffered
SSE log line-by-line and reconstructs output items. It handles
three cases:

- A late `response.completed` event with non-empty `output` →
  return that directly (this is the canonical recovery and is
  preferred over reconstruction).
- `response.output_item.done` events → keyed by `output_index`
  into a `recovered_output_items` map.
- `response.output_text.done` events → either attached to an
  existing item at the same `output_index`, or used to synthesize
  a minimal `{type: "message", role: "assistant", content:
  [{type: "output_text", text}]}` item.

The integration point in `transform_response()`:

```py
output_items = raw_response.output
if len(output_items) == 0:
    recovered = self._recover_output_items_from_logging(logging_obj)
    if recovered:
        output_items = recovered
        raw_response.output = recovered
        verbose_logger.warning(
            "Recovered empty Responses API output from raw SSE for model=%s", model)
```

The `verbose_logger.warning` is the right log level: this is a
recovery, not a normal path, and operators should see how often it
fires (an upstream regression is the most likely cause of frequent
recoveries).

The chatgpt-flavored `transformation.py` adds a thin
`_handle_raw_dict_response_item` (the +16) that routes raw-dict
output items through the same parser, ensuring the recovered items
go through the same shape-validation as the canonical ones.

# Risks

- **`_recover_output_items_from_raw_sse` is best-effort.** If the
  SSE log is itself corrupted, the bridge returns the zero-item
  result and the original `ValueError` still fires. That is correct
  but means recovery is silent on partial corruption — a counter or
  metric on "recovery attempted vs succeeded" would be valuable.
- **Synthetic message items have a fabricated ID
  (`f"msg_{output_index}"`).** Anything downstream that uses the
  message ID for dedupe or correlation across retries will see this
  fake ID. In practice this is fine for the Chat-Completions bridge
  consumer, which doesn't surface item IDs, but a comment naming
  this assumption would help.
- **Routing fix is one-directional.** "Preserve existing mode" is
  the right rule when the existing mode is `responses`, but the
  fix doesn't distinguish — it always preserves. If the shared
  backend was originally registered with the *wrong* mode (e.g.
  manual ops bug), the fix locks that wrong value in. A
  "preserve only if more-capable" rule would be safer; the current
  behavior is acceptable because in practice the first registration
  is always from the actual model definition.
- **`_recover_output_items_from_raw_sse` linear-scans the entire
  SSE log.** For very long completions this is O(N) per recovery
  attempt; fine because recovery is rare, but worth noting in a
  comment.

# Suggestions

1. Emit a counter metric (`litellm.responses.recovery_attempted` /
   `recovered`) so operators can alert on upstream regressions.
2. Document the "preserve existing mode" rule in `router.py` with
   an inline comment naming the shared-backend invariant.
3. Synthesize a `recovered: True` flag onto the recovered
   `raw_response` so debugging can distinguish recovered from
   canonical responses end-to-end.

# Verdict

**Ship it.** This is a re-land of a community PR (#25403) cleanly
rebased with appropriate co-author credit. The two fixes are
genuinely orthogonal but related — both target the ChatGPT/Responses
bridge — and the test coverage (+395 lines across three test files)
exercises shared-backend reuse, empty `output[]` SSE parsing, and
`response.output_text.done` reconstruction. The recovery path is
defensive enough (logs a warning, fails-soft to the original error)
that it cannot make things worse.

# What I learned

The general pattern here is "buffer the wire format, and on
unexpected end-of-stream conditions, try to reconstruct from the
buffered events." It generalizes far beyond LiteLLM. Any client
that sits behind a streaming protocol with a separate "completion"
event has to handle the case where the completion event arrives
malformed but the work-product events arrived fine. The cheap
defense is: never throw away the raw event log between the first
useful event and the completion event. Keep it for the duration
of the request, and have a single recovery function that walks it
when the canonical path fails. The expensive variant is to
materialize from the SSE stream as you go and treat the
"completion" event as merely an upper-bound terminator. This PR
chooses the cheap variant, which is the right trade-off when
upstream malformations are infrequent.
