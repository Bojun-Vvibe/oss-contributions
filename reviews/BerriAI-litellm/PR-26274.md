# Context

`litellm` recently shipped an MCP (Model Context Protocol) server module
under `litellm/proxy/_experimental/mcp_server/`. Two of its endpoint
families are OAuth-shaped: the **BYOK** flow (where a user types a
third-party API key into a litellm-hosted HTML form, and an MCP client
running locally — Claude Desktop, an editor extension — picks up the
result via an OAuth-style authorization-code redirect), and the
**discoverable** flow (where litellm acts as an OAuth proxy in front of
an upstream IdP). Both were introduced together and both adopted the
authorization-code-with-PKCE shape from RFC 6749 + RFC 8252.

The implementation as merged in #22850 had a cluster of issues that this
PR addresses together. The most consequential is open-redirect plus
authorization-code theft on the discoverable proxy: the `/authorize`
endpoint accepted any `redirect_uri`, encoded it into the OAuth `state`
parameter, and on `/callback` decoded it and 302'd the user with the
`code` attached as a query string. An attacker controlling an MCP client
config could trivially substitute `https://attacker/cb` and harvest the
authorization code. On the BYOK side, the authorization endpoint trusted
the OAuth `client_id` form field as the **user identity** (it had a
comment saying "external client passes LiteLLM user-id as client_id") —
a complete misreading of RFC 6749 §2.2, where `client_id` identifies the
*client application*, not the human. Anyone who could reach the endpoint
could attribute API-key writes to any user_id.

The PR also adds spec-compliance fixes to the `/token` endpoint
(error format per RFC 6749 §5.2, no-store cache headers per §5.1, exact
match of `redirect_uri`/`client_id` per §4.1.3), and a new shared
`oauth_utils.py` with a `validate_loopback_redirect_uri` helper that
codifies RFC 8252 §7.3 ("native apps redirect to loopback only"). The
combined diff is +1045/-116 across 5 files (3 source, 2 test).

# Diff walkthrough

**`oauth_utils.py` — the loopback-only predicate.** A new file, ~50
lines, that both endpoint modules now share:

```python
def validate_loopback_redirect_uri(redirect_uri: str) -> None:
    """Require a loopback ``redirect_uri`` (OAuth 2.1 §4.1.2.1 + RFC 8252
    §7.3 native-app pattern). MCP clients are native apps that listen on
    a localhost port; rejecting non-loopback URIs prevents a malicious
    client from pointing the callback at its own server to capture the
    authorization code ..."""
    try:
        parsed = urlparse(redirect_uri)
    except ValueError:
        raise HTTPException(status_code=400, detail="invalid_request")
    if parsed.scheme not in ("http", "https"):
        raise HTTPException(status_code=400, detail="invalid_request")
    if parsed.fragment:
        raise HTTPException(status_code=400, detail="invalid_request")
    host = (parsed.hostname or "").lower()
    if host == "localhost":
        return
    try:
        if ip_address(host).is_loopback:
            return
    except ValueError:
        pass
    raise HTTPException(status_code=400, detail="invalid_request")
```

Three details to praise. First, the loopback check uses
`ipaddress.ip_address(...).is_loopback`, not a string compare on
`"127.0.0.1"`. That correctly catches the `127.0.0.0/8` range plus
the IPv6 forms (`::1`, `0:0:0:0:0:0:0:1`) without enumerating them.
The author calls this out in the docstring. Second, the fragment
check — RFC 6749 §3.1.2 forbids fragments in redirect URIs — closes
a subtle smuggling: `http://127.0.0.1/cb#frag?code=...` would
otherwise be accepted, and the browser parses `code=...` as
fragment payload, which can leak via the URL bar but never reaches
the server's logs. Third, a fully-qualified `urlparse` failure path
returns 400 instead of bubbling a 500, which is small but matters
for error parity with spec-compliant clients.

The `TOKEN_NO_CACHE_HEADERS = {"Cache-Control": "no-store",
"Pragma": "no-cache"}` constant in the same file enforces RFC 6749
§5.1 / OAuth 2.1 draft-15 §4.1.3 — any intermediary that caches a
`/token` response can later replay credentials. Both endpoint modules
import and apply it.

**`byok_oauth_endpoints.py` — the identity fix.** The before/after
on `client_id` is the most important behavioral change. Before:

```python
_byok_auth_codes[auth_code] = {
    "api_key": api_key,
    "server_id": server_id,
    "code_challenge": code_challenge,
    "redirect_uri": redirect_uri,
    "user_id": client_id,  # external client passes LiteLLM user-id as client_id
    "expires_at": time.time() + _AUTH_CODE_TTL_SECONDS,
}
```

After:

```python
user_id = user_api_key_dict.user_id
if not user_id:
    raise HTTPException(status_code=401, detail="login_required")

auth_code = str(uuid.uuid4())
_byok_auth_codes[auth_code] = {
    "api_key": api_key,
    "server_id": server_id,
    "code_challenge": code_challenge,
    "redirect_uri": redirect_uri,
    "client_id": client_id,
    "user_id": user_id,
    "expires_at": time.time() + _AUTH_CODE_TTL_SECONDS,
}
```

The `user_id` now comes from a verified UI session cookie, supplied by
a new `_byok_session_auth` dependency. `client_id` is still stored, but
only as a binding token: it must match between `/authorize` and `/token`
(RFC 6749 §4.1.3), which is defense-in-depth against an attacker who
acquires an authorization code but presents a different client_id at
exchange time.

The session-cookie validator is unusually careful:

```python
def _user_id_from_session_cookie(request: Request) -> Optional[str]:
    """... The /token endpoint in this file ALSO issues master-key-signed JWTs
    (type="byok_session") for MCP-client-side use. They must not be
    accepted here as UI sessions — otherwise a leaked byok_session token
    could be replayed as a cookie to re-authorize BYOK writes. Distinguish
    by requiring a ``login_method`` claim ..."""
    ...
    payload = jwt.decode(
        token, master_key,
        algorithms=["HS256"],
        options={"require": ["exp"]},
    )
    if payload.get("type") == "byok_session":
        return None
    if payload.get("login_method") not in ("sso", "username_password"):
        return None
```

The `/token` endpoint in this same file mints HS256-signed JWTs
(`type="byok_session"`) for the MCP client to use. Both UI session
cookies and `byok_session` JWTs are signed with the same `master_key`,
so cryptographic verification alone does not distinguish them. The
explicit `payload.get("type") == "byok_session"` rejection plus the
positive `login_method` allow-list (`sso`, `username_password`) is the
type-confusion defense — without it, capturing a `byok_session` JWT
would let you re-authorize BYOK writes by replaying it as a cookie.
The `options={"require": ["exp"]}` is also defensive: PyJWT verifies
`exp` if present, but `require` ensures the token actually has one,
preventing a cookie that simply omits the field from being accepted
forever.

**`/token` endpoint — RFC compliance pass.** Several mechanical changes:

```python
if grant_type != "authorization_code":
    return _oauth_token_error("unsupported_grant_type")

record = _byok_auth_codes.get(code)
if record is None:
    return _oauth_token_error("invalid_grant")

if time.time() > record["expires_at"]:
    del _byok_auth_codes[code]
    return _oauth_token_error("invalid_grant")

if not _verify_pkce(code_verifier, record["code_challenge"]):
    return _oauth_token_error("invalid_grant")
```

Each `raise HTTPException` becomes `return _oauth_token_error(...)`.
The new helper produces `{"error": "<code>"}` per RFC 6749 §5.2 instead
of FastAPI's default `{"detail": ...}`, attaches the `no-store` headers,
and uses status 400. Spec-compliant OAuth clients parse the `error`
field; the previous `{"detail": ...}` shape would cause them to surface
"unknown error" instead of a meaningful code.

The `redirect_uri` and `client_id` exact-match enforcement is added
with a thoughtful comment about OAuth 2.1 backward compatibility:

```python
# RFC 6749 §4.1.3: if redirect_uri was sent with the authorization
# request, the token request MUST include the identical value.
# OAuth 2.1 draft-15 §4.1.3 drops this requirement — strict OAuth 2.1
# clients will omit it. Enforce equality ONLY when the client
# actually submitted a value, so we stay RFC 6749-backward-compatible
# without breaking OAuth 2.1 clients. PKCE + client_id binding
# (checked below) cover the security role redirect_uri played.
if (
    record.get("redirect_uri")
    and redirect_uri
    and redirect_uri != record["redirect_uri"]
):
    return _oauth_token_error("invalid_grant")
```

This is the kind of policy decision that's easy to get wrong: a strict
"always require redirect_uri equality" breaks OAuth 2.1, and an absent
check breaks the §4.1.3 contract. The conditional ("require equality
*if both sides submitted it*") threads the needle, and the comment
explains *why* — exactly the documentation a future maintainer needs.

**`discoverable_endpoints.py` — the open-redirect fix.** Less code
visible in this excerpt but conceptually the highest-severity item:
adding `validate_loopback_redirect_uri(redirect_uri)` at the start of
the `/authorize` handler before the URI is encoded into `state`.
Without this, the prior flow would happily 302 to any URL on
`/callback` with the authorization code attached as a query string —
the textbook "OAuth open redirect plus credential exfiltration"
chain.

# Risks / things a maintainer would catch

1. **`localhost` accepted as a hostname.** Some operating systems
   resolve `localhost` to non-loopback addresses if `/etc/hosts` is
   tampered. RFC 8252 §8.3 explicitly recommends *IP-only* loopback to
   avoid this. The author intentionally accepts `localhost` for
   ergonomic reasons, but a hardening option to require IP form when
   `STRICT_LOOPBACK=1` would be reasonable.
2. **No port restriction.** Any port is accepted on the loopback
   interface. A malicious other process running on the user's machine
   on port `47888` could capture the redirect. This is the standard
   trade-off documented in RFC 8252 §7.3 and matches normal practice,
   but worth flagging in security review.
3. **`_user_id_from_session_cookie` rejects on `payload.get("type") ==
   "byok_session"`. Open question: are there other token types this
   module mints?** A grep across the codebase for places that sign
   JWTs with `master_key` would confirm the allow-list (`sso`,
   `username_password`) is exhaustive.
4. **In-memory `_byok_auth_codes` store.** Authorization codes live in
   a process-local dict. In a multi-replica deployment, the user's
   `/authorize` may land on replica A and `/token` on replica B,
   guaranteeing `invalid_grant`. A Redis-backed store (or session
   affinity) is needed for HA. Sibling PR #26318 in this same repo
   addresses this for OAuth sessions; this PR may benefit from
   coordinating.
5. **PKCE verifier comparison is not constant-time.** `_verify_pkce`
   is not shown but if it uses `==` on hashes, a timing oracle is
   possible. `hmac.compare_digest` is the right comparator.
6. **The HTML form's `client_id` is now bound to the code but not
   validated against any registry.** Any string is accepted as the
   client_id at `/authorize` and pinned on the code. That is
   acceptable for the BYOK use case (clients are native apps with no
   pre-registration), but worth documenting that the `client_id`
   binding is anti-replay only, not actually identity-attesting.
7. **Tests touch only the request-validator boundary.** A useful
   addition would be a parametric test that walks every `/token`
   error branch (`invalid_grant` for: missing code, expired code,
   wrong code_verifier, wrong redirect_uri, wrong client_id) and
   asserts both the error body shape and the `Cache-Control`
   header — exactly the things the PR is trying to lock in.
8. **`master_key` may be rotated.** UI session cookies signed with the
   old key become invalid on rotation. That's fine for security but
   the user-visible UX is "you have to re-login on rotation" —
   document it.
9. **`require=["exp"]` is good but the PR doesn't bound `exp` upper.**
   A UI session cookie with `exp=2099-01-01` would pass validation.
   A defense-in-depth `payload["exp"] - payload["iat"] <= 24h` check
   would limit how long a leaked cookie remains valid.
10. **Loopback-only in BYOK works for desktop MCP clients but breaks
    any future *web*-hosted MCP client.** This is a deliberate
    constraint that should be in the docs so users don't ship a
    web-based MCP integration that silently 400s.

# Suggested questions for the author

1. Should the BYOK `client_id` actually be required to live in a
   pre-registered allow-list (like the discoverable proxy's client
   registration), even for native MCP clients? It would let admins
   audit which clients have ever authorized against their proxy.
2. Has consideration been given to making the `_byok_auth_codes` and
   `oauth_state` stores Redis-backed at the same time (the related
   sibling PR #26318 does this for MCP OAuth sessions)? The current
   in-memory store guarantees production failure under HA.
3. Is `_verify_pkce` doing constant-time comparison of the SHA-256
   challenge? If not, a one-line `hmac.compare_digest` swap would
   close a timing-channel risk.
4. Does PyJWT's `master_key` rotation story need any UX work — when
   the key rotates, every active UI session cookie invalidates;
   should there be a grace period using a previous-keys list?
5. Would you accept an additional `STRICT_LOOPBACK=1` flag that
   requires the IP form (rejecting the literal `localhost`), to harden
   against `/etc/hosts` tampering?
6. The `login_method in ("sso", "username_password")` allow-list is
   exhaustive of UI sessions today. Is there an enum or shared
   constant we can pin against, so a new login method introduced
   elsewhere doesn't silently fail to authorize against this gate?

# Verdict

**Ready to merge after Q&A.** This is a high-quality security-focused
hardening PR. It collapses two genuine vulnerabilities (open redirect
on the discoverable flow, identity-spoofing on the BYOK flow) and
brings spec compliance up to RFC 6749 + RFC 8252 + the OAuth 2.1
draft. The shared `oauth_utils.py` is the right shape — both endpoint
modules should pass through it for any future OAuth-shaped feature.
The questions above are mostly defense-in-depth follow-ups; the core
diff should ship now.

# What I learned

The most reusable lesson is structural: when you have two OAuth-shaped
endpoint modules (here, BYOK and discoverable), the redirect-URI
validator *must* be shared code or you will eventually have a tighter
check in one and a sloppy check in the other. The PR's
`oauth_utils.validate_loopback_redirect_uri` is the correct shape for
this — a single, well-tested predicate that codifies the spec citation
and the rationale. Any future OAuth-shaped surface in the same codebase
should be force-routed through it. The complementary lesson is
type-confusion across same-key-signed JWTs: when one module both
*verifies* incoming JWTs and *issues* outgoing JWTs signed with the
same secret, the verifier must positively check the issued tokens it
*will* accept, not just verify the signature. The
`payload.get("type") == "byok_session"` reject + `login_method`
allow-list is the explicit form of that defense; the implicit form
(just signature verification) is how cross-token replay attacks happen.
A useful pattern to enforce in any auth-system code review: every JWT
verifier should pair signature verification with a positive type
predicate, and every JWT issuer should set a unique `type` claim. That
turns "everything signed with master_key" into "this specific class of
token signed with master_key."
