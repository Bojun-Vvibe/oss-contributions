# Context

`litellm` ships a proxy with per-key and per-team spend budgets.
Budgets can be "windowed" — e.g. a key has a `1d` window with a
spend limit, and at the end of each window the spend counter
resets and the next window opens. The implementation stores the
windows as a JSON blob (`budget_limits`) on
`LiteLLM_VerificationToken` (keys) and `LiteLLM_TeamTable` (teams),
each window having its own `budget_duration` and `reset_at` fields.

A periodic background job, `ResetBudgetJob.reset_budget_windows`,
ticks every minute, walks the rows whose `budget_limits` is
non-null, and for each window whose `reset_at` is in the past it
bumps `reset_at` forward and clears the in-memory spend counter
keyed by `spend:{key|team}:{id}:window:{duration}`.

The schema column is `budget_limits Json?` (nullable JSON). The
"is non-null" filter is the part that broke.

# Problem

The previous implementation used the obvious ORM filter:

```python
all_keys = await self.prisma_client.db.litellm_verificationtoken.find_many(
    where={"budget_limits": {"not": None}}  # type: ignore[arg-type]
)
```

This raises every tick:

```
prisma.errors.MissingRequiredValueError: Unable to match input
value to any allowed input type for the field.
Parse errors: [`where.budget_limits.not`: A value is required
but not set, ...]
```

The reason is a known `prisma-client-python` limitation
(RobertCraigie/prisma-client-py#714): null-filtering on a
nullable JSON column requires a `DbNull` / `JsonNull` sentinel
that the Python client does not expose. When `None` is passed in,
the client serialiser drops the value entirely before sending it
to the query engine, which then rejects the malformed `not`
clause. The `# type: ignore` on the original line is a tell — it
silenced the type-checker's complaint but did not change the
runtime behaviour.

Net effect: the reset job has been raising on every tick for both
the key path and the team path, so no windows have been resetting.
Customers running with key-level windowed budgets would have seen
spend counters never clear after a window expired — the next
request would be blocked, indefinitely, by a stale counter.

This is a high-severity correctness bug masked as a noisy log
line.

# Design

The fix sidesteps the Prisma-client limitation by dropping to
`query_raw` for exactly the filter step:

```python
key_rows = await self.prisma_client.db.query_raw(
    'SELECT token, budget_limits FROM "LiteLLM_VerificationToken" '
    "WHERE budget_limits IS NOT NULL"
)
for row in key_rows:
    raw = row["budget_limits"]
    if not raw:
        continue
    windows: list = raw if isinstance(raw, list) else json.loads(raw)
    ...
    if changed:
        await self.prisma_client.db.litellm_verificationtoken.update(
            where={"token": row["token"]},
            data={"budget_limits": json.dumps(windows)},
        )
```

Three things to like about the shape:

1. **Read-narrowed-then-write-via-ORM.** Only the read goes
   through `query_raw`, and only because the ORM cannot express
   the filter. The write still goes through
   `litellm_verificationtoken.update(where={"token": ...}, data=...)`
   so all of the ORM's hooks (audit logging, prepared-statement
   reuse, change tracking) still apply on the path that mutates
   state. This is the right place to draw the line.

2. **Selecting only the columns needed.** The raw query is
   `SELECT token, budget_limits FROM ... WHERE budget_limits IS NOT NULL`,
   not `SELECT *`. If the table grows new columns the query stays
   tight. Two-column projection also makes the regression test's
   stub trivial: `[{"token": ..., "budget_limits": ...}]`.

3. **The inline comment justifies the workaround.** This is the
   bit reviewers most often skip:

   ```python
   # Note on raw SQL: prisma-client-python does not support null-filtering
   # on `Json?` columns (no DbNull/JsonNull sentinel — see
   # RobertCraigie/prisma-client-py#714). We use `query_raw` with
   # `IS NOT NULL` so we don't materialize every key/team row on each
   # tick of the reset job. Writes still go through the ORM.
   ```

   It links the upstream issue, names the constraint, and explains
   why the fallback was chosen over the alternatives (load every
   row, filter in Python — which the comment explicitly rules out).
   Six months from now when `prisma-client-py#714` lands, the next
   reader knows this is the place to delete.

The team path mirrors the key path with the same shape, the same
comment-by-implication, and the same write-through-ORM pattern.
Symmetry is good here — the two paths have to behave identically
under failure (one path raising must not stop the other), and the
PR's `test_reset_budget_windows_query_error_does_not_break_team_path`
explicitly defends that.

The test file (`test_reset_budget_job.py`) is the second
substantive contribution. It adds 233 lines of unit tests that
deserve a closer look:

```python
def test_reset_budget_windows_uses_is_not_null_filter(monkeypatch):
    """Regression guard for the Prisma client limitation documented in
    RobertCraigie/prisma-client-py#714: `{"not": None}` on a `Json?` column
    raises `MissingRequiredValueError`. We work around it by using `query_raw`
    with `IS NOT NULL`. If someone reverts to the ORM filter, this test fails.
    """
    job, prisma_client, _ = _make_reset_budget_windows_job(
        monkeypatch, key_rows=[], team_rows=[]
    )
    asyncio.run(job.reset_budget_windows())

    queries = [call.args[0] for call in prisma_client.db.query_raw.await_args_list]
    assert len(queries) == 2, queries
    key_query, team_query = queries

    assert '"LiteLLM_VerificationToken"' in key_query
    assert "budget_limits IS NOT NULL" in key_query
```

This test does the *boring but important* thing: it pins the
*shape* of the SQL the workaround uses, so a future PR that
reverts to ORM filtering will fail loudly with a clear
explanation in the docstring of why it can't do that. This is
the right pattern for "we had to drop one layer of abstraction
to dodge an upstream bug" — the regression test has to assert
the workaround, not just the behaviour.

The other tests cover the obvious cases:

- expired window → update + counter clear (key, team)
- unexpired window → no write
- `budget_limits` returned as a JSON-encoded string (because
  `query_raw` can come back either way depending on the engine)
  → parsed via `json.loads`
- empty/falsy `budget_limits` → skipped without error
- key path raises → team path still runs

The "string vs list" test is the one I would not have thought to
write. `query_raw` from prisma-client-python returns JSONB columns
as already-parsed Python objects on Postgres, but other engines
(or future versions) might hand back the raw string. The handler
defensively handles both with
`raw if isinstance(raw, list) else json.loads(raw)`. The test pins
the defence.

# Risks

**Quoting and identifier safety.** The raw SQL uses
double-quoted identifiers (`"LiteLLM_VerificationToken"`) which
is correct for Postgres and incorrect for MySQL. If `litellm` is
deployed against MySQL via prisma's MySQL connector, this query
will fail at the parser. Two mitigations: (a) document Postgres
as the supported backend for windowed budgets in the PR, or
(b) parameterise on the driver. A grep through the repo for other
`query_raw` uses would tell you which convention exists.

**`SELECT` columns drift if the schema changes.** If a future
PR renames `token` to `key_token` on `LiteLLM_VerificationToken`,
the schema migration tool will succeed and this raw query will
return rows with no `token` field, raising `KeyError` at the
`row["token"]` access. The ORM path would have caught this at
import time (Pydantic model validation). A small mitigation:
mention the raw query in a migration checklist, or extract the
column names to module-level constants alongside the table name.

**Per-tick row materialisation.** The comment says "so we don't
materialize every key/team row on each tick". For deployments
with millions of keys but only thousands with `budget_limits`
set, the `IS NOT NULL` filter is doing real work. Worth checking
that there's an index on `budget_limits` (a partial index
`WHERE budget_limits IS NOT NULL` would be ideal) — without it,
this is still a sequential scan per tick. Out of scope for the
fix itself but worth a follow-up issue.

**Counter clear runs only on `changed`.** If the ORM `update`
fails (transient DB blip), the in-memory counter has already
been cleared by `_reset_expired_window`. Next tick, the row will
be re-read, the window will *still* look expired, and the update
will be retried — fine — but the in-memory counter will be
re-cleared each time. For the duration of the outage the spend
attribution is whatever the cache says, which is "0". This is
the existing behaviour, not something this PR introduces, but
worth noting in case anyone is debugging "spend showed 0 for
five minutes during the DB hiccup".

**Test stubs sys.modules.** `_make_reset_budget_windows_job`
calls `monkeypatch.setitem(sys.modules, "litellm.proxy.proxy_server", fake_module)`
to dodge the heavy real `proxy_server` import. This works but it
is global state — if pytest is run with parallel workers and
another test imports `litellm.proxy.proxy_server` *between* the
monkeypatch setting and the function-under-test reading it, you
have a race. `pytest-xdist` handles this by giving each worker
its own process, so probably fine in practice. Mention in a
comment that this pattern requires single-process or
process-isolated execution.

# Suggestions

1. Add a one-line comment at the top of the new raw queries
   explaining the Postgres-identifier-quoting assumption, or use
   prisma's introspection to fetch the actual table name at
   runtime if the project supports multiple backends.

2. Consider extracting `"LiteLLM_VerificationToken"` and
   `"LiteLLM_TeamTable"` to module-level constants. The string
   appears in both the SQL and the test assertion; pinning it
   in one place makes the schema-rename failure mode less
   surprising.

3. Add a follow-up issue (or at least a TODO comment) for the
   partial index recommendation. Without an index, the `IS NOT
   NULL` scan grows linearly with table size on every tick.

4. The two existing-test assertions changed from one style:

   ```python
   assert len(updated) == 2, (
       f"Expected 2 endusers reset (1 explicit + 1 implicit), got {len(updated)}"
   )
   ```

   to:

   ```python
   assert (
       len(updated) == 2
   ), f"Expected 2 endusers reset (1 explicit + 1 implicit), got {len(updated)}"
   ```

   That looks like an autoformatter sweep that happened to land
   in this PR. Worth either splitting into a separate "format
   only" commit or noting in the PR description so future `git
   blame` doesn't attribute the formatting churn to the budget
   fix.

5. The regression test checks `len(queries) == 2`. If a future
   PR adds a third path (say, end-users with windowed budgets),
   this assertion fires and the test failure looks like "we
   broke the existing two", not "we added a new one". A small
   comment or splitting the assertion would clarify intent.

# Verdict

Approve, no blockers. This is a textbook "upstream library
limitation, document and work around it" PR. The diagnosis is
correct, the workaround is the minimum surface area, the comment
makes the workaround discoverable, the regression test pins both
the SQL shape and the behaviour, and the symmetry between key
and team paths is preserved including under partial failure. The
suggestions are quality-of-life, not correctness.

# What I learned

The thing I am taking away is the *shape* of "regression-test the
workaround, not just the behaviour". The naive instinct when you
fix a bug like this is to write a test that exercises the
behaviour: insert a row, run the job, assert the row got updated.
That's necessary but not sufficient. A future contributor looking
at the ORM filter would see "this looks cleaner than the raw
SQL" and revert the workaround, the behavioural test would still
pass (because the bug only fires on certain prisma-client-python
versions), and the bug would silently regress.

The
`test_reset_budget_windows_uses_is_not_null_filter` test asserts
the *implementation* uses `IS NOT NULL` raw SQL, with the upstream
issue link in the docstring. That converts "this is a workaround"
from a tribal-knowledge fact into an enforceable invariant. The
cost is one extra test that looks redundant; the benefit is that
the workaround cannot be accidentally undone. For any change of
the form "drop one abstraction layer to dodge an upstream bug",
this should be the default pattern.

The second lesson is about JSON-column quirks in ORM-shaped APIs.
`Json?` is a perfectly normal SQL concept (nullable JSONB column)
and a perfectly broken ORM concept (the ORM has to thread
"SQL NULL" vs "JSON null" through its sentinel system, and many
ORMs get this wrong). When you see a `# type: ignore` on a JSON
filter, treat it as a yellow flag worth investigating —
"the type system is unhappy here" is often a hint that the
runtime is also unhappy and the failure mode is just "rejected
at the engine, error logged, job silently never ran". This bug
had been raising every minute and the existing logging was
apparently not enough to surface it. Worth asking: which other
ORM-shaped operations on `Json?` columns in this codebase are
silently failing the same way?
