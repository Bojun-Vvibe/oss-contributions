# Context

OpenHands SaaS has an organization mode where admins set
"organization defaults" (LLM, condenser, verification settings)
that members inherit. The frontend uses React Router clientLoaders
with a permission-guard pattern: each protected route's loader
runs `createPermissionGuard(permission, currentPath, customRedirectPath?)`,
which redirects unauthorized users either to the configured
`customRedirectPath` or — falling back — to
`getFirstAvailablePath()`, the first route the current user *is*
permitted to see.

The org-defaults pages also depend on `getActiveOrganizationUser()`
to resolve "which org am I acting on behalf of." During an org
switch this call can transiently return `undefined` (a brief
`/me` failure is normal mid-switch).

# Problem

A textbook cross-route redirect loop, made possible by an
incomplete loop-prevention check.

When `getActiveOrganizationUser()` returned `undefined`, the
three org-defaults loaders fell through to `customRedirectPath`,
which pointed at the *personal* settings counterparts:
`/settings/condenser`, `/settings/verification`, `/settings`.
But those personal routes, in SaaS mode, unconditionally bounce
*back* to org-defaults via `requireOrgDefaultsRedirect`. Two
loaders, each redirecting to the other, and the browser tab
loops until the JS engine bails or the user closes it.

The existing safety net in `createPermissionGuard` checked
`redirectPath === currentPath` — i.e., it caught a loader
redirecting to itself. But the loop here is between *two
different paths* (`/settings/org-defaults/condenser` →
`/settings/condenser` → back). A same-path check cannot see a
two-node cycle.

# Design

Surgical: drop the `customRedirectPath` argument from the three
org-defaults loaders. With no custom redirect, the guard falls
through to `getFirstAvailablePath()`, which by construction
returns a route the user can actually land on without further
redirect — the user's terminal page (`/settings/user`) for the
unauthorized case. -3 LOC × 3 routes = -12, +3 LOC of new arg
shape, plus a 98-line targeted regression test.

The test is the load-bearing element. It mocks
`getActiveOrganizationUser()` to return `undefined`, runs the
loader, and asserts the resulting redirect URL is `/settings/user`
— i.e., a leaf node, not the personal route that would bounce
back. That single assertion locks in the structural property
this PR established.

# Risks

- **Behavioral change for the legitimate "user lacks org perms"
  case.** Previously these users landed on `/settings/...`
  (their personal counterpart, which has the same fields).
  Now they land on `/settings/user`. Arguably worse UX — they
  asked for "condenser settings" and landed on "user
  settings." But it's correct: the personal page would just
  bounce them back, so the previous behavior wasn't actually
  showing them their personal condenser settings either. A
  flash of the right page followed by an infinite loop is
  worse than landing on the wrong-but-stable page.
- **The root cause — a two-node cycle that the same-path guard
  can't detect — is unaddressed.** Three other routes happen
  to participate; any future loader that adds a
  `customRedirectPath` pointing into the org-defaults space
  re-creates the cycle. A graph-based guard
  (`Set<visited>` carried across the redirect chain) would
  fix the class. This PR fixes the instances.
- **Transient `getActiveOrganizationUser()` failures still
  swallow errors silently.** The loader treats `undefined` as
  "no permission" rather than "transient failure, retry."
  Distinguishing the two would let the loader retry once
  before redirecting — preserving the user's intended
  destination across an org switch.

# Suggestions

1. Add a cycle detector to `createPermissionGuard`: thread a
   `visitedPaths: Set<string>` through the redirect chain
   (via search params or session storage) and bail to a safe
   terminal if any path repeats. This kills the bug class.
2. Differentiate `undefined` (transient) from "permission
   denied" in `getActiveOrganizationUser()` so the loader can
   choose between retry and redirect.
3. Write the symmetry as a property test: for every loader L
   with a `customRedirectPath` P, assert P's loader does not
   redirect back into L's namespace. Catches the cycle at
   build time.

# Verdict

**Ship it.** The fix correctly stops the bleeding (no more
infinite loops, browser tab no longer crashes during org
switches), the regression test locks in the new behavior, and
the LOC delta is honest. The architectural follow-up — a real
cycle detector — is a separate piece of work, not a blocker.

# What I learned

A redirect-loop guard that only catches `redirectPath ===
currentPath` is not a redirect-loop guard; it's a self-redirect
guard. Real redirect cycles span ≥2 nodes, and the smallest
non-trivial cycle (A→B→A) is also the most common in practice
because it's exactly what you get when two routes have
"redirect to each other on permission failure" written by two
different people. The right primitive is path history (a
visited set), not pairwise equality. Until that exists, the
practical defense is to never wire `customRedirectPath` to a
route that itself runs a permission guard — let the fallback
do its job.
