# Context

OpenHands SaaS lets organization admins set org-default LLM
settings — model, base URL, API key — and propagate them to all
members. The save endpoint is `POST /api/organizations/llm`,
backed by `OrgLLMSettingsService`. Members can override with
their own BYOR ("bring your own resource") key, marked by a
`has_custom_llm_api_key` flag. When an admin chooses the
managed "OpenHands" provider, the backend is supposed to
generate a managed LiteLLM key, write it to every member, and
clear any stale BYOR markers.

# Problem

Four bugs that compound into "the org-defaults LLM page does
nothing useful":

1. **Payload silently dropped.** Frontend sent
   `agent_settings`; backend `OrgLLMSettingsUpdate` declared
   `agent_settings_diff`. Pydantic discarded the unknown
   keys, `has_updates()` returned `False`, the service
   short-circuited. No error, no log, just "save did
   nothing."
2. **No managed key on provider switch.** When an admin
   flipped to the OpenHands provider, the personal-save
   path's key-rotation logic had no equivalent on the org
   path. Members were "switched to OpenHands" without
   actually receiving an OpenHands key.
3. **Stale BYOR keys persisted.** Even when a managed key
   *was* somehow set, members' `has_custom_llm_api_key=True`
   flag was never reset, so `_get_effective_llm_api_key`
   would prefer the (now-orphaned) custom key over the
   managed one at read time.
4. **`GET` response diverged from stored state.**
   `model` leaked the SDK-internal `litellm_proxy/` prefix
   that the basic-view dropdown didn't recognize, and
   `api_key` returned either the masked placeholder or the
   raw value depending on path. Net effect: refresh the page
   and the form looked empty.

These four bugs hide each other. Fix any one in isolation and
the symptoms shift; the user still can't successfully save.

# Design

The PR rewrites the save and read paths as a single coherent
flow, with three structural moves worth calling out:

**(1) Wire-format alignment.** Rename
`OrgLLMSettingsUpdate.{agent,conversation}_settings_diff` →
`{agent,conversation}_settings`. The `_diff` suffix described an
internal storage concept that leaked into the public API. The
related `OrgUpdate` is renamed similarly, with frontend grep
verifying no caller uses the old names.

**(2) `_normalize_agent_settings` validator.** A pydantic
validator that:
- Lifts nested `agent_settings.llm.api_key` to the encrypted
  top-level `llm_api_key` column.
- Leaves `MASKED_API_KEY` (`'**********'`) in the JSON so
  serialized state stays in sync across propagation.
- **Skips re-lifting the masked placeholder.** This is the
  detail that prevents accidentally encrypting `**********`
  into the column on a no-op save — exactly the kind of bug
  that surfaces months later when someone tries to use the
  key and gets `**********` back.
- Auto-fills `base_url` via the new shared
  `resolve_llm_base_url(model, base_url, *, managed_proxy_url)`
  helper, which handles the three cases: empty → cleared,
  non-empty → pass-through, `None` + known model → inferred.

**(3) `_maybe_rotate_managed_llm_key_for_user`.** The missing
piece. On a managed-provider switch, this function:
- Reuses the acting user's current managed key if
  `verify_existing_key` says it's still valid.
- Otherwise generates fresh via `LiteLlmManager.generate_key`,
  with `{'type': 'openhands'}` metadata for `openhands/` models
  and `get_openhands_cloud_key_alias` + `delete_key_by_alias`
  for managed-proxy URL models.
- Gated on `EDIT_LLM_SETTINGS` permission (admins/owners only).

The service then makes a *single*
`update_all_members_llm_settings_async` call writing the
diff, the new `llm_api_key`, and `has_custom_llm_api_key=False`
to every member row in one DB pass. This atomicity matters: a
partial propagation would leave the org in a state where some
members read the new key and some still hit the old BYOR
column.

**(4) Read-path symmetry.** `OrgLLMSettingsResponse.from_org`
denormalizes `litellm_proxy/X` → `openhands/X` so the basic-view
dropdown can match the OSS settings response shape, and nulls
`api_key` so neither the raw secret nor the masked marker
leaks. Frontend recognizes "managed mode" via
`KNOWN_PROVIDER_DEFAULT_BASE_URLS` rather than by sniffing the
key.

# Risks

- **Cross-package coupling.** The shared `resolve_llm_base_url`
  and `MASKED_API_KEY` now live in `openhands/utils/llm.py` and
  are consumed by both the personal-save router and the
  enterprise service. A change to either constant ripples
  across both surfaces. Worth a constant import test.
- **Single DB write for propagation.** Right call for
  consistency, but on a 1000-member org this is a wide
  multi-row update. Make sure the DB has the right index for
  org-id lookups; a missing index turns this into a slow
  full-scan on every settings save.
- **`SecretStr` typing change** for `OrgMemberLLMSettings.llm_api_key`.
  Pydantic auto-wraps `str` inputs, so callers passing plain
  strings still work, but any test that compared the field
  with `==` against a plain string will now compare against a
  `SecretStr` and silently fail unless `.get_secret_value()`
  is used. Test diffs (`test_org_member_store.py` +6/-6,
  `test_org_service.py` +7/-7) suggest this was caught
  comprehensively, but third-party consumers of the model
  could be surprised.
- **No-op save protection depends on masked-placeholder
  recognition.** If the frontend ever changes the masked
  string (e.g., to `'****'`), the validator will lift the
  new placeholder into the encrypted column. A constant
  shared across frontend and backend, or a regex matching any
  all-asterisk string, would be more robust.

# Suggestions

1. Add an integration test: admin saves managed-provider
   settings → assert every member row has the new key, the
   masked JSON value, and `has_custom_llm_api_key=False`.
   Unit tests cover the service; only an integration test
   catches a future routing bug that bypasses the service.
2. Log a structured event whenever `has_updates()` returns
   `False` on an `update`-shaped request. The original bug
   was invisible specifically because this short-circuit was
   silent; one log line per silent no-op would have surfaced
   it in days.
3. Move `MASKED_API_KEY` to a place both frontend and backend
   can import (or codegen it into the TypeScript types) so
   they cannot drift.
4. Consider an explicit `clear_custom_keys: bool` field on the
   org update payload rather than implicitly resetting
   `has_custom_llm_api_key=False` on every save. Implicit
   resets are correct here but surprising; an explicit flag
   makes the contract auditable.

# Verdict

**Ship it.** This is a serious compound-bug fix with
appropriate test density (the service-test file alone is +594
LOC). The structural moves — wire-format alignment, masked-key
no-op protection, single-DB propagation, symmetric
read/write — are all the right shape. The remaining concerns
are operational (indexing, logging) and constants-management,
not blockers.

# What I learned

When a save endpoint "does nothing" silently, the bug is
almost never in one place. Here it took four independent
defects in the same flow — naming mismatch, missing rotation,
flag not reset, response shape divergence — to fully break the
feature, and *each one alone* would have produced a survivable
half-broken state. The lesson for review: when a fix touches
one symptom (say, "managed key isn't being generated"), grep
the entire flow for *all* the places that symptom could be
masked by another bug. And: any save endpoint where
`has_updates()`-style short-circuits exist must log on the
no-op path. Silent no-ops are how compound bugs accumulate.
