# Context

OpenHands has a long-running tension between *personal* settings
(per-user `agent_settings`, `conversation_settings`) and
*organization-level* settings (org defaults that members inherit).
Recent history: PR #14044 fixed an org-defaults save bug, but the
follow-up branch then drifted into building parallel
infrastructure — separate `OrgLLM*` service/store layers,
separate write models — that reintroduced the contract mismatch
between frontend and backend. This PR rolls that drift back and
re-imposes a single shape: **diff-only nested settings writes,
through a single org write model**.

Net diff is dramatically negative (1247 additions vs 2355
deletions, -1108 lines). That's a strong signal — when a "fix"
PR is net-deleting an entire service file (`org_llm_settings_service.py`,
-277), an entire store file (`org_llm_settings_store.py`, -85), and
a 813-line test file alongside, the actual content of the PR is
"un-build the wrong abstraction".

# Problem

Three concrete issues being resolved together:

1. **Two parallel write models for org settings.** `OrgUpdate`
   was the original; `OrgLLMSettings*` had been added on top.
   Frontend ended up calling whichever the recent PR exposed,
   backend ended up validating against whichever the recent
   route accepted, and the two surfaces were drifting. Symptom
   for users: changes saved in the LLM settings tab didn't
   appear when the org was reloaded, because the read path and
   the write path were against different stores.
2. **Inconsistent nested-settings payload shape.** Some endpoints
   accepted `{agent_settings: {...}}` (the full nested object),
   others accepted `{agent_settings_diff: {...}}` (a sparse
   diff). The frontend had to know which endpoint expected
   which shape, and got it wrong on at least three call sites
   (use-add-mcp-server, use-delete-mcp-server, use-update-mcp-server).
3. **Validation order.** The pre-existing code validated the
   sparse diff against the full settings schema, which rejected
   legal diffs (e.g. `{agent_settings_diff: {model: "gpt-5"}}`
   failed because the diff was missing required fields the
   full schema demanded). Validation needs to happen *after*
   merging the diff into the existing settings, not on the
   sparse diff itself.

# Design

The PR enforces three invariants:

1. **One org write model.** `OrgUpdate` is the single shape.
   The deprecated `/api/organizations/llm` endpoints survive as
   thin compatibility wrappers that translate to `OrgUpdate`
   under the hood. Old clients keep working; no new code paths
   fork off.
2. **`*_diff` is the only nested-settings write contract.**
   `agent_settings_diff` and `conversation_settings_diff` are
   the names; full nested keys (`agent_settings`,
   `conversation_settings`) on `POST /api/v1/settings` now
   return **422** instead of being silently accepted. Hard
   rejection is the right move — silent acceptance of the
   wrong shape was the original drift mechanism.
3. **Validate after merge.** The new contract: receive sparse
   diff → merge into current settings → validate the merged
   result. Tests in `tests/unit/storage/data_models/test_settings.py`
   (+34/-6) and `tests/unit/app_server/test_settings_api.py`
   (+32/-5) pin this.

The frontend mirrors all three: every settings save and MCP
mutation now uses the `*_diff` shape, mocks updated, and the
test suite (`__tests__/routes/llm-settings.test.tsx` +87/-51,
`__tests__/routes/mcp-settings.test.tsx` +22/-6) follows.

The `agent_settings_diff` naming choice is good — it makes the
sparse semantics visible at the call site, instead of relying
on the reader to know that "this is a partial update" from
context. Compare to the alternative (`agent_settings_partial`,
`agent_settings_patch`): `_diff` is the most explicit about
"what's in here is *what changed*, not *what should be*".

# Risks

- **422 on legacy payload shape is a breaking change.** Any
  external API client that was POSTing `agent_settings: {...}`
  will start getting 422s. The PR description doesn't note a
  deprecation period or a CHANGELOG entry. For an enterprise
  product, that's a real exposure — at minimum, the 422 should
  carry a body that says "use `agent_settings_diff` instead".
  The current rejection appears unconditional.
- **Net-negative refactor risks orphaning data.** Removing
  `org_llm_settings_store.py` (-85) and the `OrgLLM*` service
  layer is fine *if* no production data was being written
  through paths that landed in different tables. The PR doesn't
  surface a migration check. If `OrgLLM*` had its own table,
  there could be settings written before this PR that are now
  unreachable.
- **45 files in one PR is a lot.** The frontend, backend,
  enterprise, and shared `openhands/` package are all touched.
  Atomic merge is the right call (otherwise one half ships
  with a contract mismatch), but it makes review extremely
  hard. Splitting into "delete the wrong abstraction" + "unify
  on diff-only" + "frontend mirror" would have been
  easier — at the cost of the intermediate states being
  broken.
- **"validate after merge" semantics need a fast-fail.**
  If the merged settings fail validation, the user gets an
  error about *fields they didn't change*. Worth verifying
  the error message points at the field, not at the diff key.
- **Compatibility wrappers around `/api/organizations/llm` are
  permanent debt** unless they have an explicit removal date.
  "Deprecated as compat wrapper" tends to live forever. A
  follow-up PR with a `Deprecation` header + sunset date would
  close the loop.

# Suggestions

1. Add a `Deprecation` / `Sunset` header to the
   `/api/organizations/llm` compat wrappers, plus a logged
   warning per call.
2. The 422 response for legacy `agent_settings` /
   `conversation_settings` keys should include a body that
   names `agent_settings_diff` as the migration target.
3. Add an explicit data migration check or a release note: did
   `OrgLLM*` ever write to a separate table? If yes, document
   the reconciliation path.
4. Consider extracting the diff-merge-validate flow into a
   single `apply_settings_diff(current, diff) -> Result<Settings>`
   helper that's the *only* path used by all settings writes
   (org, personal, MCP). That way the next "let me add a new
   settings endpoint" PR can't reintroduce the drift.

# Verdict

**Ship it, with follow-ups filed.** Net-negative refactor that
restores a single contract is exactly the right move when an
abstraction has metastasized. The 422 hard-rejection of legacy
shapes is the right signal — half-measures are how the original
drift happened. File the deprecation header / migration note as
follow-ups.

# What I learned

The "Y-shaped abstraction drift" pattern is worth naming: a
core write model exists; someone adds an adjacent specialized
model (`OrgLLM*`) for one slice of the surface; the two slowly
develop independent persistence, validation, and frontend call
sites; then the two surfaces drift and the user sees
inconsistent behavior. The fix is always the same — collapse
back to one write model, keep the specialized endpoints as
thin wrappers (or delete them outright if no external client
depends on them), and *validate after merge* not before.

The reusable insight is the diff-only nested-write contract.
Every settings system that grows to "save the whole settings
blob" eventually hits the lost-update problem (two clients
overwriting each other's unrelated changes). Going to diff-only
prevents that and removes ambiguity about "did the client
intend to clear this field, or did they just not include it?"
The naming convention `*_diff` makes that intent visible at
every call site. Worth replicating in any settings-y API
designed today.

The third lesson is on PR shape. A 45-file, +1247/-2355 PR is
brutal to review but **correct as one atomic unit** because
splitting it would land intermediate states where frontend
expects diff-only and backend still accepts both, or vice
versa. The right tradeoff for contract-restoration PRs is
"large but atomic". For *additive* PRs, the opposite tradeoff
holds — break them up. Knowing which kind of PR you're writing
should drive the size decision.
