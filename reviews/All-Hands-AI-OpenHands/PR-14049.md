# Context

OpenHands has a long-running agent that may take minutes to
complete a task. Users routinely tab away while the agent works.
The product had two notification primitives — a `notification.mp3`
audible cue and a flashing browser-tab title — that fired when
the agent reached a terminal/wait state. A `enable_sound_notifications`
toggle in Application Settings governed both.

Commit `1523555ee` (`refactor(frontend): remove dead code (#10839)`)
deleted the notification utilities. The toggle in the settings
UI was *not* removed. Result: the setting silently did nothing
for an unknown number of releases, and tab-away users had no
indication when the agent finished, awaited input, or asked for
confirmation.

# Problem

Two compounding failures here, only one of which is the
notification regression:

1. **The feature itself is gone.** The audio cue and the
   title-flash both vanished, including the `notification.mp3`
   asset and the `browser-tab.ts` utility.
2. **The settings UI lies.** A toggle that no longer wires to
   anything is worse than no toggle — users believe the feature
   is on and disabled, and may file silent dissatisfaction
   instead of a bug.

The "dead code removal" PR was a real cleanup; the bug is that
a usage-edge (the settings toggle) wasn't traced back to its
producer. This is the classic "delete the consumer, leave the
producer" inverse: here it was "delete the producer, leave the
consumer (UI)."

# Design

The PR is restoration with one structural improvement: the new
`useAgentNotification` hook integrates with the **V1** agent
state system, not the legacy state model the original code was
written against. Concretely:

- `browser-tab.ts` (+42) — pure utility: alternates document
  title on an interval, with an optional favicon swap, and a
  `stop` that restores the original title. No React deps,
  trivially testable (+77 lines of tests).
- `useAgentNotification.ts` (+76) — subscribes to V1 agent state,
  fires audio + tab-flash on the three terminal/wait states
  (`AWAITING_USER_INPUT`, `FINISHED`, `AWAITING_USER_CONFIRMATION`),
  and gates the entire effect on the `enable_sound_notifications`
  setting. Stops flashing on tab focus.
- `AgentStatus` integration (+4) — the hook is mounted exactly
  where the agent state is already being read for visual
  presentation, which is the right co-location.

The test budget is the load-bearing element: 259 LOC of tests
against 165 LOC of new code (1.5×). That ratio is appropriate
for restored functionality — the bug class here is "feature
silently regresses again," and the test suite is the
permanence of the fix.

# Risks

- **Audio autoplay policies.** Browsers block `<audio>.play()`
  without prior user interaction. The hook should be safe in
  practice (the user almost certainly clicked something to
  start the agent), but a first-conversation, never-clicked
  case could throw an unhandled `NotAllowedError`. Worth
  catching and logging at debug.
- **Title-flash interval is not coalesced across tabs.** If a
  user opens two OpenHands tabs both reaching `FINISHED`,
  they'll both flash and both play sound. Annoying but not
  incorrect.
- **No favicon dot for in-tab notifications.** The PR includes
  favicon swap support but does not appear to wire a
  visible-tab indicator (e.g., a small badge when the agent
  finishes while the tab is foreground). Future enhancement,
  not a regression.
- **`.agents/skills/custom-codereview-guide.md` (+47)** ships
  with this PR. Out of scope for the user-facing fix; should
  arguably be a separate commit, but harmless.

# Suggestions

1. Add a Playwright/end-to-end smoke that toggles the setting,
   forces an `AWAITING_USER_INPUT`, and asserts the document
   title actually changed. Unit tests cover the hook; only an
   E2E test catches the wiring regression class that caused
   this bug originally.
2. When the setting is OFF, show a small "Notifications are
   disabled" hint near the setting label so users who turned
   it off years ago and forgot have a path to discoverability.
3. Wrap `audio.play()` in a try/catch and log the
   `NotAllowedError` once per session — better than a console
   error mystery for users on autoplay-strict browsers.
4. Long-term: a CI rule that flags any settings toggle whose
   bound state isn't read by at least one consumer would have
   prevented the original regression. Cheap static check.

# Verdict

**Ship it.** The restoration is faithful to the original
behavior, integrates with the current state model rather than
porting old plumbing, and the test ratio means the next dead-code
sweep will fail loudly rather than silently re-deleting the
feature.

# What I learned

A user-facing setting whose bound state is read by *zero*
producers is a worse bug than a missing setting. The fix is
process: any "remove dead code" PR should grep for setting
keys, env vars, and feature flags in the deleted blocks and
either remove the corresponding UI or call out the producer
gap explicitly. This PR also illustrates the right shape of a
restoration: don't port the old code verbatim — re-implement
against the current architecture (V1 agent state) and bring
heavier tests than the original carried, so the bug class is
permanently retired rather than re-armed.
