# Context

SQLAlchemy 2.0 introduced `mapped_column()` + `Mapped[T]` typed
mappings as the modern alternative to legacy `Column()` (with no
type info). The OpenHands codebase had been migrating
incrementally — PRs #13846–#13859 covered earlier modules. This
PR finishes the migration for the *core* application server
SQLAlchemy models: 5 files, ~59 fixed mypy `[var-annotated]`
errors. The classic "incremental cleanup, finishing wave" PR.

# Problem

Three concrete pain points the legacy `Column()` style created:

1. **No static type information.** Code that reads
   `model.created_at` had no way for mypy to know the value is
   a `datetime`. The fallback was `# type: ignore` comments
   sprinkled across call sites — both ugly and a future-bug
   factory (the `# type: ignore` outlives any change in the
   actual column type).
2. **`Mapped[Optional[T]]` ambiguity.** With legacy `Column()`,
   nullability and Python-level optionality were derived from
   two different sources (DDL `nullable=` vs the implicit
   `Optional` of the Python attribute). With
   `mapped_column()`, the `Mapped[Optional[T]]` annotation is
   the single source of truth and mypy enforces it.
3. **Stale imports.** The pre-PR file imported `Column`,
   `Boolean`, `Float`, `Integer`, `UUID as SQLUUID` —
   most unused once the `mapped_column()` migration is done,
   because Python type hints (`bool`, `float`, `int`,
   `uuid.UUID`) carry that information now.

# Design

The migration recipe is mechanical, applied uniformly across all
five files:

```python
# Before
from sqlalchemy import Column, String, Integer, Boolean
from sqlalchemy.dialects.postgresql import UUID as SQLUUID
class StoredFoo(Base):
    id = Column(SQLUUID(as_uuid=True), primary_key=True)  # type: ignore
    name = Column(String, nullable=False)                  # type: ignore
    count = Column(Integer, default=0)                     # type: ignore

# After
from sqlalchemy.orm import Mapped, mapped_column
from uuid import UUID
class StoredFoo(Base):
    id: Mapped[UUID] = mapped_column(primary_key=True)
    name: Mapped[str] = mapped_column(nullable=False)
    count: Mapped[int] = mapped_column(default=0)
```

Five things happen at once on each row:

1. The Python type annotation (`Mapped[T]`) replaces the
   SQLAlchemy first-arg type.
2. The `# type: ignore` comments go away.
3. Import surface shrinks (drop `Column`, `Integer`, etc.,
   add `mapped_column`, `Mapped`).
4. `datetime` and `UUID` are now imported as actual Python
   types because they appear in the `Mapped[...]` annotation.
5. `SQLUUID` from postgres-specific dialect is replaced by
   plain `uuid.UUID` since SQLAlchemy 2.0 picks the dialect
   automatically.

The `+124/-76` size split is consistent with this — most lines
change in place, some imports collapse, and a few `Mapped[T]`
annotations span an extra line (`name: Mapped[str | None] =
mapped_column(...)` is sometimes longer than `name = Column(
String, nullable=True)`).

# Risks

- **No runtime test coverage in the PR.** The PR says "CI will
  validate unit tests". For a pure type-annotation migration
  that's *almost* fine — the migration *should* be a no-op at
  runtime — but there are two SQLAlchemy 2.0 footguns that
  cannot be caught by mypy:
  - Default-value behavior changed subtly between
    `Column(default=...)` and `mapped_column(default=...)`
    when the type is mutable (e.g. `default=[]` or
    `default={}`). Worth scanning all five files for this
    shape.
  - `mapped_column(server_default=...)` and `default=...` are
    distinct; if any column was relying on a Python-side
    default that the legacy code left implicit, the new code
    might silently drop it. The mechanical rewrite is unlikely
    to hit this, but a pass to confirm zero columns with
    Python-mutable defaults would close the question.
- **`type: ignore` removal can mask new errors.** When `# type:
  ignore` is deleted, mypy starts re-checking that line. If the
  new `Mapped[T]` annotation is wrong (e.g. `Mapped[int]` for
  what's actually a `Mapped[Optional[int]]`), call sites get
  *new* mypy errors that were previously suppressed. The PR's
  "59 errors fixed" is a net number — without seeing the
  per-line breakdown it's not certain none of them are
  *introduced*.
- **Mapped[UUID]** with `uuid.UUID` (not the dialect-specific
  `SQLUUID`) only works correctly if SQLAlchemy 2.0's UUID
  type-coercion is enabled by the engine config. On postgres
  it's automatic; on SQLite (which test suites often use) the
  coercion may differ. Worth running the SQLite-backed tests
  specifically, not just postgres-backed.
- **No CHANGELOG entry.** The migration is internal so this is
  fine, but operators who maintain custom subclasses of these
  models (rare but possible in enterprise deployments) would
  benefit from a heads-up.

# Suggestions

1. Grep across the five files for `default=[]`, `default={}`,
   `default=set()` — any mutable Python-side default needs a
   `default_factory=list` migration to be safe under the new
   API.
2. Run the test suite under SQLite specifically (not just the
   default postgres path) to catch UUID-coercion differences.
3. Add a tiny round-trip test for at least one model in each
   of the 5 files — instantiate, persist, fetch, assert
   field values match. That catches any subtle default /
   coercion / nullability regression the type-only migration
   could introduce. Cheap insurance.
4. Future PRs of this shape should land alongside a
   `mypy --strict` run on the changed files in CI, with the
   *before* error count and *after* error count printed in the
   PR body. The current "59 errors fixed" is unverifiable
   without that.

# Verdict

**Ship it.** Mechanical migration, consistent across 5 files,
finishes a long-running cleanup. The recipe is right and the
import-shrinkage is a nice bonus. The risks (mutable defaults,
SQLite coercion) are real but low-probability for the specific
columns being migrated. File a follow-up to add the round-trip
tests if not already present.

# What I learned

Type-annotation migrations are a deceptive class of PR — they
*look* mechanical and risk-free because they don't change
runtime behavior intentionally. The hidden risk is that the
old API was permissive about a thing the new API is strict
about, and the migration silently changes that thing. The
canonical examples are:

- Mutable defaults (`default=[]` vs `default_factory=list`)
- Nullability inference (the legacy `Column` was nullable by
  default; `mapped_column(Optional[T])` is more explicit)
- Type coercion at the dialect boundary (UUID, JSON, ARRAY)

For any "modernize the typing" PR, the right test discipline is
*not* "the unit tests pass" (they will — the change is type-only)
but a small targeted *round-trip* test per migrated model. The
round-trip catches what mypy can't: defaults, coercion,
nullability semantics. Worth standardizing as the
"mapped_column migration test contract" so future waves of this
work don't have to re-derive what to test.

The other lesson is the value of incremental cleanup PRs. This
is the last wave of a multi-PR migration (#13846–#13859 +
#14065). Each PR was small enough to review individually,
the recipe was uniform across all of them, and the cumulative
result is "the entire core SQLAlchemy surface is now typed".
That's a much better arc than one mega-PR that touches every
model at once. The lesson: when a migration is mechanical and
file-scoped, *split by directory boundaries and ship per
directory*. The reviewer cost stays bounded, the rollback
surface stays small, and the cumulative win is the same.
