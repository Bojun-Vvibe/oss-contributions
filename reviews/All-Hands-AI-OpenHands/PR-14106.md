# Context

`OpenHands` is mid-migration from a V0 architecture (built around
`openhands.controller`, `openhands.memory`, `openhands.microagent`,
classic event streams) to a V1 architecture rooted in the
`openhands.app_server` and SDK packages. The V0 packages are
slated for deletion. The constraint: nothing outside the V0
packages can still import them when the deletion lands, otherwise
the rip-out PR will be a 50-file diff that nobody can review.

This PR is one of several "carve off external references"
preparation PRs. It targets `openhands.memory` specifically and
takes care of three holdouts: `app_server/user/skills_router.py`,
`core/setup.py`, and `mcp/utils.py`. The PR description lists the
remaining external references for `controller` (4 files) and
`microagent` (5 files) as future work.

# Problem

The mechanical problem is straightforward: find every external
import of `openhands.memory.*` and replace it with something
self-contained. The interesting problem is *what* the
replacement looks like in each case, because each holdout
depends on the V0 module for a different reason.

1. `skills_router.py` imports two path constants
   (`GLOBAL_MICROAGENTS_DIR`, `USER_MICROAGENTS_DIR`) and re-uses
   them as `GLOBAL_SKILLS_DIR` and `USER_SKILLS_DIR`. The
   comment "Re-use V0 path constants (single source of truth)"
   makes the original intent explicit. Removing the dependency
   means picking a *new* source of truth.

2. `core/setup.py` imports a small zoo of V0 types (`AgentController`,
   `Agent`, `State`, `Memory`, `BaseMicroagent`, `LLMRegistry`,
   `EventStream`, `Event`, `ConversationStats`) to define three
   helper functions: `create_memory`, `create_agent`,
   `create_controller`. Removing the imports means deciding
   whether the helpers are still used.

3. `mcp/utils.py` (not shown in the diff snippet I read but
   referenced in the PR description) takes a `Memory` parameter
   somewhere. Removing the dependency means changing the
   function signature.

Each replacement has to leave V1 behaviour unchanged. This is
the part where "preparation refactors" most often go wrong:
someone deletes an import, adjusts the call sites, and silently
changes behaviour because the V0 code path was load-bearing in a
way that wasn't obvious.

# Design

The three replacements take three different shapes — appropriate
because the three problems are different.

**`skills_router.py`: switch the source of truth from V0 to
the package's own location.**

```python
# before
from openhands.memory.memory import GLOBAL_MICROAGENTS_DIR, USER_MICROAGENTS_DIR

GLOBAL_SKILLS_DIR = Path(GLOBAL_MICROAGENTS_DIR)
USER_SKILLS_DIR = Path(USER_MICROAGENTS_DIR)

# after
import openhands

GLOBAL_SKILLS_DIR = Path(os.path.dirname(openhands.__file__)) / 'skills'
USER_SKILLS_DIR = Path.home() / '.openhands' / 'microagents'
```

This is the most subtle change in the PR. The before/after paths
are not literally equal — they cannot be, because the V0
`GLOBAL_MICROAGENTS_DIR` pointed at a `microagents/` directory
inside the V0 memory package, and the V1 replacement points at
a `skills/` directory inside the top-level `openhands` package.
The PR is implicitly asserting that the V1 surface has its own
`skills/` directory at that location, distinct from V0's
microagents directory.

For `USER_SKILLS_DIR`, the path stays at `~/.openhands/microagents`
— so user-installed skills continue to live in the
`microagents` directory on disk even after V0 is deleted. That
is good for backwards compatibility (existing users' skills
keep working) and slightly confusing for future readers (the
constant is `USER_SKILLS_DIR` but the path is `microagents`).
A short comment would help. See Suggestions.

**`core/setup.py`: delete the helpers entirely.**

The diff removes 100+ lines: `create_memory`, `create_agent`,
`create_controller`. The remaining file keeps
`initialize_repository_for_runtime` and the imports it needs
(`OpenHandsConfig`, `Runtime`, provider types). The implication
is: *no V1 code calls these helpers any more*. The V1 app server
constructs agents, controllers, and memory through its own
factories.

This is the cleanest outcome for the V0-removal goal — the
external dependency is gone because the *function* is gone, not
because the function was rewritten to avoid the import. It also
means the PR can be reviewed by checking one thing: are there any
remaining callers of `create_memory`, `create_agent`, or
`create_controller` in the V1 codebase? If yes, the PR breaks
them. If no, the deletion is safe and the PR is essentially
risk-free for these helpers.

A quick `grep` of the PR's file list shows the helpers are not
re-defined elsewhere, and the PR description doesn't mention
breaking changes — so the PR author has presumably already done
that grep. A reviewer should re-do it independently before
merging.

**`mcp/utils.py`: drop a parameter.**

The diff snippet I have doesn't show this file, but the PR
description says "Remove Memory dependency from openhands/mcp/utils.py".
Without seeing the diff, the most likely shape is: a function
that took `memory: Memory` as an argument, used one or two of
its accessors (probably to register MCP servers as memory
sources), and now either takes the underlying data directly or
no longer registers anything because V1 has its own MCP
registration path.

This is the highest-risk part of the PR by default — removing a
parameter changes the function signature, which is a contract
change for callers. Reviewers should check the call sites.

# Risks

**Path-constant divergence is invisible until runtime.** The
`GLOBAL_SKILLS_DIR` change moves from "wherever V0 said" to
`<package>/skills`. If the V1 package layout doesn't actually
have a `skills/` directory in the install — perhaps it's only
populated via a post-install step, or only exists in dev
checkouts — the router will silently return empty skill lists in
production. A test that spins up the router and asserts at least
the bundled skills resolve would catch this. Out-of-the-box
`Path(os.path.dirname(openhands.__file__)) / 'skills'` is fine
in the dev tree but worth confirming for the wheel build.

**`USER_SKILLS_DIR` is hard-coded to `~/.openhands/microagents`.**
Two issues:
- The directory name `microagents` is V0 terminology. If V1
  reads "skills", users who install a skill called `foo` and
  expect to find it under `~/.openhands/skills/foo` will be
  confused.
- The path bypasses `XDG_DATA_HOME` and `HOME` env-var overrides
  that are convention on Linux. For a server-deployed `OpenHands`
  the user-skills directory probably belongs under whatever the
  app's data root is, not under `~`.

The V0 path was likely identical so this isn't a regression, but
the PR is the natural moment to revisit either of these.

**`os` import was added to `skills_router.py`.** The diff adds
`import os` at the top. Two characters of nit: `os.path.dirname`
on a `__file__` is fine, but `Path(openhands.__file__).parent`
is more idiomatic and removes the new `os` import. If this file
already used `pathlib.Path` everywhere, mixing in `os.path` is a
small consistency loss.

**`create_memory` / `create_agent` / `create_controller` deletion
risk.** If *any* external integration test, example, or doc
snippet referenced these helpers, those will break silently
(import errors at import time). A `grep -R "create_memory\|create_agent\|create_controller" docs/ tests/ examples/`
would surface them. The PR description says obsolete test files
were removed (`tests/runtime/test_microagent.py`,
`tests/runtime/test_mcp_action.py`) but doesn't enumerate which
non-deleted tests still pass.

**The `mcp/utils.py` change is invisible in the PR description.**
"Remove Memory dependency from openhands/mcp/utils.py" is a
one-liner but the diff impact isn't summarised. Reviewers should
explicitly read that file in the diff.

**No changelog entry.** This PR removes public-ish helpers
(`create_memory`, `create_agent`, `create_controller` were
top-level `from openhands.core.setup import ...`) without a
deprecation cycle. For a V0→V1 migration that may be
intentional, but a `BREAKING_CHANGES.md` entry — even just "V0
removal preparation, see PR #14106" — would help anyone whose
external scripts import these names.

# Suggestions

1. Replace `Path(os.path.dirname(openhands.__file__)) / 'skills'`
   with `Path(openhands.__file__).parent / 'skills'`. Drops the
   `os` import, keeps the file `pathlib`-only, idiomatic.

2. Add a short comment over `USER_SKILLS_DIR` explaining why the
   directory name on disk is `microagents` (V0 backwards
   compat) when the constant is `USER_SKILLS_DIR`. Two-line
   comment, prevents the next person from "fixing" it.

3. In the PR description, add the explicit list of external
   call sites that were checked for `create_memory`,
   `create_agent`, `create_controller`. Even just
   "grepped `docs/`, `examples/`, `tests/` — no remaining
   callers" gives reviewers a single thing to verify.

4. For the `mcp/utils.py` change, add a one-line summary in the
   PR body of what the function signature looked like before and
   after. The "Remove Memory dependency" framing tells reviewers
   *why* but not *what* — and "what" is the part that breaks
   callers.

5. Consider a follow-up PR (or this one, if scope-creep is
   acceptable) that adds a single integration test:
   `test_skills_router_returns_bundled_skills` — assert at least
   one skill in `GLOBAL_SKILLS_DIR` is enumerated. This would
   catch the "wheel build doesn't include `skills/`" failure
   mode mentioned above.

6. The PR mentions "preparation for removing V0 after migrating
   to V1" but does not link to the umbrella issue. A single
   `#NNNN` reference would let future readers walk the chain of
   prep PRs in order.

# Verdict

Approve with the caveats. The changes themselves are correct
and small. The two things I'd want before merge: (a) the
`os` → `pathlib` cleanup (suggestion 1), (b) the explicit
"no remaining callers" attestation in the PR description
(suggestion 3). Neither blocks; both raise confidence.

The structural risk is the `USER_SKILLS_DIR` semantics — if the
intent is "V0 microagents directory, kept for compat" then the
comment in suggestion 2 is the right fix; if the intent is "V1
skills directory, but I forgot to rename" then the path string
is wrong and needs to be `~/.openhands/skills`. The PR doesn't
say which. A reviewer with V1 context should resolve this
before merge.

# What I learned

The general technique here is "prepare-for-deletion is its own
PR family". You don't delete the V0 packages in one giant rip;
you walk the dependency graph backwards from the leaves,
shipping one PR per dependent module that swaps the dependency
for something self-contained, and only when the in-degree of the
V0 packages drops to zero do you do the actual deletion. This
PR is doing exactly that for the `openhands.memory` slice.

The interesting part is the *shape* of the swaps. There are
three patterns visible in this one PR:

- **Re-host the constant.** When you depend on `OldPkg.X` for a
  literal value (a path, a string, a number), the cheapest swap
  is to move the literal into the consumer module and keep the
  same value. (`skills_router.py`.) The risk is that the literal
  diverges silently — see the `USER_SKILLS_DIR` discussion.

- **Delete the wrapper.** When you depend on `OldPkg.X` to
  construct something that the new architecture constructs
  differently, often the wrapper function in the consuming module
  is itself dead code. The cheapest swap is to delete the
  wrapper. (`core/setup.py`.) The risk is missed callers; the
  mitigation is `grep`.

- **Drop the parameter.** When you depend on `OldPkg.X` because
  a function takes one as input, sometimes the function doesn't
  actually need it any more (the new architecture has different
  entry points). The cheapest swap is to drop the parameter.
  (`mcp/utils.py`, presumed.) The risk is contract change for
  callers; the mitigation is to read every call site.

Each of these patterns has a different review checklist. Mixing
all three in a single PR is fine if the PR author is explicit
about which pattern applies to which file — and the PR
description here is *almost* explicit. Adding the call-site
attestation for the deletes and the before/after signature for
the parameter-drop would push it from "almost explicit" to
"reviewer-friendly".

The deeper lesson, applicable to any V1/V2 rewrite: the
*cleanup* PRs that prepare for the rewrite-package deletion are
where the architectural assumptions get encoded. The
`USER_SKILLS_DIR` decision — V0 microagents path vs V1 skills
path — is a real product decision that this refactor PR is
making by default. Worth flagging up to the V1 owners explicitly
rather than burying it in a single-line constant change.
