# Context

OpenHands had a recent PR (#13952) that cleaned up "deprecated"
V0 FastAPI endpoints, assuming each had a V1 replacement. The
cleanup was correct in *intent* but executed against an
incorrect assumption: not every V0 endpoint that wasn't
*explicitly* marked `deprecated=True` actually had a V1
sibling. `/api/user/git-organizations` got swept up in the
cleanup despite being a live endpoint with no V1 equivalent.

The user-visible regression: the Git Conversation Routing UI —
the tab where org admins claim Git organizations to route them
to specific OpenHands orgs — broke entirely. Every page load
404'd; every "claim" or "disconnect" action failed.

# Problem

The bug is one missing migration step plus one wrong assumption.
Concretely:

1. **Wrong assumption.** PR #13952 used `deprecated=True` as the
   filter for "this is safe to delete". That's a fine heuristic
   if the codebase is disciplined about marking endpoints as
   they get superseded, but it's a *trailing* indicator —
   anyone who shipped a new V1 sibling without remembering to
   mark the V0 deprecated leaves a landmine for the cleanup PR.
2. **Missing V1.** When `/api/user/git-organizations` should
   have been migrated to V1 (probably as part of an earlier
   batch), the migration was simply missed. The provider-side
   methods (`ProviderHandler.get_github_organizations()`,
   `get_gitlab_groups()`, `get_bitbucket_workspaces()`) all
   still exist intact in `openhands/integrations/provider.py`,
   so this was never an "intentional removal of capability" —
   the route layer just got dropped on the floor.
3. **Frontend silent on 404.** The frontend kept calling the
   V0 path and silently showed an empty list. From a user's
   perspective, "no Git organizations are connected"
   (indistinguishable from "the endpoint is gone").

# Design

A thin SAAS-only V1 wrapper, +165 lines:

```python
# enterprise/server/routes/users_v1.py (+44/-2)
@router.get("/git-organizations", response_model=GitOrganizationsResponse)
async def get_git_organizations(
    provider: Provider,
    handler: ProviderHandler = Depends(get_provider_handler),
) -> GitOrganizationsResponse:
    if not handler.has_token_for(provider):
        raise HTTPException(status_code=401, detail=...)
    if provider == Provider.GITHUB:
        return await handler.get_github_organizations()
    if provider == Provider.GITLAB:
        return await handler.get_gitlab_groups()
    if provider == Provider.BITBUCKET:
        return await handler.get_bitbucket_workspaces()
    raise HTTPException(status_code=400, detail="unsupported provider")
```

Thin is the right shape — the provider-side methods already
do the heavy lifting (token resolution, API calls, response
shaping). The route is pure dispatch + auth + error mapping.

The frontend change is one line:

```ts
// frontend/src/api/user-service/user-service.api.ts (+1/-1)
- const url = "/api/user/git-organizations";
+ const url = "/api/v1/users/git-organizations";
```

The unit test (+110/-0) covers three cases:
1. **401** when the user has no token for the requested provider.
2. **400** for unsupported provider.
3. **Happy path** for GitHub, GitLab, Bitbucket each.

That's the right test surface for a thin route wrapper —
exercise the dispatch branches and the two error responses.
No need to test the underlying provider call (it has its own
tests).

The route is correctly scoped to **SAAS-only** (lives under
`enterprise/server/routes/users_v1.py`). The provider
abstractions are in the open-source `openhands/integrations/`
tree, but the route-level wiring is enterprise — appropriate.

# Risks

- **No deprecation/sunset on the V0 path.** The V0 endpoint is
  *gone* now (deleted in #13952), so the question is moot in
  this direction. But the *pattern* — "a cleanup PR uses
  `deprecated=True` as the only filter" — is what allowed this
  bug. A guardrail in CI that compares the route table before/
  after a cleanup PR and prints "these routes are being deleted,
  here are their callers in the frontend" would prevent the
  next instance.
- **No integration test that exercises the route through the
  HTTP layer.** The unit test mocks `ProviderHandler` and tests
  the route-handler function directly. A FastAPI `TestClient`
  test that hits `/api/v1/users/git-organizations` over HTTP
  would catch routing-prefix mistakes (the kind of bug that
  sent the V0 caller to a 404).
- **Frontend has no error UI** for the case where the V1 path
  returns 401. The current behavior is "empty list", same as
  the original 404. Worth a follow-up to surface "you haven't
  connected this provider" explicitly so users aren't confused
  about whether the feature is broken or just misconfigured.
- **`Provider` enum coverage.** The dispatch covers GITHUB,
  GITLAB, BITBUCKET. If a future provider is added (Gitea?
  Codeberg?) the route silently 400s with "unsupported
  provider" instead of being updated. A `match` with
  exhaustive enum check would force the next provider author to
  touch this route. Python doesn't do exhaustive match easily,
  but a `else: raise NotImplementedError(f"Provider {provider}
  not yet supported in V1 route — see PR #14032")` would at
  least leave a breadcrumb.

# Suggestions

1. Add a CI check that diffs the FastAPI route table before/
   after a PR and lists deleted routes alongside their frontend
   callers (grep `frontend/src/api/` for the path string). Any
   PR deleting a route that's still referenced should fail CI
   unless the PR also updates the caller.
2. Add a `TestClient` integration test for the new route. The
   unit test is necessary but not sufficient — it doesn't
   verify the URL path or the auth dependency wiring.
3. Surface a distinct frontend error state for "no provider
   token" vs "no organizations". Today both look like an empty
   list.
4. Audit the rest of the post-#13952 V1 router surface for any
   *other* V0 endpoint that got deleted without a V1 sibling.
   This bug is unlikely to be the only instance.

# Verdict

**Ship it.** Surgical fix with the right scope, tests cover
the dispatch and error branches, frontend unblocked. Follow-ups
on CI guardrail and integration test should be filed
immediately to prevent a recurrence.

# What I learned

The "cleanup PR with a trailing-indicator filter" is a recurring
shape: someone writes a script that says "delete everything
matching pattern X", where X is a property the code is *supposed*
to have when it's safe to delete (here: `deprecated=True`). But
the property is set by humans, not derived from facts, so
anything where a human forgot becomes collateral damage. The
defense is to derive the safety condition from facts the system
can check itself — e.g. "no caller in any frontend file
references this path" — rather than from a manually-set flag.

The reusable insight is route-deletion as a *cross-tier*
operation. Adding a route is a one-tier change (backend);
deleting one is a two-tier change (backend + frontend) but
nothing in the typical CI pipeline enforces that. Some
codebases generate a typed client from the OpenAPI spec, which
turns route deletion into a compile-time error in the frontend
the moment the spec changes. OpenHands doesn't appear to do
that today; if it did, this bug class would be impossible.
Worth filing as an architectural follow-up.

The third lesson is on the frontend "silent 404 → empty list"
failure mode. Any frontend that calls a backend endpoint and
shows a default-empty UI when the call fails has a *latent*
indistinguishability between "no data" and "the API is broken".
Surfacing the distinction (a banner, a toast, a console
warning that becomes a Sentry event) catches this class of
bug at first deploy instead of waiting for a user to file a
ticket. Cheap to add, never regretted.
