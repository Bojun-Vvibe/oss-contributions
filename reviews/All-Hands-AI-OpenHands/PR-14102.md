# Context

OpenHands creates conversations through two distinct entry
points:

1. **Interactive UI / API.** A user opens a chat or a CLI
   client posts to `POST /conversations`. The conversation
   service constructs the conversation, attaches a set of
   "callback processors" (per-event hooks for things like
   title generation, telemetry emission, agent monitoring),
   and persists the resulting object.
2. **Webhook-driven.** A GitHub/GitLab webhook fires (PR
   opened, issue mention, etc.) and the webhook service
   creates a conversation on behalf of the platform user. It
   has historically used a slimmer construction path that
   bypasses the interactive entrypoint.

Callback processors are first-class participants in the
conversation lifecycle. `SetTitleCallbackProcessor` in
particular listens for the first user message event and asks
the LLM to summarize it into a 4-6 word title, then writes
that title back to the conversation record so it shows up
nicely in the sidebar / dashboard.

# Problem

Webhook-created conversations were missing the
`SetTitleCallbackProcessor` registration. The result: every
webhook-created conversation kept its default placeholder
title (`"New conversation"` or the raw event subject like
`"PR #4521 opened"`) forever, regardless of what the user
actually discussed in it.

This is a low-severity-per-conversation bug with a
high-aggregate-annoyance footprint — the dashboard fills up
with rows that all look identical, and finding "the
conversation about the auth bug" requires opening each one.

The root cause is structural: the two creation paths
diverged. Interactive went through `ConversationService.
create_with_processors()`; webhook went through
`WebhookService._create_conversation()` which built the
conversation object directly and registered only the minimal
processors (event persistence, the webhook ack callback).

# Design

The PR takes the right structural fix: route both paths
through a single processor-registration helper.

```python
# conversation_service.py
def default_processors(self) -> list[CallbackProcessor]:
    return [
        EventPersistenceCallbackProcessor(self.event_store),
        SetTitleCallbackProcessor(self.llm_factory),
        TelemetryCallbackProcessor(self.telemetry),
    ]

def create_with_processors(self, ...):
    procs = self.default_processors()
    # ...
```

```python
# webhook_service.py (after)
def _create_conversation(self, payload):
    conv = self.conversation_service.build_conversation(payload)
    procs = self.conversation_service.default_processors()
    procs.append(WebhookAckCallbackProcessor(payload.delivery_id))
    conv.attach_processors(procs)
    return conv
```

Three things this gets right:

1. **Single source of truth for "what processors should every
   conversation have".** `default_processors()` is now the one
   place to add or remove a baseline processor. The webhook
   path opts in via a method call, not by re-listing.
2. **Webhook-specific processor is appended, not replaced.**
   The webhook ack lives alongside the defaults, not instead
   of them. That's the right composition.
3. **No new abstraction.** The PR resists creating a
   `ConversationFactory` or a `ProcessorRegistry`. For a
   two-callsite divergence, a method on the existing service
   is the right weight.

The test seeds a synthetic webhook payload, walks the
creation path, and asserts the resulting conversation's
`processors` list contains a `SetTitleCallbackProcessor`
instance. It also asserts that simulating a user message
event triggers a title-set on the conversation. Tight
coverage for the immediate fix.

# Risks

- **Other creation paths.** Are there only two? A grep for
  `Conversation(` constructors elsewhere (CLI scripted
  bulk-create, the migration tool that imported v0
  conversations, the test fixtures) would catch any third
  path that's still bypassing `default_processors()`. The
  PR doesn't audit this.
- **Processor ordering.** `SetTitleCallbackProcessor` and
  `EventPersistenceCallbackProcessor` both react to the same
  user-message event. If title generation runs before
  persistence and the LLM call hangs or 500s, does the event
  still get persisted? Looking at the dispatch code, processors
  are awaited in registration order. This means a slow LLM in
  the title processor delays event persistence. Probably fine
  (the event is held in memory until persistence) but worth
  a note. A swap in ordering — persistence first, title
  second — would be more defensive and is one diff away.
- **LLM cost amplification.** Every webhook now triggers a
  title-summarization LLM call. For high-volume webhook
  integrations (a busy repo with many PRs), this is a
  real cost increase. The PR could mention an opt-out flag
  for orgs that want webhook conversations to keep the
  raw subject as the title.
- **Idempotence on retry.** Webhook deliveries can retry. If
  conversation creation is idempotent on `delivery_id` (the
  ack callback suggests it is), the title processor will
  not fire twice. If creation isn't idempotent, you could
  end up with duplicate conversations each running a
  title call. The PR doesn't address this; a follow-up
  audit of webhook idempotence would be welcome.

# Suggestions

1. Reorder `default_processors()` so persistence runs
   before title generation. This narrows the window where a
   slow LLM could delay event durability.
2. Add an audit step (or a regression test that introspects
   all conversation construction sites) to catch a future
   third path bypassing `default_processors()`.
3. Add a feature flag (env or per-org config) to disable
   `SetTitleCallbackProcessor` for high-volume webhook
   sources. Defaults to on; off for orgs that report cost
   concerns.
4. Consider a startup-time integrity check: assert every
   registered conversation factory returns a conversation
   with the default processors. Fail loud at boot rather
   than at the 1000th webhook delivery.

# Verdict

**Merge.** Right structural fix (single processor list,
both paths share it), good test, low risk. The cost and
idempotence concerns are follow-up material, not blockers.

# What I learned

This is the canonical "two construction paths drift"
bug. It always starts the same way: feature A is built on
path 1, then feature B requires a new entry point and the
author writes a "minimal" path 2 because path 1 has too
much ceremony, and from then on every new processor /
hook / decorator that lands on path 1 has to be
remembered for path 2 — and isn't.

The structural fixes for "two paths drift" are well-known:

- **Funnel.** Make the two entry points call a shared
  builder. This PR does that.
- **Decoration.** Make the additions plug-in-shaped so
  registering them at construction time is a single
  call, regardless of path.
- **Registration sweep.** At app startup, assert every
  known construction path produces an object with the
  required attachments. Fail boot if not.

The temptation to skip (a) and just "duplicate the
registration call into path 2" is high — it's smaller
diff, doesn't touch the funnel — but it sets up the
*third* path to drift the same way. The funnel is worth
the small refactor cost because it makes the next
divergence impossible-by-construction.

The diagnostic that surfaces these bugs in the field is
also stable: *"feature X works on path 1 but not path 2,
and nobody noticed for months"*. That phrase, more than
the code, is the symptom to keep an ear out for.
