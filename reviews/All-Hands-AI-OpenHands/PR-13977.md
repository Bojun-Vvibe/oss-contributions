# Context

OpenHands' settings page has a tiered-disclosure UI: the same
underlying schema is rendered in three views ("basic", "minor",
"advanced") with progressively more fields visible. The SDK
settings schema marks each field with a `prominence` level, and
`isFieldVisibleInView` decides which fields appear in which view.
A user might also conditionally hide a field through dependent
controls (the `isSettingsFieldVisible` predicate evaluated against
current form values).

The bug this PR fixes is one of the most common shapes of "data
loss via UI": **the previous "save" path, when called from a less-
detailed view, would write back the schema's `default` value for
every hidden field — silently clobbering whatever the user had
configured in a more-detailed view earlier.** A user who set
`mcp.tools.allowed_hosts = ["foo.com"]` in the advanced view, then
flipped to basic view to tweak the agent name and clicked Save,
would lose their MCP allowlist with no warning.

The fix is shaped well: detect which hidden fields would be reset,
prompt the user with a confirmation modal listing them, and only
emit the reset payload after explicit confirmation. +495/-19
across 10 files (4 of which are tests, 2 are i18n bookkeeping).

# Diff walkthrough

**The old behavior (the bug).** The pre-PR `buildSdkSettingsPayloadForView`
unconditionally wrote `field.default ?? null` into the payload for
any hidden field:

```ts
for (const field of getSchemaFields(schema)) {
  if (!isFieldVisibleInView(field, view)) {
    setDotted(payload, field.key, field.default ?? null);
  }
}
```

That's a destructive write masquerading as a no-op. The author
clearly knew the user expects "saving from basic view doesn't
touch advanced fields", but the implementation said the opposite.

**The new shape.** The function gains an `options.resetHiddenFields`
flag (default false) that gates the destructive write, plus the
visibility predicate now ANDs both the view-prominence check and
the dependent-control check:

```ts
const isVisible =
  isFieldVisibleInView(field, view) &&
  isSettingsFieldVisible(field, values);

if (dirty[field.key] && isVisible) {
  setDotted(payload, field.key, coerceFieldValue(field, values[field.key]));
} else if (
  resetHiddenFields &&
  !isVisible &&
  hasNonDefaultFieldValue(field, values[field.key])
) {
  setDotted(
    payload,
    field.key,
    coerceFieldValue(field, getFieldDefaultFormValue(field)),
  );
}
```

Two semantic shifts here: (a) only `dirty` *and* `visible` fields
are now written from the user's input, where before it appears the
old code wrote *all* hidden fields and inherited dirty handling
from `buildSdkSettingsPayload`. (b) the reset branch only fires
when the field's current value differs from default (the new
`hasNonDefaultFieldValue` helper), which is exactly the right
optimization — there's no point asking the user to confirm
"reset the empty field to empty".

**The detection helper.** `getHiddenSettingsFieldsToReset` is
shaped as a pure query over the schema:

```ts
export function getHiddenSettingsFieldsToReset(
  schema: SettingsSchema,
  values: SettingsFormValues,
  view: SettingsView,
): SettingsFieldSchema[] {
  return getSchemaFields(schema).filter((field) => {
    const isVisible =
      isFieldVisibleInView(field, view) &&
      isSettingsFieldVisible(field, values);
    if (isVisible) return false;
    return hasNonDefaultFieldValue(field, values[field.key]);
  });
}
```

Returning the full `SettingsFieldSchema[]` rather than just keys
or just labels is the right call — the caller wants both the
count and the labels for the modal preview, and a future caller
might want the keys too. Pure function, easy to test.

**The UI integration.** `handleSave` becomes a two-call dance:

```ts
const handleSave = (resetHiddenFields = false) => {
  if (!filteredSchema || isReadOnly) return;
  const hiddenFieldsToReset = getHiddenSettingsFieldsToReset(
    filteredSchema, values, view,
  );
  if (!resetHiddenFields && hiddenFieldsToReset.length > 0) {
    setHiddenSettingsResetCount(hiddenFieldsToReset.length);
    setHiddenSettingsResetLabels(
      hiddenFieldsToReset.map((field) => field.label),
    );
    setIsHiddenSettingsModalOpen(true);
    return;
  }
  // ...build and submit payload with { resetHiddenFields }
};
```

First click opens the modal; modal `onConfirm` re-calls
`handleSave(true)`. The single-function recursion via parameter
keeps both code paths in lock-step — no chance the modal-confirmed
path drifts from the no-modal path because they're literally the
same function.

The modal preview format is small but considered:

```ts
const formatHiddenSettingsLabelPreview = (labels: string[]): string => {
  const preview = labels
    .slice(0, HIDDEN_SETTINGS_LABEL_PREVIEW_LIMIT)
    .join(", ");
  return labels.length <= HIDDEN_SETTINGS_LABEL_PREVIEW_LIMIT
    ? preview
    : `${preview}, …`;
};
```

3-label preview with ellipsis. Reasonable default; not great when
the user has 47 things to lose (they'll see "Foo, Bar, Baz, …" and
have to take it on faith). See risks.

**The reset-on-mount.** The component clears `setIsHiddenSettingsModalOpen`,
`setHiddenSettingsResetCount`, and `setHiddenSettingsResetLabels`
whenever `scope`, `settingsSource`, or `sectionKeysSignature`
changes. Good — prevents a stale modal from re-opening with stale
labels after the user navigates between settings sections.

# Risks

1. **The modal shows labels, not values.** A user who sees "Allowed
   hosts will be reset" doesn't know whether they're losing
   `["foo.com"]` or `["foo.com", "bar.com", "baz.com"]`. For
   destructive confirmation prompts, *show what's being lost* is
   the standard UX rule. A second-level "show details" disclosure
   would close the gap without bloating the default modal.
2. **The 3-label `…` truncation.** If a user has 20 hidden non-
   default fields, the modal shows three names and an ellipsis,
   then the user clicks Reset and loses 17 settings they couldn't
   even read the names of. Either expand the limit dynamically
   (show all if ≤10) or render a scrollable list. The current
   shape is fine for the modal UX but wrong for the "informed
   consent" goal of the prompt.
3. **`hasNonDefaultFieldValue` calls `normalizeComparableValue` on
   both sides.** If `normalizeComparableValue` is a lossy
   normalizer (e.g. trims whitespace, lowercases), then a user's
   `"  foo  "` would compare equal to `"foo"` and be silently
   *not* offered for reset, even though saving from the limited
   view would still clobber it (because the reset branch is gated
   on the same predicate). That's fine because both paths use the
   same predicate, but if any third caller uses
   `hasNonDefaultFieldValue` for a different purpose later, the
   lossy comparison could surprise.
4. **The dependent-visibility check uses live `values`.** If a
   user is editing field X (which controls field Y's visibility)
   and Y was previously set to a non-default value, then changing
   X to hide Y *and clicking Save* would correctly trigger the
   prompt. But if X is *itself* hidden in basic view, the
   computed `isSettingsFieldVisible` may behave differently from
   the rendered visibility a user saw. Worth a test that covers
   "field-A-controls-field-B and both are hidden in current view".
5. **`ConfirmationModal` reuse.** The shared modal almost
   certainly defaults the confirm button to a benign style
   ("OK"). For destructive resets, the confirm button should be
   styled as destructive (red) and labeled "Reset N settings",
   not "Confirm". Not a code-correctness issue but a UX-of-
   destructive-actions one.
6. **No explicit i18n string for "this will permanently change
   your saved settings".** The single `SETTINGS$CONFIRM_RESET_HIDDEN_FIELDS`
   string carries the whole modal copy. That's fine for
   translation but the burden is on translators to capture
   urgency; an explicit "this cannot be undone" phrase in the
   key would help.
7. **The settings-source `conversation_settings` branch still
   merges with a `defaultPayload`.** The diff shows the code
   keeps that merge path, but the interaction with the new
   `resetHiddenFields=false` mode (where hidden fields are
   *omitted* from `basePayload`) needs verification: a `defaultPayload`
   merge that fills in *all* fields with defaults could undo the
   "leave hidden fields alone" intent. Tests exist; worth a
   reviewer pass to confirm coverage.
8. **State explosion.** Three new pieces of state (`isOpen`,
   `count`, `labels`) for one modal feels heavy. They could
   collapse to a single `pendingResetState: null | { count: number;
   labels: string[] }` discriminated union, which would also
   make the "modal is open" derivable from `pendingResetState !== null`
   and remove the possibility of an open modal with empty labels.
9. **The `coerceFieldValue(field, getFieldDefaultFormValue(field))`
   double-call.** `getFieldDefaultFormValue` already calls
   `normalizeFieldValue(field, field.default)`. Then `coerceFieldValue`
   runs again. If both transformations are idempotent it's fine,
   but it suggests the field-value pipeline has multiple
   normalization stages that could be unified.

# Suggested questions for the author

1. The modal preview truncates after 3 labels with an ellipsis.
   Have you considered showing the *value* (current vs default)
   rather than just the label, given this is a destructive
   confirmation?
2. Is there a test that covers the case where field A controls
   field B's visibility, A is itself hidden in the current view,
   and B has a non-default value?
3. How does the `conversation_settings` `defaultPayload` merge
   interact with the new "leave hidden fields out of payload"
   shape? Is there a risk that `defaultPayload` fills in defaults
   for fields the user wanted preserved?
4. Why three pieces of React state instead of one
   `pendingResetState | null`? Was there a reason to keep them
   separate (animation, async submission)?
5. The `ConfirmationModal` is shared. Does it support a
   "destructive" variant (red button, different label) that
   should be wired up here?

# Verdict

**Approve.** This is a textbook fix for a textbook data-loss bug:
detect the destructive case, prompt with specifics, only act on
explicit confirmation. The pure-function `getHiddenSettingsFieldsToReset`
is clean and trivially testable. The single-function recursive
shape of `handleSave(resetHiddenFields)` keeps the two paths
locked in step. The biggest gap is UX-of-the-modal, not code
correctness — the user sees three label names and is asked to
trust the system on the rest. A follow-up to enrich the modal
content (show values, scrollable list for >10 fields, destructive
button styling) would land the feature properly. Ship this PR
now; file the UX follow-up.

# What I learned

Three takeaways, all useful for any agent UI that has tiered
disclosure or progressive forms.

**First**, the "saving from a less-detailed view" case is the
single most common data-loss vector in tiered-disclosure UIs and
almost always implemented incorrectly the first time. The
intuitive shape is "the form posts whatever's in the form" —
which means hidden fields post their displayed value (often
empty or default), which means hidden fields get clobbered. The
correct shape is "the form posts whatever the user *touched*",
which requires either (a) a `dirty` map that tracks user edits
explicitly, or (b) a diff against the originally-loaded values.
This PR uses (a). For agent UIs that increasingly have
context/history/scope-switching, this is critical: a user
switching from "session-level config" to "global config" must
not have their session-level overrides silently dropped.

**Second**, the *single recursive function* shape for "modal-
gated destructive action" is a quietly excellent pattern. The
naive shape is two functions: `handleSave` and `handleSaveWithReset`,
which inevitably drift. The shape here — `handleSave(reset = false)`
which either opens the modal *or* performs the action depending
on `reset` — keeps both paths in one function body, which means
any change to the save logic automatically applies to both
paths. The modal's `onConfirm` is just `() => handleSave(true)`.
This pattern generalizes to any "preview then commit" flow: the
preview path and the commit path are the same function with a
boolean.

**Third**, the gap between *technically correct* and *informed-
consent correct* in destructive prompts. The PR's modal
*technically* warns the user, but a user who sees "Foo, Bar,
Baz, …" and clicks Reset has not been informed about the 17
other things they're losing. The standard for destructive
prompts in well-designed UIs is now "show the entire diff" — VS
Code's "discard changes" prompt shows the file count *and* lets
you click through to the diff before confirming. For agent UIs
where the underlying state is rich and structured (an MCP
allowlist, a per-host hook config, a tool whitelist), the
"informed consent" bar is high. Three labels and an ellipsis is
the modal you ship in week one; a scrollable diff with current-
vs-default values is the modal you ship in week three after the
first user complains. Build the second one first if the
underlying data is meaningful.
