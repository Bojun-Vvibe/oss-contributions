# Context

OpenHands has been adding *Agent-Client Protocol* (ACP) support
— the ability to spawn third-party coding-agent CLIs (Claude
Code, Codex, Gemini CLI, etc.) as sub-agents inside an OpenHands
conversation. The ACP server emits its own structured tool-call
events (run-this-shell, edit-this-file, fetch-this-url) which
OpenHands now needs to render in the same conversation timeline
as its own native tool calls.

This PR is the frontend half: take an `ACPToolCallEvent` flowing
in from the websocket and render it as a card in the chat UI
that visually parallels OpenHands' native execute/edit/read
cards. The companion server-side PR (#13983) is what actually
routes the ACP traffic; this PR is purely the rendering layer.

+663/-4 across 16 files. Five of those are tests, one is the
i18n declaration update, one is the translation JSON. The actual
code surface is small and well-bounded.

# Diff walkthrough

**The event type.** New file
`frontend/src/types/v1/core/events/acp-tool-call-event.ts`
defines the wire shape:

```ts
export type ACPToolKind = "execute" | "edit" | "read" | "fetch" | "other";
export type ACPToolCallStatus = "in_progress" | "completed" | "failed";

export interface ACPToolCallEvent extends BaseEvent {
  kind: "ACPToolCallEvent";
  source: "agent";
  tool_call_id: ToolCallID;
  title: string;
  status: ACPToolCallStatus | null;
  tool_kind: ACPToolKind | null;
  raw_input: unknown;
  raw_output: unknown;
  content: ACPToolCallContentBlock[] | null;
  is_error: boolean;
}
```

Several principled choices: `kind` as a literal-string discriminator
plays nicely with the existing event union; `status` is nullable
for backwards compatibility with pre-PR events; `raw_input`/
`raw_output` are typed as `unknown` rather than `any`, forcing
downstream renderers to narrow before accessing — TypeScript
hygiene done right. The `is_error` flag is *separate* from
`status === "failed"`, which the docstring explains: an ACP
server can report a tool call as an error while still marking the
call lifecycle as `completed`. That's a real distinction (a
shell tool that ran a command and the command exited non-zero)
and conflating them would lose information.

The `ACPToolCallContentBlock` is a permissive
`{ type: string; [key: string]: unknown }`. This is the right
shape for a pass-through wire object — ACP defines text, image,
resource, and resource_link blocks, and the renderer only cares
about the discriminator.

**The render gate.** `should-render-event.ts` adds a one-liner:

```ts
if (isACPToolCallEvent(event)) {
  return true;
}
```

Trivial. Worth noting that `shouldRenderEvent` is the central
choke-point for "what shows up in the chat" — every new event
type flows through here. The PR adds a single explicit case
rather than a generic fallthrough, which keeps the visible-
surface auditable.

**The dedup logic in `handle-event-for-ui.ts`** is the most
interesting piece:

```ts
if (isACPToolCallEvent(event)) {
  const existingIndex = newUiEvents.findIndex(
    (uiEvent) =>
      isACPToolCallEvent(uiEvent) &&
      uiEvent.tool_call_id === event.tool_call_id,
  );
  if (existingIndex !== -1) {
    newUiEvents[existingIndex] = event;
  } else {
    newUiEvents.push(event);
  }
  return newUiEvents;
}
```

ACP streaming emits multiple events per tool call as it
progresses (`in_progress` → `completed` / `failed`), so the UI
needs to *replace in place* rather than append. The shape is
correct: keep position stable so the card doesn't jump around
the timeline, but swap the entire object so React's reference
equality triggers a re-render. The full-object swap (rather than
a field-by-field merge) means the new event is treated as the
authoritative truth — which is right because the server is the
source of truth for the tool call's progress.

**The content renderer.** `get-acp-tool-call-content.ts`
mirrors the existing `getTerminalObservationContent` shape so
the rendered card lines up with regular OpenHands observations:

```ts
if (
  toolKind === "execute" &&
  rawInput && typeof rawInput === "object" &&
  "command" in rawInput &&
  typeof (rawInput as { command: unknown }).command === "string"
) {
  const { command } = rawInput as { command: string };
  output += `Command: \`${command}\`\n\n`;
} else if (rawInput !== null && rawInput !== undefined && rawInput !== "") {
  const inputStr = stringifyPayload(rawInput);
  if (inputStr.trim()) {
    output += `Input:\n\`\`\`json\n${inputStr}\n\`\`\`\n\n`;
  }
}

const outputStr = truncate(stringifyPayload(rawOutput).trim());
const outputLabel = isError ? "**Error:**" : "Output:";
const outputBody = outputStr || i18n.t("OBSERVATION$COMMAND_NO_OUTPUT");
output += `${outputLabel}\n\`\`\`\n${outputBody}\n\`\`\``;
```

The narrowing of `raw_input` to `{ command: string }` for
`execute` is the correct shape — proper type-narrowing through
the `in` operator and a `typeof` check, rather than a cast. The
`stringifyPayload` fallback (JSON pretty-print, then `String()`)
handles the long tail of weird payloads. The `MAX_CONTENT_LENGTH`
truncation is reused from the shared module, so it's consistent
with other event renderers.

The translation key dispatch via `getACPToolCallTitleKey` —
`ACTION_MESSAGE$ACP_RUN`, `ACP_EDIT`, `ACP_READ`, `ACP_FETCH`,
`ACP_TOOL` (default) — gives users a localized, ACP-prefixed
title that reads naturally.

# Risks

1. **`raw_input`/`raw_output` are `unknown` and rendered with
   `JSON.stringify`.** A circular object will throw inside the
   `try`/`catch` and fall back to `String(value)` — fine. But a
   payload with secrets (an `Authorization` header echoed back
   in `raw_input`, an API key in a fetched URL's query string)
   will be rendered verbatim into the chat UI, then end up in
   conversation logs, browser DevTools history, and possibly a
   shared replay link. The ACP layer should redact, but the
   renderer is the second line of defense and currently does
   none.
2. **The `command` extraction for `tool_kind === "execute"`
   only handles `raw_input.command: string`.** ACP can send a
   command as `string[]` (argv-style) for many shells. The fall-
   through to the JSON dump would surface it readably but lose
   the syntax-highlighted "Command:" treatment. Worth handling
   both shapes.
3. **`MAX_CONTENT_LENGTH` truncation is silent.** A 50KB output
   gets cut to N bytes with `...` appended — the user has no
   "show full output" affordance. For ACP tools especially
   (which can produce large file diffs or web-fetch responses),
   the truncation is more user-hostile than for native OpenHands
   tools because the ACP card *is* the only surface for that
   data.
4. **The dedup logic uses `tool_call_id` equality** — correct
   for ACP semantics. But if the server ever ships a buggy
   sequence where two unrelated tool calls share an id (replay
   bugs, ID-collision in tests), the second will silently
   overwrite the first. A dev-mode invariant check ("if the
   replaced event's `kind` matches but `tool_call_id` was
   collapsed across two distinct titles, warn") would catch this.
5. **Ordering invariants.** `newUiEvents[existingIndex] = event`
   keeps position. But what if the in-progress event arrives
   *after* an unrelated event was rendered, then completes? The
   completed event slots back into the original position —
   correct. What if the in-progress event was *never seen* and
   the first event for that `tool_call_id` is the `completed`
   one? The `findIndex` returns -1, we push. Also correct. But
   what if events arrive out of order (`completed` before
   `in_progress`)? The later `in_progress` would overwrite the
   `completed` state, downgrading the UI. ACP shouldn't emit
   that, but a server-side bug or proxy-reorder would corrupt
   the UI. Worth either a status-monotonicity check (don't
   downgrade `completed` to `in_progress`) or a server-side
   sequence number.
6. **`status: null` is silently rendered as if no status.** A
   user looking at an old event has no way to know whether
   "status not shown" means "the tool succeeded" or "we don't
   know". A small "(status unknown)" footer would be more
   honest.
7. **The `i18n.t("OBSERVATION$COMMAND_NO_OUTPUT")` fallback for
   the output body is reused from the bash observation
   renderer.** The string makes sense for shell commands but
   reads oddly for ACP `read` or `fetch` tools that return
   structured data and "no output" means "the file was empty"
   or "the URL returned 200 with empty body". A per-tool-kind
   fallback string would read better.
8. **Test coverage of `is_error: true` + `status: "completed"`
   combination.** The semantic distinction the docstring calls
   out (errored but completed) needs an explicit test that
   asserts the "**Error:**" label is used.
9. **No `rel="noopener noreferrer"` consideration for content
   blocks of type `resource_link`.** The renderer punts on the
   `content` array (it's typed but not yet rendered), so this
   doesn't bite today. When a follow-up PR adds the link
   rendering, the standard external-link safety needs to apply.
10. **The translation JSON adds five new keys** but if any
    translator only ships partial coverage, fallback behavior
    needs to degrade gracefully. Worth confirming the i18n setup
    falls back to English for missing keys rather than rendering
    the raw key (`ACTION_MESSAGE$ACP_RUN`).

# Suggested questions for the author

1. Has the secret-redaction story for `raw_input`/`raw_output`
   been discussed? An ACP server passing through an
   `Authorization` header in `raw_input` will surface it
   verbatim in chat — is that the ACP layer's responsibility or
   should the renderer also redact known-sensitive keys?
2. What's the plan for status monotonicity? An out-of-order
   `in_progress` arriving after `completed` would currently
   downgrade the UI state.
3. The truncation at `MAX_CONTENT_LENGTH` is silent. Is there a
   "show full output" affordance planned, or is the assumption
   that users with large outputs will use the raw event log?
4. The `command` field for `execute` only handles
   `string`-typed commands. ACP supports `string[]`-typed
   commands too — should that path also get the syntax-
   highlighted "Command:" treatment?
5. The ACP `content` blocks (`text`, `image`, `resource`,
   `resource_link`) are typed but not rendered. What's the
   roadmap for that?

# Verdict

**Approve.** This is a clean, narrowly-scoped frontend addition.
The type definitions are principled (`unknown` for raw payloads,
nullable status for back-compat, explicit discriminator), the
dedup-by-`tool_call_id` shape is correct for streaming ACP
events, and the content renderer mirrors the existing
observation-renderer idioms. Five test files for ~200 lines of
production code is the right ratio for an event-pipeline change.

The most important follow-up is the secret-redaction question —
ACP's design assumes the server is honest about what it puts in
`raw_input` and `raw_output`, but a robust client UI should
defense-in-depth. Second priority is the status-monotonicity
guard. Both are follow-up PRs, not blockers.

# What I learned

Two takeaways, both about rendering streaming protocol events.

**First**, the dedup-on-stable-id pattern (find by id, replace
in place, push if absent) is the canonical shape for "live-
updating card in a timeline" and works equally well for
streaming chat completions, file-edit progress, long-running
shell commands, and tool-call lifecycle events. The key
property is *position stability*: the card stays where it
appeared, only its contents change. This is what makes the UI
feel alive rather than flickery. Compare to the naive "just
append every event" shape, which produces a wall of duplicates,
or the "filter to latest per id at render time" shape, which is
O(N²) per render and loses positional anchoring. The
`handleEventForUI` shape here is the textbook pattern.

**Second**, the discipline of `unknown` over `any` for wire
payloads is small but compounds. Every downstream consumer of
`raw_input` is forced to narrow with `typeof`/`in` checks before
accessing fields. That discipline catches the "what if `command`
is an array, not a string?" case at the type-checker, not at
runtime in the user's browser. The `(rawInput as { command:
unknown }).command` cast in this PR is mildly unsightly but
functionally correct — it's narrowing through a type assertion
followed by a `typeof` check, which is the safe shape. The
alternative — typing `raw_input` as a tagged union per
`tool_kind` — would be tighter but more brittle when the ACP
server adds new tool kinds. `unknown` plus runtime narrowing is
the right trade for an evolving wire protocol.

The third meta-lesson is about layering: the PR ships a *type*,
a *render gate*, a *dedup rule*, a *title-key picker*, and a
*body renderer* — each in its own file, each independently
testable. Five files of production code, five files of tests,
one feature. That ratio of "small files, narrow contracts" is
the shape that makes a frontend codebase maintainable. The
opposite shape ("one big file with a switch statement on event
kind") is what every codebase trends toward unless someone
deliberately resists. This PR resists, and the result is easy
to review and easy to extend.
