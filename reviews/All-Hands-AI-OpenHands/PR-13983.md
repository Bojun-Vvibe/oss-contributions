# Context

OpenHands has been incrementally adding *Agent-Client Protocol*
(ACP) support — running third-party coding-agent CLIs (Claude
Code, Codex, Gemini CLI) as sub-agents inside an OpenHands
conversation. Frontend rendering of ACP tool-call events landed
in #13994 (separately reviewed). This PR is the server-side
companion: actually route an ACP-flavored conversation through
the *correct* agent-server endpoint so that the conversation
gets created against the ACP-aware controller rather than the
LLM-only one.

The change is necessarily wide because OpenHands' "agent" was
previously a single concept (`AgentSettings`) with a `LLM` field.
The SDK rework (OpenHands/software-agent-sdk#2861) splits this
into a discriminated union (`LLMAgentSettings | ACPAgentSettings`),
and this PR threads that distinction through every layer:
storage model (new `agent_kind` column + alembic migration), SDK
compatibility shim (the union isn't on PyPI yet), conversation
service (route to `/api/conversations` vs `/api/acp/conversations`),
settings router/UI (separate ACP server / model settings pages),
and frontend schema utilities.

+1463/-111 across 31 files. I reviewed the core routing service,
the alembic migration, the SDK compat shim, and the storage
model in depth, with spot-checks of the frontend route additions.

# Diff walkthrough

**The routing decision.** The new private helper
`_agent_kind_to_router_path` is the single source of truth for
the URL fork:

```python
def _agent_kind_to_router_path(agent_kind: str | None) -> str:
    if agent_kind == 'acp':
        return '/api/acp/conversations'
    if agent_kind not in (None, 'llm'):
        _logger.warning(
            'Unknown agent_kind %r; routing to /api/conversations. '
            'Add a branch to _agent_kind_to_router_path if this is a '
            'new variant.', agent_kind,
        )
    return '/api/conversations'
```

Three good choices in this small function: (a) `None` is
explicitly tolerated for legacy rows (the migration creates the
column nullable), (b) unknown variants log a warning rather than
raising — this is the right call for a *read* path that's
walking historical conversations and shouldn't blow up the list
view because of one corrupt row, and (c) the warning copy is
actionable ("Add a branch to `_agent_kind_to_router_path`")
rather than just descriptive. Unknown kinds fall back to the
LLM path because *that* path's GET will produce a clean 404 for
a missing conversation, whereas hitting an unknown ACP variant
might 404 on the *router* itself.

**The conversation-start fork.** In `_start_app_conversation`,
the ACP request type is detected via `isinstance` and the URL
is selected:

```python
is_acp_request = isinstance(
    start_conversation_request, StartACPConversationRequest
)
conversation_path = (
    '/api/acp/conversations' if is_acp_request else '/api/conversations'
)
# ...
response = await self.httpx_client.post(
    f'{agent_server_url}{conversation_path}',
    json=body_json,
    headers=headers,
    timeout=self.sandbox_startup_timeout,
)
response.raise_for_status()
info: ConversationInfo | ACPConversationInfo
if is_acp_request:
    info = ACPConversationInfo.model_validate(response.json())
else:
    info = ConversationInfo.model_validate(response.json())
```

The shape is correct: the request type drives the URL and the
response type. The variable annotation `info: ConversationInfo |
ACPConversationInfo` makes the union explicit so a downstream
attribute access against the wrong field gets caught by the type
checker. This is the discriminated-union pattern done right.

The display-model handling is the kind of detail that matters
for users:

```python
agent_obj = start_conversation_request.agent
if isinstance(agent_obj, ACPAgent):
    display_model = agent_obj.acp_model or agent_obj.llm.model
    agent_kind = 'acp'
else:
    display_model = agent_obj.llm.model
    agent_kind = 'llm'
```

For ACP agents, `agent.llm` is a sentinel (since ACP doesn't use
a direct LLM — the sub-process talks to its own provider), so
falling through to `agent.llm.model` would surface garbage in
the conversation list. The `acp_model or agent.llm.model`
fallback is the right escape hatch for partially-configured ACP
agents while preferring the meaningful name when present.

**The migration.** Migration 009 is exemplary in its
constrained shape:

```python
def upgrade() -> None:
    op.add_column(
        'conversation_metadata',
        sa.Column('agent_kind', sa.String, nullable=True),
    )

def downgrade() -> None:
    op.drop_column('conversation_metadata', 'agent_kind')
```

Nullable column with a thoughtful docstring explaining the
NULL→`"llm"` coercion. No backfill, no data migration, no
default value — historical rows stay NULL and the read path
treats NULL identically to `"llm"`. That's the *only* safe
shape for a column add on a table that may have millions of
existing rows in production.

**The SDK compat shim.** `openhands/utils/sdk_settings_compat.py`
is the most thoughtful piece of the diff. The constraint is real:
the SDK rework is on `main` but not on PyPI, and the project's
pinned SDK release (`1.17.0`) doesn't have the new symbols. The
naive shape would be "assume new symbols, break the pinned
build". The chosen shape is a try/import with explicit fallbacks:

```python
try:
    from openhands.sdk.settings import (
        ACPAgentSettings, AgentSettingsConfig, LLMAgentSettings,
        default_agent_settings, export_agent_settings_schema,
        validate_agent_settings,
    )
    _HAS_DISCRIMINATED_UNION = True
except ImportError:
    _HAS_DISCRIMINATED_UNION = False
    from openhands.sdk.settings import AgentSettings
    LLMAgentSettings = AgentSettings  # alias

    class _ACPAgentSettingsStub:
        """Sentinel — SDK 1.17.0 cannot produce ACPAgentSettings instances."""

    ACPAgentSettings = _ACPAgentSettingsStub
```

The stub class is the key insight: code elsewhere does
`isinstance(x, ACPAgentSettings)` to gate ACP behavior, and the
stub class makes that check structurally always-false on the
pinned SDK. So ACP code paths become *dead code* under the old
SDK rather than *broken code*. That's the right defensive
shape.

The `validate_agent_settings` fallback raises explicitly on
`kind == 'acp'`:

```python
def validate_agent_settings(data: dict[str, Any]) -> AgentSettings:
    if isinstance(data, dict) and data.get('kind') == 'acp':
        raise RuntimeError(
            "Stored settings contain kind='acp' but the installed "
            'openhands-sdk does not support ACP agents. Upgrade to an '
            'SDK release that includes the discriminated-union rework '
        )
    return AgentSettings.model_validate(data)
```

This is exactly the shape requested by a reviewer in a sister
PR (#14004): refuse silent coercion. A user who persisted
`kind='acp'` settings under a newer SDK and then downgraded
would, without this guard, see their ACP config silently dropped
and the agent boot as LLM — a confusing failure mode. The hard
error is the right call.

The whole module is documented as "Delete this once the SDK
ships." That discipline matters: compat shims that don't have a
clear deletion criterion become permanent.

# Risks

1. **Migration 009 makes `agent_kind` nullable forever.** That's
   the right shape for the *upgrade*, but no follow-up plan is
   visible to backfill historical NULLs to `"llm"` and then
   tighten the column to `NOT NULL`. Without that, every read
   forever has to handle NULL — a low-grade tax that compounds
   as the codebase grows. A follow-up PR to backfill + tighten
   should be in the roadmap.
2. **`_agent_kind_to_router_path` warns on unknown kinds and
   falls back to LLM.** That's fine for *reads*, but the helper
   is also reachable from contexts where falling back silently
   to LLM would be wrong (e.g. starting a new conversation from
   a corrupted settings row). A second helper variant —
   `_agent_kind_to_router_path_strict` that raises — would be
   safer for the *write* path. Currently the write path uses
   `isinstance(... StartACPConversationRequest)` directly, so
   it's not affected, but a future caller might use the helper
   incorrectly.
3. **The `acp_model or agent.llm.model` fallback masks
   misconfiguration.** A user who configures ACP but forgets to
   set `acp_model` will see the dummy LLM sentinel string in
   their conversation list. A small UI hint ("ACP model not
   set; showing LLM placeholder") would help — or a server-side
   validation that requires `acp_model` for ACP agents at
   create time.
4. **The compat shim's `_ACPAgentSettingsStub` is an empty
   class.** That works for `isinstance` checks but has a
   subtle hazard: any code that calls `ACPAgentSettings(...)`
   constructor under the old SDK gets a ValueError-free instance
   of the stub, which is *truthy* and would pass through any
   `if settings:` check. The shim should arguably make the stub
   `__init__` raise, to fail loud. Then again, no in-tree caller
   constructs `ACPAgentSettings` directly (the validator builds
   it from JSON), so the risk is theoretical.
5. **The conversation type in storage is now a discriminator
   (`agent_kind`) but the conversation **payload** isn't typed
   accordingly.** `AppConversationInfo` keeps `llm_model: str`
   and adds `agent_kind: str`; an ACP conversation has
   `llm_model = display_model` (which is the ACP model name).
   Reading code that expects `llm_model` to be a *real LLM
   model* will be confused. A field rename to `display_model` (or
   a `model_name` neutral term) would clarify; or split into
   two fields with one nullable.
6. **The `RuntimeError` in `validate_agent_settings`** is the
   right idea but uses `RuntimeError` rather than a domain-
   specific exception type. Callers can't catch it specifically
   without string-matching. A `SDKVersionMismatchError` or
   similar would be better.
7. **`_HAS_DISCRIMINATED_UNION` is exported via `__all__`** but
   leading-underscore-named. Code outside the module can read
   it but the underscore says "private". Either rename it
   (`HAS_DISCRIMINATED_UNION` if it's part of the public API)
   or drop it from `__all__`.
8. **No test for the SDK compat shim's import-failure branch.**
   The fallback is reached only when the new symbols are
   missing from the SDK — testing it requires either monkey-
   patching the import or pinning to the old SDK in a separate
   CI matrix. Without coverage, the fallback can rot silently.
9. **The 31-file scope makes review costly.** Several of these
   files are arguably orthogonal: the frontend ACP server / model
   settings pages could be a separate PR, and the alembic
   migration could land before everything else as
   infrastructure. The current single-PR shape couples backend
   routing, storage migration, SDK compat, frontend routing,
   and settings UX — any one piece reverting drags the rest.
10. **Enterprise storage** (`saas_conversation_store.py`)
    received changes too. Worth confirming the enterprise path
    has equivalent test coverage and that the `agent_kind`
    column is added to the SaaS schema via the same migration,
    not a separate one.

# Suggested questions for the author

1. Is there a roadmap for backfilling NULL `agent_kind` rows
   and tightening the column to NOT NULL once everything is
   stable? Without it, every read forever handles NULL.
2. The `acp_model or agent.llm.model` fallback masks the
   "ACP-but-no-acp_model-configured" case. Should server-side
   validation require `acp_model` when `agent_kind == 'acp'`?
3. The compat shim's `_ACPAgentSettingsStub` is an empty class.
   Has the team considered making `__init__` raise to fail
   loudly if any code accidentally constructs it under the old
   SDK?
4. `AppConversationInfo.llm_model` now stores either a real LLM
   model name or an ACP model identifier. Is renaming to
   `display_model` (or splitting into two fields) on the table?
5. Why is the SDK compat shim a separate utility module rather
   than a private helper inside the SDK itself? Pinning the
   shim into `openhands/utils/` ties its lifecycle to OpenHands
   rather than to the SDK — was that the intentional ownership
   split?
6. The 31-file scope makes review slower. Was there a reason
   not to land the alembic migration as its own PR first, then
   the routing changes, then the frontend?

# Verdict

**Approve, with the column-nullability follow-up as a tracked
issue.** The routing fork is correctly minimal (one isinstance
check, one URL pick, one response-type pick), the migration is
appropriately conservative (nullable, no backfill), and the
SDK compat shim is the most thoughtful piece of code in the diff
— stub class for `isinstance`-based gating, hard error on
attempted ACP validation under the old SDK, explicit deletion
criterion. The biggest concerns are *operational*: the
`agent_kind` column will live as nullable forever without a
follow-up, the `llm_model` field name now misrepresents its
contents for ACP rows, and the 31-file scope makes the PR
expensive to review. None of these are blockers for the
substantive correctness of the change.

# What I learned

Three takeaways, all about evolving a system that has both a
storage layer and a third-party dependency.

**First**, the *empty stub class* pattern for compatibility
shims is underused and excellent. The standard naive shape for
a missing symbol in a dependency is `if HAS_FEATURE:` checks
sprinkled throughout the codebase — which means every call site
has to remember to gate. The shape this PR uses — define
`ACPAgentSettings = _ACPAgentSettingsStub` (an empty class) —
makes `isinstance(x, ACPAgentSettings)` the gating mechanism
for every ACP-related branch, and the stub's structural
impossibility (no constructor that produces an instance,
nothing in the codebase that creates one) makes those branches
dead code under the old SDK. The cost is one stub class; the
benefit is that the call sites stay clean and no `if
HAS_FEATURE` guard can be forgotten. This generalizes to any
"feature that may not be available in this version of the
dependency" situation.

**Second**, the *fail-loud-on-silent-coercion* discipline in
`validate_agent_settings`. A user who persisted ACP settings
under a newer SDK, then downgraded (or whose CI is on a pinned
release), would have their config silently lossy-coerced to
LLM. The PR catches this explicitly and raises with an
actionable error message. This is the right shape for any
versioned settings store: when the schema can express more than
the code can interpret, the code must refuse to interpret rather
than silently dropping fields. The cost is one if-guard and a
clear error; the benefit is that no user ever sees their ACP
config silently revert to LLM and wonders why their tools
aren't loading.

**Third**, the *nullable-forever risk* of conservative
migrations. The migration here adds `agent_kind` as nullable —
the only safe choice for a column add on an existing table — and
the read path coerces NULL → `"llm"`. That's correct. But
without a follow-up plan to backfill NULLs and tighten the
column, the codebase carries that NULL-handling tax forever.
Every new read path has to remember to coerce. Every new piece
of code that joins on this column has to handle NULL. The
cumulative cost is real. The right discipline is: every nullable
column added in a migration should have an issue tracking the
backfill-and-tighten follow-up, with a target date. Without that
discipline, the storage layer accretes nullable columns
indefinitely and reads become a maze of `COALESCE(col, default)`
expressions. This is a place where storage hygiene and product
hygiene align — neither team wants the tax, but only the team
that owns the migration can schedule the cleanup.
