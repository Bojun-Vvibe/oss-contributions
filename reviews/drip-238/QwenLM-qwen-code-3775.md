- **Verdict**: **merge-after-nits**

## Context

Three different "side-query" LLM call sites — `LlmRewriter` (ACP message rewrite), `DataProcessor` (insight generation), `renameCommand` (kebab-case session renaming) — were each calling either `config.getContentGenerator().generateContent({...}, "tag")` or `config.getBaseLlmClient().generateJson({...})` directly. Each call site duplicated: model selection logic, abort-signal plumbing, thought-vs-text part filtering, system-instruction wiring, retry/error handling. PR introduces a `runSideQuery(config, options)` chokepoint exported from `@qwen-code/qwen-code-core` that all three call sites now route through.

## What's right

- **One source-of-truth for side-query semantics.** Each call site collapses to a `runSideQuery(this.config, {purpose, model, contents, ...})` call. The `purpose` field (`"acp-rewrite"` / `"insight-session-analysis"` / `"insight-qualitative-generate"`) gives the chokepoint a single place to add per-purpose telemetry, rate-limiting, model-routing, or auth-scoping without touching each consumer.
- **`LlmRewriter.ts:115-150` collapse is the cleanest.** Pre-fix the function pulled a content-generator, hand-built a `GenerateContentRequest` with `systemInstruction`/`abortSignal`/`temperature`/`maxOutputTokens`/`thinkingConfig`, then post-processed `result.candidates?.[0]?.content?.parts?.filter(p => !p.thought).map(p => p.text).filter(Boolean).join('')` to strip thoughts. Post-fix the chokepoint owns thought-stripping (note `result.text` at `:138` is already the cleaned string), so the consumer-side post-processing disappears — three steps (filter-thoughts → map-text → filter-empty → join) replaced by one field read. The "skip if rewritten too short" guard at `:143` correctly trims-once via `if (!rewritten || rewritten.length < 5)` rather than the prior trim-twice (`rewritten.trim().length < 5` then `rewritten.trim()` later).
- **`DataProcessor.ts:197-218` and `:391-413` route the JSON-schema-validated calls through the same chokepoint.** Note both flip from `getBaseLlmClient().generateJson({model, contents, schema, abortSignal})` to `runSideQuery<Record<string, unknown>>(config, {purpose, model, contents, schema, config: {thinkingConfig: {includeThoughts: true}}, abortSignal})`. Two important shifts: (a) `config: {thinkingConfig: {includeThoughts: true}}` is now explicit at the call site rather than implicit in `generateJson` defaults — better, since insight quality depends on reasoning being on, (b) the `<Record<string, unknown>>` generic preserves the prior return type so downstream parsing is unchanged.
- **Test mocks updated to track the new chokepoint shape.** `LlmRewriter.test.ts:11-39` adds a `vi.mock("@qwen-code/qwen-code-core", () => ({createDebugLogger: ..., runSideQuery: vi.fn(async (_config, options) => { ... })}))` that forwards the options to the legacy `mockGenerateContent` and re-derives the same `text` extraction the production code used to do — keeps existing test assertions (which checked `mockGenerateContent.mock.calls[0][0].model` etc.) functional. Smart shim.
- **`DataProcessor.test.ts:1170-1196` adjustment.** The "partial qualitative data" test was previously returning a 3-field `{intro, areas, opportunities}` mock — but schema validation in `runSideQuery` rejects partial objects. Fix expands the mock to a fully-populated 9-field object so only the explicitly-rejected calls land as undefined. Captures a real semantics tightening: side-query schema validation is now strict (which is the correct direction).
- **`renameCommand.test.ts:185-235` flips from `getContentGenerator().generateContent({...candidates: [{content: {parts: [{text}]}}]})` to `getBaseLlmClient().generateText({...text, usage: undefined})`** — confirms the kebab-rename path moves to the `generateText` API surface where the result is a flat `{text, usage}` rather than the nested candidates/parts shape. Both tests (`uses fastModel when configured` + `falls back to main model`) flip symmetrically.

## Risks / nits

- **`runSideQuery`'s implementation isn't in the diff** (it's exported from `@qwen-code/qwen-code-core` and presumably lives in a sibling PR or earlier commit). The review can't verify the chokepoint's own correctness — particularly: (a) does it set per-purpose timeouts? (b) does it honor an abort signal even when none is passed (`abortSignal ?? new AbortController().signal` at `LlmRewriter.ts:130` suggests yes, but the chokepoint's own behavior on absent signal needs pinning), (c) does it propagate retries on transient errors? Recommend at minimum a doc comment in the PR body linking to where `runSideQuery` is defined.
- **`abortSignal: signal ?? new AbortController().signal` at `LlmRewriter.ts:130`** creates a never-aborted controller as a fallback. Functional but wasteful (one allocation per call). Cleaner: have `runSideQuery` accept `abortSignal?: AbortSignal` and let the chokepoint create-or-omit. Tiny nit.
- **The `purpose` string is free-form.** A typo (`"acp-rewrite"` vs `"acp_rewrite"`) would silently miss any purpose-scoped routing/telemetry. Consider a `SideQueryPurpose` union type exported from core so call sites get compile-time validation.
- **Test mock for `runSideQuery` at `LlmRewriter.test.ts:25-37` re-implements the production text-extraction logic** (`r.candidates?.[0]?.content?.parts ?? []).map(p => p.text ?? "").join("").trim()`). If the chokepoint's real text extraction ever diverges (e.g. starts handling tool-call parts), the mock won't catch the drift. Acceptable as a test shim but worth a `// keep in sync with runSideQuery's text extraction` comment.

## Verdict

**merge-after-nits.** Three call sites collapse to one chokepoint with consistent semantics, a real cleanup of duplicated request-shape building + thought-stripping. Wants the `runSideQuery` definition surfaced in the PR body for verification, and a typed `SideQueryPurpose` union to prevent free-form-string drift across consumers.
