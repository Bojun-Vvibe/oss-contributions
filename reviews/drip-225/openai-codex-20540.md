## Notes
- New `FileChangeItem` first-class turn-item variant at `protocol/items.rs:142-159` (`id`, `changes: HashMap<PathBuf, FileChange>`, optional `status`/`auto_approved`/`stdout`/`stderr` with `skip_serializing_if`) is the right "promote synthetic event into the canonical item stream" shape — same pattern codex applied for `WebSearch`/`ImageGeneration` last quarter. Adding the `TurnItem::FileChange` arm at `:42` and the `id()` arm at `:443` keeps the enum exhaustive across all 8 turn-item variants.
- Legacy-event bridge methods at `protocol/items.rs:409-428` (`as_legacy_begin_event`, `as_legacy_end_event`) construct the old `EventMsg::PatchApplyBegin`/`PatchApplyEnd` shapes from the new item, with the load-bearing `success: status == PatchApplyStatus::Completed` derivation at `:425` collapsing the previously-stored `success: bool` into a derivable function of `status` — this is the right "single source of truth" cleanup since the old field was redundant with `Completed`/`Failed`/`Declined` status.
- Producer-side migration at `core/src/tools/events.rs:185-198` flips `send_event(... EventMsg::PatchApplyBegin{...})` to `emit_turn_item_started(... TurnItem::FileChange{...})`, matched by the corresponding `emit_patch_end` migration at `:497-509` dropping the now-redundant `success: bool` parameter at the four call sites (`:202/:219/:236/:250`). The legacy `EventMsg::PatchApplyBegin` arm in `event_mapping.rs:452-459` is correspondingly deleted since the unified item-completed path now generates it via `as_legacy_begin_event`.
- Test coverage at `protocol/protocol.rs:4703-4738` (`item_started_event_from_file_change_emits_patch_begin_event`) and `:4769-4805` (`item_completed_event_from_file_change_emits_patch_end_event`) pins the round-trip contract at the legacy-event boundary — asserting `PatchApplyBegin{call_id:"patch-1", auto_approved: true, ...}` and `PatchApplyEnd{success: true, status: Completed, stdout: "Done!"}` survive the new-item → legacy-event conversion. Plus `v2.rs:10324-10352` pins the `CoreTurnItem::FileChange → ThreadItem::FileChange` projection with the `convert_patch_changes` helper invocation.
- Thread-history builder updates at `thread_history.rs:361/383` (adding `TurnItem::FileChange(_)` to the no-op match arms) correctly recognize that file-change items don't need delta-rebuild support since they're emitted whole — same shape as `Reasoning`/`WebSearch`/`ImageGeneration`.

## Nits
- `as_legacy_end_event` at `protocol/items.rs:418` returns `Option<EventMsg>` keyed on `self.status.clone()?` — the `?` early-return swallows items in the in-progress state silently. A future caller wanting "show me everything including in-flight patches" via the legacy event stream would silently get nothing; an inline comment naming "legacy end-event only emitted for terminal states" would help.
- `as_legacy_begin_event` at `:409` always sets `auto_approved: self.auto_approved.unwrap_or(false)` — defaulting to `false` is conservative but a `None` value at the new-item layer is semantically distinct from `Some(false)` (was-not-recorded vs explicitly-not-approved). Probably fine for the legacy bridge but worth a comment naming the intentional information loss.
- The `TurnItem::FileChange` arm in `as_legacy_events` at `protocol/items.rs:456-459` calls `as_legacy_end_event(String::new())` — the empty turn-id string would propagate into `PatchApplyEndEvent.turn_id`. Callers reconstructing legacy events from item history won't have the original turn-id context; the comment in `protocol.rs:1884-1892` (`HasLegacyEvent for ItemCompletedEvent`) shows the right path uses `self.turn_id.clone()`. Worth a `// turn_id is filled at the wrapping event level` comment at `:457` or accepting an explicit `turn_id` argument to make the contract loud.
