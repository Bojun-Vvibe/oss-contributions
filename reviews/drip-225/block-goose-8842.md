## Nits
- `Regex::new(matcher)` at `:144` runs on every hook event — for a busy agent emitting `BeforeToolCall` thousands of times, this is unnecessary CPU. Compile once at `from_config` time (or lazily via `OnceCell`) and store the compiled `Regex` in the `HookEntry`. The current shape would also re-log the "Invalid hook matcher regex" warning every single emit if a misconfigured matcher exists, polluting logs at scale.
- `execute_command_hook` at `:243-291` uses `expanded.split_whitespace().collect()` to tokenize the command at `:251` — this breaks on commands with spaces in arguments (e.g. `~/scripts/check-safety.sh "this is one arg"`). Should use a proper shell tokenizer (`shell-words` crate) or document the limitation in the JSDoc/example. The `shellexpand::tilde` at `:249` correctly handles `~`, so the partial work was done.
- The HTTP hook path at `:293-322` constructs a fresh `reqwest::Client::new()` per invocation — should be a shared client (one per `HookManager` instance) to enable connection pooling and avoid the per-call TLS handshake cost. For a `BeforeToolCall` hook firing hundreds of times per session this is real overhead.
- No timeout enforcement on the `child.stdin.take()` write at `:262-265` — if the hook process never reads stdin, the parent would block on `stdin.write_all` indefinitely. The outer `tokio::time::timeout` at `:268` only covers `wait_with_output`, not the stdin write. Wrap the stdin operations in their own timeout.
- `parse_decision` accepts both `"block"` and `"deny"` strings at `:303`/`:316` — good for ecosystem compatibility but the `block` field at `:328` is bool-only. Inconsistency could confuse users writing custom hooks; either standardize on a `"decision"` enum or document the precedence order (`decision` → `action` → `block`) explicitly.
- `agent.rs:285-307` exposes `emit_hook` as a public method but I don't see callers in this diff — the actual integration into the tool-call lifecycle, session start/end, and reply-begin/end paths must come in a follow-up PR. Worth either including those wiring points in this PR (so the feature actually does something) or naming the follow-up issue/PR explicitly so reviewers can verify the surface area.
- The whole `hooks.rs` module is import-reordered (a separate `use anyhow::{Context, Result, anyhow}` rust-style sort at `agent.rs:7` and many similar) which makes review harder by inflating the diff. Worth splitting the import-reorder commit from the feature commit for cleaner history.

**Verdict: needs-discussion**

The feature scaffolding is well-shaped and the test coverage of `parse_decision` is solid, but `emit_hook` has no callers in this diff so the user-visible surface is zero. Worth confirming with the author whether the integration sites (agent loop's `before_tool_call`/`after_tool_call`, session lifecycle in CLI/server) are intentionally split into a follow-up PR for review-tractability, or accidentally omitted. Also worth discussing whether the per-emit regex compilation and per-call HTTP client construction are acceptable for the v1 ship or should be addressed before merge given this is a hot-path subsystem.
