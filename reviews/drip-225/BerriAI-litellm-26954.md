## Notes
- CVE-class TOCTOU fix at the rate-limit kernel boundary. New `CHECK_AND_INCREMENT_BY_N_SCRIPT` Lua script at `parallel_request_limiter_v3.py:84-160` implements all-or-nothing atomic check-and-increment in two passes inside Redis (Pass 1 reads state and validates against limit at `:101-127`; Pass 2 only runs when *every* descriptor passed and applies increments at `:130-156`). Return shape `{0, new_counter_1, ...}` on OK or `{1, descriptor_index, current_counter, limit}` on over-limit at `:124/:131` is the right "exit early without writes" pattern that cannot leave partial state.
- `atomic_check_and_increment_by_n` Python wrapper at `parallel_request_limiter_v3.py:683-739` is the surface every caller migrates to. Critical defense: the in-memory fallback path at `:738-906` is gated on `self._check_and_increment_lock = asyncio.Lock()` (declared at `:265`) so single-process deployments without Redis still get TOCTOU safety via per-process serialization. Two-tier (Redis Lua → in-memory lock) is the correct shape — multi-replica deployments rely on Redis, single-replica deployments rely on the asyncio lock, neither has a window where read and increment are separable.
- `batch_rate_limiter.py:166-211` collapses the prior 3-step (`should_rate_limit(read_only=True)` → manual capacity check at `:188-203` → `async_increment_tokens_with_ttl_preservation` at `:240-247`) sequence into one `atomic_check_and_increment_by_n` call with parallel `increments` array carrying `{requests: batch_usage.request_count, tokens: batch_usage.total_tokens}` per descriptor — eliminating the entire TOCTOU window between the read and the increment. Builds correct `OVER_LIMIT`-shape error at `:204-211` for the failure path.
- `dynamic_rate_limiter_v3.py:463-491` does the same collapse for the dynamic priority-routing case, with the load-bearing `enforced_descriptors` selection at `:471-473` (`model_saturation_check` always enforced; `priority_descriptors` enforced only when `should_enforce_priority`) so the atomic call only writes counters for descriptors that actually participate in the enforcement decision — preserving the prior "informational priority check" semantics where unenforced descriptors don't consume capacity.
- Test file `test_rate_limiter_toctou.py:35-153` proves the fix shape works against a deliberately-constructed TOCTOU window. `_make_phase1_barrier` at `:36-69` is the load-bearing test primitive: pre-fix it forces all N coroutines through `should_rate_limit(read_only=True)` *before* any reaches the increment, mimicking real concurrent-load behavior; post-fix it times out on each call because the new lock+Lua serializes execution, so only the lock-holder reaches the barrier and the others are queued. The `assert len(successes) <= max_allowed_successes` assertion at `:147` pins the post-fix invariant by exact arithmetic (5 concurrent batches of 40 tokens against TPM=100 should produce ≤2 successes, not 5).

## Nits
- `validateUrl`-style nothing here, but the Lua script's `Pass 1 → Pass 2` shape allocates a `descriptor_state` table and walks descriptors twice. For typical N=1-3 descriptors this is fine; for batch operations with many descriptors a single-pass with a write-buffer would be marginal but probably not worth the complexity given Lua scripts are bounded-time.
- `_atomic_check_and_increment_in_memory` at `:738` reads `internal_usage_cache` with `local_only=True` — correct for the "no Redis" fallback path, but worth a comment naming why we don't fall back to the dual-cache redis path (answer: the Lua script *is* the redis path, so falling through to in-memory means redis is unavailable; reading dual-cache here would just retry redis). One-line comment would prevent a well-meaning future contributor from "fixing" this.
- The `try/except` at `:705-712` catches *any* Lua failure and falls through to in-memory; this is correct (preserves availability) but masks Redis configuration bugs (e.g. CLUSTER mode key-slot mismatch on the `{descriptor_key:value}` hash-tag pattern) since the warning is at WARN level. Consider promoting to an error-level metric counter so operators see persistent fall-through rather than only seeing it in logs.
- Tests are excellent but only cover batch + dynamic-v3 paths. The `should_rate_limit` callers in `parallel_request_limiter_v3.py` (the v3 hot path itself) still uses non-atomic check+increment in some arms — worth a follow-up scoping issue confirming whether the main parallel-request hot path also needs migration to `atomic_check_and_increment_by_n`, or whether its single-step nature already provides atomicity.

**Verdict: merge-as-is**
