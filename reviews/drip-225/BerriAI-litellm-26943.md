## Notes
- Three-front SSRF/path-traversal fix at the URL-construction boundary. Arize Phoenix client at `arize_phoenix_client.py:11-19` adds `_sanitize_id` rejecting `/`, `\`, `#`, `?` characters and `..` substring before `urllib.parse.quote(identifier, safe="")` URL-encoding. Applied at the single call site `:68-69` (`safe_id = _sanitize_id(prompt_version_id); url = f"{self.api_base}/v1/prompt_versions/{safe_id}"`) — the right "fail-closed at the value layer adjacent to the URL template" shape rather than per-method validation.
- BitBucket fix at `bitbucket_client.py:13-25` is shape-correct for path-aware identifiers: splits on `/`, rejects only `..` segment (allowing intra-path slashes since BitBucket APIs nest `repos/.../src/.../path/to/file`), then per-segment URL-encodes with `urllib.parse.quote(part, safe="")`. Applied symmetrically at all three URL-construction sites: `get_file_content:91-92`, `list_files:138-139`, `get_file_metadata:232-233` — confirming no path-template surface was missed.
- AssemblyAI fix at `assembly_passthrough_logging_handler.py:208-216` inlines the same predicate (`any(c in transcript_id for c in ("/", "\\", "#", "?")) or ".." in transcript_id`) — minor inconsistency with the other two clients (no shared helper) but functionally equivalent. The inlining is defensible since this file lives in a different module tree.
- Test coverage at `test_assemblyai_unit_tests_passthrough.py:139-196` and `test_arize_phoenix.py:284-256` and `test_bitbucket_integration.py:373-415` pins the anti-behavior contract by exact value: each rejects test asserts `pytest.raises(ValueError, match="disallowed characters"|"path traversal"|"URL special characters")` so the *error message itself* is part of the contract — a future "let's combine these errors" refactor would break tests, forcing visibility. The load-bearing `assert ".." not in called_url` at `test_assemblyai_unit_tests_passthrough.py:196` and the encode-spaces test at `test_bitbucket_integration.py:402` (`assert _sanitize_file_path("prompts/my prompt.prompt") == "prompts/my%20prompt.prompt"`) prove the URL-encoding actually fires for the non-rejected path.
- Predicate ordering matters: in `_sanitize_id` at `arize_phoenix_client.py:13-18` the `/`-check fires *before* the `..`-check, which is why `_sanitize_id("../../projects")` raises `disallowed characters` (slash arm) rather than `path traversal` (dotdot arm) — and the test at `test_arize_phoenix.py:288-291` documents both arms with the exact expected error message, so the order is pinned.

## Nits
- The three clients implement `_sanitize_*` independently with slightly different shapes (Arize: identifier without slashes; BitBucket: path with slashes; AssemblyAI: inlined). A shared `litellm/_security/url_path.py` with `sanitize_identifier(s, allow_slash=False)` and `sanitize_path(s)` would dedupe and ensure the next provider integration to land doesn't re-derive (or worse, miss) the predicate. Worth a follow-up issue.
- `_sanitize_file_path` at `bitbucket_client.py:18-19` rejects `?` and `#` but accepts spaces (correctly URL-encoded later). However it does not reject other URL-meaningful characters like `;` or backslash — BitBucket's path semantics likely don't care, but a tighter explicit allow-list (`[A-Za-z0-9._/-]`) instead of a deny-list would defend against the next URL-parser quirk.
- `_sanitize_id` at `arize_phoenix_client.py:11-23` does not reject `\0` (null byte) — Python `str` would carry it through `quote` as `%00`, which most HTTP servers reject but worth defending against explicitly given the security context.
- AssemblyAI's inlined check at `:208-216` should ideally be extracted to match the helper-function pattern of the other two clients, both for grep-discoverability and to ensure the predicate stays in sync if the threat model expands.

**Verdict: merge-as-is**
