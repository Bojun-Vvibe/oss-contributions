## Notes
- Establishes per-signal HTTP OTLP endpoint contract at the config boundary. New `otlpTracesEndpoint`/`otlpLogsEndpoint`/`otlpMetricsEndpoint` fields on `TelemetrySettings` at `config/config.ts:218-223` are documented as "Used as-is without path appending" in the JSDoc — the right "explicit override means you take responsibility for the path" contract that respects operator intent. Threading at `:750-753` (the `Config` constructor) and getter accessors at `:1995-2003` are all-or-nothing additive (no breaking change to the existing `getTelemetryOtlpEndpoint()` callers). `getTelemetryOtlpEndpoint()` return type widening from `string` to `string | undefined` at `:1987` is the load-bearing breaking change for downstream consumers — the diff *should* surface every call site that previously assumed non-null, and the existing `?? DEFAULT_OTLP_ENDPOINT` fallback at `:1988` keeps the behavior backward-compatible.
- `resolveHttpOtlpUrl(baseEndpoint, signal)` helper at `telemetry/sdk.ts:51-67` is the right "smart-append the signal-specific path" shape: parses the URL, strips trailing slashes via `url.pathname.replace(/\/+$/, '')`, checks if the path already ends with the signal-specific suffix (`v1/traces`/`v1/logs`/`v1/metrics`) at `:62-64`, and only appends if not — meaning operators who set `otlpEndpoint=https://collector.example.com/v1/traces` (mistakenly thinking they need to specify the full path) get the same result as `otlpEndpoint=https://collector.example.com`. Idempotent path construction is the right default.
- `LogToSpanProcessor` at `log-to-span-processor.ts:33-141` is a thoughtful workaround for backends (per the docstring at `:21-27`: "Alibaba Cloud") that support traces+metrics but not logs over OTLP. The `onEmit` arm at `:50-95` constructs a `ReadableSpanLike` directly per log record, deriving `traceId` from `session.id` via `createHash('md5').update(sessionId).digest('hex')` at `:78-80` so all log-derived spans within a session group under one trace ID — load-bearing for backend visualization. Random per-event `spanId` at `:81` keeps spans distinct.
- The bridging activation at `sdk.ts:170-176` is the load-bearing wiring: when `logsUrl` is undefined but `tracesUrl` exists, `logToSpanProcessor = new LogToSpanProcessor(new OTLPTraceExporterHttp({ url: tracesUrl }))`. Note this constructs a *separate* trace exporter instance dedicated to log-bridging rather than reusing the main `spanExporter` — correct because the two have different shutdown lifecycles. The `flushTimer.unref()` at `:46` prevents the periodic flush from blocking process exit.
- Per-signal env var resolution test at `telemetry/config.test.ts:166-176` pins the OTEL spec contract: `OTEL_EXPORTER_OTLP_TRACES_ENDPOINT` resolves to `otlpTracesEndpoint`, `OTEL_EXPORTER_OTLP_LOGS_ENDPOINT` to `otlpLogsEndpoint`, etc. Plus the `QWEN_*` variant at `:179-189` takes precedence over `OTEL_*` (`expect(resolved.otlpTracesEndpoint).toBe('http://qwen-traces:4318/v1/traces')`) — pinning the qwen-namespace-wins contract that lets operators differentiate qwen-cli telemetry from a system-wide OpenTelemetry config.
- `validateUrl` at `sdk.ts:105-115` is a defensive predicate: invalid URLs trigger `diag.error(...)` and the signal is skipped, rather than throwing and breaking telemetry init. Right call — telemetry failures should never crash user-visible CLI flows. The `diag.setLogger(..., DiagLogLevel.ERROR)` at `:33` (down from `INFO`) is a noise-reduction sibling change that prevents OTel internal chatter from polluting CLI output.

## Nits
- Removal of `ConsoleSpanExporter`/`ConsoleLogRecordExporter`/`ConsoleMetricExporter` at `sdk.ts:18-29` and the corresponding fallback arm at `:142-147` (`else { spanExporter = new ConsoleSpanExporter(); ... }`) means an operator who enables telemetry without configuring any endpoint *and* without `telemetryOutfile` now gets… nothing (the exporters are `undefined`). Previously they got console output, which while noisy was at least debuggable. A single-line `debugLogger.warn('Telemetry enabled but no exporter configured; data will not be exported')` would help operators discover misconfiguration.
- gRPC arm at `sdk.ts:201-205` correctly warns that per-signal endpoints aren't supported but only at `debugLogger.warn` level — a user who set `QWEN_TELEMETRY_OTLP_TRACES_ENDPOINT` expecting it to work with gRPC would silently see the override ignored. Promote to a startup error or surface in the `init` summary so operators see the misconfiguration loudly.
- `LogToSpanProcessor.onEmit` at `:60-67` does `attributes[key] = typeof value === 'object' ? JSON.stringify(value) : value` — this stringifies arrays and objects but the resulting string-typed attribute loses the OTel semantic-convention shape (an array attribute should carry array semantics in the backend). Backends that index attributes by type would treat the JSON string as a regular string. A comment naming this lossy conversion, or a `_json` suffix on the key, would document the trade-off.
- `randomHexString(length)` at `:154-158` uses Web Crypto `crypto.getRandomValues` — works in Node 19+ via the global `crypto` object but not in older Node versions still in the test matrix. The `node:crypto` import at `:18` is already present for `createHash`; using `crypto.randomBytes(length/2).toString('hex')` would be both faster and version-portable.
- `LogToSpanProcessor` has no unit test coverage in this PR — the buffer-flush path, the session-id-derived trace ID, the duration-from-attribute logic at `:69-75`, and the error-status derivation at `:177-194` are all load-bearing and untested. A vitest at `log-to-span-processor.test.ts` exercising "emit 3 records with same session.id, flush, assert exporter received 3 spans with same traceId" would pin the contract.

**Verdict: merge-after-nits**
