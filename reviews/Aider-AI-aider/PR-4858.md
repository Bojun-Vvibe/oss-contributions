# Context

aider summarizes long chat histories to keep them under the
model's context limit. The existing `ChatSummary` flow is
*reactive*: when `done_messages` exceeds a threshold, aider
spins a background thread to summarize, but the *current*
turn must wait for that summarization to complete before
proceeding (otherwise the next prompt would exceed the limit).
The user-visible result is a noticeable pause whenever
compaction happens.

This PR adds a *proactive* compaction strategy — "double
buffer" — riffing on Mark Lubin's "hopping context windows"
post (linked in the docstring). The idea: at 60% of capacity,
start summarizing in the background while the model still has
full attention. When usage hits 85%, swap in the pre-computed
summary instantly. The key win is that the user never waits
for compaction; it's always already done by the time it's
needed.

+500/-0 across 4 files. New module (`aider/double_buffer.py`),
integration in `base_coder.py`, two new CLI flags, one test file.
Opt-in via `--double-buffer`.

# Diff walkthrough

**The state machine.** `DoubleBufferManager` tracks three
phases:

```python
class Phase(Enum):
    NORMAL = "normal"
    CHECKPOINT_PENDING = "checkpoint_pending"
    CONCURRENT = "concurrent"
```

`NORMAL` → no action needed. `CHECKPOINT_PENDING` → background
summarization in flight. `CONCURRENT` → checkpoint complete,
summary held in reserve, waiting to be swapped at the swap
threshold.

The init validates threshold ordering:

```python
if not (0 < checkpoint_threshold < swap_threshold <= 1.0):
    raise ValueError(
        f"Invalid thresholds: checkpoint={checkpoint_threshold}, "
        f"swap={swap_threshold}. Must satisfy 0 < checkpoint < swap <= 1.0"
    )
```

This is the right shape for "configurable but constrained" —
the user picks values, the manager refuses nonsense
combinations at construction time rather than producing
mysterious behavior at runtime.

**The threshold checks** are deliberately phase-gated:

```python
def should_checkpoint(self, current_tokens, max_tokens):
    if self.phase != Phase.NORMAL:
        return False
    # ...
def should_swap(self, current_tokens, max_tokens):
    if self.phase != Phase.CONCURRENT:
        return False
    if self.checkpoint_summary is None:
        return False
```

`should_checkpoint` only fires when no checkpoint is in flight;
`should_swap` only fires when a checkpoint has *completed* and
its summary is held. This is what makes the state machine safe
under aider's "summarize_start is called after every turn"
discipline — the manager won't double-fire.

**The background work.** `_run_checkpoint` runs in a daemon
thread:

```python
def _run_checkpoint(self, summarizer):
    try:
        result = summarizer.summarize(self._checkpoint_messages_snapshot)
        if result:
            summary_parts = []
            for msg in result:
                content = msg.get("content", "")
                if content:
                    summary_parts.append(content)
            self.checkpoint_summary = "\n".join(summary_parts)
            self._checkpoint_result = result
            self.phase = Phase.CONCURRENT
            self.generation += 1
        else:
            self.phase = Phase.NORMAL
    except Exception as err:
        self.phase = Phase.NORMAL
        self.checkpoint_summary = None
```

On success, the result lands in `_checkpoint_result` and the
phase advances. On failure or empty result, the phase resets
to `NORMAL` so the next turn will retry. The exception swallow
is wide (`Exception`) but appropriate — a background thread
should never raise.

**The integration in `base_coder.py`** swaps in the new
behavior when the flag is set:

```python
def summarize_start(self):
    if self.double_buffer is not None:
        self._double_buffer_check()
        return
    # ...legacy path...
```

The `_double_buffer_check` method orchestrates the state
machine:

```python
def _double_buffer_check(self):
    sized = self.summarizer.tokenize(self.done_messages)
    current_tokens = sum(tokens for tokens, _msg in sized)
    max_tokens = self.summarizer.max_tokens

    if self.double_buffer.should_swap(current_tokens, max_tokens):
        if self.double_buffer.phase.value == "checkpoint_pending":
            self.double_buffer.wait_for_checkpoint()
        if self.double_buffer.phase.value == "concurrent":
            swapped = self.double_buffer.complete_swap()
            if swapped:
                self.done_messages = swapped
                return

    if self.double_buffer.should_checkpoint(current_tokens, max_tokens):
        self.double_buffer.begin_checkpoint(self.done_messages, self.summarizer)
        return

    # Fallback: if too big and no double-buffer action, use standard summarization
    if self.summarizer.too_big(self.done_messages):
        # ...legacy path...
```

The fallback at the end is the safety net: if for any reason
the double-buffer machinery hasn't produced a summary by the
time the messages are *actually* too big, the legacy reactive
path takes over. That's the right belt-and-suspenders posture
for a new feature — the worst case degrades to the existing
behavior.

# Risks

1. **Threading without a lock.** `_run_checkpoint` writes to
   `self.checkpoint_summary`, `self._checkpoint_result`,
   `self.phase`, `self.generation` — all of which are read by
   the main thread. Python's GIL makes simple attribute
   assignments atomic, but the *combination* "phase = CONCURRENT
   *and* `_checkpoint_result` is set" is not atomic. A reader
   on the main thread could observe `phase == CONCURRENT` with
   `_checkpoint_result is None`. The current code defends with
   the `if self._checkpoint_result is None: log.error("invalid
   state")` check inside `complete_swap`, but the race is real.
   A `threading.Lock` (or transitioning phase *after* setting
   the result) would close it.
2. **`wait_for_checkpoint(timeout=30)` is a synchronous block
   in a "we want responsiveness" feature.** If the checkpoint
   hasn't completed by 85%, the user gets up to 30s of pause
   — exactly the pause this PR is meant to eliminate. The
   timeout is right (better than infinite wait), but the
   intended user experience needs a "I'm waiting on the
   checkpoint" indicator stronger than the verbose-only
   message currently emitted.
3. **Summary "extraction" loses structure.** `summarizer.summarize()`
   returns a list of message dicts, but `_run_checkpoint`
   joins their `content` fields with `\n` into a single
   string (`self.checkpoint_summary`). Then `complete_swap`
   returns the *original* `_checkpoint_result` (the list).
   So `checkpoint_summary` is never actually used for
   replacing messages — only for the log line and (presumably)
   future telemetry. The naming suggests otherwise. Either
   delete `checkpoint_summary` (it's dead) or make it the
   actual swap payload.
4. **`generation` is incremented on success but never on
   failure or reset.** Good — but it's also never persisted.
   Long-running aider sessions can rack up many generations,
   which would be useful diagnostic data ("the user has
   compacted 14 times in this session, maybe time for a
   `/clear`"). A passive log when generation crosses
   thresholds would surface this.
5. **`begin_checkpoint` snapshots `done_messages` via
   `list(done_messages)`** — shallow copy. The messages
   themselves are not copied. If the main thread mutates a
   message dict in-place (which it shouldn't but can), the
   summarizer thread sees the mutation. Worth a defensive
   `copy.deepcopy` if message mutations are even possible.
6. **No unit test for the race "swap arrives during
   checkpoint".** The state machine is correct in logic, but
   testing the threading behavior requires injecting delays
   or using `threading.Event` to coordinate. Worth adding.
7. **`reset()` joins the thread for 5s.** If the thread is
   genuinely stuck (a slow LLM call), aider's session-end
   waits 5s. That's tolerable but worth documenting.
8. **The 60%/85% thresholds are sensible defaults but
   model-independent.** Different models have different "I
   start losing coherence" points (Claude is forgiving up to
   90%; some models degrade visibly past 70%). A future
   refinement would be model-family-specific defaults.
9. **The opt-in flag is good** for a new feature but the
   PR doesn't show benchmarks. Without "with double-buffer:
   median pause = 0.2s; without: median pause = 6.4s" the
   maintainer is approving on faith. Adding even a hand-
   measured pair of numbers in the PR description would help.
10. **`max_chat_history_tokens` vs `summarizer.max_tokens`.**
    The integration uses `self.summarizer.max_tokens`, but
    the docstring of `should_checkpoint` says
    "model.max_chat_history_tokens". Worth confirming these
    are the same value or naming the source of truth
    consistently.

# Suggested questions for the author

1. The `phase = CONCURRENT` transition isn't atomic with the
   `_checkpoint_result` assignment. Should there be a
   `threading.Lock` around the phase transitions?
2. `checkpoint_summary` (the joined string) appears dead —
   `complete_swap` returns `_checkpoint_result` instead. Is
   the string field intended for telemetry, or should it be
   removed?
3. Are there benchmarks for the user-visible pause reduction
   compared to the legacy reactive path?
4. The 60%/85% thresholds — were these tuned empirically, or
   chosen for the principle of "summarize while attention is
   still there"?
5. The `reset()` 5s thread join is reasonable, but what
   happens if `summarizer.summarize()` legitimately takes
   longer than 5s (large history, slow model)? The thread
   keeps running as a daemon — does it pollute the next
   session?

# Verdict

**Approve, with the threading-correctness follow-up.** This is
a thoughtful feature: a real user-experience win (proactive
compaction eliminates the pause), shipped as opt-in (no risk
to existing users), with a fallback to the legacy path (no
risk if the new machinery misbehaves). The state machine is
clear, the threshold validation is principled, the integration
in `base_coder.py` is small and well-bounded.

The two real concerns are (a) the lockless phase/result
transition (likely benign in practice but a race nonetheless)
and (b) the dead-looking `checkpoint_summary` field. Neither
blocks landing — the lockless race would manifest as an
"invalid state" log line and fall through to the legacy path,
which is the desired safety net. But both deserve a follow-up.

# What I learned

Three takeaways, all about background work in interactive CLI
tools.

**First**, the *proactive vs reactive* split for resource
management. Most batch-job designs are reactive: wait until you
hit the limit, then do the expensive work. Most user-experience
designs are proactive: do the expensive work *before* you hit
the limit, so the user never sees the cost. The shift from
reactive summarization (block when too big) to proactive (start
at 60%, swap at 85%) is the same shift that distinguishes a
laggy UI from a snappy one. The cost is "you might do work that
wasn't needed" (if the conversation ends before hitting 85%);
the benefit is "no user ever waits for compaction".

The general pattern: any time you have a *resource exhaustion
event* that requires expensive recovery, ask whether the recovery
can be pre-computed when there's still resource to spare. The
answer is "almost always yes", and the cost is almost always
worth it for an interactive product.

**Second**, the *state-machine + phase guards* shape for
asynchronous coordination. The naive shape for "in-flight async
work" is a boolean: `is_checkpointing`. That's fine for one
operation but doesn't model the *result holding* phase between
"checkpoint done" and "swap consumed". The PR's three-phase
enum (`NORMAL` → `CHECKPOINT_PENDING` → `CONCURRENT` →
`NORMAL`) makes the lifecycle explicit and lets each method
reject calls outside its valid phase. The `should_checkpoint`
returning `False` when phase isn't `NORMAL` is what prevents
the manager from launching N concurrent checkpoints — much
cleaner than a "is_checkpointing" flag with manual locking.

The shape generalizes: when a feature has a "result is held
between produce and consume" phase, model it explicitly in the
state machine. Don't conflate "in flight" with "holding result"
with "idle" by reusing the same boolean — the cost in clarity
is too high.

**Third**, the *opt-in + fallback* discipline for behavior
changes. The PR's `--double-buffer` flag means existing users
see no change unless they ask for it. The fallback path inside
`_double_buffer_check` (legacy reactive summarization if
double-buffer hasn't kicked in) means even users who *do* opt
in get the legacy guarantee as a safety net. Together these
give the maintainer strong confidence that landing this PR
won't regress any existing user — the worst case is "feature
inactive" or "feature inactive with extra logging". This is
the shape that lets large-impact behavior changes ship safely.
The cost is one CLI flag and one fallback branch; the benefit
is reviewer confidence and zero rollback pressure if a bug
slips through.

The fourth meta-lesson: the linked blog post in the docstring
("https://marklubin.me/posts/hopping-context-windows/") is
small but important — when a PR implements a published idea,
linking the source helps reviewers (and future maintainers)
understand the *theory* behind the implementation, not just
the code. More PRs should do this.
