# Context

Aider edits files. By default, when running inside a git repo,
it commits its changes — that's the auditable trail of "AI
edited my code." Many users disable auto-commits (`--no-auto-commits`
or equivalent) because they prefer to review and commit
manually, or because they're using aider on files that aren't
under version control at all.

The `/add` command is how the user tells aider "include this
file in the chat context, you may edit it." It enforces a
rule: files must live inside the repo. The intent is to keep
the auto-commit feature consistent — aider can only commit
what git can see.

# Problem

The "must be inside the repo" check fired regardless of
whether auto-commits were enabled. So a user with
`--no-auto-commits` who wanted to ask aider to edit
`~/scripts/foo.py` (outside their current repo) was rejected
with a message that didn't apply to their configuration:
they had no commits enabled, so the "we need to be able to
commit it" justification didn't hold.

The bug shape is "two coupled features (in-repo restriction +
auto-commit) where one is configurable but the other always
enforces the coupling." The restriction is sensible *only*
when auto-commits are on; with auto-commits off, the
restriction is friction with no purpose.

# Design

The fix gates the "outside repo" rejection on whether
auto-commits are enabled. Effectively:

```python
if outside_repo and self.coder.auto_commits:
    raise # original rejection
elif outside_repo:
    # allow with a warning, e.g.,
    self.io.tool_warning("File is outside repo; will not be committed.")
    add_file_anyway()
```

The +5/-1 LOC is the conditional rewrite plus a user-facing
warning so the behavior is observable. No new flag, no new
config — the existing `auto_commits` setting is repurposed
to govern *both* the commit behavior and the
where-can-files-live restriction, which is the right level
of coupling because they describe the same underlying
intent.

# Risks

- **Surprise factor for users who disable auto-commits but
  still expect repo discipline.** Some users disable
  auto-commits but want aider to refuse out-of-repo edits
  as a guardrail (e.g., they don't want aider to
  accidentally edit `~/.zshrc`). The previous behavior
  enforced that incidentally; the new behavior doesn't.
  The user-facing warning helps but doesn't prevent. A
  separate `--restrict-to-repo` flag would give users the
  explicit control, but adding it isn't necessary for this
  fix.
- **No test in this PR.** The change is small enough that
  manual verification is reasonable, but a regression test
  asserting "with `--no-auto-commits`, `/add` accepts a
  path outside the repo and emits a warning" would lock
  the behavior in. Easy follow-up.
- **Path traversal implications.** Allowing out-of-repo
  files means aider can now read and write anywhere the
  user has filesystem permission. That was already true
  for users running aider outside any repo at all, so this
  PR doesn't change the security surface — but it does
  expand the scenarios where it's relevant. The warning
  message should make it visible that aider is now editing
  outside the repo.
- **Other in-repo restrictions not audited.** If there are
  other places in `commands.py` (or elsewhere) that
  enforce "must be in the repo" on the same auto-commit
  premise, they'd need the same fix. A grep for "outside
  the git repo" or similar messages would surface them.

# Suggestions

1. Add a focused regression test: with `auto_commits=False`,
   `/add ~/some/external/file.py` succeeds, file is in the
   chat, warning is emitted.
2. Make the warning message specific about the consequence:
   "File is outside the repository. It will be edited but
   not committed by aider." So the user knows what they
   gave up by being outside the repo.
3. Audit `commands.py` and `coder.py` for other "must be in
   repo" gates that are coupled to `auto_commits`. Same
   one-line conditional fix wherever found.
4. Consider an explicit `--allow-external-files` flag for
   the case "I want auto-commits but also want to edit one
   external file." Out of scope for this PR but a clean
   follow-up.

# Verdict

**Ship it.** The fix correctly identifies that the
"in-repo only" restriction was a side-effect of the
auto-commit assumption, and gates it on the same setting.
4-line change, real friction removed for a documented user
workflow. The lack of a regression test is the only thing
keeping this from being a textbook small fix.

# What I learned

When a feature has a conditional behavior (auto-commit on
vs. off), audit every code path that implicitly assumes the
"on" branch. Restrictions like "must be in the repo" often
exist *because* of a downstream feature, and when that
feature is configurable the restriction should be too.
The general rule: any check whose justification is "because
later we'll do X" should be gated on whether X is actually
going to happen. Otherwise the check becomes a phantom
constraint that frustrates users without serving its
original purpose.
