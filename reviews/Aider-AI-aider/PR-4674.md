# Context

aider supports a `git: false` configuration mode where it
operates without a git repo at all — useful for one-off
scripting, prototyping in `/tmp`, or working in directories
that aren't (and shouldn't be) git-managed. Most aider commands
work in this mode, but a small set of git-aware paths weren't
fully audited when `git: false` was added, and they bake in the
assumption that "if a file isn't in the repo, it can't be
edited".

The `/add` command (promote a read-only file to editable) was
one of those missed paths.

# Problem

Concrete repro:

1. User runs aider with `git: false`.
2. User does `/read-only README.md` (file is now in
   `abs_read_only_fnames`).
3. User does `/add README.md` to promote it to editable.
4. aider prints
   `"Cannot add /home/.../README.md as it's not part of the
   repository"` and refuses.

The error message is wrong on its face — there *is* no
repository in `git: false` mode, so "not part of the
repository" is a tautology, not a meaningful gate. The
underlying check is:

```python
if self.coder.repo and self.coder.repo.path_in_repo(matched_file):
    # promote to editable
```

When `self.coder.repo` is `None` (which is exactly what
`git: false` produces), the condition is `False` and the
promotion path is never reached. The else branch falls through
to the "cannot add" message.

# Design

Replace the single condition with a two-case branch:

```python
# aider/commands.py:866 (after)
if self.coder.repo:
    can_edit = self.coder.repo.path_in_repo(matched_file)
else:
    can_edit = abs_file_path.startswith(self.coder.root)

if can_edit:
    self.coder.abs_read_only_fnames.remove(abs_file_path)
    self.coder.abs_fnames.add(abs_file_path)
    ...
```

The `git: false` branch checks `abs_file_path.startswith(self.coder.root)`
— is the file under aider's project root? That's the right
gate for the no-git case: aider should still refuse to promote
files outside its working scope (no `/add ../../../etc/passwd`),
but should allow anything under the working directory.

The `aider/repo.py` change (+2/-1) is just an `extract local
variable` refactor:

```python
# Before:
return self.normalize_path(path) in tracked_files
# After:
normalized = self.normalize_path(path)
return normalized in tracked_files
```

This is essentially a no-op stylistic change. The PR doesn't
explain it, but it likely landed alongside debug-print
investigation while the author was tracing the bug. It's
harmless but adds noise to a 2-file PR — could have been
either separated or skipped.

# Risks

- **`startswith(self.coder.root)` is path-prefix-based, not
  realpath-based.** A file at `/home/user/projectroot-evil/foo`
  would `startswith` `/home/user/projectroot` and slip through
  the check. The `root` is *almost certainly* canonicalized
  earlier, but a `os.path.commonpath([abs_file_path, root]) ==
  root` check would be strictly more correct than `startswith`.
  Edge case, but worth tightening.
- **Symlink edge cases.** If `abs_file_path` is a symlink whose
  target is outside `self.coder.root`, the check passes (it
  looks at the symlink path, not the target). Whether that's a
  bug depends on the threat model. Probably fine for aider
  (the user is intentionally pointing at the file), but worth
  documenting the choice.
- **No regression test.** Two-line change with no test means a
  future refactor that "simplifies" the if/else back into a
  single condition will silently regress. A small test that
  exercises `/add` in `git: false` mode and asserts the file
  ends up in `abs_fnames` would lock the fix in.
- **Asymmetric error message.** The "cannot add" branch
  (when neither condition is met) still prints the
  "not part of the repository" message even in `git: false`
  mode. So if a user tries to `/add` a file outside
  `self.coder.root` with `git: false`, they get a misleading
  error. Should be conditional on `self.coder.repo` to print
  the appropriate message ("not under project root" vs "not
  part of the repository").
- **The `repo.py` cosmetic change** has zero behavior impact
  but should arguably not be in a bug-fix PR. It clutters the
  diff and review.

# Suggestions

1. Add a test in `tests/basic/test_commands.py` that exercises
   `/add` against a read-only file under `git: false`.
2. Tighten the path check from `startswith` to
   `os.path.commonpath([abs_file_path, root]) == root`.
3. Make the error message branch on `self.coder.repo` so
   "not part of the repository" is only printed when there
   actually is a repository.
4. Drop the `repo.py` cosmetic change or move it to a
   separate PR. Keeps the bug-fix PR's diff strictly focused
   on the bug.
5. Audit other `/add`-adjacent commands (`/drop`,
   `/read-only`, `/clear`) for the same `if self.coder.repo
   and ...` pattern that fails closed under `git: false`.

# Verdict

**Ship it.** Real fix to a real bug; the user impact is
"a documented mode of operation simply doesn't work for one
common command". Fix is mechanically correct, even if it
could be tighter on path canonicalization. File the test and
the error-message follow-ups.

# What I learned

This is the canonical "feature flag added late, audit
missed a code path" bug. `git: false` was added to support
no-git workflows; most code paths were updated; one wasn't.
The shape of the bug — `if self.coder.repo and ...` works
but `if self.coder.repo and ...` else branch doesn't reach
the *equivalent no-git logic* — is structurally identical
to many "feature added late" bugs across many codebases.

The fix pattern is **derive a capability from each backend,
then act on the capability uniformly**. Here, the question
"can this file be edited under the current configuration?"
should be answered by a single helper that handles both the
`repo` and `git: false` cases internally:

```python
def can_promote_to_editable(self, abs_path):
    if self.repo:
        return self.repo.path_in_repo(abs_path)
    return self._is_under_project_root(abs_path)
```

Once that helper exists, the call site becomes one line and
no future contributor needs to remember "oh, also handle the
no-git case". This PR moves halfway there (it explicitly
handles both cases at the call site) but doesn't extract the
helper, so the next call site that wants the same check has
to re-derive it.

The reusable insight is **error messages are part of the
contract**. The "not part of the repository" message
*sounds* like an explanation, but it's a leaked-implementation
detail. Under `git: false`, repository is the wrong word
entirely. Error messages that name internal concepts
(repository, index, commit) leak when those concepts are
absent in some configurations. The right error message names
*user-level* concepts (project root, file scope) regardless
of internal state. Worth a sweep across any flag-conditional
code path.
