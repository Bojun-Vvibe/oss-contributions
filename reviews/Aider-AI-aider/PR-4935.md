# Context

aider's test suite hardcodes specific model IDs to verify
behaviors like model-info lookup, vision-model branching, and
`/tokens` command output. These tests rely on `litellm`'s
embedded model database, which periodically prunes models that
the upstream providers have removed from their APIs. When
litellm removes a model, every aider test that referenced it
starts failing with `KeyError` or `model not found`.

This PR is the small periodic clean-up that follows litellm
removals — three models pruned upstream, three test references
updated or deleted.

# Problem

Three models were removed from litellm's database:

1. **`gpt-4-32k`** — pruned in [BerriAI/litellm#20795](https://github.com/BerriAI/litellm/pull/20795).
   OpenAI deprecated the 32k variant in favor of `gpt-4-turbo`
   (128k) and the gpt-4o family long ago; litellm finally
   dropped it from the static model registry.
2. **`claude-3-5-sonnet-20240620`** — pruned in
   [BerriAI/litellm#23400](https://github.com/BerriAI/litellm/pull/23400).
   Anthropic has aged out the dated 20240620 variant in favor of
   the newer Sonnet 3.5 / Sonnet 4.x line.
3. **`gpt-4-vision-preview`** — pruned in the same litellm PR.
   The vision-preview model was an early-access preview that
   was superseded by `gpt-4o` (vision is now built into the base
   model, not a separate endpoint).

Each removal manifested as a test failure on the next litellm
upgrade.

# Design

The fix is mechanically correct and minimal:

- `gpt-4-32k` reference in `tests/basic/test_models.py` is
  *deleted* (lines 46-48). The test was asserting
  `max_input_tokens == 32 * 1024`, which has no obvious
  modern-model equivalent — picking a different 32k model would
  have made the test about *that* model's properties, not about
  the original assertion. Deleting the assertion is the right
  call.
- `claude-3-5-sonnet-20240620` is replaced with
  `claude-sonnet-4-6` in `test_commands.py:539`. The
  `test_cmd_tokens_output` test doesn't care about which Claude
  it is, just that one is instantiated and produces an
  announcement. Picking the most current Sonnet is correct.
- `gpt-4-vision-preview` is replaced with `gpt-4o` in
  `test_cmd_read_only_with_image_file`. `gpt-4o` is the
  canonical successor for vision in the OpenAI lineup. The
  test exercises vision-model code paths, so any vision-capable
  model works.

The PR has no test additions — these *are* the test changes. No
production code touched. The commit count is appropriate for a
clean-up: each model gets its own conventional commit message
describing the removal.

# Risks

- **`gpt-4o` may itself eventually go away.** This is a recurring
  treadmill — every model named in a test eventually gets
  deprecated. No fix in this PR (it's not the right scope), but
  the underlying problem is "test asserts on a moving target".
  A fixture-based "give me a vision-capable model" /
  "give me a Claude model" indirection would let the litellm
  team announce the *capability mapping* once and have aider's
  tests follow automatically.
- **The deletion of the `gpt-4-32k` test removes coverage of
  the 32k-context-class model lookup.** No modern equivalent
  exists that's likely to stay stable, but the *behavior*
  (large-context model has the right `max_input_tokens` value)
  is still worth testing. A test that walks the litellm
  registry and asserts "for any model, `max_input_tokens > 0`
  and matches the registered value" would cover the same
  behavior without naming a specific model.
- **No CI guard against the next pruning event.** This PR is
  reactive — fix tests after they break. A CI step that pins
  the litellm version (or runs a daily test against latest
  litellm and announces breaks) would convert "tests broke
  in main" into "tests will break tomorrow, here's a heads-up".
- **No release-note entry.** Internal-only test fix, so this is
  fine; mentioning it for completeness.

# Suggestions

1. Refactor toward capability fixtures —
   `pytest.fixture: vision_model` returns whatever the current
   canonical vision-capable model is, picked from a small
   table at the top of `conftest.py` that's the *only* place
   to update on the next litellm pruning.
2. Replace the deleted `gpt-4-32k` test with a property-based
   test that walks the litellm model registry and verifies
   internal consistency (no specific model named).
3. Add a CI job that pins litellm to *latest* (separate from
   the production-pinned version) and runs nightly. Catches
   the next pruning event before it lands in main.

# Verdict

**Ship it.** Two-line clean-up that follows three documented
upstream removals. The commit messages reference the litellm
PRs that removed each model, which is exactly the audit trail a
reviewer wants for "why did this test change". Mechanically
correct, scoped tightly.

# What I learned

The "test-suite-as-museum-of-deprecated-model-names" pattern
is endemic to any project that integrates with a fast-moving
upstream. The temptation is to fix each pruning event
individually and move on. The high-leverage move is to
*invert the dependency* — instead of tests naming models,
tests should name *capabilities* (vision, large-context,
function-calling, reasoning) and the test fixtures should
resolve the capability to a current model name. The fixture
becomes the single point of update and the tests stay
stable across years of upstream churn.

The reusable observation is that **deletion is a valid fix**.
The `gpt-4-32k` `max_input_tokens` test had no modern
equivalent worth preserving, so the right move was to delete
the assertion entirely rather than picking an arbitrary
replacement. Many "fix the broken test" PRs over-correct by
finding *something* to assert on, which adds maintenance
burden without adding signal. If the original assertion's
domain is gone, deletion is honest.

The third lesson is the value of the upstream-PR reference in
the commit message. Each model removal in this PR cites the
litellm PR + commit SHA that pruned it. That's the audit
trail that lets a future maintainer reading the git log
understand "why is this test calling itself
`test_cmd_tokens_output` but instantiating Sonnet 4.6
instead of Sonnet 3.5?". Without the citation, the next
pruning event has to re-derive the relationship from first
principles. Always link upstream causes when fixing
downstream symptoms.
