# Context

Aider catches LLM-related exceptions emitted by LiteLLM (the
underlying LLM-router library) to present friendly messages
instead of bare tracebacks. To do this it walks LiteLLM's
exception module and dynamically builds a list of classes to
include in `except` clauses. This list is used in
`except (cls1, cls2, ...) as e:` constructs at runtime.

# Problem

Issue #4615 reported a real bug: some LiteLLM exceptions
weren't being caught. A previous fix (commit `656301c`)
attempted to widen the list by including more classes from
LiteLLM's exception module. That fix introduced a worse
regression (#4724):

> TypeError: catching classes that do not inherit from
> BaseException is not allowed

The `coder` feature — the core agent flow — crashed at
startup. Cause: LiteLLM's exception module exports a class
literally named `ErrorEventError`, which *sounds* like an
exception but is in fact a regular class that doesn't
inherit from `BaseException`. Modern Python (≥ 3.11)
strictly enforces that everything in an `except` tuple must
be an exception class, raising `TypeError` at evaluation
time if it isn't. The wide-net inclusion logic in `656301c`
let `ErrorEventError` through, every `except` site that
touched the list raised at runtime, the `coder` feature
broke completely.

The bug shape is "name-based heuristic that violated a
language invariant the heuristic didn't check." The widening
PR assumed "a class with `Error` in its name is an
exception"; Python disagreed.

# Design

Two surgical edits:

1. **Narrow the filter to actual exception classes.** Before
   appending any class to the list, check
   `issubclass(cls, BaseException)`. This is the correct
   primitive — the same predicate Python uses internally
   when evaluating an `except` tuple. Anything that isn't
   an exception is excluded regardless of name.
2. **Remove `ErrorEventError` from the explicit list** if
   it was named there.

That's the entire production change. +4/-2. The PR also
effectively reverts `656301c`'s overreach while keeping the
intent of catching more exceptions — but only ones that
actually *are* exceptions.

# Risks

- **No regression test.** A test that ensures every entry in
  the dynamically-built list is `issubclass(BaseException)`
  would lock the contract permanently. Without it, the next
  contributor who tries to widen the list could re-introduce
  the bug class. This is the single biggest improvement
  available and costs about 8 lines of test.
- **The original bug (#4615) may not be fully solved.** This
  PR fixes the *regression* (`coder` crashing) and walks
  back the *attempted* widening. If the original missing
  exception isn't a `BaseException` subclass in LiteLLM,
  the original bug returns. From the PR description it
  sounds like the author understands this and considers the
  trade acceptable: the "wider net" approach was wrong, so
  the right path is to identify the specific missing
  exception(s) and add them by name once verified to be
  real exception classes.
- **LiteLLM upstream may rename `ErrorEventError`.** If
  upstream ever fixes the misleading name (or makes the
  class actually inherit from `BaseException`), the
  filter's behavior changes silently. A debug log of which
  classes were included/excluded at startup would surface
  upstream changes immediately.
- **Python version dependency.** The `TypeError` at
  `except`-time is enforced more strictly in 3.11+; older
  Python versions might have failed differently or
  silently. The fix is correct on all versions but worth
  noting in commit-message archaeology.

# Suggestions

1. Add a test:
   ```python
   def test_litellm_exception_list_only_contains_exceptions():
       for cls in LITELLM_EXCEPTIONS:
           assert issubclass(cls, BaseException), \
               f"{cls.__name__} is not a BaseException subclass"
   ```
   Eight lines, locks the bug class permanently.
2. Log at debug what the filter included and excluded at
   import time. The next time a "missing exception"
   complaint comes in (#4615-style), the log will
   immediately show whether the missing class was filtered
   out, named differently, or absent entirely.
3. File an issue (or PR) upstream in LiteLLM about the
   misleading name `ErrorEventError`. Costs nothing, helps
   every consumer of the library.
4. Explicitly enumerate the LiteLLM exceptions aider cares
   about by name in a comment or constant, with a link to
   each. The dynamic discovery approach is convenient but
   has demonstrated its failure mode; a smaller hand-curated
   list would be more robust at the cost of slight upstream
   tracking overhead.

# Verdict

**Ship it.** The fix is the correct primitive
(`issubclass(BaseException)`) replacing a name-based
heuristic that violated a language invariant. The PR
correctly walks back the over-broad earlier attempt. The
test gap is real — that's the most valuable follow-up — but
not a blocker for fixing a P0 startup crash.

# What I learned

Two lessons here, both general. First: when writing
dynamic class discovery against a third-party module, use
the *language-defined* predicate
(`issubclass(BaseException)` for exceptions, `callable()`
for callables, etc.) rather than name patterns. Names
across libraries are not a stable interface; type
relationships are. Second: when reverting a regression
caused by a previous fix, take the opportunity to narrow
the contract a third party might break in the future. The
original `656301c` failed because it trusted a naming
convention; the fix encodes the actual invariant. That's
the kind of change that makes the bug class extinct, not
just the bug instance.
