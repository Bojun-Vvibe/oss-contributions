# Context

`aider` ships a `--no-auto-commits` mode that tells it not to
create commits automatically — the user wants control over what
ends up in git history. The promise of that flag is "I run
aider, I review its changes, I decide what to commit". Anything
aider does to the index without the user's explicit action
breaks that promise.

This PR fixes a small but real violation: even with
`--no-auto-commits`, aider was running `git add` on files it
created or modified, leaving them *staged* but uncommitted.
Staged-but-uncommitted is a state the user didn't ask for.

# Problem

The bug shape:

1. User runs aider with `--no-auto-commits`.
2. aider edits or creates a file.
3. `allowed_to_edit()` notices the file isn't tracked and runs
   `self.repo.repo.git.add(full_path)`.
4. User runs `git status` after the session.
5. The file shows as **staged**, not as **modified** or
   **untracked**.

That ruins three workflows in particular:

- **Selective commits.** A user who wants `git add -p` to
  pick which hunks to commit can't, because `git add` already
  staged the whole file. They have to `git reset HEAD <file>`
  first.
- **Per-file commits.** A user who wants to commit changes to
  file A but not file B finds both staged. Same `git reset`
  workaround.
- **Reviewing diffs against HEAD.** `git diff` shows nothing
  meaningful for a fully-staged change; the user has to know
  to use `git diff --staged` or `git diff HEAD` to see what
  aider actually did.

The bug is small in mechanism but real in user experience.

# Design

Two-line fix in `aider/coders/base_coder.py`:

```python
# Before (line 2219):
if need_to_add:
    self.repo.repo.git.add(full_path)

# After:
if need_to_add and self.auto_commits:
    self.repo.repo.git.add(full_path)
```

The same pattern repeats 14 lines later (line 2233) for a
slightly different code path that also was running `git add`
unconditionally. Both call sites get the same `and self.auto_commits`
guard.

The fix is correct because the intent of `git add` in this code
is "make sure the file is tracked so the *upcoming* auto-commit
will include it". When `auto_commits` is false, there is no
upcoming auto-commit, so the staging serves no purpose and
violates the user's stated preference.

The change preserves the **untracked → tracked** transition
that `git add` was doing — but the file simply stays untracked,
which is what `git status` will show, which is what the user
expects. If the user wants to commit it, they `git add` it
themselves.

# Risks

- **A future code path that *requires* the file to be staged
  before some operation could break.** The PR is a `git add`
  removal in a specific condition; if any downstream code
  assumes "by the time `allowed_to_edit` returns, the file is
  staged", that assumption is now violated under
  `--no-auto-commits`. Worth a quick search for callers that
  depend on the staging side-effect; the small fix scope
  suggests the author already verified this, but no test was
  added.
- **No regression test.** A test that runs aider with
  `--no-auto-commits`, edits a file, and asserts
  `git status --porcelain` shows the file as `??` (untracked)
  rather than `A ` (added) would pin this behavior. Without
  it, the next refactor that "cleans up" the
  `and self.auto_commits` clause will silently regress.
- **Two near-identical conditional blocks 14 lines apart**
  is the kind of code structure that *should* be one helper.
  The PR fixes both copies but doesn't deduplicate. The next
  bug fix in this area will need to remember to update both
  again. A `_stage_for_auto_commit(full_path)` helper would
  make the next change touch one place.
- **Behavior change is silent on existing sessions.** Users
  who had grown to expect "files are staged after edit" — and
  built workflow around that, e.g. they always
  `git commit -m "aider: ..."` after a session, expecting
  staged files — will now find an empty staging area. That's
  arguably the *correct* behavior, but the change isn't called
  out in a CHANGELOG. A release note would set expectations.

# Suggestions

1. Add a regression test in `tests/basic/test_repo.py` (or
   wherever the auto-commit tests live) that asserts the
   `git status` post-condition under `--no-auto-commits`.
2. Refactor the two staging blocks into a single helper
   `_stage_if_auto_committing(path)` so the next change
   doesn't need to update two copies.
3. CHANGELOG entry: "fix: `--no-auto-commits` no longer stages
   files; they remain untracked until you `git add` them
   yourself".
4. Audit `aider/repo.py` for any *other* code paths that run
   `git add` without checking `auto_commits`. The same bug
   class likely exists in the file-deletion / file-rename
   paths.

# Verdict

**Ship it.** Two-character fix to a real promise-violation
bug that affects every `--no-auto-commits` user. The fix is
mechanically correct and the scope is appropriately tight.
File the test and the helper-extraction as immediate
follow-ups.

# What I learned

This is a textbook "flag promise vs. internal optimization"
bug. The user-facing contract of `--no-auto-commits` is "I
control what gets committed". Internally, `git add` is being
used as a setup step for the *upcoming* commit — a perfectly
reasonable internal idiom that becomes a bug the moment the
upcoming commit isn't going to happen. The fix is to thread
the flag through to the setup step. The general lesson is
that any setup work for an action also needs to be conditional
on the action.

The reusable insight is the **staged-but-uncommitted as a
git anti-state**. Most users have a mental model of
"untracked, modified, or committed". Files that are
*staged-but-uncommitted* are an unusual state to leave for
someone else (including yourself five minutes later). Tools
that operate on git should be very conservative about
landing files in that state; if you can't immediately commit,
don't stage. The same principle applies to `git stash` and
to `git rm --cached` — both leave the index in an unusual
state that surprises users on next `git status`.

The final lesson is on the value of "the user said `--no-X`"
as a hard signal. Whenever a user passes a `--no-something`
flag, every code path that does `something` (or sets up to do
`something`) needs to honor that flag. The temptation to do
"a little bit of `something`" — like staging files when
auto-commits are off — is exactly the bug class this PR
fixes. A pre-commit checklist for any new feature: "what
flags should disable this, and have I threaded them all the
way through?"
