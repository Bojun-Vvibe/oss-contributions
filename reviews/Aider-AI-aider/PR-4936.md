# Context

ACP (Agent Communication Protocol — distinct from Anthropic's
Anthropic ACP, this is the open-protocol variant used by
multiple agent IDE integrations) lets a host application drive
an agent process over JSON-RPC. The host sends `prompt` calls,
the agent streams session updates back. The OpenHands ACP
support I reviewed earlier (PR #13994 / #13983) was on the
*host* side; this PR is the same protocol on the *agent* side
— making aider into something a host like Zed or another
ACP-compatible IDE can drive.

The adapter wraps aider's existing `Coder` class without
touching the core: it swaps aider's `InputOutput` for an
ACP-aware subclass that streams to a JSON-RPC client, and runs
`coder.run(with_message=...)` in a thread-pool worker. The
session lifecycle (new/load/resume/fork/list/cancel) is handled
by a new `SessionManager`. New CLI flag `--acp` flips main.py
into server mode.

+990/-0 across 8 files. Important provenance note: the module
docstring says `server.py` is adapted from
`~/.hermes/hermes-agent/acp_adapter/server.py` (Nous Research,
MIT). The attribution is preserved.

**The agent class** wraps the standard ACP `Agent` interface:

```python
class AiderACPAgent(acp.Agent):
    def __init__(self, session_manager: SessionManager | None = None):
        super().__init__()
        self.session_manager = session_manager or SessionManager()
        self._conn: Optional[acp.Client] = None

    def on_connect(self, conn: acp.Client) -> None:
        self._conn = conn

    async def initialize(...):
        return InitializeResponse(
            protocol_version=acp.PROTOCOL_VERSION,
            agent_info=Implementation(name="aider-acp", version=AIDER_VERSION),
            agent_capabilities=AgentCapabilities(
                session_capabilities=SessionCapabilities(
                    fork=SessionForkCapabilities(),
                    list=SessionListCapabilities(),
                ),
            ),
            auth_methods=None,
        )
```

`auth_methods=None` is "use whatever the connection layer
provides" — a fine default for stdio transports but worth a
comment for HTTP transports where it would mean "no auth
required".

**The prompt path** is the meaty one:

```python
async def prompt(self, prompt, session_id, **kwargs) -> PromptResponse:
    state = self.session_manager.get_session(session_id)
    if state is None:
        return PromptResponse(stop_reason="refusal")
    user_text = _extract_text(prompt).strip()
    if not user_text:
        return PromptResponse(stop_reason="end_turn")

    conn = self._conn
    loop = asyncio.get_running_loop()
    if state.cancel_event:
        state.cancel_event.clear()

    coder = state.coder
    original_io = coder.io
    coder.io = ACPInputOutput.from_existing(
        existing_io=original_io,
        conn=conn,
        session_id=session_id,
        loop=loop,
    )

    def _run_coder() -> None:
        try:
            coder.run(with_message=user_text)
        except Exception:
            logger.exception("Coder Error in session %s", session_id)

    try:
        await loop.run_in_executor(_executor, _run_coder)
    finally:
        coder.io = original_io

    stop_reason = "cancelled" if state.cancel_event and state.cancel_event.is_set() else "end_turn"
    return PromptResponse(stop_reason=stop_reason)
```

The IO swap pattern — temporarily replace `coder.io`, call into
the synchronous `coder.run()`, restore the original IO in
`finally` — is a clean way to plumb a new transport into a CLI-
shaped library without modifying the library. The `try/finally`
restoration is critical: if `coder.run()` raises, the original
`io` is restored before the next prompt, otherwise subsequent
local CLI invocations would erroneously try to talk to the dead
ACP connection.

**The IO subclass** streams output:

```python
class ACPMarkdownStream:
    def __init__(self, conn, session_id, loop):
        self.conn = conn; self.session_id = session_id; self.loop = loop
        self.previous_text = ""

    def update(self, text: str, final: bool = False) -> None:
        if not text: return
        added = text[len(self.previous_text):]
        if added:
            update = acp.update_agent_message_text(added)
            _send_update(self.conn, self.session_id, self.loop, update)
        self.previous_text = text
```

The "diff against `previous_text`" pattern is correct for
aider's `MarkdownStream.update()` contract — aider passes the
full accumulated text on each tick, and the streamer emits only
the new tail. The `text[len(previous):]` slice is O(N) per
update but N is small (a few tokens of streaming output), so
fine.

**The cross-thread send** uses `asyncio.run_coroutine_threadsafe`:

```python
def _send_update(conn, session_id, loop, update):
    if conn is None: return
    try:
        future = asyncio.run_coroutine_threadsafe(
            conn.session_update(session_id, update), loop
        )
        future.result(timeout=5)
    except Exception:
        logger.debug("Failed to send ACP update", exc_info=True)
```

Worker thread → asyncio event loop bridge done correctly. The
5-second `future.result(timeout=...)` prevents a wedged
connection from blocking the worker indefinitely. The `debug`
log on failure (rather than `error`) is appropriate for
streaming updates where dropping one is non-fatal.

# Risks

1. **Concurrent `prompt` calls share `_executor` (4 workers)
   but each prompt mutates `coder.io`.** If a single `Coder`
   instance is shared across sessions (the SessionManager
   probably gives each session its own Coder, but not visible
   from this excerpt), two concurrent prompts on different
   sessions could swap each other's `io` mid-flight. Worth a
   per-session lock, or a "Coder per session" invariant
   documented loudly.
2. **`auth_methods=None` is "no auth"** which is fine for
   stdio but a foot-gun if anyone ever wires this to a TCP
   transport. A defensive check ("if transport is network,
   require an auth method") would close the gap.
3. **`_extract_text` only handles `TextContentBlock` and
   things with `.text`.** Image, audio, resource, and
   embedded-resource content blocks are silently dropped.
   That's fine for an MVP (aider doesn't support them anyway)
   but the response should arguably tell the client "I
   ignored your image". Currently the client thinks the agent
   processed the full prompt.
4. **`ACPMarkdownStream.update`'s diff logic assumes the new
   `text` is a *prefix-extension* of the previous.** If
   aider's formatter ever rewrites earlier output (e.g. for
   a syntax-highlight pass that touches earlier lines), the
   slice would emit garbage. Worth a defensive check — if
   `not text.startswith(self.previous_text)`, send a "reset
   and re-emit" update.
5. **`_send_update`'s 5-second timeout per chunk** can
   accumulate. If the network is slow and aider streams 100
   chunks per turn, the worst case is 500 seconds of blocked
   worker time per turn. A per-session backoff (skip
   subsequent updates if the last one timed out) would cap
   the damage.
6. **Cancellation is cooperative.** `state.cancel_event` is
   set on `cancel(session_id)`, but the comment notes "Aider's
   Coder doesn't have an interrupt method easily callable
   from thread, but setting and reading from cancel_event
   inside IO works." That's the honest description but means
   cancellation only fires when aider next checks the IO
   (could be many seconds during an LLM call). Worth either
   threading more interrupt points through aider's core or
   documenting the cancellation latency clearly.
7. **`finally: coder.io = original_io` happens even if
   `loop.run_in_executor` itself raises before `_run_coder`
   starts.** That's correct — the swap happened before the
   `await`. But worth a test that cancellation during the
   "swapping io" window leaves the original io intact.
8. **The `_executor` is module-global with 4 workers.** That
   means a single misbehaving session can starve the others.
   In a multi-session deployment, per-session executors or a
   dynamic worker pool would scale better.
9. **No backpressure.** If the client is slow to consume
   session updates, `_send_update` can stack up. The
   `future.result(timeout=5)` per call provides per-call
   bounds, but no overall flow control.
10. **`session_id` types.** ACP defines session IDs as
    opaque strings; the aider code treats them as strings
    consistently, but `update_cwd` returning `None` for
    "session not found" is overloaded with "session not
    updatable" — slight API confusion that a typed sentinel
    would clarify.
11. **`from_existing` constructor pattern** — copying values
    from an `original_io` into a new `ACPInputOutput`. The
    excerpt cuts off before showing what fields are copied;
    if any field is missed, the ACP IO will silently
    misbehave (e.g. miss a `pretty` flag, an output format,
    a confirmation policy). A field-by-field comparison test
    against `InputOutput`'s `__init__` would catch drift.
12. **Image, audio, resource blocks rejected silently.**
    The `_extract_text` function should at minimum log when
    it encounters non-text content, both for diagnostics and
    so the user knows their image attachment was dropped.

# Suggested questions for the author

1. Does `SessionManager.create_session` produce a fresh
   `Coder` per session, or is the Coder shared? The IO swap
   pattern needs the former; a comment on the invariant
   would help.
2. The 5s `future.result(timeout=...)` per stream chunk can
   stack up under slow-network conditions. Should there be a
   per-session circuit breaker that switches to "drop
   updates" mode after N consecutive timeouts?
3. `_extract_text` silently drops image/audio/resource
   blocks. Should the agent emit a session update saying
   "I received non-text content but cannot process it"?
4. The cancellation latency depends on when aider next
   checks `cancel_event` inside the IO. Is there a target
   for "cancel within X seconds" to drive a roadmap?
5. The `auth_methods=None` is appropriate for stdio. Is
   there a follow-up PR planned for HTTP transport with
   real auth?
6. The "Hermes" provenance note is great. Is there a
   policy for periodically diffing against upstream to pull
   in fixes?

# Verdict

**Approve, with the per-session-Coder invariant documented
and the auth story tracked.** The adapter is well-shaped: it
wraps aider's existing core via the IO-swap pattern without
modifying the core, the cross-thread asyncio bridge is done
correctly with `run_coroutine_threadsafe` and per-call
timeouts, and the session lifecycle methods (new/load/resume/
fork/list/cancel) cover the standard ACP surface. The
provenance disclosure is a model of good open-source
attribution.

The biggest risk is the per-session-Coder invariant: if two
ACP sessions ever share a `Coder` instance, the IO-swap
pattern races. Either lock per-session, document the
invariant, or refactor to "swap into a per-call IO context
manager" rather than mutating shared state.

# What I learned

Three takeaways, all about wrapping a synchronous CLI library
behind an async protocol.

**First**, the *swap-restore IO* pattern is the cleanest way
to get streaming output from a library that wasn't designed
for it. Aider's `Coder` writes to `self.io`; you don't need
to thread a streaming-aware output channel through every call
site. Just swap `coder.io` for an instance that captures
everything, run the synchronous code, restore. The pattern
generalizes to any library that has a single output sink:
loggers, console renderers, telemetry emitters. The cost is
discipline — `try/finally` is mandatory or you'll leak the
swap into the next call. The benefit is *zero modification to
the wrapped library*, which means upstream changes don't
break your wrapper.

A subtle point: this only works if the IO sink is a *member
variable*, not a *global*. Aider has `self.io`, which is
swappable. A library that writes to `print()` or a global
logger requires monkeypatching with all its cost. When
designing a library you want others to wrap, *put the output
on `self`*.

**Second**, the *cross-thread asyncio* idiom. A worker thread
that needs to call into an asyncio event loop must use
`asyncio.run_coroutine_threadsafe(coro, loop)` — never
`loop.create_task(coro)` (wrong thread!) and never
`asyncio.run(coro)` (creates a new loop, deadlocks). The
returned `Future` can be awaited from the calling thread with
`.result(timeout=...)`, which lets you bound the cross-thread
call. The PR uses exactly this shape, with a 5-second
timeout per call. This is the canonical pattern for
"synchronous library + async transport" and worth memorizing.

**Third**, the *provenance disclosure* discipline in the
module docstring. Adapted code is a fact of open-source life:
useful patterns get copied, modified, re-released. The right
shape is what this PR does — name the source path, name the
copyright holder, name the license, list the changes, leave
a hint for re-merging upstream fixes. Compare to the wrong
shape: silent copy, no attribution, no merge note. The wrong
shape creates legal ambiguity, makes future maintainers
reinvent context, and hides the existence of upstream
improvements. The right shape costs five lines of docstring
and pays back every time the upstream code evolves. Every
adapted file should have this header.

The fourth meta-lesson: when you wrap a synchronous library
behind an async server, the *concurrency invariants* of the
wrapped library bleed into your design. If the library has
per-instance state (like aider's `Coder`), your server needs
"one instance per session". If the library has global state
(many CLI tools have this), your server can only serve one
session at a time. The IO-swap pattern in this PR works *only*
if Coder-per-session is the invariant; the PR doesn't make that
invariant explicit, which is the one place I'd want the author
to slow down. State invariants of wrapped synchronous libraries
are the foot-gun every async wrapper steps on eventually.
