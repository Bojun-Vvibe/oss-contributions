# Context

Aider builds a "repo map" by walking files in the user's
repository, normalizing each path through `safe_abs_path()`,
which calls `pathlib.Path.resolve()`. Resolution follows
symlinks. Real-world repositories — particularly those that
vendor projects like V8 (which contains a deliberate
`v8/v8 -> v8/v8` self-symlink) — contain circular symlinks.
Python's `Path.resolve()` raises `RuntimeError("Symlink loop
from ...")` when it detects one.

# Problem

Issue #4711: aider crashed on repo-map generation against any
repo containing a circular symlink. From the user's
perspective: `aider` exits with an unhelpful Python
traceback the moment it tries to index a perfectly valid
checkout. The blast radius is "any user who clones a project
that vendors V8, Chromium, or any of the dozens of other
upstream projects with self-referential symlinks."

The bug class is "trusting a stdlib path operation to never
raise on real filesystem inputs." `Path.resolve()` documents
the `RuntimeError`, but `safe_abs_path` — whose name promises
"safe" — was anything but, and silently propagated the raise
up through repo-map generation to the top-level CLI.

# Design

```python
def safe_abs_path(path):
    try:
        return str(Path(path).resolve())
    except (RuntimeError, OSError):
        return str(Path(path).absolute())
```

Three lines, one comment-worthy decision: the fallback is
`Path.absolute()`, *not* `Path.resolve()` with a flag. The
distinction matters. `resolve()` resolves symlinks and may
raise. `absolute()` makes the path absolute (prepends `cwd`
if relative) and **does not follow symlinks**, so it cannot
encounter a loop.

The user-facing consequence is subtle: for paths inside a
symlink loop, the returned absolute path will be the
*symlinked* path, not the *target* path. For repo-map
indexing this is correct — aider should refer to files by
the path the user sees, not the canonical resolved path,
which in the loop case isn't well-defined anyway.

`OSError` is also caught, which covers the broader family of
"path resolution failed" errors (broken symlinks, permission
denied on a parent directory, etc.). Useful generality at
zero added cost.

The test (`test_safe_abs_path_symlink_loop`) constructs the
loop in a temp dir, calls `safe_abs_path`, and asserts no
raise plus a sensible return. 15 lines. Exactly the right
test for this fix — it codifies the contract ("doesn't
crash on circular symlinks") that the function name was
already promising.

# Risks

- **Silent semantic change for the loop case.** Before:
  crash. After: returns a path that doesn't represent a
  real on-disk file (since the symlink loop has no
  terminal target). Anything downstream that does
  `os.stat` or `open` on the returned path will get
  `OSError` or `FileNotFoundError`. For the repo-map use
  case this is fine — aider will simply skip the file —
  but a future caller assuming "if `safe_abs_path` returned
  a path, it points somewhere real" could be surprised. A
  doc note on the function would harden the contract.
- **`OSError` is a broad catch.** It now masks permission
  errors and other genuine filesystem problems by silently
  falling back. For the repo-map walk this is the right
  trade (don't crash on one unreadable file), but a debug
  log on the fallback path would help diagnosis.
- **No upper bound on `Path.absolute()` work.** Not a real
  risk — it's a string operation — but worth knowing it
  doesn't touch the filesystem at all, which is part of why
  it can't loop.

# Suggestions

1. Add a one-line docstring documenting the fallback
   semantics: "Returns an absolute path. Does not guarantee
   the path resolves to an on-disk target if the input
   participates in a symlink loop."
2. Log at debug when the fallback fires:
   `logger.debug("safe_abs_path: fallback to absolute() for
   %s due to %s", path, exc)`. Cheap and the next bug
   report against this code will be much easier to triage.
3. Consider adding the same try/except pattern to any other
   `.resolve()` call sites in the codebase. A repo-wide
   grep is a 5-minute job and prevents the next instance.

# Verdict

**Ship it.** Tiny, correct, well-tested, and the choice of
`Path.absolute()` as fallback is exactly right. This is the
shape of fix that makes a tool feel reliable to users — no
new feature, no UI change, just "stops crashing on a real
repo configuration."

# What I learned

A function named `safe_*` should enumerate the unsafe
operations it wraps and explicitly handle each. Here the
unsafe operation was `Path.resolve()`'s `RuntimeError` on
symlink loops — documented in the stdlib but easy to miss
because it's a class of error you only see on certain
filesystems. The general rule: when you wrap a stdlib call
in a "safe" facade, also catch the parent classes
(`OSError`) of the documented exceptions, because the
stdlib has a long history of widening what its file
operations can raise across versions. The fallback should
preserve as much of the function's contract as possible
(here: still returns an absolute path) so callers can keep
flowing.
