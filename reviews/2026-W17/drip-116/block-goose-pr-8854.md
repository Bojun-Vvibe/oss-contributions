# block/goose#8854 — fix: remove defaults yaml

- PR: https://github.com/block/goose/pull/8854
- Head SHA: `7c60c1b3`
- Diff: +28 / -140 across `crates/goose-server/src/commands/agent.rs` (+2), `crates/goose/src/config/base.rs` (-117), `crates/goose/src/security/mod.rs` (+24)

## What it does
Removes the `defaults.yaml` mechanism from `crates/goose/src/config/base.rs` — the `defaults_path: Option<PathBuf>` field on `Config`, the `find_workspace_or_exe_root() → defaults.yaml` discovery in `Config::default()`, the `Config::new_with_defaults()` constructor, the `load_defaults()` / `merge_missing_defaults()` helpers, and the four `defaults_path: None` initializations in the existing test constructors. Three associated tests are also removed (`test_defaults_fallback_when_key_not_in_config`, `test_full_precedence_env_over_config_over_defaults`, `test_defaults_not_persisted_on_write`). One small test (`test_no_defaults_file_behaves_as_before`) is renamed to `test_missing_key_returns_not_found`.

In place of the generic mechanism, the security module gains `set_security_defaults()` (`security/mod.rs:21-32`) plus a `set_default_if_not_exist(config, key, default_env)` helper at `:16-23` that seeds two specific keys from env vars: `SECURITY_PROMPT_ENABLED` ← `DEFAULT_SECURITY_PROMPT_ENABLED`, `SECURITY_COMMAND_CLASSIFIER_ENABLED` ← `DEFAULT_SECURITY_COMMAND_CLASSIFIER_ENABLED`. Called from two places: `SecurityManager::new()` (covers CLI and any agent-construction path) and `goose-server::commands::agent::run` (`agent.rs:40`, eager seed at goosed boot before Axum accepts requests, so the desktop Settings UI sees seeded values on its first `/config` call).

## Strengths
- **The architectural critique is correct.** The `defaults.yaml` mechanism in `config/base.rs` was load-bearing for *exactly one* in-tree consumer (security flags) but added: a `defaults_path` field on every `Config`, a workspace-or-exe-root discovery dance at construction time (`find_workspace_or_exe_root`), an extra constructor variant, and 60+ lines of merge logic. For one feature with one consumer, the cost/benefit is clearly upside-down. Replacing it with two specific env vars consumed directly by the security module is a strict reduction in cognitive surface.
- The "open-source build doesn't bundle a `defaults.yaml`" framing in the PR description is honest about what's being removed: this was infrastructure for an internal/fleet build pattern that the OSS repo never used. Cleaning up unused infrastructure is exactly the right move.
- The `set_default_if_not_exist` helper at `security/mod.rs:16-23` correctly preserves the user-wins precedence: if the key is *already* set in `config.yaml` (`config.get_param::<bool>(key).is_ok()`), short-circuit; otherwise read from env and write through. Once seeded, the value lands in `config.yaml`, so the user toggle wins on subsequent runs and the UI/backend stay in sync.
- The two seeding call sites (`SecurityManager::new()` + `goose-server::commands::agent::run`) cover both the CLI path and the desktop-server boot path. The desktop UI's first `/config` call sees the seeded values — that's the right ordering for first-launch UX.

## Concerns
- **Test deletion is a strict loss of coverage at the config layer.** The three deleted tests (`test_defaults_fallback_when_key_not_in_config`, `test_full_precedence_env_over_config_over_defaults`, `test_defaults_not_persisted_on_write`) pinned three properties of the old mechanism. Two of those properties — *env > config > <fallback>* precedence and *fallback values are not persisted on write* — are *still* properties of the new design (the `set_default_if_not_exist` helper only writes if absent, and env is read once at startup, then config takes over). Equivalent tests should exist for the new path: a `test_security_defaults_seed_from_env_when_absent`, `test_security_defaults_do_not_overwrite_user_value`, and `test_security_defaults_persist_to_config_yaml_after_seed`. Without those, the next refactor of `set_default_if_not_exist` could silently regress the user-wins invariant.
- The PR description says "Manual testing locally with DEFAULT env passed" — manual is acceptable for a config-layer change, but the deleted tests *were* the automation. Replacing automation with manual verification is a regression in test posture.
- **`set_security_defaults()` is called from `SecurityManager::new()`.** That means it runs every time a new `SecurityManager` is constructed (potentially many times across the process lifetime). The early-return in `set_default_if_not_exist` (`if config.get_param::<bool>(key).is_ok() { return; }`) makes subsequent calls cheap, but each call still does a config read + lock acquisition. For a one-shot startup seeding helper, calling it once at process boot from `goose-server::run` is sufficient; calling it from `SecurityManager::new()` (which the comment at `:53` justifies as "covers CLI and any other agent-construction path") doubles up on goosed but matters for CLI. Probably fine, but worth a comment on idempotency.
- The `goose::security::set_security_defaults()` call at `agent.rs:40` runs *before* `crate::logging::setup_logging(Some("goosed"))?` finishes (it's after the call, but logging could itself emit during seeding). Any errors from the env-var parse path (`config.get_param::<bool>(default_env)` returns `Err`) are silently swallowed by the `if let Ok(parsed) = ...` pattern. If `DEFAULT_SECURITY_PROMPT_ENABLED=notabool` were set, the user gets no signal that their env var was malformed. A `tracing::warn!` on the parse-failure branch would be cheap.
- The renamed test `test_missing_key_returns_not_found` is fine, but the original `test_no_defaults_file_behaves_as_before` name actually communicated more semantically — it was the "negative space" test confirming the optional mechanism didn't degrade the default path. The new name is just a generic config-not-found test. Minor naming nit.
- `set_default_if_not_exist` accepts `&Config` and calls `config.set_param(key, parsed)` which mutates the config file. The function name (`set_default_if_not_exist`) reads as a query/check but actually has a write side-effect. A rename to `seed_default_if_not_exist` or `seed_from_env_if_absent` would advertise the side-effect more honestly.

## Verdict
**merge-after-nits** — the architectural cleanup is correct and the test deletions are at least *acknowledged* by the PR description. But replacing three deleted automated tests with "manual testing locally" is a coverage regression that should be reversed before merge: add `set_security_defaults` unit tests covering the user-wins invariant, the seed-from-env-on-absent invariant, and the config-yaml-persists-after-seed invariant. The other items (rename, idempotency comment, parse-error log) are follow-ups.

## What I learned
Generic config infrastructure with one in-tree consumer is almost always over-abstracted. The ratio "lines of mechanism : lines of consumer" was 117 : ~24 here — a 5:1 ratio of plumbing to payload. The right move when you spot that ratio is to inline the mechanism into the consumer, even if it loses generality, because the generality was theoretical anyway. Just be sure to *port the tests*, not just delete them.
