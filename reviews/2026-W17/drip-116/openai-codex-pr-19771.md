# openai/codex#19771 — fix: filter dynamic deferred tools from model_visible_specs

- PR: https://github.com/openai/codex/pull/19771
- Head SHA: `f1826252`
- Diff: +293 / -69 across `codex-rs/core/src/session/turn.rs` (-49), `codex-rs/core/src/tools/router.rs` (+34/-19), `codex-rs/core/src/tools/router_tests.rs` (+88), `codex-rs/core/tests/suite/compact_remote.rs` (+170+/-1)

## What it does
Closes #19486. Moves the "filter deferred dynamic tools out of `model_visible_specs`" filtering from `build_prompt` (called once per normal turn) up into `ToolRouter::from_config` (called once at router construction). The motivating bug: `/compact` request building also calls `router.model_visible_specs()` directly — *bypassing* `build_prompt` — so deferred dynamic tools without a `tool_search` partner leaked into the compaction request payload, producing 400s from the upstream API. Filtering at router construction fixes the issue generically for *every* caller of `model_visible_specs()`.

## Strengths
- **Right layer for the fix.** The previous design encoded "deferred tools should be invisible to the model" as a downstream filter at the `build_prompt` boundary, which is fragile: any new caller of `model_visible_specs()` (compaction, tool-discovery introspection, debug dumps) had to remember to re-apply the filter. Pushing it into `ToolRouter::from_config` (`router.rs:74-93`) means `model_visible_specs()` is always *correct by construction* — there's no second filter to forget.
- The `filter_deferred_dynamic_tool_spec` helper is moved from `turn.rs` (where it was only reachable via `build_prompt`) to `router.rs` (where it's a private detail of router construction). The implementation is unchanged — same `ToolName::plain` vs `ToolName::namespaced` matching, same `namespace.tools.is_empty()` collapse — so the migration is behavior-preserving for the existing `build_prompt` path while extending coverage to compaction.
- The early-return optimization at `router.rs:309-311` (`if deferred_dynamic_tools.is_empty() { return Some(spec); }`) preserves the zero-overhead fast path for the common case (no deferred tools), so the router-construction cost doesn't regress for the 99% of configurations that don't use `defer_loading`.
- **Two-layer test coverage.** The unit test at `router_tests.rs:142-193` constructs both a `defer_loading: true` and a `defer_loading: false` dynamic tool, then asserts (a) `find_spec` *can* still find the hidden tool by name (so `tool_search` can promote it later — important: deferred ≠ deleted), and (b) `model_visible_specs()` only returns the non-deferred one. The integration test at `compact_remote.rs:60-101` adds `assert_tools_payload_does_not_defer` to verify the actual `/compact` request body never contains a `defer_loading` declaration. That two-layer pinning closes the loop on both the API contract (`find_spec` still works) and the observable effect (compaction payload is clean).
- The PR description includes a manual end-to-end validation against the live `/compact` endpoint: raw deferred dynamic tools without `tool_search` returned 400 (current bug), filtered payload returns 200. That's the right sanity check.

## Concerns
- The filter logic is now *inside* the `let model_visible_specs = specs.iter().filter_map(...)` chain at `router.rs:80-92`, which combines two unrelated concerns: `code_mode_only_enabled` filtering and `defer_loading` filtering. Both are early-return predicates, so the `if` chain is fine, but a reader has to mentally parse two filters in the same closure. Extracting the body into a `fn should_be_visible(configured_tool: &ConfiguredTool, ...) -> Option<ToolSpec>` would make each predicate independently testable.
- The deletion of `build_prompt`'s local filter at `turn.rs:947-961` is correct (filter moved upstream) — but the `tools: router.model_visible_specs()` call at `turn.rs:949` is now a *behavior change* on every code path that goes through `build_prompt`, not just the compaction path. The unit test at `compact_remote.rs` covers compaction; is there an existing test that pins normal-turn behavior to assert deferred tools are *also* still filtered there? If `build_prompt` only had the `compact_remote.rs` test guarding it, it does — but worth confirming the normal-turn path still has coverage.
- `dynamic_tools` is captured by reference into `deferred_dynamic_tools` at `router.rs:74-78`, then `filter_deferred_dynamic_tool_spec` is called once per spec. For configurations with many specs and many deferred tools, the inner `HashSet::contains(&ToolName::plain(...))` allocates a `ToolName::plain(tool.name.as_str())` per check. Probably not hot enough to matter (router construction is one-shot at session start), but flagging.
- No test for the empty-namespace collapse path (`namespace.tools.is_empty() → None`) where *all* tools in a namespace are deferred. The existing test only deferes one of two namespaced tools, so the namespace survives. Worth a third case.

## Verdict
**merge-as-is** — correct architectural move (filter at the source, not at every caller), with both unit-level (`find_spec` still works) and integration-level (`/compact` payload is clean) tests pinning the contract. Concerns above are follow-ups.

## What I learned
When a "filter on output" has multiple callers, the filter belongs at the data-source boundary, not at the call site. Otherwise every new caller is a chance to forget. This PR is a textbook example of the refactor: identify the bug (one caller forgot), then ask "where should this be impossible to forget?" — the answer is almost always one layer up.
