## Specifics

- New `pty.ts` declares an `HttpApi.make("pty")` group covering all five JSON routes (list/create/get/update/remove) plus a separate raw-router `ptyConnectRoute` for the websocket upgrade at `pty.ts:184-234`. The split is necessary because `HttpApi` doesn't model websocket upgrades — the connect route lives outside the typed-API surface and is mounted directly on `HttpRouter`. Wiring is symmetric in `server.ts:70-79` (`ptyConnectRoute` mergedAll alongside `HttpApiBuilder.layer(PtyApi)`) and `index.ts:97-108` (Hono fallback that proxies to the experimental Effect handler when `Flag.OPENCODE_EXPERIMENTAL_HTTPAPI` is on).
- `create` handler at `pty.ts:137-148` correctly defends against shared-mutable inputs by spreading `args`/`env` into fresh arrays/objects before passing them to `pty.create()` — same pattern is repeated for `update.size` at `pty.ts:160-163`. Without this, the underlying `Pty.Service` would hold a reference to the request payload and could observe mid-flight mutation if Effect retries the handler. Good defensive copy.
- `get`/`update` map a `null`/`undefined` from `pty.get`/`pty.update` to `HttpApiError.NotFound` via `yield* new HttpApiError.NotFound({})` at `pty.ts:152` and `pty.ts:164` — both endpoints declare `error: HttpApiError.NotFound` in the schema (`pty.ts:82,94`), so the wire-error class lines up with the schema.
- `remove` at `pty.ts:168-171` returns `Schema.Boolean: true` regardless of whether the ID existed. That's deliberately idempotent (DELETE-as-idempotent matches REST norms), but the OpenAPI summary "Remove and terminate a specific PTY session" reads as if 404 should fire when the ID is missing. Pick one and document it.
- `ptyConnectRoute` uses the websocket upgrade pattern that the file shares with the existing TUI/event bridges: it constructs an adapter object exposing `readyState`/`send`/`close` over `socket.writer`, hands it to `pty.connect(ptyID, adapter, cursor)`, and pumps inbound messages through `socket.runRaw()`. The `closed` flag at `pty.ts:198-213` is a single boolean owned by the closure — both `send` and `close` short-circuit on `closed`, and `Effect.ensuring` at `pty.ts:223-228` flips `closed = true` and calls `handler.onClose()` exactly once on stream end. `Effect.catchReason("SocketError", "SocketCloseError", ...)` swallows the two expected websocket terminations cleanly; everything else `Effect.orDie`s.
- `parsedCursor` validation at `pty.ts:193-195` is conservative: requires `Number.isSafeInteger(parsedCursor) && parsedCursor >= -1`, falls back to `undefined` otherwise. The `-1` lower bound is presumably "all history" sentinel — that contract is undocumented in this PR (it lives in `Pty.Service`); a one-line comment would help.
- Tests at `httpapi-pty.test.ts:329-373` cover the happy path end-to-end (list → create → get → update → remove → 404 after delete) plus a websocket-against-missing-PTY-returns-404-before-upgrade case. Test is gated off Windows via `const testPty = process.platform === "win32" ? test.skip : test` (PTY's `node-pty` dependency is platform-sensitive). The test reuses a `command: "/usr/bin/env"` + `args: ["sh", "-c", "sleep 5"]` payload that's process-bound rather than depending on shell PATH — good portability hygiene.
- The spec-doc checklist flip at `specs/effect/http-api.md:320-345` correctly marks all six PTY routes done in one batch.

## Concerns

- `server.ts:70-79` provides `Pty.defaultLayer` *only* to `PtyApi`'s layer, but `ptyConnectRoute` at `pty.ts:233` also `.pipe(Effect.provide(Pty.defaultLayer))`. Two separate provisions of the same default-layer factory mean two `Pty.Service` instances if `defaultLayer` is not memoized (it usually returns a fresh `Layer` per call). PTY state would split across the JSON routes and the websocket route — the test suite never exercises the JSON→websocket transition (create via POST, then connect via the websocket), so this would be invisible to CI. Worth checking whether `defaultLayer` is a `Layer.scoped`-cached singleton or a factory; if the latter, hoist it.
- `update`'s `if (!info)` 404 at `pty.ts:164` swallows the `Pty.Service` distinction between "ID didn't exist" and "ID existed but update failed" (e.g. PTY already exited). Both surface as 404. That's probably wrong — a closed PTY should be 409/410 — but the PR also doesn't ship that distinction, so this is preserved-behavior, not a regression.
- The `connect` route returns `HttpServerResponse.empty()` after the socket loop ends at `pty.ts:232` — after a successful upgrade Hono/Bun should not see this body, but it's worth confirming the runtime doesn't try to write a second response on top of the upgraded socket.
- Test at `httpapi-pty.test.ts:367-373` exercises the missing-PTY-websocket path but only against the pre-upgrade 404 branch. There's no test that covers a successful upgrade-then-disconnect, so the closure-state flag, `Effect.ensuring`, and `handler.onClose()` are all uncovered.

## Verdict

`merge-after-nits` — the code is structurally clean, the defensive payload copies are right, and the tests cover the JSON CRUD path well. Two real questions before merge: (1) confirm `Pty.defaultLayer` is a singleton or move it up so the JSON and websocket routes share state, and (2) decide whether `remove` should 404 on missing IDs vs the current always-true. Adding even a smoke test for a successful websocket upgrade would also lock down the closure-state/`Effect.ensuring` path.
