## Specifics

- Adds `golang.org/x/sync/singleflight` import and a `requestGroup singleflight.Group` field on `Manager` at `manager.go:9,18`. The whole `New(...)` + `Initialize(...)` + `WaitForServerReady(...)` startup sequence is now wrapped in `s.requestGroup.DoChan(name, func() (any, error) { ... })` at `manager.go:181-220`, so N concurrent callers asking for the same LSP server name share a single underlying startup attempt and one of them returns to all N waiters.
- The previous code had three open-coded "is this client already active?" snapshots scattered across `startServer` (initial check, post-`isUserConfigured` check, post-`New` race check). The PR consolidates them via two new helpers at `manager.go:222-245`: `isActiveClient(client) bool` (returns true for `StateReady|StateStarting|StateDisabled`) and `Manager.getActiveClient(name) (*Client, bool)` that combines the map lookup with the state predicate. All three former call sites now read `if client, ok := s.getActiveClient(name); ok { ... }`. Real DRY win.
- Critical context-handling change at `manager.go:191`: `baseCtx := context.WithoutCancel(ctx)`. This decouples the shared startup goroutine from the context of whichever caller happened to win the singleflight. Without this, if caller A's `ctx` was canceled mid-startup, caller B (still waiting on `DoChan`) would see the entire LSP server initialization aborted even though B's context was still live. With `WithoutCancel`, the LSP `New(...)` and `Initialize(...)` calls run against a context that doesn't carry A's cancellation — exactly the right semantics for a shared resource.
- Caller's own `ctx` is still respected at `manager.go:213-215` via the `select { case <-ctx.Done(): ... case res := <-ch: ... }`. So caller A canceling now means "I don't care about the result anymore"; the underlying startup keeps running for caller B. This is the canonical singleflight-with-cancellable-callers pattern.
- `res.Shared` is consumed at `manager.go:218-222` to demote the log level on shared failures: the first caller logs `Error`, subsequent callers logging the same shared error log at `Debug`. Avoids N copies of the same noisy stack in the journal when many editors all simultaneously open `.go` files and the gopls binary is missing. Nice touch.
- State transitions cleaned up: `client.serverState.Store(StateStarting)` at the old `:225` (direct atomic) becomes `client.SetServerState(StateStarting)` at `manager.go:200` — uses the public setter, which is consistent with the rest of the file. On init failure at `manager.go:208`, the client is now `SetServerState(StateError)` *before* `Close()` and `s.clients.Del(name)` — the old code went straight to `Close()`+`Del`, leaving anything reading the in-memory client through a different path observing whatever state it was in pre-init (probably `StateUnknown`). The new ordering is also correct because the `singleflight` cache returns `(nil, err)` and the `getActiveClient` predicate explicitly excludes `StateError`.
- The post-init "actually warm" check has been re-shaped: previously the code had a `WaitForServerReady` call followed by a state flip; now both are inside the singleflight closure at `manager.go:212-217`, with the same `WaitForServerReady` failure path setting `StateError` (continuing without erroring out to caller — same fail-soft behavior). Preserved.

## Concerns

- The double `s.clients.Set(name, client)` race window has narrowed but isn't fully gone. At `manager.go:201-204`, after the inner `getActiveClient` re-check, the code calls `s.clients.Set(name, client)` *before* `client.Initialize(...)` is called. If a caller from a different name (impossible with singleflight per-key) or a manual external caller writes to `s.clients` in between, you could observe the half-initialized client through `Manager.GetClient`-like accessors. With singleflight per `name`, this is essentially impossible in practice but worth a comment.
- The `WaitForServerReady` failure path at `manager.go:217-219` flips state to `StateError` but then the closure still `return client, nil` (no error). So to *callers*, this looks like success — they get a `*Client` but its internal state is `StateError`. Is that intentional? Pre-PR behavior was identical (the old code logged a `Warn` and continued), so this is preserved, but it's worth documenting in a comment that "ready check failed but we still hand out the client; the next call to `getActiveClient` will reject it because `StateError` isn't in the active set, so the next request will retry". That's actually a clean state machine — just non-obvious.
- No new tests. `singleflight.Group` semantics are subtle (key collision, panic propagation, `Forget`-on-error policy not used here) — even one table-test that fires N goroutines at the same `name` and asserts `New(...)` was called exactly once would lock down the contract.
- `singleflight.Group` doesn't `Forget` on error by default, which means a failed startup is cached for the duration of in-flight callers — once the channel closes, the next call gets a fresh attempt. That's fine for this code (no `Forget` needed since the entry is removed when the inner func returns), but worth knowing for future readers.

## Verdict

`merge-after-nits` — solid refactor that fixes a real thundering-herd problem (many editors open many `.go` files at once → many parallel gopls bring-ups, only one of which the user wanted), with the right context decoupling via `WithoutCancel` and a nice log-noise demotion via `res.Shared`. Two nits before merge: add at least one concurrent-startup test against `singleflight`, and put a one-line comment on why `WaitForServerReady`-fail still returns `(client, nil)` so the next reader doesn't "fix" it.
