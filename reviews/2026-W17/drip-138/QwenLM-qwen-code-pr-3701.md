## Rationale

- **Right diagnosis of a real keyboard-flow gap.** The `/export` slash command takes a format argument (`html|md|json|jsonl`) and the suggestion list shows them, but pre-diff the arrow-key flow on a `/export`-only buffer either (a) navigated the *suggestion list* with no `setText` side effect or (b) navigated the *input history* depending on the focus state — neither of which is what a user expects. The fix at `InputPrompt.tsx:75-82` (`getExportFormatFromInput` regex matching `^\/export\s+(html|md|json|jsonl)$`) plus the `EXPORT_FORMAT_COMPLETIONS` constant at `:71-72` and the `exportFormatSuggestions` memo at `:235-249` correctly carve out the four-format set as a known completion target.
- **Test matrix at `InputPrompt.test.tsx:745-907` covers the load-bearing cells.** `:745-768` (`should submit a perfect match on Enter when suggestions were not navigated`) pins the no-arrow-then-Enter cell — the user typed `/export`, didn't press arrows, hit Enter; the buffer should submit `/export` verbatim and `handleAutocomplete` should NOT fire. `:771-800` (`should fill and submit an export format selected with arrow navigation`) pins the arrow-then-Enter cell — Down arrow fills `/export md ` (with trailing space, important for next-token typing); Enter then submits `/export md `. `:803-829` (`should keep cycling export formats after arrow navigation fills input`) pins the multi-arrow cycling cell — two Down arrows in a row produce `/export md ` then `/export json `, AND `mockInputHistory.navigateDown` is *not* called (the load-bearing negative assertion that input-history hijacking was the prior bug). `:832-895` is the integration cell that actually mounts a real `useTextBuffer` with the slash-command tree and verifies the suggestion list stays visible after the first arrow press.
- **The `completionSelectionWasNavigatedRef` and `exportCompletionSelectionIndexRef` `useRef`-tracked state at `InputPrompt.tsx:210-211` is the right structural choice.** Using refs (not state) means the cycling logic doesn't trigger re-renders on every arrow press, and the "did the user already navigate?" boolean separates the no-arrows-then-Enter path (submit perfect match) from the arrow-then-Enter path (submit selected format). This is the canonical "transient interaction state belongs in refs, persistent UI state belongs in state" split.
- **The `subCommandsByName` Map lookup at `:240-244` correctly defers to the slash-command tree's actual sub-command labels when present** (`command?.name ?? format` and `command?.value ?? format` fallbacks), so if a future PR adds an `xml` sub-command to `/export`, the suggestion display picks up the canonical label/description automatically — the format-list constant at `:72` becomes the only source-of-truth that needs updating, not the rendering path.

## Nits / follow-ups

- `EXPORT_FORMAT_COMPLETIONS` at `:72` is hardcoded in `InputPrompt.tsx` — a UI component shouldn't own the canonical list of export formats. Worth importing it from wherever `/export` is defined (probably a `commands/export/index.ts` or similar), so a future "we added XML export" PR doesn't need to also remember to update this UI-side list.
- The `getExportFormatFromInput` regex at `:78` (`^\/export\s+(html|md|json|jsonl)$`) duplicates the format list inline as a regex alternation — it would silently miss any format added to `EXPORT_FORMAT_COMPLETIONS` but not to the regex. Worth building the regex from the constant: ``new RegExp(`^\\/export\\s+(${EXPORT_FORMAT_COMPLETIONS.join('|')})$`)``.
- The PR body mentions Windows-only manual validation ("macOS and Linux manual validation were not performed"). The added Vitest tests cover the keyboard logic in a Node environment which is reasonably platform-portable, but the actual `\u001B[B` (Down-arrow ANSI sequence) handling in the real terminal can differ on macOS Terminal.app vs iTerm2 vs Linux gnome-terminal vs Windows Terminal. Worth either a CI matrix run or at minimum a smoke test on macOS before landing — Windows-only validation has surfaced TUI input regressions on Linux multiple times in this repo's recent history (see drip-135 #3694's "remove flaky TUI input tests" PR which mined 60 failed CI runs to identify the same class of platform-specific TUI flakes).
- Test name at `:745` says "should submit a perfect match on Enter when suggestions were not navigated" — the assertion at `:763-765` checks `props.onSubmit` was called with `/export` and `mockCommandCompletion.handleAutocomplete` was NOT called. That's tight, but the test setup at `:746-757` mocks `mockedUseCommandCompletion` to return `isPerfectMatch: true` — so the test is *also* implicitly anchoring "when `isPerfectMatch` is true, Enter submits without autocomplete." Worth either renaming the test to surface that, or adding an explicit `expect(mockCommandCompletion.activeSuggestionIndex).toBe(0)` assertion to lock the unmoved-cursor invariant.

## What I learned

Slash-command-with-argument completion is one of the trickiest CLI UX patterns to get right because three different intent paths (typing more chars, navigating suggestions, navigating input history) all consume the same arrow-key events at the same focus state. The "use refs to track transient interaction state" pattern at `:210-211` is the canonical way to express "this UI state matters for the *next* event but should not trigger re-renders." Mining `mockInputHistory.navigateDown` for the negative assertion at `:822` is exactly the right shape for "we routed an event somewhere new — assert the old route is no longer hit," strictly more informative than just asserting the new route fires. The hardcoded format list duplication between the constant and the regex is a future-bug magnet that one tiny refactor would resolve. The Windows-only-validation pattern is a recurring blind spot in this codebase worth a CI-level fix rather than per-PR review chasing.
