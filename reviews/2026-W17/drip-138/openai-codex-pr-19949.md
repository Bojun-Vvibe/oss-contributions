## Rationale

- **Right scope, right direction — extending the existing `MCP_SERVER_CONFIG` import path to also cover `SUBAGENTS`, `HOOKS`, and `COMMANDS` is a natural follow-up to the AGENTS_MD/SKILLS/PLUGINS migration scaffold.** The protocol schema additions at `protocol/v2.rs:1094-1102` add the three new enum variants symmetrically with `#[serde(rename = "...")]` and `#[ts(rename = "...")]` matching, and the JSON schemas at `schema/json/{ClientRequest,codex_app_server_protocol.schemas,codex_app_server_protocol.v2.schemas,v2/ExternalAgentConfigDetectResponse,v2/ExternalAgentConfigImportParams}.json` all get the same three new enum members appended in lockstep — this is the right "schema first, all surfaces in sync" hygiene.
- **The implementation symmetry at `external_agent_config.rs:189-243` is correct.** Each new variant gets the same three-step pattern (call importer, emit metric, propagate count where applicable) — `Subagents` and `Commands` get an item count via `import_subagents` and `import_commands` returning `usize`, while `Hooks` follows `McpServerConfig`'s no-count pattern since hooks are aggregated into a single feature config rather than enumerable items. The metric emission at each arm uses the existing `emit_migration_metric` plumbing so the new types instantly show up on the existing telemetry dashboard without dashboard-side changes.
- **However: the diff includes a 1155-line new file (`external_agent_config_migration.rs`) plus 308 lines of new tests (`external_agent_config_tests.rs`) that aren't visible in the diff sample, and the PR ships with an empty body (no description, no linked issue, no test plan).** A 1817-line surface-expansion PR with no PR description is the canonical "review can't actually verify the change" anti-pattern. The protocol changes are mechanically obvious; the 1155-line implementation file is where every interesting decision lives (which subdirectories does it walk? what's the conflict-resolution strategy when an existing config.toml already defines an MCP server with the same name? does it copy file contents or symlink? are file paths sanitized against directory-traversal? does it preserve hooks-source provenance for later un-import?), and none of that is documented in the PR.
- **The `serde_yaml` dependency add at `Cargo.toml:77` is a meaningful new attack surface.** `serde_yaml` has had RCE-class CVEs in the past (the alias-bomb / billion-laughs class). Importing arbitrary `.yaml` files from a user's `.claude/` directory is a controlled trust boundary in single-user mode but becomes a real concern in any shared/server deployment of the app-server. The PR doesn't mention which `serde_yaml` flags are enabled (`unbounded`/`bounded` recursion limits, etc.) or whether file-size/depth limits are enforced before parsing.
- **The `merge_missing_toml_values` call at `external_agent_config.rs:295` for MCP server config detection has subtle semantics that aren't documented in this PR.** The call asks "is the migrated content missing from the existing config?" and only adds the migration item if `should_include == true`. But the function name is `merge_missing_toml_values` (sounds like a mutating merge), the call site mutates `existing` in place (`merge_missing_toml_values(&mut existing, &migrated_mcp)`), and the return value is used as a should-include gate — a reader has to follow into the implementation to learn whether `existing` was actually written back to disk or just used for diffing. Worth a one-line `// detect-only: does not write back to target_config` comment at the call site, or splitting the function into `compute_missing_keys()` and `merge_into()` so the detect path doesn't need a mutable borrow.

## Nits / follow-ups

- The `mod` import at `external_agent_config.rs:24-25` uses `#[path = "external_agent_config_migration.rs"]` to inline a sibling file as a private submodule. This works but is unusual — most Rust code would use `mod external_agent_config_migration;` and let the standard module discovery find it. The explicit `#[path]` attribute is a flag that the file naming/visibility deviates from convention; worth a one-line rationale comment.
- The new file at `:34` (`external_agent_config_migration.rs`) is 1155 lines with eight pub-crate functions exported. Worth splitting into per-feature submodules (`mcp.rs`, `subagents.rs`, `hooks.rs`, `commands.rs`) to keep blast radius contained — a bug in `import_hooks` shouldn't risk regressions in `import_subagents` review-time.
- No tests visible in this diff sample (the 308-line `external_agent_config_tests.rs` is referenced in the PR file list but cut off in the 250-line diff sample I pulled). Need to verify the test file covers: (a) round-trip detect → import → re-detect produces no items (idempotence), (b) each importer is no-op when the source directory doesn't exist, (c) MCP-server-name collision behavior, (d) commands/subagents with non-UTF8 filenames or path-traversal-style names (`../../../etc/passwd`), (e) hooks YAML with malformed syntax (graceful failure vs panic).
- The empty PR body would block merge in any review system that requires a description; even an internal-team PR benefits from a 3-line "what changed / why / how tested" so reviewers (and `git blame` archeologists later) can orient.

## What I learned

Surface-expansion PRs that add three new enum variants across six schema files and a 1155-line implementation file *while shipping with an empty body* are the textbook "you can't actually review this" pattern — the diff mechanics are obvious (one new arm per variant, schema sync) but the value of the review is in the implementation choices that the PR body would normally call out. The right ask isn't "redo the work" but "PR body naming the conflict-resolution strategy, the trust-boundary assumptions for `serde_yaml`, and the not-tested-here cells." The `merge_missing_toml_values`-as-detect-gate at `:295` is a subtle structural hazard: function names that imply mutation but are called for their boolean return value are a future-bug magnet. The `#[path]`-inlined submodule at `:24` is a minor structural choice that signals an intentional deviation from convention; surfacing the rationale inline is cheap and prevents the next reader from "fixing" it.
