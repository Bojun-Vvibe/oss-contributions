## Rationale

- **Right shape for an installable codex skill — the SKILL.md frontmatter at `:1-4` follows the canonical name+description pattern that the install.sh harness keys on, and the `agents/openai.yaml` companion at `:1-4` provides the `display_name`/`short_description`/`default_prompt` triple that surfaces in the picker.** The skill is intentionally additive (159+/0) and adds two new files in an isolated directory; it doesn't modify any existing skill or rebuild the install pipeline, so blast radius is bounded to consumers who explicitly invoke `$codex-capacity-24h`.
- **The skill content itself is mature operational doctrine — not a stub.** `SKILL.md:7-10` (Core Rules) names the right load-bearing constraints up front: "collapse VMs into pool-level stories," "Pacific time for all daily peaks," "this is a risk analysis tool — forecast from the worst defensible scenario," and "use one decision-ready planning number, not ranges." These are the four classes of mistakes a generalist agent reaches for first when handed a capacity-forecast question (over-decomposing into per-VM noise, mixing time zones, comfort-midpoint forecasting, range-as-hedge). Anchoring them in Core Rules upstream of the workflow is the right teaching shape.
- **The Workflow section at `:31-69` correctly sequences the analysis: top-pool identification → traffic/peak pulls (with the load-bearing Pacific-day-bucket discipline at `:38-43`) → capacity signals by lane (with the load-bearing TBT-and-load-together check at `:50-52`) → forecast (with the explicit weekday/weekend/Monday-recovery branching at `:62-65`).** Step 3's `:50` "treat current TPM / load fraction only as a diagnostic load ceiling — do not sum those ceilings and call the result all-in usable capacity" is the right callout against the most common beginner mistake (linear-extrapolating from current load to a fake headroom number).
- **The Decision Rubric at `:67-71` (`OK ≥10%`, `OK but tight 5-10%`, `tight <5%`, `at risk` for over-capacity) gives a deterministic mapping from cushion percentage to label — the kind of structure that lets two operators reach the same verdict on the same data.** The explicit "based on cushion as a percentage of usable capacity, not only absolute TPM" at `:14` heads off the most common labeling mistake (calling a small absolute cushion "tight" when it's a healthy fraction of a small pool, or vice versa).
- **Evidence-link discipline at `:75-92` is unusually well-specified** — `Useful widget IDs` table at `:88-92` pre-stocks the load-bearing widget URLs (TPM, headroom, load) for the four top pools, so the agent doesn't have to guess or query. The "do/don't" examples at `:81-86` cover the three failure modes (raw-URL-suffix, Slack-syntax, link-wrapping-too-much-text) that are 90% of bad-link bugs in agent-authored Slack messages.

## Nits / follow-ups

- The skill description at `:3` is two sentences (192 chars by my count). Most skill-discovery tools index the *first sentence* of the description for triage; the second sentence ("Collapse virtual models...") is the load-bearing "use this when" detail. Worth flipping the order: lead with the trigger phrase, then the constraint summary.
- `:35` references `oaipkg monorepo` and `confs.yml` paths that are internal to the publishing system — not necessarily a bug for an internal-only skill, but worth a one-line "this skill requires monorepo access; will fail with PERM_DENIED otherwise" preflight note so a misconfigured installation surfaces a clean error rather than confusing data-empty results.
- `:88-92` hardcodes Datadog widget IDs that will drift when the dashboard is restructured. Worth a "if widget IDs return 404, fall back to dashboard root and warn the operator that the skill needs an update" failure mode rather than silent unlinked claims (which the skill already mandates at `:80`, but the hardcoded IDs make the silent-fallback the likely outcome).
- The Bad Hyperlinks block at `:84-86` correctly rejects Slack `<url|text>` syntax for Codex output — but the skill is also intended for Slack-ready reports per `:6` and `:97`. Worth one line clarifying the two surfaces (`Codex output: Markdown links; if Slack-posting via webhook, transform to <url|text>`) so the agent doesn't strip the link wrapping when handing the report off.
- `agents/openai.yaml`'s `default_prompt` is a single sentence — fine, but "Use $codex-capacity-24h to assess..." is somewhat redundant given that the picker already shows the skill name. A more directly-actionable default like "Run $codex-capacity-24h for the next 24 hours and post the report to #codex-capacity" would save a follow-up turn.

## What I learned

Operational skills like this are 80% taxonomy/discipline and 20% workflow — the load-bearing content is the Core Rules section, not the Workflow section, because the Workflow section is mechanical once the rules are internalized but the rules are what an agent reaches for *while running* to avoid each known failure mode. The "do/don't" pattern at `:97-129` is a teaching technique I should reach for more — it's strictly more informative than "do" alone because it shows the *gradient* of acceptable-vs-not, which is what the agent needs to navigate. The Pacific-time discipline (`:11`, `:38-43`) is worth porting as a generic "always specify and convert timezones at the boundary" rule for any skill that does day-bucket analysis. The hardcoded widget IDs at `:88-92` are the right tradeoff (faster + lower-failure than dashboard-traversal, accepting the drift cost) but should be paired with a graceful-degradation contract.
