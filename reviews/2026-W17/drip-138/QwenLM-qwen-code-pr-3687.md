## Verdict: merge-as-is

## Rationale

- **Right scope, right shape — `task_stop` was a kind-aware tool with a single registry; this PR widens it to two registries (subagent + background shell) with deterministic lookup precedence and identical user-facing semantics.** The pre-diff `task-stop.ts:43-83` had subagent registry as the only source-of-truth; the post-diff `:43-117` walks subagent registry first (`agentRegistry.get(taskId)` at `:50`) and falls back to `shellRegistry.get(taskId)` at `:74` only if the agent lookup misses. This is the canonical "extend without breaking the existing path" refactor — every previously-working call site still works because the agent-registry path is unchanged.
- **The two-namespace ID design (`<subagentName>-<suffix>` for agents, `bg_<8 hex chars>` for shells) makes collision practically impossible**, but the PR correctly adds a defensive test for the pathological-collision case. `task-stop.test.ts:147-178` (`prefers an agent over a shell when both have the same id (defensive)`) registers the same `'shared-id'` in both registries, asserts the agent path wins (`agentAc.signal.aborted === true`, `shellAc.signal.aborted === false`, shell entry status stays `running`). This is the right kind of test — pins a deterministic answer for an edge case that *can't happen today* but would silently produce nondeterministic behavior if a future ID-naming refactor broke the namespace isolation.
- **The `notRunningError` helper extraction at `task-stop.ts:114-127` is exactly the right deduplication.** Pre-diff, the not-running branch was a four-line `return { llmContent, returnDisplay, error }` block specific to agents. Post-diff, both registries call `notRunningError('agent', ...)` or `notRunningError('shell', ...)` and the helper formats the human-facing message with the right kind word. The error type itself is reused (`TASK_STOP_NOT_RUNNING`) — the PR body correctly cites that #3471's review made the error types kind-agnostic for exactly this kind of multi-registry future use. Symmetric `notRunningError('shell', taskId, status)` at `:75` handles the "shell already exited" case with `Background shell "bg_done" is not running (status: completed)` wording that's directly actionable for the model.
- **The `cancelled` settlement contract is correctly observable.** `task-stop.test.ts:101-122` (`cancels a running background shell`) registers a shell with a real `AbortController`, calls `task_stop`, then asserts (a) `result.error === undefined`, (b) `result.llmContent` includes the shell ID + command + output path (the load-bearing model-facing wiring — agent needs to know where to fetch logs after cancellation), (c) `shellRegistry.get('bg_a1b2c3d4').status === 'cancelled'`, and (d) the `AbortController.signal.aborted === true`. That last assertion is the load-bearing one — without it, the test would pass even if `shellRegistry.cancel()` only updated state without actually firing the abort signal that the running child process is awaiting.
- **The user-facing message at `task-stop.ts:81-86` correctly tells the agent where to find the captured output** (`captured output remains at ${shellEntry.outputPath}`) — this is what closes the loop on "I cancelled, what happened?" without forcing the agent to spawn another bash to find the log file. The parallel agent message at `:60-63` similarly names the description so the model can correlate which task it just stopped.

## Nits / follow-ups

- The lookup precedence ordering (agent first, shell second) is documented inline at `:46-49` with a load-bearing comment naming the ID-format invariant — but the comment doesn't say what should happen if the namespace invariant *is* broken in a future refactor. Worth a one-line "if you're refactoring the ID schemes and this can collide, the agent path wins by design — see the defensive test in task-stop.test.ts" pointer to the test as the canonical contract.
- `notRunningError` at `:114` takes `kind: 'agent' | 'shell'` as a string literal union — fine for two values, but if a third registry kind is added (e.g., "background hook"), the union widens and every call site needs updating. Worth considering a typed `RegistryKind` enum exported from a shared location, but acceptable as-is for two values.
- The PR doesn't touch `BackgroundShellRegistry`'s public API — the test relies on `shellRegistry.register({ ..., abortController: ac })` and `shellRegistry.cancel(taskId, Date.now())` and `shellRegistry.complete('bg_done', 0, Date.now())`. Worth a one-line "no API changes to BackgroundShellRegistry; this PR consumes its existing register/cancel/complete contract" note in the body — useful for readers landing this PR after a partial-cherrypick to understand the dependency boundary.
- The integration with the parent reasoning loop's "final task-notification carries the agent's last result" semantics is described in the inline comment at `:54-57` for the agent path but is not similarly described for the shell path. Worth a parallel "the shell's child-process exit handler settles the registry to `cancelled` and `/tasks` remains the source-of-truth for the final exit code" comment at `:71-73` to make the symmetric contract obvious.

## What I learned

The "two registries with deterministic precedence" pattern is the canonical shape for any tool that's being widened from one-source-of-truth to multi-source. The defensive test for the pathological-collision case (registering the same ID in both registries, asserting which one wins) is the right discipline even when the namespace invariant should make collision impossible — because the test pins the *answer* not just the *invariant*, so a future refactor that violates the invariant gets a deterministic-rather-than-random failure mode. The `notRunningError` helper extraction with `kind: 'agent' | 'shell'` is a textbook deduplication that took two near-identical 4-line blocks and replaced them with a single 13-line helper used twice — net +5 lines but eliminates the drift class. The "tell the model where to find the captured output" wording at `:81-86` is the right pattern for any cancellation path that leaves observable state behind: an explicit pointer to the inspection mechanism beats forcing the model to discover it. The Phase A/B/C decomposition (control plane in #3471 → registry in #3642 → tool wiring in this PR) is also a teaching example of staged complex-feature landing — each PR is independently reviewable and the dependency chain is explicit in the body.
