## Review

- New helper at `packages/opencode/src/cli/cmd/agent-match.ts:5-11` extracts the name-matching policy out of `RunCommand` into a pure two-pass resolver: first tries exact match (preserves prior behavior for casing-sensitive systems where two agents legitimately differ only by case), then falls back to `toLowerCase()` equality. The exact-first ordering is load-bearing — a user with agents `Build` and `build` running `--agent build` should get `build` not `Build`, and the early-return on exact match preserves that. Keeping the helper free of `Agent.Service` / `AppRuntime` dependencies (just operates on `AgentSummary[]`) is the right shape for testability and is what the new test file at `test/cli/agent-match.test.ts:5-16` exploits.
- Two call-site rewrites at `cli/cmd/run.ts:586-606` and `:608-628` now both go: `matchAgentName(name, list) → find original entry by matched name → fall through to warning if not matched`. Returning the *canonical* matched name (`return matched` not `return name`) is the load-bearing part of the fix — downstream session creation, agent lookup, and event emission all key on the canonical name, so passing the user's typed casing through would silently bifurcate state (session created against `Build` while agent registry holds `build`). The `if (!agent || !matched)` guard is technically redundant (`matched` undefined ⇒ `agent` undefined via the conditional `find`), but the dual-check is defensible defense-in-depth.
- The second call site (`Agent.Service.use((svc) => svc.list())`) replaces what was previously a direct `svc.get(name)` lookup. This is a small but measurable cost regression — `get` was likely O(1) hash lookup while `list()` materializes the full agent set just to do a linear scan. For users with many agents (or projects mounting many agent files) this could be noticeable. Worth either: (a) adding a `svc.getCaseInsensitive(name)` API to push the case-folding into the service so it can use its existing index, or (b) commenting why the linear scan is acceptable. Not a blocker.
- Test coverage is minimal but correct — covers exact match, two casings of case-insensitive, and undefined-on-no-match. Missing: collision test (two agents `build` and `Build` both present, `--agent BUILD` should return *one* deterministically — current implementation returns whichever `find` hits first, which is array-order-dependent and undocumented). Adding `expect(matchAgentName("BUILD", [{ name: "build" }, { name: "Build" }])).toBe("build")` would pin the behavior.
