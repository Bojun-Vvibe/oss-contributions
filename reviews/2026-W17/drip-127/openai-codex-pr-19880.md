## Review

- New `ExecExpiration::cancellation_token()` accessor at `codex-rs/core/src/exec.rs:225-233` extracts the inner `CancellationToken` from the two variants that carry one (`Cancellation`, `TimeoutOrCancellation`) and returns `None` for the timeout-only variants. The `#[cfg_attr(not(target_os = "windows"), allow(dead_code))]` is correct platform scoping ŌĆö only Windows sandbox path consumes this ŌĆö and prevents the dead-code lint on Linux/macOS builds. Match arms exhaustively cover all four variants which means adding a new variant later forces this site to be updated (compiler-enforced, the right pattern).
- The wire-up at `exec.rs:592-611` resolves the prior `TODO(iceweasel-oai)` comment about `run_windows_sandbox_capture` not supporting all `ExecExpiration` variants. Wrapping the `CancellationToken` in `WindowsSandboxCancellationToken::new(move || token.is_cancelled())` is the right shape ŌĆö keeps the sandbox crate decoupled from `tokio_util::sync::CancellationToken` (windows-sandbox-rs shouldn't need a tokio dep just to receive a "should I cancel?" signal). The `is_cancelled()` closure is Send+'static which the cancel-writer thread requires.
- The cancel-writer thread at `elevated_impl.rs:118-148` is the load-bearing piece for actually translating cancellation into an in-band `Message::Terminate` frame. Polling loop with `park_timeout(Duration::from_millis(50))` is reasonable (50ms cancellation latency is fine for sandbox tear-down; tighter polling would burn CPU for no user benefit), and the `done` `AtomicBool` provides the early-exit signal so the thread doesn't outlive the sandbox process. Two concerns: (1) the `Ordering::SeqCst` on both the load and the implied stores is heavier than needed ŌĆö `Acquire`/`Release` would suffice for this single-flag handshake ŌĆö but SeqCst is the conservative default and the perf delta is negligible at 50ms cadence. (2) `try_clone()` on the pipe write half could fail (e.g. handle-table exhaustion) and the `?` propagates that as `Result`, but the surrounding code path probably can't recover ŌĆö would benefit from a comment naming why this failure is fatal.
- The `Terminate` framed message is fire-and-forget (`let _ = write_frame(...)`) ŌĆö silent on write failure. For cancellation specifically that's defensible (if the pipe is broken, the sandbox is probably already gone) but worth a `tracing::debug!` on the error path so an operator debugging "why didn't my Ctrl-C kill the sandbox" can see whether the terminate frame actually went out.
- No test added for the cancellation-during-network-denial path. The PR title implies this fix targets a specific UX bug (network denial inside the sandbox would hang until timeout instead of unwinding immediately on outer cancel), but there's no integration test pinning that. A test that triggers a network-denial inside a Windows sandbox under an outer `CancellationToken` and asserts the sandbox unwinds in <1s would prevent regression.
