## Review

- New `plan_mode_nudge_visible: bool` field on `ChatComposer` at `chat_composer.rs:352` plus init at `:529` adds a per-composer flag that gates the nudge footer ("Create a plan? Shift+Tab use Plan mode  Esc dismiss"). Field placement among the other footer-state booleans (`disable_paste_burst`, `footer_hint_override`) is consistent with surrounding ownership; default `false` is the right initial state (don't nudge until classifier trips).
- `set_plan_mode_nudge_visible` at `:994-1000` returns `bool` indicating whether visibility *changed*, not just the new value. This is the load-bearing piece for callers to avoid redundant repaints — a classifier that runs on every keystroke would otherwise trigger unconditional re-renders. The `if self.plan_mode_nudge_visible == visible { return false; }` short-circuit is the correct guard. Worth pulling this idiom into a helper if it's repeated for other footer fields, but as a one-off it's fine.
- Render path at `:3978-3989` is correctly chained: `history_search_footer_line()` first (already in the prior arm), then `plan_mode_nudge_visible` second, then the default footer. The precedence order matters — history search is a transient modal that should always win over a soft nudge. Truncation via `truncate_line_with_ellipsis_if_overflow` is consistent with the default branch.
- New `plan_mode_nudge_line()` constructor at `:444-454` builds the footer with `magenta()` for "Create a plan?" plus inline key hints. Color choice differentiates the nudge from regular footer hints (which are typically dimmed) — good UX signaling. One concern: the `Esc dismiss` affordance implies Esc binding for dismissal, but I don't see the corresponding key handler in this diff slice. If `Esc` is already wired globally to clear footer state, that's fine, but worth verifying the binding actually fires `set_plan_mode_nudge_visible(false)` rather than no-oping.
- New `composer_input_enabled()` accessor at `bottom_pane/mod.rs:752-754` plus `input_enabled()` at `:973-975` exposes composer state to the parent pane — presumably so the plan-mode classifier only runs when input is actually editable. Trivial passthrough, no concerns.
- This PR is +261/-0 with no test changes despite adding a new user-visible UI element with a non-trivial state-machine. At minimum, a snapshot test asserting the nudge line renders correctly at narrow widths (truncation behavior) and a test toggling `set_plan_mode_nudge_visible(true)` then `(false)` to assert the change-detection contract would close the loop. The classifier logic itself (presumably in another file not in this slice) needs unit coverage too — false positives are the obvious failure mode.
- Missing from this slice: where does the classifier *trigger* the nudge? If it's keystroke-rate that's a hot path; if it's a debounced background task that's safer. Worth a comment on `set_plan_mode_nudge_visible` documenting the expected caller cadence so future maintenance doesn't accidentally promote it to a tight loop.
