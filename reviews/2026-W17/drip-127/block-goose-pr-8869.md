# block/goose #8869 — fix: correct WSL2 OS detection by removing PWD-based Windows override

- URL: https://github.com/block/goose/pull/8869
- Head SHA: `2efc795775247a6f2ffc4514aa010593c0d801a8`
- Verdict: **merge-as-is**

## Review

- Net `+4/-31` to `download_cli.sh` — almost all deletion. Removes a tangle of WSL detection branches that all tried to second-guess the OS based on `$PWD` heuristics (`[[ "$PWD" =~ ^/mnt/[a-zA-Z]/ ]]`, presence of `/c`/`/d`/`/e` mount points, presence of `powershell.exe`/`cmd.exe` in PATH). The replacement is a single comment-supported branch at `:97-99` that names the right invariant: "WSL is a Linux environment regardless of the current working directory. The PWD (e.g. /mnt/c/) does not change the kernel — always install Linux."
- This is the load-bearing realization. The prior code conflated *filesystem location* with *runtime ABI*. A user `cd`-ing into `/mnt/c/foo` is still running a Linux ELF binary on the WSL2 Linux kernel — installing the Windows `.exe` build there would either fail to execute under Linux or, worse, succeed via WSL's binfmt_misc and produce a confusing mixed installation with PATH conflicts. The correct answer is "WSL ⇒ Linux, full stop."
- The `BIN_DIR` simplification at `:60-67` similarly collapses three branches (native Windows, WSL with proc-version-match, WSL with /mnt/ PWD) into two: native Windows uses `$USERPROFILE/goose`, *everything else* (Linux, macOS, WSL) uses `$HOME/.local/bin`. This is the right shape — the prior WSL-specific branch was redundant because `/proc/version`-detected WSL would also satisfy the default Linux branch. Removing the redundancy means there's no second-source-of-truth to drift.
- One subtle removal at `:103-104` (the trailing `elif command -v powershell.exe ...` branch in the OS detection): a non-WSL Linux machine that happens to have powershell.exe in PATH (e.g., powershell-on-linux installed via apt) would previously have been misclassified as Windows. That's gone now — Linux-with-pwsh stays Linux. Good removal.
- The remaining at `:108-110` (Git Bash detection: `[[ "$PWD" =~ ^/[a-zA-Z]/ ]] && [[ -d "/c" || -d "/d" || -d "/e" ]]`) is preserved and is the *correct* place to keep PWD-based detection — Git Bash on native Windows really does run a different ABI from cygwin/msys (which is caught earlier at the `OSTYPE` branch). The asymmetry between "drop PWD detection for WSL" and "keep PWD detection for Git Bash" is exactly right because the underlying environments differ — WSL has a Linux kernel, Git Bash has a Windows kernel with a POSIX-emulation shell.
- No test in this PR. Shell installer scripts are notoriously hard to test, but a CI matrix entry that runs `download_cli.sh --dry-run` (if one exists) under (a) Ubuntu 22.04, (b) Ubuntu 22.04 simulating WSL via `/proc/version`, (c) macOS, and asserts each picks the right OS would close the loop. Not a blocker — the change is a clear net simplification and the prior code was demonstrably wrong for the WSL-with-Windows-mount case.
- Worth mentioning in release notes: users who previously relied on the buggy "WSL in /mnt/c/ installs Windows build" behavior (probably nobody intentional, but possible) will see different behavior on next install. The new behavior is "always Linux from WSL" which matches every other tool's WSL handling — but the breaking-implication should be flagged for any user who scripts goose installation in CI.
