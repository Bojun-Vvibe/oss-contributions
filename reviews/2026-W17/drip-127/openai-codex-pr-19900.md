## Review

- The colon-prefix sigil for built-in profiles (`default_permissions = ":workspace"`) is documented at `config_toml.rs:117-119` and mirrored in `core/config.schema.json:2497`. Using `:` as the namespace separator is a defensible choice — it's not a valid identifier character in TOML keys and is unlikely to collide with user-defined profile names. The doc-comment update is the right shape (names starting with `:` refer to built-in; others resolve from `[permissions]` table), and the schema description mirrors it exactly so doc/code don't drift.
- New test `default_permissions_can_select_builtin_profile_without_permissions_table` at `config_tests.rs:1192-1218` is the load-bearing assertion that sets the contract: a user with *no* `[permissions]` table at all can still select `:workspace` as their default. Asserts both the positive (project root writable) and the negative (`.git` carveout protected) — the second assert is the more important one because a buggy implementation that just rubber-stamped any `:workspace`-named profile with full FS access would pass the first assert but fail the second.
- `empty_config_defaults_to_builtin_profile_for_trusted_project` at `:1224-1264` covers the implicit-default path: when a project is `TrustLevel::Trusted` and no explicit `default_permissions` is set, the system falls back to `:workspace`. The platform fork at `:1248-1262` (Windows stays read-only because Windows sandbox isn't ready, non-Windows uses `:workspace`) is the right hedge — pinning Windows to read-only by test means a future sandbox rollout there will need to update this assertion deliberately rather than silently flipping behavior. The `.codex` metadata carveout assertion at `:1259-1262` is the same defense-in-depth as above.
- `empty_config_defaults_to_builtin_read_only_without_trust_decision` (truncated in this slice but visible by name) closes the third corner: untrusted/undecided projects get the strictest built-in. This three-cell coverage matrix (explicit `:workspace` / trusted-implicit `:workspace` / no-trust-decision read-only) is the right shape for the policy layer.
- The PR is +335/-42 but only 3 of those tests are visible in this diff slice. The actual built-in profile *registration* (where `:workspace` is defined as a `PermissionsProfile` value) isn't in the visible slice — that's where the real surface area sits. Reviewer should pull the full diff to verify the built-in profile values are actually conservative (e.g., `:workspace` should not include network access by default, or the read-only built-in should genuinely deny all writes including `~/.codex/`). Marking merge-after-nits pending that verification.
- Naming concern: `:workspace` and `:read-only` (assumed) collide somewhat with shell-style variable expansion. A future shell wrapper that does `eval` on a config string could accidentally expand `:workspace` as a substitution syntax. Documentation should call out that the colon prefix is *literal* in the TOML config and shouldn't be quoted with shell-style escapes.
- Migration story for users currently using a literal profile named `workspace` (no colon): the precedence is presumably "user-defined wins on exact name, built-in wins only when name starts with `:`" — worth an explicit test asserting that `[permissions.workspace]` without colon correctly shadows nothing and `default_permissions = "workspace"` finds the user-defined one.
