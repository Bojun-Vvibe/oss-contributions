# sst/opencode #24702 — fix(core): remap tail_start_id for compaction parts when forking sessions

- URL: https://github.com/sst/opencode/pull/24702

- Verdict: **merge-as-is**

## Review

- Single-line fix at `packages/opencode/src/session/session.ts:611` closes a real correctness bug in session forking. When a session containing `compaction`-type parts was cloned, every other field on the cloned part was being re-assigned through the new `idMap` (so `messageID`, `sessionID`, `id` all resolved into the new session's ID space) but `tail_start_id` — which compaction parts use to point at the message ID where the post-compaction tail begins — was being silently inherited from the *source* session. After fork, that ID would either be a dangling reference (if the source was deleted) or a cross-session pointer that any consumer walking from compaction → tail would follow into the wrong session's history.
- The conditional spread `...(part.type === "compaction" && part.tail_start_id && { tail_start_id: idMap.get(part.tail_start_id) })` is the right shape: type-narrows on `part.type === "compaction"` (so the spread only fires for the part variant that has the field), guards on `part.tail_start_id` truthiness (so `undefined`/missing values don't get mapped to `idMap.get(undefined)` which would be `undefined` anyway but cleaner not to call), and uses the same `idMap` that the surrounding clone loop already populates for `messageID`/etc. — so the remapped `tail_start_id` is guaranteed to resolve in the cloned session by the same invariant that makes the rest of the clone correct.
- One latent concern: `idMap.get(part.tail_start_id)` returns `undefined` if the source `tail_start_id` points at a message that wasn't included in the clone (e.g. fork from a partial range). The spread would then write `tail_start_id: undefined`, which the schema may or may not accept depending on whether `tail_start_id` is `Optional<string>` vs `string | undefined`. Worth a follow-up: either defensive skip when the lookup misses, or assert that compaction tails are always within the cloned range. Not a blocker for this PR — pre-existing forks already had the wrong reference, so any improvement is strictly better.
- No regression test for the forking-with-compaction case. Given how silent the failure mode is (cross-session reference, won't crash, just produces incorrect compaction-aware UI), a test asserting `cloned.tail_start_id === idMap.get(original.tail_start_id)` for at least one compaction part would prevent the bug from recurring. Marking merge-as-is given the surgical nature, but a follow-up test would be cheap insurance.
