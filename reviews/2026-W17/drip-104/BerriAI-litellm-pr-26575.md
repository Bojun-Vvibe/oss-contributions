## Summary

Two coupled changes in the MCP OAuth discovery path: (1) extend `_descovery_metadata()` and `_fetch_single_authorization_server_metadata()` so an issuer URL of the form `https://<azure-identity-host>/<tenant-id>/v2.0` resolves correctly (probing the issuer-rooted `.well-known/openid-configuration` and falling back to a derived metadata struct when the well-known probe also misses), AND (2) remove `Depends(user_api_key_auth)` from the `/v1/mcp/server/oauth/{server_id}/authorize` and `/.../token` routes so a browser-based OAuth dance can complete without a proxy API key. The two changes are bundled in one PR but have very different review surfaces — (1) is additive and safe, (2) is a security-relevant route-deauthing that deserves its own conversation.

## Specific observations

- `litellm/proxy/_experimental/mcp_server/mcp_server_manager.py:1684` — adds the issuer-rooted `f"{issuer_url.rstrip('/')}/.well-known/openid-configuration"` candidate URL alongside the existing `base + path` and `base`-only variants. This is the correct extension for tenant-scoped issuers where the tenant id sits between the host and `/v2.0` and the well-known doc must be appended *to the issuer itself*, not to the origin. Order of probes preserves the prior fast paths.
- `litellm/proxy/_experimental/mcp_server/mcp_server_manager.py:1726-1751` — new `_build_authorization_server_metadata` (renamed in this review to avoid naming a vendor) is a hardcoded fallback that fires only when (a) the issuer hostname is in a small allowlist of three public identity-authority hostnames (commercial, US gov, China cloud), (b) the path has exactly two segments, (c) the second segment is literally `v2.0`. All three guards must hold, and the helper returns the tenant-derived `oauth2/v2.0/authorize` and `oauth2/v2.0/token` URLs. This is a reasonable last-resort because the public well-known endpoint occasionally 404s for new tenants during propagation, but it permanently encodes a vendor-specific URL template into the generic OAuth discovery path. A `verbose_logger.info()` on the fallback firing would make it observable.
- `litellm/proxy/_experimental/mcp_server/mcp_server_manager.py:1488-1502` — `_descovery_metadata` previously *raised* `RuntimeError("OAuth discovery must not succeed without a challenge")` if the upstream MCP server returned 200 instead of a 401 challenge; this PR changes the same code path to silently fall through to a `_attempt_well_known_discovery` probe. That semantic flip is correct for the new use case (tenant-scoped issuer doesn't need to challenge first) but discards a real signal — a 200 from a server that should-have-challenged was previously an actionable misconfiguration warning. Worth keeping the prior `verbose_logger.warning` line on the fall-through path.
- `litellm/proxy/management_endpoints/mcp_management_endpoints.py:1452+1500-1560` — drops the `dependencies=[Depends(user_api_key_auth)]` on the `/server/oauth/{server_id}/authorize` and `/.../token` routes and removes the `user_api_key_dict` parameter, then in `_get_cached_temporary_mcp_server_or_404` weakens the admin-only check from `if not _user_has_admin_view(user_api_key_dict)` to `if user_api_key_dict is not None and not _user_has_admin_view(user_api_key_dict)` so passing `None` bypasses it entirely. The new test at `test_mcp_management_endpoints.py:1597-1626` (`test_mcp_authorize_and_token_routes_are_browser_callable`) explicitly *asserts* the deauthing as desired behavior — but the route is still scoped to a `server_id` that comes from a `get_cached_temporary_mcp_server` cache, and the threat model is "any unauthenticated caller can probe arbitrary `server_id`s and trigger an OAuth redirect to a redirect_uri the caller controls." The cache is described in the surrounding comment as "admin-only setup flow," but with auth removed from the *callback* surface, the redirect_uri parameter becomes an open-redirect vector unless the cached server stores an allowlist of acceptable redirect URIs. The diff does not show that allowlist check.
- The admin-only-check predicate at `:1478-1480` now uses Python truthiness on `user_api_key_dict` — `is not None` is correct (an empty/default `UserAPIKeyAuth` would still pass `is not None`) but the surrounding code's habit of using `_user_has_admin_view()` should be wrapped in a `_caller_is_browser_oauth_flow()` helper rather than coded as a `None` sentinel, so the intent is grep-able.
- 18-line tests for the discovery path (`test_descovery_metadata_probes_well_known...`, `test_fetch_single_authorization_server_metadata_supports_*_issuer_path`, `test_fetch_single_authorization_server_metadata_derives_*_metadata`) correctly mock the httpx client and pin both the well-known-success and well-known-404-then-fallback paths. Good coverage on the discovery half.

## Verdict

`needs-discussion` — the discovery extension at (1) is mergeable as-is with one log-line follow-up, but the route deauthing at (2) is a security-relevant change that should be split into its own PR with explicit threat-model justification and an open-redirect mitigation (allowlist of `redirect_uri` values stored on the cached server record, or signed state token verifying the redirect target). Bundling a "support new identity provider" feature with "remove auth from two OAuth-related routes" makes review and bisect harder, and the included test that asserts the deauthing locks in the new contract before the conversation has happened.
