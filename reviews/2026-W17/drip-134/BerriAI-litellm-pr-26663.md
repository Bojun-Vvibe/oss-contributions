## Rationale

- **Three unrelated fixes bundled in one PR** — (1) auth `access_group_ids` enforcement when `models=[]`, (2) `_update_user_db` bypass for team-key calls in spend tracking, (3) ZAI provider message content flattening for GLM chat templates. The PR title and description only cover fix #1. Each has its own root cause, its own affected surface, its own test file, and its own potential blast radius — bundling forces every reviewer to context-switch between the auth/spend/provider-transform domains and a request-changes on any one of them blocks the other two. The two unrelated fixes need to be split into separate PRs; this is the primary blocker.
- **The named auth fix itself is correctly shaped.** `auth_checks.py:2693-2710` adds the early check before the `try` block: when `valid_token.access_group_ids` is non-empty AND `valid_token.models` is empty (i.e., the silently-bypassed cell per the PR body's "`_check_model_access_helper` evaluates `len(models) == 0` as all-model access — returning `True` immediately without ever raising"), resolve `_get_models_from_access_groups()` first and pass that as the constrained `models` list to `_can_object_call_model`. Symmetrically applied to `can_team_access_model` at `:2770-2787`. The 4 unit tests added in `test_auth_checks.py` (per PR body: key in/out of group + team in/out of group) are the right cells. The existing fallback path (models non-empty but denied → check access groups) is preserved unchanged at `:2716-2731` / `:2789-2804`.
- **The spend-tracking change is a behavior-change with subtle invariants.** `db_spend_update_writer.py:507-541` adds an early return in `_update_user_db` when `team_id is not None`, with the carefully-named carve-out that *end-user* spend updates still get enqueued via `SpendUpdateQueueItem(entity_type=Litellm_EntityType.END_USER, ...)`. The comment at `:511-518` ("Without this guard, `LiteLLM_UserTable.spend` accumulates team key costs and causes false `BudgetExceededError` on personal key calls when the Redis counter is absent (e.g. after a restart)") names a real bug, but: (a) the change to `_batch_database_updates` at `:340` propagates `team_id=team_id` which is a new dependency that wasn't shown in the diff context — needs verification that all `_batch_database_updates` callers actually have a meaningful `team_id`; (b) silently dropping personal/global spend tracking for team keys means the `LiteLLM_UserTable.spend` for the user will *understate* their true spend across all keys — this may break dashboards/exports that aggregate user spend across personal+team keys without joining `LiteLLM_TeamTable`; (c) no test was added covering the "team key + end user" cell that the carve-out was specifically designed to preserve. The PR description doesn't mention this change at all.
- **The ZAI message flattening fix is reasonable but lives in the wrong PR.** `zai/chat/transformation.py:11-34` adds `_flatten_content_parts()` and a new `_transform_messages` override at `:84-99` that walks tool/assistant messages and flattens list-format `content` to plain strings before forwarding to z.ai. The named root cause ("GLM's chat template checks `m.content is string` and silently drops list-format content, same root cause as vllm-project/vllm#39614") is well-attributed and the test coverage in `test_zai_provider.py` (114 new lines, `TestZAIMessageTransformation` class with 2+ test methods including a tool-message and assistant-message cell) is the right shape — but this should be its own PR with its own issue link.

## Required changes

1. **Split this PR into 3 PRs.** The named scope (`fix(auth): enforce access_group_ids model restriction...`) only covers `auth_checks.py` + `test_auth_checks.py` + the access-group-ids parts of `_can_object_call_model`. The spend-tracker and ZAI changes need their own titles, their own PR descriptions naming their own root causes, and their own reviewers (proxy/db owners and provider-transform owners are different from auth owners).
2. **For the spend-tracker carve-out (after split):** add a unit test pinning the "team key call with `end_user_id` set still enqueues the END_USER SpendUpdateQueueItem" cell; document the user-spend-now-understates-personal+team-aggregate semantic in the PR body so dashboard/export consumers can plan; verify all `_batch_database_updates` call sites supply a meaningful `team_id`.
3. **For the auth fix itself:** the unit-test count (4 cells) is correct; consider also pinning the cell "key with `access_group_ids` set AND `models` non-empty AND model in group AND model NOT in `models`" to lock down the precedence rule (the new early check skips `models` entirely, which is intentional but should be regression-pinned).
