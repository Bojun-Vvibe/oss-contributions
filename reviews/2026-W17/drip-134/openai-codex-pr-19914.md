## Verdict: merge-after-nits

## Rationale

- **Right architectural shape: validate ‚Üí download ‚Üí install ‚Üí notify backend ‚Üí load apps from the materialized bundle.** `plugins.rs:421-485` reorders the install flow so the bundle is validated and downloaded *before* the backend `install_remote_plugin` call, then the backend is notified, then `load_plugin_apps` and `load_plugin_mcp_servers` read directly from the just-cached `result.installed_path`. The inline comment at `:455-457` ("Cache first so a backend install cannot succeed when local materialization fails") names the invariant correctly ‚Äî if the local bundle is corrupt or the download URL is missing, the backend never gets the install POST so the user-visible state stays consistent (no orphaned "installed in cloud, missing locally" rows). This is the right conservatism.
- **Remote-only behavior is the right scope and is well-tested.** `plugin_install.rs:185-274` `plugin_install_writes_remote_plugin_to_cloud_and_cache` exercises the full happy path (mock detail endpoint with `bundle_download_url`, mock installed-list, mock install POST, mock bundle GET) and verifies four cells post-install: `wait_for_remote_plugin_request_count` for both the install POST (`:230-235`) and the bundle GET (`:236-242`), `installed_path.join(".codex-plugin/plugin.json").is_file()`, `installed_path.join("skills/plan-work/SKILL.md").is_file()` (proving the `.tar.gz` was extracted in full), and the negative `!plugins/cache/chatgpt-global/{REMOTE_PLUGIN_ID}/1.2.3.exists()` (`:251-257`) pinning that the cache key uses the *display name* `linear` not the opaque plugin id. The negative-path test `plugin_install_rejects_missing_remote_bundle_url` (added at `:285+`) covers the absent-bundle-URL cell.
- **Switch from `fetch_remote_plugin_detail` to `fetch_remote_plugin_detail_with_download_urls`.** `plugins.rs:424-435` calls a new variant that returns `bundle_download_url` and `release_version` alongside the existing summary ‚Äî correctly named (the `_with_download_urls` suffix signals the explicit superset), and the existing `release_version` is then propagated into `validate_remote_plugin_bundle` so the cache layout `cache/<marketplace>/<name>/<version>` is version-keyed (good for future "two versions of the same plugin coexist" semantics).
- **Test environment gating is right.** The `CODEX_TEST_ALLOW_HTTP_REMOTE_PLUGIN_BUNDLE_DOWNLOADS` env var (`plugin_install.rs:62-63`, threaded through `McpProcess::new_with_env` at `:208-211`) cleanly separates production "HTTPS-only bundles" from the test mock server's `http://` URLs. Production code never sees this env var; the env-gated allow-http path is in `validate_remote_plugin_bundle` (`remote_bundle.rs`).
- **Auto-trigger MCP refresh + OAuth login surface after install** at `plugins.rs:472-481` correctly handles the case where the newly-installed plugin ships MCP servers ‚Äî `queue_mcp_server_refresh_for_config` is called only if `plugin_mcp_servers` is non-empty (avoiding an unnecessary refresh for skills-only bundles), and the warn-on-error rather than fail-on-error is right (the install succeeded; MCP refresh is a best-effort follow-up).

## Nits / follow-ups

- The `TODO(remote plugins)` comment at `plugins.rs:471-473` ("remote marketplaces do not yet have a local marketplace/read-path sync, so this install path reads MCP/apps directly from the just-cached bundle") flags real future work ‚Äî the install path now bypasses the marketplace abstraction and reads `plugin.json` straight from disk, which means the read-path sync needs to land before remote and local plugins behave uniformly downstream. Worth a tracking issue link.
- The `remote_plugin_bundle_install_error_to_jsonrpc` helper at `plugins.rs:744-749` collapses every variant of `RemotePluginBundleInstallError` into a generic `internal_error(format!(...))` ‚Äî losing the distinction between user-actionable errors (bad checksum, bad URL scheme) and operator errors (disk full, network failure). The catalog-error helper above it (`remote_plugin_catalog_error_to_jsonrpc` at `:711-741`) does this distinction correctly with a `match`; the bundle-install path should follow the same pattern so client UIs can render appropriate guidance.
- `+848` lines in a single new `remote_bundle.rs` file is a lot to land in one PR; would have been easier to review split into "validation primitives + tests" and "download + extraction + tests" ‚Äî but given the test coverage is already there in `plugin_install.rs` (3+ new test cases) and the validate/download are tightly coupled, this is a soft preference not a blocker.
- Confirm that `tar = { version = "=0.4.45", default-features = false }` (Cargo.toml `:354`) was deliberately exact-pinned (`=` prefix) for security/reproducibility reasons rather than as an accident ‚Äî most other workspace deps use `^` semver.
