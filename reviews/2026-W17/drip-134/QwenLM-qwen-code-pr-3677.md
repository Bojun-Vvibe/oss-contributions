## Verdict: merge-after-nits

## Rationale

- **Provider-gated parsing is the right architectural choice.** `provider/minimax.ts` (new, +31) registers a MiniMax provider that opts in to tagged thinking parsing only when the endpoint matches `*.minimaxi.com` — confirmed by the PR's "Reviewer focus" item ("Confirm tagged thinking parsing is enabled only for MiniMax"). This is the correct conservatism: other OpenAI-compatible providers (e.g., a generic vLLM endpoint serving an unrelated model that happens to emit `<think>` for prompt-template reasons) will not have their content rewritten. The opt-in flag travels through `responseParsingOptions.ts` (+9) into `converter.ts` (+30/-5) and `pipeline.ts` (+9) — clean propagation through one config struct rather than scattered conditionals.
- **`taggedThinkingParser.ts` (+80) handles the load-bearing streaming-tag-split cell.** The PR body's "Confirm streaming parsing handles tags split across chunks" reviewer item maps to the right concern — `<think>` may arrive as `<thi` + `nk>` across two SSE chunks, and a naive `content.split("<think>")` would miss it on the first chunk and emit `nk>` as visible content on the second. A standalone parser module (rather than a regex applied per-chunk) is the right shape because it can carry partial-match state across chunk boundaries. Test file `provider/minimax.test.ts` (+93) and `converter.test.ts` (+251) together exercise the unification surface.
- **CLI display fix for leading-blank chunks is independent and correct in its own right.** `useGeminiStream.ts:108-111` adds `stripLeadingBlankLines = (text) => text.replace(/^(?:[ \t]*\r?\n)+/, '')` and applies it in two places: the agent-message buffer at `:773-781` (where if the buffer's `trim().length === 0` the function returns early without finalizing the previous pending item or starting a new `gemini` row) and the thought buffer at `:854-867` (symmetric early-return + strip). The `if (newGeminiMessageBuffer.trim().length === 0) return newGeminiMessageBuffer;` early-return is the load-bearing cell — it prevents the empty-row symptom (`✦` appearing on its own line) by not creating the pending history item until there's actual content. Test `does not render leading blank content chunks as an empty assistant item` at `useGeminiStream.test.tsx:1208-1278` and the symmetric `does not render leading blank thought chunks` at `:1336-1406` pin both branches with the right cells: leading `\n\n` chunk arrives → `pendingHistoryItems` stays empty (`expect(...).toEqual([])`), then a `'哈哈'` content chunk arrives → exactly one pending item with `text: '哈哈'`.
- **Three closed issues (#3669, #3228, #3387) imply the bug had multiple independent reports**, increasing confidence that the symptoms are real and the fix is hitting the actual user-visible problem rather than a narrow reproduction.
- **`useGeminiStream.test.tsx:152-154` mock setup change** (`vi.mocked(findLastSafeSplitPoint).mockImplementation((s: string) => s.length)`) is the right scaffolding — making the safe-split-point function a no-op-by-default keeps the leading-blank tests focused on the buffer-management logic rather than tangling with markdown-split semantics.

## Nits / follow-ups

- The two `if (...trim().length === 0) return ...;` early-returns at `useGeminiStream.ts:773` and `:856` are duplicated logic across the agent-message and thought paths — worth a one-line shared helper like `shouldStartNewPendingItem(buffer)` to keep the two sites in sync if a future contributor adjusts the predicate.
- The provider-gating `*.minimaxi.com` check in `provider/minimax.ts` is a string match — worth confirming whether MiniMax has any alternate hostnames (regional endpoints, on-prem deployments) that would silently fall back to the no-thinking-parse path. The opposite-direction risk (false positive on a non-MiniMax host that happens to contain "minimaxi.com" as a substring) should be ruled out by an exact-suffix or hostname-parse rather than a substring check.
- The CLI fix and the converter fix are technically two separate concerns folded into one PR, but they share the same user-visible symptom on MiniMax (empty `✦` row + raw `<think>` tags) and are tightly coupled by the test setup, so this is a soft observation rather than a request to split.
- No end-to-end test against a real MiniMax endpoint per the PR's "Not covered" disclosure — acceptable given the hermetic mock coverage is comprehensive, but worth a manual smoke before tagging.
