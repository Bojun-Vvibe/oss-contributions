## Specifics

- Status snapshot type at `packages/core/src/lsp/types.ts` introduces `LspStatusSnapshot` with discrete counters (`configuredServers`, `readyServers`, `failedServers`, `inProgressServers`, `notStartedServers`) plus a `servers: ServerStatus[]` array carrying `{name, status, languages, transport}`. The discrete-counter shape is the right call — it keeps `/status` rendering O(1) without iterating `servers` on every redraw, and it lets `formatLspStatusSnapshot` produce the cited summary lines (`enabled, 1/1 ready`, `enabled, 0/1 ready (1 failed)`, etc.) directly. Good schema design.
- Config plumbing at `packages/core/src/config/config.ts:+54` adds `getLspStatusSnapshot()` that delegates to `lspClient.getStatusSnapshot?.()` if the client implements it, otherwise returns a synthetic "statusUnavailable: true" snapshot (verified by the test at `config.test.ts:+60+` covering "should report unavailable LSP status when client lacks a status snapshot API"). The optional-chain `?.` pattern is correct here — older custom clients without the snapshot API stay backward compatible. Also adds `setLspInitializationError` so init failures propagate from `loadCliConfig`'s `try/catch` (`packages/cli/src/config/config.ts:1259-1290`) into the snapshot's `initializationError` field.
- `loadCliConfig` flow at `packages/cli/src/config/config.ts:1259-1290`: `discoverAndPrepare()` → debug-mode snapshot dump → `start()` → debug-mode snapshot dump → `setLspInitializationError(undefined)` (success) or `setLspInitializationError(err)` (catch). The double-snapshot-in-debug pattern is well-tested at `config.test.ts:+723-744`: "should collect LSP status snapshots during debug-mode startup" asserts `getStatusSnapshot` called exactly 2 times, "should not collect during normal startup" asserts 0 times. That's the right discipline — no observable cost when debug is off.
- `/status` rendering at `packages/cli/src/utils/systemInfo.ts:208-252` produces five distinct strings: `disabled` / `enabled, initialization failed: <err>` / `enabled, status unavailable` / `enabled, no servers configured` / `enabled, N/M ready (failed/starting/not started details)`. The detail formatting at `:241-251` filters empty strings via `[…].filter(Boolean)` then joins, so the parens block elides cleanly when all sub-counts are zero. The `addField(fields, 'LSP', info.lspStatus ?? The PR body claims "recent stderr output and exit code/signal" — that lines up with the standard pattern for surfacing "command path is unsafe / command not found / requires trusted workspace / LSP connection closed" errors. ## Concerns

- PR is marked Draft and the body says "Additional tmux E2E smoke with real LSP tool calls will be added before moving this out of Draft." That's the right disposition — without a live LSP roundtrip, the snapshot counters could drift from the actual server state and the status display would lie. The unit tests cover the snapshot *shape* but not the snapshot-vs-reality correspondence.
- Relationship to PR #3615 is awkward: this PR retargeted to `main` directly so its diff is independent, but conceptually it's a follow-up. If #3615 merges first this rebases cleanly; if not, reviewers have to mentally diff against both. Suggest landing #3615 first.
- The `getStatusSnapshot` API on `NativeLspService` (`NativeLspService.ts:+63`) is a synchronous read — fine for the in-process counters, but it implies the service maintains those counters live as servers transition states. The `LspServerManager.ts:+10` change is the only manager-level addition; if state transitions (e.g. server crash mid-session, restart) don't update the counters, the snapshot becomes stale. Worth a state-transition test.
- `formatLspStatusSnapshot` ordering at `:241-251` is `failed → starting → notStarted` — that's the right severity order for an "at-a-glance" line. But "in progress" and "not started" both indicate "not yet ready" and may confuse users — a brief docs note explaining the difference (`inProgressServers` = startup spawned, awaiting initialize response; `notStartedServers` = configured but `start()` not yet called) would help.
- `initializationError` field carries `string` (per `formatLspStatusSnapshot:228`) but the catch at `loadCliConfig:1283` does `err instanceof Error ? err : String(err)` which writes an `Error` object into the field. The display path will toString it correctly via template-literal coercion at `:228` (`enabled, initialization failed: ${snapshot.initializationError}`), but the type signature in `LspStatusSnapshot.initializationError` should be `string | Error` or the assignment should call `.message`/`String(...)` to normalize.

## Verdict

`needs-discussion` — the design is solid (good snapshot schema, clean optional-chain backward compat, the debug-only-double-snapshot discipline is the right cost model, comprehensive docs). Two reasons to hold the merge: (1) PR is explicitly Draft pending tmux E2E smoke, and (2) it's logically a follow-up to #3615 which the body says "should be reviewed/resolved first." Recommend landing #3615 first, then add the state-transition test for `getStatusSnapshot` post-server-crash, then graduate this out of Draft. The `initializationError` type-vs-runtime mismatch (Error object stored in a `string` field) should also be tightened before merge.
