- `parseKagiSearchResults` at `search.go:135-167` defends against `item.URL == ""` and clamps results with `len(results) >= maxResults`. The `min(len(response.Data), maxResults)` capacity hint at `search.go:141` is harmless; it's just a slice pre-allocation. The published-date concatenation at `search.go:147-149` (`snippet + " (" + item.Published + ")"`) is a reasonable enrichment but bakes the format into the snippet — a downstream renderer can't separate them later. Probably fine for an LLM prompt context, where it'll be parsed as text anyway.
- `maybeDelaySearch` rewrite at `search.go:296-310` fixes a real lock-held-during-sleep bug: the old code held `lastSearchMu.Lock()` (defer-unlocked) for the entire `time.Sleep(minGap - elapsed)` window, serializing all callers behind one sleep. The new shape computes `delay := minGap - time.Since(lastSearchTime)` under the lock, releases, sleeps without the lock, then re-acquires to update `lastSearchTime`. This is the correct fix but it has a subtle race: two goroutines that both observe the gap as expired can both proceed and both update `lastSearchTime` to roughly `time.Now()`, giving you back-to-back searches with no gap. For the rate-limit-DDG-friendliness use case that was the *whole point* of the original lock-held design, this regresses. A `sync/atomic` CAS or a token-bucket would preserve correctness; the current rewrite trades correctness for fairness.
- Config schema additions at `internal/config/config.go:+49`: new `WebSearch` block with `Engine`, `KagiAPIKey`, plus a `ResolvedKagiAPIKey(resolver)` accessor for `$KAGI_API_KEY` env-var indirection. The `Engine()` method returns the configured engine or defaults to `"duckduckgo"` (per the README example at line +250-263 showing `search_engine: "kagi"` → `kagi_api_key: "$KAGI_API_KEY"`). Schema bump at `schema.json:+28` matches.

## Concerns

- The `maybeDelaySearch` rewrite (see above) is a fairness-vs-correctness trade. Under burst load (N parallel searches starting from idle), the new code lets all N callers compute `delay <= 0` simultaneously and proceed in parallel, defeating the rate limit that the function exists to enforce. The old code serialized them. If the intent is genuinely "minimum gap between *any* two searches", the new code is wrong. If the intent is "rough best-effort backoff", it's fine but the function name lies. Worth flagging to the maintainer.
- `searchKagi` at `search.go:91-133` doesn't call `maybeDelaySearch` at all (only `searchDuckDuckGo` does, implicitly via the existing call site). That's defensible — Kagi has a paid API and doesn't need scraper-friendliness — but it means the `SearchEngine` per-call override silently changes rate-limit behavior. A one-line comment in `searchKagi` explaining the asymmetry would help the next maintainer.
- API-key handling: `kagi_api_key_input.go:60-62` reads `cfg.Tools.WebSearch.KagiAPIKey` directly into the textinput, which means a key configured as `$KAGI_API_KEY` (env reference) will display as the literal string `$KAGI_API_KEY` in the dialog. That's probably what the user wants (they can see "yes, I configured the env reference"), but if they edit-and-save it'll round-trip as a literal env-ref string rather than the resolved value — confirming `ActionSaveKagiAPIKey` writes to config without resolving would be good.
- Test coverage at `search_test.go:+183` covers `parseKagiSearchResults` happy path (3 results), URL-empty filter, `t != 0` skip, and max-results clamp. Missing: integration test for the 401/402/403 error-mapping branches in `searchKagi` (the most likely failure mode in production).
- README addition at `:247-263` shows `kagi_api_key: "$KAGI_API_KEY"` env-ref but doesn't explain that the literal-string fallback (raw key) is also supported. Minor docs nit.
- "PR written under my supervision and testing by GPT-5.5" is in the body — fine signal, but the maintainer may want to confirm no GPT-induced subtle bug (especially the rate-limit rewrite) snuck through.

## Verdict

`merge-after-nits` — the Kagi backend wiring is well-structured (clean separation of API client, parser, config schema, dialog, action types), tests cover the parser thoroughly, and the per-call `search_engine` override is a nice touch. The blocker before merge is the `maybeDelaySearch` rewrite — confirm the intended semantics with the maintainer; if "min gap between any two searches" is the contract, the rewrite needs a CAS or token-bucket. Smaller follow-ups: error-branch tests for `searchKagi`, API-key dialog round-trip semantics, and a one-line comment on the DDG-vs-Kagi rate-limit asymmetry.
