## Specifics

- The runner change at `runner.ts:18-25,60-62,150-167` adds an explicit `cancelled: Deferred.Deferred<void>` to `ShellHandle` so cancellation intent is observable independently of fiber-interrupt status. `stopShell` now `Deferred.succeed(shell.cancelled, undefined)` *before* `Fiber.interrupt(shell.fiber)` (`runner.ts:107-111`), so the wait loop at `runner.ts:155-163` can distinguish "user cancelled" from "fiber crashed with interrupt-only cause" — the new gate `(yield* Deferred.isDone(cancelled)) || Cause.hasInterruptsOnly(exit.cause)` runs `onInterrupt` even when the child exits with a *signal failure* (the explicit before-and-after in the PR summary). This is the actual fix: previously a SIGTERM-on-cancel could surface as `Cause.fail(SignalError)` rather than an interrupt cause and skip `onInterrupt`.
- The `awaitDone` helper at `runner.ts:63-64` factors the `Cancelled → onInterrupt` projection out of the per-state branches in `ensureRunning` and applies it uniformly to `Running`/`ShellThenRun`/`Idle` (`runner.ts:120,128,134`). The old `.pipe(Effect.catch(...))` at the bottom of `ensureRunning` is removed — that catch-all only covered the *outer* awaited deferred, so a `Cancelled` raised by the new shell-await branch wouldn't have routed through `onInterrupt` consistently. Centralizing on `awaitDone` is the right cleanup.
- In `prompt.ts:720-863` the `shellImpl` body is rewrapped in `Effect.uninterruptibleMask((restore) => …)` and the read-only setup work (`InstanceState.context`, agent lookup, message/part allocation, `sessions.updateMessage`/`updatePart`) is hoisted into an inner `Effect.gen` that runs *before* anything interruptible. That ordering is what the PR claims as "ensure shell command messages and tool parts are finalized before cancellation fallback reads the last assistant result" — without it, a fast cancel would race the assistant `MessageV2`/`ToolPart` writes and the fallback's `lastAssistant` lookup would see no record at all.
- The reorder also matters for the third bullet ("keep shell progress/final output writes ordered so completed tool state cannot be overwritten by stale running updates"): putting the assistant message + initial `running` `ToolPart` writes *inside* `uninterruptibleMask` means the `state.status: "running"` snapshot is always durably persisted before `restore(...)` opens the cancellation window, so the post-cancel `finish` block can't lose to an in-flight earlier write that lands later.

## Concerns

- The new branch at `runner.ts:155-160` does `if (onInterrupt) return yield* onInterrupt; return yield* Effect.die(new Cancelled())`. Dying inside a fiber-await projection is an escalation from the previous code path that just `Effect.failCause`'d the original cause — callers without an `onInterrupt` will now see a defect rather than a typed failure. Worth documenting that `make(...)` callers without `onInterrupt` get a die on cancel; the type signature still says `E | Cancelled` which is no longer fully accurate when `onInterrupt` is absent.
- The new `cancelled` Deferred is created per shell handle but never explicitly cleaned up — `Deferred.make<void>()` allocates a small structure and the closure holds it for the shell's lifetime. Fine in isolation, but worth confirming `finishShell` releases the handle so the deferred is GC'd along with the fiber.
- `prompt.ts` is +129/−128 dominated by re-indentation under the new `uninterruptibleMask`. Reviewers will want a no-rename diff to confirm no logic outside the hoist+wrap actually moved; the test list in the PR body covers the five named behaviors but a `--ignore-all-space` sanity check would be cheap insurance.
- Test coverage cited is comprehensive ("cancel persists aborted shell result when shell ignores TERM", "cancel interrupts shell and resolves cleanly", "cancel finalizes interrupted bash tool output through normal truncation", "loop waits while shell runs and starts after shell exits", "shell completion resumes queued loop callers") and exercises both the `cancelled` deferred path and the message-ordering invariant. Good behavior coverage; no new test file added but the existing `test/session/prompt.test.ts` is already the right home.
