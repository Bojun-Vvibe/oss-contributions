## Verdict: merge-after-nits

## Rationale

- **Right diagnosis: ACP's `Session` was bypassing the auto-compression path.** Per the PR body: "ACP `Session` sent messages by calling `GeminiChat.sendMessageStream()` directly. That bypassed `GeminiClient.sendMessageStream()`, where automatic compression runs on the CLI path before model requests." This is the canonical "two surfaces share a backend, but only one of them runs the pre-flight protection" bug — same shape as several drip-W9/W13 bugs covered earlier in the program (compaction, processor registration, retry classification). The CLI path called the compression-wrapping `GeminiClient.sendMessageStream`; the ACP path called the lower-level `GeminiChat.sendMessageStream` and silently lost the compression-trigger. Result: long ACP sessions could grow past the model's context window without ever auto-compressing, leading to hard 400-class errors instead of graceful summarization.
- **The fix correctly extracts a single private helper for the wrapping.** `Session.ts:782-802` adds `#sendMessageStreamWithAutoCompression(promptId, message, abortSignal)` that: (1) gets `geminiClient`; (2) `await geminiClient.tryCompressChat(promptId, false, abortSignal)` runs the compression pre-flight; (3) re-fetches `geminiClient.getChat()` and calls `sendMessageStream` *on the post-compression chat instance*. The re-fetch at `:792` is load-bearing because `tryCompressChat` may *replace* the chat instance (per the third test case at `:339-368` which confirms this) — caching the chat reference before compression would have caused a writes-to-stale-instance bug. This is the correct contract for "compress, then resume on whatever chat is current."
- **The deletion of cached `chat` references is the right structural cleanup.** The diff removes `const chat = this.config.getGeminiClient()!.getChat()` at `:294-296` (the comment "Always fetch the current chat from GeminiClient so that /clear's resetChat() (which replaces the chat instance) is reflected here" was already acknowledging the same instance-replacement risk for `/clear`). It then replaces every `chat.X` call with either `this.#getCurrentChat().X` (for non-send operations) or the new wrapper helper (for sends). The `#getCurrentChat()` helper at `:779-781` makes the "always re-fetch" discipline explicit and grep-able. Removing the `chat: GeminiChat` parameter from `#handleStopHookLoop` at `:556-562` keeps the stale-reference risk from leaking into the stop-hook path.
- **Test coverage matches the four critical cells.** `Session.test.ts:308-440` adds an `auto-compress` describe block covering: (a) compression runs *before* `sendMessageStream` for ACP prompts (with `invocationCallOrder` assertion at `:328-330` pinning the ordering — this is the load-bearing test because order, not just call-presence, is the contract), (b) post-compression chat instance replacement is honored (`:339-368` mocks `tryCompressChat` to swap the chat ref and asserts the *new* chat's `sendMessageStream` is called, not the old one), (c) tool-response follow-up sends *also* get compression (`:370-419` — important because long tool-call/response cycles are the most common context-runaway path), (d) slash commands handled without a model send do *not* trigger compression (`:421-439` — the negative cell that pins "compression only fires when a real send is about to happen"). All four cells map to a distinct failure mode the bug could re-introduce.

## Nits / follow-ups

- **The `tryCompressChat(promptId, false, abortSignal)` call hard-codes `force=false`.** That matches the CLI path's default, but the *literal `false` at `:790` is not commented* — a future reader has to chase the `tryCompressChat` signature to understand what the boolean means. One-line `// force = false → only compress when threshold exceeded` above the call would close that gap.
- **`#getCurrentChat()` at `:779-781` and the inline `geminiClient.getChat()` re-fetch at `:792` are now two ways of saying the same thing.** Worth using `this.#getCurrentChat()` inside the wrapper too for consistency — or alternately, inline `#getCurrentChat()`'s body if it only has one call site that isn't in the wrapper. Pick one.
- **The negative-cell test at `:421-439` (`does not auto-compress slash commands handled without a model send`) is the highest-leverage test in the suite** — it pins the "no-op when no send" contract that prevents future refactors from accidentally compressing on every prompt path including no-op ones. Worth a comment in the test body noting "regression guard for #3652 — compression must not fire when slash command short-circuits the send."
- **The `#sendMessageStreamWithAutoCompression` wrapper has no explicit return-type annotation** at `:786`. TypeScript will infer it, but the inferred type is `Promise<AsyncGenerator<...>>` which is non-trivial — explicit return type aids IDE hover and prevents silent contract drift if `sendMessageStream`'s return shape ever changes.
- **The PR doesn't add a test that asserts `tryCompressChat` rejection (e.g., AbortError) propagates correctly** — i.e., if compression is interrupted mid-flight by the abort signal, the wrapper should re-throw rather than fall through to `sendMessageStream`. The current implementation does the right thing (await throws → wrapper throws), but pinning that with a test would prevent a future "swallow compression error and continue" regression.

## What I learned

The "two surfaces, one shared backend, one missing pre-flight" bug shape is a structural recurring problem in any agent CLI that grows a second invocation surface (ACP, MCP, raw HTTP API, embedded SDK). The first surface is built with all the protections inline; the second surface is wired up by someone who reads the *innermost* method and calls it directly, missing the protection layer that wraps it. The right structural fix isn't a one-off "add the protection here too" patch — it's to make the inner method *unsafe to call directly* (e.g., make `GeminiChat.sendMessageStream` private and force everyone through `GeminiClient.sendMessageStream`), or to extract the protection into a wrapper that both surfaces compose. This PR does (b) — the new `#sendMessageStreamWithAutoCompression` wrapper is the composable protection, and any future third surface (e.g., a new MCP transport) can either reuse the same wrapper or re-derive it from the same primitives. The test discipline that proves the fix is "ordered invocation" (compression *before* send) plus "stale-reference safety" (re-fetch chat after compression) plus the negative cell (no-op when no send) — exactly what this PR delivers.
