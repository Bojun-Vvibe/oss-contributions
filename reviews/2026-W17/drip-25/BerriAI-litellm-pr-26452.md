## Summary

Three-line fix to `.circleci/config.yml` that restores the
"Rerun failed tests" capability for three jobs:
`local_testing_part1`, `local_testing_part2`, and
`litellm_router_testing`.

The `circleci-tests-plugin-cli` plugin, on rerun, reads the
JUnit XML `classname` attribute (Python's dot-notation module
path, e.g. `tests.local_testing.test_router`) and pipes it
into `xargs pytest`. pytest receives that as a module path
string, can't resolve it, collects 0 items, and exits 123.

The fix inserts an `awk` preprocessor between
`circleci tests run` and `xargs pytest`:

```bash
awk '/\.py/ {print; next} {gsub(/\./, "/"); print $0 ".py"}'
```

- Lines containing `.py` (i.e. real file paths from the initial
  test run) pass through unchanged.
- Lines without `.py` (i.e. dot-notation classnames from the
  rerun) get `.` → `/` substituted and `.py` appended, so
  `tests.local_testing.test_router` becomes
  `tests/local_testing/test_router.py`.

## Key change

The diff in each of the three jobs is identical:

```diff
-              --command="xargs uv run --no-sync python -m pytest \
+              --command="awk '/\\.py/ {print; next} {gsub(/\\./, \"/\"); print $0 \".py\"}' | xargs uv run --no-sync python -m pytest \
```

The PR body shows the verification:

```
$ printf 'tests/local_testing/test_router.py\ntests.local_testing.test_router\n' \
    | awk '/\.py/ {print; next} {gsub(/\./, "/"); print $0 ".py"}'
tests/local_testing/test_router.py
tests/local_testing/test_router.py
```

Both inputs normalise to the same path. Initial-run path is
preserved; rerun path is corrected.

## Concerns

1. **`gsub(/\./, "/")` is too aggressive for any path with
   embedded dots.**
   Today, all the JUnit classnames generated by these jobs are
   under `tests/local_testing/` and don't contain dots in
   directory names. But the fix would break for any future test
   path that uses a directory name with a dot (e.g. a hypothetical
   `tests.foo.bar.test_baz` where `foo.bar` is the actual
   directory) — `gsub` rewrites *every* `.` to `/`. That'd
   produce `tests/foo/bar/test_baz.py` instead of
   `tests/foo.bar/test_baz.py`. Unlikely in practice for this
   project, but worth a comment in the YAML noting the
   assumption.

2. **The fix relies on the JUnit XML `classname` containing a
   pure dot path.**
   pytest's JUnit XML emits `classname="tests.local_testing.
   test_router"` for module-level tests, but `classname=
   "tests.local_testing.test_router.TestClassName"` for
   class-grouped tests. Under the awk rewrite, that becomes
   `tests/local_testing/test_router/TestClassName.py` — which
   doesn't exist, so pytest collects 0 again and the rerun
   silently regresses for any class-grouped test. Worth
   either:
   - asserting that the affected jobs only use module-level
     `def test_*` (no class-grouped tests), or
   - tightening the awk regex to handle the
     `Module.Class` shape — split on `.`, treat the last
     `Class`-cased segment as a class name and drop it before
     appending `.py`, or rewrite in a small Python preprocessor
     where the parsing is less fragile.

3. **No test for the test fix.**
   Inherently hard to test a CircleCI-config-only change in
   isolation. The right evidence here is a "before/after"
   screenshot or paste of the actual rerun behaviour from a
   CI run on a fork — the PR body has the awk one-liner
   verification but not the actual rerun output. Worth asking
   for one before merge so the reviewer doesn't have to take
   "this fixes it" on faith.

4. **One-line change, three identical copies.**
   The same awk preprocessor is duplicated across three job
   definitions. If any future maintainer needs to refine the
   regex (per concern #2), they'll need to update all three.
   Worth extracting the preprocessor into a shared CircleCI
   command so the next refinement is a one-place edit. Not
   blocking, but the duplication makes the YAML less
   maintainable.

## Verdict

`merge-after-nits` — the immediate problem (rerun collects 0
items, exits 123) is real, the diagnosis (classname dot-path
vs file-path mismatch) is correct, and the awk fix solves it
for the current set of tests. Two pre-merge asks: confirm the
affected jobs don't have class-grouped tests (or harden the
regex if they do), and paste a real rerun-from-CI output as
evidence. Post-merge follow-up: extract the awk preprocessor
into a shared CircleCI command so the three copies don't
drift.

## What I learned

JUnit XML `classname` is one of those interchange-format
fields with no formal spec — different runners populate it
differently (pytest uses module dot-paths; some other
runners use the actual class hierarchy; ant-style runners
use the Java FQN). Any pipeline that round-trips through
`classname` and expects to reconstruct the file path is
implicitly assuming the producer's convention. The defensive
move is to either (a) get the test list from the producer in
the format you need (pytest's `--collect-only -q` outputs
file paths directly, no classname round-trip), or (b)
treat the classname as opaque and resolve it through a
lookup table that the producer also emits. The awk-based
text rewrite this PR uses is the right fix for a specific
bug shape, but it doesn't generalise — and as soon as the
test layout changes, it'll break silently. Same lesson as
"don't grep over JSON" — fragile-string-rewrite over a
loosely-specified data format is a debt that compounds.
