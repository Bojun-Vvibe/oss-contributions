# QwenLM/qwen-code #3694 — test(cli): remove 8 flaky TUI input tests surfaced by CI history mining

- PR: https://github.com/QwenLM/qwen-code/pull/3694
- Head SHA: `f3af25b365106a22e8a76c1f6cd603d0d5718ec6`
- Diff: 0 additions, 477 deletions across 3 test files — `AuthDialog.test.tsx`, `InputPrompt.test.tsx`, `StandaloneSessionPicker.test.tsx`

## Verdict: merge-after-nits

## Rationale

- **Methodology is correct.** PR body documents the diagnostic process explicitly: mining the last 60 failed `Qwen Code CI` runs and observing each removed test fails on **2-10 distinct unrelated SHAs** with low fails-per-SHA. That is the textbook signature of a timing flake (real bugs would be reproducible on a single SHA). The top offender — `AuthDialog > should trigger OpenRouter OAuth from API key options` — failed on **10 distinct SHAs across 3 matrix legs**, which is a strong signal it's an environment issue not a code issue.
- **Common shape across all 8 removed tests is the same root cause.** Every removed test uses `stdin.write(...)` followed by a fixed `setTimeout(50-350ms)` (the `wait()` helper) while waiting for React state to settle. This is the textbook "race fixed-delay against real async event" anti-pattern. The repo already has `itWhenTuiInputReliable` (`AuthDialog.test.tsx:728`) which guards against the same flake on Windows + CI Node v20 — the PR's claim is that the guard isn't broad enough now that the flake reproduces on Ubuntu 24.x as well, and `InputPrompt.test.tsx` / `StandaloneSessionPicker.test.tsx` have no equivalent guard at all.
- **Coverage-loss is bounded.** PR body explicitly names what survives: "command-completion suppression in InputPrompt, in-preview Enter-fires-onSelect in SessionPicker, and the surviving wizard review-screen test." Looking at the surviving test in `AuthDialog.test.tsx:765-819` (the diff hunk between the two big deletion blocks shows `'shows review screen with JSON after entering model IDs'` survives) — so the wizard's load-bearing review/save path is still pinned, just not the intermediate-step navigation cells.
- **Three of the deleted tests had genuine value, but flake-immune alternatives exist.**
  - `AuthDialog > should trigger OpenRouter OAuth from API key options` (`:723-779` deleted): tested that pressing Enter twice from default selection fires `handleOpenRouterSubmit`. This is the only test for that integration. A flake-immune replacement: spy on `handleOpenRouterSubmit`, dispatch the menu-item-selected event directly via the component's exposed action handler instead of going through `stdin`. Worth a follow-up issue.
  - `InputPrompt > 'accepts and submits the prompt suggestion on Enter when the buffer is empty'` (`:232-246` deleted): the comment in the deleted test (`:241-244`) explicitly said "Enter path must NOT call buffer.insert — it passes text directly to handleSubmitAndClear. Calling insert would re-fill the buffer after it was already cleared (the microtask race bug)." That's a regression-anchor for a *real bug* (the microtask race), not just a UX assertion. Removing this loses the regression pin. Should be replaced with a non-stdin test asserting the same invariant.
  - `StandaloneSessionPicker > opens preview on Space and closes on Esc` (`:699-732` deleted): tested the Space-to-preview / Esc-to-close cell. The PR's "in-preview Enter-fires-onSelect" claim covers a different cell (Enter behavior *inside* preview), not the open/close transition itself.

## Blockers / nits

- **Add a `// TODO(flake-replacement): #<followup-issue>` marker** at the top of each affected `describe` block naming the issue tracking the non-stdin replacement test. Without this, the deletion is a permanent coverage loss; with it, this is a load-shedding move with a clear "come back to this" signal.
- **The `InputPrompt` microtask-race regression test is the only one I'd push back on as a flat deletion.** Per the deleted comment at `:241-244` ("the microtask race bug"), this test pinned a real regression — the right move is to convert it to a unit test on the buffer/handler functions directly (not the React component) so it doesn't need stdin. A 2-minute rewrite, but if it's deferred, the regression can silently come back.
- **PR body says "the existing `itWhenTuiInputReliable` guard … isn't broad enough — flake now happens on Ubuntu 24.x as well."** The minimum-cost alternative to deletion is widening the guard's `isUnreliableTuiInputEnvironment` predicate at `AuthDialog.test.tsx:725-727` to also detect Ubuntu 24.x CI runners, then converting the deleted tests to `itWhenTuiInputReliable(...)` (which already exists for the wizard tests they're sibling to). That preserves coverage on environments where they pass while skipping on the flaky ones. Worth at least mentioning in the PR body why "delete" was preferred over "expand the skip predicate."
- The `AuthDialog Custom API Key Wizard` block deletes 5 of its 7 tests. The remaining 2 (`shows review screen with JSON after entering model IDs` and one more) cover the happy path but not the intermediate navigation cells. A reviewer should confirm those intermediate cells are exercised by an e2e or integration test elsewhere — if not, the wizard's step-by-step navigation is now untested.
- The `wait` helper is removed at `AuthDialog.test.tsx:732` (the `const wait = (ms = 50) => ...` line in the `Custom API Key Wizard` describe) but `pressEnterAndWaitFor` and `navigateToCustomAdvancedConfig` (which surviving tests call) likely still use it via import — verify the surviving wizard test still compiles/runs.

## Why "merge-after-nits" not "merge-as-is"

Methodology and bias toward CI green are right. The structural concern is that 1-2 of the 8 removed tests pinned real regressions (not just UX cells), and deletion-without-replacement permanently weakens the regression net. A 30-min follow-up to convert the `InputPrompt` microtask-race test to a non-stdin unit test would close the loop.
