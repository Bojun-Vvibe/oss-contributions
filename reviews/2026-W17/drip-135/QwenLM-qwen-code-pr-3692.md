## Verdict: merge-as-is

## Rationale

- **Diagnosis is exactly right.** `SessionService.countSessionMessages` at `sessionService.ts:323-336` (pre-diff) had its own readline loop with inline `JSON.parse(trimmed)` inside `try/catch { continue }` — meaning any line that wasn't perfect-JSON was silently dropped. After PR #3606's read-side recovery introduced `parseLineTolerant` which can recover both records from a `}{`-glued line (the canonical corruption shape from interrupted appends), `countSessionMessages` was left out — so the listSessions count was lower than the resume-time record count, the exact mismatch flagged in #3681 Item 1.
- **Right shape for the fix: surface the existing helper, route the existing reader through it.** `jsonl-utils.ts:115` widens `parseLineTolerant` from `function` (private) to `export function` and the docstring at `:109-114` updates the framing from "Mirrors the silent skip in `countSessionMessages`" (which was the *bug*) to "Use this from any streaming reader that walks JSONL line-by-line and wants the same recovery semantics as `read()` / `readLines()`." That docstring rewording is load-bearing — it announces the helper as the canonical entry point for any future streaming reader so this class of bug doesn't recur.
- **The replacement at `sessionService.ts:329-335` preserves all original semantics.** Before: `const record = JSON.parse(trimmed) as ChatRecord; if (record.type === 'user' || record.type === 'assistant') uniqueUuids.add(record.uuid)`. After: `for (const record of jsonl.parseLineTolerant<ChatRecord>(trimmed, filePath)) { if (record.type === 'user' || record.type === 'assistant') uniqueUuids.add(record.uuid) }`. The `user`/`assistant` filter is unchanged, the `uniqueUuids: Set<string>` semantics are unchanged, the surrounding `for await (const line of rl)` stream is unchanged. The only behavior diff is "well-formed line yields 1 record, `}{`-glued line yields 2 records" instead of "well-formed yields 1, glued yields 0" — exactly the fix.
- **The "why streaming, not `jsonl.read() + dedupe`" rationale in the PR body is correct.** `countSessionMessages` is on the `listSessions` hot path (default page size 20), and active sessions can hold thousands of records. The issue's alternate suggestion — call `jsonl.read()` and dedupe on the full materialized array — would change the function from O(uuid-set memory) to O(record-count memory) for every listed session. The author explicitly rejects that and preserves the streaming readline shape, which is the right perf-vs-correctness call.
- **3 unit tests for the newly-exported helper** at `jsonl-utils.test.ts:84-104` cover the load-bearing cells: well-formed line → `[{ a: 1 }]` (the wrap-in-array semantics for the single-success case), `}{`-glued line → `[{ uuid: 'a' }, { uuid: 'b' }]` (the recovery case from #3606), unrecoverable input `'not-json'` → `[]` (the silent-no-crash cell). The third test is the most important regression anchor — without it, a future change to the recovery logic could start throwing on bad input and the silent-skip semantic would silently become a hard crash.
- **`Item 2 from #3681 is intentionally deferred per the issue's own guidance.`** PR body explicitly names the scope cut: write-side durability (`writeLine` / `writeLineSync` lack `fsync` / atomic-rename) is out of scope. That's the right call — the issue itself flagged Item 2 as "Open for discussion — no immediate action required." Single-purpose PR with clean scope boundary.
- **Imports are minimal and correct.** `jsonl-utils.test.ts:13-16` adds `parseLineTolerant` to the existing import group from `./jsonl-utils.js` — no new file, no new dependency. `sessionService.ts` already imports the `jsonl` namespace (the diff doesn't show an import-line change, which is correct because `jsonl.parseLineTolerant` resolves through the existing namespace import).

## Why merge-as-is

This is a textbook "missed-symmetry-after-prior-PR" fix: PR #3606 introduced `parseLineTolerant` for `read()` / `readLines()` but `countSessionMessages` was the third streaming reader in the codebase that should have been routed through it. The fix is the minimum correct diff (export the helper, route the existing reader, add 3 unit tests for the newly-public surface, update the helper's docstring to announce the canonical entry point). Memory profile is preserved (streaming, not full-load). Scope is intentionally tight (Item 2 deferred per issue guidance). The PR body's "Why streaming, not `jsonl.read()` + dedupe" section is exactly the kind of prior-art / alternative-considered narrative reviewers want.
