## Summary

Hardens the insight-generation pipeline against three real failure modes: (1) malformed LLM responses where facet fields come back `null` or with unexpected types/enum values cause aggregation crashes, (2) all-empty qualitative sections render as headers + nav links to nothing in the report UI, and (3) cached facets reused from previous runs propagate any malformed shape because no validation runs at reuse time. The fix is a normalization layer that coerces every facet field through type-safe defaults (invalid enums fall back to a documented default like `unclear_from_transcript` / `moderately_helpful` / `single_task`, null objects become `{}`, strings get trimmed) at three entry points: the LLM-response handler, the cached-facet reuse path, and the count-aggregator that previously assumed dict shapes.

## Verdict: `merge-after-nits`

The defensive-normalization shape is right, the test coverage in this PR is unusually thorough (42 tests on one file in the diff), and the UI hide-when-empty logic prevents the navigate-to-nowhere link bug. Three concerns about the normalization defaults themselves and one about cache invalidation.

## Nits / concerns

1. **Default-to-middle-of-distribution biases the data.** `Qwen_helpfulness: 'invalid'` → `'moderately_helpful'` and `outcome: null` → `'unclear_from_transcript'` are very different choices: the second is honest ("we don't know"), the first pretends to know. Recommend either:
   - Add a `'unknown'` / `'normalization_failed'` enum variant for `Qwen_helpfulness` and surface the count of normalized values in the report metadata so consumers can see how much of the data is synthetic, OR
   - Drop the session entirely when `Qwen_helpfulness` is invalid, with a `verbose_logger.warn` so the silent-discard rate is observable.
2. **`Qwen_helpfulness` field name** appears in tests at L394, L685, L716. (`Qwen` is presumably the prefix this project uses for its own scoring conventions; the codebase already establishes that naming, and this PR follows it consistently.) Just confirm the prefix is the canonical project term — if it's the *AI model* name this'll need a rename later.
3. **No log/telemetry on normalization fallbacks.** Every time a fallback fires, that's a signal that either the LLM contract has drifted or the prompt needs strengthening. Without observability, the normalization layer becomes a place where contract drift accumulates silently. A counter incremented per fallback (visible in debug logs) would let maintainers spot rising normalization rates as a leading indicator.
4. **Cache validation is reactive, not preventive.** This PR validates *on read*. A complementary fix would validate *on write* — refuse to cache a malformed facet in the first place. That stops the problem at the source and avoids re-validating the same bad data on every read forever. Worth a follow-up.
5. **42 tests for one source file is excellent**, but `should normalize cached facets before reusing them` is truncated in the diff at L1490 — the assertion shape isn't fully visible from this view. Verify the test actually asserts the *normalized* shape on the cache-hit path and not just that the cache returned something.
6. The PR description has a few rendered placeholders (`(, , , )`, `( function`) where backtick-quoted method names didn't survive markdown processing. Worth cleaning up before merge so the PR description is searchable later.
7. The body of the PR mentions "Tests" — the new file `DataProcessor.test.ts` reports `42 passed` in the body. Worth confirming `npm run test:coverage` shows the normalization functions themselves are at 100% line coverage, since they are now the trust boundary for downstream insight generation.

## What I learned

Normalization layers at trust boundaries (LLM responses, cache reads, untyped JSON) are valuable, but the *choice of default* for "this field is malformed" is a design decision with downstream consequences, not just a defensive shrug. Defaulting numeric or categorical fields to the middle of their distribution looks safe but biases aggregate statistics in invisible ways; defaulting to an explicit "unknown" sentinel keeps the data honest at the cost of forcing every consumer to handle the sentinel. This PR makes the right structural move (centralize the normalization, test the fallbacks, validate on cache reuse) but underweights the second decision — the specific defaults — by treating them as mechanical fallbacks rather than as policy.
