## Summary

Closes #19515 by introducing first-class multi-root workspaces — a single session can now include multiple folders without relying on filesystem symlinks. The PR adds: persisted `.code-workspace` files (copied into OpenCode's data directory before being managed, so external rename/delete don't mutate user-owned files), workspace CRUD HTTP routes, app + TUI dialogs for create/open, session ↔ workspace association via a new join table, SDK header propagation for the active workspace, instance root expansion across multiple roots, and LSP `workspaceFolders` support so language servers see all roots. A follow-up commit extends the feature to instruction discovery, file indexing, file watching, and TUI auth header propagation across all workspace roots.

## Verdict: `needs-discussion`

This is a major feature with the right shape (`.code-workspace` JSON as the persistence format, copy-on-import to avoid mutating user files, LSP `workspaceFolders` integration) but at +7776 / 83 files it's also a one-shot landing of a feature that touches the migration layer, server routes, two TUIs (app + cli), the LSP integration, the SDK, and 18 i18n files. Several architectural decisions deserve explicit review/discussion before this lands as a single squash.

## Concerns / discussion points

1. **Squash size.** 7776 added lines in one PR is hard to bisect later. A more reviewable shape would be: PR-1 schema + migrations only (~3000 lines incl. snapshot JSON); PR-2 server routes + SDK regen on top of PR-1; PR-3 TUI dialog + i18n; PR-4 app dialog + i18n; PR-5 LSP `workspaceFolders` + tests; PR-6 docs. That's six PRs each independently reviewable and revertable, vs. one giant squash where a single regression on the LSP side forces reverting everything including the user-facing dialogs.
2. **External `.code-workspace` file copy-on-import.** The PR description says "existing external `.code-workspace` files are copied into OpenCode's data directory before OpenCode manages them, so rename/delete actions do not mutate user-owned files." That's the right policy but raises three questions: (a) what happens when the user edits the *original* file after import (does OpenCode re-import? Show a "diverged" warning? Silently use the imported copy?); (b) what's the de-dup rule if two users on the same machine import the same file (one shared imported copy, or two?); (c) where is the imported copy stored exactly, and is it gitignored if the data dir lives inside a project? Worth pinning these as ADRs in `docs/workspace.mdx`.
3. **Session ↔ workspace FK semantics.** The `add_session_multi_root_workspace_id` migration name says "FK on sessions". On `DELETE workspace`, what happens to sessions pointing to it? `CASCADE` (delete the sessions, surprising users), `SET NULL` (orphan the sessions, confusing), or `RESTRICT` (refuse to delete the workspace, requires user to clean up sessions first)? The migration JSON should make this explicit and the docs should match.
4. **LSP `workspaceFolders`** is the integration point most likely to surface bugs — many language servers have known half-implemented multi-folder support, and `gopls` / `rust-analyzer` / `typescript-language-server` each have different quirks (rust-analyzer wants a `Cargo.toml` per root; gopls wants a `go.mod` per root; tsserver effectively folds everything into the largest root). The new test at `test/lsp/multi-root.test.ts` (+130) probably can't cover all three. Worth a dedicated bug-bash on a test workspace with mixed Go + TS + Rust roots before this lands as default.
5. **Workspace-context header propagation** — the SDK now sends a workspace identifier with each request, but what's the failure mode when the header is missing on an old client? Should the server reject (breaking old clients) or fall back to "default workspace" (introducing a possible cross-workspace leak if the default leaks across users)? Pin the answer in `multi-root-middleware.test.ts`.
6. **No e2e in this PR diff list except `packages/app/e2e/workspace.spec.ts`** — that's a single e2e spec. Worth adding a CLI-side e2e that exercises create-workspace → open-in-tui → run-tool-against-multi-root → close → reopen.
7. **i18n placeholder verification.** Worth running a quick grep over the 17 non-EN locale files to confirm the new keys actually have translated values (not literal English copy or empty strings) — it's common for big-feature PRs to land with EN-only and a "translations PR" to follow, but the docs and dialog labels will be confusing for non-EN users in the meantime.

## What I learned

Multi-root workspace support is one of those features that looks small ("just let users add a second folder") and turns out to touch every layer that ever assumed a single CWD: the bootstrap, the LSP, file watchers, the indexer, the auth-header propagation, the database schema, the SDK, and 18 translation files. The right shape for landing a feature like this is a stack of small PRs — each one independently reviewable and revertable — rather than a single 7776-line squash. The architectural decisions in this PR (`.code-workspace` JSON, copy-on-import, LSP `workspaceFolders`) all look right, but the *delivery shape* makes it hard to get high-confidence review on any one of them.
