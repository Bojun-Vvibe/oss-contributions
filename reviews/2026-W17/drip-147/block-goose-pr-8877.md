This is exemplary PR ergonomics for a 40-file change. Reviewers should follow zone order, not file order.
- **Zone 1 — ACP contract + proxy** (`crates/goose/src/acp/mcp_app_proxy.rs:1-229`, `crates/goose/src/acp/templates/mcp_app_proxy.html:1-141`, `crates/goose-sdk/src/custom_requests.rs:1-25`, `crates/goose/src/acp/server.rs:+99/−3`, `crates/goose/src/acp/transport/mod.rs:+2/−1`): introduces authenticated proxy + guest-document routes with **bounded in-memory HTML storage and CSP shaping** for inline rendering. ## Risks / nits

- 40-file PR with 2120 additions is at the upper bound of what's reviewable as a single unit. Author has done the work to make it tractable (zone guide + commit guide + per-file purpose section in PR body), but reviewers should still split-review across at least two sittings.
- `mcp_app_proxy.rs:1-229` Rust file — verify proxy-route auth tests exist; the per-process-secret design is correctly chosen but worthless if not actually enforced on every route. Want to see `cargo test -p goose acp::mcp_app_proxy` cells covering 401 path, document-cap enforcement, CSP-header presence on every response.
- CSP shaping for inline app HTML: which directives? `default-src 'none'; script-src 'unsafe-inline'` (likely necessary for MCP apps to function but undermines a lot of the protection); or is there an allowlist for embedded scripts? The CSP policy text is the load-bearing artifact here — needs to be in the PR description verbatim.
- `_goose/tool/call` nested tool call from inside an MCP app: does this respect Goose's outer permission policy, or does an MCP app get blanket access to call any tool the host knows about? If the latter, this is a privilege-escalation primitive (MCP app supplies a payload that triggers `_goose/tool/call shell.execute "rm -rf"` ). Want explicit policy-gate confirmation.
- File-size budget adjustments in `ui/goose2/scripts/check-file-sizes.mjs:+6/−1` — these are intentional per the PR. Worth a comment in the script naming *which* files got bigger and why, so future readers don't mistake the bumped budget for a slow leak.
- Generated SDK files (`ui/sdk/src/generated/{client,index,types,zod}.gen.ts`) regenerated. Reviewer should not hand-review these — they're outputs of `acp-schema.json` codegen. Verify the schema source-of-truth changed in the same commit and the generated outputs match a fresh regen.
- 4 commits is a clean stack for a feature this size. Recommend reviewer review commit-by-commit per the author's guide rather than file-by-file.
- Manual-smoke instructions (5 steps) are concrete (specifically calls out MCP AppBench as the test app); reviewer should actually run them, not just read them.

## Verdict: `needs-discussion`

**Rationale:** This is a 40-file Step-3-of-a-multi-step renderer migration with **strong PR ergonomics** (zone guide, commit guide, per-file purpose docs, manual-smoke checklist) but **two unresolved load-bearing security questions** that can't be answered from the diff alone: (1) the **CSP policy text** for inline MCP-app HTML rendering needs to be in the PR description verbatim — `default-src 'none'; script-src 'unsafe-inline'` makes this safe-ish, anything looser is a vector; and (2) the **`_goose/tool/call` nested tool call** path needs explicit confirmation that it respects the host's outer permission policy, not just whatever permissions the MCP app already has. If an MCP app payload can issue `_goose/tool/call shell.execute "rm -rf ~"` because nested tool calls bypass the outer permission gate, this is a serious privilege-escalation primitive. Both questions are answerable in a follow-up comment thread; once answered, this is `merge-after-nits` (the nits being: per-process-secret leak audit, proxy-route auth tests in Rust, CSP-header-on-every-response test, named file-size-budget bumps).
