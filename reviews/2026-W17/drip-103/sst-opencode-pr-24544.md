# sst/opencode PR #24544 — fix(session): compare message positions instead of IDs in SessionPrompt.run

- Link: https://github.com/sst/opencode/pull/24544

- Size: +19 / -6 across 1 file

## Summary

Replaces lexicographic `lastUser.id < lastAssistant.id` and `m.info.id <= lastFinished.id` comparisons in `SessionPrompt.run`'s post-step loop logic with array-position comparisons (`session/prompt.ts:1322+`). Tracks `lastUserIndex`/`lastAssistantIndex`/`lastFinishedIndex` alongside the existing `lastUser`/`lastAssistant`/`lastFinished` references, then uses index ordering for the "no more tool calls, exit loop" check and pivots the `step > 1` follow-up-user-text scan to `for (let i = lastFinishedIndex + 1; i < msgs.length; i++)` instead of filter-by-id.

## Specific-line citations

- `session/prompt.ts:1322-1342`: three new index trackers initialized to `-1` and assigned in the same conditional that captures the corresponding message reference, so the index/reference pair stays in lockstep.
- `session/prompt.ts:1366`: exit-loop guard changes from `lastUser.id < lastAssistant.id` to `lastUserIndex < lastAssistantIndex`. This is the substantive correctness fix — `MessageID.ascending()` produces lexicographically sortable IDs *as long as* every ID is generated locally in monotonic order, but the moment a session is loaded with messages that were generated out-of-order (replay, import, multi-device), the lexicographic comparison no longer matches conversation order. Position is the source of truth.
- `session/prompt.ts:1471-1474`: the `step > 1` follow-up-user-text scan flips from `for (const m of msgs) { if (m.info.role !== 'user' || m.info.id <= lastFinished.id) continue; … }` to a position-anchored `for (let i = lastFinishedIndex + 1; i < msgs.length; …)`. Beyond correctness this is also O(steps×msgs)-friendly: the loop now starts past the finished message instead of scanning all of `msgs` and filtering.

## Verdict

**merge-after-nits**

## Rationale

The diagnosis is correct: `id`-based ordering only matches array-position ordering when IDs are generated by a single monotonic counter, and `MessageV2` already gives no such guarantee for messages introduced by session resume / external import / cross-device merge — the `for (let i = msgs.length - 1; i >= 0; i--)` walk that establishes `lastUser`/`lastAssistant`/`lastFinished` already takes array position as truth, so it's actively inconsistent for the downstream comparisons to fall back to ID lex.

Two nits: (1) PR ships no test, but the bug's failure mode (loop never exits after a tool-call-free assistant turn when message IDs go out of order) is exactly the kind of regression that's easy to pin with a synthesized `msgs` fixture where the user message has a *later* ID than the trailing assistant — the existing `compaction.test.ts` shape is a fine template; (2) `lastUserIndex = -1` initialization is correct but combined with the exit-loop check, an edge case worth pinning is "no user message at all" (`lastUser === undefined`, which `lastUserIndex < lastAssistantIndex` would still evaluate against `-1 < someIndex` as truthy and could short-circuit) — the prior `lastUser.id < lastAssistant.id` would have NPE'd on the same shape, so this is technically a behavior change worth a one-line guard.
