## What it does

Promotes remote plugin install from "register-only" to "fetch-validate-download-extract-cache". After the existing `install_remote_plugin` cloud call succeeds, the new module downloads the bundle tarball over HTTPS, validates the URL/version on the way in, extracts a tar.gz into a staging tempdir under `plugins/.remote-plugin-install-staging/`, finds the plugin root, and hands off to the existing `PluginStore::install_with_version` path. Adds typed errors for every failure class and rewires the post-install MCP/apps loading from the cloud-detail snapshot to the actually-extracted on-disk bundle.

## Specific observations

- **`validate_remote_plugin_bundle` at `core-plugins/src/remote_bundle.rs:97-152` is the right shape for an external-input boundary.** Pure function, no IO, returns a typed `ValidatedRemotePluginBundle` whose fields are all *post-validation* (`PluginId` parsed, version trimmed and run through `validate_plugin_version_segment`, URL parsed and scheme-checked against `{"http", "https"}`). The "parse, don't validate" pattern means downstream `download_and_install_remote_plugin_bundle` can't accidentally see a `None` version or a `gopher://` URL — those would have been rejected at the gate. The split between this validator and the async download function is correct: validation happens synchronously before the download network round-trip is even attempted, so a bad backend response fails fast with a typed error rather than after a 60-second timeout.
- **HTTP/HTTPS scheme allowlist is the load-bearing security gate.** At `:139-149` the explicit `match parsed_bundle_url.scheme()` with an allowlist (not a denylist) is right — any future scheme that `url::Url::parse` learns to recognize (`file://`, `data:`, `gopher://`) defaults to rejected, with a typed `UnsupportedBundleDownloadUrlScheme` error carrying the offending scheme for log forensics. A denylist of `file` and `data` would have been a security footgun.
- **Tar extraction path-traversal defense at `checked_tar_output_path`** is the other load-bearing security piece. Walks tar entry path components and rejects `ParentDir` / `RootDir` / `Prefix` outright with `InvalidBundle` — exactly the right `Component`-level approach (string-comparing for `..` would miss `foo/../../etc/passwd` after path normalization, and joining-then-checking-prefix would have a TOCTOU window through symlinks). Only `Component::Normal` and `CurDir` (silent skip) reach the output path. **Concern**: there's no symlink-type handling visible in the snippet — if the tar contains a symlink entry (`b'2'` typeflag) followed by a regular file written through the symlink target, an attacker could escape the staging dir. The visible code handles `b'\0'`/`b'0'` (regular file) and `b'5'` (directory); if the tar parser doesn't explicitly reject other typeflags or skip-without-creating symlinks, this is a real CVE shape. Worth confirming the `_` arm in the typeflag match.
- **Custom tar parser instead of `tar` crate** at `extract_plugin_bundle_tar` is suspicious. The `tar` crate is mature, has battle-tested path-traversal defenses, and a hand-rolled `[u8; 512]` block reader with manual octal parsing is a maintenance and security risk. The justification likely revolves around tighter control over typeflag rejection or avoiding a dependency, but the value of "we wrote it ourselves so we know the trust surface" is much smaller than the value of "the entire Rust ecosystem audits the `tar` crate." Strong recommendation: justify the custom parser in module-level docs or migrate to `tar::Archive::unpack` with `set_unpack_xattrs(false)` and `set_preserve_permissions(false)` for the same effect with a well-audited core.
- **`spawn_blocking` wrapping at `download_and_install_remote_plugin_bundle:155-164` is correct** — extraction is sync IO and tar parsing is CPU-bound, so off-loading from the tokio reactor is the right move. `JoinError` is mapped to `InvalidBundle` which is wrong shape (a panic in the blocking task isn't a malformed-bundle problem) — should be a separate variant like `ExtractionTaskFailed { source: JoinError }` to avoid telemetry confusion.
- **Atomic install via `tempfile::Builder::tempdir_in(&staging_root)` at `install_remote_plugin_bundle`** is the right shape for staging — `TempDir` auto-cleans on drop, including on panic, so a failure mid-extraction doesn't leak a partial install. The handoff to `PluginStore::install_with_version` (presumably an atomic rename from staging to final) means the cache directory either contains a complete plugin or no plugin at this version — no torn states. Worth verifying that `install_with_version` does in fact perform an atomic rename (rename within same filesystem is atomic on POSIX; `tempdir_in(&staging_root)` ensures same-FS).
- **`extract_dir` is not explicitly cleaned after successful extraction** — relies on the `TempDir` Drop impl. If `install_with_version` *moves* (not copies) `plugin_root` out of the staging dir, the `TempDir` Drop will succeed (empty dir cleanup); if it *copies*, every install leaks an extracted tree under `plugins/.remote-plugin-install-staging/remote-plugin-bundle-XXXX/` until process exit. Worth confirming.
- **`install_remote_plugin_bundle` ordering**: in `app-server/src/codex_message_processor/plugins.rs`, the cloud `install_remote_plugin` call happens *before* the local download/extract. If the cloud install succeeds but the local download fails (network blip, malformed bundle), the user is in a "registered with backend, not installed locally" state with no automatic rollback. The post-install MCP refresh and OAuth login flows at `:466-475` are gated on `plugin_mcp_servers` non-empty so a failed extract before this point at least doesn't fire spurious OAuth flows, but the half-installed state is still observable to the next "list installed plugins" call. Either rollback the cloud install on local failure or document the half-state recovery path.
- **Loading from on-disk bundle vs. cloud detail at `:472-475`**: the rewrite from `remote_detail.app_ids.into_iter().map(AppConnectorId)` to `load_plugin_apps(result.installed_path.as_path()).await` is the right change for correctness — the plugin-on-disk's actual app IDs are the source of truth, not what the catalog claimed before download. If they disagree (catalog stale, bundle updated), trusting the bundle is right since that's what the user just got. **Subtle**: this means a malicious bundle could now declare app IDs the catalog never advertised, requesting OAuth scopes the user never opted in to via catalog selection. Worth confirming `plugin_apps_needing_auth_for_install` enforces a "must appear in catalog" check, otherwise the bundle is trusted to enumerate its own connector requirements.
- **Test rewrite at `plugin_install.rs` is comprehensive** — fixtures use the realistic `plugins~Plugin_<32hex>` ID shape via `REMOTE_PLUGIN_ID` constant, helpers `mount_remote_plugin_*` and `configure_remote_plugin_test` factor out the mock-server setup so individual tests focus on the failure cell they're pinning. The new `plugin_install_rejects_missing_remote_bundle_url` is exactly the right kind of negative test for the validator's `MissingBundleDownloadUrl` arm. The success test at `:179-216` asserts the on-disk path `plugins/cache/chatgpt-global/linear/1.2.3/.codex-plugin/plugin.json` exists *and* asserts the wrongly-named `plugins/cache/chatgpt-global/{REMOTE_PLUGIN_ID}/1.2.3` path does *not* exist — that negative assert pins the contract that local cache uses the human-readable plugin name not the backend's opaque ID, exactly the right kind of regression anchor.
- **Missing test cells**: no test for path-traversal rejection (a tar fixture containing `../../etc/passwd` should produce `InvalidBundle`), no test for unsupported-scheme rejection (`file://` URL should fail validation), no test for invalid-version rejection. These are the security-critical cells; the visible tests cover the happy path and one missing-field cell.

## Verdict

**request-changes** — substantial new attack surface (download untrusted tarballs, extract over filesystem) needs the security-critical test cells (path-traversal, scheme-rejection, version-validation) before merge, and the custom tar parser deserves either a strong justification in module docs or migration to the `tar` crate with explicit symlink handling. Half-install rollback semantics (cloud register succeeded, local install failed) need either rollback-on-failure logic or a documented recovery path. Smaller nits: `JoinError` should map to its own variant rather than `InvalidBundle`, the `extract_dir` cleanup-after-success path needs verification, and the bundle-vs-catalog app-ID trust boundary deserves an explicit check.
