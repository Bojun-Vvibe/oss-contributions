## What it does

Three bundled changes, all converging on one goal — keep `PermissionProfile` invisible on the *stable* app-server v2 wire surface while allowing it to keep flowing through internal Rust code:

1. Marks the `permissionProfile` field on five v2 message shapes (`command/exec`, `thread/{start,resume,fork}`, `turn/start`) as experimental, so the existing schema/TS regeneration filter strips them from the stable artifacts.
2. Enables `inspect_params: true` on `OneOffCommandExec` so the runtime experimental-API gate can catch a stable client that *does* somehow send `permissionProfile` and reject it with the right `command/exec.permissionProfile` reason string instead of letting the request through.
3. Fixes a previously-existing bug in the TS schema-filter scanner at `export.rs` where apostrophes in JSDoc comments (`m.content is string`-style prose) were being treated as string delimiters, so the scanner's "are we inside a string" state machine got confused and some experimental fields *that were filtered correctly from the JSON* leaked through into the stable TS.

## Specific observations

- **The `inspect_params: true` macro flag at `common.rs:584` is the load-bearing runtime piece.** Without it, the per-field experimental marker would be a documentation-only contract — a stable client could hand-craft a `command/exec` request with a `permissionProfile` field and the server would silently honor it. With it, the existing per-field experimental inspector at `experimental_api::ExperimentalApi::experimental_reason` walks the deserialized params struct and surfaces the `command/exec.permissionProfile` marker, which the dispatch layer can use to reject with a clean error instead of accepting an unstable input. The new test at `:2052-2078` is the spec for this contract — constructs the full `CommandExecParams` with `permission_profile: Some(...)` and asserts `experimental_reason` returns exactly `Some("command/exec.permissionProfile")`. A future renaming of the reason string would either update the test (intentional) or break it (regression caught).
- **Schema-deletion blast radius is large but mechanical.** 11 JSON files and 9 TS files lose 5 occurrences of the same `permissionProfile` field. The fact that this is purely additive removal (no field reshape, no rename) means downstream stable clients that never used `permissionProfile` see a 100% backwards-compatible delta. Stable clients that *were* using it (presumably none, since the field was already documented as experimental in some spirit) hit the runtime rejector instead. The git-blame story says "field used to leak into stable schema, now it doesn't" — exactly the right framing.
- **The TS-scanner fix at `export.rs:739-960` is the most subtle and the most important.** Before this PR, `ScanState` tracked `string_delim` and `escape` but had zero awareness of comments. The OpenAPI/JSDoc generator emits doc comments like `* Defaults to the user's configured permissions when omitted.` — that apostrophe in `user's` was opening a "string" the scanner never closed, so depth tracking went sideways for the rest of the file. The fix adds `block_comment` and `line_comment` state with proper `previous_char` tracking to detect `//` and `/*…*/` boundaries, and renames `in_string()` → `in_ignored_syntax()` to reflect the broader "skip this region for delimiter purposes" semantic.
- **The comment-state state machine at `:891-934` is correct but has subtle ordering.** The `line_comment` and `block_comment` checks come *first*, then the in-string check, then the comment-entry detection (`previous_char == Some('/')` plus next `/` or `*`). Order matters: if `string_delim` were checked first, a `"//"` substring inside a JSON string literal would incorrectly enter line-comment state. The current ordering (comment-state-active → string-state-active → check-for-comment-entry) handles `"foo // bar"` correctly because the string check fires before the comment-entry check. **Subtle bug shape**: the comment-entry check at `:935-944` doesn't check whether `previous_char == Some('/')` is itself inside a string — but since the string-active arm at `:910-925` returns early before ever reaching the comment-entry check, this is fine. Worth a comment in the source naming the ordering invariant since the implication is non-obvious.
- **`previous_char` reset on `block_comment` exit at `:903`** correctly nulls `previous_char = None` so a `*/` immediately followed by `/foo` doesn't mis-trigger another comment entry on the next char. Without this reset, `*/foo` would set `previous_char = '/'` (wrong, since `/` was the comment terminator not standalone) and then `f` would not enter line-comment state but `/o` further on would check `previous_char == Some('/')` against the wrong `/`. Subtle but right.
- **Test at `:2731-2810` reproduces the exact JSDoc-with-apostrophe shape that was failing.** The fixture `content` includes `"Defaults to the user's configured permissions when omitted."` inside a `/** … */` doc comment immediately preceding `permissionProfile?: PermissionProfile | null` — exactly the layout that broke the pre-fix scanner. Asserts both that the field is stripped *and* that the import line is stripped (no orphan import) *and* that an unrelated field (`sandboxPolicy`) survives. That third assert is the load-bearing regression anchor — without it a future "fix" that aggressively stripped *everything* would pass the negative asserts and silently break other fields.
- **Cleanup safety**: `TempDirGuard` at `:2740-2747` ensures the test tempdir is removed even on panic via `Drop`. Good hygiene; pattern is repeated from earlier tests in the same file (consistency check passes).
- **Concern**: the experimental marker is per-field, which means a *stable* client that sends `command/exec` *without* `permissionProfile` works as before. Good. But the docs on the field at the JSON-schema level still say "Defaults to the user's configured permissions when omitted" — there's no surfaced indication on the Rust struct or any generated artifact that this field is experimental. The experimental-API mechanism is runtime-only. Worth either (a) a `#[deprecated]`-style attribute that surfaces "experimental" in the IDL/docs, or (b) a doc-comment update on the Rust field saying "This field is gated behind the per-field experimental API and will be rejected by stable clients."
- **`export.rs` change extends `ScanState`'s field count from 3 to 6** which makes the struct's `Default::default()` initialization expand under the hood — verified the struct uses `#[derive(Default)]` (visible in surrounding context), so the new fields default to false/None correctly without an explicit impl change.
- **`split_top_level_multi` and `find_top_level_brace_span` at `:739`/`:763`** both flip from `!state.in_string()` to `!state.in_ignored_syntax()` — the rename is mechanical and complete; no other callers of the renamed predicate exist (the helper was tightly scoped).

## Verdict

**merge-as-is** — three bundled but coherent changes, all converging on "keep `PermissionProfile` off the stable wire while migration continues." The experimental-API runtime gate (`inspect_params: true` + per-field marker + new test) is the load-bearing correctness piece; the schema/TS regeneration is the mechanical cleanup that follows; the comment-aware scanner fix at `export.rs` closes a pre-existing real bug where JSDoc apostrophes were corrupting the TS-filter state machine and silently leaking experimental fields into stable TS artifacts. The new test at `export.rs:2731-2810` reproduces the exact failure shape and pins both positive (stripped) and negative (unrelated fields preserved) outcomes — that's the right shape for catching aggressive over-stripping regressions. Optional follow-up: a doc-level "experimental" annotation on the Rust field so the experimental contract surfaces in IDL not just at runtime.
