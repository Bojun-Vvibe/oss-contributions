## What it does

Closes a real user-facing bug (issue `#8437`) where typed config reads of provider timeouts (`OLLAMA_TIMEOUT`, `OPENAI_TIMEOUT`, `LITELLM_TIMEOUT`) silently failed when the value in `config.yaml` was quoted (`OLLAMA_TIMEOUT: '1200'`). The desktop app writes these values quoted, so the bug was reachable through the normal config UI without the user ever editing the YAML by hand. Pre-fix, `serde_yaml::from_value` would try to deserialize `String("1200")` as `u64` and fail with a type mismatch error, propagated as `ConfigError::DeserializeError`. Post-fix, the typed read falls back to `parse_env_value` (the same scalar parser used for `OLLAMA_TIMEOUT=1200` env-var reads) and converts the string to a JSON number, which `serde_json::from_value` then deserializes to `u64` cleanly.

## Specific observations

- **The fix at `config/base.rs:735-748` is surgical.** The pre-existing single-line `.and_then(|v| Ok(serde_yaml::from_value(v.clone())?))` is replaced with an explicit `match` on the YAML deserialization result, falling through only on the specific shape "Err *and* the value is a `String`." Two fail-safe properties:
  - **Non-string values that fail YAML deserialization fall through to the original `yaml_err`** at `:740-742`, so a malformed list or table still surfaces the YAML error class — the new branch only activates for the narrow "string scalar that the caller wanted as a number" case.
  - **`Self::parse_env_value` failure returns *that* error**, then `serde_json::from_value` failure maps back to `yaml_err.into()` (not the JSON error). The choice to surface the original YAML error class on the JSON-deserialize failure path is right because the *caller's* mental model is "I'm reading from YAML config" — surfacing a JSON error would be confusing.
- **String-requesting callers are correctly preserved.** `test_get_param_reads_quoted_numeric_yaml_as_string` at `:1196-1207` exercises `let value: String = config.get_param("XXX_TIMEOUT")?` against `set_param("XXX_TIMEOUT", "300")` and asserts `value == "300"`. This pins the contract that `String`-typed reads still see the original string and don't get round-tripped through `parse_env_value` (which would normalize `"300"` to `300` as a number). The first branch of the `match` (`Ok(value) => Ok(value)`) handles this — `serde_yaml::from_value::<String>(YamlValue::String("300"))` succeeds without ever reaching the fallback. Good.
- **`parse_env_value` reuse is the load-bearing design choice.** The PR description says "the same scalar parsing used for environment variables" — this means `OLLAMA_TIMEOUT='300'` in `config.yaml` and `OLLAMA_TIMEOUT=300` in the environment now produce the *same* typed value through the same code path. That's a real consistency win. **Subtle**: `parse_env_value` likely handles booleans (`"true"`, `"false"`), floats, and integers — meaning a `MY_FLAG: 'true'` quoted-bool config value would now parse as `bool true`, where pre-fix it would fail. This is an intentional widening of the contract; worth confirming the env-var parser doesn't have any surprising conversions (e.g., does `"on"` or `"yes"` parse as bool true?). If yes, this PR silently extends those semantics to YAML reads.
- **The `serde_json::from_value(parsed)` step at `:744`** uses JSON not YAML for the second deserialization. This is correct because `parse_env_value` returns a `serde_json::Value`, and `serde_json::from_value` is the natural deserializer for that type. The downside: if a caller's target type has serde derives that behave differently between `serde_yaml` and `serde_json` (rare but possible — for instance, untagged enums with overlapping variants), the fallback path would deserialize differently than the primary path. For provider timeouts (`u64`), this is a non-issue.
- **Test matrix at `:1175-1224` covers four cells**: numeric YAML → u64 (no-op happy path), quoted-numeric YAML → u64 (the fix), quoted-numeric YAML → String (preserves string-typed reads), invalid quoted string → u64 (negative path, asserts `Err(DeserializeError)`). These are exactly the cells the change touches. **Missing cell**: quoted boolean (`"true"` requested as `bool`) and quoted float (`"3.14"` requested as `f64`). Both should work via the new path; pinning them would document and lock the widened contract.
- **`env_lock::lock_env([("XXX_TIMEOUT", None::<&str>)])`** at the top of every test ensures the test's env doesn't inherit a real `XXX_TIMEOUT` from the shell. Good hygiene; consistent with surrounding tests in the file.
- **Error type collapse**: `serde_json::from_value(parsed).map_err(|_| yaml_err.into())` at `:744` discards the JSON error entirely (`|_|`). If the JSON-deserialize failure mode is interesting for debugging (e.g., "string `'invalid'` parsed as JSON `string` but caller wanted u64"), that information is lost. Probably fine since `yaml_err` carries a similar shape, but a `verbose_log!("fallback parse failed for key={key}, json_err={err}")` would aid diagnosis.
- **No regression risk for the all-numeric YAML happy path**: `numeric YAML → u64` test at `:1175-1184` exercises `set_param("XXX_TIMEOUT", 300_u64)` (which writes a YAML number, not a string) and asserts the read returns 300. The new `match` arm hits `Ok(value)` immediately on the first `serde_yaml::from_value` call — the new fallback code is never executed for this case. Backward-compatibility verified.
- **`parse_env_value` is a private impl method** (`Self::parse_env_value`) — the test at `:1217-1224` (`rejects_invalid_string_as_u64`) implicitly verifies its error propagation by setting `XXX_TIMEOUT="invalid"` and asserting the read returns `Err(DeserializeError)`. The `match` arm at `:743` returns `parsed?` (with the `?` propagating the error from `parse_env_value`), so a string that's neither numeric, boolean, nor any other recognized form bubbles up that error. Good.

## Verdict

**merge-as-is** — clean surgical fix for a real user-reproducible bug (desktop UI writes quoted numerics to `config.yaml`, typed reads then silently fail). The match-on-error fallback is correctly scoped to the narrow "string-typed YAML value, caller wanted a number" cell, fail-safe on the negative path (preserves original `yaml_err` class on JSON-deserialize failure so error messages stay coherent with the caller's "I'm reading YAML" mental model), reuses `parse_env_value` so YAML-quoted scalars and env-var scalars now parse through the same code path (real consistency win), and the 4-cell test matrix at `:1175-1224` covers exactly the changed branches plus the regression anchor for string-typed reads. Optional nits: extending tests to cover quoted-bool and quoted-float widens-contract documentation, and `verbose_log!` on the JSON-deserialize failure would aid diagnosis when the new fallback path hits an unexpected type.
