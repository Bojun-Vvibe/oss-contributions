- Head SHA: `a3dbffa`
- Author: cdxiaodong
- Files: 6 / +142 / −1

## Summary

Fixes DeepSeek V4 multi-turn chat failures where the upstream API rejects assistant tool-call messages whose `reasoning_content` field is missing — even an empty string is required. Adds a dedicated `patch_deepseek_v4_reasoning_messages(model, messages)` helper that walks the message list and ensures every assistant message with `tool_calls` (or `tool_call_id`) carries `reasoning_content: ""` for `deepseek-v4*` models. Patches both the OpenAI-config transform path and the raw `openai.py` HTTP-dispatch path, and adjusts `Message.__init__` to keep the `reasoning_content` slot when `tool_calls` is present.

## Specific observations

- Discriminator at `litellm/llms/openai/common_utils.py:264-273` — `requires_deepseek_v4_reasoning_content(model)` lowercases the model, strips a `responses/` prefix if present, and returns `"deepseek-v4" in normalized_model`. Substring match is the right ergonomic for the `deepseek/deepseek-v4-pro`, `deepseek/deepseek-v4-flash`, `responses/deepseek-v4-pro` cells — but it will also match a hypothetical future `deepseek-v40` or `deepseek-v4o-mini` (anything with `deepseek-v4` as a substring), which is a false positive that may not exist today but will eventually.
- Mutation pattern at `:280-294` — `for message in messages: ... message["reasoning_content"] = ""` mutates the caller's list **in place** and also returns it. This is the same anti-pattern the bedrock vector-stores PR (#26685, drip-143) explicitly fixed with `deepcopy`. If the same `messages` list flows through multiple transformers (e.g. router fallback → second provider) the mutation persists; non-DeepSeek-V4 providers shouldn't see a stray `reasoning_content: ""` on their tool-call messages because the helper short-circuits on non-V4, but defensive copy is the right shape.
- Triple-patch insertion at `gpt_transformation.py:441` (sync), `:467` (async), `openai.py:271` (raw transform), `openai.py:438` (async HTTP dispatch), `openai.py:482` (sync HTTP dispatch). Five callsites is one more than ideal — there's no single seam where "all messages destined for OpenAI-shaped chat completion" pass through, so the patch is correctly applied at each independent path. But this means a future new dispatch path will silently miss the patch unless the contributor remembers — the helper should arguably live as a transform applied higher up.
- **Source-corruption near-miss at `litellm/llms/openai/common_utils.py:296`** — the new helper ends at `:294` with `return messages`, then **`organization: Optional[str]`** appears at `:296` with no class context. Reading the diff: the helper was inserted *inside* the `class OpenAICredentials(NamedTuple):` body (the `class ... = api_base: str / api_key: Optional[str] / [HELPER INSERTED HERE] / organization: Optional[str]` shape). The result is the helper becomes a method of `OpenAICredentials`, the `requires_deepseek_v4_reasoning_content` import in `gpt_transformation.py:59` and `openai.py:60` will fail, and `organization` becomes a dangling annotation on the wrong scope. **This is a hard merge-blocker — the import statements at `gpt_transformation.py` and `openai.py` cannot work as written.**
- `Message.__init__` change at `litellm/types/utils.py:1224` — `if reasoning_content is None and not getattr(self, "tool_calls", None)` keeps the `reasoning_content` slot when tool_calls exist. Right shape for round-tripping the field through the message lifecycle. Test at `tests/test_litellm/types/test_types_utils.py` (named in PR body but not visible in my truncated diff) pins this.
- `get_supported_openai_params` at `gpt_transformation.py:172-174` adds `["thinking", "reasoning_effort"]` for any model containing `"deepseek"` lowercase — this widens beyond `deepseek-v4*` to include older `deepseek-reasoner`/`deepseek-chat`, which may or may not actually accept those params. Test `test_deepseek_v4_supported_openai_params_include_thinking_controls` at `:181-189` only pins the V4 cell, not the older models.
- Test cells at `tests/llm_translation/test_deepseek_completion.py:179-243` cover (a) supported-params include thinking controls, (b) V4 transform_request injects empty reasoning_content for tool-call message, (c) deepseek-reasoner does NOT inject (negative cell) — three-cell matrix is correct shape.

## Rationale

The actual fix logic is right-shape and the four-cell test matrix (V4-positive, V4-tool-calls-injection, reasoner-negative, types-utils slot-preservation) is the correct review surface. **But the diff at `litellm/llms/openai/common_utils.py:258-296` corrupts the source by inserting the new helper between two fields of the `OpenAICredentials` NamedTuple class body** (`api_key: Optional[str]` at `:260`, then the helper, then `organization: Optional[str]` at `:296`). This will (a) fail to import the helper from `gpt_transformation.py:59` and `openai.py:60` because the helper is now a `OpenAICredentials` method scope, and (b) produce a stray dangling `organization: Optional[str]` annotation on the wrong scope or trigger a class-body parse error. **Required: move the helper out of the `OpenAICredentials` class body to module scope (after the class closes) before merge.** Secondary nits worth addressing in the same revision: (1) `deepcopy` the messages or mutate-in-place-with-explicit-comment to make the side-effect contract intentional; (2) tighten the substring matcher to a regex `\bdeepseek-v4\b` or whitelist-based equality to avoid future false positives on `deepseek-v40`/`deepseek-v4o`; (3) narrow the `"deepseek" in model.lower()` widening at `:172` to V4 only, or add tests pinning the non-V4 DeepSeek cells the PR claims to widen.
