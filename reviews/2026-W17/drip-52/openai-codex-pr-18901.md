# openai/codex#18901 — Install standalone archives with checksum verification

- **URL**: https://github.com/openai/codex/pull/18901
- **Author**: efrazer-oai
- **Files**: `scripts/install/install.ps1`,
  `scripts/install/install.sh` (+62/-75)

## Summary

The installer (both Bash and PowerShell variants) used to fetch
the GitHub Releases API JSON to discover (a) the asset download
URL and (b) the asset's `digest` field for SHA-256 verification.
That tied installation to the GitHub API (which is
heavily rate-limited unauthenticated and was occasionally
returning 403/secondary-rate-limit responses to install scripts
in the wild). This PR switches to:

1. Constructing the asset URL directly from the predictable
   `releases/download/rust-v$VERSION/$ASSET` pattern.
2. Downloading a separately-published checksums manifest
   (`codex-installer_SHA256SUMS`, classic two-column format) and
   parsing it to find the expected digest.
3. Switching the asset itself from the npm tarball
   (`codex-npm-...tgz`) to a standalone archive
   (`codex-standalone-...tar.gz`) with a flatter layout.

## Reviewable points

- `scripts/install/install.sh:114-152` — `checksum_for_asset`
  parses the `SHA256SUMS` manifest with awk, asserts the digest
  matches `^[0-9a-fA-F]{64}$`, and lowercases via `tr 'A-F'
  'a-f'`. The shape of the file is `<digest><whitespace><filename>`
  with optional `*` binary marker (handled implicitly because
  awk uses `$2` after default whitespace split — wait, the awk
  uses `$2 == asset` which would fail for `*codex-...tar.gz`).
  Cross-checking with the PowerShell variant at lines 36-45,
  which explicitly does `$filename = $parts[1].TrimStart("*")`.
  **The Bash side does NOT trim the leading `*`**. If the
  manifest is generated by GNU `sha256sum --binary`, asset names
  will start with `*` and the awk match `$2 == "codex-..."` will
  miss. Worth confirming: if the release pipeline uses
  `sha256sum` (text mode, no `*`) this is fine; if it uses
  `sha256sum -b` it breaks the Bash installer silently. Quick
  asymmetry to flag — but the PowerShell side handles it, so the
  generator likely emits the `*` and the Bash side is buggy.

  Update: re-reading line 161-167, awk's default field split
  already drops leading whitespace; if the manifest line is
  `<digest> *<asset>` then `$2` is `*<asset>` and the
  comparison fails. **Recommend** adding
  `gsub(/^\*/, "", $2)` before the comparison, mirroring the
  PowerShell side.

- `install.sh:166` — `printf '%s\n' "$digest" | tr 'A-F' 'a-f'`
  normalizes case. Good defensive choice given some tooling
  emits uppercase hex.

- `install.ps1:32-50` — same logic, but the `$parts =
  $line -split "\s+", 2` + `TrimStart("*")` correctly handles
  both text and binary mode manifests. This is the reference
  implementation; the Bash variant should match.

- `install.sh:228` and `install.ps1:651` — asset name change to
  `codex-standalone-$npm_tag-$resolved_version.tar.gz` with new
  archive layout. Bash's `install_release` (line 558-578) drops
  the `package/vendor/$target` prefix walk and copies directly
  from `$archive_root/codex` and
  `$archive_root/codex-resources/rg`. PowerShell's `$copyMap`
  (line 660-665) does the same flatten. Consistent across both
  installers.

- `install.sh:191-195` — verification error message updated from
  "release metadata" to "release checksums" — accurate now that
  we read a manifest instead of the API JSON.

## Risks

- **The `*` in `sha256sum --binary` output is the one real bug
  candidate** (see above). If the release CI uses `sha256sum -b`
  or `gsha256sum -b`, every Bash-side install will silently fail
  with "Could not find SHA-256 checksum". The PowerShell variant
  is fine. Easy fix: add `sub(/^\*/, "", $2)` in the awk
  predicate.
- Removing the GitHub API dependency is a clear win for rate
  limits and resilience. The only thing the API gave us beyond
  the manifest was *discoverability* — confirming the asset
  exists before downloading. Now a missing asset surfaces as a
  download failure rather than a "asset not found in release"
  error. Marginal regression in error message clarity.
- The new archive layout means existing partially-installed
  caches (with the old `package/vendor/...` layout) will trigger
  `release_dir_is_complete` returning false and a re-download.
  That's the desired behavior for a layout change.

## Verdict

`merge-as-is` — but with a strong recommendation to verify the
checksums manifest format against the release CI. If `sha256sum
-b` is used anywhere, the Bash installer will break silently. The
two-line fix is `sub(/^\*/, "", $2)` in the awk block. If the
maintainer confirms text-mode manifest generation, ship as-is.

## What I learned

Two lessons here:

1. Decoupling installers from authenticated API calls is a
   resilience win — the GitHub Releases API is rate-limited per
   IP and can return 403s for legitimate install traffic; static
   download URLs + a checksums manifest beat that handily.

2. When porting parsing logic across shells (PowerShell ↔ Bash),
   the asymmetry is usually in handling of *implicit* format
   variants — leading `*` for binary mode, BOMs, CRLF line
   endings. The PowerShell side here did it right; the Bash side
   needs a one-line `gsub` to match.
