## Observations

- This is the cap on the multi-PR HttpApi bridge effort: the PR body claims SDK signature mismatches drop from `15` to `0` against the legacy Hono-derived OpenAPI. The diff shape is consistent with that claim — it's a collection of surgical, per-route normalizers in `matchLegacyOpenApi` rather than one sweeping schema change, which is the right blast radius when the goal is byte-equivalent parity with a generated artifact.
- The `/event` and `/global/event` SSE response wiring at `public.ts:80-90` is the load-bearing fix: the new `responses!["200"]` block annotates these endpoints as `text/event-stream` so HeyAPI's generator emits `.sse.get` methods rather than the JSON `.get` shape it would otherwise infer from a missing response media type. The `Event` schema dummy admission at `:74` (`if (spec.components?.schemas && !spec.components.schemas.Event) spec.components.schemas.Event = {}`) is a smell — it papers over an ordering issue where the Effect-derived spec hasn't materialized the `Event` schema yet at the time `matchLegacyOpenApi` runs. A comment noting the invariant ("downstream `$ref` will be resolved when Effect emits the real schema") would prevent a future maintainer from deleting the empty-object stub thinking it's dead code.
- `normalizeRequestSchema` at `:97-117` gains three correctness patches: (1) `flattenOptions` recurses through nested `anyOf`/`oneOf` arms before filtering null variants — without this, an `anyOf:[anyOf:[number,"Infinity"],null]` shape (which Effect emits for the `Infinity`-tolerant numeric encoding) would not collapse to `{type:"number"}`; (2) the `isFiniteNumberOption` helper at `:123-126` tightens the original `withoutNull.every(item => item.type === "number" || item.type === "string")` predicate to require that string variants only contain `["Infinity","-Infinity","NaN"]` — the old loose check would have collapsed `string | number` unions to `number`, silently losing string variants in legitimate `string | number` request bodies; (3) `if (schema.type) delete schema.allOf` at `:108` drops an `allOf` extension when a concrete type is already present, which is the correct read of Effect's habit of emitting `allOf` for literal-narrowed types where the legacy spec just had `type: "string"`.
- The `/tui/publish` PR-component-extraction at `:47-63` hoists the four anonymous `anyOf` arms into named `EventTuiPromptAppend`/`EventTuiCommandExecute`/`EventTuiToastShow`/`EventTuiSessionSelect` schemas before re-pointing the request body at them via `$ref`. This is required because HeyAPI generates discriminated union method signatures only for *named* component schemas; anonymous unions collapse to `unknown`. The `if (anyOf?.length === 4)` guard is brittle but correct for the current shape — if a fifth event variant is added, the silent fall-through means the SDK will regress to `unknown` without a typecheck failure. Worth converting to an exhaustive name-by-name match or a length-mismatch warning.
- The `parameterSchema` test helper at the test file's `:65-72` and the two new tests at `:172-189` correctly pin `start`/`limit`/`roots` query schemas as JSON-stringified equality (rather than structural deep-equal) — this is good test discipline for OpenAPI shape because key ordering changes in the spec output would otherwise silently slip through.

## Risks / nits

- The `/sync/replay` patch at `:86-91` does `structuredClone(spec.components.schemas.SyncReplayEvent)` before calling `normalizeRequestSchema` on the inlined copy. This is correct (avoids mutating the canonical component) but creates a parity risk: future schema additions to `SyncReplayEvent` will appear in the component but not in the inlined items shape unless the PR-time clone-then-normalize is re-run on every regeneration. Worth a CI assertion that the two are structurally equivalent post-normalize.
- `normalizeParameter` hardcodes parameter names `["start", "cursor", "limit", "method"]` and `["roots", "archived"]` at `:131-137`. Adding a new query parameter that's semantically a number/boolean will silently fall through to the generic `normalizeRequestSchema` arm, which may not produce the same shape the SDK expects. Consider pulling these from the source-of-truth route definitions instead of duplicating the list.
- No test for the four new per-route patches (`/experimental/workspace`, `/tui/publish`, `/sync/replay`, `/event`) at the *behavioral* level — the existing parity test (the build script's `0` mismatches comparison) is the only thing pinning them. Worth adding a unit cell that drives `matchLegacyOpenApi` on a synthetic input and asserts each of the four route-specific outputs.
- PR body says "preserve remaining legacy nullable/body/ref shapes for workspace, TUI publish, and sync replay" — the workspace null wrapping at `:42-46` only nullable-wraps `branch` and `extra`. If the legacy Hono spec also had nullable arms for other workspace fields (which the SDK signature comparison would have flagged), the fact that this PR claims `0` mismatches means there are no others, but that's an invariant worth pinning in the test.

## Verdict: `merge-after-nits`

**Rationale:** Correct cap on the parity effort with the right narrow fix shape — per-route normalizers rather than schema-engine changes. The SSE wiring and the named-component-extraction for `/tui/publish` are the right primitives for HeyAPI's codegen contract. Nits: tighten the `anyOf?.length === 4` brittle guard, document the empty-`Event`-schema stub's ordering reason, and pin the per-route patches with unit-level tests so future Effect upgrades that change the input shape can't silently regress the SDK signatures.
