## Observations

- This PR closes two genuine bugs in one focused diff. **Bug 1**: `get_local_repo_path` previously dropped the GitHub owner segment at `github_recipe.rs:131-133` (the original `let (_, repo_name) = recipe_repo_full_name.split_once('/')...` discards the owner via the leading `_`). That meant two configured recipe sources `owner-one/shared` and `owner-two/shared` resolved to the same temp clone directory `<temp>/shared`, and whichever recipe ran first would seed a checkout that the second source then read from — silent cross-repo recipe pollution. **Bug 2**: `get_folder_from_github` joined the raw `recipe_name` parameter into the temp path at `:207`, so a recipe name containing `/` or `\` could escape the intended temp child (path traversal). The fix lifts both into a shared `temp_child_name` sanitizer that whitelists ASCII alphanumerics + `-_.` and replaces path separators with `__`, all other chars with `_`.
- Sanitizer shape at `:131-148` is well-thought-through: (a) explicit `'/' | '\\' => "__"` branch for path separators (preserves a trace of where the boundary was, vs. silently collapsing), (b) `is_ascii_alphanumeric() || ch == '-' || ch == '_' || ch == '.'` whitelist for the safe set, (c) `_` for everything else. The `if child.is_empty() { "_" }` fallback at `:144-146` is defensive — without it, an input like `""` or `"///"` followed by all-separators-to-`__` would produce empty or all-underscore output respectively. Empty would create a file named directly inside `temp_dir` rather than a child directory, which would break `output_dir.starts_with(parent)`-style invariants downstream. Good.
- The `..__outside` test at `:155-159` is the load-bearing security cell. It pins that `../outside` (the canonical path-traversal payload) gets rewritten to `..__outside` — which is a *single child name* that will sit *inside* `parent` (`output_dir.starts_with(parent)` is asserted directly). Importantly, the sanitizer does **not** strip `..` — it just makes sure `..` can't *be* a path separator, so `..__outside` is a literal directory named `..__outside`, not a traversal. That's the right semantics: don't try to interpret intent, just constrain the syntactic shape.
- The owner-collision test at `:144-153` is the load-bearing correctness cell. It verifies that `owner-one/shared` and `owner-two/shared` resolve to *distinct* paths (`assert_ne!(first, second)`) and that the rendering is deterministic (`owner-one__shared` / `owner-two__shared`). Without `assert_ne!`, a future regression that accidentally collapsed owners back to repo-only would still match the explicit-string assertions — `assert_ne!` plus exact-shape asserts is the right paired discipline.
- The `format!("{owner}/{repo_name}")` at `:139` then `temp_child_name(...)` is slightly indirect — the owner and repo_name are *known* to be split by `/`, then they're re-joined and the join character is the very thing the sanitizer is asked to rewrite to `__`. Same end-state as `temp_child_name(&format!("{owner}__{repo_name}"))` would have produced, but the round-trip-through-`/` is intentional: it handles owners or repo names that *themselves* contain odd characters (which shouldn't happen for GitHub but defensively can) by routing every char through the sanitizer. Reasonable design choice; worth a one-line comment explaining the choice.

## Risks / nits

- **Backward-incompatible cache-path change**: existing checkouts at `<temp>/<repo_name>` will be orphaned after this lands — the next invocation will look at `<temp>/<owner>__<repo_name>` and re-clone fresh. Fine for `temp_dir`-based caches (OS will clean them up eventually) but worth a release-note line so users running on capped tmpfs don't get caught out.
- The sanitizer treats `:` as a generic non-whitelisted char, mapping to `_`. This is the right behavior on Linux/macOS, but on Windows `:` is a reserved drive-separator character that's already invalid in path components. So the sanitization is correct, but the test at `:153` (`assert_ne!(first, second)`) doesn't drive a Windows-shaped input. Probably fine — this is the temp-clone *child name*, not a full Windows path — but worth a Windows CI cell.
- `test_recipe_temp_paths` (the negative cell) only tests `../outside`. The other classic traversal payloads — `..\\outside` (Windows-style), encoded forms (`%2e%2e`), or null-byte injection — aren't exercised. The whitelist approach handles them correctly by construction (`%`, null, `\\` all fall to `_` or `__`), but a parameterized cell driving them would pin the contract. Trivial extension.
- `cargo test -p goose-cli github_recipe --no-default-features` could not be run locally per PR body (`libxcrun.dylib` x86_64-only on arm64 host) — the PR is relying on CI to verify the tests pass. Reviewers should check the CI run before merge.
- The PR file shows `\ No newline at end of file` at `:83` — not a substantive issue but `cargo fmt` might be configured in repo to enforce trailing newlines. Worth a quick local check.
- Title says "isolate GitHub recipe temp paths" — accurate but understates the security half (path traversal). Worth widening to mention both ("isolate per-owner + sanitize names").

## Verdict: `merge-after-nits`

**Rationale:** Both bugs are real (cross-repo pollution + path traversal), the fix is narrowly-scoped to the two sites that need it, the shared sanitizer follows the right "whitelist + explicit separator handling + non-empty fallback" shape, and both bug classes are pinned by tests with proper paired discipline (`assert_ne!` plus exact-string for the owner case, `starts_with` plus exact-string for the traversal case). Nits: add a Windows-style traversal cell, document the cache-path-incompat behavior in release notes, and widen the title to credit the security half. None of these block merge.
