## Observations

- This is a **three-layered UX change** packaged as one PR: (a) artifact-policy scoping shifts from message-wide to per-tool-call (`ArtifactPolicyContext.tsx`), (b) a new `toolCallPresentation.ts` helpers module produces deterministic input summaries / artifact actions / file affordances, (c) the underlying `ToolCard` primitive (in `ai-elements/tool.tsx` per the diff hunks at lines 3500‚Äì3763) gains an `embedded`/`showLabel` axis plus an `EmbeddedOverflowViewport` with viewport scroll/resize-observer-driven fade gradients. Each layer has merit; together the PR is large and harder to review than three sequential PRs would have been.
- Layer (a) is the load-bearing correctness change. The original `ArtifactPolicyContext.tsx:152-166` produced one display per `messageId` (only `ranking.primaryToolCallId`), filtered to write-oriented tools only. The replacement at `:189-202` iterates `ranking.candidatesByToolCallId` and produces a `ToolCardDisplay` per *tool call*, gated by the new `isArtifactActionTool` predicate. This is the right shape ‚Äî the previous code had a real bug where two write-oriented tool calls in the same message would silently collapse into one display. The new `buildToolCardDisplay` at `:78-103` correctly handles the "no allowed candidate" case by falling back to `candidates[0]` (`firstAllowedIndex === -1 ? 0 : firstAllowedIndex`), so a message with all-denied paths still gets a renderable host rather than nothing. Good defensive shape.
- The `isArtifactActionTool` predicate at `:68-79` widens the matcher from the previous `isWriteOrientedTool` to a regex-prefix match (`/^(write|writing|create|creating|save|saving)\b/`) plus three substring matches (`write_file`, `create_file`, `save_file`). The regex prefix is good ‚Äî it correctly distinguishes "writing markdown..." from a hypothetical "writing-helper-for-reads" misnomer. But the substring matches are bidirectional: `"unwrite_file"` would match `write_file` substring. Probably no real tool name like that exists, but a `\b`-bounded regex would be safer than `includes`. Tiny nit.
- The deletion of the entire 424-line `ArtifactPolicyContext.test.tsx` and its replacement by `ArtifactPolicyContext.tool-card-display.test.tsx` is concerning at this scope. The PR body justifies it as "the older broad artifact-policy test file no longer matched the new tool-card display contract," which is plausible, but losing 424 lines of test coverage in one PR ‚Äî even if a focused replacement is added ‚Äî leaves a coverage gap that's hard to assess from the diff. Recommend the reviewer cross-reference: which assertions from the old file have *direct* coverage in the new file, and which were dropped because the contract genuinely changed.
- Layer (c)'s `EmbeddedOverflowViewport` (lines 3470‚Äì3535 in the diff) is a small but well-built primitive: viewport-tracked scroll/resize/ResizeObserver wired to `showTopFade`/`showBottomFade` state, with proper teardown in the effect's cleanup function (`removeEventListener`/`disconnect`). The `useEffect` dependency on `[children]` is suspicious ‚Äî re-running the wiring on every children-change could thrash the observer. A `useLayoutEffect` plus a memoized `viewport` ref check would be more idiomatic, but for the typical tool-card output sizes this won't matter.
- `ToolInput` at `:3537-3631` cleanly separates the structured-vs-empty input cases via `hasStructuredInput` (`input !== undefined && input !== null && (typeof input !== "object" || Array.isArray(input) || Object.keys(...).length > 0)`). The `Array.isArray` guard is the right defense ‚Äî `typeof []` is `"object"` so without it the predicate would incorrectly drop array inputs to the empty branch. `summary` accepts both `ReactNode` and `(({isOpen}) => ReactNode)` for callers who want the summary to react to the open state ‚Äî flexible but adds a function-vs-node type discriminator at every render call. Worth profiling on transcripts with many tool calls.
- `ToolOutput`'s new `embedded` branch at `:3694-3725` introduces a `max-h-16` clamp on output content, which is opinionated ‚Äî a 64-px output viewport is tight for any non-trivial JSON. The fade gradients partially compensate, but for a `JSON.stringify(output, null, 2)` with 10+ keys the user must always scroll. Not a defect (the embedded variant is meant to be compact), but the `max-h-16` constant is hardcoded rather than themed, which is a future maintenance friction.

## Risks / nits

- **Three concerns in one PR.** Per usual project discipline, the artifact-policy scoping fix (a) is the only behavioral correctness change; (b) presentation helpers and (c) the `ToolCard` primitive expansion are pure UX. Splitting would have given each layer its own commit history and review surface. As-shipped this is hard to bisect: a regression in tool-card rendering won't separate cleanly from a regression in artifact-policy resolution.
- **Test coverage delta uncertain.** The 424-line test file deletion is the biggest review risk. Need a side-by-side: which test assertions from the old file are pinned in the new file, which are dropped because the contract changed, which are silently lost. A `grep -c "^\s*it("` on both files at minimum, ideally a per-assertion mapping in the PR body.
- `isArtifactActionTool` substring matches (`includes("write_file")` etc.) are not word-boundary-safe. Switch to a single regex with `\b`-anchored alternatives, or to an explicit `Set<string>` of full tool names plus the regex-prefix arm.
- `EmbeddedOverflowViewport`'s `useEffect([children])` re-creates the listeners on every `children` change. For a transcript with rapidly streaming tool output, this could thrash. Consider scoping to `[]` (one-time wiring) and relying on the ResizeObserver to detect children-driven layout changes.
- `max-h-16` (64px) hardcoded clamp on embedded `ToolOutput` is too tight for typical tool JSON. Pull into a theme token (`--tool-card-embedded-max-height`) so it can be adjusted without touching the component.
- The PR body doesn't enumerate the test files added (only mentions "focused replacement coverage"). With the scope of the PR, a PR-body table of "tests dropped ‚Üí tests added (scenarios pinned)" would significantly speed review.

## Verdict: `request-changes`

**Rationale:** The artifact-policy scoping fix is genuinely needed and correctly shaped ‚Äî same-message multi-write-tool collapsing is a real UX bug. But this PR bundles that fix with two large UX-layer additions (toolCallPresentation helpers + ToolCard primitive expansion) and deletes 424 lines of tests in the process, replacing them with a "focused" file whose coverage delta isn't enumerated. Recommend splitting: PR-A ships only the `ArtifactPolicyContext.tsx` per-tool-call scoping (with the new test file as its sole new test) so the correctness fix lands fast; PR-B and PR-C ship the presentation helpers and `ToolCard` primitive separately. This makes each layer independently reviewable, bisectable, and revertable.
