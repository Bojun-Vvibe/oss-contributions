# PR #24022 ‚Äî fix(app): prevent question dock overflow

- URL: https://github.com/sst/opencode/pull/24022
- Author: aakash2330
- Head SHA: `23b9273a68895eb7cdc8479c51ad83ed6906c035`

## Summary

UI fix for the session-page question dock overflowing when an assistant
question carries long text. Replaces a brittle `document.querySelector`-based
height measurement (which scanned the global DOM for the first sticky
element under `.scroll-view__viewport`) with a scoped traversal rooted at the
dock element itself, and pushes overflow handling up one level so the inner
question region can scroll independently. Closes #24020 and #23730.

## Specific callouts

- `packages/app/src/pages/session/composer/session-composer-region.tsx:139-150`
  ‚Äî Adds `min-h-0 max-h-full overflow-hidden` to the prompt-dock wrapper and
  `min-h-0 overflow-hidden pt-px` to its inner container. The `min-h-0` is
  the load-bearing token here ‚Äî without it, the flexbox child refuses to
  shrink below its content size and the dock pushes the rest of the layout
  off-screen. Standard CSS-flex pattern, correct fix.
- `.../session-question-dock.tsx:121-150` (`measure`) ‚Äî The old code:
  ```ts
  const scroller = document.querySelector(".scroll-view__viewport")
  const head = scroller instanceof HTMLElement ? scroller.firstElementChild : undefined
  ```
  was fragile in two ways: (1) it picked the *first* `.scroll-view__viewport`
  in the entire document, which broke any time the page mounted multiple
  scroll views (e.g. modal overlays); (2) it depended on the
  `.sticky`-classed first child being the visual top reference. The new code
  walks `root ‚Üí dock ‚Üí panel ‚Üí header[data-session-title]` and prefers the
  header's `bottom` over the scroller's `top`. Two improvements:
  - Scoped to the actual ancestor panel ‚Äî no more cross-modal interference.
  - Falls back to `scroller.getBoundingClientRect().top` if no header is
    present, instead of bailing entirely.

  Worth confirming that `[data-session-title]` is a stable selector ‚Äî if it's
  generated by another component, an explicit `data-` attribute should ship
  alongside this PR with a short comment that the dock measurement depends
  on it.
- `.../session-question-dock.tsx:144-150` ‚Äî Inset calculation:
  ```ts
  const panelBottom = Math.min(panel.getBoundingClientRect().bottom,
                               window.visualViewport?.height || window.innerHeight)
  const inset = Math.max(0, panelBottom - dock.getBoundingClientRect().bottom) +
                Number.parseFloat(getComputedStyle(dock).paddingBottom || "0")
  const max = Math.max(240, Math.floor(panelBottom - top - 8 - inset - 24))
  ```
  Clamping against `visualViewport.height` is the right call for mobile
  Safari / on-screen-keyboard scenarios where `innerHeight` lies. The hard
  floor of 240px and the magic numbers `8` (gap) and `24` (extra inset)
  should each get a named constant or at least an inline comment ‚Äî six
  months from now nobody will remember why `24` and not `16`.
- `.../session-question-dock.tsx:186-200` ‚Äî `createResizeObserver` now
  observes four elements (dock, panel, scroller, header) instead of two.
  Each is null-safe via the `instanceof HTMLElement` guards. Good ‚Äî but the
  observer array can contain `undefined` entries on first render before the
  panel/scroller exist; please confirm `createResizeObserver` tolerates that
  (most Solid implementations do, but it's worth one assertion).
- `packages/ui/src/components/message-part.css:909-948` ‚Äî Moves
  `overflow-y: auto` and the scrollbar-hiding rules from
  `[data-slot="question-options"]` (or similar) up to the parent slot. This
  is what enables the dock body itself to scroll once content exceeds the
  computed `--question-prompt-max-height`. Combined with the JS fix above,
  this is the actual user-visible behavior change. Sensible.

## Risks

- The new measurement assumes a specific DOM hierarchy
  (`root ‚Üí dock ‚Üí panel ‚Üí header/scroller`). Any future restructuring of the
  session page that breaks this chain will silently fall back to
  `top = scroller.top` or `top = 0`, producing wrong heights without an
  obvious error. Add a `console.warn` in the bail-out branches in dev mode
  to make regressions visible during development.
- `paddingBottom` is parsed as a float but could theoretically be a
  computed `calc(...)` in some browsers ‚Äî `getComputedStyle` resolves it,
  but the parser will silently give `NaN` if the resolved value is unusual.
  `|| "0"` covers the missing case but not `NaN`; consider
  `Number.parseFloat(...) || 0` instead of `... || "0"` upstream.

## Verdict

**Verdict:** merge-after-nits

Solid fix that actually addresses the root cause (global-DOM measurement)
rather than just patching the symptom. Three small follow-ups: (1) confirm
`[data-session-title]` is a stable contract and add a comment, (2) name the
240/8/24 magic numbers, (3) tighten the `paddingBottom` parse to
`Number.parseFloat(...) || 0`.
