# BerriAI/litellm #26650 — fix(mcp): preserve native tools in semantic filter hook

- URL: https://github.com/BerriAI/litellm/pull/26650
- Head SHA: `6c62c5cae5ef7ba87034d05768da32919d51d1bb`
- Verdict: **merge-after-nits**

## Review

- Closes a quietly-bad bug in `litellm/proxy/hooks/mcp_semantic_filter/hook.py` where mixed requests (MCP refs + native OpenAI function tools) silently lost all native tools after MCP expansion, because `_expand_mcp_tools` only returned expanded MCP tools and `_parse_mcp_tools` discarded everything else. Fix at `:266-276` materializes `native_tools_before_expand` *before* the expansion call by shape-matching `isinstance(t, dict) and "function" in t and "type" in t`, then concatenates `data["tools"] = native_tools_before_expand + expanded_tools`. Concatenation order (native first, MCP after) is consistent with non-expand-path ordering downstream so consumers that index by position don't reshuffle.
- New `_is_mcp_tool` classifier at `:126-150` is the load-bearing piece for the rest of the hook's filter/preserve logic. Shape-first check (`isinstance(tool, dict) and "function" in tool and "type" in tool` → always native) before falling back to `_extract_tool_info` name lookup is the right precedence — pre-expanded MCP tools also match the OpenAI shape and would otherwise be misclassified, which the docstring at `:139-144` correctly calls out and defers to the caller's `tools_expanded_from_mcp` flag for handling. The dual-classifier (shape-first then name-lookup) is slightly fragile: if upstream introduces a new MCP tool shape that *also* has both keys, the shape branch will misfire silently. A defensive `assert "name" in tool["function"]` would at least catch the malformed case.
- New `_emit_filter_metadata` at `:160-198` scopes the `litellm_semantic_filter_stats` / `litellm_semantic_filter_tools` headers to MCP-only counts (`f"{len(mcp_tools)}->{len(filtered_mcp_tools)}"`) and skips emission entirely for purely-native requests at `:194-197`. Correct policy: a `5->5` header on a request with zero MCP tools would be misleading (suggests filtering happened when it didn't). Worth confirming `verbose_proxy_logger.info` at `:189` doesn't fire at high cardinality — the message embeds three counts and a CSV which could be noisy under load.
- Native-tool preservation list-comprehension at `:271-275` is duplicated from the `_is_mcp_tool` shape check — should be extracted to a static helper `_is_native_openai_tool(tool)` so the two sites can't drift. Tomorrow when someone adds support for a third tool shape (e.g. `{"type": "code_interpreter"}`), this would be the wrong place to discover the duplication.
