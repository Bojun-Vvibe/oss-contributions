# BerriAI/litellm #26653 — fix(caching): preserve prompt_tokens_details through embedding cache round-trip

- URL: https://github.com/BerriAI/litellm/pull/26653
- Head SHA: `26064667172242162ab788508ad4b1943a87c8ec`
- Verdict: **merge-after-nits**

## Review

- Three-piece fix at `litellm/caching/caching.py`: extends `_convert_to_cached_embedding` signature with `prompt_tokens_details: Optional[dict] = None` and threads it into all three serialization branches (`:653-685`); adds new `_get_per_item_prompt_tokens_details` helper at `:687-731` that handles single-item vs multi-item distribution; calls the helper at `:745-749` in `add_embedding_response_to_cache` so each cached embedding entry carries its own per-item slice of the original `prompt_tokens_details`. Closes the bug where multimodal providers (Bedrock Titan, Nova, Vertex AI) cached the embedding but lost the `prompt_tokens_details` (image-tokens, audio-tokens etc.) on cache rehydration — downstream cost-calculation against cached embeddings was systematically under-counting modal-token usage.
- `_get_per_item_prompt_tokens_details` distribution policy at `:719-728` is the right shape for arithmetic correctness: integers use `divmod(value, num_items)` with the remainder distributed to the first `remainder` items (`quotient + (1 if idx_in_result_data < remainder else 0)`) so summing all per-item details exactly reconstructs the original `value`; floats use even division `value / num_items` (which loses precision but the loss is bounded by float epsilon × num_items); non-numeric values pass through unchanged. The integer-with-remainder pattern is correct and avoids the "9/4 = 2 each, total 8 ≠ 9" undercount that a naive `value // num_items` would produce.
- Single-item short-circuit at `:715-716` (`if num_items <= 1: return details_dict`) is load-bearing — multimodal providers that emit one embedding per call are the *primary* fix target, so the fast path needs to be exact (no division, no remainder accounting). Correctly placed before the distribution loop.
- `details.model_dump(exclude_none=True)` vs `dict` polymorphic handling at `:702-707` covers Pydantic-model and plain-dict cases but doesn't handle the case where `prompt_tokens_details` is some other Pydantic-like object (`BaseModel` subclass without `model_dump` exposed, or an `attrs`-based class). The `else: return None` branch silently drops these — for an obscure provider this means caching silently loses details with no log signal. A `verbose_logger.debug("dropping prompt_tokens_details with unsupported type")` at `:707` would help future debugging. Distribution math also has no test for the float branch (only the integer-with-remainder is explicitly covered in PR description).
