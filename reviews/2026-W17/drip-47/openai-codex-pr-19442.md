## Summary

Bedrock as a provider does not support several runtime capabilities (apps, plugins, MCP servers, tool search, tool suggest, image generation, app instructions, web search). Before this PR, every Bedrock launch path required the user (or a wrapper) to manually disable each one in their config. That's both error-prone and invisible to the app-server's `config/read` view — there was no way to say "config asks for `apps = true` but Bedrock won't honor it". This PR introduces a new `ConfigLayerSource::ProviderCapabilities { provider }` layer that sits in the config layering stack and clamps capability flags down to whatever the provider supports, with `config/read` surfacing the override origin so clients can render "disabled by amazon-bedrock" instead of silently dropping the request.

## Rationale

This is the right architectural move. The pre-PR state (callers manually disable capabilities, no first-class visibility) was a known footgun every Bedrock onboarding hit. Modeling provider clamping as a `ConfigLayer` is consistent with how the rest of the config system works and gives `config/read` clients a clean way to render "why is this off?". Three things to tighten: (1) add a negative test (non-Bedrock provider does not get the clamp); (2) confirm the layer-producer's policy table is data-driven, not a hardcoded `if provider == "amazon-bedrock"` chain — otherwise every new restricted provider needs a code change; (3) note the additive v2 enum variant in CHANGELOG so third-party app-server clients know to bump. None are blocking, but the negative test in particular is cheap and load-bearing for "did the clamp accidentally fire on the wrong provider?" regressions.
