## Summary

Restructures the MLX sampler from "one sampler per request" into a slot-registry model: sequences register via `Sampler.Add(slotID, opts, history)`, each `Sample(slots, logits)` call samples one token per row across any subset of registered slots, and per-slot ring-buffer history is maintained. When all active slots share `Options` and the penalty rings are full, the implementation runs a **single fused transform** over the whole batch via a persistent pooled history tensor; otherwise it falls back to per-slot serial processing indexed against the same pool. PR description explicitly says "Performance is unchanged for a single sequence, which is all that is exposed for now" — i.e. the batch path is dead code at the moment but is the foundation for upcoming multi-sequence serving.

The diff doesn't show `runner.go` adoption — confirm there is no caller of the old signature outside the diff. Also: by removing the no-axis convenience overload you've made the common "logsumexp over last axis" case more verbose at every call site; consider keeping `Logsumexp` as a thin wrapper that calls `LogsumexpAxis(-1, keepDims)`.
- `x/mlxrunner/pipeline.go:53` — `const pipelineSlot = 0`. Hardcoded magic number with a comment ("The runner serializes requests today so we just use a fixed slot ID"). The whole point of this PR is to enable multi-slot batching, so leaving a hardcoded `0` here means the *consumer* of the new batched API is itself single-slot. That's defensible as a staging step, but please add a `// TODO(multi-seq)` linking the issue/PR that will actually thread per-request slot IDs, or this `0` will live forever and become a footgun the day someone reuses the Runner concurrently.
- `x/mlxrunner/pipeline.go:60-62` — defer changed from `request.Sampler.Free()` to `r.Sampler.Remove(pipelineSlot)`. **Subtle resource-lifetime change:** previously the per-request sampler's GPU buffers were freed on defer; now the slot is removed from the shared registry, but does `Remove` release the pooled history tensor's slot? If `Remove` only marks the slot as inactive without deallocating its row of the pooled history, long-running runners will accumulate per-request tensors. The sample.go diff isn't fully in view but the lifecycle invariant needs to be: `Add` allocates the row, `Remove` reclaims it. Add a test that asserts `len(activeSlots) == 0` and `pooledHistory.Dim(0) == 0` (or zero peak memory growth) after N register/sample/remove cycles.
- `x/mlxrunner/pipeline.go:142-143` — `r.Sampler.Add(pipelineSlot, request.SamplerOpts, inputs)` is called **after prefill completes**, but `request.SamplerOpts` was previously consumed inside the request struct. The earlier `request.Sampler.ResetHistory(inputs)` call was removed (`:73`) — confirm no test or code path still depends on the sampler being initialized at request-construction time. The new "lazy register after prefill" is conceptually cleaner but means a prefill error path now skips registration entirely; the `defer r.Sampler.Remove(pipelineSlot)` will then attempt to remove a slot that was never added. `Remove` must be idempotent; verify in `sample.go`.
- `x/mlxrunner/pipeline.go:124` — shape change `mlx.FromValues(tokens[processed:processed+n], n).ExpandDims(0)` → `mlx.FromValues(tokens[processed:processed+n], 1, n)`. Equivalent shape `(1, n)` either way, but the new form bakes the batch dim into `FromValues` rather than expanding after. Cleaner. Same change at `:154`.
- `x/mlxrunner/sample/sample.go` (+415/−122) — the bulk of the actual change; not in my diff window. Two structural questions for the focused review:
  1. The fused-batch fast path triggers when "all slots share Options and penalty rings are full". The "rings are full" condition means the speedup only kicks in *after* the warmup window — for short generations (e.g. < `repeat_last_n` tokens) the batch path never fires and you eat the slot-registry overhead with no benefit. Quantify: at what generation length does the batch path break even with the per-slot fallback?
  2. "All slots share Options" is brittle for heterogeneous serving (different temperatures, top-p across concurrent requests). Is there a plan to bucket slots by Options compatibility, or is the assumption "in practice all sequences in a batch will be for the same request and thus same Options"?
- `sample_test.go` (+252/−62) — the test surface tripled, which is the right shape for this kind of restructure. Specifically check there's a test that asserts equivalence between batched and per-slot output bit-for-bit (same token IDs, same logprobs) for a non-trivial multi-slot, multi-step run; without that, a future fast-path bug is invisible until production diverges.

## Verdict

## Rationale

This is good infrastructure work — the slot/registry model is the right abstraction for batched sampling, and shipping it before any consumer uses it is the right sequencing (avoids a flag-day). What blocks me from a clean approval is the **lifecycle/correctness concerns** around the new shared resource: the `Add`/`Remove` symmetry on prefill-error paths, the pooled-history reclamation, the `Concatenate` early-return changing tensor identity. These are exactly the bugs that show up only under sustained multi-sequence load, which is precisely the load this PR is preparing for. Before merging, I want to see: (1) an explicit "prefill error → no slot leak" test, (2) a bit-for-bit equivalence test between batched and per-slot paths, (3) a clarifying comment or follow-up plan for the "all slots share Options" assumption. The performance story ("unchanged for single seq") is acceptable for a foundation PR but means the actual batched-path benefits won't be benchmarked until the consumer lands — keep an eye on that. Strong direction, needs one more pass before it lands.
