## Verdict: merge-as-is

## Rationale

- **Closes a real "operator sets `OLLAMA_TIMEOUT: '1200'` in `config.yaml` and the value is silently ignored" bug.** Pre-diff `base.rs:735-738` had `values.get(key).ok_or_else(NotFound).and_then(|v| Ok(serde_yaml::from_value(v.clone())?))` — meaning when the desktop app or another tool wrote a numeric value as a quoted string (the canonical YAML serialization for env-var-derived values to preserve roundtrip), `serde_yaml::from_value::<u64>(Value::String("1200"))` returned `Err`, and the caller saw a deserialize error which provider code treats as "value missing" and falls through to the default timeout. The fix at `:735-748` keeps `serde_yaml::from_value` as the fast path and *only* on failure attempts to `parse_env_value(string_value)` (which is the same scalar parser used for env vars and handles `"1200" → JSON number 1200`), then `serde_json::from_value` to coerce into the target type.
- **Two-stage fallback is the right shape.** First-attempt typed YAML deserialization preserves all existing behavior (no regression for callers reading `String`, `bool`, etc.). Second-attempt `parse_env_value` only fires for `value.as_str().is_some()` — i.e. only when the YAML value is a string scalar — so non-string YAML mappings/sequences/booleans correctly fail with the *original* `yaml_err` (the `let Some(string_value) = value.as_str() else { return Err(yaml_err.into()); };` at `:741-743`). That preserves error fidelity for the non-string-scalar failure case.
- **Error reporting on the `parse_env_value`-then-`serde_json` path correctly preserves the original YAML error.** At `:746` `serde_json::from_value(parsed).map_err(|_| yaml_err.into())` — if the env-var-style parse succeeds but coercion to the target type fails (e.g. `"hello"` parsed to JSON string `"hello"` then attempted as `u64`), the error returned is the *original* yaml deserialization error, not a confusing JSON one. That keeps log lines consistent with what the operator would expect from a `serde_yaml`-based config layer. The `_` discard on the `serde_json` error is intentional.
- **String-target-preservation is correct.** When the caller requests `String` (test at `:1206-1213` `test_get_param_reads_quoted_numeric_yaml_as_string`), the first-attempt `serde_yaml::from_value::<String>(Value::String("300"))` succeeds and returns `"300"` — never reaches the fallback. So the fix is strictly additive: anyone storing `"300"` and reading as `String` keeps the string, anyone storing `"300"` and reading as `u64` now succeeds.
- **Four-cell test coverage is the load-bearing regression anchor.** At `:1177-1219`:
  - `test_get_param_reads_numeric_yaml_as_u64`: native YAML number `300` → `u64 300` (regression-anchor for non-quoted case)
  - `test_get_param_reads_quoted_numeric_yaml_as_u64`: quoted `"300"` → `u64 300` (the new positive case, the bug fix)
  - `test_get_param_reads_quoted_numeric_yaml_as_string`: quoted `"300"` requested as `String` → `"300"` (the no-regression-for-string case)
  - `test_get_param_rejects_invalid_string_as_u64`: quoted `"invalid"` requested as `u64` → `Err(DeserializeError)` (the negative cell — non-numeric string still correctly fails)
  Those four cells are exactly the right coverage matrix: positive native, positive quoted, no-regression string, negative non-numeric. Missing only the "non-string, non-numeric YAML value (e.g. mapping) requested as u64" cell which would test the `let Some(string_value) = value.as_str() else { return Err(yaml_err.into()); };` early-return branch.

## Nits / follow-ups

- **The negative test at `:1213-1219` asserts `Err(ConfigError::DeserializeError(_))`** — confirming the error type but not the *contents* of the error. Worth at minimum a debug-format assertion that the error message names the offending value `"invalid"` so operators can find the bad config entry from the log line alone.
- **`parse_env_value` is being reused as a YAML scalar coercer** — its primary callsite is env-var parsing per the existing comment at the original site. Worth a code comment at `:744` naming the reuse: "Reuses the env-var scalar parser to coerce string-encoded numerics/booleans, since YAML and env-var scalars share the same numeric/boolean grammar." Otherwise a future env-var-parser refactor could subtly change the YAML coercion behavior.
- **The fix doesn't cover the symmetric write path.** `set_param` presumably writes the value as it received it, so a caller that does `config.set_param("X", 300_u64)` writes a YAML number, while a caller that does `config.set_param("X", "300")` writes a quoted string — both can now be read as `u64`, but the mixed-format file is still possible. Worth a follow-up that documents the read-side normalization in the API docstring so callers don't expect roundtrip.
- **No coverage for floats** (`OLLAMA_TIMEOUT: '12.5'`) or large integers exceeding `u64` range. The `parse_env_value` parser presumably handles floats, but a `f64` test cell would lock that.
- **The `serde_json::from_value(parsed).map_err(|_| yaml_err.into())` at `:746`** discards the `serde_json` error. For debugging a "value parsed as JSON but couldn't be coerced to target type" failure, the JSON error would actually be more useful than the YAML one. Worth either chaining both errors or returning a custom error variant that includes both. As-is, the operator log just sees the original YAML "expected u64, got string" message which is the same as before the fix — not catastrophically misleading, but loses information.
- **The fix is in the `Config::get_param<T>` method specifically.** If the codebase has other typed-read paths (e.g. `get_secret`, `get_extension_config`), they may have the same bug. Worth a quick `rg "serde_yaml::from_value" crates/goose/src/config/` to check.

## What I learned

The YAML "quoted numeric is a string scalar, not a number" rule is a well-known sharp edge — any time YAML is used as a config format with mixed sources (operator hand-edits, env-var sourcing, GUI app writes), you eventually get a mix of `300` and `"300"` for the same key. The "try typed deserialization first, fall back to scalar-coerce on string-only failure" pattern is the canonical fix and preserves error fidelity for non-string-scalar failures. The four-cell test matrix (positive native, positive quoted, no-regression-string-target, negative non-numeric) is the load-bearing regression anchor; the no-regression-string-target cell specifically prevents a future "simplification" that always coerces strings to numbers and breaks string-typed callers. The "preserve original error" pattern with `map_err(|_| yaml_err.into())` is a calculated trade — useful for log-line consistency but loses debug info; reviewers should evaluate whether the consistency or the debug info matters more for the team's ops workflow.
