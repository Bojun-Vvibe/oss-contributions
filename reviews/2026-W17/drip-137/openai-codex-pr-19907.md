## Verdict: merge-as-is

## Rationale

- **Closes a real auto-review focus-misdirection bug.** Pre-diff `prompt.rs:175-188` rendered every approval-request prompt through the same template — `headings.action_intro` followed by an optional "Retry reason:" block, "Assess the exact planned action below", and "Planned action JSON:" — even when the request variant was `GuardianApprovalRequest::NetworkAccess`. For a network approval the "retry reason" is always the generic "Network access to \"X\" is blocked by policy", which under auto-review pulls the model's attention onto the *blocked connection* (a downstream fact) rather than the *triggering command that initiated the connection* (the load-bearing decision factor). The fix at `:175-209` splits into a `match &request` with a dedicated `NetworkAccess { trigger, .. }` arm that uses the right framing.
- **Trigger-presence branching is correct.** The `if trigger.is_some()` arm at `:179-185` instructs the reviewer to "focus primarily on whether the triggering command is authorised by the user and whether it is within the rules" with the explicit corollary "the user does not need to have explicitly authorised this exact network connection, as long as the network access is a reasonable consequence of the triggering command" — that second clause is the load-bearing one, because without it auto-reviewers tend to fall back on a "did the user explicitly approve this exact `host:port`?" check that's almost always `no` and produces over-restrictive denials. The `else` arm at `:186-191` for the no-trigger case (rare but possible — e.g. session-resumed network access where the originating tool call has been compacted out) correctly degrades to "use the retained transcript and network access JSON to evaluate user authorization and risk" instead of fabricating a trigger context.
- **The "Planned action JSON" → "Network access JSON" rename matters.** At `:194` for network and `:204` for non-network. The label is what the model attends to first when scanning the prompt structure, so calling network-access JSON by its real name (vs the generic "planned action") avoids the model rationalizing the network call as if it were a tool-call action.
- **Snapshot test at `snapshots/...__network_access_guardian_prompt_layout.snap` is the strongest possible regression anchor.** It pins the exact output text including the load-bearing sentences, the order of `>>> APPROVAL REQUEST START` → "Below is a proposed network access request under review." → trigger-context paragraph → "Assess the exact network access below." → "Network access JSON:" → the JSON body → `>>> APPROVAL REQUEST END`. Any future drift to that wording will require a deliberate snapshot update with the diff visible in review.
- **`tests.rs:798-866` test is the right shape and uses the right negative assertions.** The positive `assert!(text.contains(...))` checks lock the new wording, and the four negative `assert!(!text.contains(...))` checks at `:847-852` lock the *absence* of the previous-template strings ("The Codex agent has requested the following action:", "Planned action JSON:", "Retry reason:", and the literal blocked-by-policy reason that was passed in but should now be suppressed) — those negatives are what catch a future refactor that accidentally re-routes the network arm through the generic template path.

## Nits / follow-ups

- **The retry reason is silently swallowed for the network-access branch.** At `:175-194`, the network arm doesn't render `retry_reason` at all (the snapshot test at `:851` confirms `!text.contains("Network access to \"example.com\" is blocked by policy.")` is the intended behavior). For the canonical "blocked by policy" reason this is correct — that string was the source of the focus-misdirection bug — but for *operator-provided* retry reasons (e.g. a custom policy hook that returns a structured "denied: corp proxy requires VPN" message), this loses information that would help the reviewer. Worth either (a) keeping retry-reason rendering for network-access requests when the reason is *not* the canonical blocked-by-policy string, or (b) documenting in the prompt-builder docstring that network-access prompts intentionally drop retry reasons.
- **The "trigger present vs absent" branch comment** at `:179` could name the resume-after-compaction case as the canonical absent-trigger scenario. As-is, future readers might assume the absent-trigger branch is unreachable.
- **No test cell for the absent-trigger arm.** The new `build_guardian_prompt_items_explains_network_access_review_scope` test only covers `trigger: Some(...)`. A second cell with `trigger: None` would lock the "use the retained transcript and network access JSON" wording.

## What I learned

The shape "rendering branch by request variant rather than one-template-fits-all" is the canonical fix for focus-misdirection in any prompt-driven decision pipeline. The two-cell positive/negative assertion pattern (positive `contains` for new wording, negative `contains` for the absence of old wording) is the load-bearing test shape — without the negatives, a future refactor that *adds* the new wording while *also* keeping the old one would pass tests but reintroduce the misdirection. The snapshot file is the canonical regression anchor for prompt content because it pins not just substrings but the full text including section ordering, blank lines, and JSON formatting — all things a `contains` chain misses.
