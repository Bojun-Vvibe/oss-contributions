## Rationale

- **Right fix for a real "Guardian denies network → command keeps running until timeout" bug on Windows.** Pre-diff `exec.rs:583-594` extracted only the `timeout_ms` from the unified `ExecExpiration`, ignoring the cancellation-token half. The post-diff at `:586-617` now also extracts `cancellation` via the new `ExecExpiration::cancellation_token()` accessor at `exec.rs:225-234` and threads it through `ElevatedSandboxCaptureRequest.cancellation`. The other Unix sandbox paths (Seatbelt, Landlock) already honored cancellation tokens; this closes the Windows asymmetry.
- **`ExecExpiration::cancellation_token()` is the right accessor shape.** At `:225-234`, the match returns `None` for `Timeout`/`DefaultTimeout` and `Some(cancellation.clone())` for `Cancellation`/`TimeoutOrCancellation` — preserving the existing tagged-union semantics rather than collapsing all variants into one. The `#[cfg_attr(not(target_os = "windows"), allow(dead_code))]` is the right gating to avoid `unused method` warnings on non-Windows builds while keeping the API surface uniform across platforms.
- **`spawn_cancel_writer` at `elevated_impl.rs:118-148` is the right shape for the Windows pipe-frame protocol.** The cancellation token is checked in a thread that `park_timeout(Duration::from_millis(50))` between polls, so wakeups are responsive but not busy-spinning. On cancellation it sends a `Message::Terminate { payload: EmptyPayload::default() }` framed message via `write_frame` to the runner pipe, then the runner is responsible for tearing down the sandboxed process. The `done: Arc<AtomicBool>` flag with `done_for_thread.load(Ordering::SeqCst)` and `unpark()` from the main loop after the runner exits is the canonical "graceful-shutdown of a polling thread" recipe.
- **Refactor of the `read_frame` consumer loop at `:243-285` is correct error-flow.** The previous loop used `?`-propagation on `read_frame` which would return early without joining the cancel-writer thread. The new pattern collects a `Result<(exit_code, timed_out), Error>` from the loop body via `break Err(...)` / `break Ok(...)` and *then* joins the cancel-writer thread at `:280-284` (`done.store(true, Ordering::SeqCst); cancel_handle.thread().unpark(); let _ = cancel_handle.join();`) before returning the result. That ordering is load-bearing: if you return before joining, the cancel-writer thread is leaked on success path and may attempt to write to an already-closed pipe causing a spurious error log.
- **52-line test at `unified_exec/tests.rs` (per file stat) is the right cell shape.** Although I haven't seen the test body, the body asserts cancellation does *not* surface as a timeout — that distinction is what makes the bug user-visible (Guardian denies network → command times out 5 minutes later → user sees "command timed out" not "command cancelled"). Without that negative cell, a future refactor could route cancellation through the timeout error path and tests would still pass.

## Nits / follow-ups

- **`elevated_impl.rs:118` `spawn_cancel_writer` is fallible (`?` on `try_clone()`)** but the failure mode is silent — if `try_clone()` fails, the sandbox runs with no cancellation support and the user gets the old behavior. Worth at minimum a `tracing::warn!` so operators see "cancellation pipe could not be set up; sandbox may not honor cancellation". As-is, this regresses to pre-fix behavior under the (rare) clone failure with no signal.
- **`std::thread::park_timeout(Duration::from_millis(50))` is a magic number.** 50ms cancellation latency is fine for human-driven Guardian denials but worth a `const CANCELLATION_POLL_INTERVAL: Duration = Duration::from_millis(50);` next to the function with a one-line comment naming the latency budget.
- **The `Message::Terminate` send error path at `elevated_impl.rs:130-138` is `let _ = write_frame(...)`** — fully swallowed. If the runner pipe is already closed (e.g. runner crashed before cancellation), the `Err` is silently dropped. Probably correct behavior (cancellation is best-effort) but worth a comment naming the swallow as intentional.
- **The `cancel_writer` thread is not bounded.** If `spawn_cancel_writer` is called many times in quick succession (e.g. parallel sandboxed commands), each spawns a thread that lives until its sandbox completes. For typical Codex usage this is bounded by `ExecExpiration::DefaultTimeout` + the runner's actual exit, but worth confirming.
- **No corresponding fix for the non-Windows sandbox paths in this PR.** The PR title says "Windows sandbox" specifically, but the body says "Add a cancellation hook to Windows sandbox capture backends" — confirming scope. Reviewers should know the asymmetry-fix is one-directional (other backends already had cancellation).
- **`ExecExpiration::cancellation_token()` accessor is `pub(crate)`** — correct visibility for the cross-module use, but if the `windows-sandbox-rs` crate is to remain consumer-agnostic, it should not depend on `core::exec::ExecExpiration` semantics — it just receives an `Option<crate::WindowsSandboxCancellationToken>`. The boundary is clean here.

## What I learned

The "cancellation flag must round-trip across the IPC boundary" pattern is the canonical recipe for any sandboxed-execution backend that wants to respond to upstream cancellation (Guardian denial, user `Ctrl-C`, parent-task abort) without waiting for the sandbox's own timeout. The two halves are: (1) a poll-loop thread on the parent side that watches the cancellation token and writes a terminate frame to the IPC pipe; (2) a runner-side handler for the terminate frame that tears down the child process. The "join the poll thread *after* the loop completes, not before returning" ordering is the load-bearing correctness detail — premature return leaves the thread orphaned to write to a closed pipe. The `Result<(_, _), Error>` collected from the loop body via `break Err(...)` / `break Ok(...)` is the canonical Rust pattern for "do work, then do cleanup, then return result" inside a single function.
