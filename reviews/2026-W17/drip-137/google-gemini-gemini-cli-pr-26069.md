## Rationale

- **Closes a real `TypeError: resolved.startsWith is not a function` crash** triggered by two yargs-shaped argv inputs: (1) `gemini --model foo --model bar` (yargs parses to `model: ["foo", "bar"]`), (2) `gemini --model` with no value (yargs parses to `model: true`). Both flow into `resolveModel(requestedModel)` which calls `requestedModel.startsWith(...)` somewhere downstream (or `isCustomModel` which does the same). The fix is defense-in-depth: coerce at both the *outer* CLI ingestion site (`config.ts:818-830`) and the *inner* core resolver (`models.ts:111-117`).
- **Outer coercion at `config.ts:818-830` matches yargs semantics correctly.** `Array.isArray(rawModel) ? String(rawModel.at(-1) ?? '').trim() || '' : rawModel === undefined ? undefined : (String(rawModel ?? '').trim() || '')` — handles three cells: array → take last, `undefined` → keep as `undefined` (so the env-var fallback chain works), anything else → `String(...)` coerce + trim. The `.at(-1)` is the right "last wins" semantic that matches the user mental model of "the last `--model` flag overrides earlier ones" — that's the documented behavior in the test at `config.test.ts:783-822` where `model: ['gemini-1.5-pro', 'gemini-2.0-flash']` resolves to `'gemini-2.0-flash'`.
- **Inner coercion at `models.ts:112-117` is the load-bearing defense-in-depth layer.** `Array.isArray(requestedModel) ? String(requestedModel.at(-1) ?? '').trim() || '' : typeof requestedModel !== 'string' ? String(requestedModel ?? '').trim() || '' : requestedModel.trim() || ''` — exactly mirrors the outer coercion shape. This matters because `resolveModel` is called from many paths beyond `loadCliConfig` (the test enumerates `isCustomModel(['a', 'b'])`, `resolveModel(true)`, `resolveModel(null)`), so fixing only the CLI-ingestion site would leave other callers (e.g. the model-config-service, settings-driven model resolution) still vulnerable. The duplicate coercion logic is a calculated DRY-vs-blast-radius trade — worth at least a shared `normalizeModelInput(input: unknown): string | undefined` helper.
- **`models.test.ts:432-446` `resolveModel` tests cover the three load-bearing cells.** `(['a', 'b']) → 'b'`, `(true) → 'true'`, `(null) → ''`. The `(null) → ''` cell is the subtle one — empty-string vs `undefined` matters because the empty string falls through to the `defaultModel` branch downstream while `undefined` would either crash or hit a different default — pinning the empty-string output is what locks the no-`--model` case to the right downstream branch.
- **`models.test.ts:277-281` `isCustomModel` array test pins the symmetric behavior.** `isCustomModel(['gemini-2.0-flash', 'gpt-4'])` returns `true` (last is custom) — without this assertion, a future refactor could change the array-handling semantics in `isCustomModel` to "any" or "all" without breaking the `resolveModel` tests, which would be a silent behavior change.

## Nits / follow-ups

- **Code duplication between `config.ts:822-826` and `models.ts:112-117` is the biggest concern.** Both implement the same `Array → last; non-string → String coerce; trim; '' fallback` chain. Worth extracting to `packages/core/src/config/models.ts::normalizeModelInput(input: unknown): string | undefined` (the choice of `undefined` vs `''` for the `undefined` input is the only semantic difference — outer wants `undefined` to keep the env/settings fallback chain alive, inner can collapse `undefined` → `''` since it's the resolution endpoint). A two-arg version `normalizeModelInput(input: unknown, { keepUndefined: boolean })` would handle both call sites.
- **`String(rawModel.at(-1) ?? '')` allocates on the hot path** for every CLI invocation. Negligible, but worth noting that the `?? ''` is redundant given the outer `String()` already converts `undefined` to `'undefined'` — actually wait, it's needed: `String(undefined) === 'undefined'` would put the literal string `"undefined"` as the model, while `String(undefined ?? '') === ''` empties it. So the `?? ''` is load-bearing; document it.
- **No test cell for `model: false`** — yargs can produce `false` for `--no-model`-style negation. The test covers `true` but not `false`. With the current coercion, `String(false ?? '').trim() || ''` evaluates to `'false'` (the literal string, not empty) — which would then flow to `resolveModel('false')` and probably hit a "model not found" downstream error rather than fall through to the default. Worth either an explicit cell pinning that behavior or a `typeof rawModel === 'boolean' ? undefined : ...` branch to treat boolean inputs as "no model specified".
- **`String(rawModel)` for non-`undefined` inputs** is broad — for `Symbol(...)` or BigInt or class instances it'd produce `'Symbol(undefined)'` / `'42n'` / `'[object Object]'` and route those through as model names. yargs typically only produces `string | string[] | boolean | number`, so this is theoretical, but the `typeof requestedModel !== 'string'` branch in `models.ts:113-115` could be tightened to enumerate the expected types and explicitly route others to `''`.
- **`models.ts:118` `config.modelConfigService.resolveModelId(normalizedModel, {...})`** is the only caller in the experimental dynamic config branch — but the rest of `resolveModel` (the non-experimental path) is not visible in this diff and presumably also uses `requestedModel`. Worth checking that the rename from `requestedModel` → `normalizedModel` was done at every site below the coercion, not just the one shown.

Worth a `createTestCliArgs(overrides)` helper to reduce noise and prevent drift when CliArgs gains a new field — the current shape requires updating every test in the file when CliArgs adds a field.

## What I learned

yargs's "duplicate flag → array, naked flag → boolean, missing flag → undefined" tri-state output is a well-known sharp edge for any CLI that types its argv with `string`. The defense-in-depth pattern (coerce at the CLI boundary AND at the resolver boundary) is the canonical answer because the resolver has callers beyond the CLI — settings files, IPC payloads, programmatic usage — each of which can deliver non-string inputs. The "last wins" semantic for repeated flags is the right user mental model and matches yargs's own behavior for non-array fields. The `(null) → ''` test cell is the subtle regression anchor that locks the empty-string-fallback contract — without it, a future "simplification" to `String(input ?? '')` (which returns `''` for `null`) vs `String(input)` (which returns `'null'` for `null`) is a silent semantic change.
