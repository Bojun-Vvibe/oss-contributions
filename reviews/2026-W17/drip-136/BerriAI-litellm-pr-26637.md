## Verdict: merge-after-nits

## Rationale

- **Closes a real "callers need pricing/limits via /v1/models without two round-trips" workflow gap.** Today, the OpenAI-compatible `/v1/models` endpoint at `proxy_server.py:7025` returns only the canonical `{id, object, created, owned_by}` shape, forcing any client wanting pricing or context-window data to do a second `/model_group/info` round-trip per model — wasteful for dashboard / picker UIs that need to show pricing inline. The PR adds an opt-in `include_model_info: bool = False` query param at `:7028` that (when true) embeds the same data `/model_group/info` returns under a top-level `"model_info"` key per entry. Opt-in is the right default — the canonical OpenAI shape stays byte-compatible for any tooling that depends on the lean response.
- **Three call sites updated symmetrically.** `proxy_server.py:7113-7117` (the per-route lookup branch), `:7146-7150` (the catch-all listing branch), and `:7228-7232` (the single-model `/v1/models/{model_id}` endpoint at `model_info()`) all wire the new flag through to `create_model_info_response`. The `or False` defensive coercion at all three sites correctly handles the `Optional[bool] = False` FastAPI binding when the query param is absent vs `?include_model_info=` (empty).
- **The `try/except Exception` at `utils.py:5723-5731` is the right policy choice.** If `llm_router.get_model_group_info(model_group=model_id)` raises (router not initialized, unknown model_id, internal exception), the endpoint still returns 200 with `model_info: {}` rather than 500'ing the entire `/v1/models` listing. That fail-quiet-on-this-cell-only shape is the right contract for an additive enrichment field — the *base* response shape is load-bearing for OpenAI-compatible clients, and a single bad model row should not break the whole list. The `verbose_proxy_logger.exception(...)` at `:5726-5730` preserves the diagnostic so operators see *why* the lookup failed even though the response succeeded.
- **The `model_info: {}` sentinel for "no router / unknown model / lookup failed" is correct.** `:5733-5735` always emits the `model_info` key when `include_model_info=True`, even if the value is empty. That's the right shape: callers can write `entry["model_info"].get("input_cost_per_token", None)` without first checking `if "model_info" in entry` — the key's *presence* is the consumer-visible contract for "this response has the enriched shape", and the value's emptiness signals "lookup unavailable for this row".
- **Test coverage covers all five cells.** `test_proxy_utils.py:118-198` covers (a) flag-off → key absent + router not called, (b) flag-on with full ModelGroupInfo → all 9 fields flow through (mode/max_in/max_out/cost_in/cost_out/tpm/rpm/supports_fn_calling/supports_vision), (c) wildcard model with `None` group_info → `{}`, (d) router exception → `{}` (verifying the catch arm), (e) no router available → `{}`. That five-cell matrix locks the contract: shape is opt-in, value is fail-quiet, exceptions don't propagate, and unknowns don't differ from successes-with-no-data. The `model_dump(exclude_none=True)` at `:5734` is the right serialization choice — it omits Pydantic-`None` fields so consumers don't have to skip them.

## Nits / follow-ups

- **The `include_model_info` parameter is not added to the OpenAPI schema docstring of `model_list` in a way that propagates to the auto-generated client SDKs.** The added docstring at `:7042-7046` is human-readable but FastAPI's openapi.json generation pulls from the `Query(...)` signature and the function docstring — worth confirming the new param appears in the rendered Swagger UI and downstream clients can introspect it.
- **No proxy-FastAPI-TestClient end-to-end test.** The unit tests cover `create_model_info_response` directly but not the full HTTP path through `model_list` and `model_info`. A `TestClient.get("/v1/models?include_model_info=true")` test asserting the response shape contains `"model_info"` keys for at least one entry would lock the wiring at the entry-point layer where the `Optional[bool]` query-param coercion actually happens.
- **`include_model_info=true` widens the per-row response substantially** (potentially 9+ fields × N models). For deployments with hundreds of model groups configured, the response size could grow 10×. Worth a docstring note or a conservative default capping (e.g. emit only the most useful fields unless an additional `include_model_info=expand` value is passed).
- **The `Optional[bool] = False` + `or False` pattern at the three call sites** is repeated 3×. A small helper `def _coerce_flag(v: Optional[bool]) -> bool: return v or False` would dedup, though the inline form is clear enough.
- **Worth confirming `get_model_group_info` returns the same shape** the `/model_group/info` endpoint returns, so consumers can use the embedded data interchangeably with a separate `/model_group/info` call without serialization drift. The test at `:138-149` constructs a `ModelGroupInfo` and asserts the embedded `model_dump` round-trips, so the contract is locked at the type level — but a comment naming "this matches `/model_group/info` exactly" would be a useful pointer.

## What I learned

The "opt-in enrichment via flag" pattern for OpenAI-compatible endpoints is the correct way to extend a response without breaking byte-compatibility. The load-bearing decisions here are: (a) the flag defaults to false so existing tooling sees zero change, (b) the response key is *always* present when the flag is on (so consumers don't have to nil-check the key existence vs the value emptiness), (c) the lookup is fail-quiet at the per-row level so a single bad model can't 500 the whole list. The five-cell test matrix at `:118-198` is exactly the right shape for this kind of opt-in field — it locks the contract at every failure mode separately. Worth noting that the `verbose_proxy_logger.exception(...)` retention even on the success path (we still return 200, but log the diagnostic) is the canonical "fail quiet to caller, fail loud to operator" pattern, which is the right policy for consumer-facing endpoints with diagnostic-bearing failure modes.
