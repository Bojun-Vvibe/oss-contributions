## Scope

PR title says "Docs: Conflicting privacy information for Google AI Pro
subscribers regarding model training" — and the diff is supposed to address
that. What actually lands is a brand-new file at the repository root named
`tos-privacy.html` whose entire contents are 8 lines of *Markdown* (not
HTML) that read like an automated bounty submission template:

```
# Bounty Contribution

## Task: Docs: Conflicting privacy information for Google AI Pro subscribers regarding model training
**Reward: $20**
**Source: GitHub-Paid**
**Date: 2026-04-28 04:35:34.749868**

// Contributed for bounty: $20
```

The PR body (in Chinese) is the same template, declaring `$20` GitHub-Paid
reward and noting the file was "auto-generated by the bounty hunter system
+ Feishu notification" (`由赏金猎人系统自动生成 + 飞书通知`).

This is a low-quality automated submission and there's no actual content
change addressing the privacy-docs conflict in #20569.

## Specific findings

- `tos-privacy.html:1-8` — the file is named `.html` but the body is
  Markdown (`#`, `##`, `**bold**`, `// comment`). No `<html>`, no `<head>`,
  no DOCTYPE. Browsers would render this as a single line of plain text;
  static-site generators would not pick it up as either an HTML page or
  a Markdown page (wrong extension for the latter). The file is functionally
  inert in every consumer.
- The file is at the repository root, not under `docs/` or any documented
  privacy-docs location. There's no companion change to navigation,
  sitemap, or any existing privacy doc to link to it.
- The PR body claims `Closes #20569` but the file content does not
  describe, resolve, clarify, or even mention what the actual privacy
  conflict is. The bounty-template metadata is the entire payload.
- Sibling PR #26076 from the same author has an identical title, body
  shape, and presumably the same file pattern — this is a duplicate
  spam submission targeting a single bounty.
- There is no modification to any actual gemini-cli source, README,
  user-facing docs, code comments, error messages, or anything else
  that would change behavior or documentation as visible to a user.

## Risk

This PR provides zero value and adds a maintenance liability: a
misnamed orphan file at the repository root that future contributors
will have to either delete or correctly migrate. Merging it would
also normalize the bounty-spam pattern (file with metadata, no actual
content fix) and invite more of the same.

The original issue #20569 (conflicting privacy info for AI Pro
subscribers re: model training) is a real documentation problem that
deserves a real fix — typically an edit to the user-facing privacy doc
or a clarification in the auth/onboarding flow that reconciles the
conflicting messages. None of that work appears in this PR.

## Verdict

**request-changes** — close this PR (and sibling #26076) as not
addressing the linked issue. The actual privacy-docs conflict needs:
(1) a real diff to the user-facing privacy or onboarding docs, (2)
located under the documented docs path (not a misnamed orphan file at
repo root), (3) with content that actually describes which message is
correct and which is being corrected, (4) co-signed by someone who can
authoritatively reconcile the conflict (likely a maintainer with
access to product/legal review).

## What I learned

The "automated bounty submission" anti-pattern is recognizable from
three signals together: (1) PR body in a different language describing
the bounty system rather than the change, (2) file content that is
metadata about the bounty rather than a fix for the underlying issue,
(3) a near-duplicate sibling PR from the same author. None alone is
disqualifying — non-English contributions are welcome, metadata files
exist legitimately, and resubmissions happen — but all three together
are diagnostic of an LLM-driven bounty harvester and should be closed
without further engagement.
