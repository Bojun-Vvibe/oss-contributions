# PR: fix(provider): preserve Azure API version

## Summary

When a user configures the Azure provider with a custom `baseURL` (i.e. Cognitive Services Responses API at `…/openai`), the `@ai-sdk/azure` SDK normally appends `?api-version=…`, but a custom baseURL bypasses that path. The result was a request that 400'd because Azure requires `api-version` on every Cognitive Services call. This patch wraps the URL on the way to `fetchFn`, and if the provider is `@ai-sdk/azure` and an `apiVersion` option is set and the URL doesn't already carry `api-version`, it appends the query parameter. A new integration test in `llm.test.ts` runs a full streaming `/responses` call against a fixture and asserts the captured request URL has `api-version=2025-04-01-preview`.

## Specific observations

- `packages/opencode/src/provider/provider.ts:1474-1486` — the IIFE pattern is fine, but the chain `input instanceof URL ? new URL(input.href) : typeof input === "string" && URL.canParse(input) ? new URL(input) : undefined` silently no-ops on `Request` objects. `fetch` will accept a `Request`, and `@ai-sdk/azure` may at some point start passing one (it currently passes `string` URLs, but this is an undocumented contract). Either narrow the type at the call site or add an explicit `instanceof Request` branch that pulls `input.url` and re-constructs.
- `packages/opencode/src/provider/provider.ts:1480` — `if (url.searchParams.has("api-version")) return input` — good, this prevents double-stamping. Worth a comment that this is intentional so a user-supplied baseURL that already carries `?api-version=…` wins. A failing test that exercises this path (user already set `api-version` in baseURL → unchanged) would lock the precedence in.
- `packages/opencode/src/provider/provider.ts:1483-1485` — the return reconstructs `URL` vs `string` based on the input type. Good. Nit: if `input` is a `string` that round-trips through `new URL(input).href`, the resulting string may have normalized differently (trailing slash, lowercased host, default-port stripping). For Cognitive Services that's fine; for a user pointing at a quirky reverse proxy it's a minor surprise. Returning `url.href` is the right call but consider documenting it.
- `packages/opencode/src/provider/provider.ts:1476` — the predicate is `model.api.npm !== "@ai-sdk/azure"`. If a user adds a fork of the Azure adapter under a different npm name, this gate skips them silently. Future-proof would be a registry-style flag on the model.api descriptor (`requiresApiVersionParam: true`); not required for this PR but worth flagging in the description as a known limitation.
- `packages/opencode/test/session/llm.test.ts:678-779` — the test calls `loadFixture("azure-cognitive-services", "gpt-5.4-mini")`, builds an opencode.json with `apiVersion: "2025-04-01-preview"` and a custom `baseURL`, drains a streamed response, and asserts `capture.url.searchParams.get("api-version")` matches. The negative case ("non-Azure provider, apiVersion option present, URL is not modified") is not covered; given the `model.api.npm !== "@ai-sdk/azure"` guard, regression coverage there would catch a future refactor that loosens the predicate.

## Verdict

`merge-after-nits`

## Reasoning

The fix targets a real interoperability gap (custom baseURL + Cognitive Services Responses API + api-version requirement), the implementation is small and the integration test exercises the streaming path end-to-end. The blocking nit before merge is the `Request` input branch — `fetchFn` officially accepts `Request | URL | string`, and silently letting `Request` through unmodified will be a head-scratcher the day someone enables it. A negative test case for the non-Azure path would round out the test matrix. Neither warrants `request-changes` on its own; together they justify a polish round.
