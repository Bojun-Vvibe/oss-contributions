## Summary

When a user attaches an image whose MIME type is outside the four formats almost every multimodal model accepts (`jpeg`, `png`, `gif`, `webp`), the new `unsupportedImageMime` check in `provider/transform.ts` replaces the part with an inline error-text part instructing the model (and via it the user) about the unsupported format. Previously the part was passed through to the provider SDK which would either reject the request or, worse, silently drop the image and continue with degraded context.

## Verdict: `merge-after-nits`

Right intent and the implementation is local and self-contained, but the supported-set is hard-coded to four formats which is more conservative than several of the actual upstream providers (Anthropic, OpenAI, and Gemini all accept additional formats — for instance HEIC via Anthropic, BMP via several providers). The `unsupportedParts` function already has a `model.capabilities.input[modality]` check downstream; pushing the supported-MIME list into the per-model capability descriptor (or at least making it overridable) would avoid silently breaking users on providers that *do* support, say, `image/heic`.

## Specific references

- `packages/opencode/src/provider/transform.ts:9` — `SUPPORTED_IMAGE_MIME_TYPES = new Set(["image/jpeg", "image/png", "image/gif", "image/webp"])`. The `Set` is module-scoped which is fine, but it bakes the assumption that this set is the same across **every** provider opencode talks to. That is currently true in practice but will rot.
- `packages/opencode/src/provider/transform.ts:18-20` — `unsupportedImageMime` rightly does `.toLowerCase()` on the incoming MIME, which handles client SDKs that emit `Image/JPEG` etc. Good.
- `packages/opencode/src/provider/transform.ts:309-315` — the replacement text mentions "Inform the user.", which is a useful hint for the model. The MIME is interpolated into the message twice (`${mime}` after the parenthesis and `format ${mime}`). Reads slightly awkwardly; consider `(unsupported image format ${mime})` once and dropping the trailing repeat.
- `packages/opencode/test/provider/transform.test.ts:1058-1095` — the two new tests cover both the `image` and `file` part shapes and assert the exact error string. Good regression lock-in.

## Nits

1. Move the supported set into `Provider.Model.capabilities.input.image_formats?: string[]` (optional; defaults to the current four), so future model entries can opt in to additional formats without code changes.
2. The error text duplicates `image/avif` ("Cannot read image/avif (unsupported image format image/avif)..."). Tighten to `Cannot read this image (unsupported format image/avif). ...` so the assistant's eventual user-facing rephrase reads naturally.
3. Consider hoisting the same gate to the `audio/` and `video/` modalities — the bug (silent drop / cryptic provider error) is identical for unsupported audio types.
