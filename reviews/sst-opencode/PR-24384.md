# PR: fix(provider): respect configured output limit

- URL: https://github.com/sst/opencode/pull/24384

- State: OPEN  (+39 / −1, 2 files)

## Summary

`maxOutputTokens` previously did `Math.min(model.limit.output, OUTPUT_TOKEN_MAX) || OUTPUT_TOKEN_MAX`, which silently capped any configured `model.limit.output` at the global `OUTPUT_TOKEN_MAX` ceiling and also fell back to the global on a `0` value via the `||` short-circuit. Users who configured a model with an output budget greater than the default ceiling (e.g. 65 536 against a 32 768 default) were getting their config silently truncated. The new expression `model.limit.output > 0 ? model.limit.output : OUTPUT_TOKEN_MAX` honors the configured value when present and only uses the fallback when the user explicitly omitted it.

## Specific observations

- `packages/opencode/src/provider/transform.ts:1037-1038` — old: `Math.min(model.limit.output, OUTPUT_TOKEN_MAX) || OUTPUT_TOKEN_MAX`. The `Math.min(...)` was load-bearing in a way that the `|| OUTPUT_TOKEN_MAX` guard masked; removing it changes the behavior for users whose configured limit exceeds `OUTPUT_TOKEN_MAX`. That is in fact the intent ("respect configured output limit"), but the PR description should call out that this is a behavior change for any provider whose model card listed an output cap above the default — the user-facing token budget will now be larger by default for those providers.
- `packages/opencode/src/provider/transform.ts:1037` — there is no upper bound on `model.limit.output` anymore. If a misconfigured or hostile model card declares `limit.output: 1_000_000`, that value flows straight through. For provider SDKs that 400 on impossible budgets this is fine; for ones that silently allocate based on the requested budget it could cost real money. Worth at least a sanity ceiling (e.g. clamp to `Number.isFinite(...) && x > 0 && x <= ABSOLUTE_MAX`).
- `packages/opencode/src/provider/transform.ts:1037` — the `> 0` predicate matches `0`, `NaN`, negative, and `undefined` to the fallback branch. Good. But if `limit.output` arrives as the string `"65536"` (e.g. from a hand-written JSON model card), `"65536" > 0` is `true` after coercion and the string is then returned and presumably handed to the SDK as a token count. A `Number(...)` wrap or a runtime `typeof === "number"` check would defend against that.
- `packages/opencode/test/provider/transform.test.ts:5-44` — two tests: configured limit > fallback, and `output: 0` falls back. The "configured limit *less than* the default" case is missing; given the old code did `Math.min`, regression coverage for "small configured limit is preserved, not bumped up to the default" would directly exercise the previous foot-gun.

## Verdict

`merge-after-nits`

## Reasoning

The fix is correct and the previous expression was a real bug — `Math.min` against a global ceiling silently overrode the user's configured budget. The two nits worth tightening before merge are (a) one more test case for "configured limit smaller than the default is honored" and (b) a defensive numeric cast to avoid a stringly-typed `limit.output` flowing through. The ceiling-removal is the intended behavior change but should be in the PR description so deployers are not surprised by larger requests post-upgrade.
