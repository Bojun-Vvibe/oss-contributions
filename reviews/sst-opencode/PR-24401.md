## Summary

Two defensive nullish-guards on tool-output handling: `truncateToolOutput` now returns `""` when `text` is falsy, and `Truncate.output` short-circuits to `{ content: "", truncated: false }` when `text == null`. Fixes a `text.split is not a function` / `Cannot read properties of undefined` crash when an MCP tool returns an entry whose `state.output` is `undefined`.

## Verdict: `merge-as-is`

A two-call-site, three-line patch addressing a real runtime crash. The `?? ""` at the call site (`message-v2.ts:854`) plus the early return inside `truncateToolOutput` (`message-v2.ts:321`) is belt-and-braces but cheap, and the `text == null` (loose equality) in `truncate.ts:87` correctly catches both `null` and `undefined`.

## Specific references

- `packages/opencode/src/session/message-v2.ts:321` — `if (!text) return ""`. This will also short-circuit on the empty string, which is fine because the existing path `text.length <= maxChars` would also have returned `text` (i.e. `""`) — net behaviour identical.
- `packages/opencode/src/session/message-v2.ts:856` — `truncateToolOutput(part.state.output ?? "", options?.toolOutputMaxChars)`. The `?? ""` is redundant given the guard inside `truncateToolOutput`, but it keeps the call site self-documenting against the schema where `output: string` (not `string | undefined`).
- `packages/opencode/src/tool/truncate.ts:87` — `if (text == null) return { content: "", truncated: false } as const`. Good — uses `==` to match both `null` and `undefined`, and the `as const` keeps the return shape narrow for downstream consumers.

## Nits

1. The schema for `ToolStateCompleted.output` should probably be tightened to `string` (with a default of `""` at the producer side) so this class of bug becomes a type error rather than a runtime check. Out of scope for this PR but worth filing.
2. Consider adding a one-liner unit test that calls `truncateToolOutput(undefined as any, 100)` — purely to lock the contract in the test file.
