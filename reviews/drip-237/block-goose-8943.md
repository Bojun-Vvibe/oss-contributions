## Context

Starting a new chat session inside a project required the user to wait on a synchronous chain of `resolveSupportedSessionModelPreference(...)` (provider-inventory probe) â†’ `acpPrepareSession(...)` (round-trip to the agent service) â†’ `createSession(...)` (server write) before the UI flipped to the chat view. Net latency: hundreds of ms per click on a cold provider inventory. This PR splits the project-session path into an *optimistic local-create* arm â€” `createLocalSession(...)` populates the store + flips the active view immediately, with the ACP binding deferred to a `prepareSessionBinding(...)` call that runs as the user starts typing or selects a model.

## Risks / nits

- **Optimistic-local sessions appear in the UI before any server-side commit.** If the user closes the window between `createLocalSession` and the eventual `prepareSessionBinding`, the local session is orphaned in client state. Worth pinning the cleanup discipline: either (a) the local session has a TTL and is GC'd if not bound within N seconds, or (b) the next app-startup reconciles unbound local sessions against the server. The PR doesn't address this â€” worth at minimum a code comment in `chatSessionStore.ts:createLocalSession` flagging the orphan possibility for follow-up.
- **`prepareSessionBinding` is called from at least three sites** (`AppShell.tsx:494`, `useChatSessionController.ts:172`, the project-change handler at `:328`). Worth a one-line code comment at the store-level definition pinning the contract: "Call this on the first user action that needs the ACP binding (model select, message send, project working-dir change). Idempotent on already-bound sessions."
- **The `knownNew: true` flag** is set in the binding call at `useChatSessionController.test.ts:284` but the production-side code path that decides when to set it isn't visible in this diff (the relevant block is in `chatSessionStore.ts:prepareSessionBinding` which has +123/-3 in the file summary but the diff is truncated). Worth confirming the production code uses `!session.acpSessionId` as the predicate (i.e. "we never bound this session before") rather than something fragile like `Date.now() - session.createdAt < 1000`.
- **`acpSessionTracker.ts` has +24/-17.** Not visible in the truncated diff but presumably this is where the local-session-id â†’ acp-session-id mapping is maintained. Worth confirming the tracker handles the race where two tabs/windows of the same client race to bind the same local session.
