## Summary

`Config::get_param` previously deserialised the YAML value through `serde_yaml::from_value` and surfaced the error directly. That meant a YAML string like `XXX_TIMEOUT: "300"` (which users frequently produce when round-tripping configs through env-var or shell-quoting paths) would fail to deserialise into `u64`. The fix adds a fallback: when YAML deserialisation fails and the value is a string, run it through `parse_env_value` (the existing env-var parser) and try `serde_json::from_value` against that. The original YAML error is preserved on the second-stage failure so error messages stay accurate.

## Verdict: `merge-as-is`

Backwards-compatible widening of an input contract. Test coverage is symmetric (numeric → u64, quoted-numeric → u64, quoted-numeric → String, invalid string → still errors), and the error mapping correctly returns the *original* YAML error rather than the JSON one, which is the more user-actionable message.

## Specific references

- `crates/goose/src/config/base.rs:735-747` — the new fallback branch. The `let Some(string_value) = value.as_str() else { return Err(yaml_err.into()); }` pattern is the right shape: if the value isn't even a string, the JSON detour cannot help, so emit the original YAML error untouched.
- `crates/goose/src/config/base.rs:743` — `serde_json::from_value(parsed).map_err(|_| yaml_err.into())` discards the JSON error in favour of the YAML one. That's deliberate and good for the common case (`"abc"` parsed to `String("abc")`, then JSON-deserialise to `u64` fails — surfacing the YAML error "expected u64, got string" is more useful than JSON's "invalid type"). Worth a brief comment noting that the JSON error is discarded *intentionally*.
- `crates/goose/src/config/base.rs:1177-1224` — four new tests, including the subtly important `test_get_param_reads_quoted_numeric_yaml_as_string` which guards against an over-eager numeric coercion that would break users who genuinely want their `"300"` to stay a string.

## Nits

1. The discarded JSON error in `map_err(|_| yaml_err.into())` deserves a one-line `// JSON error here is less informative; surface the YAML error.` comment.
2. Consider also handling the inverse: a YAML number that the user wanted as a string. Less common, but symmetric. (Probably fine to defer.)
3. `parse_env_value` is now a load-bearing dependency of the typed-config path, not just the env-override path. Worth a doc-comment on `parse_env_value` itself noting that it has two callers with different error-handling expectations.
