## Summary

Bedrock's `ContentBlock` enum carries a `ReasoningContent` variant that some models (`openai.gpt-oss-safeguard-120b` and other gpt-oss models) include in responses. The previous `from_bedrock_content_block` had a catch-all `_ => bail!("Unsupported content block type from Bedrock")` arm, so any reasoning block aborted the whole response. This PR maps `ReasoningContent::ReasoningText` → `MessageContent::Thinking`, `ReasoningContent::RedactedContent(blob)` → `MessageContent::RedactedThinking` (base64-encoded), and changes the function signature to `Result<Option<MessageContent>>` so unknown SDK-future variants can be skipped without aborting. The reverse direction (`to_bedrock_message_content`) stops dropping `Thinking`/`RedactedThinking` as empty text and replays them verbatim to satisfy Bedrock's reasoning-replay contract.

## Verdict: `merge-as-is`

This is a correct, well-structured fix to a real production bug, with strong test coverage and a careful design choice on the `Unsupported`/`Unknown` boundary that avoids silent truncation.

## Specific references

The `if !thinking.signature.is_empty() { builder = builder.signature(&thinking.signature); }` guard is right: the SDK builder errors if you pass an empty signature. The base64-decode-then-fall-back-to-skip behaviour for `RedactedThinking` (line 67-79) is the right call given that other providers (Anthropic) treat `data` as an opaque payload that may or may not be base64; aborting the whole request on a non-base64 payload would be worse than the existing skip behaviour, and the `tracing::warn!` makes it observable.
