## Summary

Replaces `expect("Failed to read permission.yaml")` + `unwrap_or_else(|e| panic!(...))` in `PermissionManager::new` with nested `match` expressions that log the error via `tracing::error!` and fall back to an empty `HashMap`. Updates the existing `#[should_panic]` test to assert the new graceful-fallback behaviour instead.

## Verdict: `merge-after-nits`

Right direction (process should not crash on disk corruption) and the code is clean, but the policy change deserves a louder signal than `tracing::error!` because falling back to "empty permissions" silently strips every prior tool-approval the user has granted. A user who has approved 50 tools and then one byte of `permission.yaml` flips on disk will be re-prompted for all 50 with no UI hint that anything went wrong.

## Specific references

- `crates/goose/src/config/permission.rs:46-69` — the new nested `match`. Structurally correct; the inner `match serde_yaml::from_str(&contents)` is the right place to recover from corruption (the most likely failure mode), and the outer `match fs::read_to_string` covers the rarer file-permission / I/O failure. The `tracing::error!` calls are properly structured (`path = %permission_path.display(), error = %e`), which is good practice and lets operators grep for these.
- `crates/goose/src/config/permission.rs:67` — the message `"Failed to read permission config — starting with empty permissions."` is informational but does not communicate the security implication: **every prior tool approval is now revoked for this session**. Consider either (a) renaming the file to `permission.yaml.corrupt-<timestamp>` and creating a fresh empty one (preserves forensic evidence + makes the recovery path obvious), or (b) refusing to start *write operations* until the user re-approves, even though startup itself proceeds.
- `crates/goose/src/config/permission.rs:75` — `fs::create_dir_all(&config_dir).expect("Failed to create config directory")` is **left as `expect`**. Inconsistent with the new policy: if reading a corrupt config shouldn't crash, neither should an EROFS / EACCES on the config directory. Either both are fatal or neither is.
- The test `test_corrupted_permission_file_falls_back_to_empty` (line 363+) covers the YAML-parse-error branch but not the `fs::read_to_string`-error branch (e.g. file exists but is unreadable due to mode `000`). The latter is the harder one to repro on Windows but is trivial on Unix with `fs::set_permissions`.

## Nits (optional)

1. Both error paths return `HashMap::new()`. Extract a `EMPTY_PERMISSIONS_FALLBACK` constant or a `recover_with_empty(reason: &str, …) -> HashMap<...>` helper so the two arms can't drift.
2. The `tracing::error!` level is appropriate; consider also emitting a one-line `eprintln!("warning: permission config at {} is corrupt; starting with empty permissions. See logs.", path.display())` so users running without structured logging actually see the message in their terminal.
3. PR title is good (`fix: gracefully handle corrupted permission config instead of panicking`) — please carry the same wording into the commit subject if rebasing for merge.
