## What it does

Dependabot bump of the `bzip2` crate (used by `goose-cli` for
archive decompression). The 0.6.0 release notes flag this as the
release that **switched the default backend from the C `bzip2-sys`
shim to the pure-Rust `libbz2-rs-sys` implementation**. 0.6.1 is a
patch on top.

## Diff notes

- `crates/goose-cli/Cargo.toml:54` — `bzip2 = "0.5"` → `bzip2 = "0.6"`.
  Caret bump.
- `Cargo.lock`:
  - `bzip2` 0.5.2 → 0.6.1, `dependencies` array now `["libbz2-rs-sys"]`
    instead of `["bzip2-sys"]`.
  - `bzip2-sys 0.1.13+1.0.8` is **removed entirely** (no other
    dependent in the graph).
  - New crate: `libbz2-rs-sys 0.2.3`.
  - `cc` and `pkg-config` build-time deps from `bzip2-sys` go away —
    cross-compilation gets simpler, no system `libbz2` required.

## Concerns

1. **Backend switch is the real change, not the version number.** The
   PR description (auto-generated by Dependabot) buries this: 0.6.0
   replaces a battle-tested ~25-year-old C library with a Rust port
   from `trifectatechfoundation`. Goose uses bzip2 for installer /
   archive paths (per the `tar = "0.4.45"` and `zip` neighbors in
   the same `Cargo.toml`). Risk surface = decompressing untrusted
   archives. Recommend running goose's existing archive-handling
   tests against a corpus of known-good and known-malformed bzip2
   files before merging — `libbz2-rs-sys` is young (0.2.x) and may
   handle edge cases differently than `libbz2`.
2. **License change worth flagging.** Per the upstream notes,
   `libbz2-rs-sys` carries the `bzip2-1.0.6` SPDX license. If goose
   uses `cargo-deny` or any license allowlist in CI, that license
   may need to be added explicitly. Check `.cargo-deny.toml` /
   `deny.toml` in the repo before merging.
3. **MSRV bump to 1.82.0.** The 0.6.0 release ships an MSRV
   requirement of Rust 1.82. Verify goose's CI toolchain is at or
   above 1.82 — if `rust-toolchain.toml` pins anything older, this
   bump will break the build.
4. The `bzip2-sys` removal means goose binaries no longer dynamically
   link against the system `libbz2.so` / `libbz2.dylib`. Net positive
   for distribution, but if any downstream packaging script lists
   `libbz2` as a runtime dep, it can be dropped.

## Verdict

merge-after-nits
