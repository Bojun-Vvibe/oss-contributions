# fix: model picker stays usable during provider loading

Two-pronged fix to a UX deadlock where the goose2 model picker disabled itself entirely while a background provider-inventory refresh was in flight, leaving users with no way to switch agents/models during the (multi-second on slow connections) refresh window.

(1) **Lift the inventory-refresh logic out of the picker component into a shared API.** Deletes the 23-line `useEffect`-driven `syncInventory` block inside `AgentModelPicker.tsx:357-378` (which fired `getProviderInventory()` on every popover open) and the helper imports `getProviderInventory` / `useProviderInventoryStore` at `:10-11`. Replaces it with a new `backgroundRefreshInventory(inventoryStore, initialEntries?)` exported helper at `features/providers/api/inventory.ts:33-72` that does the right "merge first then refresh" sequence: if no `initialEntries` supplied, fetch+merge the snapshot first so the UI sees fresh data even when the upstream-refresh RPC returns no `started` entries; if `initialEntries` supplied, skip the merge (caller already did it) and proceed straight to the refresh. The startup-path caller at `useAppStartup.ts:97-106` is the "with initialEntries" arm; the new picker-open caller in `useAgentModelPickerState.ts:166-183` is the "without initialEntries" arm.

(2) **Decouple the picker's disabled state from the loading flag.** At `AgentModelPicker.tsx:331`, `disabled={loading}` becomes `disabled={loading && !selectedAgentLabel}` — i.e. the picker disables only when *both* loading is in flight *and* there's no resolved agent label to display. Paired with the trigger-text fallback at `:340-343` flipping from `loading ? t("toolbar.loading") : (triggerModelLabel ?? selectedAgentLabel)` to `triggerModelLabel ?? selectedAgentLabel ?? (loading ? t("toolbar.loading") : null)`. The contract becomes "if we have a selected agent label, render it and stay clickable, regardless of background-refresh state" — which is the correct UX because the user's model-switching action is meaningful against the cached inventory even while the refresh is pending. The new "loading inside the popover" branch at `:491-512` adds a placeholder `PickerItem` showing the currently-selected model with an inline `Spinner` so the user sees "we're refreshing, but here's what you have right now" rather than a blank "loading…" spinner that drops the current selection from view.

The new `backgroundRefreshInventory` is paired with a 97-line vitest at `inventory.test.ts:1-97` covering three contract arms: (a) no-providers-configured returns after merge without invoking refresh, (b) refresh-with-no-started returns after merge+refresh-call, (c) initialEntries-supplied skips the redundant fetch+merge. The `refreshingRef` guard in `useAgentModelPickerState.ts:166-183` prevents duplicate-refresh-on-rapid-popover-toggle by gating on both the local ref *and* the store's `loading` flag — the right shape (component-local ref handles the "this component is mid-refresh" case, store flag handles the "another component already kicked off a refresh" case).

Three nits worth flagging but small enough not to gate:

(1) `disabled={loading && !selectedAgentLabel}` is the right behavioral shape but reads as a bool inversion puzzle — `disabled={!selectedAgentLabel && loading}` (predicate reordered to "no selection AND still loading") is the same logic with the more-readable mental model. A comment naming the contract ("keep clickable when we have a cached selection to show, even while refreshing") at the line would help the next reader more than the reorder alone.

(2) The new `handlePickerOpen` callback threads through five files (`useAgentModelPickerState` → `useResolvedAgentModelPicker` → `useChatSessionController` → `ChatInput`/`ChatInputToolbar`/`ChatView`/`HomeScreen`/`AgentModelPicker`). That's the standard prop-drill shape for callbacks owned by a controller and consumed at a leaf component, but eight files of mechanical `onPickerOpen={controller.handlePickerOpen}` threading is the kind of thing that grows unbounded — a context provider for picker callbacks would scale better past ~3 callbacks. (Not actionable for this PR, just flagging the shape.)

(3) The `refreshingRef.current = true` / `false` guard handles the duplicate-on-rapid-toggle case but doesn't *cancel* an in-flight refresh when the popover closes mid-fetch. Effect-style components that set `cancelled = true` on cleanup would have stopped the merge — this PR's ref-guard lets the merge complete and writes to the store after the popover is closed. That's almost certainly correct (you want the cached data to be fresh for the next open), but it deserves a one-line comment naming the choice.

## what I learned
"Disable the UI when state is loading" is the lazy default for async-derived UI, and almost always wrong when the loading is a *refresh* of state the user already has a usable cached version of — the correct contract is "stay interactive against cached state, show a non-blocking refresh indicator." The shape this PR takes (decouple `disabled` from `loading`, render cached selection even during refresh, show a contextual spinner for the in-flight portion) is the textbook fix; the nit-level question is just whether the refresh-trigger logic belongs in the leaf component (where this PR moved it *out* of) or in a controller hook (where this PR moved it *into*) — the controller-hook answer is correct because the same refresh might want to be triggered from elsewhere later, but it does come with the prop-drill cost flagged above.
