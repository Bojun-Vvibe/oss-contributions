# openai/codex#21278 — Move message history out of core

- **Head SHA**: `69c15d5df784297868cc9a99876e31d0ae3f2bd1`
- **Stats**: +376 / -493, ~25 files

## Summary

Extracts the cross-session JSON-Lines message-history feature from `codex-core` into a freestanding `codex-message-history` workspace crate, and lifts the two `Op` variants (`AddToHistory`, `GetHistoryEntryRequest`) out of the `protocol` crate's core-protocol enum into app-server-level `AppEvent::AppendMessageHistoryEntry` / `AppEvent::LookupMessageHistoryEntry` events. The motivation is the broader "thin core" refactor — message history is a TUI-side persistence concern, not a model-execution concern, and pinning it inside `codex-core` made `core` depend on tokio-fs/tempfile/serde_json transitively.

## Verdict

**request-changes**

## Rationale

The crate-extraction itself is clean and idiomatic Rust workspace surgery — the dep slimming, BUILD.bazel hookup, and `AppEvent` rerouting are all done right. The `protocol::Op` deletion is defensible *if* the project's policy treats those Ops as internal-only. **What blocks merge is the silent on-disk-format change** at `message-history/src/lib.rs:8,419`: the JSON record key flips from `conversation_id` to `session_id` with no `#[serde(alias = "conversation_id")]` on the corresponding struct field and no migration code. Every existing user's `history.jsonl` becomes unreadable on first turn after upgrade — `lookup_entry` will silently return `None` for the entire pre-upgrade history, and `append_entry` will write records the new code can read but the old code cannot (a downgrade-hostile mix). The fix is one of (a) keep the wire field as `conversation_id` even though it semantically holds a session id (matches the original code's stated convention "field name is preserved for backwards compatibility" — that comment is *deleted* in this diff at line 363, indicating the author considered and rejected compat), (b) add `#[serde(rename = "session_id", alias = "conversation_id")]` on the struct field plus a one-shot rewrite-on-read shim, or (c) ship a tiny migration that rewrites the file once on startup. Without one of these, a year of users' history vanishes silently after the upgrade. Secondary concerns (public-API breakage on `protocol::message_history::HistoryEntry`, `Op` removal) are real but downstream of the format-compat decision.
