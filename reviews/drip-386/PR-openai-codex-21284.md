# openai/codex#21284 — Add response.processed websocket request

- **Head SHA**: `92810dc975f459339b2fc409d3efa52b254fae2e`
- **Stats**: +232 / -29, 7 files

## Summary

Adds a new fire-and-forget `response.processed` frame on the Responses-API websocket transport, sent by the client *after* a turn's `response.completed` event has been fully processed (rollout written, hooks fired, additional contexts recorded). The intent — judging from the feature gate and the new test coverage — is to let the server-side know exactly when the client has durably absorbed a response, distinguishing "stream ended" from "turn-state mutated and persisted." This unblocks server-side state cleanup that today has to be inferred from connection idle time.

## Rationale

Clean, narrowly-scoped feature behind an `UnderDevelopment` flag with `default_enabled: false`, so there's zero risk to existing rollouts. Wire format, schema, feature registry, and tests all stay in sync. The shared `send_websocket_request` helper extraction is a legitimate refactor improvement that the diff carries for free. Three nits worth squashing: (1) `client.rs:893-906`'s `debug!` swallow should escalate to `warn!` on first failure per session — a permanently-silently-broken processed-ack would be invisible in production logs but is exactly the kind of thing a server-side cleanup feature would want to know about; (2) the test at `client_websockets.rs:182-189` hard-codes `wait_for_request(/*connection_index*/ 0, /*request_index*/ 2)` with the prewarm at 0 and create at 1 — fine for now, but a regression in prewarm semantics would shift the index; consider matching by frame `type` instead; (3) the empty `vec![]` third turn slot in the test at line 173 is a server-side template artifact — a comment naming why it's needed (the post-processed close marker?) would help future readers.
