# BerriAI/litellm#27265 — Fix Prometheus remaining rate limit gauges

- **Head SHA**: `05ef215b5a335db66e5cb1ba28cefe201844cc64`
- **Stats**: +156 / -5, 2 files

## Summary

`PrometheusLogger._set_virtual_key_rate_limit_metrics` previously read `litellm-key-remaining-{requests,tokens}-{model_group}` only from the request `metadata` dict. For paths where the upstream provider returns the per-key remaining counts in `x-ratelimit-model_per_key-remaining-{requests,tokens}` HTTP response headers (and they get stashed into `standard_logging_object.hidden_params.additional_headers` rather than into `metadata`), the gauge fell through to `sys.maxsize`, making the dashboard panel useless for those providers. This PR adds a metadata→header→`sys.maxsize` precedence chain plus an int-coercion step (the headers come back as strings).

## Specific citations

- `litellm/integrations/prometheus.py:1397-1402`: new `_coerce_prometheus_gauge_value(value)` static — `int(value)` wrapped in `try/except (TypeError, ValueError)` falling back to the raw value. The `except` branch *returns the original value*, not `sys.maxsize`, which is a deliberate choice: if a provider returns a float (`"12.5"` → `int("12.5")` raises `ValueError` → returns `"12.5"`), the gauge `.set()` call downstream will raise instead of silently posting nonsense. That's the right trade-off, but the test at `test_prometheus_virtual_key_rate_limits.py:33-36` only asserts the int-success path with `"12"` and `"345"`. Worth one more case asserting the `set()` raise on a non-coercible string, so a future refactor doesn't accidentally swallow it.
- `litellm/integrations/prometheus.py:1404-1421`: new `_get_virtual_key_rate_limit_value(metadata, metadata_key, additional_headers, header_key)` static. Precedence is metadata-not-None → headers-not-None → `sys.maxsize`. The `if metadata_key in metadata and metadata[metadata_key] is not None` guard at line 1410 correctly handles the **zero-remaining case** — `metadata["litellm-key-remaining-requests-gpt-4"] = 0` is truthy-falsy with `or`, but the membership-and-not-None test preserves it. This is exactly the bug the older `metadata.get(name, sys.maxsize) or sys.maxsize` shape (line 1421 in pre-diff) had: `0 or sys.maxsize == sys.maxsize`, silently corrupting any "rate-limit exhausted" gauge to "infinite headroom." The new test at `:62-90` (`test_virtual_key_rate_limit_metrics_preserve_zero_remaining_values`) pins this regression.
- `litellm/integrations/prometheus.py:1448-1466`: call-site refactor. Pulls `standard_logging_object` from `kwargs`, navigates `hidden_params.additional_headers` defensively (the `isinstance(hidden_params, dict)` guard at line 1453 protects against the field being `None` or a Pydantic model that hasn't been .dict()'d). Then dispatches the two helper calls with the canonical header keys `x-ratelimit-model_per_key-remaining-requests` / `-remaining-tokens`. The header keys exactly match what `additional_headers` is populated with elsewhere in litellm — verifiable via `git grep "x-ratelimit-model_per_key-remaining-requests"` (not done in this review, but worth a sanity-check at merge time).
- `tests/test_litellm/integrations/test_prometheus_virtual_key_rate_limits.py:1-107` (new file). Three tests: (a) headers-path string-coercion (`"12"` → 12), (b) zero-remaining preservation, (c) missing-everywhere → `sys.maxsize`. The fixture at lines 8-22 unregisters every collector in the global `REGISTRY` both pre- and post-yield, working around `prometheus_client`'s "duplicate timeseries" sticky-collector behavior in pytest. That's a pragmatic but slightly heavy-handed reset — it'll silently break any *other* test in the same session that relies on a specific Prometheus collector being pre-registered. If the test suite is run in isolation per-file, this is fine; in shared collection (`pytest tests/`), check for ordering flakiness.

## Verdict

**merge-as-is**

## Rationale

Real bug, correct fix, three pinning tests including the load-bearing zero-remaining preservation case. The metadata→header→default precedence is the right ordering — metadata reflects pre-request limits the proxy itself enforced, headers reflect what the upstream actually has remaining, and they should agree but if they disagree, the proxy's own bookkeeping wins. The int-coercion fallback to raw-value (rather than to `sys.maxsize` on parse failure) is a defensible "let it raise" call, though I'd prefer a `warning` log at that branch so misconfigured providers surface in proxy logs rather than as a `set()` `TypeError` later. The test fixture's wholesale `REGISTRY` unregister is the only thing I'd reconsider — a `CollectorRegistry()` instance scoped to the `PrometheusLogger` would let tests run in parallel without trampling each other — but that's a wider refactor than this PR. Ship it.
