## Specific observations

- `hook_auto_review.rs:96-104` — for each candidate hook the reviewer runs sequentially (`for hook in candidates`). For a user with 10 untrusted hooks at default `GUARDIAN_REVIEW_TIMEOUT` apiece, this serializes wall-clock time on the critical-path of turn start. At minimum bound the candidate set; ideally `tokio::join!`/`buffer_unordered` with a small concurrency cap (eg. 3). The `HookAutoReviewStartedEvent` already broadcasts `hook_count` so the UI can show progress, but if 10 hooks each cost 30s the user waits 5 minutes silently.
- `hook_review.rs:1493-1567` — `review_hook_for_security` rebuilds the entire session config (`build_guardian_review_session_config`), including reading `network_proxy.current_cfg().await`, on every single hook review. The `live_network_config` snapshot is fine to recompute, but `available_models = list_models(RefreshStrategy::Offline)` and the `preferred_model` selection are pure functions of the model registry — cache the `(review_model, review_reasoning_effort)` tuple once per `maybe_run_hook_auto_review` call instead of per-hook.
- `hook_review.rs:1432-1450` — `HOOK_REVIEW_POLICY` instructs the reviewer to "inspect that file too using read-only tools. Do not execute the hook command or any referenced script." This is policy text only; there is **no enforcement**. The `run_codex_thread_one_shot` call doesn't appear to scope tool availability — confirm whether the spawned guardian thread inherits the user's full toolset (`exec`, `apply_patch`, network access) or runs under a hardened sub-agent profile. If full toolset, a sufficiently confused model could be tricked by a prompt-injected hook source into running it.
- `hook_review.rs:1565-1568` — `tokio::time::timeout(GUARDIAN_REVIEW_TIMEOUT, ...)` — on timeout the function returns the `Err` arm without cancelling the spawned codex thread. `cancel_token.clone()` is created at line 1547 but I don't see a `cancel_token.cancel()` on the timeout path. Result: the in-flight review thread continues consuming model tokens after the wrapper has given up. Cancel explicitly on timeout.
- `hook_auto_review.rs:131-147` — `apply_result` failure path zeroes `trusted_count` / `dangerous_count` and adds them to `failed_count`, but **only after the `dangerous_hooks` list has already been mutated**. If the persist fails after a `Dangerous` verdict, the user is told via the completed notification that 0 dangerous hooks were found — yet the in-memory model knew otherwise. Either persist before pushing to `dangerous_hooks`, or surface a separate `WarningEvent` when the persist fails so the user knows the verdicts were computed but lost.
- `hook_auto_review.rs:78-93` — the candidate filter only matches `Untrusted | Modified`. That's correct, but `HookTrustStatus` is a 3-variant enum (`Trusted` is the third). If a future variant is added (`AutoTrusted`, `Stale`, etc.) this `matches!` will silently exclude it. Use an exhaustive `match` with explicit `_ => false` and a comment about why.
- `hook_review.rs:1432-1450` — the policy text uses "Codex" capitalized — that's a product name. Fine for upstream codex, but worth noting that "Mark a hook dangerous **only** for clear security risk" combined with a "should not be marked dangerous" benign list is asymmetric: a hook that's *both* benign-shaped *and* contains an exfil pattern can hit either bucket depending on model interpretation. Recommend pinning a few canonical positive/negative examples in the policy and shipping a small eval set in `hooks/src/engine/mod_tests.rs` (+138 lines added there — confirm the eval is offline-deterministic).
- `MAX_SOURCE_EXCERPT_CHARS = 32_000` (`hook_auto_review.rs:1743`) — a 32k-char excerpt is a lot of context per review when you're running this serially across N hooks. Combined with the per-hook full-config rebuild, the total review cost can be substantial. Worth a metric or log line so users can see what the auto-review feature is costing them in tokens.
- The 30+ schema files with `+2 / -1` are all slug bumps — mechanical and low-risk, but mass-touching ThreadStartResponse/ThreadResumeResponse/etc. in the same PR as the feature makes the diff hard to scan. Future PRs of this size would benefit from splitting "schema slug bump" into a separate prep PR.
- `bespoke_event_handling.rs` (+42/-0) and `thread_status.rs` (+55/-0, new) — protocol routing. No tests added in the app-server suite for the new notifications beyond `tests/suite/v2/hooks_list.rs` (+12 lines). Add at least one test that asserts `HookAutoReviewStarted` / `HookAutoReviewCompleted` round-trip through the v2 protocol.

## Verdict: `request-changes`

Real concerns: (1) **timeout doesn't cancel the underlying review thread** — token waste, fix is one line; (2) **no enforced sandbox on the guardian sub-agent** when reviewing potentially-malicious hook source — at minimum confirm the spawned thread runs under a tool-restricted profile, or document the trust assumption explicitly; (3) **persist-failure path silently drops the dangerous-hooks list** before reporting — surface a `WarningEvent`. Once those are addressed the rest of the PR is well-shaped: clean module boundaries, structured output schema, and proper protocol versioning. Suggest also splitting the schema-slug bump out of this diff.
