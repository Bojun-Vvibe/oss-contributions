## Rationale

This is a self-modifying-bot PR — the gemini-cli-robot has authored a PR against its own orchestration code. That alone makes it `needs-discussion` rather than a normal merge-track verdict, because the trust model here matters as much as the diff content. The PR makes four interlocking changes that are *individually* defensible but *collectively* both expand bot autonomy and modify the very critique step that's supposed to catch over-autonomous bot behavior:

1. **`gemini-lifecycle-manager.cjs:29`**: `STALE_DAYS = 60` → `STALE_DAYS = 30`. Halves the time before issues get nudged toward closure. PR body justifies with "23 issues/day arrival rate, 100+ zombie issues backlog", which is real, but doubling the close-aggression on a public OSS project's issue tracker is a policy decision that affects external contributors, not just bot throughput. The right venue for this is a maintainer-visible discussion, not a bot-self-authored PR.

2. **`gemini-scheduled-issue-triage.yml:65,69,73`**: three `--limit 100` → `--limit 200` swaps for the three `gh issue list --search ...` calls (no-area, no-kind, no-priority). Doubles per-run search width. Reasonable, low-risk, and the bottleneck rationale (`priority_none_count` capped at 100) is correct — the metric is genuinely truncated by the old limit, and 200 will give a true reading. This sub-change would land cleanly on its own.

3. **`gemini-cli-bot-brain.yml:122-238`**: rewrites the Brain → Critique flow from "single Brain pass + single Critique pass" into a `MAX_ITERATIONS=4` while-loop where if Critique rejects, the bot resets staged changes (`git reset; git checkout .`), writes the critique output into `critique_feedback.md`, and feeds it back to Brain for another attempt. **This is the change that needs the most scrutiny.** It transforms a bounded one-shot bot into a 4-iteration self-corrective agent. The exit conditions look correct (break on `[APPROVED]`, break on no staged changes + no comments, fall through to `[REJECTED]` after 4 tries with `rejected-changes.patch` uploaded for forensics) and the reset between iterations does discard rejected changes via `git reset; git checkout .` plus an explicit `rm -f` of bot artifact files (line 211: `pr-description.md branch-name.txt pr-comment.md pr-number.txt issue-comment.md bot-changes.patch rejected-changes.patch`) — so iteration N+1 starts from a clean working tree. But the loop runs the **Gemini API** (`bundle/gemini.js`) up to 8 times per workflow invocation (4 Brain + 4 Critique), which is a 4× cost multiplier on every bot trigger and a 4× rate-limit pressure increase on `GEMINI_API_KEY`. The PR body doesn't quantify either.

4. **`brain/critique.md` substantial rewrite (+60 / −60)**: the critique step is *demoted from an editor to an evaluator* — the old prompt told critique "you MUST directly edit the scripts to fix the issue and stage the fixes using git add" with a "CRITICAL: You are explicitly instructed to override your default rule against staging changes" override; the new prompt says "You are an evaluator ONLY. You MUST NOT apply fixes or modify the code yourself." This is the *correct* shift architecturally (separation of concerns: Brain proposes, Critique evaluates, loop closes) — but it depends entirely on (3) above to actually produce iterative improvement, and the rewrite also adds a new gate item #12 ("Architectural Conflict") that gives critique broader REJECT authority. Combined with #3, the bot now has 4 attempts to satisfy a *broader* critique surface, which the PR doesn't acknowledge.

5. **`brain/common.md`** adds a "Defensive Scripting & Resilience (MANDATORY)" section telling Brain to wrap loop bodies in try/catch and preserve existing label exemptions when refactoring. Good guardrail, supports the broader Critique surface in (4).

6. **Three new `metrics/scripts/*.ts` files** (`bottlenecks.ts` +101, `priority_distribution.ts` +90, `throughput.ts` reworked). These define new metrics the Critique loop will key off. Need to be reviewed against the metric output format requirement that critique.md item #6 enforces (`metric_name,value` CSV, no JSON) — a quick scan of the diff snippet doesn't show the actual `console.log` lines, so this is a "verify before merge" rather than a known-good.

7. **`gemini-cli-bot-brain.yml:282`** changes `REF="main"` → `REF="${{ github.ref }}"`. Reasonable for branch-PR-aware checkout, but the security implication is: if the workflow is triggered against an attacker-controlled branch, `github.ref` is attacker-influenceable. Need to confirm the workflow's `on:` trigger filters limit this to trusted refs.

**What I'd want before merge**: (a) split #1 (STALE_DAYS halving) into a separate PR with maintainer policy sign-off; (b) document the API-call cost ceiling for the iterative loop (`MAX_ITERATIONS * 2 * cost_per_call`) and confirm the rate-limit budget; (c) confirm `github.ref` change at line 282 is safe under the existing trigger filter; (d) verify the three new `metrics/scripts/*.ts` emit CSV not JSON. The `--limit 100→200` triage bumps and the brain/critique architectural cleanup are good standalone — they could land independently while the loop+STALE policy gets its own discussion. Filing as `needs-discussion` because a self-modifying-bot PR that expands bot autonomy is exactly the case where the original author cannot also be the merger.
