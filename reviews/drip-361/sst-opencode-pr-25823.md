## Rationale

This PR tries to do three things in one commit: (a) auto-strip `completed`/`cancelled` rows from the todo list both on write (`tool/todo.ts:42`) and on read (`session/todo.ts:73`), (b) register two new built-in commands `/clear-tasks` and `/清除任务` whose template tells the LLM to call `todowrite` with `[]` (`command/index.ts:104-120`), and (c) update the `todowrite.txt` system prompt to advertise the auto-cleanup behavior. The intent is reasonable — the PR body's complaint that "stale todos accumulate in the chat UI" is real — but the implementation contradicts the very contract that `todowrite.txt` (which this PR also edits) instructs the model to follow, and the prompt edit itself is grammatically broken in a way that will leak into every future TodoWrite call.

**Concrete problems:**

1. **The auto-strip on write breaks the documented contract.** The unmodified portion of `tool/todowrite.txt` (lines 1, 5, 95, 137 of the file as it exists post-PR) explicitly tells the model: "Mark tasks complete IMMEDIATELY after finishing (don't batch completions)" and "Update task status in real-time as you work". The model is instructed to *send the full list including completed items* on every write so the user can see progress. The diff at `tool/todo.ts:42-45` filters those out before persistence — which means the very next `TodoWrite` call from the model (which will, per its own training and per the unchanged surrounding prompt, re-send the same list with one item now `completed`) will silently drop the completed row from storage. The user never sees the green checkmark; the in-progress→completed transition becomes invisible. This is not "auto-cleanup of stale todos" — this is "you can never observe a task being completed". The behavior the PR body wants ("stale todos from earlier conversation turns") needs to be scoped to *prior-turn* todos, not in-turn completions.

2. **The session read-side filter at `session/todo.ts:73` doubles down on the same bug.** Even if a future writer respects "completed should persist", any consumer reading via `Todo.Service.list()` will get the rows hidden again. This means downstream code (UI, exports, audit) sees an inconsistent view of history.

3. **`todowrite.txt:1` rewrite is ungrammatical and will be sent to every model on every TodoWrite.** The new line reads: `Use this tool to create and manage a structured task list. Completed/cancelled tasks are auto-removed. for your current coding session.` — note the lowercase `for` after the period, the dangling preposition, and the broken sentence boundary. This is the *first line* of a system prompt that gets prepended to every tool description. It needs to be a single clean sentence. Also, the claim "auto-removed" directly conflicts with the rest of the prompt body (lines 95-101 in the post-PR file: "Mark tasks complete IMMEDIATELY after finishing"), which will confuse the model about whether to send completed items or not.

4. **No tests.** A change that mutates persistence semantics for `TodoWrite` and adds two new built-in command names needs at minimum a `tool/todo.test.ts` covering: (a) writing a list with mixed statuses preserves what the read API exposes for the *active session* turn, (b) the `/clear-tasks` template actually causes the model's resulting `TodoWrite([])` call to clear, (c) the zh.ts key `command.clearTasks` resolves at runtime. None of this is exercised.

5. **Built-in command registration in `command/index.ts:104-120` is not safely composable with user `cfg.command`.** The loop at line 122 (`for (const [name, command] of Object.entries(cfg.command ?? {}))`) iterates *after* the built-ins are registered, so a user-defined command named `clear-tasks` or `清除任务` will silently overwrite the built-in — that's probably intended, but the converse (a user who already has `/clear-tasks` defined will be surprised by the new built-in *not* overwriting theirs because they happen to load second) means there's no precedence documentation. Also, registering a Chinese-named slash command as a *default* (always registered, regardless of the user's locale) means English users get a `/清除任务` command in their tab-completion they can't read. That should be locale-gated, not unconditional.

6. **Mixed concerns make this hard to review safely.** The i18n string addition, the new built-in commands, the prompt rewrite, and the persistence-layer behavior change are four independent landings. The persistence change is the riskiest and the one that needs the most discussion; bundling it with two trivial command registrations and a one-character zh.ts addition pressures reviewers to LGTM the whole thing.

**What I'd want before merge:** split into 3 PRs — (1) the two `/clear-tasks` built-ins + zh.ts (probably mergeable as-is with locale gating), (2) the prompt rewrite (needs a clean rewrite of `todowrite.txt:1` and a decision about what semantics it actually advertises), (3) the auto-strip behavior, which needs to either be scoped to prior-turn cleanup only or be replaced with an explicit "archive" tier so completed items are still observable. Until then, `request-changes`.
