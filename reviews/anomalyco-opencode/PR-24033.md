# Context

`session/retry.ts` `retryable()` is the central predicate
the session-loop consults to decide whether a classified
error gets a backoff-and-retry pass. Today it has two arms:
a plain-text substring scan over `error.data.message` for
`"rate increased too quickly"`, `"rate limit"`, `"too many
requests"`; and a JSON-parse arm that re-parses the same
message as JSON and matches structured shapes like
`{ type: "error", error: { type: "too_many_requests" } }` or
`{ code: "resource_exhausted" }`.

The two arms exist because providers are inconsistent — some
emit raw text errors, some emit JSON-stringified envelopes.
Having two parallel matchers means every newly-observed
transient surface (connection reset, upstream hangup, 5xx,
timeout, "ended without sending chunks") needs editing in
both, plus expansion of the substring list. In practice the
plain-text list has stayed small while real-world transient
patterns have multiplied — so legitimate transients keep
slipping through and surfacing as hard failures.

# Problem

This PR collapses both arms into a single regex over the
lower-cased message, dropping the JSON-parse arm entirely.
The new regex has ~25 alternates spanning rate limits, 4xx/
5xx HTTP codes, connection errors, network errors, timeouts,
upstream/socket disconnects, retry-delay messages, and
provider-returned-error wrappers. After matching, it
classifies `overloaded`/`exhausted`/`unavailable` as
"Provider is overloaded" and otherwise returns the original
message verbatim.

# Design

One regex, one branch, one decision:

```
/rate.?limit|too.?many.?requests|overloaded|exhausted|
 unavailable|service.?unavailable|429|5\d\d|internal.?error|
 network.?error|connection.?(?:error|refused|lost|reset)|
 fetch.?failed|upstream|upstream.?connect|reset before headers|
 socket.?hang.?up|other.?side.?closed|ended without|
 timed? out|timeout|terminated|retry delay|
 provider.?returned.?error|server.?error/i
```

Test coverage adds 10 new plain-text cases (connection
refused, timed out, HTTP 500, overloaded, 429, provider-
returned-error, other-side-closed, reset-before-headers,
ended-without, retry-delay-exceeded). The existing JSON-
shape tests are *renamed* to "raw json strings" — but their
expected return values change too: previously a JSON-encoded
`too_many_requests` error returned the string
`"Too Many Requests"`; now it returns the raw JSON string
unchanged because the regex matched `too_many_requests`
inside the serialized JSON. Same for rate-limit codes.

# Risks and edges

- **Regex breadth.** `5\d\d` matches any three-digit
  starting with 5, which is the intent for HTTP 5xx, but it
  will also match the substring `"500"` in any text ("we
  processed 500 records"). The error path is narrow enough
  that this is unlikely to fire on user messages, but a
  provider that emits `"sent 500 tokens, hit rate cap"` —
  already retryable for unrelated reasons — will trip the
  regex on the `500` rather than the `rate cap`. Net result
  is the same (retry triggered, message passed through), so
  this is a false-positive that happens to land on the
  correct answer. Worth noting but not blocking.
- **`5\d\d` also matches HTTP 503/504 inside a structured
  JSON body that happens to be a "request_id" field.** Same
  story — false positive, correct answer. The cost of these
  false positives is one extra retry on a non-transient
  failure, which the surrounding backoff-and-attempt-cap
  handles.
- **`upstream` alone matches almost any nginx error message.**
  The intent is upstream-disconnect-style transients, but
  `"upstream model returned 422"` (a hard validation error)
  will also match `upstream`. The retry will burn time on a
  permanent failure. A tighter alternate like
  `upstream.?(?:disconnect|connect|closed|reset|error)` would
  preserve intent without the broad match. Cheap fix.
- **The JSON-parse arm drop is a behaviour change for tests
  AND callers.** The old code, given a JSON-encoded
  `too_many_requests` error, returned the constant string
  `"Too Many Requests"`. The new code returns the raw JSON
  string. Anything downstream that string-matches on
  `"Too Many Requests"` to decide UI presentation will
  silently lose its match. The renamed tests document this
  but don't flag it as a breaking change. Worth grepping the
  rest of the codebase for that constant before merge.
- **`GO_UPSELL_MESSAGE` and `FreeUsageLimitError` paths.**
  The new test suite asserts that `FreeUsageLimitError`
  inside `responseBody` returns `GO_UPSELL_MESSAGE` and that
  `APIError` with "Overloaded" in `message` returns
  "Provider is overloaded" — these paths must already exist
  in the upper part of the file (not shown in the diff)
  because the tests pass. The simplification preserves them.
- **Loss of structured-code matching means new providers
  that emit only structured error codes (no human text)
  will no longer be classified as transient.** Anthropic and
  OpenAI both include human text alongside the code, so
  this is not a regression today, but a future provider
  emitting `{ "code": "RATE_LIMIT_EXCEEDED" }` with no
  `message` field would now leak through. Mitigated because
  the JSON serialization itself contains `"rate_limit"` as
  a substring — the regex will match anyway. So this risk
  is mostly theoretical.
- **`lower.includes("overloaded")` reclassification fires
  *after* the regex match.** Order is: regex match decides
  retryability, then a sub-classification picks
  "Provider is overloaded" or returns the raw message. The
  existing `APIError` overloaded path higher in the file
  already returns "Provider is overloaded" directly, so this
  is the same string from two places. Consistent — good.
- **`iife` import drop.** The unused-after-this-PR import is
  cleanly removed. Good hygiene.

# Verdict

Approve. Net simpler, broader coverage of real transient
surfaces, well-tested. Two recommendations: (a) tighten
`upstream` to `upstream.?(?:disconnect|connect|closed|
reset|error)` to avoid matching legitimate 422-class
upstream errors; (b) grep for `"Too Many Requests"` string
matches downstream and update them, since the constant
return is gone.
