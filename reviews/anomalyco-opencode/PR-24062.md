# Context

`opencode` exposes its server through Hono. A parallel migration
is in progress to lift each route group onto the Effect-flavoured
`HttpApi` builder, which gives schema-typed endpoints, generated
OpenAPI, and uniform middleware composition. The migration is
feature-flagged behind `OPENCODE_EXPERIMENTAL_HTTPAPI`; both
implementations of each endpoint coexist until the flag flips.

The migration tracker (`specs/effect/http-api.md`) walks through
route groups roughly cheapest-first: read endpoints before
mutations, small surfaces before chatty ones, and any "obvious
seam" before anything that needs to coordinate with WebSocket
upgrades. By this PR the migration has already absorbed `auth`,
`config`, `permission`, `project`, `provider`, and `question`.
This PR adds `workspace` reads — `GET /experimental/workspace`,
`GET /experimental/workspace/adaptor`, `GET /experimental/workspace/status` —
and leaves create/remove on the legacy Hono router for a later PR.

# Problem

Three concrete things need to happen, and the PR has to do them
in a way that does not regress the legacy implementation when
the flag is off:

1. Define an `HttpApi` group whose endpoint shapes match the
   existing legacy responses byte-for-byte (so generated SDK
   clients keep working through the cutover).
2. Mount the new handler in front of the legacy Hono routes only
   when the flag is on, so the new handler captures the request
   first and the legacy router never sees it.
3. Ship a regression-grade test that proves the new handler
   actually serves the three paths against a real-ish workspace
   (not just that the routes are registered).

The implicit constraint — and the interesting one — is that the
new handler must be installed *in the Hono tree*, not as a
sibling. Workspace routes are scoped by the
`WorkspaceRouterMiddleware`, which uses the Hono request to
upgrade to a WebSocket. If the Effect handler captured the route
before that middleware installed itself, WebSocket upgrades for
*other* workspace routes (the ones still on the legacy path)
would silently break.

# Design

The new file `httpapi/workspace.ts` is a clean, self-contained
HttpApi group:

```ts
export const WorkspacePaths = {
  adaptors: `${root}/adaptor`,
  list: root,
  status: `${root}/status`,
} as const

export const WorkspaceApi = HttpApi.make("workspace")
  .add(
    HttpApiGroup.make("workspace")
      .add(
        HttpApiEndpoint.get("adaptors", WorkspacePaths.adaptors, {
          success: Schema.Array(WorkspaceAdaptorEntry),
        }).annotateMerge(...)
        // list, status
      )
  )
```

Two things to like immediately. First, `WorkspacePaths` is exported
as a `const` object — the test file imports it and feeds it
directly into a `Request(...)` constructor, so any future rename
is a typecheck failure rather than a stale string in a test. The
legacy mount in `server.ts` also reads from the same `WorkspacePaths`
constants, so there is exactly one source of truth for the three
paths.

Second, the handlers are written as one-liners over the existing
`Workspace` and `listAdaptors` entry points:

```ts
const list = Effect.fn("WorkspaceHttpApi.list")(function* () {
  return Workspace.list((yield* InstanceState.context).project)
})

const status = Effect.fn("WorkspaceHttpApi.status")(function* () {
  const ids = new Set(
    Workspace.list((yield* InstanceState.context).project).map((i) => i.id)
  )
  return Workspace.status().filter((item) => ids.has(item.workspaceID))
})
```

That `status` filter is doing real work: `Workspace.status()` is
the global connection table across *all* projects, and the
endpoint must scope it to the current project. The implementation
delegates "which workspaces does this project own" to
`Workspace.list`, builds an id set, then filters. It is the same
logic the legacy route uses (verified by reading the test which
expects empty arrays for both — so on a fresh tmp project both
implementations agree on `[]`).

The mount in `server.ts` is the part worth dwelling on:

```ts
const workspaceApp = new Hono()
const workspaceLegacyApp = new Hono()
  .use(InstanceMiddleware())
  .route("/experimental/workspace", WorkspaceRoutes())
  .use(WorkspaceRouterMiddleware(runtime.upgradeWebSocket))
if (Flag.OPENCODE_EXPERIMENTAL_HTTPAPI) {
  const handler = ExperimentalHttpApiServer.webHandler().handler
  const context = Context.empty() as Context.Context<unknown>
  workspaceApp.get(WorkspacePaths.adaptors, (c) => handler(c.req.raw, context))
  workspaceApp.get(WorkspacePaths.list, (c) => handler(c.req.raw, context))
  workspaceApp.get(WorkspacePaths.status, (c) => handler(c.req.raw, context))
}
workspaceApp.route("/", workspaceLegacyApp)
```

Order matters and the order is correct. Hono matches in
registration order; the three flag-gated `.get` calls are
registered first, so when the flag is on they win the match for
those three exact paths. Everything else under
`/experimental/workspace/*` (mutations, WebSocket upgrades) falls
through to the `workspaceLegacyApp` mount. When the flag is off,
no `.get` calls happen at all — the new code is structurally
absent and behaviour is identical to before.

The `Context.empty() as Context.Context<unknown>` cast is the
ugliest line in the diff but I think it is honest. The HttpApi
handler expects an Effect `Context` and the legitimate dependencies
(authorization, instance state) are injected through the
`Authorization` middleware and `InstanceState.context` calls
*inside* the handler, not via the outer Context. There is no
real top-level context to pass, so passing an empty one is the
truthful representation. Wrapping it in a helper would be
cosmetic.

# Risks

**Drift between schemas and legacy responses.** The Effect group
declares `success: Schema.Array(WorkspaceAdaptorEntry)`. The
legacy route has its own response shape, defined elsewhere. If
they ever diverge — say someone adds a field to
`WorkspaceAdaptorEntry` — clients hitting the experimental flag
will see the new field and clients hitting the legacy route will
not. Today they share the same TS type, so the typechecker would
flag a divergence in source, but a runtime serializer change
(e.g. `omit` somewhere) would not be caught. A small contract
test that hits both paths against the same fixture and asserts
equality would pin this.

**Auth applied uniformly?** `WorkspaceSecured = WorkspaceApi.middleware(Authorization)`
is the only auth gate on the new path. The legacy
`workspaceLegacyApp` chain pulls in `InstanceMiddleware()` and
`WorkspaceRouterMiddleware(...)`, and those middlewares may be
doing more than auth (instance scoping, request id propagation,
log fields). The new path skips all of them. For *read* endpoints
that are already filtered by `InstanceState.context`, this is
probably fine — `InstanceState.context` is the same source of
truth either way — but it is worth a one-line audit of what the
two middlewares contribute beyond auth, and a comment in the new
file noting which guarantees apply.

**Test fixture is the empty case.** The test asserts the three
endpoints return well-formed responses on a fresh tmp project:
one default `worktree` adaptor, empty workspace list, empty status
list. That's a valid smoke but it does not exercise the
`status` filter, which is the only handler doing non-trivial work.
A second test with two projects and overlapping workspace ids
would catch a regression where someone simplifies `status` to
`return Workspace.status()` without the filter — the empty-case
test would still pass.

**`ExperimentalHttpApiServer.webHandler()` per request?** The
`server.ts` lambdas call `.webHandler()` once at startup (it's
outside the `(c) =>` closure) — good, that's correct. But the
captured `handler` is then invoked per request with a fresh
`Request` constructed from `c.req.raw`. If `webHandler()` does
any internal caching keyed by the request URL, the empty
`Context` could become a confounder. Worth a note about whether
the handler is intended to be re-entrant (it almost certainly is,
but the assumption isn't documented).

# Suggestions

1. Add a one-line comment above the empty-context cast linking
   to where the real context (auth identity, instance state)
   actually gets installed inside the handler, so the next reader
   doesn't try to "fix" the cast by threading context through the
   outer scope.

2. Add a second test that creates two workspaces under one
   project and asserts the `status` endpoint returns exactly the
   project's two entries (not all global statuses). This pins the
   filter logic.

3. The migration tracker entry now reads `workspace` - `bridged`,
   but the `bridged endpoints:` line still lists three explicit
   paths. When a future PR ports the mutations, the tracker entry
   format will need to evolve from "which endpoints" to "which
   verbs". Worth thinking about now so the format stays parseable.

4. Consider exporting `WorkspacePaths` from a higher-level
   `httpapi/paths.ts` barrel so that an integration test in
   another package (the SDK?) can reference the same constants
   without reaching into the `httpapi/workspace` module.

# Verdict

Approve. This is the right size of step for a feature-flagged
migration: one route group, three read endpoints, one focused
test, no behaviour change when the flag is off, single source of
truth for paths, ordered correctly relative to the legacy
WebSocket-bearing middleware. The risks I noted are all
"someone adds a field next year" risks, not "this PR is wrong"
risks.

# What I learned

The `WorkspacePaths` const trick is the bit I am stealing. I
have written a lot of route migrations where the "new" handler
and the "old" handler each define their own path strings, and
the only thing keeping them in sync is the pull-request reviewer
remembering that `/v2/foo` should match `/foo` modulo prefix.
Defining the path strings in the new module, exporting them as
typed constants, and consuming them from *both* the new mount
and the legacy mount means the migration cannot accidentally
serve different paths from the two implementations. It costs one
extra import line per consumer and removes an entire class of
bug.

Second, the order-sensitivity of `Hono` mounts is the kind of
thing I would normally be paranoid about, and the diff here
handles it the simplest correct way: register the flag-gated
exact-match `.get`s first, then mount the legacy router for
everything that falls through. No "is the path `/experimental/workspace/...`?"
predicate, no rewriting, just registration order. The fact that
the test passes on a fresh tmp project with no workspaces is
also a quiet check that the new handler doesn't accidentally
short-circuit `WorkspaceRouterMiddleware` on adjacent paths.

Third, the `Context.empty() as Context.Context<unknown>` cast is
exactly the kind of line I would normally try to refactor away.
After tracing where `InstanceState.context` gets set
(`InstanceMiddleware` for the legacy path,
`WorkspaceApi.middleware(Authorization)` for the new path) I am
convinced the cast is honest — the outer context genuinely has
nothing in it, the dependencies are installed inside. Adding a
helper to dress it up would just hide that fact. Sometimes the
ugly line is the truthful one and the right move is to leave it
alone.
