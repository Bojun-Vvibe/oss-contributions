# Context

The AI SDK's Zod schemas for OpenAI-compatible providers
require `tool_calls[].id` to be a string. The OpenAI API spec
agrees. A handful of OpenAI-compatible servers — the PR cites
NVIDIA NIM running kimi-k2.5 — emit numeric IDs anyway. The
SDK rejects the response at the schema boundary and the entire
turn fails with a Zod validation error, far below the layer
where opencode could meaningfully recover. Closes #23886.

# Problem

The fix has to land *before* the AI SDK parses the response,
which means inside the fetch interceptor that
`provider/provider.ts` already wraps around `@ai-sdk/openai-
compatible` calls. There are two response shapes to handle:
`application/json` for non-streaming completions, and `text/
event-stream` for streaming. Plus a backwards-compat tail:
sessions persisted before this fix may have numeric `callID`
values stored in the part rows, so the part hydrator must
coerce on load too.

# Design

Three coercion sites, one shared coercer:

1. **`util/coerce-tool-call-ids.ts`** — recursive walker that
   mutates `tool_calls[].id` and `delta.tool_calls[].id` in
   place. Walks every value of every record so nested shapes
   (e.g. `{ choices: [{ message: { tool_calls: [...] } }] }`)
   get covered without hard-coding the response envelope.
2. **`provider/provider.ts` fetch interceptor** — gated on
   `model.api.npm === "@ai-sdk/openai-compatible"`. For JSON
   bodies, reads the full text, parses, mutates, re-serializes
   and returns a fresh `Response`. For SSE, wraps the body in
   `transformSSEStream`, which line-buffers on `\n`, parses
   each `data: ` payload, mutates, and re-emits.
3. **`session/message-v2.ts` part hydrator** — checks
   `data.type === "tool" && typeof data.callID === "number"`
   on each `PartTable` row and coerces to string. Pure
   defensive backwards-compat for sessions captured before the
   fetch fix landed.

Test coverage is generous — 16 unit tests in the new file plus
one combined fixture-style test in `provider.test.ts`. Covers
nested objects, mixed string/numeric IDs, primitive inputs,
null/undefined, [DONE] passthrough, invalid-JSON passthrough,
multi-event chunks, and delta-shaped streaming.

# Risks and edges

- **The JSON path consumes the response body to a string and
  rebuilds it.** That's a behaviour change for any caller
  downstream of the interceptor that relies on streaming the
  body (the gate is JSON-only, so streaming consumers should
  hit the SSE branch — but if a provider mis-sets
  content-type to `application/json` for a long completion,
  this materializes the whole body in memory. Worth a
  size guard or a `.clone()` + best-effort path that falls
  back to passthrough on bodies above some threshold.)
- **`headers: res.headers`** is reused on the rebuilt
  `Response`. The original `Content-Length` header (if the
  upstream sent one) will now be wrong after re-serialization
  because `JSON.stringify` may not produce byte-identical
  output (key ordering is preserved by V8, but whitespace and
  number formatting like `1e10` vs `10000000000` are not
  guaranteed). Most consumers ignore `Content-Length` on
  responses they read via `.json()` / `.text()`, but stripping
  it (`headers.delete("content-length")`) before constructing
  the new `Response` would be safer and is essentially free.
- **`transformSSEStream` line-buffers on `\n`, not `\r\n` or
  `\r`.** SSE per RFC allows any of those. OpenAI and most
  compatible servers use `\n` so this is fine in practice, but
  a malformed provider emitting `\r\n` would have its events
  bundled into one buffer chunk and parsed only on stream
  close. Cheap fix: split on `/\r?\n/` instead.
- **Mutation-in-place on a parsed JSON tree is fine here**
  but `coerceNumericToolCallIds` walks *every* value including
  arrays-of-strings and large argument blobs (the
  `function.arguments` field can be a long stringified JSON).
  The early `typeof obj !== "object"` guard short-circuits
  primitives so the strings inside arguments are skipped at
  cost ~O(n) of the tree size. Acceptable, but worth measuring
  on a 200K-token tool-heavy completion to confirm the recurse
  doesn't show up in flame graphs.
- **The gate is `model.api.npm === "@ai-sdk/openai-
  compatible"` exact string.** Providers wrapped via
  `@ai-sdk/openai` proper (not the compatible variant) won't
  benefit. This is correct — first-party OpenAI wouldn't
  emit numeric IDs — but if NVIDIA NIM is configured against
  the `openai` package by some users (the configs are
  near-interchangeable), they'll still hit the bug. A
  follow-up could widen the gate to a Set of npm names.
- **The `codex.ts` plugin gets a one-line `gpt-5.5` addition**
  buried in the same commit. Unrelated to the tool-call-ID
  fix. Reviewers should ask the author to split that out — it
  passes through review undetected here precisely because the
  diff focus is elsewhere.
- **The `message-v2.ts` hydrator mutates `row.data` in place**
  (`data.callID = String(data.callID)`) before spreading.
  `row.data` is the live drizzle row object — mutating it
  could affect any caching layer that re-reads the same row.
  In practice drizzle returns fresh row objects per query so
  this is benign, but a `{ ...data, callID: String(...) }`
  spread would be defensively cleaner.

# Verdict

Approve with two nits: (a) drop the unrelated `gpt-5.5`
plugin line into its own commit/PR, (b) widen the SSE line
split to `/\r?\n/` for one-character cost. The core fix is
correctly placed, well-tested, and the backwards-compat
hydrator path is exactly right for users with sessions that
predate the fix.

# Skim path

1. `coerce-tool-call-ids.ts` — confirm the recursion has a
   primitive guard at the top and only mutates when
   `typeof tc.id === "number"`.
2. `provider.ts` — confirm the gate is JSON-only on the
   non-stream branch and that the SSE branch returns *before*
   the chunk-timeout `wrapSSE` wrapper, not after (so the
   transform sees raw bytes, not already-wrapped bytes).
3. `message-v2.ts` — confirm the coercion only fires on
   `data.type === "tool"` (no false positives on text parts
   with a numeric `callID`-shaped field).
4. `provider.test.ts` — confirm the inline coerce helper in
   the test is kept in sync with the production helper, or
   replaced by a direct import.
