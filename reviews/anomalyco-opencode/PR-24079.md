# Context

The `opencode` web app shows an inline VCS diff for the current
session in its review pane. The diff is fetched by hitting the
server's `/vcs/diff` endpoint, gated by a `createQuery` whose
`enabled` predicate currently boils down to "the user wants to
see review and the project is a git repo." On normal
single-repo workspaces this is fine; on "umbrella" workspaces
that embed multiple nested git repos and worktrees, the
endpoint can be O(N) over the entire tree, with reported cases
of OOM crashes on the server side (issue #24049).

# Problem

The fix for the underlying scalability problem is non-trivial —
likely involves bounding the diff scope, paginating, or moving
to incremental computation against the working tree's modified
set rather than walking everything. That work is open-ended; the
people running into the problem today need an escape hatch
*now*. The PR explicitly frames itself as a mitigation, not the
fix: `disable_vcs_diff` lets affected users opt out of the
auto-fetch entirely while preserving today's default for
everyone else.

# Design

Three small, well-isolated pieces:

1. `config/config.ts` extends the existing `experimental`
   `Schema.Struct` with an optional boolean
   `disable_vcs_diff`. Lives next to `disable_paste_summary`,
   which establishes the precedent that "disable_X" experimental
   booleans are an accepted pattern in this surface.
2. `pages/session.tsx` ANDs the new flag into the existing
   `enabled` predicate of `vcsQuery`:
   `!sync.data.config.experimental?.disable_vcs_diff &&
   wantsReview() && sync.project?.vcs === "git"`. When the flag
   is on, the query never runs, so no `/vcs/diff` request is
   issued.
3. `sdk/js/src/v2/gen/types.gen.ts` has the regenerated SDK type
   reflecting the new optional field, with the description copied
   verbatim from the schema annotation.

The total surface is 8 additions, 1 deletion, 3 files. Classic
config-gated mitigation shape: schema → consumer → SDK
regeneration, no behavioural change for anyone who doesn't opt
in.

# Risks and edges

- **The flag's name is a behavioural promise.** "Disable VCS
  diff" reads as broader than what the patch actually does — the
  patch only disables the *auto-fetch from the app's session
  page*. A user who reads the flag name and expects all
  `/vcs/diff` calls (including from CLI subcommands or from
  manual triggers in the review pane) to be suppressed will be
  surprised. Either narrow the name
  (`disable_vcs_diff_auto_fetch`) or add a description sentence
  that explicitly says "auto-fetch on session page only."
- **The `experimental` namespace is a one-way door.** Once a
  user adopts a flag, removing it later breaks their config. The
  PR description correctly hedges that this is a mitigation
  pending a real fix; ideally the flag is documented as
  "candidate for removal once #24049 is resolved" in the schema
  annotation, so users have a heads-up.
- **No telemetry on flag adoption.** When the underlying fix
  lands, the team has no way to measure how many users still
  have the workaround on. A simple counter incremented when the
  query is suppressed would let the team know when it's safe to
  retire the flag.
- **The boolean's polarity is the harder-to-reason-about one.**
  `disable_vcs_diff: true` means "do not do the thing." A
  positive-sense flag (`vcs_diff_auto_fetch: false` defaulting
  to `true`) reads more cleanly. This is bikeshed territory and
  consistent with the sibling `disable_paste_summary`, so I
  wouldn't block on it.

# Suggestions

- Annotate the schema field with a sentence about scope and
  retirement plan: "Disable the session-page VCS diff auto-fetch
  query. Mitigation for #24049; removed when the underlying
  scalability issue is fixed."
- Add a tiny visual cue in the review pane when the flag is on,
  so a user who forgot they enabled it doesn't think the diff
  pane is broken. A single italicized line — "Auto-diff disabled
  via `experimental.disable_vcs_diff`" — would prevent a future
  bug report.
- A line in CHANGELOG / release notes naming the flag explicitly
  is high-leverage; the affected users on #24049 will grep for
  it.

# Verdict

Approve. This is exactly the right shape for a mitigation: small
diff, opt-in, no default behaviour change, schema-typed,
SDK-regenerated. The only changes I'd consider blocking on are
documentation (description sentence + changelog), and even those
are easy follow-ups.

# What I learned

The "experimental disable_X flag as time-bought mitigation"
pattern is underrated. It is tempting to either ship the proper
fix or block on it; in practice the affected user is suffering
*now*, the proper fix is open-ended, and a 3-line opt-out costs
the codebase very little (especially when the schema makes the
flag self-describing in generated SDK types). The discipline is
to leave a paper trail — issue link, retirement plan, changelog
— so the flag doesn't outlive its usefulness as a permanent
quirk no one remembers adding.
