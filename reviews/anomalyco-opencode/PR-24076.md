# Context

`opencode` runs streaming inference through Bun's native `fetch`,
which exposes `ReadableStream` for SSE / chunked-encoded provider
responses. Errors from the underlying socket layer surface as
plain `Error` objects with a free-text `message` rather than as
typed exceptions. The session-layer error machinery has two
collaborators: `fromError()` in `message-v2.ts` classifies a raw
error into a typed shape (`APIError`, `RateLimitError`,
`NamedError.Unknown`, etc.), and `retryable()` in `retry.ts`
decides whether a classified error gets a backoff-and-retry pass
before the session aborts.

# Problem

The PR is responding to a concrete failure mode: when a provider
closes the streaming connection mid-response — a common transient
condition under load, especially behind layer-7 load balancers —
Bun raises an `Error` whose message contains
`peer closed connection without sending complete message body
(incomplete chunked read)`. The existing `fromError()` switch has
no case for this string, so it falls through to
`NamedError.Unknown`, which `retryable()` treats as terminal. The
session aborts, the partially-received tool call fails Zod
discriminator validation downstream
(`No matching discriminator: type`), and the user sees a stack of
three error messages — none of which says "the network blipped,
retrying."

# Design

Two parallel changes, one per file:

1. `message-v2.ts` adds `isStreamConnectionError(e)`, a
   case-insensitive substring match against four strings:
   `peer closed connection`, `incomplete chunked read`,
   `unexpected end of stream`, `socket hang up`. A new case in
   `fromError()`'s pattern-matched switch returns an `APIError`
   with `isRetryable: true` and a stable `code:
   "STREAM_CONNECTION_ERROR"` for downstream telemetry.
2. `retry.ts` adds the same four strings (plus
   `"connection closed during streaming"`, the message the new
   `APIError` itself emits) to the `retryable()` predicate, so
   the retry executor recognizes both the raw underlying error
   and the wrapped `APIError` form.

The diff is tight: 33 additions, 0 deletions, 2 files. The
classification is conservative — it adds a case, it doesn't
re-shape the existing fall-through. That is the right shape for
an error-classification PR landing into a hot path.

# Risks and edges

- **Substring matching against vendor strings is fragile.** Bun's
  `fetch` error messages are not part of any stability contract
  and can drift across Bun releases. The PR is right to match
  several variants in parallel rather than betting on one. A
  follow-up should consider matching on the underlying error
  `cause` chain (Bun's `fetch` errors usually have a `cause`
  pointing at the socket-level error with a more stable shape)
  rather than the surface message.
- **Double-coverage is intentional but worth naming.** The same
  four strings appear in two files. If a fifth variant shows up,
  both lists have to be updated. A small shared helper exported
  from one module would prevent drift; given the scope, leaving
  it inline is defensible, but a `// keep in sync with retry.ts`
  comment would help future maintainers.
- **Retry semantics on a half-streamed response.** The PR makes
  the retry happen, but does not address the question of what
  state the partial stream leaves behind. If the provider had
  already emitted half a tool call when the connection dropped,
  the retry will start a fresh request and the partial bytes
  should be discarded by the surrounding session code. A spot
  check of the retry harness to confirm partial buffers are
  cleared before the retry attempt would close that loop.
- **No new test.** The PR has no unit test covering either the
  classifier or the retry predicate. Both are pure functions over
  string input; a five-line table-driven test would prevent the
  next maintainer from deleting one of the strings as
  "redundant."

# Suggestions

- Add a tiny `__tests__/stream-error-classification.test.ts` with
  a parameterized table: `[input message, expected isRetryable,
  expected code]`. Cover the four target strings, one near-miss
  ("socket reset" is *not* in the list — confirm intentionally),
  and the legacy `Unknown` fall-through.
- Consider `error.cause` matching as a follow-up. Bun typically
  attaches the underlying `Bun.SocketError` with a stable `code`
  property (`ECONNRESET`, `EPIPE`, etc.). Matching on those
  codes would be drift-resistant.
- Emit a structured log on the retry path so operators can
  distinguish "transient stream blip retried successfully" from
  "stream blip retried and re-failed." The current
  `STREAM_CONNECTION_ERROR` code surfaces in metadata but won't
  show up on its own log line.

# Verdict

Approve as a targeted fix. The classification is correct, the
retry plumbing already exists and the PR slots cleanly into it,
and the user-visible failure mode it addresses is real and
high-frequency. The follow-up work — `cause`-based matching,
shared string list, smoke test — is below the bar for blocking
this PR.

# What I learned

Free-text error matching against a vendor's underlying transport
errors is a rite of passage for any HTTP-streaming client. The
mature pattern is exactly what this PR is converging toward but
does not fully reach: match on the underlying typed error
(`cause` chain) when the runtime exposes one, fall back to
substring matching only when it doesn't, and centralize the
match list in a single module so the two layers (classifier,
retry policy) cannot drift. The substring-list-in-two-files form
is fine as a first cut; promoting it to a single source of truth
should happen before the third layer ever needs the same list.
