# Context

`opencode` separates concerns cleanly between (a) a backend server process
that keeps per-directory state — sessions, providers, LSP, VCS — and (b) a
SolidJS web UI that subscribes to that state via SSE plus REST. The UI uses
a "child store" pattern: each project directory the user has visited gets
its own `Store<State>`, with a top-level manager that owns lifecycle (pin,
unpin, evict) and several caches keyed by directory string. The router
header `x-opencode-directory` is the contract that ties an HTTP call to a
specific directory's state.

That contract assumes one canonical string form for any directory. On POSIX
that holds: paths use `/`, the SDK URL-encodes them, the SSE backend echoes
them back the way it received them. On Windows it does not. The same
logical path arrives in two flavors during a single session: with
backslashes when it comes through OS APIs or server-sent events, and with
forward slashes when it round-trips through URL decoding or the SDK's path
formatter. JavaScript treats `"C:\\Users\\me\\proj"` and
`"C:/Users/me/proj"` as distinct map keys, and the manager dutifully
creates two parallel store entries — one populated by the writer code path,
one read by the renderer. The result is the visible bug: on Windows, the
sidebar tile for a project shows zero sessions even though the project's
sessions are reachable by direct URL.

The fix is conceptually simple — normalize the key — but the surface area
matters because the dirty key string is reached for in dozens of places
across the UI's global-sync subsystem. The PR also rolls in two unrelated
Windows correctness fixes the author noticed while debugging the symptom.

# Diff walkthrough

The change spans `packages/app/src/context/global-sync/` (UI side) and one
file in `packages/opencode/src/server/routes/ui.ts` (backend dev-serve
plumbing). Most of the +510/-172 is the rebase noise common to all PRs
opened against this main branch right now; the meaningful surface is
~250 lines.

**`child-store.ts` — the canonical fix.** A single helper is introduced at
the top of `createChildStoreManager`:

```ts
const dirKey = (d: string) => d.replaceAll("\\", "/")
```

It is a string-only normalization, deliberately not touching the *value*
of `directory` that gets passed to API calls or persisted with sessions.
Only `Map`/`Record`/`Set` *keys* are normalized. The PR then mechanically
threads `dirKey(directory)` through every place that uses a directory
string as a structural key:

```ts
const mark = (directory: string) => {
  if (!directory) return
  const key = dirKey(directory)
  lifecycle.set(key, { lastAccessAt: Date.now() })
  runEviction(key)
}

const pin = (directory: string) => {
  if (!directory) return
  pins.set(dirKey(directory), (pins.get(dirKey(directory)) ?? 0) + 1)
  mark(directory)
}
```

Note the discipline: `mark()` is still called with the raw `directory`,
not with `key`, because `mark` itself re-normalizes inside. That keeps the
public contract of every helper "takes a directory string in any form" and
localizes the normalization to one line per function. There is some
double-work (`pin` calls `dirKey` twice in one expression, then `mark`
calls it again), which a maintainer might tighten, but the current shape
is easier to reason about: every entry point normalizes, so any call from
anywhere is safe.

The trickier section is `disposeDirectory`, where the manager has to
delete from *six* different caches keyed on the directory:

```ts
function disposeDirectory(directory: string) {
  const key = dirKey(directory)
  if (
    !canDisposeDirectory({
      directory,
      hasStore: !!children[key],
      pinned: pinned(directory),
      booting: input.isBooting(directory),
      loadingSessions: input.isLoadingSessions(directory),
    })
  ) {
    return false
  }

  vcsCache.delete(key)
  metaCache.delete(key)
  iconCache.delete(key)
  lifecycle.delete(key)
  const dispose = disposers.get(key)
  ...
```

The `canDisposeDirectory` callback still receives the raw `directory`
because *its* implementation lives outside this manager (in
`global-sync.tsx`) and may have its own normalization. Mixing `key` and
`directory` in the same call is intentional: caches owned by this
manager use `key`, callbacks owned by the caller use `directory`. This is
a good separation but it depends on every owner doing its own
normalization — which is exactly the bug pattern that produced the
issue. A defensive fix would either normalize at the API boundary
(server-side, before any directory hits the wire) or wrap directory
strings in a nominal type that forces normalization at construction.

**`bootstrap.ts` — provider revision tracking.** A separate
`providerRev` map exists outside the manager to discard stale
provider-list responses; it suffered from the same key collision:

```ts
const dirKey = (d: string) => d.replaceAll("\\", "/")

export function clearProviderRev(directory: string) {
  providerRev.delete(dirKey(directory))
}

// ...inside bootstrapDirectory:
const rev = (providerRev.get(dirKey(input.directory)) ?? 0) + 1
providerRev.set(dirKey(input.directory), rev)
```

The race the `rev` counter guards against — old provider list arriving
after a newer bootstrap call — was previously *more* likely to fire on
Windows because the `providerRev.get(input.directory) !== rev` check
would compare against `undefined` for the alternate-separator key and
always pass. So the fix has a side benefit: on Windows it stops
silently letting stale provider lists through.

**`ui.ts` — the dev-serve fall-through.** This is the unrelated bonus
fix. Previously the UI route handler had two paths: load embedded
assets from the compiled `opencode-web-ui.gen.ts`, or proxy to
`app.opencode.ai`. There was no way to develop locally against
`packages/app/dist/` without modifying the binary. The PR adds a
middle layer:

```ts
// For dev serve: try loading directly from the app dist directory
const appDistDir = path.resolve(import.meta.dirname, "../../../../app/dist")
const localUIPromise = fs.exists(appDistDir).then((exists) => {
  if (!exists) return null
  console.log("[ui] Found local app dist directory:", appDistDir)
  return appDistDir
})
```

…and a permissive dev CSP that allows `localhost:*` for the dev server's
HMR socket:

```ts
const devCsp = (hashes = "") =>
  `default-src 'self' http://localhost:* http://127.0.0.1:*;
   script-src 'self' 'wasm-unsafe-eval'${hashes ? ` ${hashes}` : ""};
   ...
   connect-src 'self' data: http://localhost:* http://127.0.0.1:*
                       ws://localhost:* ws://127.0.0.1:*`
```

The dev/prod gate is implicit: if `appDistDir` exists at startup the dev
path is preferred. That works in `bun run dev` (where the app build runs
in parallel and produces `app/dist/`) but means a *production* binary
that is somehow run from a checkout that also has an `app/dist/` will
serve dev assets and the loose dev CSP. That is unlikely but possible —
and the loose CSP is the kind of thing security review will flag.

**`extractScriptHashes`.** Genuinely useful refactor: the previous code
took a single `hash` parameter for `'sha256-…'`; the new code walks the
HTML, extracts every inline script's body, and computes a SHA-256 per
script. That is the correct behavior for index.html files that contain
multiple inline bootstrap scripts (the previous code would silently
break when a second inline script was added).

# Risks / things a maintainer would catch

1. **Normalization is leaf-local, not boundary-enforced.** `dirKey` is
   re-defined in `child-store.ts`, again in `bootstrap.ts`, again in
   `queue.ts` and `session-prefetch.ts`. Drift is the failure mode: the
   day someone adds a fifth file that uses directory-as-key without
   copying the helper, the bug returns. A single shared
   `normalizeDirKey()` exported from a `path-key.ts` would centralize
   this.
2. **No nominal-type protection.** The fix relies on every developer
   remembering to call `dirKey()` at every cache lookup site. A branded
   type (`type DirKey = string & { __brand: "DirKey" }`) would make the
   compiler enforce normalization-before-use and would have caught the
   original bug.
3. **Server side is not fixed.** The server emits the directory string
   in SSE events using whatever form the OS gave it. If a future event
   arrives with backslashes after a client has stored forward-slash
   keys for the same directory, the client's *event handlers* will
   normalize but anything that compares the SSE payload directly to a
   stored directory string (e.g. equality checks not going through
   `dirKey`) will mis-route. The fix should ideally normalize at the
   server before emit.
4. **`replaceAll("\\", "/")` is not the only Windows path issue.** Drive
   letter case (`C:` vs `c:`), 8.3 short names (`PROGRA~1`), and UNC
   prefixes (`\\?\C:\...`) can also produce divergent strings for the
   same logical directory. The fix handles only the most common case
   (separator). A `path.normalize` plus drive-letter uppercasing would
   be more robust but also more expensive.
5. **Dev-serve fallback bypasses CSP hardening in any binary that
   happens to be run from a workspace.** The implicit gate
   (`appDistDir` exists) is path-based, not flag-based. A user who
   distributes a release binary inside a checkout will get dev mode
   silently. An explicit `OPENCODE_DEV_UI=1` flag would be safer.
6. **`extractScriptHashes` regex has known limits.** It will not handle
   `<script>` tags split across lines with attributes, nor scripts
   with embedded `</script>` strings. The current `index.html`
   probably does neither, but the next inline-script-emitting build
   tool change could silently produce invalid hashes and a broken CSP.
   Ideally the build emits the hashes itself rather than re-scanning at
   serve time.
7. **No new tests.** The fix has no regression test. A
   `child-store.test.ts` that creates a manager, calls `pin("C:\\foo")`
   and `unpin("C:/foo")`, and asserts pin count is zero would be
   roughly five lines and would lock the contract.
8. **Performance: `replaceAll` runs on every cache touch.** Cheap on
   any reasonable path length, but in tight loops (e.g. session list
   filtering in the sidebar) it's still string allocation per call.
   Memoizing inside `dirKey` for hot paths would be straightforward.
9. **The PR conflates three changes.** The Windows-keys fix, the
   `loadSessions` fix to use `sdkFor(directory).session.list()` instead
   of `globalSDK`, and the dev-serve UI route handler are three
   independent topics. Each could ship separately and be reverted
   separately.
10. **Provider revision counter delete also moves.** `clearProviderRev`
    now uses `dirKey`, but any caller *outside* this PR's diff that
    deletes from `providerRev` directly (e.g. via inlining for
    performance) will re-introduce the leak. A grep for
    `providerRev.delete` would confirm coverage.

# Suggested questions for the author

1. Would you consider extracting a single shared `normalizeDirKey`
   helper (or a branded `DirKey` type) instead of redefining `dirKey`
   in each module? The current shape works but is fragile to drift.
2. Should the server-side path also normalize before emit, so the
   client never sees backslashes in the first place? The client-side
   normalization is necessary regardless, but server-side
   normalization at the boundary would make every consumer (UI, SDK
   clients, future plugins) safe by construction.
3. For the dev-serve UI fallback — is it intentional that any binary
   running from a directory that happens to contain `../app/dist/`
   gets the relaxed dev CSP? An explicit env flag would prevent the
   "release run from a checkout" footgun.
4. Could the script-hash extraction live in the build step instead of
   the request handler? It's the kind of thing that's better to
   compute once at build time and pin via a generated file.
5. The PR also fixes `loadSessions` to call `sdkFor(directory)` instead
   of the global SDK. Was that a separate Windows-only bug, or does
   this actually affect POSIX too whenever the user's `cwd` differs
   from the requested directory?
6. Are there other Windows path normalization issues you've noticed
   that are not in this PR (drive letter casing, UNC paths, 8.3 names)?
   If so, it would help to capture them as follow-up issues so the
   normalization can be hardened beyond just separators.

# Verdict

**Needs revisions.** The diagnosis is correct and the localized fix
works for the reported symptom, but the change as written hard-codes
the same `dirKey` helper into multiple files and relies on developer
discipline to call it everywhere. That is exactly the bug pattern that
produced the original issue. The "small" version of the fix that the
maintainer should ask for: extract `normalizeDirKey` to a shared
module, ideally with a branded type so the compiler enforces use, then
re-route every call site through it. Splitting the dev-serve UI route
fix and the `loadSessions` fix into their own PRs would also make this
much easier to review and revert. With those changes, this is a
straightforward Windows-correctness merge.

# What I learned

The reusable insight is the failure mode of "string-keyed map with no
canonicalization at the key boundary." JavaScript's `Map` and `Record`
will happily store two entries for what humans regard as the same path,
and the only signal something is wrong is the second-order symptom
("sidebar shows zero sessions") — there is no error, no warning, no
log line. The general defense against this class of bug is to push
canonicalization to the *type system* rather than the call sites: a
branded `DirKey` type whose only constructor is `normalizeDirKey()`
makes it impossible to accidentally use a raw string as a key. The
short-term fix in this PR ("call `dirKey` at every site") is fine for
shipping a hotfix, but every long-running codebase that uses string-
keyed maps for filesystem paths eventually rediscovers this lesson the
hard way. A second takeaway: the bug was platform-specific but the
*condition* (more than one stringification of the same path can reach
the same map) is universal — POSIX has its own version (`/foo`,
`/foo/`, `./foo`, symlinks). Windows just shipped first because the
separator difference is the most syntactically obvious. Whenever you
see a string-keyed cache, ask "is there exactly one canonical form for
this key, and where is it enforced?"
