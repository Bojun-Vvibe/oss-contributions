# Context

`session/retry.ts` decides whether a classified error gets a
backoff-and-retry pass. For HTTP 429 (rate limited), the
current implementation defers to the AI SDK's `isRetryable`
flag. That flag is true for all 429s by design — the SDK
treats rate limiting as inherently transient. Most of the time
that is correct: per-minute or per-second token-bucket
limits resolve in seconds and a single retry past a backoff
window succeeds.

# Problem

The SDK's flag does not distinguish between *windowed* rate
limits (the kind exponential backoff fixes) and *quota*
rate limits — weekly, monthly, or absolute dollar caps where
the next allowed request is hours or days away. Both surface
as 429 with `isRetryable: true`. The retry executor therefore
keeps backing off and re-trying against a quota that will not
recover this session, burning the user's terminal session and
never surfacing an actionable error. Issue #24011 reports
this against Anthropic's weekly cap — the user sees opencode
spin for minutes through full backoff cycles before failing.

# Design

A surgical predicate. Before honoring the SDK's
`isRetryable: true` for a 429, the new code inspects the error
message for any of three substrings — `"weekly"`, `"monthly"`,
`"exceeded your"` — and if any match, returns `undefined` to
short-circuit the retry policy. The session then surfaces the
quota error directly to the user in a single pass.

The choice of substrings is provider-shaped: Anthropic emits
"You have exceeded your weekly limit", OpenAI emits
"You exceeded your current quota" (matched by
"exceeded your"), and several others use "monthly". The match
is case-insensitive (or should be — see risks).

# Risks and edges

- **`"exceeded your"` is broad.** It will match
  "You exceeded your maximum input tokens" if any provider
  ever encodes a per-request token-cap rejection as a 429.
  That would be wrong — that error is sometimes recoverable
  by retrying with a shorter prompt (which higher layers may
  do). Worth tightening to
  `"exceeded your monthly"|"exceeded your weekly"|
  "exceeded your usage"` if the maintainers see false
  positives in telemetry.
- **Substring matching against vendor strings is fragile.**
  Same caveat as the related stream-error PR (#24076):
  provider error strings are not part of any stability
  contract. The four strings cover today's providers but
  Anthropic, OpenAI, and Google have all reworded these
  messages in the past year. A follow-up should consider
  matching on `error.code` / `error.type` where the SDK
  normalizes them (Anthropic exposes `error.type:
  "billing_error"` for some quota exhaustions; OpenAI uses
  `error.code: "insufficient_quota"`).
- **The new test is unit-level only.** It asserts the
  predicate's return value but does not exercise the full
  retry executor with a mocked time source to confirm
  backoff is actually skipped. Given the predicate's job is
  exactly to short-circuit, an integration test that proves
  *zero* retry attempts after a quota 429 would catch a
  future bug where the executor ignores `undefined` from
  the predicate.
- **No user-facing message change.** The user gets the raw
  provider error, which for Anthropic is reasonably clear
  ("You have exceeded your weekly limit, resets in 3 days").
  For other providers the message can be cryptic. A
  follow-up that prepends "Quota exceeded — not retrying:"
  would help — but that is UX polish, not part of this fix.

# Verdict

Approve. Tight scope, the right layer, the failure mode is
expensive and user-visible. The substring-matching concerns
are real but defensible for a hotfix; the proper fix
(matching on normalized error codes) is a larger refactor
that should not block this.

# Skim path

1. `retry.ts` — confirm the predicate runs *before* the
   `isRetryable` check, not after, so the SDK's flag cannot
   re-enable retries.
2. `retry.test.ts` — confirm the three substrings each have
   coverage and the test asserts `undefined` (skip) not
   `false` (an alternate code path).
3. Issue #24011 — confirm the original repro matches one of
   the three substrings.
