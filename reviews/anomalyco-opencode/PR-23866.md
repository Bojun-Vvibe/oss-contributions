# Context

`opencode` keeps per-project state — sessions, message history, settings —
keyed by a stable `ProjectID`. Today the ID is derived from the **root
commit** of the git repository: it shells out to
`git rev-list --max-parents=0 HEAD` and uses the first root commit hash as
the project identifier, then caches it in `.git/opencode` so subsequent
launches do not have to re-shell. The intent of this design is clear: as
long as a developer is in *the same repo*, the project state should follow
them across machines, clones, fresh checkouts, even bare-and-reclone moves.

The problem the PR fixes is that this design is *too* aggressive at
unifying. Two completely independent clones of the same upstream — `~/work/A`
and `~/play/A-experiment` — share a root commit and therefore collapse into
one `ProjectID`, which means they share session history, share state, and
fight each other. For a tool whose user model is "one project per directory
I cd into," that is the wrong default. Worse, it cannot be undone short of
deleting the cached `.git/opencode` files, because the next launch will
compute the same hash and refill them.

This PR flips the default: the canonical worktree path becomes the project
ID, encoded as `dir_<base64url(path)>`. Worktrees still share state (git
worktrees resolve to the same `commonDir`, hence the same canonical path
chosen for the ID), but two clones with the same history now diverge. The
change is small in code but large in semantics — it redefines what
"the same project" means in the tool's mental model.

# Diff walkthrough

The substantive change is in three files. I focus on `src/project/project.ts`
and `src/project/schema.ts`; the unrelated bun-lock / dialog / Dockerfile /
patch / format files in this PR appear to be rebase carry-over and are
covered separately in the sibling PR (#23755) review.

**`schema.ts` — the new ID factory.** A single line teaches `ProjectID` to
mint itself from a path:

```ts
export const ProjectID = projectIdSchema.pipe(
  withStatics((schema: typeof projectIdSchema) => ({
    global: schema.make("global"),
    fromDirectory: (directory: string) =>
      schema.make(`dir_${Buffer.from(directory).toString("base64url")}`),
    zod: zod(schema),
  })),
)
```

Two design decisions baked in here. First, the `dir_` prefix namespaces the
new ID space, which keeps it visually disambiguated from the historical
40-character hex root-commit IDs. Second, `base64url` encoding on the raw
path — not a hash — keeps the ID *reversible*: anyone with the ID can decode
it and see exactly which directory it came from. That is a privacy trade-off
worth flagging (more on that below) but it does make debugging a database of
sessions much easier.

**`project.ts` — the new resolution path.** The previous code had a small
state machine: try cache file in `dotgit`, try cache file in `worktree/.git`,
shell out to `git rev-list`, write the result to disk, fall back to
`ProjectID.global` if everything failed. The new code replaces all of that
with two helpers and a straight-line resolution:

```ts
const canonicalDirectory = Effect.fnUntraced(function* (directory: string) {
  const normalized = pathSvc.normalize(directory)
  return yield* fs
    .realPath(normalized)
    .pipe(
      Effect.catch(() => Effect.succeed(normalized)),
      Effect.map((resolved) => pathSvc.normalize(resolved)),
    )
})

const projectIDFromDirectory = Effect.fnUntraced(function* (directory: string) {
  return ProjectID.fromDirectory(yield* canonicalDirectory(directory))
})
```

`canonicalDirectory` is the linchpin: it normalizes path separators, then
runs `realPath` to resolve symlinks (with a graceful fallback if `realPath`
fails — common on Windows network shares and some FUSE filesystems). The
canonical path then becomes the ID input. Without that step you would
generate distinct IDs for `~/code/foo` versus `/Users/me/code/foo` even when
they are the same directory via a symlink.

The resolution flow now reads top-down without conditional caching:

```ts
const commonDir = yield* git(["rev-parse", "--git-common-dir"], { cwd: sandbox })
if (commonDir.code !== 0) {
  sandbox = yield* canonicalDirectory(sandbox)
  return {
    id: yield* projectIDFromDirectory(sandbox),
    worktree: sandbox,
    sandbox,
    vcs: fakeVcs,
  }
}
const worktree = yield* canonicalDirectory(
  (() => {
    const common = resolveGitPath(sandbox, commonDir.text.trim())
    return common === sandbox ? sandbox : pathSvc.dirname(common)
  })(),
)
const id = yield* projectIDFromDirectory(worktree)
```

The `git rev-parse --git-common-dir` shell-out is what makes worktrees Just
Work: for a primary checkout it returns the same `.git`, while for a worktree
it returns the *parent* repo's `.git`. By taking `dirname(common)` of that,
the code gets the parent repo's working directory and uses *that* path for
the ID — so all worktrees that share a primary repo get the same ID. This is
elegant: one `git` call, no special-casing.

The deleted code is just as instructive. Gone: the `readCachedProjectId`
helper, the `git rev-list --max-parents=0 HEAD` shell-out, and the
`fs.writeFileString(... "opencode", id)` that wrote the cached ID into
`.git/opencode`. The resolution is now stateless on disk, which means
deleting `.git/opencode` no longer changes anything (and tests assert this —
see below).

**`project.test.ts` — the test matrix.** Three tests change semantics in
exactly the right way:

```ts
test("derives stable project ID from the canonical worktree path", async () => {
  await using tmp = await tmpdir({ git: true })
  const { project: a } = await run((svc) => svc.fromDirectory(tmp.path))
  const { project: b } = await run((svc) => svc.fromDirectory(tmp.path))
  expect(b.id).toBe(a.id)
  expect(a.id).toBe(ProjectID.fromDirectory(tmp.path))
})
```

This is the determinism test — same path, same ID — and it now also asserts
the ID equals `ProjectID.fromDirectory(tmp.path)`, locking in the encoding.
The most consequential test is the rename of "separate clones should share
project ID" to its negation:

```ts
test("separate clones of the same repo should get different project IDs", async () => {
  // ... bare remote, push, clone into a second directory ...
  const { project: a } = await run((svc) => svc.fromDirectory(tmp.path))
  const { project: b } = await run((svc) => svc.fromDirectory(clone))
  expect(a.id).toBe(ProjectID.fromDirectory(tmp.path))
  expect(b.id).toBe(ProjectID.fromDirectory(clone))
  expect(b.id).not.toBe(a.id)
})
```

This is the bug. Before this PR, the assertion was `expect(b.id).toBe(a.id)`
and that is *exactly the user-reported issue*: two unrelated clones
collapsing into one project. Flipping the assertion direction in the test is
the cleanest possible documentation of the intentional behavior change.

The cache-file assertions also flip from `exists ? true : false`:

```ts
const opencodeFile = path.join(tmp.path, ".git", "opencode")
expect(await Bun.file(opencodeFile).exists()).toBe(false)
```

Resolution no longer writes anything to disk; that is now contract-tested.

**`migrate-global.test.ts` — preserving migration intent.** The migration
test, which exercises moving sessions from the legacy `global` project ID
into a per-project ID, is updated to defer `git init` until after the
seeded session lands. Previously the test relied on "git init creates a
global, then a commit promotes it"; now the trigger for promotion is having
git initialized *at all*, since the ID derives from the path.

# Risks / things a maintainer would catch

1. **Path moves silently break state.** `mv ~/code/foo ~/code/foo-renamed`
   is no longer a no-op for `opencode` — it changes the project ID, which
   silently orphans every session, setting, and history record. The
   project never explicitly notifies the user. A startup-time check for
   "this directory looks like it might have been renamed" with an opt-in
   `opencode project migrate <old-path>` command would be a kindness.
2. **Privacy: the encoded path travels with the ID.** `dir_<base64url>` is
   trivially decodable. If sessions are ever exported, synced, or shared
   (telemetry, bug reports, opencode-cloud), the ID leaks the local
   filesystem layout. A SHA-256 of the canonical path with the same
   `dir_` prefix would preserve all the desirable properties (stable per
   path, distinct across paths, worktrees still collapse) without
   exposing `/Users/<name>/`.
3. **No migration path for existing users on the *old* scheme.** The PR
   silently changes IDs for everyone who had a non-global project. After
   upgrade, the tool will create new project rows for the new IDs and
   the old rows become orphan ghosts in the database. Either a
   one-shot migration (look up sessions whose project_id is a 40-char
   hex, find the corresponding directory, mint the new ID, rewrite) or
   an explicit "compat: also accept old-style IDs" lookup is needed.
4. **`realPath` is a syscall on every project resolution.** This runs on
   every `fromDirectory` call. If `fromDirectory` is hot (e.g. called per
   request in a server), the syscall overhead is non-trivial on slow
   filesystems. A small in-memory cache keyed on the input path would help.
5. **`base64url(path)` can produce extremely long IDs.** A path like
   `/Users/abc/Library/Application Support/foo/bar/baz/quux/...` can
   easily yield IDs >100 characters. Anywhere downstream that has a
   varchar(64) constraint (database column, S3 key prefix, JSON file
   name) will break. A grep across the schema for column lengths is in
   order.
6. **Windows path separators.** `pathSvc.normalize` is called, but the
   raw `directory` string still gets passed to `Buffer.from(directory)`
   in `fromDirectory`. On Windows a path containing backslashes
   round-trips through base64url just fine, but anyone constructing the
   ID by hand with forward slashes will get a different ID. Tests are
   POSIX-only here; a Windows-style fixture would lock down the
   convention.
7. **Worktree assumption depends on `--git-common-dir`.** This works for
   modern git but relies on git being installed. The "no git binary"
   branch (`if (!gitBinary)`) now derives from the *current* directory,
   not from any common parent — so a user without git who has their tool
   installed in subdir `foo/` versus `foo/bar/` will get two different
   IDs even though they probably want them unified. Fine in practice
   (most developers have git) but worth noting.
8. **Cache file `.git/opencode` is no longer written, but old ones
   linger.** Existing repos have `.git/opencode` files from prior
   versions. They are now ignored but still on disk. A one-time cleanup
   on first run with new version would be polite.
9. **Tests do not cover symlinked clones.** The `realPath` step is the
   most fragile new behavior; a test that creates `~/A` and a symlink
   `~/B -> ~/A` and asserts `fromDirectory("~/A").id === fromDirectory("~/B").id`
   would lock down the canonicalization contract.
10. **Same PR mixes 30-file rebase noise with a 4-file behavior change.**
    A maintainer should ask the author to rebase onto a clean base or
    cherry-pick the project-related commits; right now the PR is
    impossible to bisect cleanly if a regression appears in any of the
    unrelated touched files (UI dialog, container Dockerfile, etc.).

# Suggested questions for the author

1. Have you considered hashing the canonical path (e.g. SHA-256, prefixed
   `dir_`) instead of base64url-encoding it? You keep all the desired
   properties — stable per directory, distinct across directories,
   worktrees still collapse — and you avoid leaking the filesystem
   layout in any future telemetry or sync feature.
2. What is the migration story for users with existing project state
   keyed on the old root-commit ID? Without a one-shot rewrite, they
   silently lose their session history on upgrade.
3. Does the project want a `opencode project migrate <old-path>` command
   for the rename case, or is silent loss-of-state the intentional UX?
4. Is `fromDirectory` ever called in a hot loop (per request, per file
   open, per LSP event)? If so, the new `realPath` syscall on every call
   warrants a cache. If not, ignore.
5. Have you run this against a Windows host? The base64url encoding of
   a path with backslashes will differ from the same path with forward
   slashes; locking down which form is canonical (and adding a Windows
   test) would prevent silent ID divergence on `cmd.exe` versus
   PowerShell-launched sessions.
6. Should the unrelated diff churn (UI dialog, formatter, patch tooling,
   container Dockerfile) be split out of this PR before merge to keep
   the bisect surface clean?

# Verdict

**Needs design discussion.** The implementation is clean and the test
flips correctly capture the intended behavior change. But this is a
schema-affecting change to the *meaning* of an existing identifier
namespace, and shipping it without an explicit migration story silently
detaches every existing user's session history from their projects. The
right path forward is one of: (a) ship a one-shot migration that rewrites
old hex-style IDs to the new path-derived form, (b) keep both ID forms
valid in lookups for one release with a deprecation log, or (c) gate the
new behavior behind a config flag for one release and flip the default
later. Once the migration shape is decided, the code change itself is
ready.

# What I learned

The most reusable insight here is the use of `git rev-parse
--git-common-dir` as the *single* source of truth for "are we in a
worktree, and if so, what is the parent repo." The previous code had a
multi-step dance (read cache, fall back to rev-list, write cache) that
encoded the same intent through state. The new code lets git answer the
question directly and uses that answer to canonicalize the worktree path.
This is a good pattern for any tool that sits over git: instead of
maintaining a cache that mirrors git's own internal state, ask git via
its plumbing commands and treat the result as authoritative. The cost is
a process spawn per resolution; the benefit is correctness across every
git feature you have not thought of (sparse checkouts, submodules,
worktrees, alternates) because git already knows the answer.

The anti-pattern to avoid is the one this PR removes: writing your own
metadata file inside `.git/`. It seems convenient ("free per-repo
storage!") but it conflates *your tool's* state with *git's* state, with
the result that git operations users perform behind your back (clone,
worktree add, gc, fsck) silently mutate your tool's behavior. Better to
keep tool state in a tool-owned location and ask git for facts on demand.
