# Context

`opencode` ships its own programmatic npm install path (built on
`@npmcli/arborist`) so it can install plugins, MCP servers, and other
auto-discovered packages into a project-local directory without
shelling out to a user-installed `npm` binary. Until this PR the
Arborist instance was created with only the four hard-coded options
(`path`, `binLinks`, `progress`, `audit`) and called `reify({ add,
save, saveType })`. Anything users had set in their project `.npmrc`
— registries, auth tokens, `omit=dev`, `legacy-peer-deps`,
`prefer-offline`, scoped registry overrides — was simply ignored.

# Problem

Users on private registries reported that `opencode` would silently
fail to install a package that worked fine with `npm install` from
the same directory. Diagnosis was annoying because the failure mode
was *correct API call, wrong configuration* — Arborist talked to the
public registry instead of the corporate mirror configured in
`./.npmrc`. The same shape applied to `omit=dev` (devDependencies
were always pulled), `cafile` (custom CA bundles ignored), and any
scoped `@org:registry=...` line.

The root cause is structural: the Arborist constructor accepts a
flat options object that npm's CLI normally builds from a chain of
config sources (`./.npmrc`, `~/.npmrc`, `/etc/npmrc`, env vars,
defaults). Skipping that chain skips the user's intent.

# Design

The fix loads the canonical npm config exactly the way `npm` itself
does it, using `@npmcli/config`:

```ts
const npmPath = fileURLToPath(new URL("../..", import.meta.url))

const loadOptions = (dir: string) =>
  Effect.tryPromise({
    try: async () => {
      const config = new Config({
        npmPath,
        cwd: dir,
        env: { ...process.env },
        argv: [process.execPath, process.execPath],
        execPath: process.execPath,
        platform: process.platform,
        definitions, flatten, nerfDarts, shorthands,
        warn: false,
      })
      await config.load()
      return config.flat
    },
    catch: (cause) => new InstallFailedError({ cause, dir }),
  })
```

Two details worth flagging:

1. `npmPath` is resolved off `import.meta.url`, not the install
   target. That is correct — `npmPath` is supposed to point at the
   *npm installation* (so config can resolve built-in shorthands and
   defaults), while `cwd` is the install target where the
   project-local `.npmrc` lives. Mixing those up would silently
   break scoped registries.
2. `argv: [process.execPath, process.execPath]` is a known
   `@npmcli/config` quirk: the loader expects an `argv` array shaped
   like Node's, and refuses to parse without two leading entries.
   The repeated path is fine — none of the parsed argv values feed
   the install path.

The flattened config is then spread into both the `Arborist`
constructor *and* the `reify({...})` call:

```ts
const npmOptions = yield* loadOptions(input.dir)
const arborist = new Arborist({ ...npmOptions, path: input.dir, binLinks: true, ... })
return yield* Effect.tryPromise({
  try: () => arborist.reify({ ...npmOptions, add, save: true, saveType: "prod" }),
  ...
})
```

Spreading at both layers matches npm's own internals: Arborist reads
some options at construction (registry, auth, network), others at
reify time (`omit`, `save-prefix`). Spreading once would silently
drop one half.

# Risks

- **Token leakage in error reporting.** `npmOptions` contains
  resolved auth tokens. If `InstallFailedError` ever serializes its
  `cause` to a log sink, those tokens land in logs. The PR does not
  redact. The `cause` object here comes from Arborist's own throw
  path, so the immediate risk is bounded, but a `Sanitize.npmOptions`
  helper that strips `_auth`, `_authToken`, `//*:_password`,
  `//*:_authToken` keys before any error path would be cheap
  insurance.
- **`env: { ...process.env }`.** Inheriting the host process env
  means CI-time variables like `NODE_AUTH_TOKEN` and `npm_config_*`
  apply. Usually desirable, but it does mean an `opencode`
  invocation inherits ambient secrets that the user may not realise
  are scoped to this install. A `Config({ env: filteredEnv })`
  knob would let advanced users override.
- **First-call latency.** `config.load()` does sync FS walks for
  every parent dir up to root looking for `.npmrc`. Cheap on SSDs,
  noticeable on network-mounted home directories. No caching layer
  here; if `Npm.install` is called repeatedly per session, each
  call re-walks. Memoising by `dir` would save 5–20 ms per call.
- **Test only covers `omit=dev`.** The big regression risks
  (private registries, scoped registries, `cafile`) are not
  exercised. They are harder to test hermetically, but a fixture
  with a single-line `.npmrc` setting `registry=http://127.0.0.1:0`
  and asserting that Arborist tries that host (mocked) would lock in
  the wire path.

# Suggestions

1. Add a `redact` helper applied to `cause` before it lands in
   `InstallFailedError`. The patch surface is small and the upside
   is large.
2. Memoise `loadOptions(dir)` per session — the config rarely
   changes mid-install.
3. Document in code that `npmPath` intentionally points at the
   bundled `opencode` package root, not the user project, so future
   refactors don't "fix" it.

# Verdict

**Ship it.** This is the right way to delegate config: lean on
`@npmcli/config`, don't hand-roll an `.npmrc` parser, and accept
that the user's npm intent is the source of truth. The +73/-2 line
budget is impressively tight for the amount of behaviour unblocked.

# What I learned

Whenever you embed a tool's library API instead of shelling out to
its CLI, you inherit the obligation to reproduce the CLI's config
loading. The CLI is not just `tool.do_thing()` — it is *also* a
config-resolution pipeline that the rest of the world has tuned
their workflows around. Skipping that pipeline is the most common
"works in npm but not in our tool" failure mode for any
embed-the-library architecture (npm, git, ruff, eslint, prettier).
The fix, every time, is the same shape this PR uses: import the
tool's own config crate, give it the same `cwd`/`env` you would
give the CLI, hand the resulting flat object to the API. The
moment you start re-implementing config resolution, you are
guaranteed to drift.
