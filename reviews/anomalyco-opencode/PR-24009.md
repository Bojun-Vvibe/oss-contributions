# Context

In `opencode`, a single "user message" on the wire isn't always a
single block. The session stream coalesces several distinct
`text` parts into one user turn: the literal text the human
typed, plus any *synthetic* parts injected by the runtime —
attached file digests, `@`-mention expansions, slash-command
preambles, system-injected reminders. Each part carries a
`synthetic: bool` flag (and sometimes a `source` tag) so the TUI
can decide what to show.

The TUI's job is to render the user-visible parts and hide the
synthetic ones. Up until this PR, the rendering loop short-
circuited on the *first* non-synthetic part it found and dropped
the rest. For most messages that worked — there is exactly one
typed text block. The bug shows up when a paste or an editor
roundtrip creates two non-synthetic blocks (e.g. typed text +
"continued from clipboard"), or when a future producer wants to
attach a second human-authored region to one turn.

# Problem

The original loop:

```go
for _, part := range msg.Parts {
    if part.Synthetic {
        continue
    }
    if textPart, ok := part.(TextPart); ok {
        return renderText(textPart.Text) // <-- early return
    }
}
```

If a user message has two `TextPart`s and both are
`Synthetic == false`, only the first is rendered. The second
silently disappears from the TUI but is still sent to the model,
so the model "sees" content the user can't see in their own
scrollback. That is the worst class of UI bug for an agent: the
human can't audit what the agent reacted to.

The reporter found this by pasting two-paragraph input where the
TUI's paste handler split the paste into two `TextPart`s for
preserve-newlines reasons. They saw paragraph 1 in the
transcript, asked a follow-up, and got an answer that
referenced paragraph 2 — which they had no record of having
sent.

# Design

The fix replaces the early-return with an accumulating loop:

```go
var rendered []string
for _, part := range msg.Parts {
    if part.Synthetic {
        continue
    }
    if textPart, ok := part.(TextPart); ok {
        rendered = append(rendered, renderText(textPart.Text))
    }
}
return strings.Join(rendered, "\n\n")
```

Three things to like:

1. **`\n\n` join, not `\n`.** Two separately-emitted text parts
   semantically came from two distinct producers; rendering them
   as paragraph siblings (blank line between) preserves the
   "these are two things, not one thing with a newline" intent.
   Single-`\n` would have hidden the bug visually.
2. **`Synthetic` is the only filter.** The PR resists the
   temptation to add a "first non-synthetic only" mode behind a
   flag. Good — that flag would just re-introduce the bug under
   a different name.
3. **No change to message ordering or part identity.** The
   rendered loop preserves wire order, which matters for
   reproducing what the model saw.

The added test seeds a message with `[synthetic-A, text-1,
synthetic-B, text-2]` and asserts the output is
`"text-1\n\ntext-2"`. It also adds the inverse check — a
message with only synthetic parts renders to empty string,
not to a panic. Coverage is tight for the immediate bug.

# Risks

- **Image / audio parts in user turns.** The current loop only
  handles `TextPart`. If a user message contains
  `[text-1, image-1, text-2]` — which is already possible via
  the attach UI — the image renders via a separate path
  elsewhere, and this loop will collapse text-1 and text-2 to
  `"text-1\n\ntext-2"` with the image effectively absent from
  this particular renderer's output. That's likely fine because
  images are rendered by a different component, but a comment
  saying so would prevent a future "drive-by fix" from
  re-introducing the early return out of confusion.
- **Synthetic detection vs source-of-truth.** `Synthetic` is a
  best-effort flag set by the producer. If a producer forgets
  to mark a system-injected reminder as `Synthetic: true`, the
  TUI will now show it to the user (where before it was hidden
  by the early-return collapsing it into the first typed
  block). This is arguably the correct behaviour — better to
  show a reminder you didn't intend to than to hide one you
  did — but it's a behaviour change worth calling out in the
  PR description.
- **No re-render on part arrival.** If user parts arrive
  incrementally (not currently the case for user turns, but
  the assistant path does it), the loop runs once at message
  finalization. If that ever changes, the join will need to be
  idempotent across re-renders.

# Suggestions

1. Add a regression test for the paste case that triggered
   the report: a single user message with a `\n\n`-separated
   paste where the producer chose to emit two parts. That
   matches the field repro 1:1.
2. Consider rendering the `source` tag (if present) as a small
   muted prefix when there are >1 non-synthetic parts. The user
   in the linked issue noted "I had no idea those were two
   blocks" — a subtle visual cue would have prevented the
   confusion that caused the report.
3. Audit other consumers of `msg.Parts` (export-as-markdown,
   `/history` command, the dump-session debug view) for the
   same first-non-synthetic-only pattern. The bug is the kind
   of thing that gets copy-pasted.

# Verdict

**Merge.** Small, correct, well-tested, fixes a bug whose worst
property is invisibility. The accumulator pattern is the right
default for "render N matching items" — the early-return form
should be considered a smell whenever it's applied to a
filtered iteration.

# What I learned

There's a recurring failure mode in TUI rendering for agent
chats: *the rendered transcript and the model's input drift
apart, and neither side notices*. The renderer optimizes for
"don't show clutter"; the wire optimizes for "don't lose
data". When the renderer's filter is more aggressive than the
wire's, you get this bug — content goes to the model but not
to the user's eyes. When the wire's filter is more aggressive
than the renderer's, you get the inverse — the user sees
something the model never saw, and follow-up questions land in
a vacuum.

The defensive posture is to make the renderer a strict subset
filter of the wire: anything the renderer hides must be
provably synthetic, and "first text part" is not a proof of
syntheticness. This PR enforces that subset relationship by
removing the early return. The general lesson: if your
"hide noise" rule is positional (first / last / nth), it is
almost certainly wrong as soon as the producer changes its
emission shape.
