# Context

`opencode` is a TypeScript terminal coding agent built on the Vercel AI SDK
abstraction. Like every agent, it shuttles assistant messages back to the
model on every turn so the LLM has the full conversation in context. Anthropic
recently formalized a hard constraint on this round-trip: when an extended
thinking response comes back from `claude-*` (a mix of `thinking` and
`redacted_thinking` blocks), those blocks must be sent back **byte-for-byte
identical** in subsequent requests, with their `signature` and `redactedData`
fields intact. Modify any of them and the API rejects the next turn with
`messages.X.content.Y: 'thinking' or 'redacted_thinking' blocks in the latest
assistant message cannot be modified.` That terminates the session — there is
no graceful recovery short of dropping the whole assistant turn.

The Vercel AI SDK normalizes thinking blocks to a uniform
`{ type: "reasoning", text, providerOptions }` shape. `redacted_thinking` is
particularly nasty because the SDK gives it `text: ""` and stuffs the opaque
encrypted blob into `providerOptions.anthropic.redactedData`. From a naïve
"strip empty content" perspective these blocks look exactly like the noise you
would normally want to filter — except they are load-bearing.

This PR repairs three independent code paths in
`packages/opencode/src/provider/transform.ts` (the layer that runs between the
agent loop and the model call) that all happened to mutate or drop reasoning
parts. The matter is more important than the LOC count suggests: a single
mistake here turns extended-thinking sessions into single-shot interactions,
which silently degrades model quality on the next turn and looks to users like
"the agent forgot what it was doing."

# Diff walkthrough

The substantive change is in `packages/opencode/src/provider/transform.ts`
across three hunks; the rest of the +800/-93 is rebase churn (`bun.lock`,
`package.json`, container `Dockerfile`, sidebar/avatar UI, formatter and
patch tooling) that should ideally be split off but does not affect the fix.

**Hunk 1 — empty-content filter (around line 61).** The original code
treated `text` and `reasoning` symmetrically and dropped both if their
`text` field was empty:

```ts
const filtered = msg.content.filter((part) => {
  if (part.type === "text" || part.type === "reasoning") {
    return part.text !== ""
  }
  return true
})
```

That is fine for `text` (an empty assistant text block is genuinely useless)
but catastrophic for `reasoning`, because `redacted_thinking` is *defined*
as empty text plus opaque metadata in `providerOptions`. The fix splits the
two cases:

```ts
if (part.type === "text") {
  return part.text !== ""
}
if (part.type === "reasoning") {
  // Keep reasoning parts that carry providerOptions (e.g. redacted_thinking
  // blocks which have empty text but contain signature/redactedData metadata
  // that must be preserved for the Anthropic API)
  return part.text !== "" || part.providerOptions != null
}
```

This is the cleanest of the three changes. The predicate
`text !== "" || providerOptions != null` correctly drops "truly empty"
reasoning placeholders that some streaming paths emit, while preserving any
reasoning part that carries provider metadata. There is one minor sharp edge:
the predicate keys off `providerOptions` *existing* rather than specifically
having an Anthropic key. If another provider ever populates `providerOptions`
on an empty reasoning block for unrelated reasons, those will now be retained
and shipped to Anthropic. In practice this is unlikely (transforms are model-
gated upstream) but worth a defensive check.

**Hunk 2 — assistant message reordering (around line 122).** Anthropic's API
requires that within an assistant message, content blocks come in a specific
order, and the existing transform splits a message that goes
`[..., tool_call, ..., text, ...]` so that the tool calls land in their own
message. The split keeps the model happy in the *non-thinking* case but
re-parents any `thinking` blocks present, because the thinking blocks land in
the wrong half of the split. Anthropic then sees the thinking block on a new
message and rejects:

```ts
const parts = msg.content
// Don't reorder messages containing reasoning/thinking blocks — Anthropic
// requires thinking and redacted_thinking blocks to remain exactly as returned.
if (parts.some((part) => part.type === "reasoning")) return [msg]
const first = parts.findIndex((part) => part.type === "tool-call")
```

The fix is the simplest possible: if any reasoning is present, bail out of
the reorder and return the message untouched. This is correct and minimal,
but it does inherit a behavioral asymmetry — the reorder still happens for
plain tool-using turns, which means the codebase now has *two* canonical
shapes for an assistant message depending on whether the upstream model
emitted thinking. Downstream code that walks message content needs to handle
both.

**Hunk 3 — cache-control hint placement (around line 255).** This is the
subtlest of the three. The provider transform applies an Anthropic prompt-
cache breakpoint by attaching `providerOptions: { anthropic: { cacheControl:
{ type: "ephemeral" } } }` to the *last* content block of the message. The
old code just looked at `msg.content[msg.content.length - 1]` and skipped only
two specific tool-approval types:

```ts
const lastContent = msg.content[msg.content.length - 1]
if (
  lastContent &&
  typeof lastContent === "object" &&
  lastContent.type !== "tool-approval-request" &&
  lastContent.type !== "tool-approval-response"
) {
```

If the last block was a `reasoning` block (which can happen on extended-
thinking responses where the model thinks, calls a tool, then thinks again
without a final text block), the transform would slap a `providerOptions`
mutation onto it — corrupting the byte-for-byte requirement. The fix walks
the array in reverse to find the last *non-reasoning, non-approval* block:

```ts
const lastContent = [...msg.content]
    .reverse()
    .find(
      (part) =>
        part &&
        typeof part === "object" &&
        part.type !== "tool-approval-request" &&
        part.type !== "tool-approval-response" &&
        part.type !== "reasoning",
    )
if (lastContent) {
  lastContent.providerOptions = mergeDeep(lastContent.providerOptions ?? {}, providerOptions)
  continue
}
```

Note the `continue` — if there is no eligible block at all (e.g. the message
is *only* reasoning), the cache hint is silently skipped. That is the right
default but worth flagging in a comment, because it means cache control
silently degrades on thinking-only turns. The `[...msg.content].reverse()`
makes a copy, which is correct for not mutating the caller's array; a more
allocation-aware version would just walk an index downward, but this code
is not on a hot path.

**Tests.** The test file (`test/provider/transform.test.ts`) gains a
~440-line `describe("ProviderTransform.message - preserve redacted_thinking
blocks", ...)` block that exercises all three regressions independently with
explicit Anthropic model fixtures. The matrix is good: empty reasoning with
metadata is preserved; empty reasoning without metadata is still filtered;
multiple interleaved thinking/redacted blocks survive; reordering is
suppressed in the presence of reasoning; cache hints land on the right block.
These read like maintenance-grade regression tests — each one is anchored to
a specific failure mode with a clear preconditions/postconditions structure,
which is exactly what you want for "we hit this in prod" fixes.

# Risks / things a maintainer would catch

1. **PR is a stealth rebase.** The diff includes 22 changed files but only
   `transform.ts` and the test file are part of the fix. Files like
   `dialog-edit-project.tsx`, `format/index.ts`, `patch/index.ts`,
   `plugin/codex.ts`, the container `Dockerfile`, and `bun.lock` should not
   be in the same review unit. A maintainer should ask the author to rebase
   onto a clean base or split the PR before merge — otherwise the bisect
   surface for any future regression in those files balloons.
2. **`providerOptions != null` is a coarse predicate.** It keeps any
   reasoning part with provider metadata, not specifically Anthropic
   metadata. If another provider plugin attaches `providerOptions` to an
   empty reasoning block (e.g. a future Bedrock or Vertex adapter), those
   blocks will now leak through to Anthropic. Tightening to
   `part.providerOptions?.anthropic != null` or gating the predicate on the
   model's provider would be safer.
3. **Behavioral asymmetry from the early return in Hunk 2.** Skipping the
   tool-call reorder for thinking messages means the codebase now has two
   shapes for assistant messages on the wire. Anything downstream that
   assumed reorder always runs (telemetry, cost computation, message
   chunking for UI rendering) may behave differently on thinking turns. A
   grep for callers of `normalizeMessages` would be wise.
4. **Cache hint silently dropped when message is reasoning-only.** The
   `continue` means cache breakpoints are not placed on thinking-only turns.
   This subtly changes prompt-cache hit rate for long extended-thinking
   sessions; not necessarily wrong, but the user-visible billing
   characteristic should be acknowledged.
5. **No test for the "Hunk 3 falls through to nothing" path.** All three
   tests still place a real text or tool-result block after the reasoning,
   so the `if (lastContent)` branch is always taken. A test where
   `msg.content` is *exclusively* reasoning would lock in the silent-skip
   behavior and prevent a future refactor from re-introducing the bug.
6. **Mutates input objects.** `lastContent.providerOptions = mergeDeep(...)`
   writes back to a member of the caller's array. The earlier filter step
   already mutates by replacing messages, so this is consistent with the
   file's style — but a test that calls `ProviderTransform.message` twice
   on the same input would catch any non-idempotency.
7. **`as any` casts in tests.** The test fixtures use `as any` to construct
   the model object. That is fine for this PR but it means the test does
   not actually exercise the public type contract of `ProviderTransform`.
   If `ModelMessage` shape changes upstream the tests will keep passing
   while production breaks.
8. **No guard against the SDK changing its `redacted_thinking`
   representation.** This fix encodes the assumption that the AI SDK
   continues to express redacted blocks as `{ type: "reasoning", text: "",
   providerOptions: { anthropic: { redactedData } } }`. A version bump in
   `@ai-sdk/anthropic` could change that shape (for instance to
   `type: "redacted-reasoning"`) and silently re-break the issue. A
   shallow-check helper `isRedactedThinking(part)` would localize the
   coupling and make the failure obvious in tests.
9. **No log/metric on filtering.** Today, when a reasoning block is
   filtered out as truly empty, there is no log line. If a future bug
   strips a metadata-bearing reasoning block by accident, the only signal
   will be Anthropic's 400. A debug-level log in the filter callback when
   it drops a reasoning part would be a cheap diagnostic.
10. **Cross-platform: none of these tests run end-to-end against the
    Anthropic API.** They are unit tests on the transform. A CI-skipped
    integration test that issues a real extended-thinking call and a
    follow-up turn would have caught the original regression.

# Suggested questions for the author

1. Can the unrelated changes (UI dialog, formatter, apply_patch, container
   Dockerfile, bun.lock) be split out of this PR? They make review and
   future bisect harder, and if any of them gets reverted later it will
   undo this fix as collateral.
2. Should the reasoning-preservation predicate be tightened to specifically
   `part.providerOptions?.anthropic != null`, so that a future provider
   adapter that puts metadata on empty reasoning blocks does not leak
   them into the Anthropic transform?
3. In Hunk 3, when the message contains only reasoning blocks, the cache
   breakpoint is silently dropped. Is this the intended billing/perf
   behavior, or should the cache hint be applied at the *message* level
   (`providerOptions` on the message object) as a fallback?
4. Are there other callers in the codebase that walk assistant message
   content and assume the post-reorder shape (`tool_calls` always in their
   own message)? Now that thinking-bearing messages skip the reorder,
   those callers may need updating too.
5. Is there an integration-test layer that issues a real extended-thinking
   call plus a follow-up turn? If so, can a regression test be added there
   so the next refactor of `transform.ts` cannot silently re-break this?
6. The new tests use `as any` for the model fixture — would it be worth
   factoring a shared `makeAnthropicModelFixture()` helper that returns a
   properly typed `Provider.Model`? It would remove the `as any` and let
   the type checker catch shape drift.

# What I learned

The valuable pattern here is "the wire format is load-bearing in places
that look like noise." The `redacted_thinking` block is a canonical example:
its in-memory representation (`text: ""` plus an opaque blob in side-channel
metadata) deliberately hides information from middleware so that opaque
encrypted state can survive a round-trip. Any normalization layer between
your agent and an LLM provider needs an explicit "do not touch" predicate
for these payloads, and the predicate should be expressed positively
("preserve when X") rather than negatively ("filter unless Y") — because the
default-allow shape protects you against the common "we added a new opaque
type and forgot to update the deny list" failure mode. The opposite anti-
pattern, which this PR fixes, is exactly that: the original empty-content
filter was a single condition that conflated two semantic categories
(reasoning and text) because they happened to share a key. The lesson is
that as soon as a transform handles two types with the same code path, it
should be split, even if the bodies are identical, so future maintainers
have a stable place to attach divergent logic. The cache-hint hunk teaches
a related lesson: when reaching for "the last element" of an array of
heterogeneous content, prefer `findLast`-style selection with an explicit
eligibility predicate over indexing — the predicate is the documentation of
what you actually meant.
