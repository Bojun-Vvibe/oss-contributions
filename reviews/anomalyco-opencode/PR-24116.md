# Context

`packages/opencode/src/snapshot/index.ts` implements the
session-shadow-snapshot revert path. When the user (or an
agent) reverts a snapshot, the layer batches the affected
files by source commit and shells out one `git checkout
<hash> -- <files...>` per batch. The file lists come from
the snapshot's tracked-paths set and can grow with the
worktree size, especially in monorepos where a single
turn touches hundreds or thousands of files (think a
codemod, a generated-code refresh, a `pnpm install` that
shuffles `node_modules` symlinks). Closes #23758.

# Problem

Posix exec has a hard cap on argv+env byte length —
`ARG_MAX`. On Linux glibc it's typically 128KB-2MB
depending on stack size and kernel; on macOS it's 256KB
by default. When the file list to revert pushes the
flattened argv past that ceiling, `execve` returns
`E2BIG` and `git checkout` fails before it ever runs.
The user sees a revert that "didn't work" with a
non-obvious error code, and the snapshot is now in a
half-applied state because earlier batches in the loop
*did* succeed.

The trigger is environment-dependent (depends on
absolute path lengths, shell, kernel limits) so this is
the kind of bug that lurks until a power user runs into
it on a deep nested monorepo and cannot reproduce it on
a fresh clone.

# Design

```ts
const result = yield* git(
  [
    ...core,
    ...args(["checkout", "--pathspec-from-file=-", "--pathspec-file-nul", first.hash]),
  ],
  {
    cwd: state.worktree,
    stdin: feed(list.map((item) => item.rel)),
  },
)
```

The fix routes the file list through git's
`--pathspec-from-file=-` flag, which reads the pathspec
list from stdin instead of argv. `--pathspec-file-nul`
makes the entries NUL-delimited, which is the only safe
choice when paths can contain newlines or special
characters (rare but real on case-insensitive filesystems
with weird filenames). The change is +5/-1 in one file.
Argv now has a small, bounded set of flags — the
unbounded list moves to a stream.

This is the canonical fix for E2BIG. `xargs` users have
been doing it forever; git ships first-class support
specifically because revert/checkout/restore on large
file sets is the obvious case where it matters.

# Risks and edges

- **`item.rel` vs `item.file`.** The old code used
  `item.file`; the new code uses `item.rel`. The diff
  doesn't tell us what shape these are. Worth confirming
  that `rel` is the worktree-relative path that git
  actually expects in pathspec — git's pathspec parser
  is finicky about absolute vs relative depending on
  context. If `file` was absolute (likely, since it's
  what the old argv was), the switch is correct because
  pathspecs from `--pathspec-from-file` are interpreted
  relative to the current directory (which is
  `state.worktree`). But the swap is silently load-
  bearing — a regression test that touches a file two
  levels deep and asserts on the actual checkout would
  catch a mistake here.
- **`feed()` helper is not in the diff.** Reviewer
  needs to confirm it (a) writes the bytes, (b) closes
  stdin so git knows the list is done, and (c)
  appends the trailing NUL after the last entry (or
  not — `--pathspec-file-nul` reads NUL-separated, not
  NUL-terminated, so the last entry should NOT have a
  trailing NUL or git may treat it as an empty
  pathspec, which is "match everything" and would
  silently revert the whole tree). This is the single
  most dangerous edge in the PR; a one-line bug here
  becomes a `git checkout HEAD -- ''` that nukes
  unrelated working-tree changes.
- **`feed(list.map((item) => item.rel))` builds the
  full string in memory before piping it.** For a
  20K-file revert that's ~1MB of paths — fine. For
  pathological cases (1M files in a monorepo refresh)
  it's ~50MB, which is still fine on modern boxes but
  makes the chunked-write streaming version a future
  improvement worth noting if revert sizes ever grow
  enough to matter.
- **No batch-size cap was removed.** Looking at the
  surrounding code (not in the diff), the layer still
  batches by source commit. So very large reverts that
  touch files from many distinct commits still issue
  many checkouts; only the per-batch argv size went
  away. That's the correct scope.
- **No new tests.** The PR claims "Ran `bun test
  test/snapshot/snapshot.test.ts`" but does not add a
  regression test. The right test is: build a snapshot
  with N files where N×average-path-length pushes
  argv past 128KB, revert, assert success. On macOS
  with `getconf ARG_MAX = 1048576` you'd need ~10K
  files at avg path 100 chars; on Linux often less.
  Without this, a future refactor that "simplifies"
  back to argv would silently reintroduce the bug
  with no test to catch it.
- **Error path on the `git` command itself is
  unchanged.** The existing `if (result.code !== 0)`
  block still handles failures the same way — that's
  correct, the change doesn't alter error semantics,
  just the size envelope.
- **Cross-platform.** Windows doesn't have `E2BIG`
  per se but has its own ~32KB `CreateProcess`
  command-line cap, which this fix also dodges. So
  the change is unambiguously better on every
  platform.
- **Snapshot-revert atomicity is unchanged.** Still
  no transaction across batches. If the second batch
  fails after the first succeeded, the worktree is
  partially reverted. That's pre-existing and out of
  scope, but worth noting that the new path raises
  the success ceiling per batch, which means a user
  who used to fail at 5K files and now succeeds at
  500K files has a much wider blast radius if a
  later batch fails. A "revert plan / dry-run /
  validate-then-apply" mode is the structural fix;
  this PR is the right tactical fix to ship now.

# Verdict

Approve. Correct direction, canonical mechanism,
right-sized diff. Two requests before merge:

1. Add a regression test that constructs a snapshot
   with enough files to exceed argv limits on the
   tightest target platform (Linux ARG_MAX ≈ 128KB)
   and asserts revert succeeds.
2. Confirm `feed()` semantics for trailing NUL — if
   it adds one, the last pathspec is empty, which
   silently reverts the entire worktree. This needs
   to be unit-tested explicitly.
