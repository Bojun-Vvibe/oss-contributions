# Context

`opencode` lets users set per-provider HTTP headers via
`provider.options.headers` in `opencode.json`. The AI SDK
forwards those headers through `config.headers()`, but
`session/llm.ts` also attaches a default `User-Agent:
opencode/<version>` at the per-request layer through
`withUserAgentSuffix` and a `combineHeaders` pass. HTTP header
keys are case-insensitive; the SDK lowercases on merge; a user
header named `User-Agent` and the framework header
`User-Agent` collide on the same lowercased key, and the
last-write-wins ordering puts opencode's default on top.

The downstream effect is that gateways and proxies that gate on
a specific `User-Agent` (Kimi's CLI tier, several enterprise
LLM proxies, Cloudflare WAF rules tuned for a known client)
silently reject opencode requests even though the user
configured the header correctly.

# Problem

PR #12072 had previously removed an earlier fetch-level header
merge after it caused a different regression. That removal was
correct in scope but it left a second collision in place at the
session layer that no test covered. The bug therefore looked
like "opencode ignores my header" from the user's side and
like "we already fixed that" from the maintainer's side — a
classic regression that survives because no test asserts the
post-merge header value at the wire.

# Design

The fix is a one-line predicate plus its usage: only inject
the default `opencode` User-Agent when the user has not already
configured one on the provider. Concretely the `withUserAgentSuffix`
call is gated by checking whether the resolved provider header
bag already contains a case-insensitive `user-agent` entry. If
it does, the user's value is left untouched. If it does not,
the default suffix is appended exactly as before.

Model-level headers (set per-model in config) and plugin-hook
headers (added by `tool.before` / `request.before` plugins)
keep working unchanged because they are spread after the base
header object — the merge order there is the user's intent
overriding the framework, which is what they want.

The test is the load-bearing part of this PR. It stands up a
real Bun HTTP server, configures a custom
`@ai-sdk/openai-compatible` provider pointed at it, sets
`User-Agent: KimiCLI/1.30.0` plus `X-Custom-Trace` in
`options.headers`, drives a real stream through `LLM.stream`,
and asserts the captured request headers contain
`KimiCLI/1.30.0`, do not contain `opencode/`, and forward
`x-custom-trace`. This is end-to-end at the wire — exactly
the layer that the bug lived at and that the prior tests
missed.

# Risks and edges

- **The `user-agent` lowercase check must match the SDK's
  normalization.** The PR uses
  `Object.keys(headers).some(k => k.toLowerCase() === "user-agent")`
  which works regardless of input casing. Worth confirming the
  same predicate runs against the *resolved* header bag (after
  any plugin hooks) and not just the raw config — the diff
  appears to do this correctly inside `LLM.stream`, but a
  future plugin that strips the user's UA between config and
  request would silently re-enable the default. A follow-up
  should consider running the check at the latest possible
  point, immediately before the fetch.
- **The test asserts absence of `opencode/` substring.** That is
  fine for the regression but if the SDK ever adds an
  unrelated `opencode/x-trace` header for telemetry, the
  assertion will flake. Tighter would be
  `expect(req.headers.get("user-agent")).toBe("KimiCLI/1.30.0")`.
- **Plugin authors may now want a way to *force* the default
  suffix back on**, e.g. an OpenTelemetry plugin that wants
  attribution. There is no escape hatch. Given the rarity of
  that use case it does not need to land in this PR, but the
  design space is now closed; reopening it later means
  another header.
- **No CHANGELOG / docs update.** The behavior change is
  user-visible and worth one line in the provider docs page
  ("setting `User-Agent` in `options.headers` now wins").

# Verdict

Approve. The fix targets the exact collision, the gating
predicate is conservative (only fires when the user opted
in), and the regression test exercises the bug at the wire
layer where the previous test gap lived. The follow-up items
are small and can land separately.
