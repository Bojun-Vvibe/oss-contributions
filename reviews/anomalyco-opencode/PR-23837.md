# Context

`opencode` exposes two TUI entrypoints that accept `--session <id>`:
the local `opencode --session ses_xxx` flag and the `opencode attach`
remote subcommand. Both bring up the full Solid-based TUI before
checking whether the session even exists. If the ID is malformed
(`not-a-session`) or simply unknown to the server (`ses_missing`), the
user landed in a fully-rendered TUI shell with an empty body and no
indication of what went wrong. Worse, the lookup happened inside a
`createEffect(async () => …)` with `throwOnError: true`, which on
failure surfaced as an unhandled promise rejection rather than a toast
or graceful navigation back to home.

# Problem

The session-route effect (`routes/session/index.tsx`) was an
`async` function passed directly to `createEffect`. SolidJS's
`createEffect` does not await the returned promise, so any rejection
escapes into the global unhandled-rejection channel. The lookup also
re-read `route.sessionID` *after* the await, which means a fast
navigation to a different session during the in-flight request could
trigger spurious toasts/navigation against the new session ID. And
there was no validation of the ID format before the network call —
clearly invalid IDs round-tripped to the server only to come back
with whatever shape the SDK chose to throw.

# Design

Three changes, all complementary.

**(1)** A new `validate-session.ts` module gives both entrypoints a
single shared pre-flight:

```ts
const result = SessionID.zod.safeParse(input.sessionID)
if (!result.success) {
  throw new Error(`Invalid session ID: ${result.error.issues.at(0)?.message ?? "unknown error"}`)
}
await createOpencodeClient({ baseUrl: input.url, ... })
  .session.get({ sessionID: result.data }, { throwOnError: true })
```

This runs *before* the TUI mounts. `attach.ts` and `thread.ts`
identically:

```ts
try {
  await validateSession({ url, sessionID: args.session, directory, ... })
} catch (error) {
  UI.error(errorMessage(error))
  process.exitCode = 1
  return
}
```

The user sees a one-line error on stderr and gets exit code 1, which
makes the failure mode scriptable.

**(2)** The session-route effect is restructured so the async work is
captured in a synchronous outer effect that closes over `sessionID`:

```ts
createEffect(() => {
  const sessionID = route.sessionID  // snapshot at effect-creation
  void (async () => { ... })().catch((error) => {
    if (route.sessionID !== sessionID) return  // stale, ignore
    toast.show({ message: errorMessage(error), variant: "error", duration: 5000 })
    navigate({ type: "home" })
  })
})
```

The `if (route.sessionID !== sessionID) return` guard is the key bit:
it prevents stale rejections from a navigated-away-from session from
nuking the current view. The same guard is applied to the success-
path scroll: `if (route.sessionID === sessionID && scroll) scroll.scrollBy(...)`.

**(3)** `errorMessage()` in `util/error.ts` learns to drill into
`error.data.message` (line 29-37 of the diff), which is where the
SDK's HTTP-error shape stashes the server-side message. Without this,
"Session not found" came out as `[object Object]`.

# Risks

- **Double network call on the local TUI path.** `thread.ts` now calls
  `session.get()` once during validation, then the route effect calls
  it again on mount. For an existing session this is wasted work
  (~2 round-trips become ~3). A small in-process cache keyed by
  `sessionID` with a 1s TTL would close that gap.
- **`process.exitCode = 1; return` instead of `process.exit(1)`.**
  Correct (lets cleanup hooks run) but only works because nothing
  else has been started yet. If a future change hoists any setup
  before the validation block, the process will keep running with a
  pending exit code.
- **`SessionID.zod.safeParse` happens twice if both validate-session
  *and* the SDK re-validate**. Negligible, but worth a comment.
- **The new `attach.ts` validation runs before any auth header
  rotation.** If the auth function is stateful (refresh tokens), the
  first validation call may use a stale token while the post-mount
  call uses a fresh one. Probably fine; worth eyeballing the auth
  layer.

# Suggestions

1. Factor the `try/catch + UI.error + exitCode + return` block into a
   helper (`runOrExit`) — `attach.ts` and `thread.ts` have it
   verbatim, which is a future drift hazard.
2. The stale-effect guard pattern (`if (route.sessionID !== sessionID)
   return`) could move into a tiny `withFreshRoute` helper — there are
   probably other route effects that need the same protection.
3. `errorMessage()` is now scanning four shapes; consider replacing
   the cascading `if`s with a small ordered table of `(predicate,
   extractor)` pairs.

# Verdict

**Ship it.** This is a small, well-targeted UX fix that closes a
genuine reliability hole (unhandled rejection on bad session ID) and
gets the right composability (shared `validateSession`, scriptable
exit code, clean error message). The stale-effect guard is also a
real bug fix that goes beyond the stated goal of the PR.

# What I learned

`createEffect(async () => …)` is a footgun in any reactive framework
that does not await effect bodies (Solid, MobX `autorun`, Vue
`watchEffect` in some configurations). The two failure modes are
(a) unhandled rejections leaking out, and (b) stale-closure bugs when
the reactive source changes during the await. The pattern this PR
adopts — synchronous outer effect, captured value, IIFE for the
async work, explicit `.catch()` with a freshness guard — is the
canonical way to make `createEffect` and `async` cooperate. It is
verbose enough that it deserves to be wrapped in a helper the moment
it appears in two places.
