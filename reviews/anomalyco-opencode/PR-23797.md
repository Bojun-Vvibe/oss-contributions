# Context

`opencode`'s file-modifying tools (`write`, `edit`, `apply_patch`) read a
file, build a unified diff, ask permission, write back, and finally pipe
the result through the formatter. This pipeline went through `string`
the whole way, which is a perfectly fine model for the file body — but
it is the wrong model for files that begin with a UTF-8 byte-order mark
(`U+FEFF`). The BOM is a *sidecar* signal: tools like Notepad on
Windows, the Visual Studio C# compiler family, and a number of build
systems either rely on its presence or get visibly upset when an
unrelated tool flips it. Up until this PR the agent silently flipped it
in either direction depending on which transform happened to run last,
and that showed up downstream as "diff is clean but git suddenly
reports the whole file changed" or as a visible `\ufeff` glyph in the
diff preview.

# Problem

Three independent failure modes were tangled together. (1) `read` would
return the BOM as the first character of the file string, so an LLM
edit that quoted "the first line" would actually be quoting the BOM
plus the first line; the resulting diff preview rendered the BOM as a
funky glyph (the issue screenshot in the PR description shows this).
(2) `write` and `edit` wrote whatever string the model produced, so a
file that started with a BOM lost it the first time the model
rewrote line 1. (3) The formatter step ran *after* the write and
could either add or strip a BOM independently of the original file's
state, with no feedback loop to reconcile.

# Design

The fix introduces `packages/opencode/src/util/bom.ts` (31 LOC) as a
single chokepoint for BOM handling and threads a `bom: boolean` flag
through every tool's internal state:

```ts
export function split(text: string) {
  if (text.charCodeAt(0) !== BOM_CODE) return { bom: false, text }
  return { bom: true, text: text.slice(1) }
}

export function join(text: string, bom: boolean) {
  const stripped = split(text).text   // idempotent
  if (!bom) return stripped
  return BOM + stripped
}
```

Reads go through `Bom.readFile`, which decodes with
`new TextDecoder("utf-8", { ignoreBOM: true })` and then runs `split`,
returning `{ bom, text }`. The agent's view of the file content is
*always* the BOM-stripped string — diffs are clean, the LLM never sees
`\ufeff`. Writes go through `Bom.join(text, desiredBom)`, where
`desiredBom = source.bom || next.bom` (preserve if either the original
file *or* the new content carried a BOM).

The cleverest piece is `Bom.syncFile`, which runs *after* the
formatter:

```ts
const current = yield* readFile(fs, filePath)
if (current.bom === bom) return current.text
yield* fs.writeWithDirs(filePath, join(current.text, bom))
```

This closes the loop: whatever the formatter did to the BOM, the tool
re-asserts the desired state. Note that `format.file()` was changed
from `Effect<void>` to `Effect<boolean>` so callers know whether a
formatter actually ran — the BOM re-sync is skipped when no formatter
fired, avoiding a redundant read+write on every untouched file. That
type-change ripples through `format/index.ts` cleanly.

`apply_patch.ts` carries the same flag through the per-file change
record (`bom: boolean` in `fileChanges`), and `patch/index.ts`'s
`deriveNewContentsFromChunks` returns `{ unified_diff, content, bom }`
so the patch path is symmetric with the tool path.

# Risks

- **`split()` is O(1) but called per read.** Fine. No perf concern.
- **The formatter contract change.** Returning `boolean` instead of
  `void` is a breaking change to the `Format.Interface` shape. There is
  exactly one external caller (`Format.file` in the same file) so the
  blast radius is contained, but plugins or future callers must be
  audited. A small `formatted: boolean` named return type would have
  been more self-documenting than a bare `boolean`.
- **`source.bom || next.bom` favors adding BOMs.** If a file currently
  has no BOM but the LLM emits one (perhaps copy-pasted from a Windows
  source), the file gains one. That is probably the right default
  (preserve user intent on either side) but it can be surprising; a
  config knob or repo-level normalization would be a future
  improvement.
- **`apply_patch` "delete" path also tracks `bom`** (line 178) even
  though the file is being deleted. Harmless dead state, but worth
  noting for future cleanup.

# Suggestions

1. Promote `Bom.syncFile`'s return value to a named tuple
   `{ wrote: boolean, text: string }` — callers currently throw it
   away in `write.ts` but use it in `edit.ts`, which is asymmetric.
2. Add an integration-style test that runs the formatter (e.g.
   `prettier`) end-to-end on a BOM file. The current tests stub
   `getFormatter`, so the BOM-after-formatter path is only proven by
   inspection.
3. Document the BOM-preservation rule in the user-facing docs — users
   familiar with other agents may expect BOM stripping by default.

# Verdict

**Ship it.** The fix is well-scoped, the new utility is properly
isolated, and the test coverage covers the three regression classes
(`bom-overwrite`, `first-line-edit`, `apply-patch-preview`). The
formatter-loop fix is the genuinely subtle part and the code reads
correctly.

# What I learned

When a transform pipeline carries data of two semantically distinct
"levels" (here: file *body* and file *encoding sidecar*), conflating
them into one string type is a latent bug waiting for a Windows user.
The right shape is an explicit `{ payload, sidecar }` pair with a
single chokepoint that does `split` on read and `join` on write. The
formatter loop is the place this matters most, because a third party
(prettier, gofmt, etc.) gets to mutate the file *between* your write
and the next read — without a re-sync step the agent's notion of
"current bom state" goes stale instantly.
