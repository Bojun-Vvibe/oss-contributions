# Context

`opencode`'s test suite includes a `cross-spawn-spawner` test that
spawns a tiny Node child, asks it to write `"stdout"` to one stream
and `"stderr"` to another, then asserts that both streams round-trip
through the spawner harness intact. On Linux and macOS this test was
green for months. On Windows CI it was flaky enough that it was the
single largest contributor to retries on the Windows runner.

# Problem

The original test code was:

```ts
const [stdout, stderr] = yield* Effect.all([
  decodeByteStream(handle.stdout),
  decodeByteStream(handle.stderr),
])
```

`Effect.all` *defaults to sequential execution.* So this means: read
the entire `stdout` stream to EOF first, *then* attach a reader to
`stderr`. On Linux and macOS the kernel buffers a few KB of pipe
output before the child blocks; the `"stderr\n"` write fits inside
the pipe buffer regardless of whether anyone is reading, so the
child exits cleanly and the late `stderr` reader picks up the
buffered byte after the fact.

On Windows the pipe semantics are different. `node` attaches the
child's stderr to a Windows pipe handle whose lifecycle is tied to
the child process. By the time the test attaches a reader to
`handle.stderr` (after the stdout pump completed and the child
exited), the underlying handle may already have been closed, the
buffered data may have been discarded, or the `decodeByteStream`
helper may emit empty before the kernel surfaces the
already-written byte. The result was an intermittent
`expect(stderr).toBe("stderr")` failure with `stderr === ""`.

This is the classic shape of a flaky Windows CI test: the bug is
*latent* on POSIX because the kernel hides the ordering, and
*manifest* on Windows because the kernel exposes it.

# Design

The fix is one option flag:

```ts
const [stdout, stderr] = yield* Effect.all(
  [decodeByteStream(handle.stdout), decodeByteStream(handle.stderr)],
  { concurrency: 2 },
)
```

`concurrency: 2` tells Effect to fork both decoders simultaneously,
so both pipe readers attach before the child has had a chance to
exit. Now the kernel ordering doesn't matter — both readers are
already waiting when the child writes its bytes.

The PR description quantifies the validation: `--rerun-each=200`
locally with the original sequential pattern fails immediately;
the concurrent version passes. That's a strong signal.

# Risks

- **Hides the actual underlying race.** The fix makes the test
  reliable but the *spawner* itself may still mis-handle a slow
  consumer of `handle.stderr` in production. If real users do
  `stream.on("data")` on `handle.stdout` first and only attach to
  `stderr` after the child exits, they may still hit the original
  Windows bug. The right follow-up is a separate test that
  intentionally lags the `stderr` reader and asserts the spawner
  buffers correctly — which would verify the *production* path,
  not just the test scaffolding.
- **`concurrency: 2` is a magic number.** If a future variant of
  the test adds a third stream, the concurrency must be raised in
  lockstep. `concurrency: "unbounded"` (or just `Effect.allPar`,
  if the codebase uses it) would be self-adjusting and arguably
  more correct here, since "read these N streams simultaneously"
  is the actual intent.
- **No comment explaining *why* concurrency matters.** The PR
  description is excellent, but six months from now a reader of
  the test sees an unexplained `{ concurrency: 2 }` and may
  "simplify" it back. A one-line `// MUST be concurrent: stderr
  pipe handle on Windows can close before sequential reader
  attaches` would lock in the intent.

# Suggestions

1. Replace the magic `2` with `"unbounded"` and add the comment
   above. Costs nothing, prevents the regression class.
2. Add a *separate* test in the spawner package itself (not just
   the integration test) that asserts the spawner buffers
   stderr correctly when its consumer attaches late. This is the
   real bug; the test fix is just a workaround.
3. If `Effect.all` parallelism defaults to sequential is a
   project-wide footgun (it is, in my experience with Effect
   codebases), add an ESLint rule or a wrapper helper
   `Effect.allPar` and prefer it in tests by default.

# Verdict

**Ship it.** A 4-line fix that converts a flaky Windows CI failure
into a reliable test, with proper validation (`--rerun-each=200`).
The reasoning is sound; my suggestions above are about hardening
the surrounding code, not about this PR.

# What I learned

`Effect.all` defaulting to sequential is a real footgun for
anyone whose mental model comes from `Promise.all` (which is
implicitly concurrent). The same shape exists in many "run
multiple effects" libraries: Scala ZIO's `ZIO.collectAll` is
sequential, `ZIO.collectAllPar` is concurrent; Haskell's
`mapM` vs `mapConcurrently`; Rust `futures::future::join_all`
*is* concurrent, but `try_join_all` semantics differ. Whenever
you bridge from a `Promise.all` mental model into one of these
libraries, the bug class is *predictable*: stream-reader tests
that pass on POSIX and flake on Windows. The general rule:
when reading from N independent IO sources where any one of
them might be small/closed/fast, always use the explicitly
concurrent combinator, even if "obviously" reading them in
sequence would also work. Sequential consumption of N pipes is
a Windows-flake generator.
