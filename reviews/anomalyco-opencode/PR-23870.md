# Context

`opencode`'s session compaction kicks in when a session's token usage
crosses a budget; an LLM is asked to summarize the older history into
a single anchored summary message, which then replaces those older
turns in the *sent* history. The persistent record is untouched (this
is the send-vs-store split working correctly), but the agent's view
of the conversation is reduced to `summary + recent tail`. The
existing implementation worked for the first compaction but degraded
sharply on second and subsequent compactions, and would catastrophically
fail (drop the entire tail) when even one recent turn alone exceeded
the budget.

# Problem

Three independent bugs were tangled together:

**(1) Re-compaction drift.** Each compaction call always selected
"the last assistant summary" and re-summarized the messages *after*
it, but it included earlier compaction summaries in the conversation
history fed to the LLM. So the second compaction summarized
`(summary_1, turns_a..b)`, the third summarized `(summary_2,
turns_b..c)` *but also re-included* `summary_1` and the user message
that requested it. The summary kept lengthening with each cycle and
the prompt structure drifted because the LLM wasn't being told it
was *updating* an anchored summary — it was being asked to start
fresh.

**(2) Tail collapse.** The tail-selection loop walked recent turns
right-to-left and bailed the moment a single turn exceeded `budget`,
returning `{ head: input.messages, tail_start_id: undefined }` (i.e.
"no recent context survives, summarize everything"). One oversized
tool output anywhere in a recent turn forced full summarization.

**(3) Unbounded tool outputs in summary prompt.** The history fed to
the summarizer LLM passed through `MessageV2.toModelMessagesEffect`
with no per-tool-output cap. A single 200KB grep result could blow
the summarizer's *own* context.

# Design

**Re-compaction drift fix.** A new `completedCompactions()` walks the
history and returns a list of `{userIndex, assistantIndex, summary}`
records — one per past compaction. The selection step then *hides*
those indices when feeding history to the summarizer:

```ts
const prior = completedCompactions(history)
const hidden = new Set(prior.flatMap(item => [item.userIndex, item.assistantIndex]))
const previousSummary = prior.at(-1)?.summary
const selected = yield* select({
  messages: history.filter((_, index) => !hidden.has(index)),
  ...
})
```

The most recent prior summary is then injected into the prompt as the
*anchor*:

```ts
const anchor = input.previousSummary
  ? [
      "Update the anchored summary below using the conversation history above.",
      "<previous-summary>", input.previousSummary, "</previous-summary>",
    ].join("\n")
  : "Create a new anchored summary from the conversation history above."
```

The new `compaction.txt` system prompt is rewritten to match: the LLM
is told it is an "anchored context summarization assistant" and
explicitly instructed to *update* the previous summary, not start
over. The output template is now a fixed Markdown skeleton with named
sections (`## Goal`, `### Done/In Progress/Blocked`, `## Next Steps`,
etc.) so re-summarizations preserve structure across cycles instead
of drifting into prose.

**Tail collapse fix.** A new `splitTurn()` helper kicks in when the
oldest still-fitting turn does not fit into the remaining budget. It
walks *forward* inside that one turn looking for a sub-range whose
estimated token count fits:

```ts
for (let start = input.turn.start + 1; start < input.turn.end; start++) {
  const size = yield* input.estimate({ messages: input.messages.slice(start, input.turn.end), model })
  if (size > input.budget) continue
  return { start, id: input.messages[start]!.info.id }
}
```

So instead of "summarize everything", the loop falls back to
"summarize everything *plus* the early portion of the oldest tail
turn", preserving as much recent context as the budget allows.

**Tool output cap.** A new `TOOL_OUTPUT_MAX_CHARS = 2_000` is passed
to `MessageV2.toModelMessagesEffect` as `toolOutputMaxChars`, capping
each tool output before it goes to the summarizer.

# Risks

- **`completedCompactions()` requires a parent-pointer hop**
  (`users.get(msg.info.parentID)`). If a parent message is missing
  from history (pruning, corrupt store) the compaction silently
  skips. A `log.warn` on this branch would help diagnose mystery
  re-summarization regressions.
- **`splitTurn()` cuts mid-turn**, which can produce a tail that
  starts at, e.g., a tool result with no preceding tool call. The
  message-v2 reorder pass should clean this up downstream, but a
  test exercising "tail starts at orphan tool_result" would harden
  this — the test file additions cover split tails generally but I
  did not see a test specifically for orphan-tool-result.
- **`previousSummary = prior.at(-1)?.summary`** picks only the most
  recent prior summary. If the user manually tweaked an older summary
  and the most recent compaction came out badly, the bad summary
  becomes the anchor. Probably the right default but worth a knob.
- **Hardcoded `TOOL_OUTPUT_MAX_CHARS = 2_000`.** No config knob. For
  agents that lean heavily on long tool outputs (read large file +
  reason about it) 2KB may be aggressive. A `compaction.tool_output_max_chars`
  config field would cost nothing.

# Suggestions

1. Make the prompt template a separate `.txt` file alongside
   `compaction.txt` so it is editable without a rebuild.
2. Add a `log.info` line every time `splitTurn` succeeds — this is
   exactly the diagnostic you want when debugging "why did my agent
   forget the last 3 turns".
3. Consider memoizing `estimate(...)` keyed by message ID — `splitTurn`
   re-estimates overlapping suffixes O(n) times.

# Verdict

**Ship it.** The three fixes are independently sound and the test
coverage (compaction.test.ts +259 lines, message-v2.test.ts +70,
messages-pagination.test.ts +64) is appropriate for the regression
classes. The anchored-summary pattern is exactly the right shape for
multi-cycle compaction.

# What I learned

The pattern "feed previous summary in as an anchor and ask the LLM to
*update* it" is the right primitive for any cumulative summarization
loop. The naive alternative — re-summarize from scratch every time —
loses information monotonically because the summarizer has no
incentive to preserve details that the previous summary deemed
important. Combine that with a fixed structural template (named
sections, "(none)" placeholders for empty ones) and you get
diff-friendly summaries: a reviewer can compare summary N vs N+1
section-by-section and see exactly what changed. The split-turn
fallback is the other reusable insight: when a budget-fitting loop
can't fit a unit, the right move is usually "fit the part that does
fit" rather than "give up on this whole budget round".
