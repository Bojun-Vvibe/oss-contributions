# Context

`opencode` sessions are persisted as an ordered list of "turns":
each turn is either a user message, an assistant message (possibly
with tool calls), or a tool result. The compaction path summarizes
the older prefix of a session when the running token estimate
crosses a budget, replacing a slice of turns with a single
synthesized "summary" assistant turn so the model keeps seeing a
bounded prompt.

Compaction has two pre-conditions worth naming explicitly: the
slice being summarized must be (a) a *valid* conversation on its
own — no dangling `tool_use` blocks, no assistant turns that
reference tool results outside the slice — and (b) *non-empty in
the informational sense*, i.e. it contains at least one message
the summarizer can meaningfully condense. Violating either
produces Anthropic 400s ("`tool_use` without matching
`tool_result`") or a worthless summary ("the assistant said
nothing") that still burns output tokens.

# Problem

This PR fixes (b). Under streaming interruptions (network blip,
user Ctrl-C, provider 529), the session writer occasionally
commits an assistant turn whose `content` array has length zero —
no text, no tool_use, nothing. That empty turn is benign for
display (the UI just renders nothing), but when it sits inside a
compaction slice it causes two failure modes:

1. The summarizer receives a transcript where alternating
   `user → assistant → user → assistant` is broken by an empty
   assistant turn. Depending on provider, this either 400s on the
   submit (Anthropic rejects empty `content`) or produces a
   summary that hallucinates content for the empty turn to make
   the alternation "work".
2. The compaction heuristic counts turns, not informational
   content, so empty turns inflate the "we have enough to
   compact" signal. Users reported compaction firing on
   sessions that, once empties were stripped, had only two
   substantive exchanges — far below the intended threshold.

The reported symptom in the linked issue is the Anthropic 400. The
quality regression from #2 is the more insidious one.

# Design

The patch is small and correct:

```ts
// compact.ts
-  const slice = session.turns.slice(0, cutoff)
+  const slice = session.turns
+    .slice(0, cutoff)
+    .filter((t) => !isEmptyTurn(t))
+
+  if (slice.length < MIN_COMPACT_TURNS) {
+    return { compacted: false, reason: "slice_too_thin_after_filter" }
+  }
```

Where `isEmptyTurn` is:

```ts
export function isEmptyTurn(t: Turn): boolean {
  if (t.role !== "assistant") return false
  const content = t.content ?? []
  if (content.length === 0) return true
  return content.every(
    (b) =>
      (b.type === "text" && (b.text ?? "").trim() === "") ||
      (b.type === "thinking" && (b.thinking ?? "").trim() === "")
  )
}
```

Three observations:

1. **Only assistant turns are filtered.** Correct — an empty
   *user* turn is never committed by the write path (the input
   layer rejects it upstream), and an empty *tool_result* turn
   would indicate an actual bug elsewhere that the compactor
   should not paper over. Scoping the filter to assistant turns
   is the right restraint.
2. **`thinking` blocks are counted as empty when blank.** This
   matters: a thinking-only turn that the provider truncated
   mid-stream (producing an empty `thinking` text) is exactly
   the case #1 is trying to rescue. Good.
3. **`MIN_COMPACT_TURNS` is new** (set to 4). The pre-filter
   check didn't need a floor because turns-since-compact was
   already checked against a threshold; the post-filter check
   does, otherwise filtering 10 empties out of 11 turns would
   try to summarize a 1-turn slice.

The test in `session.compact.test.ts` seeds a transcript with
alternating empty assistant turns and asserts both the filter
behaviour and the "too thin" early return. Good coverage of the
happy path.

# Risks

- **Orphaned tool_use after filter.** If the empty assistant
  turn sat between a `tool_use` on turn N-1 and a `tool_result`
  on turn N+1, filtering it out doesn't break anything, because
  the tool_use/tool_result pair is preserved across sibling
  turns. But the filter could in principle remove an assistant
  turn that held a *real* tool_use block but no visible text —
  the `isEmptyTurn` predicate already guards against this
  (only fully-empty or blank-text-only turns qualify), but a
  comment in the predicate saying "do not extend this to
  tool_use-only turns" would future-proof the intent.
- **Re-introduction via streaming.** The underlying bug — the
  writer committing empty turns — is not fixed here, only
  papered over at compaction time. Display, export, and the
  non-compaction Anthropic submit paths still see the empties.
  If any other downstream consumer reads raw turns (e.g. the
  `/history` export command, the `.md` dump writer), the 400 or
  the bad summary can resurface there.
- **Monotonicity of the "too thin" return.** `compacted:
  false, reason: "slice_too_thin_after_filter"` means the
  caller does *not* advance `lastCompactedIndex`. So the next
  turn added will re-evaluate, and if it pushes the filtered
  count to 4, compaction will fire. Correct behaviour, but
  worth confirming with a test that seeds the session near the
  boundary and adds one substantive turn — currently the test
  file only covers the "too thin → stay false" branch.

# Suggestions

1. Fix the writer too. A short PR that rejects a zero-length
   `content` at `Session.writeAssistantTurn()` (with a
   diagnostic log, not an exception — we're on an
   interruption-recovery path) would stop generating these
   turns in the first place.
2. Add a `toolUseOnly(t)` sibling predicate and assert at the
   call site that `!isEmptyTurn(t) || !toolUseOnly(t)` — the
   inversion makes the intent explicit and the test harness
   can lock it.
3. Log a `compact.filter_dropped_empty_turns` metric (count,
   not identity) so the rate of this getting hit is
   observable. If the writer fix lands, that counter should
   trend to zero; if it climbs, something upstream regressed.

# Verdict

**Ship it, then chase the writer.** This is the right compaction-
layer fix: cheap, local, provably safe, buys time. The real
defect — committing empty assistant turns — should be a
follow-up, because "fix it at every consumer" is an unbounded
game.

# What I learned

The general pattern here is *defense-in-depth around a write
path that can't yet be made transactional*. Streaming
interruption is the enemy of "a turn is either fully there or
not there at all": you end up with partial turns on disk and no
easy way to retroactively prove validity. The options are:

- Make the write path transactional (a staging file, an fsync,
  an atomic rename). Expensive per turn.
- Make every read path tolerant to partial turns. This PR is an
  instance of that strategy.
- Periodically GC empties in a background pass. Adds a
  coordination problem (what if compaction races with GC?).

The pragmatic answer is usually "do (b) until it becomes a mess,
then do (a)". This PR picks (b) cleanly. The moment a fourth
consumer needs the same filter, that is the signal to escalate
to (a) — because at that point the tolerance logic has drifted
four places and will diverge.
