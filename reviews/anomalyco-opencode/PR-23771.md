# Context

LSP defines two ways for a server to deliver diagnostics:

1. **Push** (`textDocument/publishDiagnostics`): server sends a
   notification whenever it has new diagnostics. The client just
   listens. Works for TypeScript, ESLint, gopls.
2. **Pull** (`textDocument/diagnostic` and `workspace/diagnostic`):
   server registers a `diagnosticProvider` capability; the client
   *asks* for diagnostics whenever it wants them. Used by Roslyn
   (C#), Kotlin LSP, modern rust-analyzer.

`opencode`'s previous LSP client only spoke push. The symptom
described in the PR is striking: open a C# file, edit it, wait for
errors to surface — they never do. The client `touchFile`'d the
buffer, slept 1s waiting for a push that was never coming, and
returned an empty diagnostics list. Users saw "opencode just doesn't
know about C# errors" and had to alt-tab to their IDE to verify.

# Problem

The PR description spells out three iterations:

- **Pre-PR (push-only).** No errors surface at all for pull-only
  servers.
- **First attempt (pull + smart waiting).** Worked, but added 3–4s
  of latency on warm files and could time out at 10s on cold ones,
  because the smart waiting policy was Roslyn-specific and re-ran
  per-server settle loops.
- **This PR (generic pull-aware client).** No server-specific
  branches, no settle timers. After every `didOpen`/`didChange`,
  fire current-document pulls in parallel for every registered
  diagnostic identifier and return on the first match. Warm C#
  edit returns 3 `CS0103` errors in ~0.7s.

# Design

The client now maintains *two* diagnostic stores and merges them
on every read:

```ts
const pushDiagnostics = new Map<string, Diagnostic[]>()
const pullDiagnostics = new Map<string, Diagnostic[]>()
const mergedDiagnostics = (filePath: string) =>
  dedupeDiagnostics([...(pushDiagnostics.get(filePath) ?? []),
                     ...(pullDiagnostics.get(filePath) ?? [])])
```

The pull path is wired through new handlers for
`client/registerCapability` / `client/unregisterCapability`. When a
server registers a `textDocument/diagnostic` capability, its
`registerOptions.identifier` and `workspaceDiagnostics` flag get
remembered. Two helpers then issue the actual requests:

- `requestDiagnosticReport(filePath, identifier)` →
  `textDocument/diagnostic` (current document)
- `requestWorkspaceDiagnosticReport(filePath, identifier)` →
  `workspace/diagnostic` (broader sweep that the server may use to
  return cross-file diagnostics)

Both are wrapped in a 3s `DIAGNOSTICS_REQUEST_TIMEOUT_MS`. The
`mergeResults` helper distinguishes three states per file:

- **handled** — at least one pull provider responded
- **matched** — the response actually contained diagnostics for
  the requested file
- the byFile map records every related document the pull surfaced

This three-state design is the key insight. It lets the
`waitForFreshPull` loop fire requests in parallel against every
registered identifier and return the moment *any* of them matches,
without waiting for the slow ones. The previous "smart waiting"
attempt apparently waited for *all* registered providers to settle,
which is what produced the 3–4s tax.

A second important detail: the publishDiagnostics handler still
seeds the push cache silently on the very first publish for
TypeScript:

```ts
if (shouldSeedDiagnosticsOnFirstPush(input.serverID) && !pushDiagnostics.has(filePath)) {
  pushDiagnostics.set(filePath, params.diagnostics)
  return
}
```

TypeScript pushes aggressively on first open with stale
diagnostics; the second push is the real one. Suppressing the
event on the first push avoids a UI flicker. This is the kind of
empirical patch that no spec-only implementation would have.

# Risks

- **`shouldSeedDiagnosticsOnFirstPush` is hard-coded by serverID.**
  It is documented and defensible, but the comment lives in code
  and not in any test. A test fixture LSP that intentionally
  publishes a stale → fresh sequence would lock in this behaviour.
  Otherwise a future "let's clean this up" sweep silently breaks
  TypeScript again.
- **`endPosition()` splits on `/\r\n|\r|\n/`** but does not handle
  `\u2028` / `\u2029` line terminators. JavaScript source files in
  the wild occasionally contain these (they are valid in strings
  and historically valid as line terminators in JS source). For
  text-document sync version computations this is unlikely to
  matter; if it ever does, the diagnostic offsets will silently
  drift by a line. Worth a comment if not a fix.
- **Three timeouts (`DOCUMENT_WAIT`, `FULL_WAIT`, `REQUEST`) are
  separate constants but interact.** A reader has to hold all
  three in their head to predict latency. A small table comment
  ("warm: bounded by REQUEST=3s; cold open: bounded by
  DOCUMENT_WAIT=5s; init: bounded by FULL_WAIT=10s") would help.
- **`dedupeDiagnostics` keys on JSON.stringify of five fields.**
  If two providers return the same diagnostic with different
  `data` blobs (LSP code-action attachment), only one survives —
  whichever wrote first. Code actions on the dropped one are lost.
  Probably fine for `opencode`'s read-only diagnostics view, but
  should be revisited if/when the agent starts honoring code
  actions.
- **`workspace/diagnostic/refresh` is acknowledged with `null`**
  but the client does not actually re-pull. A server that asks
  for refresh expects the client to invalidate its cache; right
  now the cache stays warm. Probably benign for the agent
  use-case (next `touchFile` re-pulls anyway) but worth a comment.

# Suggestions

1. Add a test using a fake LSP fixture that registers pull
   diagnostics, returns a related-document report, and asserts
   `mergedDiagnostics` covers both files.
2. Replace the boolean `touchFile(file, true)` argument with the
   new "full" / "incremental" string enum *everywhere*, then make
   the boolean form a deprecation shim. The PR already does this
   for `debug/lsp.ts`; check the rest of the call sites.
3. Add a metric counter for `pull_diagnostic_request_total{server,
   matched}` so users can later diagnose "why is C# slow on this
   project" without re-instrumenting.

# Verdict

**Ship it.** This is the correct re-design. The merged push+pull
store with three-state result reporting is the right primitive —
it lets future server-quirk patches be local rather than rippling
into the wait loop. The empirical TypeScript first-push seeding is
the kind of detail that distinguishes a spec implementation from a
*production* implementation.

# What I learned

The pattern "run N independent providers in parallel and return on
the first match" is the right shape for any client that talks to a
heterogeneous fleet of backends with unknown latency. The wrong
shape — "wait for all providers to settle" — is what produced the
3–4s tax in the previous attempt. The fix is essentially the
*token bucket race* pattern: fire requests in parallel, accept the
first usable answer, let the slower ones complete and contribute
their diagnostics asynchronously without blocking the read.

The deeper insight: when a protocol gives you push *and* pull, you
can't pick one. You have to support both, merge their outputs, and
de-duplicate. The previous opencode client picked push because
TypeScript happened to be the dominant case; that is exactly the
"my dominant case is the spec" mistake LSP clients keep making.
The right architecture treats push and pull as *two derived views
of the same diagnostic state*, with the merged store as the
source of truth.
