# BerriAI/litellm#26935 — refactor: scope /health response to caller's models and tidy display fields

- **PR**: https://github.com/BerriAI/litellm/pull/26935
- **Head SHA**: `ee36a4a3a7a14129bf8051ae2b425dae194040c9`
- **Files**: `litellm/proxy/health_check.py` (+1/-0), `litellm/proxy/health_endpoints/_health_endpoints.py` (+118/-6), `tests/test_litellm/proxy/health_endpoints/test_health_endpoints.py` (+272/-0)
- **Verdict**: **merge-after-nits**

## Context

Closes a real information-disclosure shape bug in the `/health` endpoint plus a small auth-data hygiene fix. The pre-fix `health_endpoint()` had explicit `### FILTER MODELS FOR ONLY THOSE USER HAS ACCESS TO ###` comment with `pass` / `pass` placeholders at `_health_endpoints.py:862-864` — a literal TODO that allowed *any* authenticated key to see *all* models in `/health` regardless of `user_api_key_dict.models` scoping. Plus `api_base` and `api_version` (provider routing fields, not part of the health response shape) leak the underlying provider URL, which lets non-admin keys learn which Azure region / OpenAI org / internal host the deployment talks to. Three new tests pin the response shape.

## What's right

- **Filter is implemented at the right layer.** `_health_endpoints.py:948-952` adds `allowed_models = set(user_api_key_dict.models); _llm_model_list = [m for m in _llm_model_list if m.get("model_name") in allowed_models]` only when `len(user_api_key_dict.models) > 0` (i.e., when the key has explicit model scoping). Keys without explicit scoping (e.g., admin keys) keep the prior unscoped behavior. Correct discipline — the filter is opt-in by key configuration, not opt-out.
- **Cached background-health-check path also filtered via `_filter_health_check_results_by_model_ids` at `:778-803`.** This is the load-bearing fix — previously, when `use_background_health_checks` was enabled, the cached `health_check_results` dict was returned verbatim (`return health_check_results` at the prior `:866`), bypassing the model-scoping entirely. New helper filters by `model_id` (since cached entries don't carry `model_name`), with an explicit conservative-drop policy at `:797-799`: "Endpoints without a model_id (e.g. CLI-model entries that predate the model_id wiring) are dropped conservatively — we cannot prove they belong to the caller, so they are excluded rather than leaked." This is the right fail-closed default at the trust boundary.
- **`_strip_api_base_from_health_result` at `:752-771` correctly preserves shape.** Returns a copy with `api_base` removed from each `healthy_endpoints[i]` and `unhealthy_endpoints[i]` dict. Doesn't mutate the original (callers may share the dict). Non-dict entries pass through unchanged.
- **Admin role check is robust.** `_is_proxy_admin` at `:733-749` handles both enum-typed `LitellmUserRoles.PROXY_ADMIN` and raw-string `"proxy_admin"` (since `user_role` on `UserAPIKeyAuth` can arrive in either shape depending on the auth path), via `role_value = role.value if hasattr(role, "value") else role`. The `_PROXY_ADMIN_ROLES = frozenset({...})` at `:725-730` covers both `PROXY_ADMIN` and `PROXY_ADMIN_VIEW_ONLY` — view-only admins still see `api_base` (they need it to debug), which is the right design.
- **`_post_process` at `:920-928` is uniformly applied** at all three return paths: CLI-model branch at `:946`, cached-background-checks branch at `:980`/`:982`, and live-check branch at `:996`. No path can accidentally bypass the api_base stripping.
- **Diagnostic comment at `:962-973`** explains the `len(user_api_key_dict.models) > 0` + empty `allowed_model_ids` corner case ("scoped key has accessible model_names but none of the matching deployments expose a model_info.id; background health-check cache will return an empty result"). Logs at debug level so operators can spot deployments missing `model_info.id`. Right discipline — empty-result is correct fail-closed behavior, but operators need the signal to know why.
- **Display denylist tightening.** `health_check.py:31` adds `"api_version"` to `ILLEGAL_DISPLAY_PARAMS`. Matches the existing pattern (`api_key`, `aws_access_key_id`, `aws_secret_access_key`, `exception`, `litellm_metadata`). Small, isolated, low-risk.
- **Test coverage covers all three concerns.** PR body cites `test_health_endpoint_filters_model_list_by_user_access`, `test_health_endpoint_filters_background_cache_by_user_access`, `test_clean_endpoint_data_drops_routing_fields`. Author ran `uv run pytest ... -v # 30 passed`.

## Risks / nits

- **PR body says "🧹 Refactoring / ✅ Test" but this is a security fix.** The pre-fix `pass / pass` placeholder was an *information-disclosure bug* — any authenticated key could enumerate the full model list. Recommend recategorizing the PR as a security fix (or at minimum, a "Bug fix") so it gets the right reviewer attention and shows up in the security changelog. Filing as "refactor" undersells the impact.
- **`_strip_api_base_from_health_result` only strips `api_base`, but `api_version` stripping happens at a different layer** (`ILLEGAL_DISPLAY_PARAMS` in `health_check.py`). These two concerns should ideally be co-located — either strip both at the same layer (in `_strip_api_base_from_health_result`, rename to `_strip_routing_fields_from_health_result`), or document at each site that the other layer covers the complementary field. Current shape risks future drift where someone tightens one and forgets the other.
- **Conservative drop of model-id-less endpoints at `:797-799`** is correct security behavior but means *admin* keys with `len(models) > 0` (a possible config) could lose visibility into their CLI-model entries. Recommend the helper be skipped entirely for admin keys (they should see everything), with the filter only applied to non-admin scoped keys.
- **No test for the admin-bypass path** — i.e., that an admin key sees `api_base` in the response. The three new tests cover the non-admin scoping but not the explicit "admin sees everything" contract. Recommend a fourth test pinning that contract so a future "always strip api_base" change doesn't silently break admin debugging.
- **`_filter_health_check_results_by_model_ids` produces `healthy_count`/`unhealthy_count` at `:809-810`** — these are derived from the post-filter list, which is correct, but if the original cached `health_check_results` had additional top-level fields (e.g., `last_run_at`, `total_endpoints`), they're silently dropped. Recommend either preserving non-endpoints fields via `{**results, "healthy_endpoints": ...}` or documenting the strip-unknown-fields choice.

## Verdict

**merge-after-nits.** Closes a real information-disclosure bug (the literal `pass / pass` TODO at `:862-864` was unscoped model enumeration for any authenticated key) with the right fail-closed defaults and uniform `_post_process` application across all three return paths. The "scope to caller's models" + "strip routing fields" pair are the right contract for a `/health` response. Strongest nits: relabel as security/bugfix not refactor, co-locate `api_base` and `api_version` stripping at one layer, add an admin-bypass test, and either preserve unknown top-level cached-result fields or document the strip.
