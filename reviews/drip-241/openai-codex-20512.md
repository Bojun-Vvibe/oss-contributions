## Context

Architectural cleanup: image-view tool results were previously reconstructed by `app-server` from a legacy `ViewImageToolCall` event in `bespoke_event_handling.rs:984-1005` (a 22-line block manually constructing `ItemStartedNotification` + `ItemCompletedNotification` from the legacy event). This PR moves the responsibility to *core* (where it belongs — core is the producer of structural items) by introducing a `TurnItem::ImageView` variant, emitting it directly from the `view_image` handler, and keeping the legacy `ViewImageToolCall` event as a *compatibility output* generated from the new item lifecycle. The same architectural pattern already applied to other tool families (web search, image generation) extended to one more case.

## What's right

- **Producer-side ownership inversion is the load-bearing fix.** Pre-PR, `app-server/src/bespoke_event_handling.rs:984-1005` held image-view-specific protocol-construction logic (manually building `ThreadItem::ImageView` and emitting `ItemStarted`/`ItemCompleted`), violating the principle that app-server should be a thin transport layer over core's structural items. Post-PR `:984-986` collapses to `EventMsg::ViewImageToolCall(_) => {}` — a single no-op arm because the item lifecycle is now produced upstream in core.
- **Core-side emission at `view_image.rs:152-156` is uniform with other handlers.** `let item = TurnItem::ImageView(ImageViewItem { id: call_id, path: event_path }); session.emit_turn_item_started(turn.as_ref(), &item).await; session.emit_turn_item_completed(turn.as_ref(), item).await;` — uses the existing `emit_turn_item_started`/`emit_turn_item_completed` plumbing that web-search and image-generation handlers already use. No new infrastructure invented; the new item type slots into the existing pattern.
- **Legacy event preserved via `as_legacy_events()` at `items.rs:414-419`.** `TurnItem::ImageView(item) => vec![EventMsg::ViewImageToolCall(ViewImageToolCallEvent { call_id: item.id.clone(), path: item.path.clone() })]` — the legacy event is now *derived* from the structural item rather than emitted independently. This keeps `thread_history.rs` and other legacy-event consumers (`mcp-server/src/codex_tool_runner.rs:368-376`, `core/src/session/turn.rs:1465-1471`) working without changes to their handling logic.
- **Empty-handle arms are explicit, not silent.** `protocol.rs:1860` `TurnItem::ImageView(_) => Vec::new(),` in `ItemStartedEvent::as_legacy_events()` — image view emits no separate "begin" legacy event because it's atomic (start and complete arrive together). The `Vec::new()` arm is correct (not an oversight) and worth the explicit empty-arm rather than `_ => unreachable!()` since the discriminator must be exhaustive.
- **`thread_history.rs:357-385` correctly opts out for the replay path.** PR body: "Kept `thread_history.rs` on the legacy `ViewImageToolCall` replay path, with `ImageView` item lifecycle events ignored there." Diff at `:360,381` adds `| codex_protocol::items::TurnItem::ImageView(_)` to the no-op arm of `ThreadHistoryBuilder` — which is intentional because `thread_history` reconstructs from the legacy event side, not the new item side, to maintain backward compatibility with existing rollout files.
- **`v2.rs` `From<CoreTurnItem> for ThreadItem` impl at `:6420-6424`** correctly maps `CoreTurnItem::ImageView(image) => ThreadItem::ImageView { id, path }` with explicit field mapping (no shape mismatch).
- **Test coverage at `view_image.rs:299-348` exercises both new and legacy paths.** New assertions: `item_started.expect("view image item started event emitted")` matched against `TurnItem::ImageView(item) => assert_eq!(item.id, call_id); assert_eq!(item.path, abs_path)`, plus the same for `item_completed`, plus the legacy `legacy_event.expect("legacy view image event emitted")` with `assert_eq!(legacy_event.call_id, call_id); assert_eq!(legacy_event.path, abs_path)`. Three assertions pinning the contract: new item start, new item complete, legacy event emission. Exactly the right coverage for a producer/legacy-fan-out change.
- **`v2.rs` test at `:10321-10331`** also pins the `From` conversion contract directly via `ThreadItem::from(image_view_item)` round-trip assertion.
- **Two trivial enum-arm reorderings** (`view_image.rs:1-3` import reorganization, `session/turn.rs:1465-1471` and `codex_tool_runner.rs:368-376` reordering `ViewImageToolCall` to be alphabetized after `ImageGenerationEnd`) are non-functional cleanup riding along.
- **Verification commands cited in PR body.** `cargo test -p codex-protocol -p codex-app-server-protocol -p codex-rollout -p codex-rollout-trace -p codex-mcp-server -p codex-app-server --lib` plus the focused `cargo test -p codex-core --test all view_image_tool_attaches_local_image` plus `just fix` for formatting. Reviewer can re-run these.

## Risks / nits

- **Test only asserts the *legacy* event still fires** but doesn't assert the *order* of `item_started` → `item_completed` → `legacy_event`. If a future producer-side refactor accidentally interleaves them or fires them in reverse order, this test would still pass. Not blocking, but a minor strengthening opportunity.
- **No assertion that the `ItemStarted` and `ItemCompleted` carry the *same* `ImageView` payload** (they could in principle drift if a future refactor mutates the item between calls). Quick fix: assert `item_started.id == item_completed.id && item_started.path == item_completed.path`.
- **`bespoke_event_handling.rs:984` collapsing to `EventMsg::ViewImageToolCall(_) => {}`** is correct now but a future maintainer reading just this file may wonder why the arm exists at all. A one-line comment `// item lifecycle is now produced by core; legacy event is a compatibility output and needs no app-server handling` would be helpful.

## Verdict

**merge-as-is.** Clean architectural cleanup moving image-view item production from app-server's bespoke handling into core, parallel to the existing pattern for web search and image generation. The legacy event is preserved as a derived compatibility output via `as_legacy_events()`, so no upstream consumer breaks. Test updates correctly assert all three contracts (new item start, new item complete, legacy event). Optional polish: ordering assertions in the test, identity assertions between `item_started`/`item_completed`, and an explanatory comment on the now-empty `bespoke_event_handling.rs` arm.
