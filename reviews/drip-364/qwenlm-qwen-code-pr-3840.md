## What it does

Closes a class of silent-overwrite bugs where an Edit/WriteFile tool call is generated by the model based on a `read_file` snapshot that's now stale ‚Äî because another agent, a hook, the user's editor, or a parallel tool run mutated the file in the meantime. The fix is a new `FileReadCache.staleWriteReason(absPath, stats)` helper (`fileReadCache.ts:158-185`) plus a pre-flight `fs.statSync` + check in both `EditToolInvocation.execute` (`edit.ts:361-385`) and the parallel guard in WriteFile, returning the new `ToolErrorType.FILE_STALE_BEFORE_WRITE` (`tool-error.ts:26`) with a recovery-worded user-facing message instead of clobbering the on-disk content.

The fence semantics are deliberately narrow:
- **First-time edits / brand-new files / files-the-model-named-without-reading**: not blocked. `staleWriteReason` returns `null` for `unknown` cache state (`fileReadCache.ts:175-177`), so `old_string=""` create-from-empty flows and direct-name writes work unchanged. The `'proceeds normally when the model has never read the file'` test at `edit.test.ts:1003-1019` pins this.
- **Read ‚Üí external mutation ‚Üí Edit**: blocked. The integration test at `edit.test.ts:961-995` exercises the full flow: record a Read, mutate the file behind the cache's back (with `fs.utimesSync` to force mtime drift past the cache's millisecond resolution), attempt an Edit, assert `ToolErrorType.FILE_STALE_BEFORE_WRITE` and that the on-disk content is the *external* write, untouched.
- **Tool's own back-to-back writes**: not blocked. The `'does not block a follow-up write after the tool's own write'` test at `fileReadCache.test.ts:147-156` pins this ‚Äî the tool calls `recordWrite()` post-write so the cache is in sync with disk and a follow-up write to the same file from the same tool doesn't self-fence.

## Why this is the right shape

1. **Recovery-worded message, not a stack trace.** `fileReadCache.ts:178-184` returns `${absPath} has been modified since this session last touched it (${ageSeconds}s ago). Re-read the file before writing ‚Äî its contents are no longer what you saw.` That's exactly what the model needs to read to recover ‚Äî it tells it the file is stale, gives an age hint for context, and prescribes the next action (`re-read`). Compare to a cryptic `EBUSY` or `EPERM`, which models typically retry verbatim.
2. **Test coverage is thorough at both layers.** `staleWriteReason` itself has 5 unit tests at `fileReadCache.test.ts:110-156` (unknown‚Üínull, fresh‚Üínull, mtime-drift‚Üímessage, size-drift‚Üímessage, post-write‚Üínull). EditTool gets the integration test plus the negative case. WriteFileTool gets a parallel `fileReadCache.clear()` in `beforeEach` (`write-file.test.ts:88-95`) with a load-bearing comment explaining why: the module-scoped cache would otherwise accumulate stale entries at OS-reused inodes across tests.
3. **The `getFileReadCacheDisabled()` config flag escape hatch is plumbed correctly** (`edit.ts:364`, `write-file.test.ts:73`). Users who consciously want the old "blast through" behavior can set it; default is on. The check is a fast `if (!this.config.getFileReadCacheDisabled())` *before* the `statSync`, so disabled-cache users pay zero overhead.
4. **`fs.statSync` failure path is silent and falls through to existing error handling** (`edit.ts:380-383` `} catch { /* defer */ }`). That's right: if the file is missing, the existing `calculateEdit` path will produce its own well-tested error; the fence is *additive*, not load-bearing.

## Nits / concerns

- **`statSync` is sync I/O on the tool hot path.** Each Edit/WriteFile now does an extra synchronous `fs.statSync` before the actual edit. For local FS that's microseconds, but on slow filesystems (network mounts, fuse, throttled containers) it adds latency. Consider `fs.promises.stat` (the rest of the path is already `async`) ‚Äî purely a perf nit, no behavior change.
- **Age formatting clamps to seconds.** `Math.round((Date.now() - prior) / 1000)` (`fileReadCache.ts:182`) rounds 400ms down to `0s ago`, which reads as a lie ("0 seconds ago, but stale?"). For `< 1s` cases, prefer `<1s ago` literal or use `Math.max(1, ...)`.
- **The fence relies on `FileReadCache.check` returning `state === 'stale'` with an `entry`.** That contract is exercised by the existing `check` tests (referenced by line numbers in the PR diff context but not shown), but a one-line invariant comment in `staleWriteReason` saying "depends on `check()` returning `{state:'stale', entry}` only when there *was* a prior interaction" would protect against a future `check()` refactor silently breaking the fence.
- **No `replace` tool variant.** This PR fences Edit (the `replace`-by-`old_string`/`new_string` tool) and WriteFile. If qwen-code has a separate `replace_in_file` / `apply_patch` / `multi_edit` tool, that path needs the same fence ‚Äî otherwise users will route around the guard. Worth a follow-up grep for any other in-tree tool that ends in a `fs.writeFileSync` call without going through the cache check.
- **Concurrency: TOCTOU between the `statSync` and the actual write.** Another mutation can land in the microsecond window between `staleWriteReason()` returning `null` and the tool's own `writeFileSync`. That's not solvable without `O_EXCL`/`flock` and is fine for the model-error-recovery use case this PR is targeting; just call it out so future contributors don't expect atomicity.
