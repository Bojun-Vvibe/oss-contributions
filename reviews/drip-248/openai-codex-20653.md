# openai/codex #20653 — [codex] report running command durations

- Link: https://github.com/openai/codex/pull/20653
- Head SHA: `07c8fe4a8b34296b71b58578abbb3afe3daea15d`
- Author: nornagon-openai

## Summary
Promotes `commandExecution.durationMs` from a finalize-only field to an always-present elapsed-runtime value while a sandboxed command is running, then a final value on completion. Three coordinated edits: (1) emit `durationMs: 0` at command start, (2) record per-call `Instant` start times in app-server `ThreadState`, (3) refresh elapsed durations whenever a live `thread/resume` or `thread/read(includeTurns=true)` snapshot is materialized so a second client joining a live thread sees server-authored elapsed time instead of having to local-guess. Wire value is reported as elapsed milliseconds (clock-skew-safe), not as a server timestamp. Schema descriptions updated across 18 generated JSON files plus protocol docs.

## Line-level observations
- `codex-rs/codex-app-server/src/thread_state.rs` — new free helper `fn elapsed_duration_ms(duration: std::time::Duration) -> i64 { i64::try_from(duration.as_millis()).unwrap_or(i64::MAX) }` saturates at `i64::MAX` rather than panicking on a >292M-year process, the right shape for a wire-facing converter. New field `running_command_started_at: HashMap<String, Instant>` on `ThreadState` is keyed by `call_id` (matches the `ExecCommandBegin/End` event identifier) and stores `Instant` (monotonic) rather than `SystemTime` so it's immune to wall-clock jumps.
- `thread_state.rs` `active_turn_snapshot()` rewritten to take a mutable copy of the snapshot and call `refresh_running_command_durations(std::slice::from_mut(&mut turn))` before returning. This is the load-bearing fix: every code path that used to return a stale snapshot now goes through one refresh point. The `refresh_running_command_durations` body destructures only `ThreadItem::CommandExecution { id, status: CommandExecutionStatus::InProgress, duration_ms, .. }` and skips others — so completed commands keep their final value (correct), and any future status variant added to the enum gets the safe default of "no refresh."
- `track_current_turn_event` extends from a single `EventMsg::TurnStarted` arm to a `match` adding `EventMsg::ExecCommandBegin(payload) => running_command_started_at.entry(payload.call_id.clone()).or_insert_with(Instant::now)`. The `or_insert_with` is correct against duplicate `Begin` events (rare but observed on retries) — first start wins, so elapsed never goes backwards. The teardown side (removing on `ExecCommandEnd`) was not visible in the slice I read; if it isn't present the `HashMap` grows monotonically per-thread which would matter for very long-running threads. Worth confirming.
- New unit tests `refreshes_running_command_durations_for_live_turns` and `leaves_completed_command_duration_unchanged` at the bottom of the file pin both directions of the invariant. The first asserts `elapsed >= 1_500` after sleeping 1.5s — slightly fragile under heavy CI load (Instant elapsed is monotonic but the sleep is at-least, not exactly), but the lower-bound shape is the right assertion for a duration test.
- `codex-rs/app-server-protocol/schema/json/*` — 18 schema files all get the same description rewrite from `"The duration of the command execution in milliseconds."` to `"Elapsed command runtime in milliseconds. Final once the command completes."` plus matching prose updates in `docs/.../protocol.md` to "Running `commandExecution` items include elapsed `durationMs`; final items include parsed `commandActions`, `status`, `exitCode`, and final `durationMs`." This is the right place to encode the contract change — clients renderering live snapshots now have a typed signal rather than having to read the PR.

## Verdict
`merge-after-nits` — confirm there is an `ExecCommandEnd` arm that removes the entry from `running_command_started_at` so the map doesn't grow unbounded across long-lived threads, and consider relaxing the `assert!(elapsed >= 1_500)` test to `>= 1_400` or use a polling/await pattern to avoid CI flake under load. The architectural shape (monotonic `Instant`, single refresh chokepoint in `active_turn_snapshot`, saturating i64 conversion, schema-side contract update across all 18 emitters) is correct.
