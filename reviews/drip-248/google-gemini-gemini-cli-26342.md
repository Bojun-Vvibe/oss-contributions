# google-gemini/gemini-cli #26342 — fix(core): reset session-scoped state on resumption

- Link: https://github.com/google-gemini/gemini-cli/pull/26342
- Head SHA: `0b34951524f4145b83810450dab0fb84c4100d8e`
- Author: cocosheng-g

## Summary
Closes a "session state split" bug: at startup the CLI mints a fresh "startup" session ID (Session A), but when the user then resumes a previous session (Session B) via the Session Browser or `--resume`, several internal services and counters stayed bound to Session A. The visible symptoms cited in the PR body include Task Tracker and Plan Mode keeping Session-A scope, plus quota/credit notification flags carrying across sessions. Fix moves the reset block from the `resetNewSessionState()` wrapper into `setSessionId()` itself so every session-id transition (resume OR new) flushes 14 session-scoped fields, and adds a small `SkillManager.reset()` helper so the active-skill set clears too. Tiny diff: +22 / -1 across two files.

## Line-level observations
- `packages/core/src/config/config.ts:1809-1822` — inside `setSessionId()`, after the existing `this._sessionId = sessionId; this.storage.setSessionId(sessionId); this.trackerService = undefined;` lines, the PR appends 14 explicit clears: `approvedPlanPath = undefined`, `topicState.reset()`, `skillManager.reset()`, `latestApiRequest = undefined`, `lastModeSwitchTime = performance.now()`, `compressionTruncationCounter = 0`, `quotaErrorOccurred = false`, `creditsNotificationShown = false`, `modelAvailabilityService.reset()`, `modelQuotas.clear()`, `lastRetrievedQuota = undefined`, `lastEmittedQuotaRemaining = undefined`, `lastEmittedQuotaLimit = undefined`, `lastQuotaFetchTime = 0`, `hasAccessToPreviewModel = null`. The `lastModeSwitchTime = performance.now()` (rather than `0` or `undefined`) is correct — downstream consumers compute elapsed since this anchor and a `0` would produce a multi-decade-large delta.
- `packages/core/src/config/config.ts:1830-1832` — `resetNewSessionState(sessionId: string): void { this.setSessionId(sessionId); }` shrinks to a one-line wrapper since the body moved. The prior `this.approvedPlanPath = undefined` is removed from this method (now lives in `setSessionId`), so all callers — both `resetNewSessionState` and any direct `setSessionId` invocation — get the same flush. This is the load-bearing structural fix: previously only the "new session" path got the wipe, and the "resume" path that calls `setSessionId` directly inherited the stale state.
- `packages/core/src/skills/skillManager.ts:29-35` — new `reset(): void { this.activeSkillNames.clear(); }` with a one-line JSDoc explaining "Resets session-scoped state (active skill names)." The other field on `SkillManager` (`this.skills`) is intentionally NOT cleared because it's the loaded skill registry which is process-scoped, not session-scoped — correct distinction. A future `SkillManager.reset()` reader needs to understand this scoping decision; the JSDoc covers it.
- The reset list reads as a bit of a grab-bag that grew organically — there's no central registry of "session-scoped fields" to enforce that a future field added to `Config` will be remembered here. A follow-up could introduce a typed `SessionScopedState` struct that is recreated by reference assignment instead of 14 individual lines, eliminating the "next person adds field, forgets to reset" footgun. As-is the fix is correct but the maintenance shape invites regression.
- No new tests in this PR. For a fix whose entire premise is "these 14 things were silently leaking across session boundaries," at least one regression test that calls `setSessionId("B")` after dirtying a representative subset (`approvedPlanPath`, `quotaErrorOccurred`, `skillManager.activeSkillNames`) and asserts they're reset would lock the contract. Otherwise a future contributor refactoring `setSessionId` could silently re-introduce the leak with no failing test.

## Verdict
`merge-after-nits` — add a regression test asserting `setSessionId(B)` clears at least 3 of the 14 fields (one boolean, one map/set, one timestamp) so the contract is locked, and consider a follow-up to consolidate the 14 ad-hoc fields into a `SessionScopedState` substruct so future additions can't accidentally bypass the reset. The structural decision (move reset into `setSessionId` so resume and new-session share the path) is exactly right for the reported symptom.
