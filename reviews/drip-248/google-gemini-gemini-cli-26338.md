## Summary
Adds an experimental Auto Memory inbox flow (`experimental.autoMemory`, default off): a background extraction agent scans past conversation sessions on session startup and proposes durable memory updates as unified-diff `.patch` files in a project-local inbox at `<projectMemoryDir>/.inbox/<kind>/extraction.patch`. The load-bearing design choice is **nothing is applied automatically** — the user reviews each entry in `/memory inbox` and approves or dismisses. Two storage tiers (`private` per-project, `global` shared) with sibling-`.md` topic files. Three modes: off, manual review (this PR), and a future auto-apply gate. +4397 / -94 across 26 files. Includes two live-LLM evals (`auto_memory_contract.eval.ts`, `auto_memory_modes.eval.ts`) that pin the contract: extraction agent must use canonical filename `.inbox/<kind>/extraction.patch` and must rewrite-in-place rather than create siblings.

## Line-level observations
- `packages/core/src/services/memoryPatchUtils.ts` (new, ~390 lines) — defines `InboxPatch`, `InboxMemoryPatchEntry`, `InboxMemoryPatch` interfaces and `getAllowedMemoryPatchRoots(config)` accessor. The `getAllowedMemoryPatchRoots` function is the load-bearing trust boundary — it returns the only paths under which a `.patch` may be applied, so a malformed or attacker-influenced patch claiming `--- a/etc/passwd` would be rejected at apply time. The shape (allowlist-of-roots returned from a config-aware function) is correct; the apply-side enforcement was not visible in the diff slice and deserves verification.
- `packages/core/src/config/storage.ts` plus `packages/core/src/config/scoped-config.ts:48-58` — new `hasScopedMemoryInboxAccess()` predicate and `runWithScopedMemoryInboxAccess<T>(fn: () => T): T` AsyncLocalStorage scope. This is the right shape for "only certain code paths may touch the inbox" — the apply path scopes itself, all other paths see `false`. Combined with the path-allowlist this gives two independent gates (path-based and call-site-based) which is appropriate for a feature whose entire premise is "an LLM proposes file mutations."
- `evals/auto_memory_contract.eval.ts:200-260` — first eval pins **two** contract invariants in one test: (1) `inbox.privateFiles.length <= 1` AND `inbox.globalFiles.length <= 1` so the agent never creates `extraction-2.patch` or `extraction-3.patch` siblings, (2) when a file IS produced, the name must be exactly `extraction.patch`. Both assertions are tight. The companion eval at `:267-330` pre-seeds an existing `extraction.patch` and asserts the agent merges new findings into the same file (preserves the old hunk, adds the new hunk) rather than orphaning the prior content. This is the right discipline for a feature where the agent's output IS a contract — pinning the canonical filename means a future model regression is caught.
- `packages/core/src/agents/skill-extraction-agent.ts` (new, ~250 lines visible) plus `packages/core/src/agents/local-executor.ts:528-557` — extraction runs as a typed `LocalAgentDefinition<TOutput extends z.ZodTypeAny>` with structured output, which means the agent's filename/path/hunk choices are *parser-validated* before the patch ever reaches the inbox. The `LocalAgentExecutor` change at the same range threads the new agent definition through the existing executor harness — minimal new code path, maximum reuse of existing execution & tracing.
- `packages/cli/src/ui/components/SkillInboxDialog.tsx` (new, ~450 lines) and matching `.test.tsx` — the user-facing review surface. The dialog shows pending patches with approve/dismiss; the test file at `:1068-1318` exercises both interactions. The default-off (`experimental.autoMemory: false`) plus the explicit-approval gate means the worst case for an enabled-but-misbehaving extraction agent is a noisy inbox, not silent file mutations — fail-safe by design.
- `packages/cli/src/acp/commands/memory.ts` and `packages/cli/src/ui/commands/memoryCommand.ts:130-185` — `InboxMemoryCommand` extends the existing memory command surface. The `:171-189` arm shows the inbox listing flow now sources entries via the new `memoryPatchUtils` helpers rather than ad-hoc filesystem walks, which centralizes the canonical-path logic. Worth verifying that the listing helper rejects files that aren't named `extraction.patch` so a stray hand-edited `.patch` in the inbox can't slip through.
- `scripts/check-inbox.js` and `scripts/seed-test-inbox.js` (both new) — operator-grade utilities for CI / dev. The `seed-test-inbox` script is the right answer to "how do I test the dialog locally without waiting for the extraction agent to run" and `check-inbox` gives CI a way to assert empty state.

## Verdict
`needs-discussion` — feature shape is impressive (typed agent, structured output, two-tier storage, explicit-approval gate, contract-pinning evals) but the +4397 line surface area and the LLM-as-file-author trust model deserve a focused security review before merge. Specific concerns: (a) confirm the patch-apply path enforces `getAllowedMemoryPatchRoots()` allowlist and rejects `..` traversal, (b) confirm the inbox listing rejects non-canonical filenames so a stray `.patch` written by another tool can't be applied, (c) the live-LLM evals are a strong contract pin but cost real money on every CI run — recommend gating them behind a `runOnce` or PR-label trigger, (d) document the worst-case threat model (a hostile session transcript that influences extraction-agent output to propose a malicious patch) explicitly in the PR body so the security boundary is reviewable. The fail-safe defaults (off by default, manual approval only) absorb most of the risk; the gates and eval pinning are the right shape; the surface is just too big to merge without an explicit security pass.
