# BerriAI/litellm #26991 — fix(bedrock): forward output_config.effort for adaptive-thinking Claude

- Link: https://github.com/BerriAI/litellm/pull/26991
- Head SHA: `f52cbae60c259807e21af19f28b99743780b4f6c`
- Author: mateo-berri

## Summary
Closes a customer-reported silent-drop bug on the LiteLLM Anthropic-via-Bedrock route where `output_config.effort` for Claude Sonnet 4.6/4.7 (adaptive-thinking models) was stripped from every Bedrock surface (Converse, Invoke, `/v1/messages`, `/completion`) across v1.83.7 → v1.84.0 — the parameter never appeared in the wire body. Root cause: Bedrock Converse exposes Anthropic's `effort` parameter through `additionalModelRequestFields.thinking.effort` (per AWS adaptive-thinking docs), not as a top-level `output_config` block, but Anthropic Messages API callers naturally send the `output_config: {effort: ...}` shape and the `inference_params.pop("output_config", None)` line at `:1283` was unconditionally stripping it before any translation. Fix folds `output_config.effort` into `thinking.effort` for adaptive-thinking models *before* the strip, with explicit guards against overriding caller-set `thinking.effort`. +354/-8 across 5 source files and 3 test files.

## Line-level observations
- `litellm/llms/bedrock/chat/converse_transformation.py:453-470` — new `_handle_reasoning_effort_parameter` arm wraps the existing `AnthropicConfig._map_reasoning_effort()` call with an adaptive-thinking-only branch: `if AnthropicModelInfo._is_adaptive_thinking_model(model) and "thinking" in optional_params and optional_params["thinking"].get("type") == "adaptive" and "effort" not in optional_params["thinking"]`, then maps via a literal `effort_map = {...}` and assigns `optional_params["thinking"]["effort"] = effort_map.get(reasoning_effort, reasoning_effort)`. The four-condition guard is the right belt-and-suspenders shape: model accepts adaptive thinking AND thinking block exists AND it's adaptive-typed AND caller didn't already set effort. The `effort_map.get(..., reasoning_effort)` pass-through default lets the caller bypass the OpenAI-style `low/medium/high` mapping by passing the raw Bedrock value.
- `converse_transformation.py:53-93` — new `_fold_output_config_effort_into_thinking(inference_params, model)` static helper performs the load-bearing fix: rejects when not adaptive-thinking, pulls `effort` only when `output_config` is a `dict` and `effort` is a non-empty `str`, then either merges into existing `thinking` dict (only if `"effort" not in thinking`) or constructs a new `{"type": "adaptive", "effort": effort}` block. The `if isinstance(thinking, dict): if "effort" not in thinking:` arm correctly preserves caller intent — explicit `thinking.effort` wins over `output_config.effort`. The else-arm constructs a fully-formed adaptive thinking block rather than leaving `output_config` for the (already-existing) Anthropic legacy translator.
- `converse_transformation.py:1257-1273` — call site for `_fold_output_config_effort_into_thinking` is placed *immediately before* the existing `inference_params.pop("output_config", None)` strip at the bottom of `_prepare_request_params`, with a 7-line operator-grade comment citing "AWS adaptive thinking docs" and explaining the rationale. The ordering is correct: fold first, then strip, so the value reaches `additionalModelRequestFields.thinking.effort` before the cleanup. The dead-code `inference_params.pop("output_config", None)` at the prior `:1283` is removed since the new helper plus the surviving pop at `:1273` cover all cases.
- `litellm/llms/bedrock/chat/invoke_transformations/anthropic_claude3_transformation.py:14` and matching `messages/invoke_transformations/anthropic_claude3_transformation.py` — both Invoke surfaces import `AnthropicModelInfo` and apply the same fold. This is critical: the bug report explicitly tested across Converse AND Invoke AND `/v1/messages` AND `/completion`, so a fix that only touched Converse would leave 3 of 4 surfaces broken. The PR covers all four.
- `anthropic_beta_headers_config.json:9-10` flips `"effort-2025-11-24": null` to `"effort-2025-11-24": "effort-2025-11-24"` — turns a previously-no-op beta header into an actually-emitted one. Worth confirming with the AWS docs cited that the beta header is required; a `null` to non-null change here can flip the request to be rejected by Bedrock if the model deployment doesn't yet have the beta whitelisted. Also `litellm/types/llms/bedrock.py` introduces typed shapes for the new fields which gives downstream consumers IDE-level validation.
- Three test files cover Converse + both Invoke surfaces. The Converse test asserts the request body's `additionalModelRequestFields.thinking` dict contains `effort: "high"` after the call — the right shape, since the bug was a wire-body absence not a return-shape problem. Missing: a negative test asserting that for a *non-adaptive* model the `output_config` is still stripped (not folded into a non-existent `thinking` dict, since the fold guard at `:73` early-returns).

## Verdict
`merge-after-nits` — confirm the `effort-2025-11-24` beta header flip is intentional and that all production Bedrock Claude 4.6/4.7 deployments accept it (a `null → non-null` change here is a non-trivial wire-body delta that could regress callers whose Bedrock model isn't yet on the beta), and consider adding a negative test that a non-adaptive Claude model still gets `output_config` cleanly stripped without picking up a stray `thinking` block. The four-surface coverage (Converse + two Invoke + Messages) and the explicit "fold before strip" ordering are exactly right for the reported symptom.
