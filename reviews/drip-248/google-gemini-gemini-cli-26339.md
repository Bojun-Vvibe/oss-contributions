# google-gemini/gemini-cli #26339 — fix(cli): prevent Escape from clearing input buffer (#17083)

- Link: https://github.com/google-gemini/gemini-cli/pull/26339
- Head SHA: `584e8021c549127abf0febde85d1f428010fef6f`
- Author: cocosheng-g

## Summary
Closes #17083: pressing `Escape` during a tool execution unexpectedly wiped whatever the user had typed into the prompt buffer. Root cause is that the cancellation path in `AppContainer.cancelHandlerRef` unconditionally called `buffer.setText('')` whenever a tool was executing, and `Escape` shared that path with `Ctrl+C`. Fix threads a new `clearBuffer?: boolean` flag through the entire cancel chain — `cancelHandlerRef → onCancelSubmit → cancelOngoingRequest → onCancelSubmit` — so that `Ctrl+C` (hard interrupt, default `clearBuffer=true`) keeps clearing while `Escape` (soft cancel, explicit `clearBuffer=false`) preserves the user's text. +169/-112 across 5 files (3 source + 2 test).

## Line-level observations
- `packages/cli/src/ui/AppContainer.tsx:1127-1141` — `cancelHandlerRef` widens its signature from `(shouldRestorePrompt?: boolean) => void` to `(shouldRestorePrompt?: boolean, clearBuffer?: boolean) => void`, and `onCancelSubmit` now propagates `clearBuffer` through to `cancelHandlerRef.current(false, clearBuffer)`. The default for `clearBuffer` is `false` at the `onCancelSubmit` arrow (`clearBuffer: boolean = false`) — this is the conservative choice and correct: any caller that forgets to pass the flag gets the safe "preserve user text" behavior, not the destructive "wipe" behavior. Fail-safe defaults at the API boundary.
- `AppContainer.tsx:1324-1341` — the body of `cancelHandlerRef.current` is rewritten. The prior tool-executing arm `if (isToolExecuting(pendingHistoryItems)) { buffer.setText(''); return; }` is removed entirely; now `buffer.setText('')` only fires inside the new arm `if (!shouldRestorePrompt) { if (clearBuffer) { buffer.setText(''); } return; }`. The relocation is the load-bearing fix: clearing is now gated on the explicit caller intent, not on the accidental coupling to "is a tool currently running." The `if (!clearBuffer && isToolAwaitingConfirmation(...))` early-return at `:1326` is also correctly tightened — only skip the cleanup if we're *not* clearing, since a hard Ctrl+C should still clear during an awaiting-confirmation prompt.
- `packages/cli/src/ui/hooks/useAgentStream.ts:127-140` — new `useKeypress` listener fires when `key.name === 'escape' && !isShellFocused` AND when the streaming state is `Responding | WaitingForConfirmation`, calling `void cancelOngoingRequest(false)` (note: `clearBuffer=false`). The `isShellFocused` guard correctly defers Escape handling to the shell input when it has focus. The `isActive` predicate scopes the listener so Escape during idle state doesn't accidentally fire — important because `useKeypress` would otherwise compete with other Escape consumers in dialog/dropdown components.
- `packages/cli/src/ui/hooks/useGeminiStream.ts:807-822` — `cancelOngoingRequest` gains a `(clearBuffer: boolean = true)` parameter. Default `true` here is the *opposite* of the `AppContainer.tsx` default, but that's intentional: the legacy callers (Ctrl+C handler) call `cancelOngoingRequest()` without args and should keep the destructive default; the new Escape caller in `useAgentStream.ts` passes `false` explicitly. The arm `if (turnCancelledRef.current) { if (clearBuffer) { onCancelSubmit(false, true); } return; }` correctly handles the "cancel called twice in a row" case — the second call still gets to clear if it's a Ctrl+C, even though the first cancel has already done the heavy lifting.
- Test updates: `useAgentStream.test.tsx:205` flips the assertion from `expect(mockOnCancelSubmit).toHaveBeenCalledWith(false)` to `(false, true)` for the abort case (legacy Ctrl+C semantics preserved), and `useGeminiStream.test.tsx:1640` adds `(false, false)` for the Escape case. Both are tight assertions that lock the new two-argument contract. Missing: a test asserting that `Escape` during `isToolAwaitingConfirmation === true` does NOT call `cancelOngoingRequest` at all (the current `isToolAwaitingConfirmation` early-return at `AppContainer.tsx:1326` is now coupled with `!clearBuffer`, which is a subtle combination worth pinning).
- The two opposite default values (`AppContainer.onCancelSubmit` defaults `clearBuffer=false`; `useGeminiStream.cancelOngoingRequest` defaults `clearBuffer=true`) are correct given the call sites but increase cognitive load for future contributors. A one-line comment at each default explaining "soft cancel default" vs "legacy hard-cancel default" would help the next reader.

## Verdict
`merge-after-nits` — add a regression test that asserts Escape during `isToolAwaitingConfirmation` is a no-op (covers the new `!clearBuffer && isToolAwaitingConfirmation` early-return), and add inline comments at the two default-value sites explaining the asymmetric defaults. The structural shape — a `clearBuffer` flag threaded through the cancel chain with conservative defaults at the entry point and legacy-preserving defaults at the leaf — is exactly the right fix for a "two distinct user intents share one code path" bug.
