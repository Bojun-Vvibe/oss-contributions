## Verdict: merge-as-is

## Rationale

This is a textbook "move the helper to the right crate" refactor. The function being moved (`sandbox_mode_from_permission_profile`) is a pure projection from one wire shape (`PermissionProfile`) to another (`SandboxMode`) — neither shape is TUI-specific, both live in the protocol crate, and the helper's only consumer is the three sibling functions in `app_server_session.rs` that build `ThreadStartParams` / `ThreadResumeParams` / `ThreadForkParams` (themselves protocol types). Putting the projection in the protocol crate lets the *same* projection be reused by any future non-TUI client that also needs to build `ThreadStartParams` against the legacy sandbox field — currently zero such clients exist in this diff, but the abstraction is correctly placed.

The lossiness is properly documented at the function-doc level: the `/// This is intentionally lossy and should only be used when a / client cannot send the newer permissions selection API.` line is the right way to mark "this is a backward-compat shim, not the future". The new function name (`thread_sandbox_override_for_permission_profile`) is also better than the old (`sandbox_mode_from_permission_profile`) — it surfaces both the call site (the `sandbox` *override* field on a thread-start params) and the directional intent (override-when-permissions-not-supplied) in the name itself. This is a real readability win.

The 75-line regression at the new location locks all four match arms with named fixtures and explicit `assert_eq!` against the expected enum variants, and the pre-existing TUI-side test is rewired rather than dropped, so coverage strictly increases. The control-flow shape change in the three call sites (`then(...).flatten()` → `if/else` with a direct `None` return) is incidental but reads better after the move because the helper is no longer returning `Option<...>` from a `then` closure that itself was wrapped in a `.flatten()`. Net zero behavior change, +18 net lines (mostly the new doc comment + the new test fixture build), correct placement, properly tested. Merge-as-is.

## What I learned

The single most-actionable signal a reviewer should look for in a "move helper to a different crate" refactor is whether the *call sites* changed shape at all. If the call sites become a one-line direct invocation (as here, `thread_sandbox_override_for_permission_profile(&config.permissions.permission_profile(), config.cwd.as_path())`) and the helper's internal logic is byte-identical to the deleted version, the move is safe and the only risk is the new visibility making the helper available to unintended call sites later. Conversely, if the move came with even small body changes (an `if let` reordered, an extra `.unwrap_or_default()`, a swapped match arm), those are the things to dig into — the "move" framing camouflages real behavior changes inside what looks like a refactor. Checked here, the body is byte-identical, so the refactor is honest.
