## Verdict: merge-after-nits

## Rationale

This is a real functionality gap being closed. Pre-PR, the MCP hook system documented that `pre_mcp_call` could return `extra_headers` to be forwarded on the outbound MCP tool call, but the OpenAPI-backed-server code path had a special case that dropped those headers and emitted a warning instead — meaning anyone running an OpenAPI-backed MCP server (which is the common case for MCP-via-REST integrations) couldn't use hook header injection at all. The fix wires the hook-supplied `extra_headers` through the same ContextVar plumbing already used for `_request_auth_header`, which is the right shape: `contextvars.ContextVar` is the correct primitive for per-request state in an asyncio path, the static-then-request layering matches the auth precedent, and the LIFO token reset in the `finally` block prevents leakage across requests.

The composition at `_build_openapi_request_extra_headers` correctly orders the three header sources: OAuth2-derived headers (lowest precedence, since they're a baseline auth shape), then `mcp_server.extra_headers ∩ raw_headers` (mid-precedence, the server-config-declared forwarding allowlist intersected with what the inbound request actually carried — case-normalized at `:2407-2409` which is critical because HTTP headers are case-insensitive and a `dict` lookup isn't), then `hook_extra_headers` (highest precedence, since the hook is the runtime policy decision and should be able to override either of the static channels). This ordering matches the auth-header precedent and is the right policy shape.

The test inversion is honest. The pre-existing test was locking the *old* warning-and-drop contract (`mock_logger.warning.assert_called_once()` + the substring check on the message); the new test locks the *new* forward-and-merge contract (`mock_call.call_args.kwargs["request_extra_headers"] == {"Authorization": "Bearer jwt"}`). The test was renamed to match (`..._warns_and_continues_on_hook_headers` → `..._forwards_hook_headers`) so the test name no longer lies about what it's checking.

Nits before merge:

1. **Two near-identical helpers across the manager and the local-server.** `_build_openapi_request_extra_headers` (manager, +30 LOC at `mcp_server_manager.py:2393-2424`) and `_get_request_extra_headers_for_openapi_tool` (local-server, +24 LOC at `server.py:1127-1150`) duplicate the OAuth2-headers + server-config-extras intersection logic. The only difference is that the manager one accepts an additional `hook_extra_headers` parameter for the layered composition; the local-server one doesn't. A single helper with the hook-extras parameter defaulting to `None` would deduplicate cleanly and prevent silent drift on the next change. Minor refactor.
2. **The case-normalization pass at `:2407-2409` of the manager (and at `:1138-1140` of `server.py`) only normalizes the *lookup* side, not the storage side.** When the resolved `header_value` is written back into the `extra_headers` dict at `:2415` / `:1146`, the *original-case* header name from `mcp_server.extra_headers` is used as the key — but the value came from a case-insensitive lookup against the raw inbound headers. This is fine for the immediate downstream consumer (httpx will lowercase header names on send anyway), but if a future caller treats `extra_headers` as case-sensitive, the asymmetry could surprise. Worth a one-line comment naming the contract: `# header keys carry the server-config canonical case; values are looked up case-insensitively against inbound`.
3. **No regression test for the precedence ordering.** The new test asserts the hook-headers path lands the right header value, but doesn't lock the documented precedence (`hook > server-config > OAuth2`). A small additional test that supplies all three with the same header name and asserts the hook-supplied value wins would prevent a future "let's reorder the layers for cleanliness" patch from silently breaking the hook contract.
4. **The pre-PR warning at `:2683-2691` is now dead code that was deleted (`-9 lines`).** Good — leaving the warning in place after wiring the actual forwarding would have been confusing. But there's no migration note for operators who had built dashboards/alerts off that warning string. A one-line CHANGELOG entry naming the behavior flip would help.

None of these block merge — the headline change is correct and tested. They're cleanup before stamping.

## What I learned

The pattern of "dropping data and emitting a warning" is almost always a placeholder for "we haven't built the proper plumbing yet" — and reviewing this PR is a good example of how to flip such a placeholder cleanly. The right move when removing a warning-and-drop branch is (a) confirm the test that was locking the old behavior is *also* updated to lock the new behavior (not just deleted), (b) check that the precedence ordering between the new data path and any existing competing data paths is documented, and (c) verify the new path uses the *same* primitive as the closest sibling (here, the `_request_auth_header` ContextVar). All three are clean here. The duplication between `mcp_server_manager.py` and `server.py` is the one real refactor opportunity, but it's small enough to be a follow-up rather than a merge blocker.
