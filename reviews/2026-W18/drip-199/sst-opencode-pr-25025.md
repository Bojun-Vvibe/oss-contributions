## Rationale

This is a clean dictionary-spelling cleanup — `adaptor` is an accepted UK variant but the codebase had been mixing `adapter` (TypeScript ecosystem default, used everywhere else in the file) with `adaptor` only in this subsystem, so picking one and sticking is correct. The PR is consistent: 22 files, 163 lines moved without delta, no behavior change, no schema-shape change beyond the field-name. The author also remembered to regenerate the SDK and OpenAPI output (visible in the auto-generated artifact diffs) so the public client surface lands the rename in the same commit, which is the right call for an unreleased experimental route.

The headline risk is the wire-route rename at `/experimental/workspace/adaptor → /experimental/workspace/adapter`. This is gated behind the `experimental/` prefix so the contract is "may break without notice" by convention, but anyone running off a recent `main` against a stale TUI or pinned SDK will see the route 404 silently — the PR doesn't add a redirect or a deprecated alias for one release. For an experimental route this is defensible (the whole point of the prefix is to free the maintainer from compat overhead), but a one-line note in `CHANGELOG.md` or release notes calling out the renamed route would help downstream consumers who pinned an SDK version.

The other nits are minor: the test file list in the PR body (`test/control-plane/adapters.test.ts test/plugin/workspace-adapter.test.ts ...`) shows the test files were also renamed from `adaptors` to `adapters`, which is good, but I'd also want to grep the entire repo for any surviving `Adaptor`/`adaptor` substring to make sure no doc/comment/log line drifted out of sync — a lazy `rg -i 'adaptor' packages/` should return zero. The plugin-API surface change (`registerAdaptor → registerAdapter`) is a breaking change for any third-party plugin that called `registerAdaptor(projectID, type, adaptor)`; for an experimental plugin API this is OK but again warrants a one-line CHANGELOG mention so plugin authors aren't surprised. None of this blocks merge — fine to land as-is and follow up with the CHANGELOG, but worth the author's two-minute pass.

## What I learned

When a single-token rename touches a directory name, the file content, the wire-route segment, the SDK export, the spec docs, and the test fixtures all in one PR, git's diff renderer will usually show some files as `rename ... similarity N%` and others as paired delete+create — depending on whether the identifier substring count exceeded the similarity threshold. The latter pattern (delete+create) makes the PR look bigger than it is and obscures whether anything was actually re-thought during the rename. Reviewing this kind of PR, the right move is to compare the deleted file against the created file line-by-line (which I did for `adaptors/index.ts` vs `adapters/index.ts`) rather than trust the diff summary — a 100% rename should produce a logically zero-line diff, and any deviation from that should be called out as either a deliberate hidden change or as scope creep.
