## Concerns / nits

1. **The `normalize_extension_entry` semantics at `config/extensions.rs:53-58` is a real behaviour change**: any non-hidden platform extension entry persisted with `enabled = false` is now silently flipped to `enabled = true` on every config load. This intentionally fixes a UX bug (users couldn't disable `developer`) by making non-hidden platform extensions **always enabled**. Two concerns:
   - Users who deliberately disabled a non-hidden platform extension (e.g. opted-out of `goosehints` if it's non-hidden) will silently see it re-enabled on next load, with no migration notice.
   - The change is conflated with the unrelated extension-management UI rework. Recommend extracting it into its own commit with the rationale "non-hidden platform extensions are always on; if a user wants to disable them they need to mark them hidden" so the next reader doesn't think it's a bug.
2. **`tool_call_meta` parses extension name from `tool_name.split_once("__")` at `acp/server.rs:540`**. This is a string-shape contract — if any extension's tool name happens to contain `__` legitimately (or if the prefix-naming convention changes), the parsed `extensionName` will silently be wrong. Worth either (a) a comment naming the convention "tool names follow `<extension>__<tool>` prefix shape" or (b) sourcing the extension name from the call site's known context (the calling code knows which extension owns the request).
3. **`error: "Goose could not connect this extension when the chat started."`** at `acp/server.rs:3263-3268` is hardcoded English. Either route through the existing i18n machinery (the PR adds 47 i18n keys, so the infra is right there) or document why this particular string is intentionally not localized.
4. **`failed` is the only non-connected status** at `acp/server.rs:3253-3258`. Reality is more nuanced: an extension might be `disabled`, `not-yet-loaded`, `loading`, `crashed-during-handshake`, etc. Collapsing all non-connected to `"failed"` loses signal. A second pass with a richer status enum would help operator debugging — file a follow-up.
5. **`#[derive(Debug, Default, Clone, Serialize, Deserialize, JsonSchema, JsonRpcRequest)]` with `Default`** at `goose-sdk/src/custom_requests.rs:151`: a default `GetSessionExtensionStatusRequest { session_id: "".to_string() }` will silently dispatch to "no session" — confirm the handler at `acp/server.rs:3183` rejects empty `session_id` early rather than treating it as a hint to use a default session.
6. **44 files** is large. The breakdown — backend ACP additions (5 files), config-extensions normalization (2 files), UI rework (~30 files), generated SDK (4 files), i18n (6 files) — could be split into "backend additive ACP method" + "extension normalization" + "UI rework" + "i18n", each independently reviewable. Acknowledged that UI/backend coupling makes a clean split hard; the minimum useful split is "backend + SDK + i18n" landing first, "UI rewrite" second.
7. **Tools-list per-extension** at `acp/server.rs:3239-3247` calls `agent.list_tools(&internal_id, Some(extension.name().to_string()))` inside the per-extension loop. For workspaces with many extensions this is N round-trips through the agent surface; `agent.list_tools(internal_id, None)` once and grouping client-side would be O(1) rather than O(N). Probably not a hot path but worth a follow-up profiling note if extension counts grow.
8. **`ContextPanel.test.tsx` adds 4 lines, no other UI tests added for the `ExtensionsSettings` rewrite** — actually wrong: re-reading, `ExtensionsSettings.test.tsx` is +63, `extensionCategories.test.ts` +83, `extensionUsage.test.ts` +98. So the rewrite has ~244 LOC of new tests, which is appropriate. Disregard this nit; it's a positive signal.
