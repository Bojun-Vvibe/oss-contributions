## Concerns / nits

1. **PowerShell AST parsing is process-spawning.** `parse_with_powershell_ast` (at `powershell_parser.rs:34`) routes through `parse_with_cached_process(&mut parser_processes, ...)`. The exec-policy hot path is now `cfg(windows)` → spawn-or-reuse a parser process per command-evaluation. Caching mitigates per-command spawn cost, but `commands_for_exec_policy` is called for *every* exec attempt. If the cached process dies (Windows defender, AV scanner, terminal session teardown), there's a re-spawn cost on the next command. Worth a brief note in the PR description about expected per-command latency on Windows and whether the cache survives across `exec_policy.rs` callers.
2. **The fallback chain at `exec_policy.rs:710-718` now has three branches** (bash → powershell → single-command-prefix). When `parse_powershell_command_plain_commands` returns `Some(commands)` with `commands.is_empty()`, the `&& !commands.is_empty()` guard correctly falls through to the next branch. This matches the bash branch above (`commands.is_empty()` triggers fallback at `:706`). Good consistency. However, there's no test for the empty-script fallback (`powershell.exe -NoProfile -Command ""` → expect `(vec![original_command], false)` from the trailing fallback). The `commands_for_exec_policy_falls_back_for_empty_shell_script` test at `:719` covers bash; a `cfg(windows)` mirror would close the symmetry.
3. **`parse_powershell_command_plain_commands` is `pub`** in `powershell.rs:79` but the single caller is in `exec_policy.rs`. If this is intentional (for consumer crates), document the API contract. If not, consider `pub(crate)` to match the safety helper's scoping.
4. **Test coverage for `pwsh.exe`** (cross-platform PowerShell Core) — the tests at `:660` and `:710` only exercise `powershell.exe`. `extract_powershell_command` presumably handles both, but a third test exercising `pwsh.exe -Command "echo blocked"` would lock the cross-platform-Pwsh path against future divergence.
5. **The PR title** "Use inner powershell script/command in execpolicy" is accurate but understates the impact: this re-enables the `execpolicy_blocks_shell_invocation` Windows test that's been TODO'd out, plus adds three new prefix-rule tests. Worth a one-line "Closes #<issue>" in the body if there's a tracking issue for the TODO.
