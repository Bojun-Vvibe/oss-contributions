## Concerns / nits

1. **Concurrency gate at `Session.ts:291-296` checks `this.pendingPrompt` but doesn't acquire a lock.** If a rewind and a new prompt race (rewind passes the check, then a `pendingPrompt` is set before `chat.truncateHistory` runs), the truncation happens mid-flight. The `pendingPrompt` set/clear surface isn't visible in this diff slice — worth confirming there's a single-threaded event-loop ordering guarantee, or making the check + truncation atomic via a guard. The PR body's "destructive — irrecoverable" framing is correct; the race window deserves an explicit comment.
2. **`stripThoughtsFromHistory()` at `:313` is called unconditionally** after truncation. If the user is rewinding to keep the original assistant turn (turn 0 → keep the first model response), stripping reasoning content from already-replayed assistant messages may cause the next replay to render those messages without their reasoning blocks. Worth confirming this is intentional vs an accident of "we strip thoughts whenever we mutate history".
3. **Startup-context detection at `#hasStartupContext`** matches model output literally on `'Got it. Thanks for the context!'` (constant at `:272`). If the system prompt is localized (qwen-code does ship multi-locale prompts) or if the model paraphrases ("Got it, thanks!"), the detection silently fails and `startIndex = 0`, causing rewind-to-turn-0 to delete the startup context. The detection should be either a structural marker (e.g., a metadata flag on the message) or a regex-with-margin, not a literal string equality.
4. **`conversationRewound` reducer is the highest-blast-radius change** but the actual reducer body in `useMessageHandling.ts` isn't fully reviewable from this diff slice. The PR body lists what it should drop ("messages, tool calls, plans, usage stats, permission prompts, waiting/thinking states") — that's 6 distinct sub-stores. A test (`useWebViewMessages.test.tsx` is mentioned in the PR commands) that asserts each sub-store is reset and *no extra state mutations happen* would be the right safeguard. Otherwise a future state addition will silently survive rewind.
5. **`MessageMeta` exposing an "Edit" button on assistant messages** (per PR body) is a UX choice that may not match the model's mental model of conversation. Editing an assistant message is normally either (a) regenerate from this point, (b) edit-and-pretend-the-model-said-this, or (c) edit-and-rewind-to-the-prior-user-turn. Worth confirming which semantics the assistant-message edit triggers and documenting it in the component's prop comment.
6. **No e2e/integration test in the actual VSCode extension** (per PR's own "Not covered" section). For a destructive feature, that's a real coverage gap. The unit tests cover the truncation math but not the cross-process IPC (companion ↔ agent ↔ webview reducer). At minimum, a recorded vitest scenario that walks the full happy-path message flow would lock the contract.
