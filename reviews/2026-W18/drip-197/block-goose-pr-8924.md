Previously, if a user added a custom provider that doesn't require an API key, the CLI was still persisting whatever value happened to be in the `api_key` local. The `requires_auth.then_some(...)` collapses to `None` when `requires_auth == false`. Single-line correctness fix.

## Concerns / nits

1. **Two implementations of the same trim-empty-API-key normalizer.** `normalize_custom_provider_api_key` at `config_management.rs:108-113` and `normalize_custom_provider_upsert` at `acp/server.rs:1106-1109` both inline the trim + `(!api_key.is_empty()).then_some(api_key)` logic. They should share one helper in the SDK or in `declarative_providers.rs`. Otherwise a future change (e.g., "also reject API keys < 8 chars") will land in one path and not the other, leading to silently divergent validation between CLI and ACP entry points.
2. **`api_key_set: bool` on the read DTO** (`custom_requests.rs:120-121`) is the right choice but worth confirming it's not derivable as `api_key.is_some()` *after* `normalize_*` has run â€” i.e., a stored `Some("   ")` would (without normalization-on-read) show `api_key_set: true` while attempting auth would fail. The normalization happens on *write*, so reads from a config file written by an older binary that didn't normalize could still expose the bug. Worth a one-shot migration that normalizes existing on-disk credentials, or a `normalize-on-read` shim in the DTO conversion.
3. **6921 added lines in a single PR** â€” this is a feature-complete vertical slice (CLI + ACP server + UI + tests + i18n + SDK regen), which is fine for the user-impact statement, but very hard to review in one pass. The Rust side (8 files, ~1300 lines) and the UI side (41 files, ~5600 lines) could be two PRs landing in sequence: backend lands first with integration tests as the contract, UI lands second consuming the now-stable ACP surface. Out of scope to restructure now; flag for future similar PRs.
4. **`acp-schema.json` regen at +617/-20** is a generated artifact but the PR doesn't surface the source-of-truth diff. Reviewers can't easily tell if the schema changes are exactly the new methods or also include unintended mutations to existing methods. A `cargo run --bin gen-acp-schema` reproducible step in the PR description would help.
5. **No URL validation visible in the diff scan** for provider `api_url`. `customProviderValidation.ts` mentions URL validation in its description but the actual content of the file isn't visible in this slice. Important to confirm the URL validator: (a) requires `https://` (or explicitly allows `http://localhost`), (b) rejects `file://`, `javascript:`, etc., and (c) doesn't allow paths that escape the base URL. Custom providers are an arbitrary-URL surface and a malformed provider URL is a credential-exfil primitive.
6. **`acp_custom_provider_methods_test.rs` is +604 lines for ~20 ACP methods.** Coverage looks dense but the file isn't visible in this diff slice â€” worth confirming it covers (a) create with empty api_key + requires_auth=true â†’ error, (b) update doesn't allow changing `name` (or does, with appropriate side-effects), (c) delete idempotency, (d) catalog template instantiation round-trip.
7. **The ACP method names are visible in `acp-meta.json` (+30 lines)** but not summarized in the PR body. Worth listing them: presumably `customProvider/create`, `customProvider/read`, `customProvider/update`, `customProvider/delete`, `customProvider/template/list`, `customProvider/catalog/list`. A reader of this PR has no easy way to enumerate the new public ACP surface.
