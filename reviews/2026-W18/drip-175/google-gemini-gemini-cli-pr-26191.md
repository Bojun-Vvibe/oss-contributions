# google-gemini/gemini-cli#26191 — fix(core): reduce default API timeout to 60s and enable retries for undici timeouts

- URL: https://github.com/google-gemini/gemini-cli/pull/26191
- Head SHA: `7d860c77625a5b37a19b725738bc1d7b156ca543`
- Files: 3 changed (+29 / -2); `fetch.ts` one-liner, `retry.ts` 9-line code change, `retry.test.ts` 19-line test addition

Fixes #18030 — API requests could hang for up to 5 minutes waiting on undici's default timeout when the backend or local proxy went unresponsive, with no retry. Two-part fix: (1) reduce `defaultTimeout` in `packages/core/src/utils/fetch.ts:31` from 300_000ms to 60_000ms, and (2) map undici-specific timeout codes (`UND_ERR_HEADERS_TIMEOUT`, `UND_ERR_BODY_TIMEOUT`, `UND_ERR_CONNECT_TIMEOUT`) to the standard `ETIMEDOUT` code in `getNetworkErrorCode` at `packages/core/src/utils/retry.ts:87-94` so the existing backoff resilience logic that already handles `ETIMEDOUT` automatically picks them up.

The retry.ts change is the genuinely clever part — rather than adding three new branches to the retry decision logic (where would they go? how do they interact with the existing `ETIMEDOUT` handler?), the PR collapses all three undici codes onto the canonical `ETIMEDOUT` at the `getNetworkErrorCode` boundary. This keeps the retry-decision call sites blissfully unaware that undici exists as an HTTP layer detail, and means future undici versions that introduce new `UND_ERR_*_TIMEOUT` variants only need one more `||` clause here rather than touching every retry policy. The test at `retry.test.ts:457-471` explicitly asserts the retry happens and the function is called twice (`expect(mockFn).toHaveBeenCalledTimes(2)`) — that's the right assertion shape because it pins both "the error was classified as retryable" and "the retry actually happened" rather than just "the promise resolved."

The 60s default is more arguable. 60s is a reasonable balance for chat completions (which usually finish in <10s) but might be aggressive for long-running operations like large embedding batches or the `gemini-2.0-pro` thinking-budget responses that legitimately run 90-120s. The PR doesn't surface this as a configurable knob and doesn't document the change in user-visible docs — users currently relying on the 5-minute timeout for legitimately-slow operations will start seeing premature `ETIMEDOUT` failures (which now retry, which compounds). I'd want either (a) a `GEMINI_REQUEST_TIMEOUT_MS` env var with the 60s as default, or (b) a per-operation timeout map where `generate` keeps a longer ceiling than `embeddings`/`countTokens`.

**Verdict: merge-after-nits** — add the env-var knob OR document the change prominently in the changelog with the "if you have legitimately-slow workloads, set X" guidance; add explicit test cases for `UND_ERR_BODY_TIMEOUT` and `UND_ERR_CONNECT_TIMEOUT` parallel to the `UND_ERR_HEADERS_TIMEOUT` case at lines 457-471 since the diff adds three codes but only tests one (a future refactor that drops one of the three would silently regress); and confirm the request-cancellation path correctly handles the case where the user hits Ctrl+C during the 60s window so we don't make a retry attempt the user already abandoned.
