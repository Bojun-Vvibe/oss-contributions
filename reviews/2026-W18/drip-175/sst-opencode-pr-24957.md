This PR rebases @aaryan-rampal's stalled #16447 onto the current Effect.ts-migrated `dev` branch and threads a `mode: "conversation" | "conversation_and_code"` parameter through the revert API. The split itself is well-justified — Slack-style "undo my last prompt without rolling back the file edits the agent already made" is a real workflow that the current single-button revert can't express. The TUI surface at `packages/opencode/src/cli/cmd/tui/routes/session/dialog-message.tsx:20-67` cleanly factors the duplicated `onSelect` body into a `revert(mode)` higher-order callback, so the two new menu entries ("Revert conversation" / "Revert conversation and code") share one code path — that's the right shape for this kind of fan-out.

Where it gets shaky is the persisted-mode behavior at `packages/opencode/src/cli/cmd/tui/routes/session/index.tsx:599-605` and `packages/app/src/pages/session/use-session-commands.tsx:328-329`. Both call sites read `session()?.revert?.mode` / `info()?.revert?.mode` and pass it as the keyboard-shortcut default. The PR description claims "the revert mode persists across sequential reverts," but I can't see where that field is *written* — `packages/opencode/src/session/revert.ts` adds 19 lines but the diff snippet I have doesn't show the persistence write. If the keyboard shortcut silently uses last-mode while the menu shows both options as equally weighted, users who clicked "Revert conversation" once and then hit the keyboard shortcut expecting "code too" are going to lose work that they thought they were rolling back. This needs either an explicit `persisted: true` indicator in the dialog or a confirmation toast on shortcut-triggered revert.

The 628-line test file is impressive in coverage breadth but the diff also smuggled in three unrelated additions to `message-v2.ts`: a new `stream` async generator at lines 984-1033, a `parts` lookup at lines 1042-1048, and a `get` function at lines 1051+. These have nothing to do with revert mode — they look like a forward-port from another branch that got tangled in the rebase. The PR author already self-flagged "I have not included unrelated changes in this PR" but the new `MessageV2.stream` / `parts` / `get` helpers definitely qualify. Also note the indentation-bug at `dialog-message.tsx:101` (`            if (part.type === "file") agg.parts.push(strip(part))` — 12 spaces in a block indented 18) which the auto-formatter will catch but is currently in the diff.

**Verdict: request-changes** — split out the unrelated `MessageV2.stream`/`parts`/`get` additions into their own PR, fix the dialog-message.tsx indentation, document where `revert.mode` is persisted (or remove the persistence claim), and add a test that asserts the keyboard-shortcut path picks up the correct persisted mode after a "conversation" select followed by an unrelated message.
