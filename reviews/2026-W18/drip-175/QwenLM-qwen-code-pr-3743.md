# QwenLM/qwen-code#3743 — fix(cli): prevent file paths from being treated as slash commands

- URL: https://github.com/QwenLM/qwen-code/pull/3743
- Head SHA: `776a2ab41cffb56e5ccaa5b1503e659074a1b683`
- Files: 4 changed (+163 / -3); 33-line test addition in `slashCommandProcessor.test.ts`, 12-line guard in `slashCommandProcessor.ts`, 85-line test addition in `commandUtils.test.ts`, 33-line helper in `commandUtils.ts`

Fixes #1804 — users typing absolute file paths like `/Users/zhoushuo/Desktop/AI_operator/dw-operator-skill 帮我安装` or `/api/apiFunction/接口的实现` were getting "Unknown command" errors and having their input dropped, because `isSlashCommand()` only checked for a leading `/` without validating the first token looks like a command name. The fix introduces a `looksLikeCommandName()` helper at `packages/cli/src/ui/utils/commandUtils.ts` (the new code is at line ~58 based on the test imports) that uses a `[a-zA-Z0-9:_-]` (plus dot, per the test for `gcp.deploy`) regex to validate the first token, and `isSlashCommand` rejects any first-token containing path separators, non-ASCII chars, or whitespace.

The defense-in-depth at `slashCommandProcessor.ts:452-460` is the right call — the early return inside `handleSlashCommand` independently re-checks `looksLikeCommandName(firstToken)` so even if a future caller bypasses `isSlashCommand` (or some refactor drops the guard there), file paths still fall through to model processing rather than producing "Unknown command" errors. This is a well-applied "trust nothing at the boundary" pattern. The character set chosen (`[a-zA-Z0-9:_-]` plus `.`) covers the four real command-name shapes in this codebase: bare names (`help`), MCP-style colons (`mcp:server__tool`), kebab-case (`pr-review`), snake_case (`issue_triage`), and extension-qualified dotted names (`gcp.deploy`) — the `gcp.deploy` test at `commandUtils.test.ts:108-111` is good evidence the author thought about extension authors who use dotted namespacing.

The test coverage is genuinely thorough — 30+ cases across `looksLikeCommandName` and the updated `isSlashCommand`, including the explicit Chinese-character regression at `commandUtils.test.ts:198-201` (`/接口实现`, `/文档 查看内容` → false). The integration test at `slashCommandProcessor.test.ts:858-874` exercises the full path through `handleSlashCommand` for both an English-only path and a Chinese-text-with-path case, and asserts `false` from both — the right shape because returning `false` is the contract for "fall through to model." One small gap: the regex at `commandUtils.ts` (visible only via the documented contract) accepts `123` as a valid command name per the test at `commandUtils.test.ts:107`. That's almost certainly intentional (someone might define a command literally named `1` or `2` for a numeric menu) but it has the side effect that `/123 something` will trigger the slash command flow even though no such command exists, producing the same "Unknown command" error this PR is trying to eliminate. Worth at least a comment.

**Verdict: merge-after-nits** — add an explicit test case for `/123 args` documenting the intentional behavior (or rejecting purely-numeric tokens, the choice is fine either way as long as it's documented); consider whether bare `/?` should also short-circuit to autocomplete the way bare `/` does (currently the test asserts only `/` does); and add one fuzz test that runs 100 random POSIX-shaped paths through `isSlashCommand` and asserts every one returns `false` — that's the property the PR is really claiming and it's currently only covered by case-by-case examples.
