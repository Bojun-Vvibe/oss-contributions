# openai/codex#20207 — fix(tui): make /copy work inside tmux without passthrough

- URL: https://github.com/openai/codex/pull/20207
- Head SHA: `9beb7718d8244a49c8cc75ea309056b92afd1b93`
- Files: 1 changed (+333 / -42); single file `codex-rs/tui/src/clipboard_copy.rs`

Fixes #19926 — when running inside tmux without `set -g allow-passthrough on`, the previous DCS-wrapped OSC 52 path silently dropped writes to the host terminal's clipboard. Codex would log success while the user's clipboard didn't actually change, which is a particularly nasty class of UX bug because the user has no way to discover the failure short of pasting and getting nothing. The fix introduces a tmux-native clipboard path that uses `tmux load-buffer` / `tmux set-buffer -w` (the implementation is below the visible diff window — code lives at `codex-rs/tui/src/clipboard_copy.rs:312+` based on the new `tmux_clipboard_copy` reference at line 49) and falls back to OSC 52 only when the tmux path is unavailable.

The refactor introduces a new `CopyEnvironment { ssh_session, wsl_session, tmux_session }` struct at `clipboard_copy.rs:88-92` to replace the prior two-bool parameter list, and a new `terminal_clipboard_copy_with` indirection at lines 161-180 that dispatches between tmux and OSC 52 based on `tmux_session`. This is the right shape — the prior signature was already at the bool-parameter-explosion threshold and adding a third positional bool would have been impossible to read at the call site. The error-message branching at lines 105-117, 134-153, and 165-179 is verbose but explicit: SSH-with-tmux, WSL-with-tmux, and local-with-tmux each produce distinct error strings naming both the tmux failure mode and the OSC 52 fallback failure mode, which is exactly what you want when triaging clipboard bug reports because the failure pattern tells you which layer to look at.

Two concerns. First, the error-message branching is repeated three times — the `if environment.tmux_session { ... } else { ... }` to format `"... ; terminal fallback: ..."` vs `"... ; OSC 52 fallback: ..."` could be a single helper that takes `tmux_session` and the inner error string, eliminating about 20 lines of duplication. Second, `is_tmux_session()` at line 388 checks `TMUX` and `TMUX_PANE` env vars, but if a user is in nested tmux (e.g. SSH'd into a remote tmux from local tmux) `TMUX` is set on both sides — the inner tmux's `tmux load-buffer` will write to the *remote* tmux's clipboard, which is the bug we just fixed for OSC 52, just one layer deeper. Worth a comment acknowledging the limitation, and ideally a check against `TMUX_PANE` containing the inner-tmux socket path to detect the nested case.

**Verdict: merge-after-nits** — extract the duplicated error-message branching into a helper, add a comment about the nested-tmux limitation (or implement detection), and confirm the `cargo test -p codex-tui` coverage actually exercises the SSH+tmux+OSC52-fallback path since the PR body lists "tmux-preferred, fallback, and combined-failure paths" without explicitly naming the SSH×tmux cross product.
