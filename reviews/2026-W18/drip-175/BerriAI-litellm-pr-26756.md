# BerriAI/litellm#26756 — fix(proxy): self-heal Prisma read paths + harden reconnect state machine

- URL: https://github.com/BerriAI/litellm/pull/26756
- Head SHA: `aa2ef4120098981cb6160d94347743e2e77ffb61`
- Files: 4 changed (+546 / -12); helper in `exception_handler.py` (+137), `utils.py` rewire (+31/-11), two new test files (+255 + +123)

Two related fixes layered on top of the existing Prisma reconnect plumbing — both genuinely important. The first restores the reconnect-and-retry-once branch on `PrismaClient.get_generic_data` that 1.83.x dropped relative to 1.82.6 (issue #25143); without it, a single `httpx.ReadError` flap during `_update_config_from_db`'s four-way `get_generic_data` fan-out fires four `db_exceptions` alerts and leaves a stale config window. Rather than inlining the pattern (the PR author correctly observes that `auth_checks._fetch_key_object_from_db_with_reconnect` and `_health_endpoints` already had two drifting inline copies), the fix factors it into a single canonical `call_with_db_reconnect_retry` helper at `litellm/proxy/db/exception_handler.py:142-261`. The helper's contract is well-thought-out: zero-arg `coro_factory` callable (so retry creates a fresh awaitable, avoiding `RuntimeError: cannot reuse already awaited coroutine`), explicit `reason` tag for telemetry, and most importantly the at-most-one-retry invariant baked into the control flow at lines 233-261 — no infinite loop possible by construction.

The exception-chaining at lines 247-255 is the right call — if `attempt_db_reconnect` itself raises (lock cancellation, internal failure), the helper preserves the *original* `first_exc` and chains the reconnect error as `__cause__`. Without this, `failure_handler` / `db_exceptions` alerts would surface the reconnect-internal error and mask the actual transport problem, making the bug harder to triage than before the fix. The `_coerce_timeout` helper at lines 117-125 is a defensive guard against tests that mock `prisma_client` and leave timeout slots as `MagicMock` instances — slightly ugly but pragmatic, and the alternative (relying on test setup discipline) has lost this race before in this file.

The second fix at `litellm/proxy/utils.py:4196-4233` is the genuinely high-value one: `_run_reconnect_cycle` was clearing `self._engine_confirmed_dead = False` *before* awaiting `_do_heavy_reconnect()`. If the heavy reconnect raised (timeout, missing `DATABASE_URL`, recreate failure), the flag was left cleared, and the next attempt could silently demote to the lightweight reconnect path even though the engine was genuinely dead. Moving the reset into the success branch at lines 4226-4232 (after `await asyncio.wait_for(_do_heavy_reconnect(), timeout=...)` returns successfully) closes a state-machine bug that was silently making a class of outage scenarios unrecoverable without a process restart. This is the kind of fix that should ship as its own PR with a CHANGELOG entry.

**Verdict: merge-after-nits** — split the two fixes into two PRs (the `_engine_confirmed_dead` flag-reset bug deserves its own change with a CHANGELOG note since it changes recovery semantics in the failure case); follow up with the auth + health inline-copy migrations the PR body promises so the canonical helper has three call sites instead of one (otherwise it's just an indirection layer); and add a property test asserting that across N random sequences of `(transport_error, reconnect_outcome)` the helper never makes more than 2 calls to `coro_factory` total — that's the load-bearing invariant and it's currently only covered by case-by-case unit tests.
