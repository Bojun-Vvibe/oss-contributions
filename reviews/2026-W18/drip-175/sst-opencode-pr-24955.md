# sst/opencode#24955 — fix(mcp): detect transport close and update status to failed

- URL: https://github.com/sst/opencode/pull/24955
- Head SHA: `5c8a76a1114fa38a745e5e95b3efff6f4bf92aec`
- Files: 1 changed (+13 / -0); single file `packages/opencode/src/mcp/index.ts`

Tight, surgical bug fix for #23997 — MCP servers were appearing as `connected` (green) in the TUI/API even after their underlying transport had closed unexpectedly (server crash, network drop, process exit). Root cause is that `client.onclose` from the MCP SDK was never wired to anything, so the local connection-status dictionary `s.status[name]` only ever transitioned `connected → connected` and never `connected → failed`. The fix introduces a `dropConnection(s, name, client, reason)` helper at `packages/opencode/src/mcp/index.ts:475-482` that atomically (a) guards against stale closures via the `s.clients[name] !== client || s.status[name]?.status !== "connected"` check, (b) flips the status to `{ status: "failed", error: reason }`, (c) deletes the client and cached tool defs, and (d) fires `client.close()` for cleanup. The `onclose` hook at lines 497-499 then forks the helper through `bridge.fork(Effect.sync(...))` so it runs in the right Effect runtime context.

The stale-closure guard at line 476 is the load-bearing line and the right call — without it, a fast reconnect/disconnect/reconnect cycle could race where the *new* client gets stomped by the *old* client's `onclose` arriving late. By comparing identity (`s.clients[name] !== client`) rather than just checking the status slot, this correctly no-ops when the slot has already been replaced. This is the exact pattern used elsewhere in the file for the `setNotificationHandler` callbacks and is consistent with the codebase's existing `EffectBridge` discipline.

Two things I'd want before stamping. First, there's no test — for a 13-line fix that's already been validated locally per the PR checklist this is forgivable, but the stale-closure guard at line 476 is precisely the kind of invariant that quietly regresses three refactors later. A `mcp/index.test.ts` case that constructs two clients sharing a name, calls `onclose` on the *first* one after the second has been installed, and asserts the second is untouched would lock that contract. Second, `void client.close()` at line 481 fires-and-forgets a Promise — if `close()` rejects (transport already torn down, file descriptor already returned), the rejection becomes an `UnhandledPromiseRejectionWarning` rather than going through the established logger. Either `await` it inside an `Effect.tryPromise` wrapper, or `.catch(() => {})` it explicitly with a debug log.

**Verdict: merge-after-nits** — add the stale-closure regression test and harden the `void client.close()` call with an explicit catch; both are 5-minute follow-ups that would land cleanly on this same diff.
