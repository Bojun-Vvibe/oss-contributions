# block/goose#8900 — fix: model picker stays usable during provider loading

UX fix for the goose2 frontend — the model picker was flashing "Loading" on startup, becoming temporarily unclickable, and refusing to refresh discovered providers without an app restart. Net deletion overall (+118/-99), which is unusual for a UX fix and is a strong signal that the PR is doing the right thing by *removing* the duplicated startup logic rather than papering over the symptoms. The big consolidation is in `useAppStartup.ts:7-10, 81-129` where the inline `refreshConfiguredProviderInventory` (60+ lines with its own `INVENTORY_POLL_DELAYS_MS` constant and `sleep` helper) gets extracted into a new `backgroundRefreshInventory(store, initialEntries?)` function in `inventory.ts` — that function is then reused by both the startup path and the new picker-open refresh path, eliminating the prior two-implementations-of-the-same-loop drift risk.

The button-label priority reorder at `AgentModelPicker.tsx` (the diff shows `triggerModelLabel ?? selectedAgentLabel` becoming the primary, with "Loading" only as a tertiary fallback) is the correct fix for the flash — the prior code displayed "Loading" whenever `loading` was true, regardless of whether we already had a cached provider name from a previous session. The conditional disable (`loading && !selectedAgentLabel`) at the same site is also right: the button should only be unclickable when we genuinely have nothing to show, not during a background refresh that we can keep working through. The `handlePickerOpen` callback at `useAgentModelPickerState.ts:166-182` uses a `refreshingRef` ref to coalesce concurrent picker-open events — that's the right primitive (a `useState` would cause re-renders and lose the no-op semantics; a module-level lock would leak across hook instances). The `useProviderInventoryStore.getState().loading` guard at line 167 ensures we don't pile a background refresh on top of an in-flight foreground load.

The prop-threading from `handlePickerOpen` through `useResolvedAgentModelPicker → useChatSessionController → ChatView/HomeScreen → ChatInput → ChatInputToolbar → AgentModelPicker.onOpen` is admittedly ugly (six layers of pass-through) but it's idiomatic for this codebase per the surrounding context — the goose2 frontend doesn't use a global event bus or context for these UI signals, and introducing one for this single callback would be the wrong scope expansion. The dynamic `import("@/features/providers/api/inventory")` at `useAgentModelPickerState.ts:171-181` and `useAppStartup.ts:99` is a reasonable code-splitting choice if `inventory.ts` pulls in heavy dependencies, but if not it's just adding a microtask boundary for no reason — worth confirming.

**Verdict: merge-after-nits** — confirm the dynamic `import("@/features/providers/api/inventory")` is actually buying code-splitting (otherwise just import statically, the indirection costs more than it saves); add a test for `handlePickerOpen` that asserts (a) two synchronous opens result in one refresh call, (b) `loading=true` short-circuits the refresh, and (c) the `refreshingRef` is reset even when `backgroundRefreshInventory` rejects; and consider adding a small visual indicator in the picker dropdown that a background refresh is in flight so users understand why the model list might update under them.
