# BerriAI/litellm#26779 — fix(proxy) add support setting budget_duration on individual member budgets

- URL: https://github.com/BerriAI/litellm/pull/26779
- Head SHA: `cd3a3623efcbc0fafc9e9f21d86095f61f1614fb`
- Files: 5 changed (+343 / -15); pydantic types in `_types.py`, helper in `common_utils.py`, endpoint in `team_endpoints.py`, two test files

Fixes #25509 by closing a gap where team-member budgets accepted `max_budget`/`tpm_limit`/`rpm_limit` but had no way to express the reset cadence (`30d`, `1mo`, etc.), making them effectively lifetime caps even when the parent team had a `team_member_budget_duration`. The PR adds an optional `budget_duration: Optional[str]` field to `TeamMemberUpdateRequest` at `litellm/proxy/_types.py:3863-3868` with the documented "pass null for explicit lifetime, omit to inherit team default" tri-state contract — that's the right shape for "add a new optional field to an existing endpoint without breaking pre-existing PUT bodies."

The inheritance logic at `litellm/proxy/management_endpoints/team_endpoints.py:2729-2739` uses `data.model_fields_set` to distinguish "explicitly set to None" (use null = lifetime) from "field not present in payload" (inherit team default), which is the only correct way to do this in pydantic v2 — a plain `if data.budget_duration is None` would conflate the two cases and silently override an explicit lifetime with the team default. This is a subtle invariant and the test at `tests/test_litellm/proxy/test_team_member_update.py` (+209) presumably locks it but I'd want explicit confirmation that there's a test asserting `PUT {budget_duration: null}` produces a budget with `budget_duration=None` while `PUT {}` produces one with the team's inherited value. The `_upsert_budget_and_membership` helper at `litellm/proxy/management_endpoints/common_utils.py:357-475` correctly threads the new param through both the create path (lines 461-473) and the update path (lines 415-420), and crucially calls `get_budget_reset_time(budget_duration=...)` at both sites so `budget_reset_at` stays consistent with `budget_duration` — a bug here would mean the budget shows "30d" but never actually resets.

The "disconnect when all None" guard at lines 384-389 was correctly extended to include `budget_duration is None`, otherwise a `PUT {budget_duration: "30d"}` with no other fields would have hit the disconnect branch and silently deleted the membership budget. The new test `test_upsert_budget_duration_only_creates_budget` at lines ~261+ in `test_upsert_budget_membership.py` locks exactly this case, which is excellent — that's the test that catches the next person who refactors the disconnect guard.

**Verdict: merge-after-nits** — add an integration test for the tri-state distinction (omitted vs explicit-null vs explicit-string) at the endpoint layer not just the helper layer, since `model_fields_set` semantics are easy to break in pydantic config tweaks; document the inheritance behavior in the OpenAPI spec / docs page so users know the team default applies; and confirm the cherry-picked `81518b7` from #25560 cleanly handles the case where `team_default_budget_id` exists but `_team_budget_row` is None (the `if _team_budget_row else None` ternary at line 2735 silently swallows a "team has stale default budget pointer" misconfiguration that should probably warn).
