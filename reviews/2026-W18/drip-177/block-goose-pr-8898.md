## Reasoning

For a release PR, the bar is "does the version bump match the branch and
do the cherry-picks line up with what's in the changelog?" The version-bump
mechanics here look mechanically correct. The concerning piece is the
`canonical_models.json` change being included in a release PR — release
PRs should ideally be pure metadata so the audit trail of "what shipped in
1.33.2" is just version bumps + the cherry-picked commits. A 700-line model
inventory edit landing in the same PR as the version bump means
- ops debugging "why did this model stop resolving in 1.33.2?" will land
  on the release commit instead of the PR that actually removed the model.
- if the cherry-picks themselves removed those model entries, that's fine
  (the diff is just the cherry-pick rolling forward); if instead the model
  removal was authored *into* the release branch separately, it has no
  associated review.

Given this is an automated bot PR and the model inventory change is the
substantive part, the reviewer's job is:

1. Confirm the model-inventory deletions trace back to specific
   cherry-picks (`git log release/1.33.2 -- crates/goose/src/providers/canonical/data/canonical_models.json`)
   rather than uncommitted edits to the release branch.
2. Confirm `CHANGELOG.md` (presumably one of the unshown files) calls out
   the canonical-model removal — users with workflows pinned to
   `aihubmix/Kimi-K2` or other dropped IDs deserve a heads-up.
3. Confirm no version-bump artifact got missed — `goose-bench` or other
   non-published crates that opt out of `workspace.version` would silently
   stay on 1.33.1.

The release-script-driven shape of the PR (per the body's "Push the release
tag to trigger the release" instructions) means there's no real "code" to
review here — it's a process gate.

## Verdict

**merge-after-nits** — before tag, run `git log v1.33.1..release/1.33.2
-- crates/goose/src/providers/canonical/data/canonical_models.json` and
confirm every line removed in this PR's `canonical_models.json` delta
maps to an explicit cherry-picked commit (otherwise that 700-line removal
has no per-change author attribution and ops can't bisect a "model
suddenly missing" report back to a single PR); add a CHANGELOG line for
the canonical-model removal so users pinned to `aihubmix/Kimi-K2` or other
dropped IDs aren't surprised by 1.33.2 silently breaking their config;
confirm any non-publish workspace crates (e.g. internal `goose-bench`)
that opt out of `version.workspace = true` are handled correctly by the
release tooling — a missed crate would ship as a stale 1.33.1 alongside
the 1.33.2 sibling crates.
