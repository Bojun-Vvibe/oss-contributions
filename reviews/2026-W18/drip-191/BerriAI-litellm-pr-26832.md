# BerriAI/litellm #26832 — fix(mcp): forward extra_headers for OpenAPI-backed MCP servers

- **URL:** https://github.com/BerriAI/litellm/pull/26832
- **Head SHA:** `9dac38e952237380abf20980cd382416d9ee1500`
- **Files:** `litellm/proxy/_experimental/mcp_server/mcp_server_manager.py` (+38/-15), `litellm/proxy/_experimental/mcp_server/openapi_to_mcp_generator.py` (+12/-0), `litellm/proxy/_experimental/mcp_server/server.py` (+18/-0)
- **Verdict:** `merge-after-nits`

## What changed

Closes a parity gap where SSE/HTTP MCP transports honored `server.extra_headers` (forwarding selected inbound headers to the upstream tool server) and `pre_mcp_call` hook-injected `extra_headers` (e.g. `MCPJWTSigner`), but the OpenAPI-backed transport silently dropped both. The previous code at `mcp_server_manager.py:2509-2516` (deleted) explicitly logged a warning saying "header injection is not supported for OpenAPI servers; headers will be ignored. Use SSE/HTTP transport to enable hook header injection." — a real functional gap that forced MCP-via-OpenAPI users to either fork the transport or disable header-bound auth.

Fix is a `contextvars.ContextVar`-based per-request override:

1. **New `_request_extra_headers` ContextVar** at `openapi_to_mcp_generator.py:34-39`:
   ```python
   _request_extra_headers: contextvars.ContextVar[Optional[Dict[str, str]]] = (
       contextvars.ContextVar("_request_extra_headers", default=None)
   )
   ```
   Sits parallel to the existing `_request_auth_header` ContextVar at `:30-32`.

2. **Tool handler injection** at `openapi_to_mcp_generator.py:325-329`:
   ```python
   effective_headers = dict(headers)
   request_extra = _request_extra_headers.get()
   if request_extra:
       effective_headers.update(request_extra)
   override_auth = _request_auth_header.get()
   if override_auth:
       effective_headers["Authorization"] = override_auth
   ```
   Per-request extras applied *before* the Authorization override, so `Authorization` from the auth ContextVar wins if both are set — the exec-time priority is `auth-override > extras > base`.

3. **Two call-site wirings:**
   - `_call_openapi_tool_handler` at `mcp_server_manager.py:2079-2105`. The `raw_headers`-from-inbound-request path normalizes header keys to lowercase (`{str(k).lower(): v ...}`), then iterates `server.extra_headers` to extract each by case-insensitive lookup, building `extra_headers_dict`. Then merges `hook_extra_headers` (from `pre_mcp_call` hooks) on top with `extra_headers_dict.update(hook_extra_headers)` at `:2099-2103`. Sets the ContextVar via `_extra_token = _request_extra_headers.set(extra_headers_dict)` at `:2104` and resets in `finally` at `:2120`.
   - `server.py:execute_mcp_tool` at `:2127-2148` — does the same allowlist-extraction shape inline for the local-tool path (`_handle_local_mcp_tool`), wrapping the call with the same `_request_auth_header.set` / `_request_extra_headers.set` / `try / finally / reset` pair.

4. **Wiring the call args** at `mcp_server_manager.py:2547-2553`: the warning-and-drop branch is replaced with passing `raw_headers=raw_headers, hook_extra_headers=hook_result.get("extra_headers")` to `_call_openapi_tool_handler`.

## Why it's right

- **Closes the parity gap honestly.** SSE/HTTP transports already do this header-allowlist-then-merge dance; making OpenAPI follow the same shape is the minimum-surprise fix. The previous warning-and-drop was correct behavior given the constraint, but the constraint was a missed wiring, not a fundamental incompatibility.
- **ContextVar is the right abstraction here.** `tool.handler` at `:2087` is a closure compiled at OpenAPI-spec-load time — it doesn't take a `raw_headers` parameter, can't be retroactively widened without breaking the autogenerated MCP tool surface, and runs across an `await` so thread-locals would lose context. ContextVars are the canonical Python solution for this exact "I need to thread per-request data through a closure I can't refactor" problem, and the codebase already uses the pattern at `:30-32` for `_request_auth_header`. Symmetric extension.
- **`finally` reset is correct.** Both call sites pair `_extra_token = _request_extra_headers.set(...)` with `_request_extra_headers.reset(_extra_token)` in `finally` (`mcp_server_manager.py:2104-2122` and `server.py:2143-2148`). If the handler raises, the ContextVar still resets — no leak across requests.
- **Auth-override priority is preserved.** `effective_headers.update(request_extra)` runs before the `Authorization` set from `_request_auth_header.get()` at `:328-329`, so a user that has both an `Authorization` extra-header and an OAuth2 auth-override gets the override winning. This matches the SSE/HTTP transport priority and prevents accidental auth-stripping via misconfigured `extra_headers`.
- **Header-name preservation.** The normalize-to-lowercase step at `:2086-2090` and `server.py:2129-2133` is *only* for the case-insensitive lookup; the value is stored back under the *configured* `header` casing (`extra_headers_dict[header] = val` at `:2095` and `server.py:2138`). This means an upstream API expecting `X-Tenant-ID` (mixed-case) gets exactly that, even if the inbound request used `x-tenant-id`.

## Nits / blockers

1. **Lazy import in a hot path.** `from litellm.proxy._experimental.mcp_server.openapi_to_mcp_generator import _request_extra_headers` at `mcp_server_manager.py:2065-2067` runs *every* `_call_openapi_tool_handler` invocation. Python caches the import, so the cost is one dict lookup, but it sits inside the call boundary and obscures the dependency. Lifting to module-top would clarify the dependency graph and shave a microbenchmark hair. The existing top-level imports in this file already include other `_experimental.mcp_server.*` modules, so circular-import risk is presumably already-resolved.

2. **Two near-identical extraction blocks.** The header-allowlist-then-extract logic is duplicated verbatim between `mcp_server_manager.py:2086-2096` and `server.py:2128-2140`. If `server.extra_headers` ever needs another transform (e.g. tenant-prefix stripping, or excluding hop-by-hop headers like `Connection`/`Keep-Alive`), there are now two places to update. A `_build_extra_headers_dict(server, raw_headers)` helper in a shared module would be a one-PR follow-up.

3. **No regression test in the diff slice.** The PR ships zero test changes despite touching three production files and closing a documented behavior gap. A test that asserts `pre_mcp_call` hook-injected `extra_headers` actually reach an OpenAPI-backed handler's outgoing HTTP request (e.g. via `respx` / `httpx_mock`) would lock the contract — without one, a future refactor that drops the ContextVar set/reset (or moves it outside the `try`) silently regresses without any test failing.

4. **Hook-injected headers can override allowlisted inbound headers.** `extra_headers_dict.update(hook_extra_headers)` at `:2099-2103` lets `pre_mcp_call` hooks shadow user-allowlisted inbound headers. That's correct for the `MCPJWTSigner` case (hooks should be authoritative), but worth a one-line comment explaining the precedence — operators reading this will reasonably wonder whether `server.extra_headers` is the floor or the ceiling.

5. **`tool.handler(**arguments)` race.** The `_extra_token = _request_extra_headers.set(extra_headers_dict)` at `:2104` is set before the `await tool.handler(**arguments)` at `:2107`. Since `tool.handler` is async and `_request_extra_headers.get()` is read inside it, ContextVar copy-on-task-spawn semantics mean if `tool.handler` does `asyncio.create_task(...)` internally without explicit context propagation, the spawned task sees the *parent* context's `_request_extra_headers` value — which is correct. But if `tool.handler` uses a thread-pool offload (e.g. `loop.run_in_executor` for blocking I/O), the offloaded callable sees `None`. The current OpenAPI handler appears to use `httpx` async, but the assumption is load-bearing and worth a comment.

## Risk

Low. Pure additive change — the previous warning-and-drop path is replaced with a propagate-and-merge path. The `try / finally / reset` pattern at both call sites prevents ContextVar leakage. The Authorization-override priority is unchanged (verified at `:328-329`). Worst-case if `extra_headers_dict` grows large or contains malformed values, the upstream HTTP request errors visibly and the existing OpenAPI error path at `:2117-2123` returns it as a `CallToolResult` with `isError=True`. The biggest *operational* risk is a misconfigured `server.extra_headers` allowlist accidentally exposing inbound auth headers (e.g. `Cookie`, `X-Forwarded-User`) to a third-party tool server — but that's a config-time risk that exists for SSE/HTTP transports today, this PR just makes OpenAPI behaviorally consistent.
