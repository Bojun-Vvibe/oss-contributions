## What this does

Reintroduces a generated Python SDK at `packages/sdk/python/`, produced from
`packages/sdk/openapi.json` via `openapi-python-client` plus a hand-written
overlay (`scripts/_overlay/server.py`, `__init__.py`). Adds a regen check
workflow at `.github/workflows/python-sdk.yml` so any PR touching
`packages/sdk/openapi.json` or the Python tree must run
`python scripts/generate.py --check` and have it report clean. Hand-written
surface is just `__init__.py`, `server.py` (Python mirror of the JS
`createOpencodeServer` — sync + async), `pyproject.toml`, README, the regen
script, and the overlay package. Everything under
`packages/sdk/python/src/opencode_ai/api/default/*.py` and
`packages/sdk/python/src/opencode_ai/models/*.py` is regenerated.

## Why this is the right shape

The previous Python SDK got reverted in `13b2cf50` precisely because it had no
regen CI and silently drifted from the spec. This PR ships the SDK and the
drift detector in the same commit, which is the correct ordering. The
`scripts/generate.py --check` mode lets contributors regenerate locally and
makes the CI job a pure no-op verification, which is the cheapest possible
pre-merge signal.

## What's worth flagging

1. **593 files in one PR.** Even though most are generated, this makes
   per-file review pointless and the diff itself uncacheable on the GH side
   (`gh pr diff` 406s with "exceeded the maximum number of files (300)"). I
   could only review the manifest (file list) plus the hand-written overlay
   surface — see `packages/sdk/python/scripts/_overlay/server.py` (228 LOC),
   `packages/sdk/python/scripts/generate.py` (128 LOC), and
   `packages/sdk/python/pyproject.toml` (78 LOC). Suggest landing the
   generator + workflow + skeleton overlay in one PR and the generated tree
   in a follow-up auto-regen PR signed by the bot, so future-archaeology can
   tell what was hand-edited.

2. **Regen workflow trigger scope.** `python-sdk.yml` (83 LOC) needs to fire
   on at least: `packages/sdk/openapi.json`, `packages/sdk/python/**`, and
   `.github/workflows/python-sdk.yml` itself. If it only watches
   `packages/sdk/python/**` then a spec change without regen will silently
   pass CI on the spec-only PR and only fail later. Worth confirming the
   `paths:` filter in the workflow.

3. **Overlay/generated boundary.** The overlay pattern
   (`scripts/_overlay/__init__.py`, `scripts/_overlay/server.py`) is the
   correct shape, but a `README.md` note in `packages/sdk/python/` calling
   out "files under `src/opencode_ai/api/` and `src/opencode_ai/models/` are
   generated, do not edit" would save a future contributor an afternoon. The
   PR's README addition is 104 LOC; worth grepping it for that warning.

4. **Smoke tests against live `opencode serve`.** The PR description says
   `pytest tests/` runs 4 smoke tests against a spawned server. Suggest
   making sure those are in a separate `pytest -m smoke` mark and the CI
   matrix runs them on at least one Python version, so Bun availability in
   CI doesn't silently skip them.

## Verdict

**`merge-after-nits`** — the shape is right and the regen-CI-with-SDK-pairing
is exactly the lesson from the prior revert, but a 593-file PR for review
is genuinely hard, and the workflow `paths:` filter + overlay README warning
are worth confirming before merge.

## Nits

- Consider splitting into two commits within the PR for reviewer sanity:
  "skeleton + generator + workflow" then "generated artifacts".
- Verify `python-sdk.yml` fires on `packages/sdk/openapi.json` changes too,
  not just changes inside the Python tree.
- Add a top-of-file generated-warning header in the regenerated modules
  (`# DO NOT EDIT. Generated by scripts/generate.py from openapi.json`) so
  drift attempts in IDEs are immediately obvious.
