## Reasoning

The bug being fixed is concrete and high-impact: schemars 1.x emits `Option<T>` as either `type: ["T", "null"]` or `anyOf: [T, {type: "null"}]`, and Vertex Gemini (when proxied through Bifrost) rejects both forms — the former because Gemini's schema dialect doesn't allow array-typed `type`, the latter because nullable unions in `anyOf` aren't supported. The runtime symptom is "your tool call schema is invalid" coming back from the upstream, which is non-obvious to debug from the user side. Normalizing at the OpenAI-format layer is the right place: it sits on the path from goose's internal tool definitions to the wire format that providers (including the Bifrost-routed ones) consume.

The semantic argument the PR body makes — "optional-ness is already conveyed by the field being absent from `required`" — is correct and is exactly why dropping the null variant is safe. JSON Schema considers a field optional if it's not in the parent's `required` array, regardless of whether its `type` includes `null`. So `{required: ["command"], properties: {timeout_secs: {type: "integer"}}}` is semantically equivalent to `{required: ["command"], properties: {timeout_secs: {type: ["integer", "null"]}}}` for any conforming validator.

The two nits worth raising: (1) the `*schema = replacement;` at line 760 in the `anyOf` branch wipes sibling keys. If a future schemars version emits `{description: "...", anyOf: [T, {type: "null"}]}` the description is lost. The fix is one extra step — copy non-`anyOf` keys from the original `obj` onto the unwrapped variant before assigning. Worth a defensive patch even if no current input triggers it. (2) The function is non-recursive on the `anyOf` branch: if the unwrapped variant `T` is itself a nullable union (rare but possible in deeply nested schemars output), it won't be re-normalized. The outer `ensure_valid_json_schema` recursion handles `type == "object"` descent, so deeply-nested object cases are covered, but a hypothetical `Option<Option<i64>>` would not be. Both nits are theoretical with current schemars output, but worth one defensive commit. Approve and merge once the description-preservation case is addressed (or explicitly punted to a follow-up).
