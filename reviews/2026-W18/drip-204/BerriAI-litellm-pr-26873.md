## Reasoning

The change closes a real gap in the OpenAPI-to-MCP bridge: a customer-configured `extra_headers` list on `MCPServer` is *declared* but had no transport plumbing into the per-request `tool_function` invocation. The fix uses ContextVar — same mechanism as the existing `_request_auth_header` — so the integration cost is one new ContextVar and a tightly scoped set/reset around the local-tool dispatch. That's the right level of abstraction for per-request, async-safe state in a Python web stack.

The case-insensitivity at `server.py:2042-2046` is necessary because incoming HTTP headers are case-insensitive by spec but raw dict iteration in some frameworks preserves the wire casing. Lowering both sides for the lookup, then storing under the *configured* name (`header_name`, not the lowered form), means downstream HTTP libraries see the canonical casing the operator declared. That's correct and worth the comment I called out above so it isn't "optimized" away.

Two minor things that keep this off `merge-as-is`: (1) the `extra_headers` list comes straight from `MCPServer` config without a denylist for sensitive headers (`Cookie`, `Set-Cookie`, `Host`, hop-by-hop headers like `Connection`, `Keep-Alive`, `Transfer-Encoding`). If an operator misconfigures and forwards `Cookie`, the upstream API gets the proxy's session cookie. Probably out of scope here, but worth filing a follow-up issue. (2) The tests cover correctness but not thread/async-loop isolation — a `pytest-asyncio` test that fires two concurrent `tool_function` calls with different ContextVar values would catch any future refactor that accidentally promotes the ContextVar to a module global. Both nits are non-blocking; the patch is solid.
