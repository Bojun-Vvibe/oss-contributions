# BerriAI/litellm PR #26853 — fix(anthropic): claude code v2.1.100+ thinking-block stream rejection

- PR: https://github.com/BerriAI/litellm/pull/26853
- Head SHA: `f92c73dbc7fe6b65ff533c4a92ba698d69dc73c1`
- Files touched: `litellm/llms/anthropic/experimental_pass_through/adapters/streaming_iterator.py` (both `__next__` sync and `__anext__` async paths) plus a regression suite covering text-only, thinking-only, tool-use, and text-then-tool-use streams. Net +670/-38.

## Specific citations

- The bug is exactly at `litellm/llms/anthropic/experimental_pass_through/adapters/streaming_iterator.py:107-114` (pre-PR shape). The first-chunk emission unconditionally pushed `{"type": "content_block_start", "index": ..., "content_block": {"type": "text", "text": ""}}` regardless of what the underlying model would actually emit first. When the model returned a thinking block first (Anthropic's claude-code v2.1.100+ default with extended thinking on), the stream then tried to deliver a `thinking_delta` against a `content_block_start` that had been declared as `text` — and Anthropic's client-side validator at v2.1.100+ rejects this with `"Content block is not a text block"`, severing the stream.
- The fix at `:108-141` is a peek-then-emit pattern: iterate the first real chunk via `for first_chunk in self.completion_stream`, call `self._should_start_new_content_block(first_chunk)` to set `self.current_content_block_start` to whatever shape the chunk implies (text / thinking / tool_use), then emit `{"type": "content_block_start", ..., "content_block": self.current_content_block_start}` carrying the correct type. The translated first delta is held on `self.holding_chunk` until the first `content_block_delta` flush at `:163-167` so it lands inside the block rather than after the next `content_block_stop`. The `__anext__` mirror at `:281-310` is the exact same shape with `async for`.
- The empty-`input_json_delta` guard at `:120-129` (and the async mirror at `:289-298`) is a real subtlety — OpenAI-style tool_use start chunks carry `arguments=""` which translates to a `partial_json=""` `input_json_delta` that should not be emitted (Anthropic clients reject empty input_json_delta). The guard sets `self.holding_chunk = None` for that specific shape so the empty delta is discarded rather than buffered.
- The regression suite locks the headline contract at multiple shapes: `test_content_block_start_is_text` (text-first), `test_content_block_stop_between_thinking_and_text` (the `cbs_indices`/`stop_indices` ordering invariant — first stop must come between the two starts), `test_full_thinking_event_sequence` (full message_start → content_block_start(thinking) → ... → message_stop), and the tool_use shapes `test_content_block_start_is_tool_use` / `test_tool_use_deltas_have_correct_type` / `test_text_then_tool_use`. The async mirror class `TestAsyncNonThinkingTextStream` exercises the `__anext__` path with the same shapes.

## Verdict: merge-as-is

## Rationale

This is a real money-on-the-table fix — claude-code v2.1.100+ with extended thinking is the default for a large slice of the Claude Code user base, so any upstream-Anthropic-shaped stream that ran through the litellm pass-through adapter was failing for those users with `"Content block is not a text block"` and severing mid-conversation. The pre-PR diagnosis is precisely correct: the unconditional `{"content_block": {"type": "text", "text": ""}}` emission at the first-chunk site was a hardcoded assumption from the era when streams started with text, and it became a bug the moment thinking blocks could lead. The peek-then-emit fix is the minimum surface that resolves it — there's no smaller change that could correctly type the first `content_block_start`, because the type information lives in the first chunk, not in the stream metadata. The empty-`input_json_delta` guard at `:120-129` is the kind of subtle adjacent fix that only surfaces once you start peeking at the first chunk; without it, the peek would synthesize a `partial_json=""` delta that itself would be rejected, trading one bug for another. The regression suite is the right shape: it locks the per-block-type `content_block_start` invariant (text / thinking / tool_use), the ordering invariant (first `content_block_stop` between the two `content_block_start`s when thinking precedes text), and the async-path parity. The `__next__` and `__anext__` paths are kept in sync by literal duplication rather than extraction — that's a known maintainability nit (a helper coroutine peek-and-translate would let one path call the other), but extracting it now would balloon the diff and obscure the headline fix; that refactor belongs in a follow-up. Worth confirming pre-merge: the regression doesn't appear to lock the case where the model returns a thinking block followed immediately by a tool_use (skipping text entirely) — if that's a shape claude can produce, a `test_thinking_then_tool_use` case would close the last gap.
