## Specific citations

- The headline cross-trust fix is at `litellm/proxy/common_utils/callback_utils.py:419-435`. Pre-PR, `get_logging_caching_headers` would forward any `pillar_response_headers` dict from `_metadata` straight onto outbound response headers — meaning a request that injected `pillar_response_headers: {"x-admin-token": "..."}` into a metadata field via a callback or upstream proxy would echo it back to the caller. Post-PR, the forward is gated by a new `is_trusted_pillar_metadata = _metadata.get(TRUSTED_PILLAR_RESPONSE_HEADERS_METADATA_KEY) is True` check, plus a per-key `key.lower().startswith("x-pillar-")` filter so even a trusted pillar can only emit headers in the `x-pillar-*` namespace. The companion mark on the producer side is at `litellm/proxy/guardrails/guardrail_hooks/pillar/pillar.py:147` (`metadata_store[TRUSTED_PILLAR_RESPONSE_HEADERS_METADATA_KEY] = True`) — this is the only call site that legitimately sets the trust flag.
- The Bedrock-guardrails dynamic-body denylist is at `litellm/proxy/guardrails/guardrail_hooks/bedrock_guardrails.py:74` (`_BEDROCK_DYNAMIC_BODY_DENYLIST = frozenset({"content", "source"})`) and applied at `:414-428`. Pre-PR, `bedrock_request_data.update(self.get_guardrail_dynamic_request_body_params(...))` would let user-controlled dynamic params clobber `content`/`source` (the actual guardrail input fields) — a request could effectively make Bedrock evaluate a different payload than the one the proxy just authorized. Post-PR, those two keys are filtered out at `update` time.
- The audit-log change-by extraction at `litellm/proxy/hooks/key_management_event_hooks.py:43-69` (and again for the key-update hook at `:105+`) replaces the inline `litellm_changed_by or user_api_key_dict.user_id or litellm_proxy_admin_name` triple-fallback with a centralized `get_audit_log_changed_by(...)` helper — the helper is presumably the canonical resolution rule (single source of truth for "who did this") rather than three call sites all reimplementing it.

## Verdict: merge-as-is

## Rationale

This is a real cross-trust-boundary tightening with three distinct fixes bundled in one PR — and each one is small, surgical, and locked behind named constants (`TRUSTED_PILLAR_RESPONSE_HEADERS_METADATA_KEY`, `_BEDROCK_DYNAMIC_BODY_DENYLIST`) rather than magic strings. The pillar header-leak fix is the most consequential: pre-PR, any code path that could write into the request `_metadata` dict could synthesize arbitrary outbound response headers, which is exactly the kind of cross-customer leak primitive that turns into a CVE; post-PR, the producer-side trust marker at `pillar.py:147` is the only legitimate setter, the forwarder narrows to the `x-pillar-*` prefix, and both halves are necessary for the fix (a single-side fix would be defeated by the other half). The Bedrock-guardrails denylist closes a guardrail-bypass — the pre-PR shape let dynamic body params overwrite the very fields the guardrail evaluates, which means a guardrail-cleared request could still send a different payload to Bedrock than the one the proxy approved; the `frozenset({"content", "source"})` filter is the minimum surface that closes that bypass without breaking legitimate dynamic-body params. The `get_audit_log_changed_by` extraction is housekeeping but the right kind — when the change-by resolution rule is duplicated across multiple audit-log writers, one of them eventually drifts. The 734/-147 size includes a substantial test surface in the redact-messages refactor (the `_redact_responses_api_output_dict` / `_redact_model_response_dict_choices` helpers at `:79-105` and `:142-162` of that file) so the headline behavior changes ride alongside parity locks.
