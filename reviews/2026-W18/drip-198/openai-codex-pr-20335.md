## Verdict: merge-after-nits

## Rationale

Solid hook-bus extension landing the same shape that `function` and `mcp` handlers already use, with the right test surface (839-line integration suite plus the two `post_tool_use_response` body-shape unit tests). The `String → HookToolName` flip at `mcp_tool_call.rs:93` is exactly the kind of named-newtype tightening the PR series has been pushing (see drip-196 #20321 trust metadata, drip-197 #20336 powershell unwrap), and the new `PermissionRequestPayload::new(...)` constructor centralizes a struct-literal init that was duplicated across at least three callers — net positive for the next change that needs to add a field to that payload. Three nits worth surfacing before merge: (1) `dynamic_tool_post_tool_use_response` at `:108-122` returns `Err(FunctionCallError::RespondToModel(...))` on serialize failure, but `post_tool_use_payload` at `:46-54` swallows the error via `.ok()?` and silently emits no payload — a `tracing::warn!` on the discarded error path would prevent silent hook-bus drops; (2) the dynamic-tool naming convention `dynamic__<app>__<tool>` is now load-bearing across hook policies but lives only as a literal in the new test rather than as a documented constant in `hook_names.rs` — worth a `pub const DYNAMIC_TOOL_PREFIX = "dynamic__"` plus a one-paragraph rustdoc on `HookToolName::for_function_tool`; (3) the parity check between `mcp.rs:32` (`for_function_tool`) and the prior `HookToolName::new(invocation.tool_name.display())` shape is a wire-format change for any consumer that grepped the previous form — a CHANGELOG entry calling out "MCP hook tool names now go through `for_function_tool`" would close the migration loop.
