## Specific citations

- The behavioral change is exactly one literal at `packages/core/src/tools/grep.ts:468`: the `system grep` strategy's argument vector goes from `['-r', '-n', '-H', '-E', '-I']` to `['-r', '-n', '-H', '-E', '-I', '-i']`. The `-i` flag is the GNU/BSD grep "ignore case" switch, so a model-issued `pattern: "HELLO"` will now match `hello world` in the file body. The change is bracketed inside the `if (grepAvailable)` branch at `:466`, so it only applies when the system-grep strategy is selected (i.e., when the `git grep` strategy was rejected by `isCommandAvailable('git')` upstream).
- The regression at `packages/core/src/tools/grep.test.ts:317-345` constructs a `GrepToolParams` with `pattern: 'HELLO'` and force-routes the strategy to `system grep` by mocking `invocation.isCommandAvailable` to return `true` for `grep` and `false` for `git`. The post-condition is `expect(execStreaming).toHaveBeenCalledWith('grep', expect.arrayContaining(['-i']), ...)` which locks the `-i` presence at the `execStreaming` boundary rather than at the assembled-args level — that's the right granularity, since it's robust to the order of flags in the array.

## Verdict: merge-after-nits

## Rationale

This is a 1-character source change that closes a real consistency gap: the git-grep strategy (the preferred path when `git` is available) had been case-insensitive for some time, while the system-grep fallback (used in non-git workdirs and on systems without git) was case-sensitive. The agent-facing contract of "GrepTool finds the same results regardless of which strategy fires" was therefore silently broken depending on whether the user's cwd was a git repo. The fix is correct and minimal. Three nits before merge: (1) the agent-facing tool description in `grep.ts` (above the `class GrepToolInvocation` definition, not in this diff) likely doesn't mention case-insensitive matching as a contract, so a model that wants exact-case matching has no escape hatch — worth either adding `case_sensitive: boolean` as an optional `GrepToolParams` field (defaulting to false to preserve this PR's behavior) or documenting in the tool description that GrepTool is always case-insensitive, so the model knows to post-filter when exact case matters; (2) the third strategy in this file (the JS-fallback `ripgrep`-style or the in-process search at the bottom of the strategy chain — not shown in this diff) needs the same audit, otherwise the consistency claim only holds across two of the three strategies; (3) the regression at `grep.test.ts:317-345` only locks that `-i` is present in the args array — it doesn't lock the actual matching behavior end-to-end (i.e., that `pattern: "HELLO"` against a file containing `hello world` returns a match). A second integration-style test that runs the real strategy against a temp file would be more durable than the args-shape assertion, because the args-shape test would still pass if a future refactor moves the `-i` flag to a position where grep ignores it (which doesn't happen in practice for grep, but the principle applies).
