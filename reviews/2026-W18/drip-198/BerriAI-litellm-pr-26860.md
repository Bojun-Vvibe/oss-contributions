## Specific citations

- The new helper is at `litellm/litellm_core_utils/url_utils.py:49-66`: `encode_url_path_segment(value, *, field_name)` does `urllib.parse.quote(value_str, safe="")` after rejecting `None`, `""`, and the dot-segment values `"."`/`".."`. The dot-segment rejection is load-bearing — `quote(..., safe="")` deliberately leaves RFC 3986 unreserved characters including `.` unescaped, so `quote("..", safe="")` returns `".."` and a downstream HTTP client like `httpx` will normalize `https://api.anthropic.com/v1/files/..` into `https://api.anthropic.com/v1/` and silently change the request target. The companion `encode_url_path_segments(value, *, field_name)` at `:69-86` splits on `/` first and rejects empty segments, so leading/trailing/consecutive slashes also fail closed.
- The Anthropic call sites adopt the helper across four endpoints: batch-retrieve at `litellm/llms/anthropic/batches/transformation.py:126` (`{api_base.rstrip('/')}/v1/messages/batches/{encoded_batch_id}`), batch-results download at `litellm/llms/anthropic/files/handler.py:93-95`, file retrieve at `litellm/llms/anthropic/files/transformation.py:189`, file delete at `:212`, and file-content fetch at `:275`. Each call site uses `encode_url_path_segment(<id>, field_name="<id_name>")` and substitutes the encoded value into the f-string that builds the upstream URL.
- The Azure-AI agents handler at `litellm/llms/azure_ai/agents/handler.py:79-100` encodes `thread_id` and `run_id` separately on each of `_build_messages_url`, `_build_runs_url`, `_build_run_status_url`, `_build_list_messages_url` (one helper call per id, both are user-controllable since they come back from prior LLM-shaped responses). Bedrock invoke-agent at `litellm/llms/bedrock/chat/invoke_agent/transformation.py:101-110` does the same triple-encode for `agent_id` / `agent_alias_id` / `session_id` on the `/agents/{}/agentAliases/{}/sessions/{}/text` URL. Azure Document Intelligence at `litellm/llms/azure_ai/ocr/document_intelligence/transformation.py:218-228` adopts the helper for `model_id` even after the existing `model.split("/")[-1]` cleanup, which closes the case where `model_id` itself contains URL-meaningful characters.

## Verdict: merge-as-is

## Rationale

This is exactly the right shape for a path-segment-encoding sweep: one small reusable helper with a deliberate dot-segment rejection (which is the sharp edge that `quote(..., safe="")` leaves on the table by default), and adoption across every call site that interpolated a user-controllable identifier into an f-string upstream URL. The dot-segment rejection at `:49-66` is the load-bearing piece — without it, a `batch_id=".."` would silently retarget an Anthropic batch-retrieve request to the directory above `/v1/messages/batches/`, and there's nothing in the rest of the stack that would catch it before the request goes out. The call-site adoption is mechanical but exhaustive: Anthropic batches/files/skills, Azure-AI agents, Azure Document Intelligence, Bedrock invoke-agent and count-tokens. The `field_name="batch_id"` / `field_name="run_id"` etc. discipline means the resulting `ValueError` traceback at the rejection site tells the operator exactly which input was malformed, which is meaningfully better than a generic "URL parse error". The +1083/-100 size is mostly test surface — the regression count for a sweep this broad is the right insurance against a future URL builder being added that bypasses the helper. Two trivial nits worth flagging post-merge in a follow-up rather than blocking on: (1) `encode_url_path_segments` rejects empty segments which means a legitimate `/foo//bar/` style path (some upstreams allow it) can't be expressed through this helper — the doc comment is clear that this is intentional fail-closed behavior, but a `allow_empty_segments=False` kwarg would make the policy explicit; (2) there's no centralized lint or CI check that enforces "any new f-string of the shape `f"{api_base}/...{user_input}..."` must route through these helpers" — a follow-up grep-based CI lint would prevent the next contributor from re-introducing the bypass.
