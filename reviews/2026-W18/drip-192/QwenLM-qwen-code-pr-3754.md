## What changed

Two-piece structural change to qwen-code's `/review` skill:

1. **Six new `qwen review <subcommand>` CLI helpers** that replace the bash-style `gh api` / `git` invocations the SKILL.md previously asked the LLM to inline-shell-out. The LLM now calls `qwen review fetch-pr`, `pr-context`, `load-rules`, `deterministic`, `presubmit`, `cleanup` and reads typed JSON. All shell-outs go through `execFileSync` (no shell, so quoting/escaping is identical macOS/Linux/Windows). Tool detection uses `which` (POSIX) / `where` (Windows) at `lib/gh.ts` and `lib/git.ts` (around `:1280-1326` per grep).

2. **SKILL.md pipeline expansion**: 5 → 9 parallel review agents (Correctness/Security split + new Test Coverage + 3 undirected personas), iterative reverse audit (cap 3 rounds), and Step 9 gains self-PR detection + CI status check + 4-class existing-comment classification (Stale > Resolved > Overlap > NoConflict).

The load-bearing operational rules added in `presubmit`:

- **Self-PR rule** (around `:1976` per grep, in `presubmit.ts`): GitHub rejects both `APPROVE` and `REQUEST_CHANGES` events on a self-authored PR (HTTP 422). The new flow auto-downgrades both to `COMMENT` so the review *posts* (substantive findings still go in the inline `comments[]` array) rather than silently failing at submit.
- **CI rule**: when verdict is `APPROVE` but check-runs / commit statuses show any failure or all-pending, downgrade `APPROVE → COMMENT` and prepend the failed check names to the body. Rationale: a static-read LLM cannot see runtime test failures; approving on red CI is misleading.
- **Stacking** (DESIGN.md `:2227`): if a review is *both* self-PR *and* on red CI, both reasons are included in the body even though the event is `COMMENT` either way — informational only, but lets a future maintainer reading the comment know why an LLM that found no Critical issues did not approve.

## Why it's right

- **Replacing inline shell-outs in SKILL.md with typed CLI subcommands is the correct direction.** SKILL.md as it existed required the LLM to construct and parse `gh api graphql` queries verbatim, which is high-token, brittle to gh-CLI version drift, and cross-platform-fragile (POSIX shell quoting differs from PowerShell). Centralizing into `qwen review presubmit <pr> <sha> <owner/repo> <out>` (around `:2019`) emits a single canonical JSON artifact at `<out>`, and the LLM reads structured fields rather than re-parsing prose. Less drift surface, fewer tokens per invocation, identical behavior across OSes.
- **`execFileSync` everywhere (no shell)** — verified at `lib/gh.ts:1252-1256`, `lib/git.ts:1307-1311`, `cleanup.ts:206`, `cleanup.ts:221`. This is the security-load-bearing choice. A shell-out path takes a string and lets the OS shell parse it (so `pr-name="$(rm -rf /)"` becomes a code-execution gadget); `execFileSync('git', ['worktree', 'remove', wt, '--force'], ...)` passes args as an `argv` array straight to `execvp`, no shell interpolation. For a tool that processes attacker-controllable strings (PR titles, branch names, commit messages can all contain shell metachars), `execFileSync` is the only safe primitive.
- **Self-PR auto-downgrade is non-obvious-but-correct.** Without it, an LLM running `/review` on its own PR would synthesize a complete review, hit `gh pr review --approve`, get HTTP 422, and either (a) crash, (b) silently lose the entire review, or (c) try to retry endlessly. The downgrade-to-COMMENT path keeps the substantive findings (the `comments[]` array survives) while neutralizing only the formal review-event verdict. The PR's Step 9 description calls out `state=COMMENTED` as the verified end state on a self-PR smoke test against `#3717` — a real validation point.
- **CI rule is the more interesting one.** The model genuinely cannot run the tests; if all the model checked is "code looks fine on read", and CI says it doesn't compile / test fails, the model's `APPROVE` is wrong-by-omission. Downgrading to COMMENT and *prepending the failed check names to the body* is the right shape — the human reviewer immediately sees "model approved but CI is red on these checks", and decides whether to override.
- **Cleanup is idempotent** at `cleanup.ts:206-264`. `git worktree remove --force` and `git branch -D` are inside try-blocks so a missing worktree/branch doesn't error. Plus the `cleanup.ts` per-target temp-file cleanup uses a prefix-glob pattern (greppable via `tmpFile` filename prefix per `paths.ts:1386`). The PR description's "runs cleanly when worktree/ref/temp files don't exist" is verified by the idempotency design.
- **Cross-platform path strategy is honest.** The PR description notes that SKILL.md's previous `/tmp/qwen-review-*` references were moved to `.qwen/tmp/qwen-review-*` because `os.tmpdir()` returns `/var/folders/...` on macOS — using a project-local dir avoids that mismatch entirely. Less elegant than `os.tmpdir()` (project-local clutters the repo) but more maintainable than papering over the OS difference at every call site.

## Nits / asks

1. **Windows smoke not run.** PR test plan checkbox `[ ] Windows smoke test — not run locally`. The path/`which`/`where` logic is verified statically per the description, but real Windows CI is the only thing that catches `\r\n`-vs-`\n` parsing drift, `gh.exe`-vs-`gh` exec resolution, or PowerShell-specific worktree-path edge cases (e.g. `C:\Users\...` with backslashes). At minimum, the project's existing Windows CI matrix should run the new `review` command tests; if there's no Windows CI for `packages/cli`, this PR should add at least one. Worth flagging as an explicit known-unknown in the merge commit.

2. **2284-line PR is *operationally* hard to land.** The diff touches 14 files including a 1k+ line skill DESIGN.md, six new TS subcommand files, and three lib helpers. Even with the body's clear structural split, this PR will pile up review latency. Splitting into (a) "introduce `qwen review` subcommand surface + lib helpers" and (b) "rewire SKILL.md to use them + add presubmit logic" would let (a) merge fast as pure addition (no behavior change, SKILL.md still uses inline gh) and (b) carry only the SKILL.md flip + `presubmit` logic for focused review. Ship as a stack if Sapling/graphite is in use here.

3. **Cleanup deletes worktree, branch, *and* temp files** at `cleanup.ts:206-221`. The `git branch -D` is destructive — if the user manually checked out the review branch in another worktree to inspect it, `branch -D` errors (good) but the prior `worktree remove` may have already succeeded. Order matters: try `worktree remove` first (which only fails if the worktree has unpushed commits / dirty state), and only attempt `branch -D` if the worktree removal succeeded. The current ordering is exactly that — but a comment naming the ordering invariant would prevent a "cleanup branch first to avoid worktree-still-checked-out" reorder regression.

4. **`presubmit.ts` 4-class existing-comment classification (Stale > Resolved > Overlap > NoConflict, only Overlap blocks)** is a nuanced rule that future maintainers will have to keep aligned with the SKILL.md description. Encoding the 4-class precedence as an `enum CommentResolution { Stale, Resolved, Overlap, NoConflict }` with explicit `priority()` method (rather than implicit positional ordering in a list) would prevent a "let me reorder these for readability" patch from inverting the precedence silently.

5. **`deterministic` subcommand handles TS/JS/Python/Rust/Go but not C/C++/Java/Ruby/Swift.** That's a deliberate scoping choice (the body says future passes can plug in mypy/clang-tidy/checkstyle without changing the subcommand contract via the `ToolDef` registry pattern). But the SKILL.md should explicitly say "if the changed files include unsupported languages, deterministic returns empty findings — fall back to LLM-only review for those" so the LLM doesn't silently emit "no deterministic findings → looks clean" on a Java PR.

6. **End-to-end smoke ran on `#3717` and `#3673`** (per test plan) — both qwen-code repo PRs. The PR doesn't claim to have run against PRs in other repos (different gh-CLI auth context, different default-branch-protection rules, different CI shapes). For a tool meant to be cross-repo, at least one validation against an out-of-org PR (e.g. a fork-of-fork PR with no ci runs at all) would shake out the "no check_runs key in the response" branch.

7. **Cleanup of `.qwen/tmp/qwen-review-*` won't clear leftover dirs from older runs that predate the convention.** The `tmpFile` prefix at `paths.ts:1386` is forward-only — a one-shot `cleanup --orphans` mode that finds and removes any `.qwen/tmp/qwen-review-*` older than N days would catch the partial-run-failures-from-yesterday case.

## Risk

Medium. Surface area is large and touches both an LLM-driven skill (where bug taste is "model emits the wrong review") and a CLI surface (where bug taste is "shell-out destroys the user's repo"). The `execFileSync`-everywhere choice and the cleanup ordering substantially de-risk the latter. The former is best mitigated by (a) the smoke tests on real PRs the author already ran, and (b) the self-PR + CI downgrade rules that fail-safe toward `COMMENT` rather than authoritative verdict. Recommend landing as a stack and getting Windows CI into the matrix.
