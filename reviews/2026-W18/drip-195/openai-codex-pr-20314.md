## Concerns / nits

1. **`include_environment_id: bool` on the tool-options structs at `local_tool.rs:11, 16`** — same boolean-parameter-at-API-boundary smell as #20299 nit 1. When a future tool gains a *third* gating axis (e.g., per-environment-allowlist or per-environment-permission-profile), `bool` won't compose. Worth shaping as `enum EnvironmentExposure { Hidden, OptionalParameter, RequiredParameter }` upfront — same call-site cost today.
2. **The probe's `assertContains`/`assertLacks` helpers at `probe.ts:251-263`** match against `JSON.stringify` of pending messages, which is fragile to field-ordering changes. Consider `expect.objectContaining`-style matchers — `JSON.stringify` round-trips will silently drift if `serde_json` ever gains a deterministic-output flag.
3. **The probe's 20-second deadline at `probe.ts:222`** (`Date.now() + 20_000`) is reasonable for local CI but may be tight under macOS GitHub-runner load. The `waitFor` loop polls every 25ms — at 20s deadline that's 800 iterations of `findIndex` over `pending`. Fine for `pending.length < ~50` but could spike CPU if a noisy notification stream piles up. Bound `pending.length` (or drain irrelevant notifications) for resilience.
4. **`UnifiedExecRequest::environment` field is now mandatory (`Arc<Environment>`) at `unified_exec/mod.rs:97`** — every existing constructor site needs the field. The diff updates `process_manager_tests.rs:179` (`environment: turn.environment.as_ref().expect("turn environment").clone()`) but a `expect("turn environment")` panic in a test fixture is a foot-gun; better to construct an `Environment::local()` default for the test rather than assume the turn always has one.
5. **Two `ToolError::Rejected("exec_command is unavailable in this session")` call sites are deleted** at `unified_exec.rs:294-302, 334-339` — that error class was the *only* pre-existing user-visible signal that something was wrong with environment plumbing. Now that the environment is mandatory at request construction, the error class moves earlier (closer to the parser). Confirm that the new earlier rejection produces an equally clear user-facing message — silent behavior changes here would be hard to diagnose.
6. **No CHANGELOG / migration entry** in the diff slice. Wire-protocol additions to `ShellToolCallParams` are backward-compat (the field is `Option<String>` with `serde(default)`), but the *model-facing tool surface* changing — the JSON schema gaining or losing `environment_id` based on `include_environment_id` — is a behavior change for downstream consumers reading the tool catalog. Worth an explicit release-note line.
7. **Probe is not wired into CI in the diff** (no workflow file change visible). The probe is a debug/dev tool only unless `app-server/tests/suite/...turn_start.rs` covers the same matrix — confirm coverage parity or wire the probe up.
8. **The new `EnvironmentContextEnvironment::index` field in tests at `environment_context_tests.rs:626-635`** suggests an ordering invariant (primary = index 0). Worth a debug-assert at construction sites that `environments[0].index == 0` and `environments[i].index == i` so a future serializer that doesn't preserve order can't silently break the "primary environment" contract enforced by `primary_environment()` at `turn_context.rs:775`.
