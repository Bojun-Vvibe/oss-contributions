## Concerns / nits

1. **`is_file_change_output: bool` parameter at `event_mapping.rs:21`** — boolean parameters at API boundaries are a known smell. Specifically, when a future `EventMsg` variant is added that needs a *third* file-change-output mode (e.g., a file-change *delta* that's neither begin nor end), this signature will need to grow to `enum FileChangeOutputMode { ... }` and every call site updates. Worth doing as `enum FileChangeOutputMode { Standalone, EmbeddedInPatch }` upfront — same call-site cost today, much cheaper to extend tomorrow.
2. **No regression test in the PR diff** for `item_event_to_server_notification` itself — the function is the entire reason the module exists. The `thread-manager-sample` change exercises it as a smoke test but a unit test in `event_mapping.rs` covering at minimum (a) one `Begin`/`Update`/`End` triple, (b) the file-change branch with both values of the bool, (c) the `DynamicToolCallResponse` branch (which has the most branching: `success` → `Completed`/`Failed`, optional `duration_ms` cast via `i64::try_from(...).ok()`) would lock the projection contract and prevent silent wire-format drift when a v3 protocol field is added.
3. **`duration_ms = i64::try_from(response.duration.as_millis()).ok()` at `event_mapping.rs:38`** silently drops sub-millisecond resolution and silently drops durations > i64::MAX milliseconds (≈292 million years — fine in practice). The `.ok()` swallows the overflow case; consider `.unwrap_or(i64::MAX)` and a debug log at the conversion site so a pathological response doesn't disappear from analytics. Not a blocker.
4. **Public re-export at `lib.rs:17`** (`pub use protocol::event_mapping::*`) leaks every helper added to that module into the public crate API forever. Either narrow the re-export to `pub use protocol::event_mapping::item_event_to_server_notification` or document the module as the public surface. This is forward-compat hygiene — once it ships as `pub use ...::*` it can't be tightened without a breaking-change minor.
5. **`thread_id: &str, turn_id: &str` taken by reference then immediately `.to_string()` at `:30-31`** — taking the strings owned (`thread_id: String, turn_id: String`) avoids the small allocation for callers that already have an owned `String`. Most callers in `bespoke_event_handling.rs` are likely synthesizing these from owned data, so the API shape `&str` forces a redundant clone path. Minor perf nit.
6. **49-LOC, 1-PR architectural split** is fine for a stateless helper extraction, but the doc-comment "Callers remain responsible for any surrounding state checks" is a future foot-gun — a CI grep that fails if `item_event_to_server_notification` is called outside a list of allow-listed sites would prevent the next contributor from bypassing the state checks. Belt-and-suspenders.
