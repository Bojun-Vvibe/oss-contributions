# Context

The `codex` core has a turn loop where each turn starts with the
user's input, the model proposes tool calls, the tool runtime
executes them, and their results feed back as
`function_call_output` items in the next iteration. The shape is
the standard OpenAI Responses API: every `function_call` item must
be paired with a matching `function_call_output` keyed by `call_id`.

This PR adds a new (experimental, feature-flagged) protocol field
on `turn/start`: `prefetchedToolResults`. The motivating use case
is "the host application has *already* invoked some tools (e.g., a
hosted IDE pre-fetched the open file's contents, the recent git
log, the current selection) and wants to seed the next turn's
context with those results without round-tripping through the
model first." Without this, the host has to either (a) inject the
content as plain text in the user message (wasteful: it gets
re-tokenized as user content) or (b) wait for the model to invoke
the tool itself (extra latency + model has to know to ask).

The PR is +697/-143 across 24 files, but the substantive logic
sits in four: the protocol type
(`app-server-protocol/src/protocol/v2.rs`), a new variant on
`ResponseInputItem` (`protocol/src/models.rs`), the validation
that gates the new field
(`app-server/src/codex_message_processor.rs`), and the turn
loop's recording of the prefetched items
(`core/src/session/turn.rs`). The remaining changes are
mechanical signature plumbing through `TurnTaskInput` and a
couple of test files.

# Diff walkthrough

**Protocol surface.** A new optional field on `TurnStartParams`:

```rust
/// EXPERIMENTAL - Raw Responses API `function_call` and matching
/// `function_call_output` items to place before the user input for this turn.
#[experimental("turn/start.prefetchedToolResults")]
#[ts(optional = nullable)]
pub prefetched_tool_results: Option<Vec<JsonValue>>,
```

The field is `Vec<JsonValue>` rather than `Vec<ResponseItem>` —
deliberately untyped at the protocol layer because the validation
needs to do shape-checking with custom error reporting (see the
parser below). The `#[experimental(...)]` attribute is the gating
mechanism: only clients that opt in to the experimental feature
flag see the field in the generated TS bindings. That's the right
call for a behavior-changing addition that hasn't yet been
audited against the prod schema.

**The new `ResponseInputItem::FunctionCall` variant in
`protocol/src/models.rs`.** Previously the input-item enum had
`Message`, `FunctionCallOutput`, etc., but no way to inject a
*function call* item from the input side — the system always
generated those itself in response to model output. Now:

```rust
FunctionCall {
    name: String,
    #[serde(default, skip_serializing_if = "Option::is_none")]
    #[ts(optional)]
    namespace: Option<String>,
    arguments: String,
    call_id: String,
},
```

And a corresponding `From<ResponseInputItem> for ResponseItem`
arm that maps to `ResponseItem::FunctionCall { id: None, name,
namespace, arguments, call_id }`. Setting `id: None` on the
output is the correct choice (the id is internal and assigned by
the model service). The `serde(default,
skip_serializing_if = "Option::is_none")` on `namespace` handles
the case of legacy clients that don't set it. Clean.

**The validation parser in
`codex_message_processor.rs:6109`.** This is the most carefully-
written part of the PR and the place where the security-relevant
contract sits. ~100 lines:

```rust
fn parse_prefetched_tool_results(
    values: Option<Vec<serde_json::Value>>,
) -> Result<Vec<ResponseInputItem>, JSONRPCErrorError> {
    let Some(values) = values else {
        return Ok(Vec::new());
    };

    let mut parsed = Vec::with_capacity(values.len());
    let mut calls = HashMap::<String, usize>::new();
    let mut outputs = HashMap::<String, usize>::new();

    for (index, value) in values.into_iter().enumerate() {
        let item = serde_json::from_value::<ResponseItem>(value).map_err(|err| {
            Self::invalid_prefetched_tool_results_error(format!(
                "prefetchedToolResults[{index}] is not a valid response item: {err}"
            ))
        })?;

        match item {
            ResponseItem::FunctionCall { name, namespace, arguments, call_id, .. } => {
                if call_id.is_empty() {
                    return Err(...empty call_id...);
                }
                if calls.insert(call_id.clone(), index).is_some() {
                    return Err(...duplicate call_id...);
                }
                parsed.push(ResponseInputItem::FunctionCall { name, namespace, arguments, call_id });
            }
            ResponseItem::FunctionCallOutput { call_id, output } => {
                ...same checks...
                parsed.push(ResponseInputItem::FunctionCallOutput { call_id, output });
            }
            _ => {
                return Err(...must be function_call or function_call_output...);
            }
        }
    }

    for (call_id, output_index) in &outputs {
        let Some(call_index) = calls.get(call_id) else {
            return Err(...output for missing call_id...);
        };
        if output_index < call_index {
            return Err(...output must appear after call...);
        }
    }
    for call_id in calls.keys() {
        if !outputs.contains_key(call_id) {
            return Err(...call without output...);
        }
    }

    Ok(parsed)
}
```

This is doing real validation work, not just shape-checking:

- **Type whitelist**: rejects anything that isn't a `FunctionCall`
  or `FunctionCallOutput`. A client can't sneak an
  arbitrary-shape input item (a fake assistant message, a fake
  reasoning trace) into the prefetched batch — only the two item
  types the feature is intended for.
- **Non-empty call_id**: a `call_id == ""` would silently pass
  most other validators. Explicit check.
- **No duplicate call_ids**: prevents an attacker from binding
  one fake output to two different calls (which could confuse the
  matching logic in the turn loop).
- **Every call has an output and vice versa**: the central
  contract of the Responses API. If the host injects a call
  without an output, the model sees an "unanswered" tool call and
  may freeze or hallucinate. If it injects an output without a
  call, the API rejects the whole turn.
- **Output must come after its call**: enforces the ordering
  contract, so the items can be appended to the conversation in
  the order received.

What's *missing* from this validation is also worth noting. There's
no check that the tool name in the `FunctionCall` corresponds to a
tool the agent actually has registered. A host could inject a
prefetched call for a tool the model doesn't know about; the model
will see an unfamiliar tool call in its history and may either
ignore it or imitate it. The PR doesn't address this — see Risks.

**The turn-loop integration in `session/turn.rs`.** The prefetched
items are recorded into the conversation *before* the user input:

```rust
let prefetched_tool_result_items = prefetched_tool_results
    .into_iter()
    .map(ResponseItem::from)
    .collect::<Vec<_>>();
let additional_contexts = if input.is_empty() {
    Vec::new()
} else {
    ...
    user_prompt_submit_outcome.additional_contexts
};
if !prefetched_tool_result_items.is_empty() {
    sess.record_conversation_items(&turn_context, &prefetched_tool_result_items)
        .await;
}
if !input.is_empty() {
    let initial_input_for_turn: ResponseInputItem = ResponseInputItem::from(input.clone());
    let response_item: ResponseItem = initial_input_for_turn.into();
    sess.record_user_prompt_and_emit_turn_item(turn_context.as_ref(), &input, response_item)
        .await;
}
```

Two correctness details. First, the `if input.is_empty() &&
prefetched_tool_results.is_empty() && !sess.has_pending_input().await
{ return None; }` guard at the top of `run_turn` correctly handles
the case where the only thing the host wants to do is inject
prefetched tool results with no user input — that should still
trigger a turn (the model will respond to the new tool results).
Second, the prefetched items are recorded via
`record_conversation_items` (no turn-item event emitted to the
host) while the user input is recorded via
`record_user_prompt_and_emit_turn_item` (which does emit). That's
the right asymmetry — the host already knows about the prefetched
items (it just sent them), so re-emitting them as turn events
would be redundant.

# Risks / things a maintainer would catch

1. **No tool-name validation.** A host can inject
   `FunctionCall { name: "rm_minus_rf", arguments: "{}", call_id: "fake" }`
   even if the agent has no `rm_minus_rf` tool. The model sees
   this as part of its conversation history. Effects on model
   behavior are hard to predict and almost certainly bad. The
   parser should validate that `name` resolves to a registered
   tool (with allowance for the namespace prefix).
2. **No content-size limits.** `arguments` and `output` can be
   arbitrary-size JSON. A host that injects a 10MB prefetched
   tool result will blow past the model's context window and
   produce an obscure model error rather than a clear
   `INVALID_REQUEST` rejection at the parser layer. A
   per-item-size and total-batch-size cap would be cheap and
   diagnostic.
3. **`namespace` is preserved but not validated.** If a tool's
   namespace is wrong (e.g., the host says `namespace:
   "filesystem"` but the actual tool is registered as
   `namespace: "fs"`), the model sees the wrong namespace and
   may not connect this prefetched call to the live tool. Worth
   either validating against registered namespaces or
   documenting that the host is responsible for getting it
   right.
4. **Outputs-before-calls check uses `index` from the original
   `values: Vec`.** The check `output_index < call_index` is
   correct, but if multiple outputs reference the same (single)
   call, only one of them is detected as duplicate; the dedupe
   check happens at insert time. OK with the current dedupe
   logic but easy to break in a future refactor.
5. **No cap on number of prefetched items.** A host can inject
   1000 fake tool calls. The parser will validate them all (O(N)
   pairwise) but the resulting context bloat will silently blow
   the conversation. A `max_prefetched_items: 100` check would
   prevent this footgun.
6. **The "experimental" gate is opt-in but shipping this opens
   the door to behavior the prod schema doesn't model.** If a
   non-experimental client sends `prefetchedToolResults`, they
   get... what? Silently ignored? Hard error? The PR description
   says "behind the experimental feature flag and does not
   change the prod schema" — confirming the client-gating
   behavior end-to-end (including how the schema validator
   handles the unknown field for non-experimental clients) would
   be reassuring.
7. **`record_conversation_items` is called without emitting turn
   items.** That's intentional, but if a host *crashes* between
   sending the prefetched items and waiting for the turn
   response, the items are now in the conversation history with
   no host-visible record. On reconnect the host might re-send
   the same items, doubling them. Idempotency via call_id check
   on record could help.
8. **Thread-safety of `parse_prefetched_tool_results` is fine
   (it's a static function, no shared state), but the mutation
   of `sess` via `record_conversation_items` followed by
   `record_user_prompt_and_emit_turn_item` is two separate
   async calls.** A concurrent turn-cancel between the two
   leaves a half-written turn (prefetched items in history, no
   user input recorded, no turn event emitted). Probably already
   handled by the cancellation token but worth confirming.
9. **No tests for the type-whitelist branch.** The diff shows
   tests for valid happy-path, but I don't see one for "host
   sends a `Message` item in `prefetchedToolResults` and gets a
   400". Worth locking down.
10. **`Vec<JsonValue>` at the protocol layer means OpenAPI/TS
    generation can't statically describe the contents.** The
    PR uses `#[experimental]` to gate this, but downstream SDK
    consumers won't get type-safe access to prefetched items.
    Acceptable for the experimental phase; should become a
    proper typed `Vec<PrefetchedItem>` before stabilization.

# Suggested questions for the author

1. Is there a planned tool-name-validity check before
   stabilization? Without it, hosts can inject calls for
   nonexistent tools and confuse the model in hard-to-debug
   ways.
2. What happens when a non-experimental client sends
   `prefetchedToolResults`? Silently dropped, or 400'd? Either
   is defensible but should be documented.
3. Is there appetite for per-item and per-batch size caps in
   the parser? Even loose caps (1MB per item, 10MB per batch)
   would prevent context-window blowups.
4. The `namespace` field is preserved as-is. Should the parser
   validate that `(namespace, name)` resolves to a registered
   tool, or is "host knows what it's doing" the contract?
5. After this lands and is exercised by real hosts, would you
   consider promoting the Vec<JsonValue> at the protocol layer
   to a typed `Vec<PrefetchedItem>` enum so downstream SDKs get
   type safety?

# Verdict

**Approve as experimental.** The validation logic is unusually
careful — the type whitelist, the call/output pairing, and the
ordering check are exactly what's needed at this trust boundary.
The integration into the turn loop preserves the right asymmetry
(prefetched items recorded silently, user input emits a turn
event). Tool-name validation and size caps are the two follow-ups
I'd want to see before stabilization, but neither is a blocker
for the experimental gate.

# What I learned

The interesting design lesson here is **using a parser as the
trust boundary**. The PR could have just deserialized the input
into `Vec<ResponseInputItem>` and trusted serde — that's the
default Rust API design instinct. Instead, it deserializes one
item at a time, runs structural and relational validation
between items, and emits *contextual error messages* that
identify the offending item by index. The result is a parser that
both rejects malformed input *and* tells the host exactly where
the problem is (e.g., "prefetchedToolResults[2] function_call
call_id must not be empty").

This is the "parse, don't validate" pattern from the Haskell /
Rust functional-programming idiom, applied to a security-
relevant boundary. The validated `Vec<ResponseInputItem>` that
emerges from `parse_prefetched_tool_results` is *constructively
correct* — every call has an output, every output has a call,
ordering is correct, types are restricted to the two allowed
variants. Code downstream of the parser doesn't need to re-check
any of these invariants. That's the payoff for paying upfront in
a careful parser: the invariants become *type-witnessed* further
down the pipeline.

The complementary lesson is about **error message ergonomics at
trust boundaries**. The error strings here include the exact
item index, the field name, the failing constraint. That makes
debugging integration issues fast for the host developer. The
contrasting failure mode — "Invalid prefetched_tool_results" —
is what happens when error messages are written for the
*implementer* rather than the *caller*. The PR consistently
chooses the latter audience, which is the correct choice for any
RPC surface.
