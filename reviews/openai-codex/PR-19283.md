# Context

The Windows elevated sandbox path spawns a runner
process under a sandbox-user logon via
`CreateProcessWithLogonW` and communicates over a pair
of parent-created named pipes (one inbound, one
outbound). The parent creates the pipes with a DACL
that grants `GA` (generic-all) to the sandbox user
SID, then waits on `ConnectNamedPipe` for the runner
to attach.

There are two elevated codepaths in tree:
`elevated_impl.rs` (one-shot capture for
`exec`-style invocations) and `runner_client.rs`'s
`spawn_runner_transport` (session/unified-exec). Both
re-implement the same pipe creation + connect + ack
sequence with subtle drift.

# Problem

The DACL is the only access check today. That's a
necessary defense but not a sufficient one: any
process running under the sandbox user account that
notices the well-known pipe-name pattern
(`\\.\pipe\codex-runner-{u128}-{in|out}`) before the
real runner does could connect first. The u128
randomness mitigates discovery but not race-after-
disclosure, and any sandbox-user process that can
read parent argv or strace-equivalent on the launcher
side could observe pipe names.

Once a wrong-but-authorized process holds the pipe,
the parent will happily write `SpawnRequest` frames
to it and trust whatever it gets back as runner
output. That's a confused-deputy boundary right at
the edge where the parent transitions from trusted
to sandbox-user authority. Anything we can do to
fail closed at the connect handshake is a multiplier
on the existing ACL.

# Design

Two changes layered:

**1. PID gate on `connect_pipe`.** After
`ConnectNamedPipe` returns (or returns
`ERROR_PIPE_CONNECTED`), call
`GetNamedPipeClientProcessId` and compare against the
PID we got back from `CreateProcessWithLogonW` in
`pi.dwProcessId`:

```rust
if client_pid != expected_runner_pid {
    return Err(io::Error::new(
        io::ErrorKind::PermissionDenied,
        format!("named pipe client pid {client_pid} did not match runner pid {expected_runner_pid}"),
    ));
}
```

This is exactly the right fail-closed shape. Wrong
PID → `PermissionDenied` → caller cleans up the
spawned process and aborts. Doc comment on the
module is updated from "permissive ACLs" to
"sandbox-user-scoped ACLs", which is the truthful
description of what's already there.

**2. Unify `elevated_impl.rs` onto
`spawn_runner_transport`.** The one-shot capture path
gains the PID check by virtue of going through the
same plumbing as the session path. The +46/-267
delta is dominated by the deletion: ~260 lines of
duplicated CPWL / pipe / handshake code go away and
are replaced by a function call into the shared
helper.

Net: the security improvement and the dedup land
together, which is the right way to do this — the
dedup means future hardening only has to land in one
place. PR's own framing in "What changed" is
accurate.

# Risks and edges

- **PID reuse.** Windows PIDs are reused after process
  exit. There is a (very narrow) window between
  `CreateProcessWithLogonW` returning and the runner
  actually connecting where, theoretically, the runner
  could exit early and another sandbox-user process
  could acquire the same PID. Realistically this
  requires (a) the runner to crash before connect, and
  (b) Windows to recycle the exact PID before
  `ConnectNamedPipe` returns. The check still narrows
  the attack surface dramatically vs no check, and a
  closer-to-airtight version would also call
  `OpenProcess(PROCESS_QUERY_LIMITED_INFORMATION,
  FALSE, expected_runner_pid)` before the connect and
  hold the handle alive until after the PID
  comparison, guaranteeing the PID isn't reused. Worth
  noting as a follow-up; the current PR is already the
  correct directional fix.
- **Two pipes, two checks.** Both `h_pipe_in` and
  `h_pipe_out` are gated against the same expected
  PID. Correct — the runner is one process opening
  both ends. If the runner ever became a multi-
  process design (one for stdin, one for stdout, e.g.
  to satisfy a future security boundary) the check
  shape would need to change. Not an issue today.
- **`GetNamedPipeClientProcessId` requires
  `FILE_READ_ATTRIBUTES` on the pipe handle.** The
  parent-created server handle has full access by
  default so this is fine, but worth a comment in the
  code so a future "tighten access mask" refactor
  doesn't quietly break the PID check.
- **The error message includes both PIDs.** Good for
  debugging. Doesn't leak anything sensitive — both
  PIDs are local to the host and the parent already
  has full visibility into them.
- **`spawn_ready` handling moves into the shared
  helper.** The deleted `read_spawn_ready` in
  `elevated_impl.rs` is replaced by whatever
  `spawn_runner_transport` does internally. Reviewer
  should confirm the error semantics match — in
  particular, that a runner that exits before sending
  `SpawnReady` still surfaces the same
  "runner pipe closed before spawn_ready" error
  shape. From the diff, the helper already had this
  logic, so this should be a clean lift.
- **No new tests.** The PR says cargo tests are
  unchanged: 65 passed, 5 pre-existing failures.
  That's accurate but it means there's no automated
  coverage for either (a) the PID gate firing on a
  wrong PID, or (b) the dedup-induced behavior
  parity. The first is hard to unit-test on Windows
  (you'd need to fake-spawn a process and race the
  connect), but the second is straightforward — a
  test that drives both `run_windows_sandbox_capture`
  and the session path through a shared mock should
  assert they call into `spawn_runner_transport`. A
  module-level "I hold the contract" test would be
  enough.
- **Cross-platform compile.** All changes are
  inside `mod windows_impl` / `windows_sandbox`
  crates that are already conditionally compiled.
  Non-Windows builds are unaffected.
- **Removed imports list.** The deletion list at
  the top of `elevated_impl.rs` includes the obvious
  ones (`CreateNamedPipeW`, `ConnectNamedPipe`,
  `CreateProcessWithLogonW`, `quote_windows_arg`,
  `to_wide`, etc.). Verify none of them are still
  referenced by an inline helper that wasn't deleted.
  The compiler will catch this but the diff is large
  enough that a quick `cargo check` is the right
  smoke.

# Verdict

Approve. Correct security improvement, executed via
the right structural change (dedup onto the shared
transport). Two follow-ups worth opening as
issues:

1. PID-reuse hardening via `OpenProcess` handle
   pinning before connect.
2. A module-level "both elevated paths use
   `spawn_runner_transport`" contract test, so the
   dedup doesn't quietly drift apart again.
