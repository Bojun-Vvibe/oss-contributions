# Context

`codex` represents shell-like tool calls through `unified_exec`,
which fans out into both fresh command invocations and
`write_stdin` against an existing PTY-backed process. Output
truncation in this stack happens at two layers: `ContextManager`
clamps what the model sees in its in-memory context window, and
the per-tool `max_output_tokens` parameter clamps what gets
serialized into the persisted rollout on disk. Above both of
those sits the per-turn `truncation_policy`, which represents
the *budget* the current turn is willing to spend on tool
outputs.

# Problem

The PR title undersells it (it's flagged `chore:` but is
actually a correctness fix). Before this change, `unified_exec`
and `write_stdin` were passing `max_output_tokens: None` (or the
caller's value, with no policy clamp) into the manager. The
result was a two-faced behaviour:

- The model never *saw* the over-budget output, because
  `ContextManager` truncated it before it landed in the
  in-memory message buffer.
- The full untruncated output was still being persisted to the
  rollout file on disk.

This is the kind of bug that is invisible during development —
everything looks fine in the TUI, the model behaves sensibly,
the agent loop terminates — and only surfaces when someone tries
to replay a session, ship rollouts to a downstream system, or
audit token usage. The disk-side blob can be many times larger
than what the model actually consumed, and any tooling that
trusts rollout-level token counts is silently lying.

# Design

A small new helper, `effective_max_output_tokens`:

```rust
fn effective_max_output_tokens(
    max_output_tokens: Option<usize>,
    truncation_policy: TruncationPolicy,
) -> usize {
    resolve_max_tokens(max_output_tokens).min(truncation_policy.token_budget())
}
```

Two call sites adopt it: the fresh-exec branch and the
`write_stdin` branch. Both now pass `Some(max_output_tokens)`
into the downstream request structs and the
`ExecCommandToolOutput` shape, where they used to pass `None` or
the unclamped caller value.

The `min(...)` semantics is the right shape: an explicit
caller-side limit narrows the cap further, but the policy budget
is always an upper bound. The existing `ContextManager` clamp is
explicitly preserved as a backup, which is the correct
defense-in-depth posture.

The bulk of the diff (172 of 189 added lines) is the test file:
a new `wait_for_raw_unified_exec_output` helper that taps the
`RawResponseItem` event stream to capture what gets persisted to
rollout (as opposed to what hits the model's context). That's
exactly the right thing to test — the bug was specifically about
disk persistence diverging from model-visible context, so the
test has to assert against the persistence layer directly.

# Risks and edges

- **`turn.truncation_policy` ownership.** The handler now reads
  `truncation_policy` once at the top of the dispatch, then
  passes it into the helper. If the policy is ever mutated
  mid-tool-call (it shouldn't be, but I can't prove it from this
  diff), the second call to the helper in the `write_stdin`
  branch would use a stale value. A `let policy =
  turn.truncation_policy.clone()` (or a debug-asserted
  invariant) at the top would document the assumption.
- **Existing `max_output_tokens: None` callers downstream.** The
  shift from `None` to `Some(N)` in `ExecCommandToolOutput`
  changes the wire shape. Anything downstream that
  pattern-matched on `None` ("caller did not specify a cap")
  will now see a value and may behave differently. The PR's
  diff suggests the downstream consumers all treat `Some(N)`
  uniformly, but a one-line audit comment naming the consumers
  would help.
- **No assertion on the *original* buggy behaviour.** The new
  test asserts the post-fix behaviour. A regression-targeted
  assertion — "the rollout output's token count never exceeds
  `policy.token_budget()`" — would catch a future re-introduction
  of the bug even if the helper were silently bypassed at one
  call site. Property-style rather than example-style.
- **Sandbox-denial path.** The third call site (the
  sandbox-denial early-return branch) also adopts
  `Some(max_output_tokens)`. That's correct, but it means a
  denial response now carries a token-budget figure even though
  no execution happened. Consumers reading rollout for token
  accounting need to know that "denied" rows still consume
  budget; a one-sentence doc comment would help.

# Suggestions

- Promote the test helper's invariant into a property assertion:
  iterate over a small grid of `(caller_cap, policy_budget)`
  pairs and assert `output_tokens <= min(caller_cap,
  policy_budget)` for every combination. That's the property the
  fix establishes; let the test enforce it generically.
- Add a `// kept for defense-in-depth` comment next to the
  `ContextManager` truncation site, since the PR description
  explicitly designates it as backup. Without that comment,
  someone refactoring later might delete it as "dead now that
  unified_exec respects the policy itself."
- Consider renaming the PR commit subject to `fix:` —
  classification matters for the changelog, and "rollout
  persisted untruncated output despite policy" is squarely a
  fix.

# Verdict

Approve. This is a real correctness fix with thoughtful test
coverage, a small and reviewable surface, and the right defense
posture (don't remove the existing safety net just because the
new gate is in place). The follow-ups are small.

# What I learned

The two-layer truncation pattern (model-visible context vs
persisted rollout) is a footgun unique to systems that try to
keep a separate audit trail from what the model saw. The instinct
to truncate aggressively for the model is correct; the instinct
to *not* truncate the rollout (so you can replay later) is also
correct. The bug shape is in the seam — assuming the
context-manager clamp is "good enough" for both views, when in
fact each view needs its own gate. The fix is mechanical once
named, but naming it requires the discipline to ask "where does
the truncated value actually land?" at every layer that touches
a token-bearing buffer.
