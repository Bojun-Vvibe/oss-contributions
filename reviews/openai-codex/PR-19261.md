# Context

`codex` config supports a layered model: a base config can be
overlaid by per-project, per-workspace, or per-user layers,
each living in a different folder. Within an `[agents.*]`
section, `AgentRoleToml.config_file` may point to a child
TOML containing the actual role definition. The documented
contract is that `config_file` is resolved *relative to the
layer the agent role was declared in* — so a project layer
can ship `agents/my-role.toml` next to its `codex.toml` and
that path resolves the same way regardless of the user's
CWD.

The mechanical piece that makes this work is `AbsolutePathBuf`,
a wrapper that requires a base path at deserialization time
so any relative path it sees can be resolved immediately
rather than carried as ambiguous text.

# Problem

The layer loader was deserializing `[agents.*]` tables
*without* installing the layer's folder as the active base
path. `AgentRoleToml.config_file` therefore tried to construct
its `AbsolutePathBuf` with no base and failed at runtime with
`AbsolutePathBuf deserialized without a base path`. The
documented behavior — "you can use a relative `config_file`"
— was unreachable.

The reporter (#19257) hit this with the most natural
spelling: a project-level `codex.toml` declaring
`config_file = "agents/my-role.toml"`.

# Design

The fix threads the layer config folder through the
`[agents.*]` deserialization step using the same
folder-as-base mechanism the rest of the layer loader
already uses for the rest of the TOML body. The change to
`agent_roles.rs` is small (8 +, 2 −) and isolates the new
behavior to the one section that needed it.

The bulk of the diff (57 +) is in `config_tests.rs`: a new
test creates a user config layer with a relative
`config_file` for an agent role, drives the loader, and
asserts both the resolved absolute path and that the inner
role config deserializes successfully. The test layout
mirrors what the reporter's repro would look like end-to-end,
which is the right shape — it would have caught the bug.

# Risks and edges

- **Other `AbsolutePathBuf` fields in layered sections.** The
  fix is scoped to `[agents.*]`. If any other layered
  section also embeds an `AbsolutePathBuf`, the same bug
  exists there and is not fixed by this PR. A grep for
  `AbsolutePathBuf` in the config crate would confirm. If
  others exist, the fix should generalize to install the
  base path once at layer-load time rather than per-section.
- **Base-path semantics across layers.** When two layers
  both declare an agent role with the same name (later layer
  wins), the relative `config_file` in the winning layer
  resolves against *its* folder. That is almost certainly
  the right behavior, but the test does not assert it. A
  second test with a base + override layer where each
  declares a different relative `config_file` would lock
  this in and prevent a future refactor from silently
  flipping the base.
- **Symlinks and `..`.** `AbsolutePathBuf` does not appear
  to canonicalize. A relative path of `../shared/role.toml`
  would resolve to a path that escapes the layer folder.
  That is probably fine — it matches how POSIX paths work
  and is what users would expect — but if codex ever adds a
  layer-sandboxing constraint (e.g. project layers cannot
  reference paths outside the project root), this resolver
  is the place to enforce it.

# Verdict

Approve. This is the smallest fix that resolves the reported
bug, the test is end-to-end and would have caught it, and
the change does not affect non-relative paths or other
layered sections. Worth a follow-up issue to audit other
`AbsolutePathBuf` uses in the layer loader.

# Skim path

1. `agent_roles.rs` — confirm the base path is installed

2. `config_tests.rs` — confirm the new test uses a layer
