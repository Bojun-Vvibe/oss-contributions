# openai/codex PR #19606 — permissions: make runtime config profile-backed

## Summary

Refactors codex-rs's permission/sandbox plumbing so that the runtime carries a single `PermissionProfile` value end-to-end rather than the ad-hoc `(SandboxPolicy, FileSystemSandboxPolicy, NetworkSandboxPolicy)` triple it threaded through `build_exec_request`, `requested_permissions_trust_project`, and the app-server message processor. Introduces a new `compatibility_sandbox_policy_for_permission_profile` helper that derives the legacy `SandboxPolicy` shape on demand for the existing exec/sandbox machinery, and renames the profile constants (`PermissionProfile::Disabled`, `::workspace_write()`, `::read_only()`, `::External { network }`) so call-sites no longer construct sandbox policies directly.

## Verdict: `needs-discussion`

The direction is correct — the triple was a smell, and consolidating it into a single profile-of-record makes future permission-model evolution much easier. But +1383 / −655 across the app-server, core, protocol, and analytics crates is a one-shot rework of a security-critical surface, and the diff makes several subtle behavioural choices that I want to flag before stamping merge:

## Specific references

- `codex-rs/app-server/src/codex_message_processor.rs:2273-2320` — the new `effective_permission_profile` computation. The flow is: derive `(file_system_sandbox_policy, network_sandbox_policy)` from the requested profile, mutate `file_system_sandbox_policy` via `Self::preserve_configured_deny_read_restrictions`, then *reconstruct* the profile via `PermissionProfile::from_runtime_permissions_with_enforcement(profile.enforcement(), &fs, net)`. The mutate-then-reconstruct ordering is load-bearing: if `preserve_configured_deny_read_restrictions` ever forgets to mutate in place and instead returns a value that gets dropped, the deny-read restrictions silently disappear. Worth a `#[must_use]` on the helper or restructuring as `let fs = preserve_…(fs, …)` so the data flow is impossible to mis-wire.
- `codex-rs/app-server/src/codex_message_processor.rs:2295-2298` — the `can_set` check still uses the legacy-shaped `sandbox_policy` (now derived on the fly via `compatibility_sandbox_policy_for_permission_profile`). On the fall-through path (line 2317) when `can_set` rejects, the code now constructs a profile from `SandboxEnforcement::from_legacy_sandbox_policy(&policy)` even though `policy` is the *configured* `self.config.permissions.sandbox_policy.get().clone()` rather than the *requested* one. Confirm this is intentional — the previous code returned the configured triple here, the new code rebuilds a profile from the configured legacy policy, which should be observably equivalent but is one round-trip more.
- `codex-rs/analytics/src/analytics_client_tests.rs:163` — `sample_permission_profile()` now returns `CorePermissionProfile::Disabled.into()` instead of `…::from_legacy_sandbox_policy(&SandboxPolicy::DangerFullAccess)`. Those are *not* the same thing semantically: `Disabled` means no sandboxing applied, `DangerFullAccess` was the legacy way to express the same outcome. The test now exercises a different code path (the `Disabled` branch in the new profile-aware analytics serialisation). Confirm the analytics payload remains byte-identical for the `Disabled` case, otherwise downstream dashboards may move.
- `codex-rs/app-server/src/codex_message_processor.rs:10152-10169` — `requested_permissions_trust_project`. The new logic computes `(file_system_sandbox_policy, network_sandbox_policy)` from the profile, derives a legacy `sandbox_policy` for the `matches!` check against `WorkspaceWrite | DangerFullAccess | ExternalSandbox`. Functionally equivalent to the old `to_legacy_sandbox_policy(cwd).is_ok_and(...)`, but the old form would *fail closed* on a profile that couldn't be converted (e.g. a malformed external-sandbox config) — the new form derives a policy unconditionally, which fails *open* (the `matches!` may still match). Worth a test for the malformed-profile case.
- `codex-rs/app-server/src/codex_message_processor.rs:10685-10688` — the test `requested_permissions_trust_project_uses_permission_profile_intent` now uses `PermissionProfile::Disabled` / `::workspace_write()` / `::read_only()`. Coverage of the `External` branch is preserved by `external_profile = PermissionProfile::External { network: NetworkSandboxPolicy::Restricted }` at line 10672, but the previous test also exercised `from_legacy_sandbox_policy(&SandboxPolicy::ExternalSandbox { network_access: Restricted })` which validated the legacy-→-profile conversion path. That conversion still happens at the API boundary (line 2310 fallback path) — please add a test specifically for that conversion before merging, since the rename means the existing test no longer covers it transitively.

## What needs to be discussed

1. The `Disabled` rename. `from_legacy_sandbox_policy(&SandboxPolicy::DangerFullAccess) → PermissionProfile::Disabled` is a rename that erases user-visible nomenclature ("Danger" was a deliberate scare-word). Operators reading config files or telemetry will lose that signal. Either keep the `Danger` name in serialised form via `#[serde(rename = "DangerFullAccess")]` or document the migration in CHANGELOG.
2. `compatibility_sandbox_policy_for_permission_profile` — the name signals this is meant to be a temporary bridge. Is there a follow-up issue for ripping it out, and what's the deprecation horizon? Otherwise the "compatibility" qualifier rots.
3. With −655 deletions concentrated in the app-server processor, please confirm the reduction is mechanical (variable-count compression from 3 → 1) and not removing any error-path branches. The two `JSONRPCErrorError` returns in the modified block look preserved but the diff is large enough that I can't be 100% sure without a side-by-side.
