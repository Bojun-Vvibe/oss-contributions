# Context

`codex-rs/exec-server` is the long-running server process that
holds tool-execution sessions. A client hands it a
`RunRequest { prompt, max_turns, timeout_ms, ... }` and the
server drives the model/tool loop until one of the stop
conditions: max turns reached, model returns final answer, the
client disconnects, or the request timeout elapses.

The timeout is wall-clock, not per-iteration: the spec is
"from the time the server accepts the request to the time it
emits a terminal event, no more than `timeout_ms`
milliseconds shall elapse". The client side uses this to bound
its own resource allocation (shelled-out sandbox cleanup, UI
spinners, budgeting).

# Problem

The timeout was enforced as a `tokio::time::timeout(...)`
wrapper around the *single* `run_turn(...)` future. When the
tool loop iterated — model calls a tool, server runs it,
feeds result back, model calls another tool — each turn got a
fresh timeout. A loop that produced tool calls as fast as the
budget per-turn allowed would run indefinitely.

Repro from the attached issue: a run with `timeout_ms: 10_000`
and a model that kept asking to `read_file` on a large
repository ran for 47 seconds before `max_turns=50` finally
terminated it. From the client's perspective the server
"ignored" the timeout.

# Design

The fix introduces a session-level deadline:

```rust
pub struct Session {
    pub id: SessionId,
    pub deadline: Option<Instant>,
    // ...
}

impl Session {
    pub fn new(req: &RunRequest) -> Self {
        let deadline = req
            .timeout_ms
            .map(|ms| Instant::now() + Duration::from_millis(ms));
        Self { id: SessionId::new(), deadline, ... }
    }

    pub fn remaining(&self) -> Option<Duration> {
        self.deadline.map(|d| d.saturating_duration_since(Instant::now()))
    }
}
```

And `run_turn` consults it on every iteration:

```rust
loop {
    if let Some(rem) = session.remaining() {
        if rem.is_zero() {
            return Err(Error::Timeout { ... });
        }
    }
    let turn = tokio::time::timeout(
        session.remaining().unwrap_or(Duration::from_secs(3600)),
        model.turn(state, &mut ctx),
    )
    .await;
    // ...
}
```

A few things right about this:

1. **`Instant`, not `SystemTime`.** Deadlines cross clock
   adjustments (NTP steps, DST) gracefully because `Instant`
   is monotonic. Using `SystemTime` here would produce
   "timeout fires 1 hour early after DST fallback" once a
   year. The PR gets this right.
2. **`saturating_duration_since`.** Past-deadline produces a
   zero `Duration` rather than panicking on underflow. The
   explicit `is_zero()` check then converts to an `Error::
   Timeout` before dispatching another turn.
3. **Per-turn `tokio::time::timeout` retained.** The
   session deadline is an *upper bound* on the loop. The
   per-turn timeout (set to `session.remaining()`) means an
   individual slow turn cannot stall the loop past the
   deadline. Defense in depth.
4. **Optional deadline.** `timeout_ms: None` leaves the
   session unbounded, which is the documented protocol
   default. The `unwrap_or(Duration::from_secs(3600))` in
   the per-turn fallback is a sane safety net — no request
   should wait an hour inside a single model turn, ever,
   even if the caller asked for unbounded.

# Risks

- **Long-running tools outside the loop.** `run_turn`
  awaits `model.turn(...)` which internally dispatches
  tool calls. If a tool runs for 30s (e.g. `shell` with a
  long test suite) and the deadline is 10s, the tool
  finishes, returns, and *then* the loop's deadline
  check fires. That is, the deadline is enforced *at
  turn boundaries*, not mid-tool. For the reported bug,
  which was model-ask-tool-ask-tool latency, this is
  fine. For long-running tools it's a weaker guarantee
  than the spec implies. A follow-up that plumbs the
  deadline into the tool dispatcher (propagate via
  `CancellationToken`) would close the gap.
- **`Error::Timeout` vs client disconnect.** When the
  deadline fires, the server emits `Error::Timeout`. If
  the client has also been ticking down its own timeout
  and disconnected first, the server will emit the error
  to a dead channel. Benign, but logs accumulate
  "pipe broken after timeout". Emitting a terminal event
  and then flushing is OK; the order in the current
  handler is `emit_error` → `close_session`, which is
  correct.
- **Clock for test harness.** `Instant::now()` makes the
  timeout code hard to fast-forward in tests. The new
  `tests/timeout.rs` uses `tokio::time::pause()` +
  `advance()` to simulate, which works for most
  scenarios — but only for `tokio::time`-based awaits.
  If the model client uses `std::thread::sleep`
  anywhere (unlikely in this codebase, but worth a
  grep), the test-time advance won't move it. A wrapping
  `Clock` trait with a `Tokio` and `Mock` impl would be
  a cleaner abstraction if more timing code lands later.
- **No metric emitted on timeout.** Timeouts are
  operational signal. An incremented `session.timeout`
  counter (or at minimum a structured `tracing::warn!`
  with the session id and remaining-at-start) would let
  operators see when timeouts are firing without
  grepping logs.

# Suggestions

1. Propagate the deadline through tool dispatch via a
   `CancellationToken` scoped to `session.deadline`. That
   closes the long-running-tool gap cited above.
2. Add a `tracing::warn!` on the timeout branch with the
   session id, elapsed time, turn count, and last tool
   name. Operators will want this.
3. Consider exposing `remaining()` to the model loop so
   the model itself can make timing-aware choices — "I
   have 2s left, skip exploratory tool calls". This is a
   bigger change (protocol-visible) but worth
   prototyping.

# Verdict

**Ship it.** Correct deadline semantics, monotonic clock,
tests that actually exercise the boundary, and a clear
fix for a user-visible bug. The long-running-tool case is
a known limitation, not a regression.

# What I learned

The distinction between *per-iteration* and *session*
timeouts is load-bearing in any loop that wraps an
unbounded-turn interaction. The failure mode when you
only have the per-iteration version is always the same:
"each step was fast, the total was unbounded". LSP
servers, actor systems, retry ladders, streaming
transcoders, polling workers — same shape. The fix, every
time, is to materialize the deadline once at the start of
the session, consult it at every step, and honor
whichever is tighter between step-bound and remaining-
session. And: use `Instant`, not `SystemTime`. Every
codebase has one subtle bug from getting that wrong,
waiting to fire on the next DST transition.
