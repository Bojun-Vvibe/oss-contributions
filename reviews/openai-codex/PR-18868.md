# Context

`codex` runs each agent session inside a sandbox that includes a
local network proxy. The proxy's job is to mediate every HTTP and
HTTPS connection the agent's tools attempt — block disallowed
hosts, log audit events, and (in `Limited` mode) optionally MITM
HTTPS traffic so it can enforce HTTP-method policies that would
otherwise be invisible behind a `CONNECT` tunnel. The MITM
machinery already exists; what this PR adds is a **per-host
request-rewriting layer** layered on top: declarative TOML config
that says "for `api.github.com`, on `POST /repos/...`, strip the
`User-Agent` header and inject an `Authorization` header from
`$GITHUB_TOKEN`."

The motivating use cases are exactly what coding agents need:
- Inject API tokens from the host's secret store rather than from
  the model's context (the model never sees the secret).
- Strip headers the model added that the host doesn't want
  reaching the upstream (telemetry headers, agent fingerprint).
- Force-MITM specific hosts even when the proxy is in `Full` mode
  (which normally tunnels CONNECT without inspection), so that the
  per-host hook actually applies.

The PR is +1789/-28 across 15 files. The new module
`network-proxy/src/mitm_hook.rs` is ~1045 lines (the bulk of the
diff), with `http_proxy.rs` getting the integration into the
CONNECT path. Tests in `mitm_tests.rs` (extensive, this is a
security-critical surface). README updated.

# Diff walkthrough

**The configuration shape.** The new `MitmHookConfig` declares a
hook per (host, matcher, actions) tuple:

```rust
pub struct MitmHookConfig {
    pub host: String,
    #[serde(rename = "match", default)]
    pub matcher: MitmHookMatchConfig,
    #[serde(default)]
    pub actions: MitmHookActionsConfig,
}

pub struct MitmHookMatchConfig {
    pub methods: Vec<String>,
    pub path_prefixes: Vec<String>,
    pub query: BTreeMap<String, Vec<String>>,
    pub headers: BTreeMap<String, Vec<String>>,
    pub body: Option<MitmHookBodyConfig>,
}

pub struct MitmHookActionsConfig {
    pub strip_request_headers: Vec<String>,
    pub inject_request_headers: Vec<InjectedHeaderConfig>,
}

pub struct InjectedHeaderConfig {
    pub name: String,
    pub secret_env_var: Option<String>,
    pub secret_file: Option<String>,
    pub prefix: Option<String>,
}
```

The shape is principled. The `host` is exact-match (no wildcards
at this layer), the matcher is a conjunction of constraints
(method *and* path-prefix *and* query *and* header), and the
actions are a small declarative set (strip headers, inject
headers). The `body` field is reserved (validation explicitly
returns "not yet supported"), which is the right way to land a
schema with future-extension intent.

The `InjectedHeaderConfig` *requires exactly one* of
`secret_env_var` or `secret_file` (validated explicitly); both or
neither is rejected. Critically, this means an injected header's
value is **never** in the config file — only a *reference* to
where to fetch it at runtime. That's the right secret-handling
posture for a config that may be checked into a repo.

**The two-phase compilation.** `validate_mitm_hook_config` runs
first and returns descriptive errors for every shape violation
(empty host, empty methods, empty path_prefixes, invalid header
names, etc.). Then `compile_mitm_hooks_with_resolvers` does the
actual `BTreeMap<String, Vec<MitmHook>>` build, with two
injectable resolvers (env var lookup, secret-file read) — the
test suite uses fake resolvers, the prod path uses the real ones.
This dependency-injection shape is great for testing and makes
the secret-resolution boundary explicit.

**The matcher language.** Both `path_prefixes` and value-
matchers (in `query`/`headers`) accept three forms:
- `"foo"` — literal
- `"literal:foo"` — explicit literal (escape hatch for a string
  that starts with `pattern:` or `literal:`)
- `"pattern:foo*"` — glob, compiled via the `globset` crate

The `compile_glob_matcher` uses `literal_separator: true` for
paths (so `pattern:/repos/*` does NOT match `/repos/foo/bar` —
correct, slash is a path separator) and `literal_separator: false`
for header/query values (so `pattern:Bearer *` matches the whole
remaining value — correct). Two different separator semantics for
two different domains, both correct.

**The runtime evaluation.** `evaluate_mitm_hooks` returns a
three-state result:

```rust
pub enum HookEvaluation {
    NoHooksForHost,        // host not in the config
    Matched { actions },   // host matched and a hook fired
    HookedHostNoMatch,     // host has hooks but none matched this request
}
```

The third state is the critical one — see Risks. It exists
because the `http_connect_accept` integration uses
"host-has-any-hooks?" to decide whether to force-MITM the
CONNECT tunnel, but the hooks themselves can still decline to
apply. In the `Full` mode, a CONNECT to a hooked host that
doesn't actually match any hook will be MITM'd (so request
inspection happens), then forwarded through unchanged. That's the
right "fail-open" choice for a host that the user opted into
hooking but for a request the user didn't write a hook for.

**The CONNECT-path integration.** This is the most subtle part of
the diff:

```rust
let host_has_mitm_hooks = match app_state.host_has_mitm_hooks(&host).await {
    Ok(has_hooks) => has_hooks,
    Err(err) => {
        error!("failed to inspect MITM hooks for {host}: {err}");
        return Err(text_response(StatusCode::INTERNAL_SERVER_ERROR, "error"));
    }
};
let connect_needs_mitm = mode == NetworkMode::Limited || host_has_mitm_hooks;

if connect_needs_mitm && mitm_state.is_none() {
    // CONNECT needs MITM whenever HTTPS policy depends on inner-request inspection,
    // either for limited-mode method enforcement or for host-specific MITM hooks.
    ...
    return Err(blocked_text_with_details(REASON_MITM_REQUIRED, &details));
}

req.extensions_mut().insert(ProxyTarget(authority));
req.extensions_mut().insert(ConnectMitmEnabled(connect_needs_mitm));
req.extensions_mut().insert(mode);
if connect_needs_mitm && let Some(mitm_state) = mitm_state {
    req.extensions_mut().insert(mitm_state);
}
```

The change in semantics: previously, MITM was only required in
`NetworkMode::Limited`. Now it's required in `Limited` *or* on
any host that has a hook configured (regardless of mode). This is
correct: a hook can't fire on a tunneled CONNECT since the proxy
can't see the inner request. The new `ConnectMitmEnabled`
extension carries this decision down to `http_connect_proxy`,
which previously checked `mode == NetworkMode::Limited` directly
and now checks the carried flag — so the two ends of the CONNECT
flow stay in sync. Without this signal you could have a
race-prone bug where the accept side decided MITM was needed but
the proxy side didn't actually do it.

**The fail path on missing MITM state.** If the config requests a
hook on a host but MITM state is unavailable (CA not configured,
TLS terminator not initialized), the CONNECT is *blocked* with
`REASON_MITM_REQUIRED` rather than silently passed through.
That's the right fail-closed posture for a security feature.

# Risks / things a maintainer would catch

1. **`evaluate_mitm_hooks` returns `HookedHostNoMatch` for a
   hooked-host non-matching request — the request then proceeds
   un-modified.** That's intentional but it means the hook is
   effectively "best-effort": a malicious or buggy model can
   craft a request that *almost* matches the hook (different
   path, different method) and bypass the inject/strip behavior.
   For the *strip* action this is dangerous: a user who wrote
   `strip Cookie on POST /api/*` expects Cookies to be stripped
   from *all* sensitive traffic to that host, but a `GET /api/*`
   would slip through unmodified. Documentation should make this
   explicit: matchers are *whitelist*, not *blacklist*. A `default
   action: deny` mode would close the gap.
2. **No body inspection.** The PR explicitly defers `body` to a
   future release, but the implication is that a hook can't
   distinguish two POSTs to the same path with different
   payloads. For inject-token use cases that's fine, but for
   strip-PII cases the absence is meaningful.
3. **Glob `literal_separator` for paths is correct but
   `pattern:/repos/**` (double-star) behavior is not exercised
   in the diff I see.** `globset` supports `**` as
   "any-including-separators", which is what users will want for
   `/repos/**/issues`. Tests should lock this down.
4. **Host equality is exact (`normalize_hook_host` strips
   wildcards and rejects them).** That's deliberately
   conservative but means a user who wants to hook
   `*.googleapis.com` has to enumerate every subdomain. A
   future enhancement should consider `*.example.com` shape (DNS-
   wildcard semantics), with the same security trade-offs as
   browser cookie wildcards.
5. **`secret_file` requires absolute path.** Good. But there's
   no permission check on the file itself — a world-readable
   secret file is silently accepted. A warning ("secret_file is
   world-readable, recommend chmod 600") would be cheap.
6. **`secret_file` is read once at compile time.** A rotated
   secret on disk doesn't affect a running proxy until the proxy
   reloads its config. For long-running agent sessions this can
   surprise users. Either read-on-each-injection (with caching)
   or document the rotation behavior.
7. **`HeaderValue::from_str(&format!("{prefix}{secret}"))` will
   fail with an obscure error if the secret contains
   header-invalid bytes (CR, LF, NUL, high-bit chars).** The
   error from `HeaderValue::from_str` is helpful but the user
   may not realize their secret-file's trailing newline is the
   problem. The code does `value.trim().to_string()` in the
   default file resolver, which handles trailing newlines, but a
   secret with a literal newline in the middle would fail. A
   pre-check on the secret value with a clear "header values
   cannot contain CR/LF" error would be friendlier.
8. **Injected header overwrites silently.** If the original
   request already has an `Authorization` header, the inject
   action replaces it without warning. That's almost always the
   intent, but a user testing might wonder why their tool's
   token isn't being used. An audit-event field saying
   "overwrote pre-existing header" would help debug.
9. **`host_has_mitm_hooks` is `async` — and called once per
   CONNECT.** If the underlying lookup is `O(N)` over hook
   count, large config = slow proxy. Probably fine for the
   target deployment scale but worth a benchmark.
10. **The `if connect_needs_mitm && let Some(mitm_state) =
    mitm_state` Rust syntax (let-chains)** requires Rust 2024 /
    a recent toolchain. Worth confirming the project's MSRV
    permits this.
11. **README updates aren't reviewed in this batch but are
    advertised.** The matcher language ("literal:" / "pattern:"
    prefix syntax) is novel — users will rely heavily on the
    README to get it right. If the README isn't extensive on
    examples, expect a long tail of misconfiguration tickets.
12. **`McpToolCall` and other action types aren't relevant
    here**, since this PR is purely network-layer. But the
    interaction with the existing `guardian` (which can deny
    network actions) isn't shown in the diff: does the guardian
    see the *injected* request or the *original*? Almost
    certainly the original (guardian runs before the proxy), but
    the system test should assert this.

# Suggested questions for the author

1. Is the "matcher whitelist" semantics intentional, or should
   there be a per-host "default-deny" mode that blocks any
   request that doesn't match an explicit hook? The current
   design is fine for *augment* use cases but not for *enforce*
   ones.
2. The `secret_file` is read once at compile time. Has rotation
   been considered, either via re-read-on-injection (with mtime
   caching) or via SIGHUP-based config reload?
3. Will `pattern:/repos/**` be supported (double-star meaning
   "any path including separators")? Worth a test if so, and
   worth explicit documentation either way.
4. The injected header silently overwrites pre-existing values.
   Should the audit log record both the injected source and the
   overwrite event?
5. With the hooks table consulted on every CONNECT, has there
   been a perf benchmark on configs with many hooks (say, 100
   hosts × 5 hooks each)?
6. The interaction with the guardian's "approve/deny network
   action" path: does the guardian see the original request
   (pre-strip, pre-inject) or the modified one? This matters for
   audit fidelity.

# Verdict

**Approve, with documentation follow-ups.** The code itself is
high-quality: dependency-injected secret resolvers, three-state
evaluation result, fail-closed CONNECT handling when MITM state
is missing, exact-host matching, explicit reservation of the
unimplemented `body` field. The most important issue is
*semantic*, not *code*: users will think of strip-actions as
*defensive* (preventing leaks) but the matcher-whitelist shape
makes them *opportunistic*. That gap will produce bug reports
and a follow-up PR adding default-deny mode. The README needs
extensive examples; otherwise this is the most security-mature
proxy hooks design I've seen in any agent stack.

# What I learned

Two reusable insights, both about secret-handling design.

**First**, the "config references secrets, never embeds them"
discipline is the right default for any agent-config schema.
This PR's `InjectedHeaderConfig` is a great template: the field
is *either* `secret_env_var: String` (a name to look up) *or*
`secret_file: AbsolutePath` (an absolute path to read), and the
validator rejects both-or-neither. That structurally prevents the
"oh I'll just paste the API key into my .toml for testing" foot-
gun. The XOR pattern enforced by validator (rather than `oneOf`
in JSON schema) is appropriate for Rust config; the same shape
works equally well in JSON-Schema-driven systems via
`oneOf`+`required`. Either way, the principle is: the *only*
place a secret value can live is in the resolver's return value
at runtime, never in the config artifact.

**Second**, the dependency-injection of resolvers
(`compile_mitm_hooks_with_resolvers<EnvFn, FileFn>`) is the
right shape for any code that touches the environment or the
filesystem. The prod path closes over `env::var` and
`fs::read_to_string`; the test path closes over fake hashmaps
and fake file contents. The boundary is *one function call* —
not a trait, not a struct, not a registry. That's the minimal
form of dependency injection in Rust and it pays for itself the
moment you write the first test. Compare to a more elaborate
`SecretResolver` trait with `EnvSecretResolver` and
`FileSecretResolver` implementations — the trait gives you
nothing extra here because there's no pluggable-third-impl use
case. Function-typed parameters are underrated.

The third meta-lesson: when a security feature has a
"can't-do-the-thing-without-X" precondition (here: can't do
host-hooks without MITM state), **fail closed at the precondition
check, not at the hook-evaluation moment**. The CONNECT-path
integration does exactly this: if the host has any hooks but
MITM state is unavailable, *block the CONNECT entirely*, don't
silently let it through and skip the hook. That preserves the
invariant "if you configured a hook, you can trust it ran or you
got an error". Silently skipping security features is the
canonical way to undermine a config-file-driven security model.
