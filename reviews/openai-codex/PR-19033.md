# Context

`codex` sessions carry two adjacent state objects that are easy
to confuse:

- `SessionConfiguration` — the *live* session-wide policy
  (approval mode, sandbox policy). Mutates when the user changes
  permissions mid-session via `/approval` or equivalent.
- `original_config_do_not_use` / `per_turn_config` — snapshots
  taken at session start (or at turn boundaries), kept around
  for replay/restore semantics. The name itself begs the
  question.

The MCP (Model Context Protocol) connection manager is a long-
lived per-session object that holds the *current* approval and
sandbox policy and uses them to decide whether to elicit user
confirmation for tool calls or to auto-accept. It needs to be
re-synced any time the policy changes.

# Problem

Concrete repro from the PR description: start a session in a
restrictive permission mode, switch to Full Access mid-session,
then trigger an MCP tool call. The expected behaviour is "Full
Access → empty elicitation auto-accepts → tool runs". The
observed behaviour was "tool call denied".

Root cause: two cross-wired reads from the wrong config object.

```rust
// session/mcp.rs:229 (before)
&turn_context.config.permissions.approval_policy,

// session/turn_context.rs:615 (before)
mcp_connection_manager
    .set_sandbox_policy(per_turn_config.permissions.sandbox_policy.get());
```

Both `turn_context.config` and `per_turn_config` are *snapshots*
taken before the user toggled to Full Access. The MCP coordinator
got synced from the stale snapshot, so it kept the old (restrictive)
policy even though the live `SessionConfiguration` had been updated.
The result: empty elicitations that should auto-accept under Full
Access were being denied under the stale policy.

This is a classic "two near-identical names, one fresh, one stale,
written by different authors at different times" bug. It is
essentially undetectable by reading either file in isolation —
both the `mcp.rs` line and the `turn_context.rs` line *look*
correct.

# Design

Two-character fix, one in each file:

```rust
// session/mcp.rs:229 (after)
&turn_context.approval_policy,

// session/turn_context.rs:615 (after)
mcp_connection_manager
    .set_sandbox_policy(session_configuration.sandbox_policy.get());
```

`turn_context.approval_policy` is the *resolved* live policy on
the turn context (already pulled from the live
`SessionConfiguration` by the time the turn is constructed).
`session_configuration.sandbox_policy` is the live session-wide
sandbox policy, not the snapshot.

The fix is correct because both replacements read from the same
"live" surface, removing the asymmetry where the *approval*
side was being synced from the snapshot but the *sandbox* side
was being synced from a *different* snapshot. Now both go
through the live session configuration in lockstep.

# Risks

- **No regression test.** The PR's test plan is `cargo test
  -p codex-core --lib mcp_tool_call::tests` — that exercises
  the MCP path but doesn't reproduce the specific
  start-restrictive → switch-to-Full-Access → call-tool
  sequence. Without a test, the next refactor that re-introduces
  `turn_context.config.permissions.approval_policy` (an easy
  mistake — both `turn_context.config` and `turn_context`
  itself expose `approval_policy`) will silently re-introduce
  the bug.
- **`original_config_do_not_use` is still in the codebase.**
  Its mere presence is a footgun: every author has to remember
  "don't read from this for live policy". The fix removes two
  uses, but doesn't address the underlying naming/architecture
  problem. A follow-up to either (a) rename it more
  aggressively (`config_at_session_start_do_not_use_for_live_policy`),
  (b) wrap it in a newtype that doesn't expose
  `.permissions.approval_policy` directly, or (c) delete it
  entirely if no remaining caller needs it, would prevent the
  next instance.
- **Sandbox policy and approval policy are two different live
  reads in this fix** (`turn_context.approval_policy` vs
  `session_configuration.sandbox_policy.get()`). Symmetric
  would be cleaner — both from `session_configuration` or both
  from `turn_context`. The asymmetry is harmless today but
  makes the call sites harder to reason about as "this is the
  live policy sync block".
- **Behavioural change is silent for existing sessions** that
  *don't* switch policy mid-session — they were already
  reading the right thing because snapshot == live in that
  scenario. So the upgrade is safe by default but the fix
  surface is invisible to anyone who doesn't toggle
  permissions mid-session.

# Suggestions

1. Add an integration test under
   `codex-rs/core/tests/` that exercises the
   start-restrictive → switch → call-MCP path end-to-end and
   asserts the tool runs. This is the only test shape that
   actually pins the fix.
2. Audit the rest of the codebase for any other reads of
   `turn_context.config.permissions.*` or
   `*_per_turn_config.permissions.*`. Each one is a
   candidate for the same bug.
3. Consider replacing `original_config_do_not_use` with a
   newtype `SessionStartSnapshot` that exposes only the
   fields legitimately needed at later replay (history,
   instructions) and *does not* expose `permissions` at all.
   Compile-time prevention of this bug class.
4. Make the MCP coordinator `set_*` calls consume a single
   `&LivePolicies` struct rather than two separate
   `set_approval_policy(...)` / `set_sandbox_policy(...)`
   calls. That way the call site can't accidentally pass
   approval-from-live + sandbox-from-snapshot.

# Verdict

**Ship it.** Two-line fix to a real, reproducible permission-
escalation regression. The fix is correct. The follow-up work
(test, naming, architectural prevention) is non-blocking but
should be filed as separate issues immediately so the next
refactor doesn't undo this.

# What I learned

This is the canonical "stale snapshot vs live state" bug, and
it's worth memorizing the failure mode because it recurs in
*every* long-lived session/connection codebase. The shape is
always the same:

1. There's a long-lived object (here: `MCPConnectionManager`).
2. It caches some policy that can change over the session
   lifetime.
3. There are *multiple* code paths that re-sync it — one per
   policy change, one per turn boundary, one per replay, etc.
4. Some of those re-sync paths read from a snapshot (because
   they were originally written for a code path where snapshot
   == live), and some read from the live state.
5. As soon as the user does the thing that makes
   snapshot != live, *some* of the re-sync paths apply stale
   data and the cached policy on the long-lived object is now
   inconsistent.

The cure is structural: the long-lived object should expose
*one* re-sync entry point that takes a `&LivePolicies` view,
and the type system should make it impossible to construct
that view from a snapshot. Removing the footgun entirely is
much higher leverage than fixing each call site by hand. The
two-line patch is the right *immediate* fix; the architectural
fix is the right *next* PR.
