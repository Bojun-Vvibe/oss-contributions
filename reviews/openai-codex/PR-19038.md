# Context

`codex` ships *managed* feature flags that come from two channels:
the user's local config (`features = { ... }`) and a remote
"requirements" surface (`feature_requirements`) that lets a fleet
admin pin features on/off across a deployment. Until this PR,
unknown keys in `feature_requirements` produced a hard
`InvalidData` error at config-load time. The same shape of
unknown keys in the *config-side* `features` block already
fail-open with a warning. The asymmetry is the bug.

# Problem

Two real-world failure modes followed from the asymmetry:

1. **Forward-incompatible deployments.** A fleet admin pushes a
   `feature_requirements` containing a flag that newer codex
   builds know about but older builds don't. Older clients that
   haven't picked up the rolling update simply *fail to start*.
   That's a catastrophic UX for what should be a forward-compatible
   surface — the whole point of feature flags is graceful
   degradation.
2. **Legacy alias rejection.** `collab` was the old key for
   `multi_agent`. Configs in the wild still carry `collab = true`.
   Pre-PR, those configs hard-failed on load. Users had to
   manually rewrite their config (often without realizing why
   the new build refused to start).

The fix unifies the two surfaces: unknown keys *warn and ignore*,
legacy aliases *warn and translate*. Same behavior, both paths.

# Design

The change rotates `parse_feature_requirements` from
`-> Result<BTreeMap<...>>` to `-> BTreeMap<...>` (it can no
longer fail) and threads an optional `&mut Vec<String>`
(`startup_warnings`) through it:

```rust
fn parse_feature_requirements(
    feature_requirements: FeatureRequirementsToml,
    source: &RequirementSource,
    mut startup_warnings: Option<&mut Vec<String>>,
) -> BTreeMap<Feature, bool> {
    for (key, enabled) in feature_requirements.entries {
        if let Some(feature) = canonical_feature_for_key(&key) {
            pinned_features.insert(feature, enabled);
            continue;
        }
        if let Some(feature) = feature_for_key(&key) {
            push_feature_requirement_warning(
                &mut startup_warnings,
                format!("Using legacy `features` requirement `{key}` from {source}; \
                         prefer canonical feature key `{}`", feature.key()),
            );
            pinned_features.insert(feature, enabled);   // <- still applies
            continue;
        }
        push_feature_requirement_warning(
            &mut startup_warnings,
            format!("Ignoring unknown `features` requirement `{key}` from {source}"),
        );
    }
    pinned_features
}
```

Two distinct warning shapes, two distinct semantics:
- **Legacy alias** → translate to canonical, apply, warn.
- **Unknown key** → drop, warn.

The `Option<&mut Vec<String>>` plumbing exists because some
call sites (`validate_explicit_feature_settings_in_config_toml`)
don't have a warnings sink — the function is called for
*validation*, not load. Passing `None` there is correct: that
path didn't surface warnings before either, and isn't the
right place to start.

The two new public constructors —
`from_configured` (no warnings) and
`from_configured_with_warnings(&mut Vec<String>)` —  are a clean
way to evolve the API without breaking existing callers.
Internally they both delegate to a private
`from_configured_with_optional_warnings(Option<&mut Vec<String>>)`.

The `tracing::warn!` *in addition to* the `startup_warnings.push`
is the right call. The `Vec` is for end-user-visible startup
banners; `tracing::warn!` is for ops logs. Same message, two
audiences.

# Risks

- **Translates "config rejected" into "config silently
  half-applies".** If a fleet admin typoes `colab` (one `l`),
  pre-PR the config was rejected and they'd notice immediately.
  Post-PR, it warns into a log they may not be reading and the
  feature simply isn't pinned. That's *better* than rejecting
  good configs because of one bad line, but the warning needs
  to be loud enough — and the test only asserts *substring
  presence*, not log surface.
- **Behavior change for fleet admins.** Anyone whose CI
  validates "this requirements file loads cleanly" by parsing
  and checking exit code now gets exit 0 with a warning instead
  of exit 1. If their CI greps for "ERROR" they're fine; if it
  only checks exit code, they lose the gate. Worth noting in
  release notes.
- **`startup_warnings` is now an unbounded vec.** A pathological
  requirements file with 10k unknown keys produces 10k warnings.
  Realistically this is fine (config files are bounded), but
  some defensive cap or dedup would harden it. Not a blocker.
- **The legacy-alias message names the canonical key but not
  *when* the alias will be removed.** If the project plans to
  drop `collab` entirely in version N+2, the warning should say
  so. Otherwise the warning gets ignored forever and the alias
  becomes permanent.
- **No test for the *combined* case** where one requirements
  file contains a canonical key, a legacy alias, and an unknown
  key all at once. Order-dependent bugs in the loop body would
  slip through. Cheap to add.

# Suggestions

1. Add a "deprecated since vN, will be removed in vN+2" sentence
   to the legacy-alias warning. Forces the migration calendar
   into the message itself.
2. One combined-case test: canonical + legacy + unknown in the
   same `feature_requirements` block, assert all three warnings
   appear and the canonical+legacy keys land in `pinned_features`.
3. Consider rate-limiting / deduping the warnings vec at insert
   time (no need to push the same "Ignoring unknown
   `made_up_feature`" twice if the requirements file repeats
   it).

# Verdict

**Ship it.** Removes a forward-incompatibility footgun, unifies
two adjacent surfaces under the same fail-open policy, and
preserves operator visibility through both `tracing` and the
`startup_warnings` banner. The `Option<&mut Vec<String>>`
threading is mildly ugly but appropriate for the call-site
diversity.

# What I learned

The pattern "two parallel input surfaces with subtly different
strictness policies" is a quiet UX killer. Users rarely
distinguish "this came from my config file" from "this came
from my fleet's requirements file" — they just see "codex
won't start". Any project that has multiple config inputs
(local file, env vars, remote requirements, CLI flags) needs
a *consistent* unknown-key policy across all of them, and
that policy should almost always be **warn and continue**, not
hard-fail. Hard-fail is correct for *malformed* input
(`InvalidData`); warn-and-continue is correct for *unknown*
input (forward compatibility). This PR draws that line
correctly.

The other reusable bit is the dual-sink warning pattern:
push to a `Vec<String>` for the user-visible banner *and*
emit a `tracing::warn!` for the ops log. Same message, two
audiences, no risk of one going stale relative to the other
because they share a single helper function. Cheap to
replicate anywhere a startup-time validation produces
non-fatal complaints.
