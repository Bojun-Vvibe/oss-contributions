# Context

`codex` has two parallel agent-orchestration paths: a legacy
"agents" subsystem with `max_threads` as a per-config knob, and a
newer `multi_agent_v2` feature whose lifecycle does not honor
`max_threads`. The two were silently coexisting in `Config::load`:
turning on `multi_agent_v2` *and* setting `agents.max_threads`
produced a config that loaded cleanly but where the
`max_threads` value did nothing. Users who had set
`max_threads = 4` and then opted into `multi_agent_v2` saw their
limit silently ignored, were confused that v2 was over-spawning,
and filed reports that read like "v2 doesn't respect my thread
cap".

# Problem

The bug is structural, not behavioral: it isn't that `max_threads`
is *enforced wrong* — it is that the config schema accepts two
mutually-exclusive options without saying so. Anyone reading the
docs who follows "set `max_threads` if you want a thread cap" and
then "enable `multi_agent_v2` for the new lifecycle" produces a
nonsensical configuration that the loader cheerfully accepts.

The right shape for this kind of "two things should never both be
set" constraint is a config-load-time validation — fail early,
fail loud, with a message that names *both* knobs and which one
to remove.

# Design

Seven lines in `Config::load`:

```rust
let agent_max_threads_from_config =
    cfg.agents.as_ref().and_then(|agents| agents.max_threads);
if features.enabled(Feature::MultiAgentV2)
    && agent_max_threads_from_config.is_some() {
    return Err(std::io::Error::new(
        std::io::ErrorKind::InvalidInput,
        "agents.max_threads cannot be set when multi_agent_v2 is enabled",
    ));
}
```

The check runs *after* `features.enabled(Feature::MultiAgentV2)`
is computed, so it correctly accounts for the full feature
resolution pipeline (defaults + config + overrides + cloud
requirements). It runs *before* the existing
`agent_max_threads = cfg.agents.as_ref()...` line, so the bad
config never gets partially applied.

The error type (`InvalidInput`) is the right `io::ErrorKind` for
a user-supplied configuration violation; `InvalidData` would be
the wrong choice (that signals corrupted format, not policy
mismatch). The error message names exactly the two knobs and is
not framed as "and the system rejects this" — it is framed as
"this combination is invalid", which gives the user a clear
remediation: drop one or the other.

# Risks

- **No corresponding doc update.** A user reading the
  `multi_agent_v2` docs has no way to know `max_threads` is now
  actively rejected with this combination. The first encounter
  is the error message at startup, which is still much better
  than silent acceptance, but a sentence in the v2 docs would
  catch this earlier.
- **Asymmetric.** The check rejects `max_threads` in v2 but
  doesn't say *what to use instead*. The v2 lifecycle has its
  own concurrency knobs; the error message could point to them
  (`use multi_agent_v2.parallelism instead`, if such a knob
  exists). Without that, the user fixes the error by dropping
  `max_threads` and may still end up over-spawning, just now
  silently.
- **`agent_max_threads_from_config.is_some()`** treats *any*
  value (including `Some(0)`) as a violation. That's strictly
  correct (the user did set the knob), but a value of `0` is
  arguably a no-op already, so rejecting it is slightly
  user-hostile. The pragmatic answer is "if you set it
  explicitly, you meant something by it" — current behaviour is
  defensible.
- **Test coverage is good for the success path** (`cargo test
  multi_agent_v2_rejects_agents_max_threads`), but the PR
  description notes the surrounding crate suite has unrelated
  pre-existing failures. Worth flagging that the v2 enable +
  no-`max_threads` *positive* case is also covered by some
  test, otherwise a future "simplify the check" patch could
  remove this guard and regress.
- **No changelog entry.** Operators who discover this on
  upgrade will appreciate a CHANGELOG line — this turns a
  silent no-op into a hard error, which is a behavioural change
  on the strict reading.

# Suggestions

1. Add the changelog line, framed as
   "Configuration: `agents.max_threads` is now rejected when
   `multi_agent_v2` is enabled. Remove `max_threads` from your
   config or disable `multi_agent_v2`."
2. If the v2 lifecycle has its own concurrency knob, name it
   in the error message.
3. Audit the rest of `Config::load` for the same shape — knobs
   that are silently no-ops under certain feature combinations.
   These are all candidates for the same treatment.

# Verdict

**Ship it.** Seven lines of defensive validation that converts
a confusing silent failure into a loud config-time error. The
gold standard for this class of fix.

# What I learned

"Silent no-op when feature flags interact" is a recurring bug
class in any codebase with version-N and version-N+1 of a
subsystem coexisting in one config schema. The pattern always
looks the same: legacy knobs survive the schema migration,
they parse fine, the new code path simply doesn't read them,
and users who carry old config forward get behavior that
contradicts what their config file says. The fix is always a
small explicit check at config-load time.

The deeper insight is structural: as soon as you have
`feature_a` and `feature_b` where some config keys only apply
to one of them, the type-system-level fix is to *split the
config*. `agents.max_threads` should arguably live under
`agents.legacy.max_threads`, not at `agents.max_threads`,
specifically so that the v2 schema doesn't have a place to put
it. With a flat `agents.*` namespace shared by both, every
v2-incompatible knob requires a hand-coded reject like this
one. Splitting the namespace would let the type system enforce
the constraint mechanically — a v2 config struct simply
wouldn't have the field. Worth keeping in mind for the next
schema round.
