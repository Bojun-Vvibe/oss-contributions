# Context

`codex`'s `exec-server` runs shell commands as child processes
and streams their stdout/stderr back to the agent. The runner
multiplexes three event sources per process: the child's exit
status (via `wait`), the stdout pipe close, and the stderr pipe
close. When all three have fired the runner emits a final
`completed` event; until then the process is "live" from the
runner's perspective.

The previous code emitted `completed` as soon as the *exit
status* was observed — i.e. the moment `wait()` returned. That
seemed obviously correct (the child is dead; what else is
there?), but on Linux the kernel can deliver the SIGCHLD-driven
exit notification *before* the agent has drained the stdout
pipe of the last buffered chunk. The runner would emit
`completed`, the consumer would close the stream, and the last
1-4 KB of output — usually the most useful 1-4 KB, because that
is where the result is — would be lost.

The reporter found this on builds where the build tool prints
its summary line as the last thing before exiting; about 1 in
40 invocations dropped the summary line.

# Problem

The reordering between exit and pipe-EOF is not a bug in the
kernel; it's the documented behaviour. From `pipe(7)`: writes
to a pipe are buffered, and the read end will see EOF only
after the write end is fully closed AND the buffer is drained.
The child's exit closes the write end, but pending bytes
already in the kernel buffer remain readable until the reader
consumes them.

The previous runner treated `wait` as the synchronization
point. That is wrong any time the reader hasn't fully drained
the pipe — and the reader is on a separate task, so the runner
has no way to know.

# Design

The fix introduces an explicit "close handshake":

```rust
// runner.rs
let exit_status = child.wait().await?;
// Don't emit completed yet; wait for both pipe tasks to
// signal end-of-stream.
let _ = stdout_done.await;
let _ = stderr_done.await;
emit(Event::Completed { exit_status });
```

Where `stdout_done` and `stderr_done` are `oneshot::Receiver`s
sent from the spawned reader tasks when they observe EOF.

Three things worth noting:

1. **`wait()` still happens first.** This matters for cleanup
   correctness — the child is reaped, no zombie remains, and
   `exit_status` is captured. The change is purely the order
   of *event emission*.
2. **`oneshot`, not `mpsc`.** Each pipe signals exactly one
   event (EOF), so `oneshot` is the right channel. Using
   `mpsc` would have invited later confusion about whether
   multiple drain events were possible.
3. **No timeout on the close await.** The PR comment notes
   that the kernel-level pipe drain is bounded by the
   buffer size (typically 64 KB), so unbounded await is
   fine in practice. I'd still want a defensive timeout
   (see Risks) — but the omission is justifiable.

The added test `process_close.rs` spawns a child that prints
"summary\n" and immediately `_exit(0)`s. The test polls the
event stream and asserts the final `Stdout("summary\n")`
arrives strictly before `Completed`. Without the patch, the
test fails ~10% of the time on the CI runner (per the PR
description). With the patch, 1000 iterations are clean.

# Risks

- **Pathological producer holds the pipe open.** If the child
  forks a subprocess that inherits stdout and outlives the
  parent, the pipe doesn't close on parent exit. The runner
  would block forever in `stdout_done.await`. Today the
  exec-server is invoked for short-lived build commands so
  this is theoretical, but a `tokio::time::timeout` of, say,
  5 seconds with a `WARN` log on expiry would cap the worst
  case. The PR does not add this.
- **Cancellation paths.** If the agent cancels the run mid-
  flight, the runner's `child.kill()` path needs to also
  close the read sides of the pipes (or drop the receivers)
  so the `_done` awaits unblock. Looking at the diff, the
  cancellation path drops the receivers, which causes the
  `oneshot::Receiver::await` to resolve with `Err(_)` —
  hence the `let _ =` discards on both lines. Correct.
- **Order between stdout_done and stderr_done.** They are
  awaited sequentially. If stderr drains long after stdout,
  the runner blocks on stdout, then on stderr, total =
  max(stdout_drain, stderr_drain). That's fine; sequential
  await on independent oneshots is equivalent to `join!` for
  the wall-clock case where both have already fired.
  Switching to `tokio::join!` is a micro-optimization that
  wouldn't change behaviour but would make the
  "we want both" intent more obvious.
- **Reader-task panic.** If the reader task panics before
  signalling EOF, the oneshot is dropped and the await
  resolves with `Err(_)`. Today that just means we don't
  block forever — but we also don't *log* the panic. A
  warn-on-recv-error would help future debugging.

# Suggestions

1. Replace the two sequential awaits with
   `tokio::join!(stdout_done, stderr_done)` for clarity.
2. Add a configurable drain timeout (default: 5 seconds,
   override via env), so a runaway grandchild can't stall
   the runner. Log at `WARN` on expiry with the pid.
3. Differentiate the recv-error case from the recv-ok case
   in logs — today both are silent. A `tracing::debug!` on
   `Err(_)` makes it possible to spot reader-task panics.
4. Document the close-handshake invariant in the runner's
   module docstring. This is the kind of code that gets
   "simplified" by a future contributor who doesn't know
   why the await is there.

# Verdict

**Ship.** Correct fix, real bug, good test. The drain-timeout
gap is worth a follow-up but doesn't block this PR — current
exec-server use cases don't expose it.

# What I learned

The mental model that bites people here is "child process
exit means I/O is done". It doesn't, on Unix. Process exit
closes file descriptors held by the process, but data already
in pipe buffers remains in flight. The reader sees EOF only
after the buffer is drained, and the EOF is what tells you
"I have all the bytes" — not the exit status.

This is the second instance I've seen this month of a runtime
losing trailing output to early completion signaling (the
other was an LSP wrapper that closed its stdout consumer when
the language server's `exit` notification arrived, dropping
the last `publishDiagnostics`). The pattern is general:
*process / protocol exit ≠ stream EOF; treat them as
independent and synchronize on EOF for I/O completeness*.

For agent runtimes specifically this matters because the
"summary line" — the thing the agent most needs — almost
always sits at the tail of output. Losing the last 4 KB
silently is far worse than losing a random 4 KB in the
middle, because the user's first-order signal of "did this
work" is the exit code plus the last few lines.
