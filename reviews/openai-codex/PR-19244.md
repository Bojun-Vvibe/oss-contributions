# Context

The `codex-app-server` crate exposes the Codex backend over
multiple transports for clients (TUI, IDE plugins, the test
harness, third-party MCP-style clients). On Unix, the
transport historically opened a raw bidirectional stream over
a Unix domain socket — bytes in, bytes out, no framing
contract beyond what the application layer imposed. The TCP
and named-pipe transports, in contrast, already speak
WebSocket (specifically the standard HTTP `Upgrade: websocket`
handshake), giving them framing, ping/pong keepalives, and a
versioned protocol envelope for free.

# Problem

Two transports speaking different framings is a maintenance
hazard. Every new feature on the WebSocket side (binary
frames for tool blobs, close codes for protocol errors,
ping/pong for half-open detection) had to be re-implemented
or stubbed for the Unix path. Worse, clients that wanted to
work over either transport — increasingly the case as Codex
Apps grow — had to ship two code paths. A simple
operational symptom: if the TUI's process forks behind a
load balancer that drops idle connections at 60s, the
WebSocket path notices via missed pongs but the Unix path
just hangs.

# Design

Switch the Unix socket transport to perform the standard
WebSocket HTTP Upgrade handshake over the Unix stream. The
client opens the socket, sends an HTTP/1.1 `GET /` with
`Upgrade: websocket` and the standard
`Sec-WebSocket-Key` / `Sec-WebSocket-Version` headers; the
server responds with `101 Switching Protocols` and the
matching `Sec-WebSocket-Accept`; from there both sides
speak RFC 6455 frames. This is exactly what TCP already
does — the only difference is the underlying byte stream.

The diff is small (16 +, 10 −) because the bulk of the
WebSocket implementation already exists in the shared
transport layer; this PR is wiring the existing handshake
over the Unix `Stream` rather than implementing anything
new. The test file changes update the existing Unix
transport test to drive a real upgrade over the Unix
stream — a meaningful upgrade in test fidelity, since the
old test bypassed the framing entirely.

# Risks and edges

- **Backward compatibility for existing Unix clients.** Any
  client speaking the old raw-stream protocol over the
  Unix socket will break — the server now expects an HTTP
  upgrade as the first bytes. The PR description does not
  call out a transition path. If there are
  out-of-tree clients (third-party IDE plugins, MCP-style
  bridges, the user's own scripts) this is a breaking
  change that warrants either a version bump on the
  transport, a config flag to fall back to raw, or at
  minimum an explicit CHANGELOG line. Worth asking before
  merge.
- **`Sec-WebSocket-Key` over Unix is theatrical.** The
  handshake's CSRF-ish properties (Origin checks,
  Sec-WebSocket-Key proof-of-handshake) were designed for
  browsers crossing origins. Over a Unix socket bound by
  filesystem permissions there is no analogous threat;
  the handshake is pure protocol overhead. That is fine
  — the win is shared code with TCP, not security — but
  worth being explicit about so a future reviewer does
  not "optimize" the handshake away.
- **README scope.** The README update touches the Unix
  socket section. Worth checking that the TCP section
  doesn't also describe a distinction that no longer
  applies, so the docs land internally consistent.
- **Test coverage of close/ping behavior.** The PR
  switches *to* WebSocket but the test still appears to
  cover only the basic message round-trip. The whole
  point of unifying transports is to get
  ping/pong/close-code semantics on the Unix path; a
  follow-up test that proves a missed pong closes the
  Unix connection (matching TCP) would lock in the
  intended behavior.

# Verdict

Approve, conditional on either (a) confirmation that no
out-of-tree clients depend on the raw-stream protocol, or
(b) a CHANGELOG / migration note. The internal-consistency
win is real and the diff is appropriately small.
