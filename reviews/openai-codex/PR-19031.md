# Context

Codex talks to local MCP (Model Context Protocol) servers over
stdio: it spawns a child process, pipes JSON-RPC over its
stdin/stdout, and treats the child as a tool provider. MCP server
configs may specify a `cwd` for the child; if they don't, the
launcher needs a *fallback*. The launcher also has to resolve the
`command` field — which can be either an absolute path
(`/usr/local/bin/foo`), a name to find on `PATH` (`foo`), or a
relative path (`./scripts/foo`). The relative case is where this
PR lives.

The CLI and the Electron app both shell out through the same
`rmcp-client` crate, but they hand it different runtime
environments — most importantly different fallback cwds. The CLI
inherits the user's shell cwd; Electron has its own runtime cwd
that may be the app bundle root.

# Problem

When a server config omitted `cwd`, the local stdio launcher
resolved the `command` against the *process* cwd of whatever was
hosting `rmcp-client` — which for Electron is not where the user
expects. A config like:

```json
{ "command": "./bin/my-mcp-server" }
```

worked from the CLI (because the CLI's cwd was the user's project
root) and silently failed from Electron (resolved against the
Electron runtime cwd, file not found). The two surfaces silently
diverged on the same config.

# Design

Two coordinated changes:

**(1) Thread an explicit `fallback_cwd` through the launch path.**
`mcp_connection_manager.rs` and `app-server/src/codex_message_processor.rs`
now pass the runtime's notion of "where the user is" into the
launcher. `stdio_server_launcher.rs` accepts it and, when the
server config has no `cwd`, uses the fallback for both the
child's working directory *and* for resolving a relative
`command`. The change to `program_resolver.rs` (+13/-8) is the
smallest of the three: it accepts an optional base directory and
joins relative commands against it before falling back to `PATH`
resolution. Absolute paths and bare program names are unchanged.

**(2) Lock the contract with a focused regression test.**
`local_stdio_server_uses_runtime_fallback_cwd_when_config_omits_cwd`
is +63 LOC of test for what reads as a small fix. That ratio is
correct: this is exactly the kind of bug that comes back if the
parameter ever gets dropped at one of the four call sites. The
test sets up a relative-command server config with no `cwd`,
hands the manager a fallback cwd pointing at a temp dir
containing the executable, and asserts the child launches and
handshakes.

# Risks

- **Resolution order is now config.cwd → fallback_cwd → PATH.**
  Reasonable, but if a user's PATH happens to contain a binary
  with the same name as their relative command, the old behavior
  would find the PATH binary while the new behavior finds the
  relative one. That's the correct precedence — explicit
  relative path beats implicit PATH — but worth a one-line note
  in the MCP server config docs.
- **Fallback cwd is an additional parameter on a hot config-loading path.**
  The two existing callers were updated; any third caller
  added later (e.g., a future "test this MCP server" sub-command)
  has to remember to plumb it. No `Default` to fall back to —
  good — but future contributors will see `None` as a tempting
  shortcut. A `#[must_use]` on the launcher entry point, or
  wrapping the cwd in a newtype with no `Default`, would
  reinforce the discipline.
- **Tests that run with a fixed temp cwd may mask a real bug**
  if the production fallback ever resolves to something
  unexpected (e.g., `$HOME` instead of the project root). The
  *value* of the fallback is decided upstream and isn't covered
  here.

# Suggestions

1. Add a parity test that exercises the same config from both
   the CLI and the app-server entry points to guarantee the two
   surfaces stay aligned — the underlying bug was a parity bug,
   not a launcher bug.
2. Surface the resolved command path in launch logs (debug
   level). When a user reports "my MCP server isn't starting,"
   knowing which of the three resolution branches fired (config
   cwd, fallback cwd, PATH) collapses the diagnosis to one
   round-trip.
3. Consider rejecting relative `command` values when neither a
   config `cwd` nor a fallback cwd is supplied, with a
   structured error pointing at the config field — better than
   a "file not found" deep in the spawn path.

# Verdict

**Ship it.** The fix matches the bug shape (CLI/Electron parity
on relative resolution), the contract is locked in by a
targeted integration test, and the parameter plumbing is
explicit at every layer rather than fished out of a global.

# What I learned

When two frontends share a backend library but differ in the
ambient environment they hand it (cwd, env vars, file
descriptors), any backend code that *reads* from that ambient
environment instead of from explicit parameters is a parity bug
waiting to happen. The cure is the same as in #19046 above:
push the implicit dependency up into an explicit parameter. The
test budget should scale with the number of frontends, not the
size of the diff — a 22-line behavior change earned a 63-line
test, and that's the right ratio when the bug class is
"silent divergence between two surfaces."
