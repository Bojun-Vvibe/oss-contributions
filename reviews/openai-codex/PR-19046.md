# Context

`codex-exec-server` is the long-running process that hosts an
agent's exec capabilities — running shell commands, reading and
writing files — under a sandbox profile. Filesystem permissions
inside the exec-server are modeled by `FileSystemSandboxContext`,
which knows how to translate a high-level `PermissionProfile`
(or a legacy `SandboxPolicy`) into concrete allow/deny decisions
on absolute paths. The wrinkle is that any rule referencing a
"current working directory" — `:cwd`, `:project_roots`, a
relative deny glob — needs *some* concrete cwd to resolve
against. Where does that cwd come from?

This PR is a structural cleanup that follows up on #19016, which
fixed remote exec-server contexts so Docker-backed filesystem
requests resolve against the *request* cwd inside the container,
not the test runner's cwd on the host.

# Problem

`FileSystemSandboxContext::new(SandboxPolicy)` was an implicit
constructor: given a `SandboxPolicy`, it would synthesize the
missing cwd by calling `AbsolutePathBuf::current_dir()`. The
exec-server runs as a long-lived process whose `cwd` has nothing
to do with the request being served, so any cwd-dependent rule
silently resolved against ambient process state. In the test
suite that ambient state was the cargo runner's cwd. In
production it would be wherever the operator launched the
binary. Both are wrong.

Worse, the API shape *invited* the bug. The constructor took a
`SandboxPolicy` and returned a fully-formed context, which read
to a caller as "everything you need is here." A new caller adding
filesystem-handler logic would naturally reach for `::new` and
inherit the wrong cwd.

# Design

The PR removes `FileSystemSandboxContext::new` outright and
forces every call site onto one of three explicit constructors:

- `from_legacy_sandbox_policy(policy, cwd)` — explicit cwd.
- `from_permission_profile(profile)` — profile is self-contained,
  no cwd needed.
- `from_permission_profile_with_cwd(profile, cwd)` — profile plus
  request cwd for cwd-dependent rules.

The pivot is in `fs_sandbox.rs` (+56/-66) — net negative because
the synthesis path is gone, replaced by callers passing what they
already know. `server/file_system_handler.rs` is updated to
thread the request's cwd through; `environment.rs` and
`tests/file_system.rs` get the same treatment, but the test file
also stops routing through legacy `SandboxPolicy` projections —
it now constructs `PermissionProfile` values directly. That
matters: tests that go through legacy projections were the
primary place where the implicit `current_dir()` lookup got
exercised, and removing them removes the last camouflage for the
bug.

The environment regression test now uses an explicit *restricted*
profile with no synthetic cwd, which proves the new shape: a
profile that carries no cwd-dependent rules doesn't need a cwd.

# Risks

- **API break for downstream embedders.** Anyone outside the
  workspace using `FileSystemSandboxContext::new` will fail to
  compile. For a workspace-internal type with no public
  guarantees this is fine — and it's better to break the build
  than silently inherit a wrong cwd — but the changelog should
  call this out.
- **Profile-only path still silent.** `from_permission_profile`
  (no cwd) is correct for profiles that don't reference `:cwd`
  / `:project_roots`, but if a caller hands in a profile that
  *does* reference them, the resolution will fail at runtime,
  not at construction. A debug-assert at profile-load time, or
  a typed distinction (`Profile` vs `ProfileWithCwdRefs`),
  would catch that statically.
- **Net-zero LOC hides a behavior change.** A code-coverage diff
  won't flag this PR as risky, but the test that previously
  passed because of ambient cwd is now exercising a different
  code path. Reviewers should confirm the new tests actually
  cover the cwd-dependent rules the old tests were silently
  validating.

# Suggestions

1. Mark `FileSystemSandboxContext` non-`Clone` for one cycle to
   force any cached instance to be reconstructed against the
   current request — defensive but cheap.
2. Add a doc-comment on `from_permission_profile` (no cwd)
   listing the profile rule kinds that *require* a cwd, so the
   right constructor is obvious at the call site.
3. Long-term: collapse `from_legacy_sandbox_policy` once
   `PermissionProfile` is the only inbound shape. Leaving the
   legacy entry-point around indefinitely re-creates the
   "two ways to do the same thing" problem this PR is solving.

# Verdict

**Ship it.** The PR is exactly the right shape for an API
correctness fix: identify the implicit dependency, delete the
constructor that hides it, force every caller to pass it
explicitly. Net zero LOC, large reduction in the
"ambient state silently leaked into a security-relevant
decision" surface area.

# What I learned

A "convenience" constructor that synthesizes a missing parameter
from process global state is one of the highest-leverage
correctness bugs you can introduce. It looks like ergonomics; it
is actually a hidden coupling between a request-scoped decision
and a process-scoped value. The fix is almost always the same:
delete the constructor, make callers pass the value, and let the
type system enumerate the call sites for you. The hardest part
is convincing yourself the LOC delta of "+56/-66" is worth more
than a "+5/-2" patch that adds a `cwd: Option<...>` parameter —
it is, because the optional parameter just moves the implicit
synthesis one layer up.
