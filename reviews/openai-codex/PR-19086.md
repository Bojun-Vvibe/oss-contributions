# Context

Codex's app-server exposes a JSON-RPC surface where the agent core
asks app-server clients (the TUI, the IDE plugin, etc.) to approve
permission requests via `item/permissions/requestApproval`. The
permission shape has migrated through two generations: a legacy
`{read: [paths], write: [paths]}` projection and a richer v2 shape
called `entries`, where each entry is a structured
`FileSystemSandboxEntry` with `path`, `access_mode`, and special-path
support (e.g. `:cwd`, `:project_roots`). The two are kept in the
serialized payload together — `read`/`write` for compatibility,
`entries` as the canonical v2 form. All three fields are marked
optional in the schema so clients can negotiate which they consume.

# Problem

The legacy-to-v2 conversion happily populated `read` and `write` from
the canonical entries… and then forgot to populate `entries` when the
*reverse* projection was needed, i.e. when the request originated
from a legacy `SandboxPolicy`. So
`AdditionalFileSystemPermissions.entries` was silently `None` on
every legacy-origin permission request, even though the request was
genuinely asking for filesystem access.

The downstream consequence was bad. The app-server intersects the
client's *response* with the original *request* — anything the
response grants that wasn't in the request is dropped, on the
sound principle of "no escalation through approval." A v2-only
client that read `request.entries` to render the approval UI would
see "no filesystem access requested," return an empty grant, and
the intersection would deny everything. So a request meant to ask
"can the agent write to `/tmp/foo`?" would silently come back as
"no." The user would see the agent stall on filesystem ops with no
indication that the approval round-trip had silently zeroed out.

# Design

Two structural moves.

**(1) Populate `entries` in the v2 conversion.** In `v2.rs` the
`AdditionalFileSystemPermissions` builder now translates the
legacy `read`/`write` lists into a synthesized list of
`FileSystemSandboxEntry { path, access: Read | Write }` and stores
that on `entries`. Crucially the legacy `read`/`write` lists are
*also kept*, so a v1 client that only knows `read`/`write` still
gets a usable request. This is the right backwards-compat posture
and it is documented at the schema level: every `read` and
`write` field across the JSON schemas now carries `description:
"This will be removed in favor of `entries`."` — explicit deprecation
without yet breaking. The TypeScript shape gets the same JSDoc:

```ts
export type AdditionalFileSystemPermissions = {
  /** This will be removed in favor of `entries`. */
  read: Array<AbsolutePathBuf> | null,
  /** This will be removed in favor of `entries`. */
  write: Array<AbsolutePathBuf> | null,
  globScanMaxDepth?: number,
  entries?: Array<FileSystemSandboxEntry>,
};
```

**(2) Round-trip regression coverage.** Three new test sites:
the v2 conversion test in `protocol/v2.rs`, the app-server
`request_permissions_round_trip` test in
`request_permissions.rs`, and the TUI app-server-approval test in
`app_server_approval_conversions.rs`. Together they exercise the
conversion in the agent core, the app-server JSON-RPC round trip,
and the TUI render path. The TUI test
(`converts_request_permissions_into_granted_permissions`) is the
clearest expression of the contract: given a request with
`entries: [...]` and `read/write` populated, the TUI must produce
a grant whose `entries` round-trip back to the same set after
the app-server intersection.

# Risks

- **Two sources of truth for the same data.** During the
  deprecation window, `read/write` and `entries` must stay
  consistent. The conversion code does this in one direction
  (legacy → entries), but a contributor adding a new field to a
  `FileSystemSandboxEntry` (e.g. a glob filter, a deny flag) needs
  to remember that the legacy projection cannot represent it.
  Worth a `TODO(remove-legacy-readwrite)` marker at the
  conversion site.
- **Schema deprecation is via description text only.** JSON Schema
  has no `deprecated` keyword in the strict-2020-12 dialect (it
  exists in OpenAPI), so the `description` line is the only
  signal. A linter rule that fails on new uses of `read`/`write`
  in client code would harden the deprecation.
- **Conversion is cwd-blind.** The synthesized entries use
  literal paths from the legacy lists, not special-path
  references like `:cwd`. If the agent later needs to compare
  these grants against a request that *uses* `:cwd`, the literal
  path won't match the special path even though they refer to
  the same directory. This is a known limitation of the legacy
  shape (no cwd indirection), not a regression.
- **The TUI gets a partially-populated request shape.** During the
  rollout window, a new TUI build that only renders `entries`
  paired with an old core that emits `read`/`write` would still
  see empty entries — though this PR is exactly the fix for the
  inverse direction, so the bidirectional coverage matters.

# Suggestions

1. Add a `#[deprecated(note = "use `entries`")]` attribute on the
   Rust `read`/`write` fields so any in-tree consumer that still
   reads them gets a compile-time warning. Cheap and locks the
   migration end-state in.
2. Track removal of `read`/`write` in a follow-up issue with a
   target version, and update the schema description to include
   that version.
3. Consider adding a CI gate that diffs the generated JSON
   schemas against the previous published version — schema drift
   on deprecation-window fields is exactly the kind of accident
   that causes the rest of the deprecation to slip.

# Verdict

**Ship it.** The fix is structurally sound (populate the
canonical field in the request, keep the legacy projection for
compat), the schema-level deprecation messaging is in place, and
the test coverage spans the three layers (protocol, app-server
RPC, TUI render). The bulk of the +137 LOC is regenerated
fixtures, which is the right cost for this kind of protocol
change.

# What I learned

When a protocol is migrating from shape A to shape B, the most
dangerous moment is when the producer thinks "I'll fill in B,
and let consumers transition," but actually fills in A and leaves
B empty when A → B conversion is needed. The intersection logic
on the consumer side then silently zeroes out the intent. The
fix is a discipline: any time you hold a richer shape (B), emit
*both* the legacy shape (A) and the rich shape — never one or
the other. Then on the consumer side, prefer B if present, fall
back to A if absent. The deprecation lives in the schema
description and a linter, not in the conversion code, so
there's no chance of a future contributor "cleaning up" the
dual-write into a single-write that re-introduces the bug.
