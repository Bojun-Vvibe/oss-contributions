# Context

`codex` has been incrementally migrating its permissions model
away from a single flat `SandboxPolicy` enum (network on/off,
filesystem entries) toward a richer `PermissionProfile` that can
express *who is enforcing the sandbox*. The motivation is that
"sandbox" means three meaningfully different things in
production: Codex constructs the sandbox itself (Seatbelt,
Landlock), Codex disables the outer sandbox entirely (the
opt-out `DangerFullAccess` mode), or some upstream caller has
already arranged isolation and Codex should not double-sandbox
(`ExternalSandbox`, used by hosted runtimes).

# Problem

The pre-PR `PermissionProfile` shape was a flat record:
`{ network: Option<...>, file_system: Option<...> }`. That shape
cannot distinguish between the three enforcement modes — a
profile with `network: None, file_system: None` means
ambiguously either "no policy, full access" or "external sandbox
manages it." When profiles get round-tripped through the
app-server v2 API or persisted into rollout files, both
`DangerFullAccess` and `ExternalSandbox` were degraded to the
same lossy "all-Nones" record on the wire and could not be
faithfully reconstituted on the other side.

# Design

The PR splits `PermissionProfile` into a tagged union with three
variants:

```
ManagedPermissionProfile   { type: "managed",   network, fileSystem }
DisabledPermissionProfile  { type: "disabled" }
ExternalPermissionProfile  { type: "external",  network }
```

Within `Managed`, filesystem permissions are themselves a tagged
union: `RestrictedPermissionProfileFileSystemPermissions
{ type: "restricted", entries[], globScanMaxDepth }` vs
`UnrestrictedPermissionProfileFileSystemPermissions
{ type: "unrestricted" }`. The `glob_scan_max_depth` field is
typed `NonZeroU32` (`minimum: 1` in the JSON schema).

Partial-grant overlays — what was previously folded into the
same shape — get carved out into a separate
`AdditionalPermissionProfile` for command/session/turn-scoped
overrides. That's a consequential split: the analytics events
that used to type their `additional_permissions` field as
`Option<PermissionProfile>` now type it as
`Option<AdditionalPermissionProfile>`, and the rename ripples
through the `GuardianReviewedAction::{Shell,UnifiedExec,Execve}`
variants. This is the right shape (a partial overlay is not a
full policy), but it's the change with the largest blast radius
across the diff.

Three explicit compatibility moves stand out:

1. The deserializer accepts the legacy pre-tagged shape (no
   `type` field) and infers the variant. This keeps existing
   rollout files loadable.
2. `SandboxPolicy` (the legacy enum) remains available as a
   compatibility projection at the config/API boundary. The
   internal model is the new union; the boundary types are
   preserved.
3. `DangerFullAccess` round-trips as `disabled`,
   `ExternalSandbox` round-trips as `external`, and managed
   "unrestricted filesystem + restricted network" stays
   `managed` rather than being mistaken for `external`. The PR
   description names this last invariant explicitly, and the
   tests `permission_profile_file_system_permissions` and
   `preserving_deny_entries_keeps_unrestricted_policy_enforceable`
   nail it down.

The "preserve deny entries when projecting unrestricted" piece
is the most subtle bit: an "unrestricted" filesystem with deny
entries is not really unrestricted — it's `:root = write` plus
explicit deny rules. The projection back into runtime policies
preserves both, which is the only way `:root = write + deny
/etc` actually enforces in the sandbox layer.

# Risks and edges

- **2068 / 867 across 73 files is a large blast radius.** The PR
  is largely mechanical (rename + tag-thread + schema regen),
  but a refactor at the permissions boundary is exactly the
  shape where a small mis-thread becomes a security regression.
  The `analytics/events.rs` rename from `PermissionProfile` to
  `AdditionalPermissionProfile` is the right call; reviewers
  should still individually verify each `GuardianReviewedAction`
  variant's enforcement semantics didn't drift.
- **Schema-typed JSON wire change.** v2 schema files for
  `CommandExecParams`, `Thread{Start,Resume,Fork}{Params,
  Response}`, and `TurnStartParams` all change shape. Any
  external client wired against the v2 schema (which the PR
  description correctly flags as experimental) will need a
  coordinated update. The PR doesn't include a migration note
  or a version bump in the schema; if downstream consumers exist
  they need a heads-up.
- **`NonZeroU32` for `glob_scan_max_depth` is a quiet
  invariant.** The serde behaviour is "absent or NonZero"; a
  zero value coming over the wire would deserialize-fail rather
  than silently default. That's correct, but it also means a
  legacy rollout with `glob_scan_max_depth: 0` (if any exist)
  would refuse to load. A grep through historical rollout
  fixtures to confirm `0` never escaped would close that risk.
- **`DangerFullAccess` → `disabled` mapping merits a sanity
  test on the runtime side.** The PR has plenty of round-trip
  tests at the protocol level. What's worth adding: an
  end-to-end test that constructs a session with the new
  `disabled` profile, runs a `unified_exec`, and confirms no
  outer sandbox is layered on top. That's the property users
  actually care about, and the protocol-level round-trip
  doesn't directly assert it.
- **`ExternalSandbox` and `external` semantics.** The new
  `ExternalPermissionProfile` carries only `network`, not
  filesystem. The PR description says "filesystem isolation is
  enforced by an external caller." A reviewer should confirm
  that *all* internal call sites that previously decided
  filesystem behaviour from a flat profile now correctly
  short-circuit when the variant is `external` — a missed call
  site would silently apply Codex's filesystem rules on top of
  the external one, which is mostly harmless but defeats the
  purpose of declaring `external`.

# Suggestions

- Add a runtime end-to-end test for each enforcement variant
  (`managed`, `disabled`, `external`) confirming the actual
  sandbox behaviour matches the declared variant. Protocol
  round-trip tests are necessary but not sufficient.
- Bump the v2 schema version (if there's a versioning convention
  in this codebase) or add a CHANGELOG entry calling out the
  wire-shape change. Even for an experimental surface, a
  heads-up is cheap insurance.
- Consider extracting the "preserve deny entries on
  unrestricted" projection into a single dedicated function
  with a name that documents the invariant
  (`project_unrestricted_with_deny_entries`). Right now the
  invariant lives in the prose of one of the test names;
  promoting it to code makes it harder to silently break.
- The `AdditionalPermissionProfile` rename is sound, but a
  grep-and-comment pass at each `additional_permissions` field
  declaration noting "partial overlay, not a full policy" would
  help the next reader.

# Verdict

Approve with the runtime end-to-end test as the one suggestion I
would press hardest on. The protocol model is meaningfully
better — the previous shape was lossy in exactly the places that
matter for security (cannot distinguish "no sandbox" from
"external sandbox") and the new shape names the distinction
explicitly. The compatibility layer is well-considered and the
test coverage at the protocol layer is solid. The remaining work
is end-to-end assertions and downstream-client coordination.

# What I learned

When a permissions abstraction has multiple "I don't apply"
variants — disabled, external, none-supplied — collapsing them
into a single absent-fields shape is a security-relevant bug,
not a representation choice. The corollary: every time a type
has a representation that means "ambiguous between three
distinct behaviours," that ambiguity will eventually be
resolved by reading the field as whichever behaviour the reader
expected. Tagged unions exist precisely to refuse the
ambiguity at the type-system level. The cost is a one-time
mechanical refactor across 73 files; the benefit is that the
distinction can never again be lost in a round-trip.
