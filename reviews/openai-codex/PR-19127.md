# Context

`codex-rs` ships a `multi_agent_v2` mode where a parent agent can
spawn sub-agents to handle subtasks. Each sub-agent runs its own
turn loop with a forked copy of the parent's conversation history,
returns a final-channel response, and is reaped. The parent treats
the child's final response as the result of a `spawn_agent` tool
call.

When this feature first shipped, the design included a small block
of model-visible "you are a newly spawned agent" prose that was
appended to the child's `developer_instructions` at spawn time. The
block was wrapped in `<spawned_agent_context>...</spawned_agent_context>`
markers and explained the protocol: "you can spawn sub-agents",
"your final-channel response goes back to your parent", "the prior
conversation was forked", "treat the next user message as your
assigned task". The intent was to give a freshly-spawned agent a
self-explanation of its role.

This PR removes that block.

# Problem

The "explain the protocol to the spawned agent" idea is reasonable
in isolation, but as the multi-agent feature stabilised the block
turned out to be doing two things, only one of which was wanted:

1. **It changed the model's prior** for spawned children versus
   first-turn parents. A fresh model invocation that *should* be
   indistinguishable from a normal turn (just with a forked
   history) instead saw a developer message marking it as a
   "spawned agent in a team of agents". In practice this nudged
   the model into more orchestration-y behaviour even on simple
   spawned tasks where the parent already provided sufficient
   instruction.

2. **It bloated the developer instructions** of every spawned
   turn by ~600 characters of fixed prose, which inflates input
   tokens linearly with the spawn fan-out and cannot be cached
   against the parent's prefix (because the parent's developer
   instructions don't have it).

The PR description frames it as "the parent/configured developer
instructions should carry through normally, but the dedicated
`<spawned_agent_context>` block is no longer desired." The simpler
read: someone measured spawned-child behaviour with and without
the block, decided the no-block behaviour was at least as good,
and is now removing complexity.

# Design

The change is almost pure deletion:

- `codex-rs/core/src/context/spawn_agent_instructions.rs` — the
  whole 14-line `SpawnAgentInstructions` struct and its
  `ContextualUserFragment` impl, gone.
- `codex-rs/core/src/context/mod.rs` — the `mod` declaration and
  `pub(crate) use` re-export, gone.
- `codex-rs/core/src/tools/handlers/multi_agents_v2/spawn.rs` —
  the `if let Some(mut existing_instructions) = ...` block that
  appended `SpawnAgentInstructions.render()` to
  `config.developer_instructions`, gone.
- `codex-rs/core/tests/suite/subagent_notifications.rs` — two
  tests renamed and rewritten to assert the *new* invariant.

The most interesting line is the spawn-handler diff:

```rust
-        let spawn_agent_instructions = SpawnAgentInstructions.render();
-        config.developer_instructions = Some(
-            if let Some(mut existing_instructions) = config.developer_instructions.take() {
-                if !existing_instructions.ends_with('\n') {
-                    existing_instructions.push('\n');
-                }
-                existing_instructions.push_str(&spawn_agent_instructions);
-                existing_instructions
-            } else {
-                spawn_agent_instructions
-            },
-        );
```

What's left after the deletion: `config.developer_instructions`
keeps whatever value `apply_role_to_config` and
`apply_spawn_agent_runtime_overrides` left in it, which by
construction is the parent's instructions (forked into the child
config) plus any role-specific overrides. The PR's claim that
"parent/configured developer instructions carry through normally"
is mechanically accurate — there is no other code path that was
depending on the appended block.

The test rewrite is the part I want to look at carefully because
it's where invariants become assertions. The renamed test
`spawned_multi_agent_v2_child_inherits_parent_developer_context`
used to assert four things about a child request:

```rust
// before
assert!(body_contains(&child_request, "Parent developer instructions."));
assert!(body_contains(&child_request, "<spawned_agent_context>"));
assert!(body_contains(&child_request, SPAWNED_AGENT_DEVELOPER_INSTRUCTIONS));
assert!(body_contains(&child_request, CHILD_PROMPT));
```

After the rewrite:

```rust
// after
assert!(body_contains(&child_request, "Parent developer instructions."));
assert!(body_contains(&child_request, CHILD_PROMPT));
```

So the post-condition is "the parent's developer instructions
appear in the child's request body, and the child prompt also
appears". The negative assertion that the spawned-agent wrapper
is *absent* is implied by the matcher used to find the request
in the first place:

```rust
.find(|request| {
    body_contains(request, CHILD_PROMPT) && !body_contains(request, SPAWN_CALL_ID)
})
```

The matcher distinguishes "the child's first request" (contains
the prompt, does not contain the parent's spawn tool-call id)
from "the parent's follow-up after the child returns" (contains
a `function_call_output` for the spawn id). It does *not*
explicitly assert "this body does not contain
`<spawned_agent_context>`".

This is a small but real test-rigor gap. See Suggestions.

# Risks

**Loss of orientation in deeply-nested spawns.** The block was
arguably most useful when a sub-sub-agent was spawned: by that
point the conversation history is heavily forked and the model
might benefit from being told "you are *also* a spawned agent
and your response goes back to *your* parent, not the root user".
Without the block, the model has to infer this from the shape of
the conversation. For a flat spawn (root spawns one child) this
is probably fine; for `next_thread_spawn_depth > 1` it's worth
verifying the model still routes its final-channel response
correctly. The PR's verification list does not mention a
deep-spawn test. Worth a follow-up.

**Behaviour change is invisible to existing clients.** A user
who built tooling around "spawned agents announce themselves"
gets a silent behaviour change. The PR has no `CHANGELOG.md`
entry, no protocol-version bump, no flag. For an experimental
multi-agent v2 surface that is probably acceptable, but if any
external consumers exist a one-line note in the multi-agent
README would be courteous.

**The negative assertion is implicit.** The new test does not
explicitly assert `!body_contains(child_request, "<spawned_agent_context>")`.
If a future PR re-introduces the block (e.g. someone adds it
back conditionally, gated on a new flag), the test still passes
because the positive assertions don't care. The matcher's
`!body_contains(req, SPAWN_CALL_ID)` is doing a different job.

**Wording removal in tests.** The constant
`SPAWNED_AGENT_DEVELOPER_INSTRUCTIONS` is removed entirely. If a
follow-up PR ever wants to assert "this exact string does not
appear in spawned children", the constant would need to come
back. Cheap to add now as a single line.

# Suggestions

1. Add an explicit negative assertion in
   `spawned_multi_agent_v2_child_inherits_parent_developer_context`:

   ```rust
   assert!(!body_contains(&child_request, "<spawned_agent_context>"));
   assert!(!body_contains(&child_request, "newly spawned agent"));
   ```

   Cost: two lines. Benefit: the test now actively defends the
   intent of the change, not just the parent-inheritance side
   effect. If anyone re-introduces the block, this test fails
   loudly.

2. Add a brief comment in `multi_agents_v2/spawn.rs` at the spot
   where the block used to be appended, noting the prior design
   and why it was removed. Without it, the next person who reads
   the spawn handler and notices "we never give the spawned
   child any orientation" might re-add it. One-paragraph comment
   cheaper than a re-debate.

3. Consider running the multi-agent eval suite (whatever the
   internal benchmark is) at depth=2 and depth=3 to confirm the
   removal does not regress deep-spawn final-channel routing.
   Not blocking, but worth checking before this lands in a
   release.

4. If a `CHANGELOG.md` exists for `codex-core`, a one-line entry
   under "behaviour changes" — "spawned agents no longer receive
   a `<spawned_agent_context>` developer fragment" — would help
   anyone debugging "why does my child agent suddenly behave
   differently?".

# Verdict

Approve with one strong nit: please add the explicit negative
assertion (suggestion #1). The change itself is the right
direction — measure-and-delete is the right way to retire
prompt-engineering scaffolding once it has served its purpose.
The 7-add / 54-delete ratio is a healthy signal: the design got
simpler, not more elaborate.

# What I learned

The general lesson here is about "scaffolding prose" in agent
systems: small fixed strings appended to model prompts to
explain the agent's role, the protocol, the expected output
format. They are easy to add ("just tell the model what it is")
and very hard to remove because there is rarely a forcing
function. The team here had one — someone presumably noticed the
block was either neutral or net-negative on real eval traces —
and used it to delete the scaffolding cleanly.

The narrower lesson is about test invariants. The old test
asserted four things. Two were "the protocol works" (parent
instructions inherited, child prompt arrives) and two were "the
scaffolding is present" (the wrapper marker, the exact prose).
When the scaffolding was removed, the test rewrite kept the
"protocol works" assertions and deleted the "scaffolding present"
assertions — which is correct — but did not add the symmetric
"scaffolding absent" assertions. The matcher's
`!body_contains(req, SPAWN_CALL_ID)` looks like a negative
assertion if you skim it, but it is doing identification, not
intent-checking. Two lines fix it.

Third: the diff is a nice case of "delete a struct, delete its
mod declaration, delete its one call site, rewrite two tests".
That is roughly the minimum surface area for retiring a
`ContextualUserFragment` impl. If your codebase makes this kind
of removal awkward — e.g. if the `mod.rs` re-export is part of a
public API and removing it is breaking — that is a signal the
fragment system is exposing too much. `codex-rs` keeping these
as `pub(crate)` is what makes a clean deletion possible.

Finally: I notice the spawn handler still has
`apply_spawn_agent_runtime_overrides` and `apply_spawn_agent_overrides`
right next to where the deleted block was. Those two functions
are presumably about runtime config (model id, reasoning effort)
rather than prompt scaffolding, but the symmetry is suggestive.
A future cleanup pass might want to rename the deleted block's
neighbours to make it clearer that "overrides" are
runtime-config-only and prompt-shaping is no longer in the spawn
handler at all.
