## Summary

Wires Goose's slash-command recipe surface into ACP. Builds an `AvailableCommand` list from configured `slash_commands` (recipe metadata supplies description and input hint), pushes it via `AvailableCommandsUpdate` on session create / load / fork, and emits a visible `Running recipe: /<command>` chunk on the prompt path so ACP clients show user-facing feedback when a recipe is invoked. Only commands explicitly registered in `slash_commands` are exposed (not every discovered recipe).

## Notes

- `crates/goose/src/acp/server.rs:1568-1576` (`command_input_hint`) — selects the input hint with this precedence: (1) parameter named `args`, (2) first parameter without a default, (3) first parameter. Reasonable, but undocumented in the function. A doc comment would prevent later refactors from "fixing" the ordering and silently breaking client UX. Also: if `params` is `Some(vec![])`, `params.first()` returns `None` and the function returns `None` — the `?` on line 1569 only short-circuits on `parameters: None`, not on an empty vec. Confirm this is intended (no input hint vs an empty hint).
- `crates/goose/src/acp/server.rs:1578-1612` (`build_available_commands_from_slash_commands`) — discovers via `crate::slash_commands::list_commands()`, then for each mapping: checks `recipe_path.exists()`, reads the file via `tokio::fs::read_to_string`, parses with `Recipe::from_content`. All three failure modes (`!exists`, read error, parse error) silently `continue`, so a malformed recipe drops out of the list with no warning to the user. Add a `tracing::warn!` for at least the parse-error case so users notice broken recipes; silent drops are a debugging trap.
- `crates/goose/src/acp/server.rs:1614-1626` (`send_available_commands_update`) — single-shot send of the full command list. No incremental update or diff support. Acceptable for v1 but won't scale if recipes start being added/removed dynamically; document that clients should treat each `AvailableCommandsUpdate` as a full replacement.
- `crates/goose/src/acp/server.rs:1628-1632` (`is_builtin_agent_command`) — linear scan over `crate::agents::execute_commands::list_commands()` per prompt. List is small (under 10 commands per the static `COMMANDS` array near `execute_commands.rs:43`), so O(n) is fine. If it ever grows, switch to a `&'static HashSet`.
- `crates/goose/src/acp/server.rs:1832-1836` and `:2247-2249` — `send_available_commands_update` is invoked on the new-session and load/fork paths. Good. Confirm there is no auth-failure or capability-mismatch path that creates a usable session *without* hitting one of these — if so, that session never sees the command list.
- `crates/goose/src/acp/server.rs:2293-2313` — the "Running recipe: …" notification path. Three nested guards: (1) parse_slash_command succeeds, (2) command is *not* a builtin agent command, (3) recipe path exists. Correct ordering — builtin commands like `/mode` should not get the recipe banner. The notification fires *before* `thread_manager.append_message`, so if append fails the user sees the banner for an action that didn't happen. Low-risk but worth flipping the order so the banner only emits after the message commits.
- `crates/goose/src/acp/server.rs:2293` — `let message_text = user_message.as_concat_text();` is called for every prompt, even when no slash command is present. `as_concat_text()` likely allocates; for high-volume prompt paths consider a cheap `starts_with('/')` precheck on the first text content block before doing the full concat.
- `crates/goose/src/acp/server.rs:1538-1559` import block — bundles in `AvailableCommand`, `AvailableCommandInput`, `AvailableCommandsUpdate`, `UnstructuredCommandInput` from `sacp::schema`. Confirm the pinned `sacp` version exposes these (the PR doesn't bump `Cargo.toml` in the visible diff). If they're new in an unreleased `sacp`, this PR needs to wait or pull a version bump.
- **No tests.** PR body explicitly acknowledges this ("I haven't included any actual unit tests, but I am happy to add some if this approach is accepted"). For a 173-line ACP-protocol change, at least a unit test for `command_input_hint` precedence and one integration-style test that asserts `AvailableCommandsUpdate` fires on `new_session` would lock the contract. Manual ACP-client validation is not enough for protocol-shaped surfaces.
