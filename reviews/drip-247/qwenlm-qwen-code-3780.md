## Summary
Adds optional per-model pricing for cost estimation in the `/stats model` command. New `modelPricing` settings key holds a `Record<modelId, {inputPerMillionTokens?, outputPerMillionTokens?}>` map; when configured, the `model` subcommand of `/stats` walks `session.stats.metrics.models` and emits an "Estimated cost: $X.XXXX" line per model based on accumulated `prompt`/`candidates` token counts. When `modelPricing` is absent, the cost line is omitted (no behavior change).

## Line-level observations
- `packages/cli/src/config/settingsSchema.ts:969-986` — new `modelPricing` settings key with `requiresRestart: false` (correct: pricing is read at command-invocation time, no startup wiring) and `showInDialog: false` (correct: nested-object value isn't sensible to edit through the settings dialog UI, JSON-edit only). The TypeScript shape `Record<string, { inputPerMillionTokens?: number; outputPerMillionTokens?: number }>` matches the description's example `{"qwen3-coder": {"inputPerMillionTokens": 0.30, "outputPerMillionTokens": 1.20}}` exactly.
- The `description` string at `:983-985` includes a worked example with realistic per-million-token figures — operator-friendly. But it doesn't document what happens when only one of `inputPerMillionTokens` or `outputPerMillionTokens` is set (does the omitted side count as $0 or skip the model entirely?). Should be clarified before merge.
- `packages/cli/src/ui/commands/statsCommand.test.ts:153-200` — `'stats model shows cost when pricing is configured'` test uses `1_000_000` prompt + `500_000` candidates against `inputPerMillionTokens: 0.3, outputPerMillionTokens: 1.2`, asserts `"Estimated cost: $0.9000"` in the output. Math: `1.0 * 0.3 + 0.5 * 1.2 = 0.30 + 0.60 = 0.90` ✓. Four-decimal-place format pinned by string match, which is a reasonable contract for sub-cent pricing cases.
- `:202-241` — `'stats model does not show cost when pricing is not configured'` test asserts `not.toContain('Estimated cost')` so the omission path is locked. Correct conservative shape: a config-absent user sees zero new output.
- `:243-329` — `'stats model shows cost per model when multiple models have pricing'` test sets up `model-a` and `model-b` with distinct pricing and metrics, then exercises the per-model dispatch. The test sets up the multi-model state but the diff cuts off before showing the assertions — confirm in full review that both models' `Estimated cost:` lines appear (not just one).
- The cost-computation code itself (in `statsCommand.ts` / model subcommand action) isn't visible in the diff snippet — critical things to verify: (a) is `cached` token count correctly priced? Cached tokens often have different (or zero) pricing, and the test uses `cached: 0` so this path isn't exercised; (b) is `thoughts` (reasoning tokens) treated as input or output for billing? Different providers do different things; (c) is there protection against partial pricing where only one of the two `*PerMillionTokens` is set?
- The test mock at `:113-118` uses `(... as unknown as Record<string, unknown>)['sessionStartTime'] = undefined` to delete a typed field — ugly but standard for tests that need to break the type contract to test edge cases. The same pattern at `:160-163` and `:295-298` (`['merged'] = {modelPricing: ...}`) is consistent.
- No fail-loud behavior on typo'd model IDs in `modelPricing`. If a user configures `modelPricing.{"qwen-coder": ...}` but the actual session uses `"qwen3-coder"`, the cost line is silently omitted with no diagnostic. A warning ("the following models in modelPricing weren't used this session: ..." or "the following used models have no pricing configured: ...") would help detect config drift.
- Pricing values are stored as plain `number` not a tagged currency type. If a user configures `0.30` thinking "30 cents per million" but the code expects "30 dollars per million" the user-visible cost will be off by 100x. The `inputPerMillionTokens: 0.3` example in the description disambiguates by example but a `currency: "USD"` field (or even just a unit comment in the schema) would make this less footgun-prone.

## Verdict
`merge-after-nits` — clarify the partial-pricing semantics in the schema description, add a test for the only-input-pricing-configured case, consider a one-line "configured pricing didn't match any used model" warning to detect config drift, and confirm the multi-model-pricing test's full assertions cover both models. Cost math and shape are right.
