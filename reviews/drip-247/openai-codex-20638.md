## Summary
Adds trace-safe image-input metadata to the `codex.user_prompt` OTel span by introducing `events/image_input_telemetry.rs` (~275 lines) that walks the prompt's `UserInput` items and emits per-image `source` (one of `"data_url" | "remote_url" | "local_file"`), plus best-effort `mime_type`, `width`/`height`, `byte_length`, and `extension` fields. Aggregate `image_types` and `mime_types` strings get attached to the span; per-image details land in a `details_json` attribute. Critically the implementation **never** records local paths, full remote URLs, or base64 payloads — only the trace-safe metadata.

## Line-level observations
- `events/image_input_telemetry.rs:7` — `IMAGE_HEADER_READ_LIMIT: u64 = 64 * 1024` caps local-file header reads at 64KiB, which is more than enough for PNG/GIF/JPEG/WEBP magic + dimension parsing and bounds the I/O cost so a span emission can't be turned into a multi-megabyte read by an adversarial input. Correct discipline.
- `events/image_input_telemetry.rs:36-42` — explicit match arms for `UserInput::Image { image_url }` (remote/data URL), `UserInput::LocalImage { path }` (filesystem), with `Text` / `Skill` / `Mention` filtered to `None` and a catch-all `_ => None` arm. The catch-all is concerning: if a future `UserInput` variant adds another image-bearing shape (e.g. `UserInput::DroppedFile` for desktop drag-drop), it'll silently slip through telemetry-blind. Better discipline would be exhaustive matching with a `compile_error!`-on-add convention.
- `events/image_input_telemetry.rs:81-101` — `data_url_detail()` correctly strips the `data:` prefix, splits on the first comma, and rejects non-`image/*` MIME types with `if !mime_type.starts_with("image/") { return None }`. The `is_base64` predicate at `:97` checks for the `;base64` parameter via `.eq_ignore_ascii_case`. The `byte_length` is computed only when base64 is present (since URL-encoded data URLs have no fixed-width byte mapping) — correct conservatism.
- `events/image_input_telemetry.rs:126-133` — `read_image_header()` uses `file.take(IMAGE_HEADER_READ_LIMIT)` chained read which is the idiomatic bounded-read shape. No `unwrap`, errors fall through to `unwrap_or_default()` at `:111` so a permission-denied file returns an empty header rather than crashing the span emission. Trace-safety preserved.
- `events/image_input_telemetry.rs:143-176` — `image_info_from_header()` parses PNG (offsets 16-24 for IHDR width/height), GIF87a/89a (offsets 6-10 little-endian), JPEG (delegated to `jpeg_info_from_header`), and detects WEBP container by `RIFF...WEBP` but doesn't extract width/height for WEBP. The PNG/GIF parsers correctly use `try_into().ok()?` for fixed-array conversions so a truncated header returns `None` rather than panicking.
- `events/image_input_telemetry.rs:180-193` — JPEG SOF marker walking is correct in shape (`0xff` marker prefix, skip stuffing bytes via the `while bytes[i] == 0xff` loop) but the diff was truncated mid-function so I can't verify the SOF0/SOF2 dimension extraction. The bounded-read at `:131` caps any pathological JPEG with no SOF marker in the first 64KiB at a finite walk.
- The `image_type_from_mime` and `mime_type_from_extension` helpers (referenced at `:64-71`, `:108-110`) aren't shown in the diff snippet but follow standard MIME-table lookup shape.
- Privacy/safety property: searching the diff for `path.display()`, `image_url` as a serialized field, or `payload` shows none — the `ImageInputDetail` struct's serializable fields are deliberately limited to `source` (enum-like static str), `image_type`, `mime_type`, `width`, `height`, `byte_length`, `extension`. The PR's stated invariant ("no local paths, no remote URLs, no base64 payloads") holds at the type level.
- Validation list (`just fmt`, `cargo test -p codex-otel`, `just fix -p codex-otel`, post-rebase test re-run) is the right minimum surface for an OTel-only change. No integration test against a real span collector but the unit tests for header parsing should cover the per-format dimension extraction.
