## Summary
Adds a New Relic AI Monitoring custom logger as an opt-in extension, paired with a *separate* container image (`docker/newrelic/Dockerfile`) that bakes in the `newrelic` Python agent + a New-Relic-specific entrypoint that wraps the proxy with `newrelic-admin run-program`. The split-container approach is the correct response to the maintainer's prior request not to bundle the New Relic agent into the base image. Wires `newrelic` into `_custom_logger_compatible_callbacks_literal`, adds a `NewRelicInitParams` typed config, and surfaces the integration in the UI's `callback_configs.json` registry with one dynamic param (`NEW_RELIC_AI_MONITORING_RECORD_CONTENT_ENABLED`).

## Line-level observations
- `docker/newrelic/Dockerfile:1-18` — uses `ARG BASE_IMAGE=docker.litellm.ai/berriai/litellm` + `ARG BASE_TAG=main-stable` so the New Relic image rebases on the canonical published image. `python -m ensurepip && python -m pip install --no-cache-dir 'newrelic>=12.1.0,<13'` is correctly version-pinned with a major-version cap so a `newrelic 13.x` API break doesn't silently land in the next rebuild. `--no-cache-dir` keeps image size down.
- `docker/newrelic/entrypoint.sh:1-10` — preserves the `SEPARATE_HEALTH_APP=1` branch (delegates to `supervisord -c /etc/supervisord_newrelic.conf`), else `exec newrelic-admin run-program litellm "$@"` so the New Relic Python agent wraps the proxy. The `# ⚠️ KEEP IN SYNC: Changes to docker/prod_entrypoint.sh logic should be reviewed here` comment at `:2` is the right discipline for a copy-pasted fork; matched by the same comment in `docker/prod_entrypoint.sh:2`. But the supervisord variant at `docker/newrelic/supervisord.conf:1-50` is also a near-fork of `docker/supervisord.conf` with the same `KEEP IN SYNC` warning — three-place duplication is starting to smell, recommend a future refactor to a templated supervisord config (jinja or a tiny Python preprocessor) before this gets to a fourth integration.
- `litellm/__init__.py:152` — adds `"newrelic"` to the `_custom_logger_compatible_callbacks_literal` `Literal[...]` union, so `litellm.callbacks = ["newrelic"]` becomes type-safe at the user-facing surface. Paired with `newrelic_params: Optional[Union[NewRelicInitParams, Dict]] = None` at `:390` mirroring the existing `datadog_params` shape exactly — symmetric with the established convention.
- `litellm/integrations/callback_configs.json:248-262` — UI registry entry has `supports_key_team_logging: false` (correct: New Relic is process-global, not per-key/team), one dynamic param with type/ui_name/description/required, and a logo path (`newrelic.png`) that needs to actually exist in the assets dir — confirm the asset is checked in.
- `litellm/integrations/newrelic/newrelic.py` — the diff snippet shows only the file header but the PR claims +912 lines; can't verify the full implementation here. Critical things to verify in a follow-up review: (a) does `NewRelicLogger` correctly handle the case where the `newrelic` agent is *not* installed (the base litellm image)? It must degrade gracefully (warn-once and no-op) rather than crash on import, otherwise users on the base image who flip the callback on by mistake will hit `ImportError` on every request. (b) Does it record token counts, latency, and error rate via the AI Monitoring custom-event API rather than as plain log lines? (c) Does the `NEW_RELIC_AI_MONITORING_RECORD_CONTENT_ENABLED=false` path actually scrub message content from the emitted events?

If a user wants to *temporarily* disable the New Relic agent without rebuilding the image (e.g. `NEW_RELIC_DISABLE=1` for local debugging), there's no path. A `if [ -z "$NEW_RELIC_DISABLE" ]` guard would be a one-line addition.
