## Summary
Adds a `goose plugin install <git-url>` CLI command that clones a git repository and installs it as a plugin if it matches a supported manifest format. First-pass scaffolding supports the `gemini-extension.json` manifest (Gemini extension format) and imports any embedded skills. Lays out a `goose::plugins` module with a `PluginFormat` trait surface, format-specific installers under `plugins/formats/` (only `gemini.rs` so far), and shared utilities (`plugin_install_dir`, `copy_dir_all`, `write_install_metadata`, `FormatNotSupported`).

## Line-level observations
- `crates/goose-cli/src/cli.rs:636-642` — new `PluginCommand` enum with one `Install { url: String }` subcommand. Clean clap shape, `#[command(about = ...)]` attribute matches the convention used by the adjacent `RecipeCommand` and `GatewayCommand`.
- `crates/goose-cli/src/cli.rs:857-862` — `Plugin` enum variant on `Command`. Wiring at `:1066` (`get_command_name`) and `:1865` (dispatch to `handle_plugin_subcommand`) is symmetric with `Recipe`. Telemetry attribution will pick up `"plugin"` automatically.
- `crates/goose-cli/src/commands/plugin.rs:4-26` — `handle_plugin_install` is intentionally thin: delegate to `goose::plugins::install_plugin(url)`, then pretty-print the install summary (format / name / version / source / location / imported skills). The `console::style("✓").green()` and `style(&install.name).bold()` follow the existing CLI presentation idiom in `recipe.rs` / `info.rs`.
- `crates/goose/src/plugins/formats/gemini.rs:13-17` — `GeminiManifest { name: String, version: String }` is a minimal shape. The actual Gemini extension manifest spec has more fields (`description`, `commands`, `mcpServers`, etc.); this PR is honest in its body about being "minimal scaffolding," and only deserializing the two fields required for naming/versioning is the right surface for a first pass.
- `crates/goose/src/plugins/formats/gemini.rs:34-46` — `install_from_manifest` correctly checks `manifest_path.is_file()` and returns `FormatNotSupported` (not an error) when missing, so the multi-format dispatch in the parent module can fall through to the next format-attempt cleanly. `serde_json::from_str` with `.with_context(|| format!("Failed to parse {}", ...))` gives actionable error messages.
- `crates/goose/src/plugins/formats/gemini.rs:43` — `validate_extension_name(&manifest.name)` is called but the validator's body isn't shown in the diff snippet. Critical: this is the trust boundary — `manifest.name` is interpolated into the `install_root.join(&manifest.name)` filesystem path at `:51`, so a malicious `name = "../../etc/passwd"` would write outside the install root. The validator MUST reject path-traversal characters (`..`, `/`, `\`, leading `.`) and ideally enforce a `[a-zA-Z0-9_-]+` whitelist.
- `crates/goose/src/plugins/formats/gemini.rs:52-58` — refuses to overwrite an existing install with `bail!("Plugin '{}' is already installed at {}", ...)`. Correct conservative discipline; a future `--force` flag could be added but defaulting to refuse is right.
- The CLI command takes a `git-url: String` and clones it. The `install_plugin` function isn't shown in the diff snippet but the security surface is large: (a) does it support `file://` URLs? (b) does it shell out to `git` or use a Rust git library (`gix` / `git2`)? (c) does it pin a commit SHA or always clone the default branch? (d) does it check out submodules? Each of these has supply-chain implications and should be documented.
- `crates/goose/Cargo.toml:144` — drive-by stylistic change `tempfile = { workspace = true } → tempfile.workspace = true`. Equivalent semantics, matches the more concise modern Cargo style. Innocuous but unrelated to the PR's stated scope.
- `crates/goose/src/lib.rs:30` — `pub mod plugins` exports the new module. Correct visibility; downstream UI/desktop crates can import the install API for their own plugin-install flows.
- PR body explicitly lists future work: open-plugins / claude marketplace formats, GUI install, updating. Each is a real follow-up but landing the scaffolding first is the right shape.
- No tests in the visible diff. For a `git clone → manifest parse → filesystem write` path this is concerning — at minimum a unit test for `install_from_manifest(source, checkout_dir, install_root)` against a fixture directory containing a valid `gemini-extension.json` (and a parallel test for the `FormatNotSupported` error when the manifest is absent) would lock the contract. The factored signature with `install_root` parameter at `:32-36` clearly enables this.

## Verdict
`needs-discussion` — the scaffolding shape is right, but two open questions block confident merge: (1) what does `validate_extension_name` actually do, and is it strict enough to prevent path-traversal exploits via crafted manifest `name` fields? (2) what's the security model on the git clone (commit pinning, submodules, supported URL schemes)? Plus: at least one fixture-based unit test of the format-installer happy path. Once those are addressed this can drop to merge-after-nits.
