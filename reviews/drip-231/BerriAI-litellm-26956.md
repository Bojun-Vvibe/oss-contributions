## Summary

Closes a real correctness bug in the MCP semantic filter hook where native (non-MCP) tools were being silently dropped from the request when the filter was active, because the hook returned only the filtered MCP tools and discarded everything else. Fix partitions the request's tools into MCP-router-tools vs native, only runs the semantic filter on the MCP subset, and re-merges into the original tool list while preserving request order. Also fixes an unrelated FastAPI bug where `operationId` was always stamped with `_put` regardless of HTTP method (the snapshot generator was non-deterministic), producing the snapshot `_delete`/`_get`/`_post`/etc. corrections seen across the JSON.

The discard of native tools was the silent dataloss; this restores them to the response.

- `hook.py:252-266`: new `_get_tool_name(tool)` static helper handles three tool shapes: dict with `function.name` (OpenAI function-format), dict with top-level `name` (MCP-format), and object with `.name` attribute (typed `MCPTool`). Returns `""` defensively for anything unrecognized so downstream `if not tool_name` predicates short-circuit safely. Replaces three copy-pasted name-extraction snippets at the old `:294-298` (now collapsed to one call at `:381`).
- `hook.py:268-284`: `_is_mcp_router_tool(tool)` checks both `tool_name in tool_map` (exact match) and the `_name_matches_canonical(tool_name, canonical_name)` callback for normalized matches — covers the case where the MCP server published tool `do_thing` but the canonical name is `mcp_do_thing` or vice versa. The `getattr(self.filter, "_name_matches_canonical", None)` access keeps backward compatibility with filter implementations that don't expose the canonical-match callback.
- `hook.py:286-298`: `_partition_mcp_router_tools` is a clean linear-scan partition into `(native_tools, mcp_tools)` tuples. The order is deliberate — the no-MCP-tools early return at `:300-305` (`if not mcp_tools: ... return None`) skips the entire semantic filter when no MCP tools are present, avoiding wasted embedding compute on requests that don't even use MCP.
- `hook.py:300-322`: `_merge_filtered_tools_preserving_request_order` is the careful merge. Uses `Counter` to track how many of each filtered MCP tool name remain, then walks the original `tools` list in order — native tools are passed through unconditionally, MCP tools are passed through only if the counter for their name is `> 0` (decrementing on each pass). This preserves the original request order *exactly* (important for tool-choice prompts that depend on positional ordering) and correctly handles the case where the same MCP tool name appears multiple times.
- `hook.py:308-310`: the walrus `if (tool_name := self._get_tool_name(tool))` filter inside the `Counter` comprehension correctly skips empty-name tools so they don't pollute the counter with `Counter({"": 3})` — defensive against malformed tool entries.
- `hook.py:380`: the previous bespoke name-extraction at `_get_tool_names_csv` is replaced with the new `_get_tool_name(tool)` call — consistency win, single source of truth.
- `_lazy_openapi_snapshot.py:46-65`: new `_set_deterministic_operation_ids(routes)` function replaces FastAPI's auto-generated `operation_id` (which non-deterministically picks the first method registered for a route, producing the bogus `_put` suffix even on `_delete`/`_get` ops) with `f"{route.name}{route.path_format}_{sorted(route.methods)[0].lower()}"`. The `sorted(route.methods)[0]` choice is debatable (alphabetically first, so `delete` wins over `put` for a route registered for both) but it's deterministic and reproducible — the snapshot diffs at `_lazy_openapi_snapshot.json:3572-27117` are purely the expected resnapshot output of this regenerator change.

## Verdict

**merge-after-nits**

The semantic-filter bug is a real correctness issue (silent dataloss of native tools when MCP is also present in the request) and the fix is well-architected — three new helpers (`_get_tool_name`, `_is_mcp_router_tool`, `_partition_mcp_router_tools`, `_merge_filtered_tools_preserving_request_order`) each with a single clear responsibility, the `Counter`-based merge preserves request order exactly, and the no-MCP-tools early-skip avoids wasted embedding compute. Nits: (a) the test diff at `test_semantic_tool_filter.py:138-203` only deletes the chatty `print(...)` debug lines — there's no new test arm exercising the multi-tool partition+merge path, which is the new contract; should add a test that constructs a request with `[native_tool_1, mcp_tool_a, native_tool_2, mcp_tool_b]` and asserts the filtered output preserves the four-element order with the correct subset; (b) the `_lazy_openapi_snapshot.py` `operationId` determinism fix is a separate concern and arguably should be its own PR — the 32+ snapshot lines being touched here will pollute git blame for anyone investigating the semantic-filter logic later; (c) the `getattr(self.filter, "_name_matches_canonical", None)` private-attribute access at `:277` couples the hook to filter internals — a stable public canonical-match API on the filter would be a cleaner contract.
