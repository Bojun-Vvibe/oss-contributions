## Summary

Large feature PR adding "extension status and usage" UI surfaces to the goose2 desktop app. Three layers: (1) new ACP request `_goose/session/extensions/status` returning per-extension config + connection status + tool list + error, (2) centralized status-shaping logic that previously lived duplicated across settings and chat-context surfaces, (3) revised goose2 React views (sidebar, chat context panel, settings) that consume the new shared model to distinguish *active*, *on-demand*, and *failed* extensions.

## Specific code references

- `crates/goose-sdk/src/custom_requests.rs:9-107`: new `ExtensionConfigDto` enum tagged on `type` covers seven extension shapes (`sse`, `stdio`, `builtin`, `platform`, `streamable_http`, `frontend`, `inline_python`). Each variant uses `#[serde(skip_serializing_if = "Option::is_none")]` for optional fields and `#[serde(default)]` for `available_tools: Vec<String>` so older clients sending without the field don't break. The seven-variant exhaustiveness covers every existing extension config shape — this is the right serde contract for an ACP boundary type.
- `custom_requests.rs:110-117`: `ExtensionConnectionStatusDto` enum is `serde(rename_all = "snake_case")` with explicit `#[default] Available` variant. The `Available` default is the load-bearing choice — an extension that hasn't been connected yet defaults to "available to be invoked on demand" rather than "connected" or "failed", which matches the on-demand UX the PR is implementing.
- `custom_requests.rs:120-129`: `SessionExtensionStatusDto { config (flattened), config_key, status, tools (default Vec), error (skip_serializing_if None) }` — the `#[serde(flatten)]` on `config` means the JSON wire shape merges the `ExtensionConfigDto` fields (including discriminator `type`) directly into the parent object instead of nesting under a `config:` key. This matches the convention already used across `acp-schema.json` for tagged-union DTOs and is what the React layer expects.
- `custom_requests.rs:131-144`: new `GetSessionExtensionStatusRequest { session_id }` and `Response { extensions: Vec<...> }` registered via the `#[derive(JsonRpcRequest/Response)]` proc-macro at method `_goose/session/extensions/status`. Snake-case method path, camelCase request fields (`#[serde(rename_all = "camelCase")]`) — consistent with the existing ACP method conventions seen at `acp-meta.json:55-59` for the sibling `_goose/session/extensions` method.
- `acp-meta.json:58-62`: the new method registered in the ACP catalog. Important for the SDK code-generators that consume this file — without registration, generated TypeScript clients won't know the method exists.
- `acp-schema.json:252-422`: regenerated schema for the new request/response and the `SessionExtensionStatusDto` `oneOf` over the seven extension types. The seven `oneOf` branches each have their own `required: ["type", "name", "description"]` minimum-field contract — strict enough to reject malformed config payloads at the schema-validation layer.

## Open questions / concerns (substantive)

- **Migration behavior at `crates/goose/src/config/migrations.rs`** (file referenced in body but not visible in the truncated diff window): the PR body says "Adds migration behavior for extension on-demand defaults while preserving required manager behavior." Migrations that flip defaults silently are exactly the class of change that needs a one-line release-note callout — users with existing `~/.config/goose/config.yaml` need to know whether their previously-explicit settings are preserved or whether the migration overrides them.
- **71-file diff exceeds GitHub's 300-file API limit warning is not present here, but the ratio of test files to UI files is light**: `ui/goose2/src/features/extensions/` has tests at `lib/__tests__/extensionCategories.test.ts`, `lib/__tests__/extensionUsage.test.ts`, `lib/__tests__/extensionsWidgetData.test.ts`, `ui/__tests__/ExtensionModal.test.tsx`, `ui/__tests__/ExtensionsSettings.test.tsx`, `hooks/__tests__/useExtensionModalForm.test.ts` and the chat-side `hooks/__tests__/useExtensionsWidgetData.test.ts`, `widgets/__tests__/ExtensionsWidget.test.tsx`, `shared/api/__tests__/acpNotificationHandler.test.ts` — that's 9 test files for 71 production files. For a feature touching the chat context panel, sidebar, settings modal, ACP server dispatch, and migration logic, the test coverage feels thin on the React side — particularly missing is an end-to-end test that the new ACP request actually populates the goose2 sidebar correctly when one extension is `Failed` and another is `Connected`, which is the primary user-visible promise of the PR.
- **Frontend ACP custom-dispatch coupling**: the new dispatch at `crates/goose/src/acp/server/custom_dispatch.rs` (visible only in the file list) routes the new `_goose/session/extensions/status` request through the same custom-dispatch machinery as the existing extension management requests. Routing concentration through one module is good for maintainability but means a serde-deserialization regression on any one DTO shape can take down the whole custom-dispatch surface — would be worth a smoke test that all eight ACP custom requests (the existing seven + the new one) survive an invalid-payload fuzz arm.
- **i18n coverage**: the diff touches en + es chat/settings/sidebar locale files but the project may have additional locale targets — worth confirming with the i18n owners that en + es is the intentional shipping scope rather than an oversight.

## Verdict

**needs-discussion**

This is a substantial, well-architected feature that addresses a real UX gap (extension status visibility), but the size and breadth of the change warrant maintainer discussion before merge. Specifically: (1) the migration behavior at `config/migrations.rs` flipping extension on-demand defaults needs explicit release-note documentation since user configs are at stake; (2) the 9:71 test-to-production-file ratio is light for a feature touching this many user-visible surfaces — at minimum needs an end-to-end test covering the failed+connected mixed-state rendering that's the PR's primary promise; (3) the ACP DTO contract (`ExtensionConfigDto` + `SessionExtensionStatusDto`) becomes a stable wire contract once shipped — worth a maintainer sign-off on the seven-variant exhaustiveness and the `serde(flatten)` decision before SDK consumers build against it; (4) confirm i18n shipping scope (en + es only?) is intentional. The Rust-side ACP additions at `custom_requests.rs:9-144` are clean and self-contained; if the maintainer wants to land this incrementally, splitting (a) ACP request/schema additions, (b) migration behavior, (c) UI integration into separate PRs would make each individually reviewable and reversible.
