## Summary

Defines explicit behavior for HTTP-protocol OTLP telemetry endpoint routing — appends signal-specific paths (`/v1/traces`, `/v1/logs`, `/v1/metrics`) to the base `otlpEndpoint` when using HTTP, with new per-signal endpoint overrides (`otlpTracesEndpoint`/`otlpLogsEndpoint`/`otlpMetricsEndpoint`) that take precedence and are used verbatim. Also reads the standard OpenTelemetry env var names (`OTEL_EXPORTER_OTLP_TRACES_ENDPOINT`, etc.) as a fallback. gRPC behavior is documented as service-based routing with no path appending.

## Verdict

**merge-after-nits**

Well-scoped change that defines explicit, documented behavior for what was previously implicit OpenTelemetry SDK default behavior. The four-rule contract (HTTP path-append, signal-suffix detection, per-signal override precedence, gRPC service-routing exemption) is documented in both code comments and prose docs, with test pins for both the precedence arms and the default-endpoint-resolution arms. The TypeScript getter return-type widening from `string` to `string | undefined` is a deliberate design choice that forces every caller to handle the missing case — good for safety. Nits: (a) the "URL already ends with a signal path, it is used as-is" rule from `telemetry.md:38-39` should have a regression test arm — pass `http://collector:4318/v1/traces` as `otlpEndpoint` and assert no double-suffix appending; the visible diff doesn't show that test; (b) the `getTelemetryOtlpEndpoint` body at `config.ts:1987` still has `?? DEFAULT_OTLP_ENDPOINT` fallback even though the return type now says `| undefined` — this is internally inconsistent; either the return type should stay `string` or the fallback should move out; (c) the docs note that gRPC "does not append paths" is correct but it would be worth one explicit `otlpProtocol: 'grpc'` test arm asserting the per-signal endpoints are *ignored* in gRPC mode, since the current tests only cover the HTTP path; (d) per-signal endpoint env vars currently lack CLI flag equivalents per the table at `telemetry.md:24-26` ("CLI Flag" column shows `-`) — worth a follow-up issue noting the asymmetry with the base endpoint which does have a CLI flag.
