## Rationale

This PR does two interlocking things: (a) it exposes a new v2 HTTP endpoint `GET /api/model` that returns `Schema.Array(ModelV2.Info)` so external clients can discover available models programmatically, and (b) it does a brand rename from `Modelv2` → `ModelV2` (capital V) across all v2 model identifiers, schema brands, and `Effect.fn` trace tags. The brand rename is the dominant edit by line count (it touches `session/processor.ts:436-438`, `session/prompt.ts:981-983`, plus all 6 `Effect.fn` names in `v2/model.ts:154-180`, plus the `Schema.brand("Model.ID")` → `Schema.brand("ModelV2.ID")` and `Schema.brand("Model.ProviderID")` → `Schema.brand("ModelV2.ProviderID")` and `Schema.Class<Info>("Model.Info")` → `Schema.Class<Info>("ModelV2.Info")` at lines 5, 8, 98 respectively), but it's mechanical and self-contained.

The new endpoint is implemented in three small files. `server/routes/instance/httpapi/groups/v2/model.ts:1-24` defines the `ModelGroup` with a single `HttpApiEndpoint.get("models", "/api/model", { success: Schema.Array(ModelV2.Info) })` annotated with `summary: "List v2 models"` and `description: "Retrieve available v2 models ordered by release date."`, then `.middleware(Authorization)`. The handler at `server/routes/instance/httpapi/handlers/v2/model.ts:6-12` is a four-line `HttpApiBuilder.group` that resolves `ModelV2.Service` and forwards `model.all()` directly. The wiring at `server/routes/instance/httpapi/handlers/v2.ts:7-11` adds `modelHandlers` to the `Layer.mergeAll(...)` and provides `ModelV2.defaultLayer`. The `groups/v2.ts:7` change adds `.add(ModelGroup)` to the `V2Api` chain.

The Authorization middleware on the new group is exactly right — even though "list available models" is information-disclosure-class rather than mutation-class, the v2 API surface is consistently auth-gated and breaking that convention for the model list would set a bad precedent. The `Schema.Array(ModelV2.Info)` response type means the SDK regen at `packages/sdk/js/src/v2/gen/types.gen.ts` (+91 lines) and `sdk.gen.ts` (+38 lines) gives JS clients a fully-typed model array including pricing, capability, provider, and variant detail — that's the headline win the PR body advertises.

The new `v2/plugin-registry.ts:1-39` is a partial scaffold for a v2 plugin system (a `Service` class with `register(input)` and `trigger(name, input)` interface, currently with `trigger` being a no-op pass-through). This is unused in the rest of the diff (no caller invokes it) and the interface signature has TODOs implicit in it (e.g., `trigger`'s return is `Effect.Effect<Parameters<Plugin.Hooks[Name]>[0]>` — same shape as the input, suggesting "before-hook returning possibly-modified input" semantics, but never composes the registered plugins). Worth either gating it behind a clearer "scaffold, not wired" comment or pulling it into a follow-up PR — the model listing API and the brand rename are independently mergeable.

Two nits: (1) **PR body says "Testing: Not run; commit was created before this PR request."** — for a 277-line change that touches the core `v2/model.ts` schema brands referenced from `session/processor.ts` and `session/prompt.ts`, getting at least `bun test packages/opencode` to green before merge is non-negotiable. The brand rename in particular will silently break any downstream consumer that imported `Modelv2` (lowercase v) by name from `@/v2/model`; the diff at line 192 confirms the re-export is also renamed (`export * as Modelv2 from "./model"` → `export * as ModelV2 from "./model"`), which means callers using `import { Modelv2 } from "@/v2/model"` will get a TypeScript compile error rather than a runtime mystery, but this still needs a typecheck pass. (2) The Schema brand identifiers `"ModelV2.ID"` / `"ModelV2.ProviderID"` / `"ModelV2.Info"` are *runtime* brand strings that live inside Effect's Schema metadata; if any persisted state (sessions on disk, DB rows) was stored with the old brand-tagged shape and a new build tries to re-deserialize with the new brand, it'll mismatch. Confirm whether `Schema.brand` participates in serialized output at the boundary — if it doesn't (likely), this is fine; if it does, this is a migration concern. (3) `plugin-registry.ts` should probably not land in the same PR — pull it out.

The endpoint addition is right-sized and the brand rename is mechanically clean, but the unrun test suite plus the unused plugin-registry scaffold make this a `merge-after-nits` rather than `merge-as-is`. Run the test suite, drop the `plugin-registry.ts` into a follow-up, and this is good.
