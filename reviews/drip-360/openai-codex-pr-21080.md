## Rationale

This PR pushes the "match thread by exact name" decision down from the TUI client into the `thread/list` app-server endpoint by introducing a new `ThreadListSearchMode` enum (`Contains` | `ExactName`) on `ThreadListParams`. The protocol-side change is properly mechanical: the new enum is defined once in `app-server-protocol/src/protocol/v2.rs:4340-4346` with `#[serde(rename_all = "camelCase")]` and `#[ts(rename_all = "camelCase", export_to = "v2/")]`, then surfaces consistently across the JSON Schema (`ClientRequest.json:3631`, `codex_app_server_protocol.schemas.json:16126`, `codex_app_server_protocol.v2.schemas.json:14012`, `v2/ThreadListParams.json:24`), the regenerated TypeScript fixtures (`v2/ThreadListParams.ts:55-58` adds `searchMode?: ThreadListSearchMode | null` and the new `v2/ThreadListSearchMode.ts` exports the union), and the `v2/index.ts:366` re-export. All of this is bot-generated by `ts-rs` so the only human-authored protocol diff is the 13-line block in `v2.rs`, which is correct.

The server-side wiring in `app-server/src/request_processors/thread_processor.rs:9-15` adds `search_mode: Option<ThreadListSearchMode>` to the private `ThreadListFilters` struct and then threads it through the destructure at line 1827 and the construction at line 1850 — clean, no behavior change for callers that don't set `searchMode`. The default-to-`Contains` semantic is correctly documented at `v2.rs:4329` ("Optional search matching mode. Defaults to substring title matching.") and re-stated in `app-server/README.md:324` ("`searchMode` — optional search matching mode for `searchTerm`: `contains` (default) or `exactName` to narrow to exact user-facing thread names.").

The TUI integration at `tui/src/lib.rs` (+45/−28) is the actual *use* of the new field — startup `--resume <name>` and `/resume <name>` lookups now request `searchMode: Some(ExactName)` instead of doing client-side post-filtering on a `Contains` result, which is the cleanup the PR title advertises. The new test at `app-server-protocol/src/protocol/v2.rs:8244-8253` (`thread_list_params_accepts_state_db_only_flag` extended with `"searchMode": "exactName"`) plus the 60+ new lines in `app-server/tests/suite/v2/thread_list.rs` properly pin both the deserialization shape and the server-side filtering behavior. The test-client fixture at `app-server-test-client/src/lib.rs:1130` correctly adds `search_mode: None` to keep the existing call site compiling.

Two nits: (1) `ThreadListFilters` in `thread_processor.rs:9-15` carries both `search_term: Option<String>` and `search_mode: Option<ThreadListSearchMode>` independently — the implicit invariant is "if `search_mode == Some(ExactName)` then `search_term` should be `Some(...)` too", but nothing in the type system enforces this. An `ExactName` request with `searchTerm == null` is a no-op silently treated as "no filter" rather than a validation error, which could hide a TUI-side bug where the resume path forgets to set the term. Worth either an `assert!`/`debug_assert!` at the top of the filter pass or merging the two fields into an enum like `enum ThreadListSearch { None, Contains(String), ExactName(String) }`. (2) The README delta at line 324 says "exact user-facing thread names" but the variant is named `ExactName` — confirm whether "name" matches the *extracted title* (which is what `searchTerm` filters on per line 322) or the user-supplied alias; if they're the same field, fine, but the README should say "exact title match" for terminology consistency with the rest of the doc.

The protocol surface change is small, the regen is mechanically clean, and the test coverage is appropriate. Merge after the README terminology nit.
