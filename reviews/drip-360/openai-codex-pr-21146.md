## Rationale

This is **PR 1 of a 4-PR rollout** to enable V8 in-process sandboxing for the source-built Bazel V8 path. The scoping discipline is excellent ŌĆö this PR deliberately enables sandbox-on by default at the Bazel level (`.bazelrc:186-187` adds `common --@v8//:v8_enable_pointer_compression=True` and `common --@v8//:v8_enable_sandbox=True`) while introducing a new `v8-release-compat` config (`.bazelrc:191-192`) that **opts published `rusty_v8` artifacts back out** so existing Cargo consumers continue to work. The release/canary workflows (`.github/workflows/rusty-v8-release.yml:114` and `v8-canary.yml:108`) explicitly invoke `--config=v8-release-compat`, preserving the old contract until PR 3 in the rollout flips it.

The Python plumbing in `.github/scripts/rusty_v8_bazel.py` is a clean threading of an optional `bazel_configs: list[str] | None = None` parameter through `bazel_output_files` (line 63), `bazel_build` (line 90), `ensure_bazel_output_files` (line 112), `single_bazel_output_file` (line 194), `merged_musl_archive` (line 206), and `stage_release_pair` (line 247), with a new `--bazel-config` CLI flag using `action="append"` (line 309) so multiple configs can be stacked. The `crate_features_select` block in `MODULE.bazel:329-340` correctly enumerates all 8 Cargo target triples that consume the source-built path (excluding `*-windows-msvc`, which still uses upstream non-sandboxed prebuilts per the comment on line 330) and wires `v8_enable_sandbox` only for those ŌĆö this is the right level of granularity, since it keeps the Rust feature in lock-step with the Bazel-built `.a` archive.

The standout piece is `codex-rs/v8-poc/src/lib.rs:15-23`, which adds `linked_v8_has_sandbox()` calling the FFI symbol `v8__V8__IsSandboxEnabled()` (a symbol `rusty_v8` already exposes for its own internal verification tests), plus the new test `sandbox_feature_matches_linked_v8` at line 67 that asserts `linked_v8_has_sandbox() == cfg!(feature = "sandbox")`. This is exactly the right invariant pin: it catches the failure mode where a Cargo target's feature flag drifts out of sync with the Bazel-built artifact's actual compile-time configuration, which is the single highest-risk regression in a multi-toolchain build like this. The PR body even calls out keeping this invariant test permanent past rollout PR 4 ŌĆö that's correct.

Two nits: (1) the `try { ... } catch { ... }` over `serializeTerminalToText` doesn't apply here, but the analogous concern is the `let _ = fs::remove_dir_all(...)` pattern ŌĆö there isn't one in this PR, but the `unsafe extern "C"` block at `v8-poc/src/lib.rs:18` is calling into `rusty_v8`'s test-only symbol; if upstream `rusty_v8` ever decides this symbol is non-public and renames it, the build will break with a confusing "undefined reference" link error rather than a clear "API changed" message ŌĆö a comment pointing at the upstream commit/file where this symbol is exposed would help future-debug. (2) The `BUILD.bazel:6-9` `select` clause uses `@rules_rs//rs/experimental/platforms/constraints:windows_msvc` to gate the sandbox feature off ŌĆö this is correct but `experimental/` paths in bazel rules tend to move; pin a specific `rules_rs` version in `MODULE.bazel` if not already.

Net: the rollout sequencing is conservative (sandbox-on for the path the team controls, opted-out for the path released downstream), the invariant test is exactly right, and the surface area is genuinely additive. Approve after a one-line comment pointing at upstream's `v8__V8__IsSandboxEnabled` source location.
