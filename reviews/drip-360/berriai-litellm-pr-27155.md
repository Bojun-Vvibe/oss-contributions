## Rationale

CircleCI's "Rerun Failed Tests" button only appears on jobs that meet three preconditions: the test step uses `circleci tests run` (not bare `pytest …`), JUnit XML is uploaded via `store_test_results`, and the JUnit emits `file`/`classname` attributes (which pytest does by default). This PR migrates every remaining pytest-based job in `.circleci/config.yml` (30 jobs total per the PR body) from the bare-`pytest` form to the `circleci tests run`-wrapping form, so the rerun button works uniformly across the suite. The mechanical pattern is the same in every job — see e.g. `langfuse_logging_unit_tests` at config.yml:350-365 going from `uv run --no-sync python -m pytest -v tests/local_testing -x --junitxml=test-results/junit.xml --durations=5 -k "langfuse"` to a `mkdir -p test-results` + `TEST_FILES=$(circleci tests glob "tests/local_testing/**/test_*.py")` + `echo "$TEST_FILES" | circleci tests run --verbose --command="xargs uv run --no-sync python -m pytest -v -x --junitxml=test-results/junit.xml --durations=5 -k \"langfuse\""` block. All original pytest flags (`-n N`, `--timeout`, `--retries`, `--cov`, `-k`, etc.) are preserved.

The two non-mechanical migrations are called out correctly in the PR body and verified in the diff: (a) `llm_translation_testing` (config.yml:560-575) previously used a `IGNORE_DIRS=("tests/llm_translation/realtime")` shell array to assemble `--ignore=...` flags; the migrated form drops the ignore array and instead filters the glob output with `grep -v "^tests/llm_translation/realtime/"` since `circleci tests run` consumes a flat file list, not directory-with-ignores. This is semantically equivalent — both forms exclude the same subtree — and the fact that the now-unused `IGNORE_DIRS` shell scaffolding is fully deleted (rather than commented out) is the right cleanup. (b) `agent_testing` (config.yml:640-650) follows the same pattern, replacing `--ignore=tests/agent_tests/local_only_agent_tests` with a `grep -v "^tests/agent_tests/local_only_agent_tests/"` filter on the glob output.

The PR body also calls out three jobs (`proxy_spend_accuracy_tests`, `proxy_build_from_pip_tests`, `db_migration_disable_update_check`) that were ran pytest but were missing `store_test_results` entirely — meaning rerun would not have worked even with `circleci tests run`. The PR adds the missing `store_test_results` step to those jobs, which is a strictly-additive correctness fix unrelated to the rerun migration, and worth keeping bundled here. Validation per PR body: `python3 -c "import yaml; yaml.safe_load(open('.circleci/config.yml'))"` parses to 44 jobs and `circleci config validate` returns valid.

Two nits worth flagging before merge: (1) The migrated form changes `tests/<dir>` (directory argument that pytest expands at runtime, including all `conftest.py` and any non-`test_*.py` files like `helpers.py` that pytest might collect via its own conftest hooks) to an explicit `test_*.py` glob computed at job-definition time. For most jobs this is fine, but if any of the migrated suites relies on pytest's default collection rules to pick up `*_test.py`-suffix files (Go-style naming, occasionally seen in Python projects), those will silently be dropped. Worth a one-line `find tests/ -name '*_test.py' -not -name 'test_*.py'` audit before merge. (2) The `xargs` invocation has no explicit `-d '\n'` separator, so any test path that contains a space or shell metacharacter will be word-split and break in confusing ways — extremely unlikely in this repo, but `xargs -d '\n' uv run …` would future-proof it. (3) `grep -v "^tests/llm_translation/realtime/"` (config.yml:572) requires the trailing slash to avoid accidentally matching a hypothetical `tests/llm_translation/realtime_helpers.py` — current tree is fine, but worth pinning a unit-test on the glob output if the suite grows.

This is a high-leverage CI ergonomics fix (per-job rerun rather than full-job rerun on a transient flake) and the diff is mechanically uniform enough that the risk surface is concentrated in the two `IGNORE_DIRS` → `grep -v` translations, both of which are correct in this PR. Merge after the `xargs -d '\n'` nit.
