## Rationale

Profile-guided fix for a real Windows hang: `gemini --resume` and `/resume` were taking 10–15 seconds to render their first paint because `getAllSessionFiles` (in `packages/cli/src/utils/sessionUtils.ts:246`) was fanning out via `Promise.all` over every `*.jsonl` in `~/.gemini/projects/<hash>/chats/`, and per file `loadConversationRecord` (in `packages/core/src/services/chatRecordingService.ts:106`) was (a) doing a synchronous `fs.existsSync(filePath)` (line 118) — which under N parallel calls all on one JS thread blocked the event loop and prevented even the `SessionBrowserLoading` placeholder from painting — and (b) streaming the *entire* JSONL through `readline` + `JSON.parse` per line just to compute `messageCount`, `firstUserMessage`, `hasUserOrAssistantMessage`, and the latest `$set` patch.

The fix has three correct moving parts. (1) **Per-session metadata sidecar** `<base>.meta.json` co-located with each chat file, containing only the listing-needed fields (`sessionId`, `projectHash`, `startTime`, `lastUpdated`, `kind`, `summary`, `directories`, `messageCount`, `userMessageCount`, `hasUserOrAssistantMessage`, `firstUserMessage`, plus a `version: 1` for forward migration). Written atomically by `ChatRecordingService` on init / `pushMessage` / `updateMetadata` / `rewindTo`, and removed alongside the `.jsonl` in `deleteSessionAndArtifacts`, `deleteCurrentSessionAsync`, and `cleanupExpiredSessions`. (2) **Synchronous existsSync drop** in `loadConversationRecord` — the `for await (const line of readline)` already errors via the existing `try/catch`, so `ENOENT` is now silently swallowed in the catch to keep log parity. (3) **Lazy migration**: read path in `getAllSessionFiles` prefers the sidecar; on miss / malformed / version mismatch it falls back to the existing JSONL stream parse and opportunistically backfills the sidecar. No upgrade step needed.

The retention-cleanup pass at `sessionCleanup.ts:131` correctly opts *out* of lazy migration via the new `lazyMigrate: false` flag — exactly right, because cleanup is about to delete the session anyway, so writing a fresh sidecar would be wasted I/O. The new `unlinkSidecarIfPresent` helper at `sessionCleanup.ts:56-69` correctly tolerates `ENOENT` (sidecar absent → success) and warns-but-doesn't-fail on other errors, with the call sites at line 218 and line 245 both wrapping the `.jsonl` `unlink` and the sidecar `unlink` together. The new `getSessionMetadataSidecarPath` helper exported from `@google/gemini-cli-core` keeps the path computation in one place.

Test coverage at `sessionUtils.test.ts:619-822` is genuinely strong — five new cases pin the discrimination: (a) sidecar present with version=1 is the source of truth even when the chat file is empty/unreadable; (b) sidecar missing → fall back to JSONL parse and backfill, with `expect(sidecarContent.version).toBe(1)` confirming the migration write happened; (c) sidecar with `version: 999` (unknown) is correctly ignored and falls back; (d) sidecar with `kind: 'subagent'` is correctly excluded from the listing (subagents aren't user-facing sessions); (e) the sidecar file itself is not surfaced as a session entry. The core service test at `chatRecordingService.test.ts` (+119 lines) exercises the write side of the same contract.

Two nits: (1) The atomic sidecar write is described in the PR body as "tmp + rename" — the actual implementation isn't in the visible diff snippet but the pattern is the right one; worth confirming the rename target is on the same filesystem as `~/.gemini/projects/...` (Windows `%LocalAppData%` is normally on the same volume as the temp file the user uses, but if anyone overrides `TMPDIR` to a different drive, `fs.rename` becomes `EXDEV` and the atomic write silently degrades to a copy-then-delete). (2) The PR's checklist still has Windows validation unchecked (`[ ] Windows`) — for a fix whose entire motivation is a Windows-specific hang, getting at least one Windows maintainer to time `--resume` before/after on a 30-session history would close the loop. The macOS `7214/7214` and `6750/6750` test counts are encouraging but don't exercise the actual perf claim. (3) The `firstUserMessage` field in the sidecar is a string; if a user's first message is large (a pasted file or long prompt), the sidecar grows proportionally. Worth a length cap (e.g., truncate to 256 chars for listing purposes) to keep the sidecar tiny.

This is the right fix for the right cause, the lazy-migration design means there's no upgrade step for existing users, and the test discrimination is exactly what's needed to prevent regressions on the fast path. Approve after the Windows perf measurement and the `firstUserMessage` truncation nit.
