## Rationale

This is a clean reproduction of an established pattern (commit `2c41f3c29` did the same thing for the `policies` field) applied to a real cross-PR collision: PR #16034 made the key edit form *always* submit `allowed_routes` in its update payload, and PR #25445 added `_check_allowed_routes_caller_permission` on the backend that rejects any non-admin attempting to *set* `allowed_routes`. The intersection means a team admin clicking "Save" on a key without touching `allowed_routes` gets a 403 even though they changed nothing — backend reproduction in the PR body confirms (`curl /key/update with allowed_routes: ["llm_api_routes"] → 403`, same call without the field → 200).

The fix is at `key_info_view.tsx:202-205`: before calling `keyUpdateCall`, if `areStringListsEqual(formValues.allowed_routes, currentKeyData.allowed_routes)` returns true, `delete formValues.allowed_routes` so the field is omitted from the wire payload entirely. The two helper functions at `key_info_view.tsx:52-72` are right-sized for the job — `normalizeStringList` accepts both array and comma-separated-string shapes (handles `[]`, `["a","b"]`, `"a,b"`, and `"a, b "` uniformly by trimming and dropping empties), and `areStringListsEqual` does an order-sensitive element-by-element comparison after normalization. Order-sensitive is the right choice here because the backend treats `["a","b"]` and `["b","a"]` as two distinct values for permission-check purposes; if they were treated as set-equal, this fix would silently swallow legitimate reorderings.

The test coverage at `key_info_view.test.tsx:780-867` pins three scenarios that together prove the discrimination: (a) `allowed_routes: ["management_routes"]` submitted unchanged → `keyUpdateCall` invoked with `expect.not.objectContaining({ allowed_routes })` (the bug-fix path), (b) `allowed_routes: []` submitted when the key already had `[]` → also stripped (the previously-empty-and-still-empty no-op case, which would otherwise round-trip an empty array and trip the same backend check), (c) `allowed_routes: []` submitted when the key had `["management_routes"]` → kept in the payload as `expect.objectContaining({ allowed_routes: [] })` (the genuine clear case must survive). The third test is the critical one — without it, an over-eager strip would silently make the "clear all routes" UI button a no-op, which would be a worse UX bug than the original 403.

Two nits: (1) `normalizeStringList` at `key_info_view.tsx:52-65` correctly handles the array and string shapes but silently returns `[]` for any other input (number, object, null, undefined). For comparison-of-existing-vs-new this is the right behavior (any non-list/string is "no routes"), but if the form ever surfaces a `null` from the backend distinct from `[]` (e.g., "field never set" vs "field explicitly empty"), this fix collapses both into the same equivalence class. Worth confirming the backend's `KeyResponse` schema treats `allowed_routes: null` and `allowed_routes: []` identically — if it doesn't, this fix needs a tri-state. (2) The `delete formValues.allowed_routes` at line 204 mutates the caller's object in place; the surrounding code does this for other fields too (the `policies` precedent does the same), so this is internally consistent, but a defensive copy at the top of `handleKeyUpdate` would make this function easier to refactor later. Not a blocker.

The fix is small, the precedent is documented in the PR body, the failure mode is real and reproduced with curl, and the test coverage hits both the strip-correct and keep-correct sides of the discrimination. Merge after the `null`-vs-`[]` confirmation.
