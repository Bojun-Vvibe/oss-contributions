## Summary

Closes a real UX bug in the goose2 frontend where session replay was rebuilding all messages with the *current wall-clock time* instead of the original `created` timestamps from the saved Message records, and was inventing fresh UUIDs for the displayed messages instead of using the original message IDs. Result was that re-opening a session showed every message timestamped "now" with random IDs that didn't correspond to the persisted `Message.id` — breaking deep-links, sort-by-time UIs, and any frontend code that wanted to correlate ACP replay events back to the on-disk record. Fix attaches `{messageId, created}` metadata to every replay-emitted ACP notification (user/assistant chunks, tool calls, tool results, agent thoughts) on the Rust side, and threads that metadata through the TypeScript notification handler into the rebuilt message records.

## Reasoning

This is the right shape for a replay-fidelity fix: identify the lost data (original `created` and `id` on the persisted `Message`), pick a clean wire-shape to carry it (`_meta.goose.{created, messageId}`), centralize the extraction on both sides (`replay_message_meta` builder in Rust, `acpReplayMetadata.ts` reader in TS), and pin the contract by exact-value test assertions on both sides of the wire. The merge-vs-replace distinction (`replay_message_meta` for fresh sites, `merge_replay_message_meta` for sites that already had goose meta) shows the author thought through the integration with the existing `mcpApp` payload path.

Three nits: (a) the `merge_replay_message_meta` helper takes `Option<Meta>` by value but only consumes it on the `Some` branch — taking `Option<Meta>` and unconditionally `unwrap_or_default()` works but a `&mut Option<Meta>` or a builder pattern would be slightly more efficient on the cold path. (b) The `.meta(...)` chain at the chunk/tool-call construction sites silently coerces every `replay_message_meta` call into the new `_meta` field; if any future ACP spec extension adds a top-level `_meta.protocol` field, the chunk constructor's existing meta-merge behavior needs verification — worth a maintainer comment. (c) The Rust `Message.created` is `i64` unix seconds in this codebase; the `serde_json::json!(message.created)` serialization emits a JSON number which the TS side then `* 1000`s — works correctly but the seconds-vs-milliseconds boundary should be commented explicitly at one of the two sides so a future maintainer doesn't accidentally double-convert.
