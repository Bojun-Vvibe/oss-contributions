## Summary

Closes #3454 by hoisting the `@`-picker's two synchronous main-thread bottlenecks — the `fdir` recursive crawl in `crawler.ts` and the `new AsyncFzf(allFiles, …)` index construction in `fileSearch.ts` (the constructor is synchronous despite the name) — into a `worker_threads` worker. On home-directory or 100k-file monorepo trees the CLI previously froze the Ink render loop for 1–9 seconds on first `@` press; this branch keeps the cursor blinking and prompt input responsive throughout. Also wires up a bundled-ripgrep fast path (`rg --files`) that wins by 3–4× past ~50k files (gated by `QWEN_FILESEARCH_USE_RG=0` to fall back to fdir), pre-warms the index at CLI boot in `AppContainer.tsx`, and fixes a stack of self-audit issues (zombie worker on transient crawl error, dispose-time `whenReady()` waiter rejection, monotonic refresh tokens, worker error→synthetic-exit, `parentPort.close()` over `process.exit(0)` for queue draining, `.gitignore` fingerprint in the singleton key, `FileSearch.dispose?()` interface addition, worker module-init wrapping, picomatch compile-error degradation to empty results, and the `prepare-package.js` whitelist for the worker file).

- New `packages/core/src/utils/filesearch/fileIndexCore.ts` (+180): pure class holding the crawl, fzf index, and result cache. Worker-safe (no main-thread imports) and unit-testable without spawning a real worker — the right architectural split because the fzf+fdir logic itself is pure data work and shouldn't be coupled to the worker boundary. Test arm at `fileIndexCore.test.ts` (+107) exercises it directly.
- New `packages/core/src/utils/filesearch/fileIndexWorker.ts` (+131): thin `parentPort` pump wrapping `FileIndexCore`. Module-init is wrapped in `try/catch` per the self-audit list, so picomatch compile errors or missing-binary errors degrade to empty-result responses rather than throwing into the UI thread. - New `packages/core/src/utils/filesearch/fileIndexService.ts` (+586): main-thread singleton keyed by `(projectRoot + ignore-fingerprint + options)` that owns the worker, multiplexes reqId-based `search` calls, fans out streaming partial updates, and disposes stale-key instances when `.gitignore` edits or cwd changes produce a new key. The `optionsKey()` including the `.gitignore`/`.qwenignore` fingerprint is the load-bearing invalidation correctness fix — without it, edits to ignore patterns would silently keep returning stale results from the cached singleton until process restart. - New `packages/core/src/utils/filesearch/ripgrepCrawler.ts` (+387): `rg --files` backend using the bundled ripgrep binary. The default-on-rg with `QWEN_FILESEARCH_USE_RG=0` opt-out is the right shape: the slow case is the one users actually feel, and small-tree perf is well under the 200ms loading threshold either way.

This is the load-bearing UX fix because a cold first-`@` press would otherwise still pay the full crawl cost (now off the main thread, but still latency-visible). ## Reasoning

This is a substantial well-architected perf PR closing a real frozen-UI bug on the most-used CLI feature (`@` picker). Architecture is right: pure `FileIndexCore` separates concerns from the worker boundary, `FileIndexService` owns the singleton lifecycle and stale-key invalidation, the worker is a thin pump. The five rounds of self-audit fixes documented in the PR body show the author hardened against real failure modes (zombie worker, lost waiters, mid-search refresh, worker crash, `process.exit(0)` losing queued sends, stale ignore fingerprint, missing dispose, init failure, picomatch compile errors, missing-from-tarball). The 200ms loading-spinner threshold and the pre-warm together solve both axes (cold-start slowness and feel-fast UX). The opt-out env var (`QWEN_FILESEARCH_USE_RG=0`) is the right escape hatch for users on systems where rg is broken or banned.

Concerns worth maintainer attention: (a) the testing matrix is honest — only macOS validated personally, Windows/Linux marked with ❓ — but a worker_threads + esbuild+ripgrep+sandbox change of this size needs at least one Windows confirmation before merge given the explicit Windows worker spawn latency callout. (b) The bundled-ripgrep approach ships a native binary in the npm tarball — worth confirming the bundling story handles all four platform/arch combos (macos-x64, macos-arm64, linux-x64, win-x64) without bloating the install size unacceptably; the `package-lock.json` diff is mostly `peer: true` flag flips so the actual binary-bundling is presumably handled elsewhere. (c) The "first time the patched workflow actually runs end-to-end will be the next dev/RC release" risk pattern from #26966 also applies here — a sandbox-permitted-or-not arm for worker thread creation is flagged but not tested. (d) No explicit benchmark regression-detection — if a future PR accidentally re-introduces a synchronous main-thread call into the at-completion path, nothing in CI would catch it.

Nits: the `ripgrepCrawler.ts` size (+387) suggests it could be split further (binary discovery vs invocation vs streaming-output parsing); the `Composer.tsx` removal of `ConfigInitDisplay` is a real UX change ("Initializing…" banner is gone) that's worth a separate one-line callout in release notes.
