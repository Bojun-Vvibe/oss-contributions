## Summary

Closes #26916 — the Anthropic→OpenAI Responses adapter was replaying prior assistant `thinking` blocks (Claude's hidden reasoning state) as visible OpenAI `output_text` items, which can leak hidden reasoning text back into a later OpenAI request and alter model behavior. Fix omits both `thinking` *and* the previously-untreated `redacted_thinking` block types from the Responses input transformation, preserving adjacent visible text and existing tool-use translation. Bundled in the same PR is a deterministic-operation-id fix for the lazy openapi snapshot generator (a semi-related concern that should arguably be its own PR).

## Reasoning

The thinking-block fix is a real correctness/privacy issue — leaking Claude's hidden reasoning into a subsequent OpenAI turn can cause the OpenAI model to either (a) parrot reasoning text into its visible output, or (b) be steered by reasoning text the user never intended to expose. The fix is minimal (one elif branch) and the test coverage is good (three arms: thinking-with-text, thinking-only, redacted-thinking-with-text). The PR body's PYTHONPATH-prefixed local pytest invocation (`83 passed, 2 warnings`) is the right verification shape for a targeted change.

The bundled `_lazy_openapi_snapshot.py` operation-id determinism fix is the concerning part. It's a real bug (FastAPI's auto-id picks first-registered method nondeterministically) but it's *unrelated* to the Anthropic-thinking-block issue, will pollute git blame for anyone investigating either change later, and — more importantly — uses `methods[-1]` (alphabetically last) where the near-simultaneous #26956 used `methods[0]` (alphabetically first). One of these two choices needs to win consistently before either ships, otherwise the snapshot will flip between PRs.

Nits: (a) split the `_lazy_openapi_snapshot.py` change into its own PR; (b) reconcile the `methods[0]` vs `methods[-1]` choice with #26956; (c) the inline comment `# Anthropic thinking blocks are hidden reasoning state` is good — consider also adding a `// see #26916` reference so future maintainers can find the original incident report; (d) `redacted_thinking` should arguably also be added to any tool-use serialization paths in case Claude ever emits redacted blocks alongside tool calls — worth a grep to confirm no other call site is writing them through.
