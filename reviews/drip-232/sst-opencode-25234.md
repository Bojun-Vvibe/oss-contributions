## Summary

Pure documentation correctness PR. The published SDK docs at `sdk.mdx` were referencing the legacy v1 SDK on every code sample and on the `typesUrl` link, even though `@opencode-ai/sdk/v2` is the supported import path and the v1 surface is being retired. Bug also catches three concrete API-shape drift items the v1 docs were silently lying about: the `file.read` response discriminator (was `raw|patch`, actually `text|binary` in v2), the permission-reply method name (was the autogenerated `postSessionByIdPermissionsByPermissionId`, actually `permission.reply` in v2), and the `typesUrl` pointing at `packages/sdk/js/src/gen/types.gen.ts` instead of the v2-generated `packages/sdk/js/src/v2/gen/types.gen.ts`.

## Specific code references

- `sdk.mdx:7`: `typesUrl` redirect from `packages/sdk/js/src/gen/types.gen.ts` to `packages/sdk/js/src/v2/gen/types.gen.ts` — the load-bearing fix because every `<a href={typesUrl}>` link in the doc body (used in 6+ table rows) was sending readers to a stale or nonexistent v1 types file. One `const` swap fixes the entire reference graph.
- `sdk.mdx:31, :55, :75, :99`: four `import { ... } from "@opencode-ai/sdk"` → `from "@opencode-ai/sdk/v2"` swaps across the four canonical examples (`createOpencode`, `createOpencode` with config, `createOpencodeClient`, type imports). Symmetric and complete — no v1 import is left in the doc.
- `sdk.mdx:321`: the canonical fix that catches the actual API change — `postSessionByIdPermissionsByPermissionId({ path, body })` table row replaced with `permission.reply({ path, body })`. The old name was the autogenerated openapi-fetch shape that v2 has properly grouped under the `permission.*` namespace; the doc was telling readers to call a method that doesn't exist on the v2 client.
- `sdk.mdx:363`: `file.read` return-type discriminator union changed from `{ type: "raw" \| "patch", content: string }` to `{ type: "text" \| "binary", content: string }`. This is the second actual API correctness fix — `raw` and `patch` are not valid v2 discriminators, and a consumer pattern-matching on the doc-stated union would silently fall through to the default arm forever on real responses.

## Reasoning

Three things make this a clean docs PR rather than a mechanical find-replace: (a) the `typesUrl` swap is the right shape — the rest of the doc references it via `<a href={typesUrl}>` rather than hardcoding the path, so one constant flip propagates everywhere; (b) the two API-shape fixes (`permission.reply`, `file.read` discriminator) are real semantic corrections rather than just import renames, which is the kind of drift that accumulates silently in docs and bites integrators; (c) PR is precisely scoped — single file, no peripheral changes, no "while I'm here" formatting churn.

The one thing missing is a corresponding redirect or callout for users still on the v1 SDK who land on this page from a search result — the doc now reads as if v2 is the only SDK, but the v1 package is presumably still installable from npm. A one-line "If you're still on v1, see [link]" header would prevent confused-user issues from filing duplicates of the bug this PR closes.
